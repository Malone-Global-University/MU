WORKFLOW NOTE

Title
Bottom Navigation CSS Conflict Resolution

Date
2026-05-17

Area
Website navigation system; bottom page navigation component; CSS modularization

Problem
The bottom navigation section at the bottom of the page was rendering incorrectly. The “Continue Browsing” heading and navigation buttons were appearing on the same horizontal row, causing layout distortion and inconsistent spacing. The visual structure of the bottom page navigation no longer matched the intended centered stacked layout.

Intended Goal
The bottom navigation section should render as a clean centered block with:

* The “Continue Browsing” heading on its own line
* Consistent spacing between buttons
* Proper alignment and wrapping behavior
* Independent styling isolated from the main navigation system

Root Cause
The `.bottom-nav` selector was defined in two separate CSS files:

* `09-bottom-nav.css`
* `03-navigation.css`

The later-loaded rules in `03-navigation.css` converted `.bottom-nav` into a flex container, unintentionally affecting all child elements including the `<h2>` heading. This created a cross-file cascade conflict and violated CSS component isolation principles.

Steps Taken

1. Reviewed the HTML structure for the `.bottom-nav` component and inspected rendered layout behavior from the provided screenshot.

2. Compared `.bottom-nav` definitions across:

   * `09-bottom-nav.css`
   * `03-navigation.css`

3. Identified conflicting layout instructions:

   * One file used standard block layout
   * The other forced flexbox layout

4. Determined that the flexbox rules in `03-navigation.css` were overriding the intended component behavior.

5. Recommended removing all `.bottom-nav` styling rules from `03-navigation.css`.

6. Consolidated `.bottom-nav` ownership into `09-bottom-nav.css`.

7. Rebuilt the bottom navigation styles with:

   * Centered section layout
   * Independent heading styling
   * Inline-block button layout
   * Stable spacing and sizing rules
   * Accent-colored heading underline

Key Change
The `.bottom-nav` component was isolated into a single dedicated stylesheet to eliminate cross-file cascade conflicts.

Outcome
The bottom page navigation system now renders predictably with:

* Proper heading placement
* Correct button alignment
* Improved spacing consistency
* Cleaner component separation
* Reduced CSS override risk

The layout behavior is now stable and easier to maintain across the MGU site architecture.

Validation
Validation was performed by:

* Comparing the broken rendered screenshot against the CSS rules
* Tracing selector overlap between files
* Confirming that removing the flexbox override restores intended vertical layout behavior
* Verifying that the revised styles preserve button alignment without affecting unrelated navigation systems

Rules Going Forward

* Each reusable UI component should have a single authoritative stylesheet.
* `03-navigation.css` should only manage primary/global navigation systems.
* `09-bottom-nav.css` should exclusively control `.bottom-nav`.
* Avoid redefining the same component selector across unrelated CSS modules.
* Layout systems such as flexbox should not be globally applied to reusable component containers unless intentionally scoped.

Notes
This issue exposed an emerging modular CSS maintenance concern within the MGU architecture. As the platform grows, selector ownership and stylesheet boundaries become increasingly important to prevent silent cascade collisions. Future refactors may benefit from adopting stricter component naming conventions or layered CSS architecture practices.
