WORKFLOW NOTE

Title
Stabilized DOCX Compile-Script Pipeline for Chapters 1–28

Date
2026-05-09

Area
MGU High School Physics Textbook / DOCX chapter production / shared compile-script workflow

Problem
The early textbook chapter workflow was inconsistent. Some chapters were created from Markdown, some from JavaScript DOCX compile scripts, and some required troubleshooting after links, rendering, or layout failed. Several DOCX files appeared longer than expected because tables or bordered blocks rendered incorrectly, creating stretched pages, large blank vertical areas, or unusable table layouts. Chapter 19 and Chapter 15 were the clearest examples: the source scripts were structurally valid, but the delivered DOCX output had malformed rendering in Word.

There was also uncertainty about whether each chapter required a recreated `lib.js`, whether verification needed to be decided case by case, and whether Claude’s changing output batch size meant the chapter content was drifting or becoming too thin.

Intended Goal
Create a repeatable, mechanical pipeline for producing editable DOCX chapter handouts from Claude-generated JavaScript compile scripts. Each chapter should use the same shared `lib.js`, compile quickly, render cleanly, preserve the MGU textbook style, and deliver a reliable short DOCX file plus ZIP fallback. The assistant should not redesign the document, rewrite the library, or reinterpret the chapter unless an actual compile or layout failure occurs.

Root Cause
The workflow was still evolving. Early chapters involved mixed source types, including Markdown and compile scripts. The shared helper library had not yet been treated as a fixed build dependency, so there was too much file-by-file reasoning about whether to rebuild `lib.js`, how to verify output, and whether each table/layout issue came from source content or the rendering process.

The major layout failures were not caused by malformed chapter scripts. They were caused by bad or inconsistent builds/rendering against the document helper setup. Once the same source scripts were recompiled with the stable shared library, the malformed stretched-page issue disappeared.

Steps Taken

1. Established the compile-script model as the preferred production method for the high school physics textbook.

2. Confirmed that Claude’s chapter scripts are semantic document builders, not final prose-only manuscripts. The scripts call helper functions such as `chapterTitle`, `sectionHead`, `bodyP`, `formulaP`, `makeTable`, `warningBox`, `noteBox`, `stepLine`, `keyTermDef`, and `makeDoc`.

3. Identified and preserved the shared `lib.js` file as the stable document-build library for the project.

4. Created an offline MGU DOCX Compile Kit containing `lib.js`, `package.json`, `README.md`, and `example_chapter_script.js`.

5. Standardized the assistant-side compile workflow: copy the uploaded `chXX.js` script into a clean work folder, use the existing shared `lib.js`, patch only the output path, run Node, and create the finished DOCX.

6. Standardized verification: check DOCX ZIP/package integrity, render the DOCX to page images, create a numeric-order contact sheet, inspect the visual layout, and deliver the DOCX only after the render passes.

7. Corrected the contact-sheet process so page images are ordered numerically, not lexicographically.

8. Standardized delivery names so the short chapter filename is the primary download link, such as `ch23.docx`, with `ch23_docx.zip` as the fallback.

9. Rebuilt problematic chapters from source scripts instead of trying to manually repair broken DOCX files.

10. Recompiled Chapter 19 from the uploaded script using the stable shared library after a badly malformed DOCX showed stretched blank pages and garbled tables.

11. Confirmed the corrected Chapter 19 opened cleanly in Word and rendered at the expected page count, approximately 15 pages in the user’s Word environment.

12. Recompiled Chapter 15 using the same stable-library method after a similar prior malformation concern. The corrected Chapter 15 rendered cleanly at 16 pages.

13. Continued compiling later chapters using the stabilized process: Chapters 23, 24, 25, 26, 27, and 28 were compiled successfully from JavaScript scripts into DOCX files with clean layouts.

14. Confirmed that sandbox render page counts may differ from Microsoft Word page counts, and that local Microsoft Word pagination is the authority for final page count.

15. Confirmed that the chapter content is consistent in structure and depth after formatting errors are corrected. Most chapters are landing in a normal usable range, commonly around 15–17 pages in Word for later chapters.

Key Change
The project moved from a file-by-file experimental conversion process to a stable deterministic DOCX build pipeline using a shared `lib.js` library.

Outcome
Chapters 1–28 now have a clearer production history and a stable forward workflow. The DOCX chapters are editable, visually consistent, and useful as complete handout-style first drafts. The pipeline is now fast enough to support repeated chapter production, with recent compile times dropping to roughly under a minute for clean scripts.

The biggest improvement is reliability. Instead of deciding how to handle each chapter from scratch, the assistant now treats each uploaded `chXX.js` as input data for the same build system. The document design remains consistent because `lib.js` controls the typography, headings, tables, callouts, formulas, captions, key terms, problem sets, headers/footers, and page structure.

Validation
Validation was performed through several layers:

The JavaScript scripts were compiled with Node using the shared `lib.js`.

The resulting DOCX files were checked as valid ZIP/DOCX packages.

The DOCX files were rendered into page images.

Numeric-order contact sheets were created for visual inspection.

Malformed builds were identified by visible symptoms: stretched blank pages, excessive page count, table layout failure, and long vertical borders.

Chapter 19 was rebuilt and confirmed clean in Microsoft Word at 15 pages.

Chapter 15 was rebuilt and confirmed clean at 16 pages.

Chapters 23–28 compiled back-to-back with clean rendered layouts and successful DOCX delivery.

The user confirmed that recent files opened correctly and rendered perfectly in Word, even when sandbox render page counts differed.

Rules Going Forward
Use the existing MGU DOCX Compile Kit and shared `lib.js`.

Do not recreate `lib.js` unless there is a real missing-helper failure.

Do not redesign the document during compile.

Do not reinterpret the chapter content unless the source script fails.

Patch only the output path to produce `/mnt/data/chXX.docx`.

Always check DOCX package integrity.

Always render the DOCX to page images.

Always create a numeric-order contact sheet.

Inspect the contact sheet before delivery.

Deliver the short DOCX filename as the primary link.

Provide a ZIP fallback.

If a DOCX opens with stretched pages, blank-page malformation, or garbled tables, rebuild from the source script using the stable shared library before attempting any manual repair.

Treat Microsoft Word pagination on the user’s computer as the final page-count authority.

Notes
The compile-script workflow is preferred because it is more efficient than asking Claude for full Markdown chapters. Claude can often generate two or three compile scripts for roughly the same token cost as one full Markdown chapter, making the process approximately 200–300% more efficient for building a complete first-draft textbook.

The DOCX handouts are not necessarily the final textbook form. They are a “thin but good” complete manuscript layer. Later, selected sections can be expanded manually into fuller Markdown or semantic HTML. DOCX can also be converted to Markdown using Python, Pandoc, or another converter without requiring AI to rewrite the chapter.

The pipeline is now stable enough to continue beyond Chapter 28 using the same process.
