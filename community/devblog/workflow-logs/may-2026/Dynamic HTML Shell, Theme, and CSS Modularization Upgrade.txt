WORKFLOW NOTE

Title
Dynamic HTML Shell, Theme, and CSS Modularization Upgrade

Date
12:50 PM 5/8/2026
Area
Sitewide HTML structure, shared components, navigation, footer, breadcrumbs, and CSS architecture

Problem
MGU pages relied too heavily on repeated hand-coded headers, footers, breadcrumbs, bottom navigation, and large CSS files. This made sitewide changes difficult, increased the risk of broken navigation, and made renamed or moved pages harder to maintain.

Intended Goal
Create a scalable page system where shared HTML sections can be controlled from central component files, page-specific dates can be preserved, breadcrumbs can become more flexible, and CSS can be split into smaller theme-aware files.

Root Cause
The site had grown large enough that manually maintaining repeated layout code across thousands of pages was no longer practical. Shared interface elements were mixed into individual pages instead of being treated as reusable site infrastructure.

Steps Taken

1. Built a new HTML shell using placeholder containers for shared sections.
2. Added JavaScript fetch logic for loading the header, footer, bottom navigation, and breadcrumb areas.
3. Created reusable HTML partials for `header.html`, `footer.html`, and `bottom-nav.html`.
4. Moved page-specific upload and update dates into `data-uploaded` and `data-updated` attributes on the `<body>` tag.
5. Updated the footer so the visible timestamp is injected from the page’s own body attributes.
6. Split the main CSS into smaller modular files by function, including theme, base, header, navigation, content, tables, utilities, responsive rules, breadcrumbs, bottom navigation, and hero styling.
7. Created separate primary theme files for major MGU areas: Department, Library, Community, News, and Main.
8. Used color-coded themes to visually separate major site domains and reduce navigation confusion.
9. Reorganized Community and Malone Panther areas so clubs, forums, public discussion, and newspaper-related content have clearer lanes.
10. Standardized the page structure around hero, breadcrumb, on-page navigation, overview, bottom navigation, and footer sections.

Key Change
Shared page structure was moved from manually repeated page code into centralized reusable components supported by modular CSS and domain-specific theme files.

Outcome
MGU now has a more scalable front-end structure. Headers, footers, breadcrumbs, bottom navigation, timestamps, and visual themes can be managed more consistently across thousands of pages. The site is easier to maintain, easier to expand, and clearer for visitors moving between Departments, Library, Community, News, and Main pages.

Validation
The injected header and footer loaded correctly, the theme toggle remained connected to the loaded header, the footer displayed page-specific dates from body attributes, and the themed CSS files successfully changed the visual identity of different site areas.

Rules Going Forward
Shared navigation and layout elements should be centralized whenever possible. Page-specific content should remain in the page. Dates should be stored on the page body, not hardcoded inside the shared footer. Major domains should keep their own theme identity. CSS should remain modular and organized by function to simplify troubleshooting.

Notes
This marks a major transition from individually maintained static pages toward a lightweight, scalable publishing system. Future cleanup should reduce repeated fetch logic, refine breadcrumb generation, and continue testing bottom navigation behavior on desktop and mobile.
