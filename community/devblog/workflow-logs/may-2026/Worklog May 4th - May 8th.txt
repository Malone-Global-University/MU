The purpose of themes is to solve navigation problems. The Department and MGU Library are structured similarily. To avoid confusion as to whether a person in at a lesson page from the Departmental courses or at an extended research page at the Library themes are used to color code and made instantly clear which domain you are currently in. This solves getting lost bouncing between the two and adds visual interest.


The main pages are the same color as they always were while Community and the School Newspaper recived new colors. The non-teasl areas have been promoted with news themes to ease the expected dramatic growth about to take place. As the site scales having colored domians prevent data sprawl. This is a very intelligent design that is exceptionally simple and devastatingly effective. For example may areas that were in Community were designed as future commentary boards have been moved to the Newspaper. This separates the types of discussions the site will host creating clearly marked lanes. Both the Civic center (student topics) and the Commons (public chat) used to be under Community but are now under Forums in the Malone Panther. They will host discussions about news events as newspapers do. This makes the Newspapers more active and much larger.


Taking its place is the new section titled "MGU Clubs". Clubs are extracurricular organizations that students and faculty create to roundtable ideas and promote projects. This gives social representation and accountability a place to exist on the site. People make clubs to share and promote their interest.


As MGU grows more clubs will appear cateering to various hobbies and interests. By separating Forums and Clubs both areas can scale without tangling into each other.


2. Body - The body is the main container for the entire page. I added a timestamp attribute tot he body tag. This solves the problem of accidently overwriting upload dates. When converting to dynamic footers all pages get the same footer which erases the dates. To fix this each page has a unique date serial id in the body tag which harcodes the correct dates. Additionally I will start keeping a database for when pages wewre born, updated, and had maintenance performed. That will give a second source of published artifact for provenance.


3. Header - Headers, footers, and the body are the three main parts of any html document.

The main Header on MGU is a combined header + top navigation menu + school branding (logo).
This must be consistent on every page. When adding or changing the header it is tedious to copy and paste updated html across hundreds of pages. At this moment MGU has over 3,000 pages. Most are seeds for planned content. Before scaling to over 10,000 mature pages there needs to be a way to upgrade the header across all pages simultaneously. 

There are three good options:

Option 1: Use a node.js cms pipeline and regenerate all pages once a new update happens. This is local and deterministic using few resources. 

Option 2: Use a lightweight JS script and inject only what is deterministic and handcode everything that is actual content.

Option 3:  Build all pages from metadata.


Without getting too technical option 2 was chosen because it was simplest and sits between the other options in terms of scalability. Option 1 will hit a upper ceiling around 5,000-8,000 pages.  Option 3 is the best for over 30,000 pages but right now would be too much code to maintain. Prototypes of option 3 exist but deploying them now would be premature for the scope of this website. When the site reaches enough maturity to warrant option 3 I will deploy and document the process as a learning opportunity.



4. Hero - The "Hero" is a banner that extend 100% the width of the screen. In MGU it is used to dispay the title for the page Using a h1 heading. 5. Breadcrumbs have been updated (same as the header, footer, and bottom nav) with place holder divs for JavaScript injected HTML. This method of injecting HTML is probably the simplest and my favorite beacause it is so lightweight. If the code were heavier (more lines or more complicated logic) it likely would: slow page load times; and complicate troubleshooting (running extra code called "exceptions" everywhere). Breadcrumbs are another element that needs to be injected because it is deterministic (that word again). "Determinisitic" means that from a single set of inputs you get a guarenteed output. This means that a program is taking code that a human created and converting it into another file type, like converting markdown or a word .docx into a PDF. The opposite would be if you used AI to write a report for you, you would get an unknown output that is random (we do not know what AI will say beforehand so we cannot guarentee results). 

Breadcrumbs need to be deterministic, that is dynamically generated, because pages can move to different folders or get renamed. This alters urls in the breadcrumbs. Dynamically creating breadcrumbs uses scripting languages to write the hierarchy of route for you. The single set of "inputs" is {root, subdomain ... your_file.} This is a specific path that programs like python or javascript can read and then write the actual html for a pages breadcrumb links. So if filename paths change the scripting language will check routes as a page is loading and then write the html block before it finishes loading. Compared to AI powered site this is a "dumb" feature. Deterministic implies that no decisions need to be made by either a human or AI agent, the instructions for how to proceed are already determined. 

Tradeoffs from overusing scripts are slow load times. The scripts MGU uses are fast loading, however when loading a page you will see a split second delay where the page loads first and then the header, or other element load second. This is okay since the delay is minimal and the parts that lag are usually navigation items. You are not likely to go to a page just to immediately go to a different one. That is just poor surfing habits.

The main benefits of using dynamic loading is that less time is spend maintaining code and more time is spent creating new lessons and other content. Obviously content gives more value to users than code maintenance so the bias towards dynamic scripting is justified.

6.  On this page - Often at the top of deeply written pages there exists a "on this page" blcok for jumping to different sections of the current page. While not the most elegant title it is the eaziest to standardize across most pages. Focus on this point should be logic over aesthetics. The more standardized the site is the more stable, and the more stable the more content can be safetly introduced. Some pages do not need this navigation block because of short pages. For longer pages they are very useful. Think of this as the table of contents to the page. 

7. Overview - This is another standardization section. Previous introdcutory sections varied from page to page, were hand written, and more custom to the topic. History would have "About the Past", etc giving each page a personal touch. That is now sadly gone. Too many competing formats made scaling for many pages impossible. I have to write CSS for elements and that is hard to do when they are diverging. Plan: use generic titles for now, scale site, then bring customized title back. In a phyiscal school each classroom would indeed have quirky notes and drawings to reflect each teachers personality. On a website some of that gets sacrificed for clarity. On a corporate website it is all but gone to give a serious, consistent, authoritative tone across all pages. Like a textbook it is written in one voice to avoid ambuiguity. 

8. Bottom Nav - This is a group of links that exit the page and go to higher subdomains. These varied by department and even by page in terms of style, placement, verbiage, number of links, and were they linked to. The trusa issue is that some display good on web but bad on mobile and vice versa. I do not have a complete answer for this one yet. The first step in finding a solution is to inject the same HTML and them tweak CSS until all devices show quality navigation links. Currently the code and HTML chosen is the best fit for current use. The number of links have been limite to three, and they are all pointing to #top-of-page, /department/, and /index (the home page). This allows consistent styling and predictablew behavior. Once the logic is established for a given element we can then add features or UX to make it more pleasant to use. 

9. Footer - This contains Social media links, branding, and upload/update dates. The main issue here was using footers in conjunction with headers to delineate pages and set boundaries for where content can exist. Of personal interest was including dates to track udpates and new pages when site first launched. Over time this becomes less serious. New sites often get neglected or abandoned. Update timestamps lets viewers know the site is being maintained which encoourages them to contune using it. If a site has not been updated for some time and many things are wrong with it, then it is an adandoned project and defunct. MGU is not like that. It is a serious institution that is built for permanance. The Founder compares this site to any HBCU such as Tuskgeegee. 

 Footers primarily exist to structure the page and let you know you are indeed at the end of the bottom of the page. Historically pagees on dialup loaded piece by piece and could get cut off. If you saw a footer you knew the page had loaded fully and nothing was "hidden" below. In professional writing footers display gravitas. Often the Bigger the company the bigger the footer, loaded with as many or more links and dropdowns as the main nav menu. In theory that is over the top and considered poor taste. In reality experts routinely push the limits of design for high risk, high reward. MGU maintains a sensible footer with just enough detail to quietly demonstrate its purpose. We do not support the practice of making the footer compete with the top nav, especially for an education site. We do not bury the best parts of out site in the bottom margins of our pages.
