WORKFLOW NOTE

Title
MGU Physics Chapters 1–28 DOCX/PDF-to-Semantic-HTML Conversion Workflow

Date
2026-05-09

Area
MGU High School Physics textbook production pipeline; Chapters 1–28; DOCX, PDF, Markdown, and semantic HTML conversion workflow.

Problem
The physics textbook chapters existed in multiple working forms: DOCX compile scripts, rendered DOCX files, PDFs, Markdown drafts, and HTML exports. Early HTML conversions from PDF were usable, but each chapter risked small styling drift because standalone HTML files sometimes carried their own embedded CSS or chapter-specific visual assumptions.

The larger issue was not the chapter content itself. The content was consistent because the upstream production pipeline was stable. The problem was making sure downstream HTML conversion stayed clean, semantic, CSS-independent, and repeatable across chapters.

Intended Goal
The correct state is a layered, repeatable textbook pipeline where chapter content is generated efficiently, rendered deterministically, visually checked, and converted into stylesheet-ready semantic HTML without redesigning the chapter each time.

The desired HTML output should contain clean structure only: `<article>`, `<header>`, `<section>`, `<h1>`–`<h3>`, `<p>`, `<table>`, `<ol>`, `<ul>`, `<dl>`, `<aside>`, `<figure>`, and stable class hooks for textbook elements. No embedded CSS should be included because final styling will come from shared external stylesheets.

Root Cause
The initial risk came from treating each chapter conversion as an independent document-design task. When each file is converted separately, the assistant may make small layout or CSS decisions that gradually drift across chapters.

The underlying structure, however, was already stable. The chapters were produced from a consistent AI-assisted DOCX compile-script pipeline using shared helper functions and a shared library. That means the correct approach is not to redesign each chapter, but to mechanically preserve the existing semantic structure and convert it into a clean downstream format.

Steps Taken

1. Converted Chapters 1–5 from PDF into standalone HTML to test the feasibility of PDF-to-HTML conversion.

2. Identified that per-chapter CSS could drift if each HTML file carried its own styling.

3. Shifted the conversion target from styled standalone HTML to CSS-independent semantic HTML.

4. Established the semantic HTML pattern: one chapter per `<article>`, numbered sections as `<section>`, review/problem sets as ordered lists, key terms as definition lists, tables with captions, and callouts as `<aside>` blocks.

5. Converted later PDF chapters into semantic HTML using stable class hooks such as `key-point`, `common-mistake`, `tip`, `historical-note`, `bridge-note`, `study-tip`, `worked-example`, and `the-rule`.

6. Confirmed that the PDFs were not arbitrary documents but deterministic outputs from a stable compile-script pipeline.

7. Documented that Claude is most efficient when generating JavaScript DOCX compile scripts rather than full Markdown chapters.

8. Confirmed the preferred authoring route: Claude generates `chXX.js` compile scripts, ChatGPT compiles them with the shared `lib.js`, DOCX files are rendered and checked, and DOCX/PDF/Markdown/HTML become downstream outputs.

9. Standardized the DOCX compile workflow: use the existing shared `lib.js`, patch only the output path, compile the uploaded script, check package integrity, render pages, create a numeric-order contact sheet, inspect for layout failures, and deliver the short DOCX link plus ZIP fallback.

10. Preserved the distinction that local Microsoft Word pagination is the authority for final page count, while sandbox rendering/contact sheets are QA tools.

11. Converted Chapters 6–9, 11–13, 16, 18–20, and 21–28 into semantic HTML or zipped semantic HTML outputs as the workflow matured.

12. Converted Chapter 23 separately into a zipped CSS-free semantic HTML file so it could be added to the Chapter 21–28 HTML set.

Key Change
The major correction was shifting from “generate styled HTML per chapter” to “extract and preserve stable semantic chapter structure, with no embedded CSS, using shared external stylesheets later.”

Outcome
The MGU Physics textbook now has a clearer production model for Chapters 1–28.

The working model is:

Claude generates compact DOCX compile scripts.
The shared `lib.js` renders deterministic DOCX/PDF output.
DOCX/PDF files serve as controlled manuscript artifacts.
Markdown and semantic HTML are downstream conversion formats.
HTML files remain CSS-free and stylesheet-ready.

This avoids wasting AI compute on repeated full Markdown generation or manual format conversion. It also prevents visual drift by separating content structure from styling.

Validation
The workflow was validated through repeated chapter conversions and render checks.

Chapters 1–5 proved PDF-to-HTML conversion was viable.
Chapters 6–9 and later chapters confirmed the semantic HTML pattern was repeatable.
Chapter 20 confirmed the “no CSS, semantic only” rule.
Chapters 21, 22, and 24–28 were bundled into a zipped semantic HTML delivery.
Chapter 23 was converted afterward and zipped separately.
The DOCX production workflow was validated earlier through successful compilation and QA of later chapters, especially after the Chapter 15 and Chapter 19 layout issues were fixed by recompiling against the stable shared library.
Contact sheets and rendered page images were used for visual QA, while Microsoft Word remained the final pagination authority.

Rules Going Forward
Do not redesign the document per chapter.

Do not create embedded CSS in semantic HTML conversions.

Do not let each chapter invent its own visual system.

Use the existing DOCX compile kit and shared `lib.js` for compile-script workflows.

Patch only the output path unless there is a real failure.

Treat uploaded `chXX.js` files as semantic chapter data for deterministic rendering.

For HTML conversion, preserve structure rather than style.

Use stable class hooks for callouts, examples, notes, warnings, tables, exercises, key terms, summaries, and checkpoints.

Use one shared external stylesheet later to control the appearance of all chapters.

When converting PDFs or DOCX files, expect the same chapter structure unless there is visible evidence of a real extraction problem.

Notes
The project’s current approach is practical and efficient. Full Markdown remains attractive as a clean source format, but for this project Claude produces DOCX compile scripts much more efficiently. Because the compile scripts and shared library are stable, the DOCX/PDF outputs are not chaotic conversion targets; they are controlled intermediate artifacts.

The best long-term model is layered production:

Compile script as compact semantic source.
DOCX as editable manuscript.
PDF as print/share artifact.
Markdown as optional intermediate.
Semantic HTML as web publication format.
External CSS as the design layer.

This keeps the project moving quickly while preserving the ability to later expand selected chapters into deeper textbook prose, polished Markdown, EPUB, or fully styled web lessons.
