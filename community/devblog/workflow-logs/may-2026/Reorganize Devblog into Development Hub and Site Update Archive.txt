WORKFLOW NOTE

Title
Reorganize Devblog into Development Hub and Site Update Archive

Date
2026-05-19

Area
`/community/devblog/`
`/community/devblog/site-updates/`
`/community/devblog/site-updates/2026/`
`/community/devblog/site-updates/2025/`

Problem
The old Devblog page was being used to list site updates in chronological order, but it had grown into a long screenshot-heavy archive. Each update card used an image thumbnail, including PNG screenshots, which increased page weight and made the page less practical as an ongoing weekly update system. The page also mixed the broader purpose of a development blog with the narrower purpose of a site-update archive.

Intended Goal
The Devblog should have a clean top-level landing page for development-related areas, while site updates should live in a dedicated archive section. The update archive should be text-first, grouped by year, and light enough to keep growing without loading many screenshot thumbnails on the main archive pages.

Root Cause
The original structure treated `/community/devblog/index.html` as both the main Devblog homepage and the complete update archive. The old page contained repeated `.blog-post` entries with `.blog-image` screenshot thumbnails and dated links such as `2026/5-11-2026`, `2026/4-29-2026`, and older 2025 update paths.  As the update list grew, that design became harder to maintain and heavier to load.

Steps Taken

1. Created a new `/community/devblog/index.html` as a broader Development Blog landing page.

2. Added Devblog area links for Site Updates, Design Notes, Workflow Logs, Technical Notes, Content Production, and Experiments.

3. Reinterpreted the old screenshot-based Devblog page as the source for the Site Update Archive rather than the whole Devblog homepage.

4. Created a dedicated `/community/devblog/site-updates/index.html` page for the update archive.

5. Moved the year-based archive concept under the Site Updates section.

6. Created matching year archive index pages for:
   `/community/devblog/site-updates/2026/index.html`
   `/community/devblog/site-updates/2025/index.html`

7. Converted the old screenshot-card model into lightweight text-based update listings with dates, titles, categories, summaries, and links to the existing dated update pages.

8. Preserved the existing dated-post URL pattern so older update pages can remain in place without needing a full URL redesign.

9. Verified the structure visually in the browser: the Devblog homepage now shows the broader Devblog areas, the Site Updates page shows the archive landing page, and the 2026 page shows the year-specific update list.

Key Change
The Devblog was split into a general development hub, while the chronological update archive was moved into `/community/devblog/site-updates/`.

Outcome
The Devblog now has a cleaner information architecture. Visitors can first choose the development area they want, while site updates have their own focused archive. The update archive is faster and easier to scan because it no longer depends on loading screenshot thumbnails on the archive pages.

Validation
The new pages were checked in the browser. The breadcrumb path correctly showed the hierarchy from Home to Community to Devblog to Site Updates, and the 2026 archive showed the extended path into the year folder. The Devblog landing page displayed the six development areas, while the Site Updates archive displayed year navigation and the year archive pages displayed update lists.

Rules Going Forward
Keep `/community/devblog/` as the broad Devblog landing page. Keep chronological update entries inside `/community/devblog/site-updates/`. Add new weekly updates to the correct year folder and update that year’s `index.html` in newest-first order. Do not restore screenshot thumbnails as required elements on archive listing pages; screenshots should only be used inside individual dated update posts when they add value.
