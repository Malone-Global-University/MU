WORKFLOW NOTE

Title
MGU Publishing Pipeline and Site Ledger Architecture Planning

Date
2026-05-17

Area
MGU website architecture, lesson-generation pipeline, static publishing workflow, SQLite maintenance ledger, JSON lesson schema, JavaScript rendering systems

Problem
The MGU website had scaled to roughly 11,500 mostly static HTML pages, making manual maintenance increasingly difficult. Repeated structures such as headers, footers, breadcrumbs, navigation systems, lesson metadata, and page layouts created growing maintenance debt across thousands of files. Existing workflows also required excessive direct HTML authoring, slowing educational content production and making large-scale updates cumbersome. The emerging lesson-generation prototype lacked a fully articulated architectural framework connecting Markdown authoring, JSON lesson generation, static rendering, and maintenance diagnostics.

Intended Goal
Establish a scalable static-first publishing architecture that allows lessons and educational content to be authored rapidly in Markdown, transformed into structured JSON lesson records, rendered into static HTML pages through reusable templates, and monitored through a lightweight SQLite maintenance and diagnostics ledger. The system should preserve the performance, SEO, portability, and durability advantages of static HTML while dramatically reducing manual maintenance overhead.

Root Cause
The original site architecture relied heavily on manually maintained static HTML pages where both content and structural presentation logic were tightly coupled. At small scale this approach was manageable, but at approximately 11,500 pages repetitive structural components and metadata became operational liabilities. The absence of structured content schemas, reusable generation templates, and centralized maintenance tracking caused scaling friction as the site evolved into a large educational publishing platform.

Steps Taken
1.
Evaluated the current MGU architecture and identified the transition from a simple static website into a scalable publishing platform with emerging dynamic infrastructure.

2.

Analyzed the limitations of fully static workflows at large scale, particularly the maintenance burden associated with repeated headers, footers, breadcrumbs, navigation systems, metadata blocks, and lesson structures.

3.

Reviewed the current use of dynamically loaded JavaScript components including:

* header injection
* footer injection
* breadcrumb generation
* theme toggles
* shared navigation systems

4.

Established that lightweight JavaScript enhancement is beneficial when used for reusable interface components rather than for full page rendering.

5.

Developed a preferred long-term MGU architecture centered on:

* Markdown authoring
* structured JSON lesson metadata
* reusable templates
* automated static HTML generation
* lightweight JavaScript enhancement
* SQLite-based maintenance diagnostics

6.

Compared four content-management architectures:

* static-only systems
* dynamically loaded JavaScript systems
* JSON-generated publishing systems
* MySQL database-driven CMS systems

7.

Concluded that MGU’s ideal architecture is primarily a JSON-generated static publishing system enhanced with lightweight JavaScript while preserving static HTML deployment.

8.

Reviewed the uploaded `auto.js` and `render.js` prototype workflow and identified the publishing pipeline structure as:

```text
Markdown
  ↓ auto.js
lesson JSON
  ↓ render.js
static HTML
```

9.

Reviewed the uploaded Lesson 9.2 JSON file (“Initial Public Offerings”) and identified that the schema already supports:

* lesson metadata
* unit relationships
* track identity
* breadcrumbs
* previous/home/next navigation
* table of contents generation
* structured lesson sections
* educational workflow content

10.

Interpreted the lesson JSON object as an intermediate structured publishing layer functioning similarly to a CMS record without requiring a live database.

11.

Recommended future improvements to the lesson schema including:

* metadata validation
* stable base-path routing
* centralized shared CSS
* Markdown-to-HTML conversion improvements
* separation of SEO descriptions from hero descriptions
* expanded maintenance metadata

12.

Reviewed and analyzed the lightweight SQLite maintenance ledger system intended to track:

* creation dates
* update dates
* maintenance schedules
* breadcrumbs
* canonical URLs
* Open Graph metadata
* schema markup
* MathJax usage
* word counts
* reading times
* image counts
* internal/external links
* recently modified pages
* short/stub pages

13.

Discussed CSV fundamentals and how CSV, SQLite, Node.js scanners, and maintenance reports integrate into the larger MGU publishing ecosystem.

14.

Reviewed the proposed Node.js scanner/report architecture involving:

* `upgrade-webpages-table.js`
* `scan-site.js`
* `generate-reports.js`

15.

Analyzed proposed database schema upgrades adding fields such as:

* `file_size_bytes`
* `filesystem_created_at`
* `filesystem_modified_at`
* `word_count`
* `reading_time_minutes`
* `has_breadcrumbs`
* `has_canonical`
* `has_open_graph`
* `has_schema_markup`
* `image_count`
* `internal_link_count`
* `external_link_count`

16.

Diagnosed and explained SQLite locking (`SQLITE_BUSY`) issues and documented safer operational workflows involving:

* closing DB Browser before writes
* stopping lingering Node.js processes
* using `busy_timeout`
* improving SQLite write handling with `better-sqlite3`

Key Change
The most important architectural shift was redefining MGU from a manually maintained static HTML website into a static-first automated publishing platform built around Markdown authoring, structured JSON lesson schemas, reusable rendering templates, and SQLite-based maintenance intelligence.

Outcome
The MGU architecture was clarified into a scalable layered system that separates:

* content authoring
* structured lesson metadata
* rendering logic
* interface enhancement
* maintenance diagnostics

The publishing pipeline now has a clearly defined conceptual model supporting future automation, faster lesson production, centralized maintenance, scalable navigation systems, and consistent static HTML deployment without requiring a heavy live CMS.

Validation
Validation occurred through architectural walkthroughs of the:

* `auto.js` → JSON generation process
* `render.js` → HTML rendering process
* Lesson 9.2 JSON schema structure
* SQLite maintenance ledger design
* Node.js scanning/report-generation workflow

The resulting architecture was tested conceptually against:

* scalability requirements
* maintainability requirements
* SEO considerations
* static-hosting constraints
* educational publishing workflows
* future course/lesson automation needs

Rules Going Forward

* Markdown should remain the primary authoring format for educational content.
* Structured metadata should be stored separately from presentation logic.
* Static HTML should remain the final published output format whenever possible.
* JavaScript should be used primarily for shared interface enhancement rather than full content rendering.
* JSON lesson files should function as the intermediate structured publishing layer.
* SQLite should remain an internal maintenance and diagnostics ledger rather than the authoritative content source.
* Shared CSS and templates should govern presentation rather than repeated inline structures.
* Metadata validation and standardized lesson schemas should be enforced before rendering.

Notes
The current MGU architecture is approaching the functionality of a lightweight static CMS while avoiding the operational burden of a full database-driven platform. The combination of:

* Markdown authoring
* JSON lesson schemas
* template rendering
* static HTML deployment
* SQLite diagnostics

creates a highly scalable educational publishing system appropriate for large static knowledge libraries. Future improvements may include:

* automated sitemap generation
* incremental builds
* search-index generation
* content validation pipelines
* broken-link scanning
* asset indexing
* unit/course auto-index generation
* metadata-driven navigation maps
