Checking Broken Links or Wrong Paths

PowerShell can help find old links, incorrect file paths, missing local files, and repeated path mistakes across a large website folder tree.

Overview

Broken links and wrong paths are common problems in large websites. A page may link to a file that no longer exists. An image path may point to the wrong folder. A stylesheet may still use an old location. A script tag may load a JavaScript file from a path that has changed.

PowerShell can help locate these problems by searching inside HTML files, listing files that actually exist, comparing expected paths with real paths, and exporting reports for review. This is especially useful when a website contains thousands of pages.

This tutorial focuses on local file and path checking. It helps inspect the files stored on your computer, such as a local C:\MGU website folder. Checking live internet links requires a different workflow, but many website path problems can be found before uploading by checking the local project.

Website Paths and Local Paths

A website path and a Windows file path may point to the same file but look different. A website path usually uses forward slashes. A Windows path usually uses a drive letter and backslashes. 

/component/dept.css
C:\MGU\component\dept.css

On the live website, /component/dept.css means “start at the website root, open the component folder, then load dept.css.” On the local computer, that same file may be stored at C:\MGU\component\dept.css.

Common Path Problems

Common website path problems include old stylesheet links, outdated script paths, moved image folders, missing leading slashes, wrong folder names, incorrect capitalization, and links to files that no longer exist.

A path can look almost correct and still fail. For example, /component/main.css and /component/dept.css are different files. A single extra folder name, missing slash, typo, or old path can cause a page to load the wrong resource.

Search for an Old Stylesheet Path

One of the safest ways to check for wrong paths is to search for a known old path. For example, if department pages should use /component/dept.css, search for old main.css references. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern "/component/main.css" -SimpleMatch

This command searches inside every HTML file under C:\MGU for /component/main.css. It does not change anything. It only reports matches.

Create a Clean List of Files with a Wrong Path

For a cleaner report, show only the unique files that contain the old path. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern "/component/main.css" -SimpleMatch |
  Select-Object -ExpandProperty Path |
  Sort-Object -Unique

This is useful before a correction because it shows which pages may need attention.

Save a Wrong Path Report

Save the matching files to a text file so you can review them later. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern "/component/main.css" -SimpleMatch |
  Select-Object -ExpandProperty Path |
  Sort-Object -Unique |
  Set-Content "C:\MGU\reports\files-using-main-css.txt"

This creates a report of HTML files that still contain the old stylesheet path.

Confirm That the Correct Path Exists

After searching for old paths, confirm that the correct local file exists. 

Test-Path "C:\MGU\component\dept.css"

If the command returns True, the file exists at that local path. If it returns False, the path is wrong or the file is missing.

Check Multiple Component Files

A website may depend on several shared component files. You can check them one at a time. 

Test-Path "C:\MGU\component\header.html"
Test-Path "C:\MGU\component\footer.html"
Test-Path "C:\MGU\component\bottom-nav.html"
Test-Path "C:\MGU\component\main.js"
Test-Path "C:\MGU\component\dept.css"

This helps confirm that the shared files loaded by many pages exist in the expected local location.

Search for Links in HTML Files

HTML links often appear in href attributes. You can search for href to find pages that contain links. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern 'href="' -SimpleMatch

This may produce many results, so it is often better to search for a narrower path, folder, or file name.

Search for Image and Script Paths

Images and scripts often use src attributes. Search for src to locate image references, script references, and other loaded resources. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern 'src="' -SimpleMatch

To narrow the search, look for a specific folder path. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern 'src="/image/' -SimpleMatch

Search for a Specific Folder Path

If a folder was renamed or moved, search for the old folder path before changing files. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern "/library/tutorial/" -SimpleMatch

This finds HTML files that still mention the old path. Replace the example path with the folder path you are investigating.

Extract Local href Links from HTML Files

For a more advanced check, you can extract local href values from HTML files. This command searches for href values that begin with a forward slash. 

$Root = "C:\MGU"

Get-ChildItem -Path $Root -Filter "*.html" -File -Recurse |
  ForEach-Object {
    $File = $_.FullName
    $Content = Get-Content -Path $File -Raw

    [regex]::Matches($Content, 'href="(/[^"#?]+)"') |
      ForEach-Object {
        [PSCustomObject]@{
          SourceFile = $File
          Link = $_.Groups[1].Value
        }
      }
  }

This creates objects showing the source HTML file and the local website link found inside it. The command reads files and reports links; it does not modify anything.

Convert a Website Path to a Local Path

To check whether a root-relative website path exists locally, convert the forward-slash website path into a Windows path under the website root. 

$Root = "C:\MGU"
$WebPath = "/component/dept.css"

$LocalPath = Join-Path $Root ($WebPath.TrimStart("/") -replace "/", "\")
Test-Path $LocalPath

This checks whether /component/dept.css exists as C:\MGU\component\dept.css.

Find Missing Local Links

This more advanced example extracts local href links from HTML files, converts them into local Windows paths, and reports links whose target files do not exist. 

$Root = "C:\MGU"

$BrokenLinks = Get-ChildItem -Path $Root -Filter "*.html" -File -Recurse |
  ForEach-Object {
    $SourceFile = $_.FullName
    $Content = Get-Content -Path $SourceFile -Raw

    [regex]::Matches($Content, 'href="(/[^"#?]+)"') |
      ForEach-Object {
        $Link = $_.Groups[1].Value
        $LocalPath = Join-Path $Root ($Link.TrimStart("/") -replace "/", "\")

        if (!(Test-Path $LocalPath)) {
          [PSCustomObject]@{
            SourceFile = $SourceFile
            Link = $Link
            ExpectedLocalPath = $LocalPath
          }
        }
      }
  }

$BrokenLinks |
  Export-Csv "C:\MGU\reports\missing-local-href-links.csv" -NoTypeInformation

This creates a CSV report of local href links that do not appear to match an existing local file or folder. Review the report carefully because some links may be dynamic routes, anchors, generated pages, or paths that are valid on the live site but not present as simple local files.

Find Missing Local src Files

The same idea can be used for src paths, such as images and scripts. 

$Root = "C:\MGU"

$MissingSrc = Get-ChildItem -Path $Root -Filter "*.html" -File -Recurse |
  ForEach-Object {
    $SourceFile = $_.FullName
    $Content = Get-Content -Path $SourceFile -Raw

    [regex]::Matches($Content, 'src="(/[^"#?]+)"') |
      ForEach-Object {
        $Link = $_.Groups[1].Value
        $LocalPath = Join-Path $Root ($Link.TrimStart("/") -replace "/", "\")

        if (!(Test-Path $LocalPath)) {
          [PSCustomObject]@{
            SourceFile = $SourceFile
            Src = $Link
            ExpectedLocalPath = $LocalPath
          }
        }
      }
  }

$MissingSrc |
  Export-Csv "C:\MGU\reports\missing-local-src-files.csv" -NoTypeInformation

This is useful for finding image paths, script paths, or other local resources that point to missing files.

Watch for Extensionless Website Links

Some websites use extensionless links. For example, a page may link to /department/science/physics/index instead of /department/science/physics/index.html. Depending on hosting settings, both may work online, but the local file usually has the .html extension.

A simple local Test-Path check may report a path as missing if the link omits .html. This does not always mean the live website link is broken. It may mean your site uses extensionless routing.

Ignore Page Anchors When Checking Files

A link may include an anchor after a number sign. The anchor points to a section inside a page, not to a separate file. 

/library/tutorials/powershell/index.html#overview

When checking whether the file exists, the anchor should be ignored. The examples above use patterns that stop before the number sign.

Ignore Query Strings When Checking Files

Some links include query strings after a question mark. A query string passes extra information to a page, but it is not part of the local file path. 

/search/index.html?q=physics

When checking local files, the file path is /search/index.html. The query string should not be treated as part of the file name.

External Links Are Different

Links that begin with http:// or https:// point to external websites or full web addresses. Local Test-Path checks cannot prove whether those websites are working. 

https://example.com

Use local path checking for files in your own project folder. Use web-request testing only when you specifically want to check live URLs.

Find http Links

A useful content check is searching for old http links. Many modern sites prefer https links when available. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern 'http://' -SimpleMatch |
  Select-Object Path, LineNumber, Line |
  Export-Csv "C:\MGU\reports\http-links.csv" -NoTypeInformation

Review the report before changing anything. Some http links may be intentional, but many are worth checking.

Find Accidental Double Slashes

Another useful check is searching for accidental double slashes in local paths. This should be reviewed carefully because https:// also contains double slashes. 

Get-ChildItem -Path "C:\MGU" -Filter "*.html" -File -Recurse |
  Select-String -Pattern 'href="//', 'src="//' -SimpleMatch

This searches for href or src attributes that begin with a double slash. Review results before deciding whether they are wrong.

Watch for Capitalization Problems

Windows is often forgiving about capitalization, but web servers and hosting environments may be stricter. A local file named Dept.css may not be the same as a link to dept.css on every server. 

/component/dept.css
/component/Dept.css

Keep website paths lowercase and consistent when possible. This reduces broken-path problems when files are uploaded to a live host.

Safe Habits

Checking links and paths should begin as a read-only task. Search first. Export reports. Review results. Do not immediately replace or delete anything based on one scan.

Some reported paths may not be truly broken because of extensionless routing, generated pages, anchors, query strings, or live-server behavior. Treat reports as a review list, not an automatic judgment.

Summary

PowerShell can help find broken links and wrong paths by searching HTML files, checking whether local files exist, extracting href and src values, and exporting reports.

Use Select-String to find known bad paths. Use Test-Path to confirm whether expected local files exist. Use reports to review possible problems before making changes.

For large website maintenance, path checking is one of the best ways to prevent missing stylesheets, broken images, incorrect script references, and outdated internal links.
