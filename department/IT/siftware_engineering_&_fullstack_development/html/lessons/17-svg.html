Overview

SVG (Scalable Vector Graphics) is an XML-based vector format ideal for crisp, resolution-independent images: icons, logos, illustrations, and data visualizations. This lesson covers when to use inline SVG versus referencing SVG as an <img> or CSS background, basic drawing primitives, sizing with viewBox and preserveAspectRatio, accessibility techniques, reusable symbols with <symbol> and <use>, styling approaches, and practical examples you can drop into projects.

When to use SVG vs raster images

  • Use SVG for logos, icons, simple illustrations, charts, maps, and any graphic that benefits from crisp scaling, small file sizes for simple shapes, or DOM access for animation/interaction.
  • Use raster (PNG/JPEG/WebP) for complex photographs or when the artwork contains many gradients/photorealistic textures â€” raster is usually smaller and more appropriate for photos.
  • Prefer SVG for UI icons and graphics that must remain sharp on high-DPI screens and zooming contexts.

Inline SVG vs referenced SVG (tradeoffs)

Inline SVG (<svg> in the HTML)

Embedding SVG directly in HTML gives you full access to its DOM: you can style elements with CSS, manipulate nodes with JavaScript, and add ARIA/title/desc elements for accessibility. Inline SVG is best for interactive graphics, animated icons, and illustrations that need runtime manipulation. 

<svg viewBox="0 0 100 100" role="img" aria-labelledby="logoTitle logoDesc">
  <title id="logoTitle">Malone University Logo</title>
  <desc id="logoDesc">Circular logo with stylized M inside.</desc>
  <circle cx="50" cy="50" r="45" fill="currentColor" />
  <text x="50" y="55" text-anchor="middle" font-size="40" fill="#fff">M</text>
</svg>

Referenced SVG (<img src="logo.svg"> or background-image)

Referencing an external SVG via <img>, background-image, or <object> keeps HTML lean and allows caching across pages. However, referenced SVGs have limited styling and scripting access from the hosting page unless you inline or fetch/inline them via JS. Use referenced SVG when the graphic is decorative or shared across many pages where you don't need to alter internals. 

<img src="/icons/logo.svg" alt="Malone University">

Short tradeoffs checklist

  • Need interactivity/animation/ARIA inside SVG? â†’ Inline SVG.
  • Shared decorative asset, cacheable, not interactive? â†’ External SVG via <img> or CSS.
  • Icons used as a sprite? â†’ Consider an external SVG sprite or icon font depending on workflow.

Basic SVG shapes and attributes

SVG has simple primitives you can combine to build complex graphics:

Rectangles

<rect x="10" y="10" width="80" height="40" rx="6" ry="6" fill="#0a7" />

Circles & ellipses

<circle cx="50" cy="25" r="20" />
<ellipse cx="50" cy="50" rx="30" ry="15" />

Lines, polylines, polygons

<line x1="0" y1="0" x2="100" y2="100" stroke="#222" />
<polyline points="10,10 50,20 90,10" fill="none" stroke="#333" />
<polygon points="50,10 90,90 10,90" />

Path (powerful, complex shapes)

<path d="M10 80 Q 52.5 10, 95 80 T 180 80" fill="none" stroke="#000" />

Text

<text x="50" y="50" text-anchor="middle" dominant-baseline="middle">Label</text>

viewBox and sizing

The viewBox defines the internal coordinate system of the SVG and enables responsive scaling: 

<svg viewBox="0 0 100 50" width="100%" height="auto">â€¦</svg>

viewBox="minX minY width height" â€” once set, the SVG scales to its container while preserving the internal proportions. Use preserveAspectRatio to control alignment and stretching.

Styling SVG

You can style inline SVG with CSS selectors or inline presentation attributes. For external SVGs referenced by <img>, you cannot style internals from the parent CSS (except limited filters). Inline SVG supports CSS cascade, classes, and currentColor to inherit text color. 

<svg class="icon" viewBox="0 0 24 24">
  <style>
    .icon .stroke { stroke: currentColor; stroke-width: 1.5; fill: none; }
  </style>
  <circle class="stroke" cx="12" cy="12" r="9"></circle>
</svg>

/* Then in site CSS */
.icon { width: 1.5rem; height: 1.5rem; color: #0a7; }

Tips: prefer CSS over inline presentation attributes for maintainability; use currentColor to make icons match surrounding text color.

Reusability: <symbol> and <use>

Define reusable fragments inside an <svg> or external sprite using <symbol> and instantiate them with <use href="#icon-id">. This is efficient for icon libraries. 

<!-- sprite.svg -->
<svg xmlns="http://www.w3.org/2000/svg">
  <symbol id="icon-star" viewBox="0 0 24 24">
    <path d="M12 17.3L5.6 20l1.1-6.3L2 9.3l6.4-.9L12 3l3.6 5.4 6.4.9-4.7 4.4L18.4 20z"/>
  </symbol>
</svg>

<!-- usage inline (if sprite in DOM) -->
<svg class="icon" aria-hidden="true">
  <use href="#icon-star"></use>
</svg>

<!-- usage referencing external sprite.svg (note browser support and CORS) -->
<svg class="icon" aria-hidden="true">
  <use href="/images/sprite.svg#icon-star"></use>
</svg>

When referencing an external sprite via <use>, be aware of CORS and some cross-browser differences â€” in many cases inlining the SVG sprite into the HTML is simplest and most compatible.

Accessibility for SVG

Accessibility requires that SVGs convey the same meaning as other images or UI controls. Use accessible names and descriptions, donâ€™t rely on visual-only cues, and decide whether the SVG is decorative or meaningful.

Decorative SVG

If decorative, hide from assistive tech:

<svg aria-hidden="true" focusable="false">â€¦</svg>

Meaningful SVG

For informative graphics, provide a title and a description and ensure the SVG is reachable by screen readers:

<svg role="img" aria-labelledby="svgTitle svgDesc" viewBox="0 0 100 50">
  <title id="svgTitle">Enrollment growth chart</title>
  <desc id="svgDesc">Line chart showing steady growth from 2019 to 2024.</desc>
  <!-- shapes -->
</svg>

Complex diagrams

For complex diagrams, include a textual long description near the SVG or link to a full text alternative. Use aria-describedby pointing to that paragraph for more context.

Interactive elements inside SVG

If interactive elements (links, buttons) exist inside an inline SVG, ensure they are keyboard-focusable (tabindex="0" when needed) and have accessible names (title/aria-label). Test with keyboard and screen readers.

Performance & security considerations

  • Simple SVGs are usually smaller than equivalent raster icons â€” but very complex SVGs with many nodes can be heavy; optimize SVG output (remove metadata, unnecessary groups, comments).
  • Minify and optimize (SVGO or similar) as part of the build pipeline.
  • Be cautious when inlining SVG from untrusted sources â€” SVG can contain scripts and external references; sanitize or avoid inlining unknown SVGs.
  • Use width/height or CSS aspect-ratio to reserve layout space and avoid layout shift.

Examples: putting it into practice

Inline accessible icon (preferred when interactive or styled)

<button class="icon-btn" aria-label="Favorite">
  <svg viewBox="0 0 24 24" width="24" height="24" role="img" aria-hidden="true">
    <use href="#icon-star"></use>
  </svg>
</button>

Referenced SVG for decorative site logo

<img src="/images/logo.svg" alt="Malone University">

Responsive SVG illustration

<svg viewBox="0 0 1200 600" width="100%" height="auto" preserveAspectRatio="xMidYMid meet">
  <!-- scalable illustration that fills container while preserving aspect ratio -->
</svg>

Practical workflow tips

  • Create icons in a vector editor (Figma, Illustrator), export clean SVG, then run an optimizer (SVGO) to remove unnecessary attributes and reduce size.
  • Use an inline sprite or a build-time inlining step for icon systems to balance caching and styling needs.
  • Prefer currentColor for icon fills/strokes to make icons theme-aware.
  • Test across browsers and assistive tech â€” sometimes SVG behavior differs for <use> across engines.

Key Takeaways

  • SVG is ideal for resolution-independent graphics (icons, logos, charts); choose it over raster for UI assets that must scale cleanly.
  • Inline SVG offers DOM access, styling, and scriptability â€” use it when interaction or accessibility markup is required.
  • Referenced SVGs are cache-friendly and good for decorative assets; remember their styling and scripting limitations from the parent document.
  • Always provide accessible names/descriptions for meaningful SVGs (<title>, <desc>, role, or aria-labelledby), and hide purely decorative SVGs from assistive tech.
  • Optimize SVG assets in your build pipeline, use viewBox and reserved sizing to avoid layout shift, and prefer currentColor for themeability.
