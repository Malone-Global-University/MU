Stage 3 â€” Advanced â†’ Lesson 15

Overview

HTML5 expanded forms with built-in validation, new input types, and helper elements that improve UX and reduce the need for JavaScript. This lesson covers the most useful validation attributes (required, pattern, min, max, minlength, maxlength), the <datalist> element, proper use of placeholder, and how to build accessible validation flows using the Constraint Validation API and progressive enhancement.

The goal: make forms that guide users, give timely feedback, and remain robust even when JavaScript is disabled.

Built-in HTML validation attributes

Browsers perform lightweight checks before submitting a form when validation attributes are present. These checks provide immediate feedback and help prevent obviously invalid submissions.

required

Marks a control as mandatory. The browser prevents submission until a value is provided. 

<label for="email">Email</label>
<input id="email" name="email" type="email" required autocomplete="email">

Type-driven checks (type="email", type="url", type="number", etc.)

The type attribute instructs browsers on expected syntax and can trigger device-appropriate keyboards on mobile. 

<input type="url" name="website" placeholder="https://example.org" autocomplete="url">

pattern

Use a regular expression to validate the control's value. Add title to provide a human-readable hint for the expected format. 

<label for="sku">SKU (3 letters followed by 4 digits)</label>
<input id="sku" name="sku" pattern="^[A-Za-z]{3}\d{4}$" title="Format: ABC1234">

min, max, step (numeric & date)

Constrain numeric and date/time inputs: 

<label>Age: <input type="number" name="age" min="13" max="120"></label>

<label>Event date: <input type="date" name="event" min="2025-01-01" max="2026-12-31"></label>

minlength & maxlength

Set length constraints for text inputs and textareas. Note that maxlength is enforced, whereas minlength participates in validation but does not restrict typing. 

<textarea name="bio" minlength="20" maxlength="1000" rows="6"></textarea>

novalidate on forms

Use novalidate to disable native validation (e.g., when implementing custom JS validation). If you disable native validation, ensure you implement equivalent checks on both client and server.

Constraint Validation API (brief)

The DOM's Constraint Validation API allows progressive enhancement: show improved inline feedback while still leaving native validation as a fallback.

Common methods/properties (example only â€” no heavy JS required):

  • input.checkValidity() â€” returns true if the input passes validity checks.
  • form.reportValidity() â€” shows native validation messages for invalid controls and returns false if invalid.
  • input.validity â€” object describing which constraint failed (e.g., valueMissing, patternMismatch).
  • input.setCustomValidity(message) â€” set a custom validation message (pass an empty string to clear it).
<script>
/* small enhancement example (optional) */
const form = document.querySelector('#signup');
form.addEventListener('submit', e => {
  if (!form.checkValidity()) {
    e.preventDefault();
    form.reportValidity(); // shows standard browser messages
  }
});
</script>

Important: never rely solely on client-side validation. Always validate and sanitize on the server.

<datalist> â€” suggested values

<datalist> provides a list of suggested values for an input while still allowing free-form typing. Itâ€™s great for UX when you want to suggest common values without restricting the user. 

<label for="city">City</label>
<input id="city" name="city" list="cities" autocomplete="address-level2">
<datalist id="cities">
  <option value="Columbus">
  <option value="Cleveland">
  <option value="Cincinnati">
</datalist>

Notes: <datalist> is not supported exactly the same across all browsers (styling and UX differ), but it degrades gracefully â€” users can still type a custom value.

placeholder â€” guidance, not replacement

The placeholder attribute shows short hint text inside inputs. It should never replace a visible label â€” placeholders are transient (disappear when the user types) and can be problematic for accessibility. 

<label for="twitter">Twitter handle</label>
<input id="twitter" name="twitter" placeholder="@yourhandle" autocomplete="username">

Use placeholders for example values or short hints, but always keep a label. If you need persistent helper text, use an adjacent hint element and connect it via aria-describedby.

Input hints: autocomplete, inputmode, and accessibility

Improve mobile and desktop UX with these attributes:

  • autocomplete â€” use tokens like email, given-name, cc-number to enable browser autofill.
  • inputmode â€” suggest a keyboard layout (e.g., numeric, tel, decimal) especially useful for type="text" inputs that expect numbers.
  • Keep labels clear and associate help text via aria-describedby.
<input name="phone" type="tel" inputmode="tel" autocomplete="tel" placeholder="+1 (555) 555-5555">

Accessible validation UX patterns

Validation messages must be announced to assistive technologies and must clearly indicate what is wrong and how to fix it.

  • Associate messages â€” put error text in an element referenced by aria-describedby on the invalid control.
  • Use aria-live for form-level messages that appear after client-side validation or submission.
  • Set aria-invalid="true" on fields that fail validation so screen readers are informed.
  • Focus management â€” when the form submission fails, move focus to the first invalid control or the error summary so keyboard users land immediately where they need to act.
<label for="username">Username</label>
<input id="username" name="username" required aria-describedby="username-error">
<div id="username-error" class="error" aria-live="polite"></div>

Provide both a field-level message (near the control) and an optional error summary at the top of the form that links to each problematic control.

Server-side validation & security

Client-side checks are convenience. Server-side validation is mandatory. Always:

  • Validate types, lengths, patterns, ranges on the server.
  • Sanitize and escape data before rendering or storing it to prevent injection attacks.
  • Enforce business rules server-side even when client code restricts inputs.
  • Use HTTPS and protect form endpoints with CSRF tokens for state-changing requests.

Examples: validated form patterns

Simple contact form with required and pattern

<form action="/contact" method="post" id="contact-form">
  <label for="name">Full name</label>
  <input id="name" name="name" type="text" required minlength="2" aria-describedby="name-help">
  <div id="name-help" class="hint">Enter your first and last name.</div>

  <label for="email">Email</label>
  <input id="email" name="email" type="email" required autocomplete="email">

  <label for="phone">Phone (optional)</label>
  <input id="phone" name="phone" type="tel" inputmode="tel" placeholder="+1 (555) 555-5555">

  <button type="submit">Send</button>
</form>

Practice

  1. Create 15-forms-validation-practice.html and build:
    • A multi-field sign-up form with required fields, type="email", minlength, and a numeric age input with min and max.
    • A field that uses <datalist> for suggestions (e.g., campus locations) while allowing custom input.
    • At least one pattern-validated field and an explanatory title for guidance.
  2. Implement accessible error handling:
    • Show field-level messages and connect them with aria-describedby.
    • Set aria-invalid="true" on invalid fields and focus the first invalid field on submit.
  3. Test behavior:
    • Disable JavaScript â€” form should still submit (server will validate) or be prevented by native browser checks where applicable.
    • Enable JS and add a small enhancement using reportValidity() or setCustomValidity to display friendlier messages.
    • Run an accessibility audit and keyboard-only test to ensure messages are announced and focus management works.

Key Takeaways

  • Use built-in attributes (required, pattern, min, max, minlength, maxlength) to reduce friction and provide immediate feedback.
  • <datalist> gives helpful suggestions without preventing custom input â€” it degrades gracefully across browsers.
  • placeholder is a hint, not a label â€” always use visible labels and persistent help text when needed.
  • Enhance validation with the Constraint Validation API for better UX, but always validate on the server for security.
  • Make validation messages accessible (use aria-describedby, aria-invalid, aria-live, and proper focus management).
