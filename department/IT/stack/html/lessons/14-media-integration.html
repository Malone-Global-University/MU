Overview

Multimedia (audio and video) makes learning experiences richer but introduces accessibility, performance, and legal/privacy requirements. This lesson covers the <video> and <audio> elements, how to provide multiple source formats, using <track> for captions/subtitles, transcripts, and best practices for accessible and performant media.

After this lesson you'll be able to embed playable media with captions, choose fallback formats, provide transcripts, and make informed decisions about autoplay, lazy-loading, and privacy.

<video> â€” basics and attributes

The HTML <video> element provides native playback for video content. Use multiple <source> children to offer alternate formats so the browser can choose a supported codec. 

<video controls width="720" preload="metadata" poster="/media/thumb.jpg" crossorigin>
  <source src="/media/lecture-720p.webm" type="video/webm; codecs="vp9, vorbis"">
  <source src="/media/lecture-720p.mp4" type="video/mp4; codecs="avc1.42E01E, mp4a.40.2"">
  <track kind="captions" srclang="en" label="English captions" src="/media/lecture.en.vtt">
  Your browser does not support the video element. <a href="/media/lecture-720p.mp4">Download the video</a> or read the transcript <a href="/media/lecture-transcript.html">here</a>.
</video>

Common attributes:

  • controls â€” show native playback UI (recommended for accessibility).
  • autoplay â€” auto-start playback (most browsers require muted for autoplay to work).
  • muted â€” start muted (useful with autoplay).
  • loop â€” replay when finished.
  • preload â€” auto, metadata, or none (metadata usually recommended for performance).
  • poster â€” image shown before playback.
  • crossorigin â€” when loading from another origin and you need to manipulate media with canvas or require CORS-enabled requests.

Notes: Always include a fallback message and a download/transcript link so users on older browsers or with assistive tech have alternatives.

<audio> â€” embedding sound

The <audio> element supports native audio playback and also accepts multiple <source> elements for format fallbacks. 

<audio controls preload="metadata">
  <source src="/media/podcast-128.mp3" type="audio/mpeg">
  <source src="/media/podcast-128.ogg" type="audio/ogg">
  Your browser does not support the audio element. Please download the file <a href="/media/podcast-128.mp3">here</a>.
</audio>

For audio-only content provide a full transcript on the page in addition to captions (if available) so Deaf or Hard-of-Hearing users can access the material.

<track> â€” captions, subtitles, and descriptions

The <track> element attaches timed text tracks to media. Typical uses:

  • captions (kind="captions"): text equivalent of spoken audio + relevant sounds â€” essential for Deaf/hard-of-hearing users.
  • subtitles (kind="subtitles"): translation of spoken content for viewers who don't understand the language.
  • descriptions (kind="descriptions"): audio descriptions of visual content for blind users (delivered as an alternate audio track or as text descriptions).
  • chapters (kind="chapters"): navigable chapter markers.
  • metadata (kind="metadata"): timed data for scripts (not visible to users by default).
<track kind="captions" srclang="en" label="English" src="/media/lecture.en.vtt">
<track kind="subtitles" srclang="es" label="EspaÃ±ol" src="/media/lecture.es.vtt">

WebVTT is the recommended format for captions/subtitles. Each <track> should have a descriptive label and correct srclang.

Best practices for captions

  • Provide captions for all videos with spoken audioâ€”this is essential for accessibility and often required legally.
  • Keep captions synchronized, accurate, and include non-speech information (e.g., [applause], [music]) when relevant.
  • Offer captions in multiple languages when your audience requires it.
  • Provide a transcript (HTML text) in addition to captions for searchability and easier access.

Transcripts & alternatives

A full transcript (plain HTML text) is strongly recommended for all audio and video content. Transcripts support:

  • Users who prefer reading or have limited bandwidth.
  • Search engines (improves discoverability).
  • Assistive technologies that can read the text more easily than timed captions.
<!-- Provide a visible transcript below or linked next to the media -->
<section aria-labelledby="transcript-heading">
  <h2 id="transcript-heading">Transcript</h2>
  <p>Speaker: Welcome to the course...</p>
  <!-- more transcript text -->
</section>

Accessibility considerations

  • Always include controls unless you provide a fully accessible custom interface. Native controls are keyboard-friendly and announced by assistive tech.
  • For autoplayed content, avoid starting with sound â€” autoplay should be muted and never surprising to users.
  • Provide captions and transcripts. Captions are not a substitute for transcripts but both are complementary.
  • Ensure media players are keyboard operable (play/pause, seek, volume) and that focus order is logical.
  • Use descriptive labels for tracks and provide language metadata (srclang) for each caption track.

Performance & optimization

Media files are large. Optimize delivery to improve UX:

  • Encode multiple bitrate variants and use adaptive streaming (HLS/DASH) for large, long videos â€” the server/player delivers appropriate quality for the user's network.
  • Serve modern codecs (AV1/VP9/WebM, H.264 for compatibility) and provide fallbacks. Prioritize WebM/AV1 for size when supported, but keep MP4/H.264 for broad compatibility.
  • Use preload="metadata" or preload="none" for non-critical media to avoid unnecessary bandwidth usage.
  • Provide poster images so the page looks stable while video loads.
  • Use CDNs for media hosting and set long cache headers.
  • Lazy-load offscreen media and defer heavy players until user interaction when appropriate.

Privacy & hosting considerations

Embedding third-party-hosted media (YouTube, Vimeo) is convenient but carries privacy and tracking implications:

  • Third-party embeds may load trackers and cookies. Use privacy-enhanced embed modes when available (e.g., YouTube's privacy-enhanced domain) or self-host media when privacy is a priority.
  • Inform users if external providers are used (privacy policy / consent banners) and respect consent before loading analytics/tracking players.
  • If you need to manipulate frames or media from another origin (e.g., draw video to <canvas>), ensure CORS is configured and use the crossorigin attribute appropriately.

Responsive & CSS tips

Make media responsive and fluid so it fits different viewports: 

/* CSS: make video responsive */
.responsive-video {
  position: relative;
  padding-bottom: 56.25%; /* 16:9 aspect ratio */
  height: 0;
}
.responsive-video video {
  position: absolute;
  top: 0;
  left: 0;
  width: 100%;
  height: 100%;
}

Use a wrapper with an aspect-ratio (CSS aspect-ratio) or the padding-bottom technique. Ensure controls remain visible and usable on small screens.

Practice

  1. Create 14-media-practice.html that includes:
    • A responsive <video> with at least two <source> formats and a <track kind="captions"> file (WebVTT).
    • An <audio> example with an accessible transcript on the same page or linked nearby.
    • A poster image and sensible preload settings to minimize initial bandwidth.
  2. Test for accessibility and behavior:
    • Confirm captions can be enabled/disabled and are synchronized.
    • Tab through the page to ensure the media controls are keyboard-accessible.
    • Verify the transcript is discoverable by search and readable without a media player.
  3. Optional advanced practice:
    • Experiment with adaptive streaming (HLS) using a small HLS manifest and a lightweight JS player â€” note this requires server support or a hosting service.
    • Measure page load impact with and without preloading the poster/metadata and with different preload values.

Key Takeaways

  • Provide captions (<track kind="captions">) and transcripts for all audio/video content â€” they are essential for accessibility and discoverability.
  • Offer multiple source formats (WebM/AV1, MP4/H.264) to maximize compatibility and efficiency.
  • Use native controls for accessibility unless you implement a fully accessible custom player.
  • Optimize media delivery with modern codecs, adaptive streaming for long videos, CDNs, and sensible preload choices.
  • Be mindful of privacy when using third-party players; prefer self-hosting when user privacy is a priority.
