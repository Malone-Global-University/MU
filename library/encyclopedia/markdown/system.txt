You’ve essentially rebuilt your system into a clean, modular static pipeline. Here’s how each script now fits together, in plain terms.

build-content.js → “Content Compiler” (Markdown → JSON)
This is your foundation layer.

It reads raw .md files from your private markdown/ folder, extracts frontmatter (title, domain, tags, SEO, etc.), and converts the body into both raw markdown and rendered HTML.

It outputs structured .json files into content/.

What matters:

This is the single source of truth
All metadata lives here (not in HTML anymore)
If something is wrong downstream, it usually starts here

Think of it as:
authoring system → structured content model

build-html.js → “Renderer” (JSON → HTML pages)
This is your presentation layer.

It takes each JSON file and injects it into your HTML template:

Title, SEO, OpenGraph, Twitter tags
Navigation + breadcrumbs
Hero section (domain, tags)
Main body (htmlBody)
Footer + timestamps

It outputs fully rendered pages into domain folders like:

/science/
/math/
/martial-arts/

What matters:

No parsing logic here — just rendering
Clean separation from content logic
Easy to redesign UI without touching content

Think of it as:
structured data → final website pages

build-index.js → “Library Builder” (JSON → Navigation Pages)
This builds your browsing experience.

It reads all JSON files and creates:

Main encyclopedia index (/encyclopedia/index.html)
Domain/category pages (/science/index.html, etc.)

What it uses:

title
domain
output.url

What matters:

No HTML scraping anymore
Purely driven from structured JSON
This is your “library shelves”

Think of it as:
content catalog → human navigation

build-search-index.js → “Search Catalog Builder” (JSON → search.json)
This builds your search database.

It reads JSON and creates a lightweight search.json file:

title
domain
url
content (from htmlBody stripped to text)

What matters:

No HTML parsing hacks anymore
Clean, predictable data
Fast client-side search

Think of it as:
content database → searchable index

search.js → “Client Search Engine” (Browser Runtime)
This is what users interact with.

It:

Loads search.json
Listens to user input
Filters results based on:
title
domain
content
Dynamically displays matches

What matters:

Runs entirely in the browser
No backend required
Works instantly once JSON is loaded

Think of it as:
search UI → live filtering engine

search.json → “Search Data Store”
This is the output of build-search-index.js.

It contains flattened, searchable data:

Lightweight
No HTML
Optimized for quick filtering

Think of it as:
read-only search database

Final System Flow (Clean Version)
You write content
→ markdown/*.md
Compile content
→ build-content.js
→ content/*.json
Render site
→ build-html.js
→ /science/*.html, etc.
Build navigation
→ build-index.js
→ /index.html, /science/index.html
Build search
→ build-search-index.js
→ /search.json
User experience
→ search.js runs in browser
What You Actually Built

Not just an SSG — a content pipeline with separation of concerns:

Markdown = authoring
JSON = data model
HTML = presentation
Index = navigation
Search = retrieval

That’s a modern architecture, just without a backend.

The Key Upgrade You Achieved

Originally:

HTML was both output and database (fragile)

Now:

JSON is the database
HTML is purely rendered output

That’s the turning point. Everything becomes easier from here:

Filtering
Tag systems
Related content
Multi-level taxonomy
API-style reuse later if you want it

If you want next step direction, the most impactful upgrades would be:

tag index pages (cross-domain linking)
subdomain nesting (true hierarchy)
related-content auto-generation

Those turn this from a static encyclopedia into an actual knowledge system.
