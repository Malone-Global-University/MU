Chapter 6 — Front Matter

Chapter Introduction and Learning Objectives

Chapter 5 introduced the HTML document skeleton: the <!DOCTYPE>, the <html> element, the <head>, the <body>, and the basic structure that every page shares. You learned how to write a valid starter page and how to nest elements correctly.

This chapter goes inside the <body> and teaches you how to structure written content. Most of what exists on the web is text — headings, paragraphs, articles, descriptions, instructions, labels, and explanations. HTML gives you specific elements for all of these, and the choice of element matters. Not for decoration, but for meaning.

By the end of this chapter you will understand why <h1> is not just a way to make text bigger, why <p> is not just a way to add space, and why elements like <main>, <section>, <article>, <header>, <footer>, and <aside> exist alongside simple headings and paragraphs. You will understand semantic HTML — writing code that says what things are, not just what they look like.

Opening Question

Two developers build pages that look identical in a browser. One wraps every piece of content in <div> elements styled with CSS classes. The other uses <h1>, <p>, <h2>, and <p> elements that CSS then styles to match. Side by side on screen, the two pages are indistinguishable. So does it matter which approach a developer chooses? By the end of this chapter, you will understand why the answer is an emphatic yes — and why the choice between meaningless containers and meaningful elements shapes who can use a page, how easily it can be found, and how easily it can be maintained.

The Basic Units of Written Content
Text on the Web · Headings · Paragraphs
Section 6.1

Text Content on the Web

Open almost any web page and look at what it contains. A news article has a title, a publication date, several paragraphs of body text, and maybe a pull quote. A documentation page has a large heading, several sub-headings, code blocks, and explanatory paragraphs. A product page has a product name, a price, a description, a list of features, and customer reviews.

All of this is text content. And all of it carries structure — not just visual structure, but logical structure. A product name is more important than a feature description. A section heading divides one topic from another. A paragraph groups a single idea together.

HTML was designed to encode this logical structure. When you write an HTML document, you are not designing a visual layout. You are describing what each piece of content is. Visual layout is CSS's job. Structure and meaning is HTML's job.

Who Depends on Structure
Assistive Technology
Screen readers read HTML structure to understand the page. A heading lets a user list every section and jump directly to one. A paragraph is announced as a block of continuous text. Visual appearance is irrelevant — only what you label in HTML is communicated.
Who Depends on Structure
Search Engines
A search engine trying to understand your page pays attention to your headings, paragraphs, and semantic landmarks. Pages with clear, logical structure are easier to index and easier to rank correctly.
Who Depends on Structure
Future Developers
When you return to a project months later, or someone else works on it, clean structural HTML tells the story of the page without requiring anyone to read every pixel of the rendered output.
The Goal of This Chapter

The goal is not to make text look a certain way. The goal is to mark text accurately so that browsers, assistive technologies, search engines, and future developers all understand what the page contains. Appearance comes later, in Part III, once the meaning is already correct.

Section 6.2

Headings and Heading Order

HTML provides six heading elements:

<h1>Heading Level 1</h1>
<h2>Heading Level 2</h2>
<h3>Heading Level 3</h3>
<h4>Heading Level 4</h4>
<h5>Heading Level 5</h5>
<h6>Heading Level 6</h6>

These six elements represent a hierarchy, not a set of font sizes. <h1> is the highest-level heading on the page — the main title or primary subject. <h2> introduces major sections under that title. <h3> introduces sub-sections inside an <h2> section. And so on through <h6>, which is rarely used.

Most pages you build in this course will use <h1> through <h3> regularly, and <h4> occasionally. You will rarely have reason to use <h5> or <h6> in a typical page.

The Heading Hierarchy Rule

The most important rule about headings is: do not skip levels. A page that jumps from <h1> to <h3> has broken its outline. A page with two different <h1> elements in the same main content area may confuse assistive technologies about what the primary subject is.

Think of headings like an outline you might write before an essay:

I. Introduction to Astronomy         <- h1
   A. The Solar System               <- h2
      1. The Sun                     <- h3
      2. The Planets                 <- h3
   B. Stars and Galaxies             <- h2
      1. Types of Stars              <- h3
II. Observation Techniques           <- h1 (or h2, depending on page context)

Each level of that outline maps to a heading level in HTML.

One <h1> Per Page

As a strong general guideline for this course, use exactly one <h1> per page. It should represent the single most important heading on that page — the title of the article, the name of the product, the main subject the page covers. If your site's name lives in a <header> as a logo, the <h1> should still identify the content of that particular page, not just the site name.

What Headings Are Not

Headings are not ways to make text big or bold. Many beginners reach for <h2> because it makes text a certain size in the browser's default stylesheet. This is a mistake. Size is a CSS concern. If you want text to be large, set its font-size in CSS. Choose your heading level based on the document's structure, not its visual appearance.

Common Mistake

The second heading below is being used as a visual caption, not a major section heading. This misrepresents the page structure and causes problems for screen readers. HTML provides <figcaption> for captions, covered in Chapter 8.

Wrong — Using h2 Just to Make Text Bigger
<h2>Welcome to my page</h2>
<p>Some content here.</p>
<h2>This is a caption</h2>
<img src="photo.jpg" alt="A mountain landscape">

Default Heading Appearance

A Good Habit to Build Now

Browsers apply default styles to headings — <h1> is typically the largest, <h6> the smallest. These are browser defaults, not permanent rules. Once you add your own CSS in Part III, you can make any heading look however you want, and its appearance will no longer match its level unless you design it that way deliberately. Choose your HTML based on meaning, then use CSS to control appearance.

Section 6.3

Paragraphs

The paragraph element <p> wraps a unit of related text that belongs together as a single thought or idea:

<p>The web was invented by Tim Berners-Lee in 1989 as a way to share research
documents among scientists at CERN. It was built on top of the existing
internet infrastructure and used a new system of linked documents.</p>

<p>Within a few years, the web had expanded far beyond scientific research.
Commercial websites, news organizations, and individuals began publishing
content, and browsers became some of the most widely used software on
any platform.</p>

Each <p> element is a separate block. Browsers insert spacing between paragraphs by default, which you will later control with CSS.

What Goes Inside a Paragraph

A <p> element should contain prose — sentences that read together as connected text. You should not put block-level elements like headings, lists, or other paragraphs inside a <p>. Nesting a block inside a <p> creates invalid HTML that browsers may silently correct in unpredictable ways.

Invalid — Block Element Inside a Paragraph
<p>Here is some information.
  <h3>A Subheading</h3>
  And some more information.
</p>

The <h3> should live outside the paragraph, not inside it.

Paragraphs Versus Line Breaks

A common beginner question is: "What is the difference between ending a paragraph and inserting a line break?"

HTML provides <br> — a self-closing element — for a single line break within flowing content. A line break is not a new paragraph; it is a hard line break within the same paragraph.

<p>123 Main Street<br>
Springfield, IL 62701</p>

This is a typical use of <br>: an address that needs lines to break without starting a new paragraph. Addresses are a single unit of information, so they belong in one <p>, but they are traditionally formatted on separate lines.

Avoid This Habit

Using multiple <br> elements to create visual spacing between sections of text is poor practice. Use paragraphs, margins, and CSS spacing instead.

Paragraphs Are Not the Only Text Container

Not all text belongs in a <p>. A list item goes in <li>. A table cell goes in <td>. A navigation link sits in an <a> inside a list. A heading uses one of the <h1> through <h6> elements. As this chapter continues, you will see many other ways to contain and organize text content.

Section 6.4

Emphasis and Strong Importance

HTML provides two inline elements for emphasizing text within a paragraph or heading:

<em>emphasized text</em>
<strong>strongly important text</strong>

<em> — Stress Emphasis

The <em> element marks text that has stress emphasis — the kind of vocal stress you would apply when reading aloud. Compare these two sentences:

Stress Emphasis Changes Meaning
The same words, two different emphases
  • Version A"I said you should call her, not him." — written as I said you should call <em>her</em>, not him.
  • Version B"I said you should call her, not him." — written as I said <em>you</em> should call her, not him.
  • Why it mattersBrowsers typically render <em> in italics by default, but the semantic meaning is stress emphasis, not italics. Screen readers may change their vocal tone when reading <em> content.

<strong> — Strong Importance

The <strong> element marks text that is strongly important — content that carries weight, severity, or urgency:

<p>Before proceeding, <strong>back up your data</strong>. This operation
cannot be undone.</p>

Browsers render <strong> in bold by default. Again, the semantic meaning is importance, not visual boldness. Use CSS to control visual weight.

What <em> and <strong> Are Not

Do not use these elements for purely visual effect. If you want italic or bold text for stylistic reasons — naming a book title or highlighting a term — HTML provides other elements, or you can apply CSS directly.

ElementSemantic MeaningDefault StyleTypical Use
<em>Stress emphasisItalicChanging meaning through vocal stress
<strong>Strong importanceBoldWarnings, key points, urgent instructions
<i>Alternate voice, mood, or offset contextItalicBook and film titles, technical terms on first use, foreign phrases
<b>Drawing attention without implying importanceBoldKeywords in a summary, product names in a review
 
<p>The novel <i>Moby-Dick</i> was published in 1851.</p>
<p>Search for <b>organic cotton</b> to filter the results.</p>
Practical Guidance

In practice for this course, you will use <em> and <strong> most often, because they carry semantic weight. Use <i> for titles and technical terms. Avoid using <b> and <i> purely for visual styling — that is CSS's job.

Section 6.5

Line Breaks and Horizontal Rules

You have already seen <br> for line breaks. Two additional minor elements are worth covering briefly.

<br> — Line Break

As covered above, <br> creates a single hard line break within a paragraph or other inline content. It is self-closing and does not take a closing tag:

<p>To apply, send your materials to:<br>
Admissions Office<br>
Room 204, Building A</p>

Use <br> only when the line break is meaningful — addresses, poetry, song lyrics, and similar content where line breaks carry semantic or conventional significance. Do not use <br> to add visual spacing between sections. Use CSS margins and padding for spacing.

<hr> — Thematic Break

The horizontal rule element <hr> represents a thematic break between sections of content within a document — a shift in scene in a story, a section break in a long article, or a major topic transition:

<p>This section covers the history of the web.</p>

<hr>

<p>This section covers modern web development tools.</p>

Browsers render <hr> as a horizontal line by default. You can style it with CSS. <hr> is a void element, meaning it has no content and no closing tag.

Semantic Meaning, Not Decoration

<hr> marks a thematic break, not just a visual line. If you want a decorative divider with no semantic meaning, use a CSS border or background instead of <hr>.

Section 6.6

Semantic HTML

You have now encountered <h1> through <h6>, <p>, <em>, <strong>, and a few other elements. All of them share an important quality: they communicate what something is, not just how it should look.

This is the core principle of semantic HTML. The word "semantic" means "relating to meaning." Semantic HTML is HTML that uses elements chosen to match the meaning of the content, not the desired visual appearance.

Compare these two approaches to the same recipe content:

Non-Semantic — "Div Soup"
<div class="big-text">Recipe: Classic Tomato Soup</div>
<div class="body-text">This recipe serves four people and takes about
thirty minutes to prepare.</div>
<div class="big-text">Ingredients</div>
<div class="body-text">You will need tomatoes, olive oil, onion, garlic,
broth, salt, and pepper.</div>
Semantic — Meaningful Structure
<h1>Recipe: Classic Tomato Soup</h1>
<p>This recipe serves four people and takes about thirty minutes
to prepare.</p>
<h2>Ingredients</h2>
<p>You will need tomatoes, olive oil, onion, garlic, broth, salt,
and pepper.</p>

Both might look similar in a browser if the CSS is designed that way. But only the second version tells a machine — a screen reader, a search engine, a browser's reading mode, or a developer — what the content actually is.

The <div> element is a generic block container with no semantic meaning. It groups content visually but says nothing about what that content represents. The <span> element is the inline equivalent — also generic, also meaningless on its own.

<div> and <span> are useful tools, and you will use them in this course, especially when you need a container for CSS layout purposes. But they should not replace semantic elements when semantic elements exist and apply. The principle is: use the most specific semantic element that correctly describes your content.

Why Semantic HTML Matters

Benefit 1
Assistive Technology Users
Screen readers build navigation menus from headings. Voice control software lets users say "click the heading" or "navigate to main content." Keyboard users rely on landmark regions.
Benefit 2
Search Engines
A page that correctly marks its title as <h1>, its sections as <h2>, its navigation as <nav>, and its main content as <main> is far easier to index and rank.
Benefit 3
Developers Maintaining the Code
Semantic HTML is self-documenting. Reading <article> tells you something. Reading <div class="article-thing-box"> tells you less.
Benefit 4
Browsers and Reading Modes
Browsers have reader modes that strip visual styling and show only content. These modes rely on semantic elements to identify what to show.
Benefit 5
Future Compatibility
HTML standards evolve. Pages written with correct semantics tend to remain valid and functional longer than pages that abuse generic containers.
A Standard, Not a Style Preference

Semantic HTML is not optional for professional web development. It is the foundation of accessible, maintainable, and well-indexed websites.

Section 6.7

<main>, <section>, and <article>

Chapter 5 introduced the three-part skeleton: <html>, <head>, and <body>. Everything visible to the user lives inside <body>. But <body> can contain many different kinds of content — primary content, navigation, sidebars, headers, and footers. HTML5 introduced several landmark elements to describe the roles of these regions.

<main> — The Main Content

The <main> element wraps the primary, unique content of the page. There should be exactly one <main> element per page, and it should not contain content that repeats across pages, like navigation menus, site headers, or site footers.

<body>
  <header>
    <!-- Site header: logo, navigation -->
  </header>

  <main>
    <!-- This page's unique content: the article, the product, the form, etc. -->
    <h1>Understanding the Water Cycle</h1>
    <p>The water cycle describes the continuous movement of water on,
    above, and below Earth's surface.</p>
  </main>

  <footer>
    <!-- Site footer: copyright, links -->
  </footer>
</body>
Skip to Main Content

Screen readers allow users to jump directly to <main> to skip repeated navigation — a feature often called "skip to main content." When you use <main> correctly, this works automatically.

<section> — A Thematic Grouping

The <section> element represents a standalone thematic section of a document or page — a group of related content that has a heading. It is not a general-purpose container. If you are reaching for <section> but cannot give the section a heading, you may want <div> instead.

<main>
  <h1>Our Services</h1>

  <section>
    <h2>Web Design</h2>
    <p>We create responsive, accessible websites for businesses of all sizes.</p>
  </section>

  <section>
    <h2>Graphic Design</h2>
    <p>Our design team produces logos, branding, and print materials.</p>
  </section>
</main>

Each <section> groups a distinct theme — one for web design, one for graphic design — and each has its own <h2>.

<article> — Self-Contained Content

The <article> element represents a piece of content that is self-contained and independently distributable. If you could take the content out of the page, publish it somewhere else, and it would still make complete sense on its own, it is likely an <article>. News articles, blog posts, product listings, forum posts, and user reviews are good candidates:

<main>
  <h1>Latest News</h1>

  <article>
    <h2>New Library Opens in Downtown</h2>
    <p>The Riverside Public Library opened its doors this morning to
    more than three hundred visitors.</p>
  </article>

  <article>
    <h2>Annual Science Fair Announced</h2>
    <p>The regional science fair will take place on March 15th at the
    community center.</p>
  </article>
</main>

Choosing Between <section> and <article>

The distinction is not always obvious. Here is a simple test:

Decision Test
Section, article, or div?
  • ArticleCould this content stand alone, outside of this page, and still make sense? If yes, it is likely an <article>.
  • SectionIs this a thematic grouping within a larger whole, not intended to stand alone, but with its own heading? If yes, it is likely a <section>.
  • DivIs this just a layout container with no semantic purpose and no heading of its own? If yes, use <div>.

These elements can also nest. An <article> can contain multiple <section> elements. A <section> can contain <article> elements. The structure should reflect the content, not your desired layout.

Section 6.8

<header>, <footer>, and <aside>

Three more landmark elements help describe the regions of a page or a piece of content.

<header> — Introductory Content

The <header> element represents introductory content for its parent element. There are two common uses.

Page-level header: At the top of the page, the <header> typically contains the site name or logo, the main navigation, and sometimes a search bar.

<body>
  <header>
    <a href="/">
      <img src="images/logo.png" alt="Riverdale Academy">
    </a>
    <nav>
      <ul>
        <li><a href="/">Home</a></li>
        <li><a href="/courses">Courses</a></li>
        <li><a href="/contact">Contact</a></li>
      </ul>
    </nav>
  </header>

  <main>
    <!-- page content -->
  </main>
</body>

Article-level header: Inside an <article>, a <header> introduces that article — its title, author, and publication date.

<article>
  <header>
    <h2>City Council Approves New Budget</h2>
    <p>Published by <span>Maria Chen</span> on <time datetime="2025-09-12">September 12, 2025</time></p>
  </header>
  <p>The city council voted 7-2 in favor of the new budget proposal
  at Tuesday evening's meeting.</p>
</article>

A page can have multiple <header> elements — one at the page level and one inside each article — because each introduces the content of its parent element.

<footer> — Concluding Content

The <footer> element works like <header> in reverse. At the page level, it typically contains copyright information, legal links, contact details, and secondary navigation. Inside an article, it might contain the author's biography, related links, or a citation.

<footer>
  <p>© 2025 Riverdale Academy. All rights reserved.</p>
  <nav>
    <ul>
      <li><a href="/privacy">Privacy Policy</a></li>
      <li><a href="/accessibility">Accessibility Statement</a></li>
    </ul>
  </nav>
</footer>

Like <header>, a page can have multiple <footer> elements.

<aside> — Tangentially Related Content

The <aside> element represents content that is tangentially related to the surrounding content — content that could be removed without affecting the main content flow. Pull quotes, sidebars, related links, advertisements, and supporting background information are typical uses.

<main>
  <article>
    <h2>The History of the Internet</h2>
    <p>The internet began as a military research network called ARPANET
    in the late 1960s.</p>

    <aside>
      <h3>Key Date</h3>
      <p>ARPANET sent its first message on October 29, 1969, between
      UCLA and Stanford.</p>
    </aside>

    <p>Over the following decades, the network expanded to include
    universities and eventually the general public.</p>
  </article>
</main>

An <aside> can appear inside an <article> as supplemental content within that article, or it can appear as a sibling of <main> to serve as a page sidebar.

Section 6.9

Landmarks and Page Meaning

The elements you learned in Sections 6.7 and 6.8 — <main>, <section>, <article>, <header>, <footer>, and <aside> — are called landmark elements. They identify the major regions of a page. There is one more landmark element that belongs in this list: <nav>.

<nav> — Navigation

The <nav> element marks a set of navigation links — links that help the user move around the site or page. A page can have multiple <nav> elements: one for the primary site navigation, one for a table of contents, one for footer links.

<header>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
      <li><a href="/resources">Resources</a></li>
    </ul>
  </nav>
</header>

The aria-label attribute on <nav> gives the navigation a human-readable name, which helps screen reader users distinguish between multiple navigation regions on the same page. You will learn more about aria-* attributes in Chapter 29.

How Landmarks Help Users

When a page uses landmark elements correctly, users of assistive technologies can do something remarkable: they can jump directly to any major region of the page. Screen readers provide keyboard shortcuts and menus that let users say, "Go to main content," "Go to navigation," or "Go to footer" — and the screen reader jumps immediately to the correct <main>, <nav>, or <footer> element.

A Real Accessibility Problem

This is why using <div> for everything is a real accessibility problem, not just a style preference. A page full of <div> elements has no landmarks. A user relying on keyboard navigation has no way to skip to the content they need.

A Typical Landmark Structure

Here is what a well-structured page outline looks like using landmark elements:

<body>
  <header>                    ← Page header: logo, site nav
    <nav>                     ← Primary navigation
    </nav>
  </header>

  <main>                      ← Unique page content
    <h1>Page Title</h1>

    <section>                 ← Major section
      <h2>Section One</h2>

      <article>               ← Self-contained piece
        <header>              ← Article header
          <h3>Article Title</h3>
        </header>
        <p>Content...</p>
        <footer>              ← Article footer
          <p>Author info</p>
        </footer>
      </article>

    </section>

    <aside>                   ← Tangential sidebar
      <h2>Related Links</h2>
    </aside>

  </main>

  <footer>                    ← Page footer
    <nav>                     ← Footer navigation
    </nav>
  </footer>
</body>

This structure might seem like a lot of elements for a simple page. For very short pages — a single article, a contact form, a landing page — you will use fewer of these elements. But for any page with multiple regions, meaningful sections, and navigation, this kind of structure is appropriate and expected.

Section 6.10

Building a Structured Article Page

This section brings together everything covered in the chapter. You will build a structured, semantically correct article page step by step. The page represents a single article on an educational website. It has a site header with navigation, a main content area with the article, a sidebar with related links, and a footer.

Step 1: Plan the Content

Before writing code, identify what each piece of content actually is:

  • Site name and main navigation → <header> containing the site name and <nav>
  • Article title → <h1> inside <article> inside <main>
  • Article publication info → <header> inside <article>
  • Article sections → <section> inside <article>, each with an <h2>
  • Emphasized terms → <em> or <strong> inside <p>
  • Sidebar with related links → <aside> inside <main>
  • Footer copyright and legal links → <footer>

Step 2: Build the Page Skeleton

Start with the HTML document structure from Chapter 5, then add landmark regions:

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>How Photosynthesis Works — Science Today</title>
</head>
<body>

  <header>
    <!-- Site header goes here -->
  </header>

  <main>
    <!-- Main content goes here -->
  </main>

  <footer>
    <!-- Site footer goes here -->
  </footer>

</body>
</html>

Step 3: Build the Site Header

<header>
  <p><strong>Science Today</strong></p>
  <nav aria-label="Main navigation">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/articles">Articles</a></li>
      <li><a href="/topics">Topics</a></li>
      <li><a href="/about">About</a></li>
    </ul>
  </nav>
</header>
Note

The site name uses <strong> inside a <p> here because it is not a heading — it is the site name in the header. Once CSS is applied, it will look like a logo. You could also use a heading element for the site name; conventions vary. What matters is that you make a conscious choice based on meaning.

Step 4: Build the Article

<main>

  <article>

    <header>
      <h1>How Photosynthesis Works</h1>

Chapter 6 — Back Matter

Chapter Summary

  • Section 6.1HTML describes meaning, not appearance. Assistive technology, search engines, and future developers all depend on accurate structural markup rather than visual presentation.
  • Section 6.2<h1> through <h6> form a hierarchy, not a set of font sizes. Use exactly one <h1> per page and never skip heading levels.
  • Section 6.3<p> contains prose. Block elements should not nest inside a paragraph. <br> creates a meaningful line break within a paragraph — it is not a substitute for a new paragraph or for CSS spacing.
  • Section 6.4<em> marks stress emphasis and <strong> marks strong importance. <i> and <b> exist for alternate voice and attention-drawing text without implying emphasis or importance.
  • Section 6.5<br> and <hr> are void elements with semantic purpose — a meaningful line break and a thematic break — not visual spacing tools.
  • Section 6.6Semantic HTML chooses elements based on what content is, not how it should look. Generic containers — <div> and <span> — exist for when no semantic element applies.
  • Section 6.7<main> wraps a page's unique content (one per page). <section> groups themed content with a heading. <article> marks self-contained, independently distributable content.
  • Section 6.8<header> introduces, and <footer> concludes, the content of their parent element — at the page level or inside an <article>. <aside> marks tangentially related content.
  • Section 6.9<nav> marks navigation links. Landmark elements together let assistive technology users jump directly to any major region of a page.
  • Section 6.10A complete article page combines a correct heading hierarchy, semantic content elements, and landmark regions — all built before a single line of CSS.

Key Terms

semantic HTML
HTML written to describe the meaning of content, not its visual appearance.
heading hierarchy
The ordered use of <h1> through <h6> to represent the outline of a document without skipping levels.
landmark element
An HTML element that identifies a major region of a page, such as <main>, <nav>, <header>, <footer>, <aside>, and <section>.
<main>
The element that wraps the primary, unique content of a page. One per page.
<section>
A thematic grouping of content, typically with its own heading. Not a general-purpose container.
<article>
A self-contained, independently distributable piece of content, such as a news article, blog post, or product listing.
<header>
Introductory content for its parent element. Can appear at the page level or inside <article>, <section>, and other elements.
<footer>
Concluding content for its parent element. Can appear at the page level or inside <article> and other elements.
<aside>
Content that is tangentially related to the surrounding content; a sidebar, pull quote, or supporting note.
<nav>
A set of navigation links.
<em>
Stress emphasis on text; typically rendered in italics by default.
<strong>
Strongly important text; typically rendered in bold by default.
<br>
A hard line break within a paragraph or other inline content. A void element.
<hr>
A thematic break between sections of content. A void element.
<div>
A generic block-level container with no semantic meaning. Used for layout purposes when no semantic element applies.
<span>
A generic inline container with no semantic meaning. Used for styling purposes when no semantic element applies.
