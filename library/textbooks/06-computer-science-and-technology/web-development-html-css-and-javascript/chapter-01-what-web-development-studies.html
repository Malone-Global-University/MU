Chapter 1 — Front Matter

Chapter Introduction and Learning Objectives

Open a browser. Type an address. A page appears. That page did not arrive by magic. Someone wrote it. Someone organized its words, images, and links into a structure the browser could read. Someone chose its colors, fonts, and layout. Someone decided which buttons would respond when clicked and what would happen when a form was submitted.

This chapter opens the door to that work. Before you write a single tag, before you learn what a selector is or how a browser renders a page, you need a map of the territory. You need to understand what web development actually is, what the three core browser languages do, how websites differ from web apps, why accessibility and responsiveness matter, and how to approach this subject as a serious student. That is exactly what this chapter provides.

Opening Question

A recipe site wants every dish to be readable on a phone, a tablet, and a desktop without building three separate websites. A school wants its contact form to work for a student who cannot use a mouse. A small business wants a navigation menu that collapses into a single tappable icon on a narrow screen and expands into a full menu on a wide one. What do these three problems have in common? Each one is solved with the same three browser languages: HTML for structure, CSS for style, and JavaScript for behavior. By the end of this chapter, you will have the vocabulary and the mental map to understand what that means.

Section 1.1

What Web Development Studies

Web development is the practice of building those pages and sites. It is the work of creating the documents, interfaces, visual designs, interactive systems, and project structures that make up the web.

Web developers write code. But they also make decisions. They decide how content is organized, how a page should look on a phone compared to a laptop, whether a button is reachable by keyboard, whether an image communicates meaning to someone who cannot see it, and whether a form clearly tells a user what went wrong when they fill it out incorrectly.

Web development sits at the intersection of writing, design, logic, and communication. You do not have to be an artist to be a web developer. You do not have to be a mathematician. You do need patience, careful attention to detail, the ability to read carefully and follow structure, and the willingness to experiment, break things, and fix them.

This textbook teaches the core of browser-native web development: HTML for structure, CSS for style and layout, and JavaScript for behavior and interaction. These three languages run in every web browser without any additional software. They are the foundation of all web work, from the simplest personal page to the most complex web application.

A student who finishes this course will be able to:

  • Write well-structured, semantic HTML documents
  • Apply CSS to control visual design and layout, including responsive behavior across screen sizes
  • Use JavaScript to add interaction, read and change page content, and process form input
  • Organize multi-page web projects with clean file structure and working links
  • Build accessible pages that work for a wide range of users
  • Test, debug, and publish a complete small website
Study Note

That is the destination. This chapter is the first step. Everything here is conceptual groundwork — you will not write a full website yet. You will build the mental model that makes everything else in this textbook make sense.

Section 1.2

Websites, Web Pages, and Web Apps

These three terms are often used loosely in conversation, but it is worth being precise about them.

A web page is a single document. It has a URL. It can be opened in a browser. It is written in HTML and may include CSS and JavaScript. A web page has content: text, images, links, headings, paragraphs. Your school's contact page is a web page. A recipe on a cooking site is a web page.

A website is a collection of related web pages organized together under a shared domain name or project. The cooking site that contains thousands of recipes is a website. Your school's full online presence is a website. Websites have navigation: links that connect pages together into a coherent structure.

A web application, often called a web app, is a website that does things. It is not just information displayed for reading. A web app accepts user input, stores and retrieves data, processes requests, and produces personalized results. A student grade portal is a web app. An online store with a shopping cart is a web app. A weather service that shows your local forecast is a web app.

Where This Textbook Draws the Line

The line between a complex website and a simple web app is not always clear. In practice, the distinction matters less than understanding that web pages are made of HTML, CSS, and JavaScript, and that this textbook teaches the foundation needed for all of them.

This textbook focuses on static and lightly interactive websites: pages with real structure, real style, real responsiveness, real accessibility, and real JavaScript interaction, but without server-side processing, databases, or user account systems. Those topics belong to later courses. The foundation you build here carries directly into all of them.

Section 1.3

HTML, CSS, and JavaScript

Every page in a web browser is built from three languages working together. Understanding what each one does — and only what each one does — is one of the most important concepts in this textbook.

HTML: Structure and Meaning

HTML stands for HyperText Markup Language. HTML is the language used to write the structure and content of a web page.

When you write HTML, you describe what things are. You say: this is a heading. This is a paragraph. This is a navigation menu. This is an image, and here is its alternative description. This is a list. This is a form with a label and an input.

HTML does not decide what things look like. HTML does not make things move or respond. HTML says what things are.

HTML Example
<h1>Welcome to the Learning Center</h1>
<p>Browse our library of resources below.</p>

This HTML says: there is a level-one heading with the text "Welcome to the Learning Center," followed by a paragraph. That is all it says. It says nothing about color, font size, placement on the page, or what happens when a user clicks anything. That is CSS and JavaScript's job.

CSS: Style and Layout

CSS stands for Cascading Style Sheets. CSS is the language used to control the visual presentation of a web page.

When you write CSS, you describe how things look. You say: headings should be dark blue and 32 pixels tall. Paragraphs should have generous line spacing. Navigation links should be arranged in a horizontal row with space between them. On small screens, cards should stack into a single column.

CSS does not describe content. It does not say "there is a heading here." It says "whatever headings exist should look like this."

CSS Example — Styling the HTML Above
h1 {
  color: #1a2e4a;
  font-size: 2rem;
}

p {
  line-height: 1.7;
  color: #333333;
}

This CSS says: level-one headings should be rendered in a dark navy color at a large size, and paragraphs should have comfortable line spacing in a dark gray color. This style applies to any h1 or p element on the page — not just one specific element.

JavaScript: Behavior and Interaction

JavaScript is the programming language of the browser. JavaScript adds behavior and interaction to web pages.

When you write JavaScript, you describe what should happen when events occur. You say: when the user clicks this button, show a hidden menu. When the form is submitted, check whether the email field is filled in. When the page loads, fetch a list of resources and display them as cards.

JavaScript can read and change HTML. It can apply and remove CSS styles. It can respond to user input, validate forms, load data, and update content without refreshing the page.

JavaScript Example
const button = document.querySelector("#toggle-menu");

button.addEventListener("click", function () {
  const menu = document.querySelector("#main-nav");
  menu.classList.toggle("open");
});

This JavaScript says: find the element with the id toggle-menu, and when the user clicks it, find the element with the id main-nav and toggle a class called open on it. That CSS class might change the menu from hidden to visible.

You do not need to understand this code fully yet. The point is that JavaScript responds to things happening and changes the page in response.

The Three Languages Together

HTML, CSS, and JavaScript are designed to work together. Each one has a specific role:

LanguageRoleHandles
HTMLStructure and meaningContent, headings, paragraphs, links, images, forms, lists, tables
CSSStyle and layoutColors, fonts, spacing, layout, responsive behavior, visual design
JavaScriptBehavior and interactionEvents, DOM changes, form validation, dynamic content, data fetching
Keep the Roles Separate

One of the most important habits you will build in this course is keeping these three roles separate. When a new student wants to change how text looks, they sometimes reach for HTML. When they want to respond to a click, they sometimes reach for CSS. These instincts are wrong and lead to messy, hard-to-maintain code. HTML structures, CSS styles, JavaScript behaves. Keep them separate and your code will be readable, maintainable, and professional.

Section 1.4

Structure, Style, and Interaction

To build intuition for the three-language system, consider a physical analogy. Imagine a classroom.

A Classroom Analogy
Structure, style, and interaction in a physical space
  • StructureThe architecture: the walls, the floor, the doors, the windows, the rows of desks, the whiteboard at the front. Structure defines what is present and how it is organized. Without structure, there is nothing to work with.
  • StyleThe visual design: the color of the walls, the type of lighting, the font on the signs, the size of the desks, the arrangement that leaves comfortable space between rows. Style does not change what is in the room, but it profoundly affects how the room feels and functions.
  • InteractionThe behavior: the lights that turn on when someone flips the switch, the door that opens when pushed, the projector that responds to a remote, the bell that signals the end of class. Interaction is what happens when someone acts on the environment.

Remove the structure and nothing else makes sense. Remove the style and everything is present but harsh and unusable. Remove the interaction and the room is static and unresponsive.

Websites work the same way. HTML is the architecture. CSS is the visual design. JavaScript is the behavior. Together they create a usable, meaningful, responsive, accessible digital space.

Why Separation Matters

You might wonder: why keep these three things separate? Why not just put everything in one place?

Separation exists because it makes maintenance possible. Consider a website with one hundred pages. If the colors of all the headings are controlled by a single CSS file, changing the color everywhere takes one edit in one file. If the colors were written directly into every HTML element on every page, changing the color would require opening one hundred files and making hundreds of changes. Separation makes large projects manageable.

Separation also makes teamwork possible. A content writer can focus on HTML. A visual designer can focus on CSS. A programmer can focus on JavaScript. Each person can work on their layer without breaking the others.

Separation makes accessibility possible. Screen readers, search engines, and other tools read the HTML structure to understand what a page contains. If structure and style are mixed together, those tools receive confusing information.

Looking Ahead

You will practice this separation throughout the entire course.

Section 1.5

Browsers and Users

When a user opens a web page, a piece of software called a browser reads the HTML, CSS, and JavaScript files and renders them into the visual page the user sees.

Browsers are translators. They read code and produce a visual, interactive experience. They handle fonts, layout, colors, and interactivity. They also handle errors: when a web developer writes broken HTML, most browsers will try to make sense of it anyway rather than showing nothing.

The major web browsers include Chrome, Firefox, Safari, Edge, and others. Each browser has its own rendering engine — the software component that interprets HTML and CSS — and while they largely agree on how to render code, they sometimes differ in small ways.

Users are the people who use websites. Users have different devices, different screen sizes, different input methods, different connection speeds, different abilities, and different goals. A user might be using a laptop with a mouse, a phone with a touchscreen, a tablet with a keyboard attached, or a desktop computer with a specialized assistive device.

Good web development means building for real users with real diversity, not just for the device sitting on your own desk.

How the Browser Reads a Page

When a user visits a URL, the browser requests the HTML file from a server. The browser reads the HTML, finds references to CSS files and JavaScript files, requests those too, and begins rendering the page.

The browser constructs something called the Document Object Model, or DOM — an internal representation of the page's structure. CSS is applied to the DOM to produce the visual layout. JavaScript can read and modify the DOM to change the page.

You will learn more about this process in Chapter 2 (How the Web Works) and Chapter 22 (The DOM). For now, the key mental model is: the browser receives your code and turns it into what the user sees and interacts with.

Browser Developer Tools

One of the most important facts about browsers is that they come with built-in developer tools. In almost any browser, pressing F12 or right-clicking on a page and choosing "Inspect" opens a panel that shows you the HTML structure of the page, the CSS rules applied to each element, a JavaScript console, network activity, and tools for simulating different screen sizes. You will use these tools constantly throughout this course. Chapter 4 covers developer tools in depth.

Section 1.6

Accessibility and Inclusive Design

When a web page is accessible, it can be understood and used by the widest possible range of people, including people who use assistive technologies.

Some users navigate the web entirely by keyboard, without a mouse. Some users use screen readers, which are software programs that read the text and structure of a page aloud. Some users have low vision and rely on the ability to zoom in significantly or to use high-contrast display modes. Some users have cognitive differences that make dense, poorly organized pages difficult to process.

Accessibility is not a special feature added to a finished website. Accessibility is woven into the decisions you make from the very beginning: how you write HTML, how you label forms, how you structure headings, how you describe images, how you write link text, how you choose colors, and how you build interactive components.

An accessible website is not just a website that works for disabled users. It is a website that works better for everyone. Clear heading structure helps screen reader users navigate, but it also helps all users scan. Descriptive link text helps visually impaired users understand where a link leads, but it also helps all users who are skimming a page. High color contrast helps users with low vision, but it also helps users reading on a bright, sunny day.

The Foundation of Accessibility

In this textbook, accessibility appears in every part of the book, not just in the final quality chapter. You will write accessible HTML from Chapter 5 onward. You will learn accessible form design in Chapter 9 and Chapter 24. You will learn accessible interactive components in Chapter 27. You will learn accessibility auditing in Chapter 29.

The most important accessibility concept to absorb right now is this: the HTML structure of a page is how assistive technologies understand what a page contains. If your HTML says a heading is a heading, a screen reader can navigate from heading to heading. If your HTML puts heading-like text in a generic element with big font styling, a screen reader sees no headings at all. Structure and meaning — the job of HTML — are the foundation of accessibility.

Section 1.7

Responsive Design and Different Screens

Websites are not viewed on one type of screen. A website might be visited on a phone with a 375-pixel-wide screen, a tablet, a laptop, a large desktop monitor, or a television. It might be viewed in portrait orientation or landscape orientation. The text size might be set larger by a user who needs it.

Responsive design is the practice of building websites that adapt to different screen sizes and contexts. A responsive website does not create separate designs for every possible device. Instead, it creates a flexible layout that rearranges itself based on the available width.

Responsive design is not optional on the modern web. More than half of all web traffic comes from mobile devices. A website that only works well on a large desktop is a broken website for more than half of its users.

CSS is the primary tool for responsive design. CSS includes a feature called media queries that allows you to write different styles for different screen sizes. You will learn this in depth in Chapter 17.

Consider a simple example. On a wide desktop screen, a set of product cards might be arranged in four columns side by side. On a narrow phone screen, the same cards might stack into a single column so each one is readable without horizontal scrolling. The content is the same. The HTML is the same. Only the layout changes — and CSS handles that change automatically based on the screen width.

CSS Example — A Preview of the Idea
/* Default: single column for small screens */
.card-grid {
  display: flex;
  flex-direction: column;
}

/* On screens 700px wide or wider: four columns */
@media (min-width: 700px) {
  .card-grid {
    flex-direction: row;
    flex-wrap: wrap;
  }
}

This CSS says: by default, cards stack vertically. But on screens 700 pixels wide or wider, they arrange into a row. You do not need to understand every word of this yet. The concept is what matters now: CSS responds to the screen.

Throughout This Book

Responsive design, like accessibility, appears throughout this textbook, not only in the chapters explicitly devoted to it.

Web Development in Daily Life

Web development is one of the most immediately visible disciplines in computer science. The things you build in this course are things you can open in a browser and show to anyone.

Consider what web development touches in ordinary life:

More practically, web development skills open direct pathways to careers in front-end development, full-stack development, UI/UX design, web accessibility, content management, digital marketing, database-backed web applications, and more. The skills in this course are foundational to all of them.

Section 1.9

How to Study Web Development Successfully

Web development is a skill subject, not a knowledge subject. Knowing the names of every HTML element is much less useful than being able to write valid, semantic HTML. Knowing what Flexbox is matters much less than being able to build a responsive card layout with it. This distinction matters for how you study.

Study Habit Guide
Eight habits for effective web development study
  • Read code carefullyRead every character of an example. Notice the angle brackets, which elements are inside which, the indentation, what is in quotes. Web development mistakes are often a single missing character. Read slowly, like a careful proofreader.
  • Type code by handType each example yourself instead of copying and pasting. Typing forces your hands and eyes to encounter every character and builds the muscle memory that makes coding faster over time.
  • Experiment deliberatelyWeb development has immediate visual feedback. After each example, change a color, a font size, a margin value, or add a new element and see what happens. Prediction followed by experimentation is how intuition develops.
  • Make mistakes on purposeRemove a closing tag to see what breaks. Create a CSS conflict and watch which rule wins. Intentional mistakes in a safe practice environment teach more than any explanation.
  • Use the browser constantlyEvery file you write should be opened in a browser. Write a few lines, open the browser, see what you have, adjust, refresh. The browser is your primary tool from the first line of code.
  • Read error messagesErrors will happen. Do not panic. Read console errors, validator warnings, and silent CSS failures carefully — they usually tell you exactly what is wrong and where. Learning to read error messages is a professional skill.
  • Build the running projectThroughout this course you will build a real multi-page website called the Student Learning Site. Do not skip the project work — it is where abstract concepts become concrete.
  • Keep a code notebookKeep a personal record of things that confused you, things that surprised you, and solutions you found to problems. This notebook will be valuable when you encounter the same issues again.

Key Terms

accessibility
The property of a website that allows it to be used by people with a wide range of abilities and assistive technologies.
browser
A software application that reads HTML, CSS, and JavaScript files and renders them as visual, interactive web pages. Examples include Chrome, Firefox, Safari, and Edge.
Cascading Style Sheets (CSS)
The language used to control the visual presentation and layout of HTML documents.
Document Object Model (DOM)
The browser's internal tree-structured representation of a web page's HTML, which JavaScript can read and modify.
HTML (HyperText Markup Language)
The language used to define the structure and meaning of web page content.
JavaScript
The programming language of the browser, used to add behavior, interaction, and dynamic content to web pages.
media query
A CSS feature that applies different styles based on screen size or other properties of the device.
rendering engine
The component inside a browser that interprets HTML and CSS and produces the visual layout.
responsive design
The practice of building websites that adapt their layout to different screen sizes and devices.
screen reader
Assistive technology software that reads the text and structure of a web page aloud for users who cannot see the screen.
semantic HTML
HTML written with elements chosen for their meaning, not just their visual appearance.
web application (web app)
A website that accepts user input, processes data, and produces dynamic results.
web development
The practice of building websites and web pages using structured content, visual design, interactivity, accessibility, and project organization.
web page
A single document written in HTML, accessible via a URL in a browser.
website
A collection of related web pages organized under a shared domain name.

Common Misconceptions

Misconception 1
"HTML controls how things look."
HTML does not control visual appearance. A heading written in HTML is a heading because of its meaning, not because it is large and bold. Those visual properties are applied by the browser's default CSS or by CSS you write yourself. An <h1> element could be styled to appear small and pink. It would still be a heading in meaning and structure. Never use HTML to try to control visual appearance. That is CSS's job.
Misconception 2
"Web development is mostly about memorizing tags and properties."
Experienced web developers do not hold the entire HTML specification in memory. They know the most common elements, properties, and patterns, and they look up the rest when needed. What matters is understanding why you use elements, what they communicate, and how the three-language system fits together. This understanding is transferable. A list of memorized tags is not.
Misconception 3
"JavaScript is just for making pages 'fancy.'"
JavaScript handles essential functionality on the modern web: form validation that protects users from submitting bad data, accessible interactive components like accordions and tabs, dynamic content loaded from a server, and responsive behaviors that CSS alone cannot provide. JavaScript is not decoration. It is the behavior layer of a serious system.
Misconception 4
"Accessibility is for a small percentage of users and can be added at the end."
Accessibility affects a much larger percentage of users than most people expect — not just users with permanent disabilities, but also users in temporary or situational contexts: someone with a broken arm, someone using their phone in bright sunlight, someone on a slow connection, someone using a keyboard instead of a mouse because the mouse battery died. Accessibility built in from the beginning is easy. Accessibility retrofitted onto a finished website is expensive, incomplete, and often painful.
Misconception 5
"Responsive design means making a 'mobile version' of the site."
Responsive design means making one website that works at all sizes. It does not mean building a separate website for phones and another for desktops. A responsive site has a single HTML structure and a single CSS stylesheet that adapts its layout based on the available width. Managing two separate sites is twice the work and usually results in worse experiences for users.

How the Web Works

Chapter 1 gave you a map of the field. Chapter 2 examines the web as a system. When you type a URL into a browser, a chain of events begins: a request travels to a server, the server sends back files, the browser receives them and renders the page. You will learn the difference between the internet and the web, what clients and servers are, what happens during an HTTP request, what domain names and hosting are, and how local files differ from published websites. Understanding how the web works as a system will help you make better decisions about file organization, linking, paths, and eventually publishing.
