Chapter 3 — Front Matter

Chapter Introduction and Learning Objectives

A finished website can feel like a single, seamless thing — a page with text, color, images, and buttons that all just work. But no website is actually one thing. Every website is a collection of separate files, stored in a structured set of folders, waiting to be found and assembled by a browser. The HTML file holds the words. A separate CSS file holds the colors and layout. A separate JavaScript file holds the behavior. Separate image files hold the photographs and icons. None of this comes together automatically — it comes together because the HTML file contains precise instructions for where to find each of the other pieces.

This chapter teaches you how that organization works. You will learn what files and folders are in a web project, how file extensions tell the browser what kind of content it is looking at, how the special file index.html behaves, and — most importantly — how paths describe the route from one file to another. Broken paths are one of the most common problems beginning web developers encounter, and almost all of them are preventable once you understand how paths actually work. By the end of this chapter, you will be able to build a clean, professional project folder before you write a single line of HTML.

Opening Question

Imagine you build a page on your own computer. You add a stylesheet, a few images, and a script. Everything looks perfect when you open it in your browser. Then you copy the project to a different computer, or upload it to a web server — and suddenly the images are missing, the page has no styling, and the script does not run. Nothing about the HTML itself changed. What happened? By the end of this chapter, you will understand exactly why this happens, and you will know how to build projects that do not break when they move.

Section 3.1

Websites as File Systems

When you visit a website and see a polished page with text, images, colors, and interactive buttons, it can feel like a single thing. But every website is actually a collection of separate files stored in an organized folder structure.

A typical web page requires at minimum:

  • An HTML file that holds the structure and content
  • A CSS file that holds the visual styling
  • A JavaScript file that holds the interactive behavior
  • Image files for any photographs, icons, or graphics

When a browser loads a web page, it does not load one file. It reads the HTML file first, then follows instructions inside that HTML file to load each additional resource: the stylesheet, the scripts, the images, and anything else the page needs.

If any file is missing, misnamed, or stored in the wrong folder, the browser cannot find it. The result is a broken page: missing styles, missing images, or JavaScript that does not run. The HTML itself may still display, but the page will look incomplete or broken.

This is why web developers think carefully about file organization before they begin writing code. A well-organized project folder prevents an entire category of errors.

The Mental Model

Think of a web project as a folder on your computer, not as a single document. Every file inside that folder serves a purpose. Every file has a name, a location, and a type. The browser finds files by following a path from one location to another — and the rest of this chapter is about how to write that path correctly.

Section 3.2

Files and File Extensions

Every file on a computer has two parts to its name: the base name and the extension. The extension comes after the final dot and tells the operating system and the browser what kind of file it is.

ExtensionFile TypePurpose
.htmlHTML documentPage structure and content
.cssCSS stylesheetVisual styles and layout
.jsJavaScript fileBehavior and interaction
.pngPNG imagePhotographs, graphics
.jpg / .jpegJPEG imagePhotographs, high-resolution images
.gifGIF imageAnimated images, simple graphics
.svgSVG imageScalable vector graphics, icons
.webpWebP imageModern compressed image format
.jsonJSON data fileStructured data
.icoIcon fileBrowser tab favicon
.woff2Web fontCustom font files

Each extension tells the browser how to interpret the file. When a browser receives a .css file, it reads it as styling rules. When it receives a .js file, it executes it as JavaScript code. When it receives a .png, it renders it as an image.

File Extension Visibility

Some operating systems hide file extensions by default. This is a common source of confusion for beginners. A file might appear to be named style when its real name is style.css. If you accidentally save a file as style.css.txt because extensions were hidden, the browser will not recognize it as a CSS file and your styles will not work. You should always work with file extensions visible — enable this in File Explorer's View settings on Windows, or in Finder preferences on a Mac.

A Note on Text Files

HTML, CSS, and JavaScript files are all plain text files. This means you can open and edit them in any text editor — you do not need special software to read or write them. What makes an HTML file different from a CSS file is not its internal format — both are just text — but its extension and the conventions that govern what goes inside it.

Section 3.3

Folders and Project Organization

Web projects use folders to group related files together. A folder (also called a directory) is a container that holds files and other folders.

A small project might only have three or four files, and organizing them into subfolders may feel unnecessary at first. But consider what happens as a project grows: a site with ten pages has ten HTML files; a site with images has dozens of image files; a site with multiple stylesheets or scripts has multiple .css and .js files; a site with data has JSON files. Without organization, all these files pile into one folder and become difficult to manage. With organization, each type of file lives in a logical place and the project remains readable.

The Standard Web Project Structure

A beginner web project typically uses this folder structure:

my-project/ index.html about.html contact.html css/ main.css js/ main.js images/ logo.png hero.jpg team-photo.jpg data/ resources.json
Subfolder
css/
Holds all stylesheet files. Every CSS file the project uses lives here, separate from the HTML.
Subfolder
js/
Holds all JavaScript files. Behavior code stays out of the HTML and CSS folders.
Subfolder
images/
Holds all image files — logos, photographs, icons, and graphics used across the site.
Subfolder
data/
Holds JSON and other structured data files, introduced later when pages need to render external data.

HTML files often live at the root of the project folder, where they are easy to find, though some projects use a pages/ subfolder for pages other than the home page.

Naming the Root Folder

The outer folder is the project root. Give it a meaningful name like my-portfolio, school-club-site, or learning-project. Avoid spaces in folder names; use hyphens or underscores instead.

Sections 3.4 – 3.6
Finding Files: index.html, Relative Paths, and Path Symbols
index.html · Relative Paths · ./ and ../
Section 3.4

The Role of index.html

Web servers have a special convention: when a browser requests a folder address without specifying a filename, the server automatically looks for and serves a file named index.html.

What This Means in Practice

If you visit https://example.com/, the server looks for example.com/index.html and serves it automatically.

If you visit https://example.com/about/, the server looks for example.com/about/index.html.

This is why the home page of almost every website is named index.html. It is not a rule you must follow during local development, but it is the standard convention for published websites. If you name your home page home.html instead, the server may return an error when someone visits your root address.

index.html in Every Subfolder

Some multi-page sites use a folder structure where each section of the site has its own folder, and each folder contains its own index.html:

my-project/ index.html ← home page about/ index.html ← about page contact/ index.html ← contact page

In this structure, a user visiting example.com/about/ gets the about page automatically. This creates clean URLs without .html extensions visible in the browser bar.

For your early projects, you can keep all HTML files at the root and link to them directly by name:

my-project/ index.html about.html contact.html

Both approaches work. Use whichever makes sense for the size and structure of your project.

Section 3.5

Relative Paths

A path is the address of a file within a file system. Paths tell the browser where to find a resource relative to the current file.

There are two kinds of paths: relative paths and absolute paths. Relative paths are by far the most common type used inside a web project. You will write relative paths hundreds of times as a web developer.

A relative path describes how to find a file starting from the current file's location. It does not start with a domain name or a drive letter. It starts with the file or folder name, or with a special symbol like ./ (current folder) or ../ (go up one folder).

Example Project Structure

my-project/ index.html about.html css/ main.css images/ logo.png hero.jpg js/ main.js

Reading Relative Paths from index.html

index.html lives in the project root. From index.html, here is how you would write relative paths to other files:

To Reach This FileRelative Path from index.html
about.htmlabout.html
css/main.csscss/main.css
images/logo.pngimages/logo.png
js/main.jsjs/main.js

Since index.html is in the root folder, you simply write the path from the root down into any subfolder.

Reading Relative Paths from a File Inside a Subfolder

Now suppose you have an HTML file inside a subfolder, like this:

my-project/ index.html pages/ about.html css/ main.css images/ logo.png

From pages/about.html, the paths are different because you are starting one level deeper:

To Reach This FileRelative Path from pages/about.html
index.html../index.html
css/main.css../css/main.css
images/logo.png../images/logo.png

The ../ means "go up one folder." From inside the pages/ folder, you need to go up one level to reach the root, and then down into css/ or images/.

The Forward Slash Rule

In web paths, folders are separated by forward slashes /, not backslashes \. This is true even on Windows, where the file system normally uses backslashes. In HTML, CSS, and JavaScript files, always use forward slashes in paths.

Section 3.6

Understanding Path Symbols

Three special path symbols appear frequently in web development. Learn to recognize them immediately.

Symbol
./ Current Folder
Means "start from the current folder." Often optional but makes the intent explicit. Example: ./images/logo.png means "look in the current folder, then inside images/, then find logo.png."
Symbol
../ Go Up One Folder
Means "go up one folder from my current location." Chain these together to go up multiple levels: ../../css/main.css goes up two levels, then into css/.
No Prefix
Same Folder
A filename or folder name with no prefix is looked for in the same folder as the current file. Example: about.html looks for that file beside the current one.
Worked Example
Tracing a path two levels deep
my-project/ index.html pages/ blog/ post.html css/ main.css images/ photo.jpg
  • Starting pointThe current file is pages/blog/post.html — two levels deep from the project root.
  • TargetWe want to reach css/main.css, which sits at the root level.
  • The path../../css/main.css — the first ../ goes up from blog/ to pages/; the second ../ goes up from pages/ to the root; then css/main.css goes down into the css/ folder and finds main.css.

Section 3.7

Absolute Paths at an Introductory Level

An absolute path provides the complete address of a file from a fixed starting point. There are two kinds of absolute paths in web development.

File System Absolute Paths

These begin from the root of your operating system's file system:

Windows: C:\Users\YourName\Documents\my-project\index.html Mac / Linux: /Users/YourName/Documents/my-project/index.html
Why You Should Avoid These

These paths work on your local computer but break immediately when you publish your site, because another server does not have the same folder structure as your computer. You should almost never use file system absolute paths in web projects.

Root-Relative Paths

Root-relative paths begin with a single forward slash /:

/css/main.css /images/logo.png /about.html

The / at the start means "start from the root of the website," not the root of your operating system. A root-relative path of /css/main.css tells the browser to go to the top of the domain and look for css/main.css there.

Root-relative paths work well on published websites but may not behave as expected when you open HTML files directly from your desktop by double-clicking them, before setting up a local server. For this reason, relative paths without a leading / are generally recommended for beginner projects.

When You Will See Absolute Paths

Absolute paths appear in two common situations: links to external websites (https://www.example.com/page.html) and links to resources on a content delivery network, such as a font service. For everything inside your own project, prefer relative paths.

Section 3.8

Linking CSS, JavaScript, Images, and Pages

Now that you understand paths, you can see how they are used in HTML. Paths appear in several different HTML elements and attributes.

Example 3.8.1
Four ways paths show up in HTML
  • StylesheetA <link> element in the head connects an external stylesheet: <link rel="stylesheet" href="css/main.css">. The href attribute holds the path to the CSS file.
  • ScriptA <script> element near the bottom of the body connects a JavaScript file: <script src="js/main.js"></script>. Scripts are typically placed just before the closing </body> tag so the HTML is fully loaded before the script runs.
  • ImageAn <img> element uses src to point to an image file: <img src="images/logo.png" alt="Company logo">.
  • Page linkAn <a> element uses href to link to another HTML file: <a href="about.html">About Us</a>.
Checking Paths Before You Write Them

Before writing any path, ask yourself two questions: Where is the current file located? Where is the target file located? Then trace the route from the current file to the target file using the path symbols you learned in Section 3.6. If you answer both questions correctly, the path will work.

Section 3.9

Naming Files for the Web

File naming matters more in web development than in everyday computer use. Web servers and web browsers are sensitive to filenames in ways that a desktop application is not.

Rules for Web-Safe Filenames

Use Only Lowercase Letters

Web servers on Linux (which hosts the majority of websites) are case-sensitive. If your image file is named Logo.PNG but your HTML references logo.png, the server will report a file-not-found error. Local development on Windows or Mac may work because those systems are case-insensitive, hiding the problem until you publish.

✓ logo.png    ✗ Logo.PNG    ✗ LOGO.PNG

Use hyphens to separate words. Avoid spaces entirely. A filename like my logo.png becomes my%20logo.png in a URL, because spaces must be URL-encoded, which is harder to read and type. Hyphens are standard and readable.

✓ team-photo.jpg    ✓ main-stylesheet.css    ✗ team photo.jpg

Keep names short and descriptive. A file named img1.jpg tells you nothing. A file named hero-background.jpg tells you exactly what it is and where it is used.

✓ hero-background.jpg    ✓ contact-form.js    ✗ img1.jpg    ✗ final-version-REAL.html

Avoid special characters. Do not use characters like #, ?, &, %, !, @, (, ), or + in filenames — these characters have special meanings in URLs.

Use consistent extensions. Always include the correct file extension and use the lowercase version. Prefer .jpg over .JPG, .html over .HTML.

Keep folder names lowercase and hyphenated as well: ✓ css/, ✓ about-us/ rather than ✗ CSS/, ✗ My Images/.

Section 3.10

Common Path Mistakes

Even experienced web developers occasionally break links with path errors. Learning to recognize common mistakes now will save you significant debugging time.

Mistake 1 — Wrong Starting Point
A file in pages/about.html writes href="css/main.css", which the browser reads as pages/css/main.css — a file that does not exist.
The fix: Always ask yourself where the current file is located, not just where the target file is. From inside pages/, the correct path is ../css/main.css.
Mistake 2 — Wrong Case
HTML references Images/Logo.PNG but the file is actually stored as images/logo.png.
The fix: Use consistent lowercase filenames and match them exactly in your HTML. This will fail on a case-sensitive server even though it may work locally.
Mistake 3 — Missing Extension
href="css/main" is missing the .css extension. The browser will request a file literally named main, not main.css.
The fix: Always include the file extension in paths.
Mistake 4 — Backslashes Instead of Forward Slashes
src="images\photo.jpg" uses a backslash, which is the Windows file system convention, not the web convention.
The fix: Always use forward slashes / in HTML, CSS, and JavaScript, regardless of which operating system you use to write the code.
Mistake 5 — Extra Spaces
src=" images/photo.jpg" has a leading space, causing the browser to look for a path that literally starts with a space character.
The fix: Check paths carefully for accidental spaces, especially near quote marks.
Mistake 6 — Absolute File System Path
href="C:\Users\YourName\Documents\my-project\css\main.css" works on one specific computer and breaks everywhere else.
The fix: Always use relative paths inside a project.
How to Debug a Broken Path

When a resource does not load, open your browser's developer tools (covered in detail in Chapter 4) and check the Console or Network panel. The browser will show an error message identifying which file it could not find and what path it tried to load. This is the fastest way to diagnose and fix path errors.

Section 3.11

Building a Clean Starter Project

Now that you understand files, folders, paths, extensions, and naming, you can build a clean starter project structure before you write any HTML, CSS, or JavaScript. A clean starter project gives you a professional foundation for every project you build, a consistent habit that prevents path errors, a structure that scales as your project grows, and a framework that mirrors how professional web projects are organized.

Step-by-Step
Creating a starter project
  • Step 1Create the project root folder. Give it a lowercase, hyphenated name, such as my-learning-site/.
  • Step 2Create the required subfolders inside the root: css/, js/, and images/. You can add a data/ folder later if your project uses JSON files.
  • Step 3Create the starting files: an empty index.html in the root, an empty main.css inside css/, and an empty main.js inside js/.
  • Step 4Open the entire project folder — not just one file — in your text editor. Most modern editors, like Visual Studio Code, show the full file tree in a sidebar.
  • Step 5Add the HTML skeleton to index.html, with correct relative paths to the CSS and JavaScript files.
  • Step 6Open index.html in a browser and check the developer tools console for errors. If the CSS and JavaScript files load without errors, your paths are correct.
my-learning-site/ index.html css/ main.css js/ main.js images/

The starter index.html file, with the skeleton you will learn in full detail in Chapter 5:

<!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>My Learning Site</title> <link rel="stylesheet" href="css/main.css"> </head> <body> <h1>Welcome to My Learning Site</h1> <script src="js/main.js"></script> </body> </html>

Notice the paths used here: href="css/main.css" finds main.css inside the css/ folder, and src="js/main.js" finds main.js inside the js/ folder. Both paths are relative, starting from index.html's location in the project root.

Key Terms

absolute path
A path that specifies a file's full location from a fixed root, such as a domain root or operating system root. Contrast with relative path.
directory
Another word for folder. A container that holds files and other folders.
extension
The letters after the final dot in a filename, indicating the file type. For example, .html, .css, .js, .png.
file
A named collection of data stored on a computer. Web projects consist of multiple files of different types.
file path
The address of a file on a computer or website, specifying its location within a folder structure.
folder
A container that holds files and other folders. Used to organize web project files by type or purpose.
index.html
The default filename that web servers look for when a browser requests a folder address. Conventionally used for home pages and section landing pages.
path separator
The forward slash / used between folder names and filenames in web paths.
project root
The outermost folder of a web project, containing all other files and folders for that project.
relative path
A path that describes how to find a file starting from the current file's location. Does not begin with a domain name or drive letter.
root-relative path
A path that begins with /, starting from the root of the website domain.
ellipsis notation ( ... )
In file trees and documentation, notation used to indicate that additional files or folders exist but are not shown.
../
A path symbol meaning "go up one folder level."
./
A path symbol meaning "the current folder." Often optional but sometimes used for clarity.

Project Structure Checklist
Use this checklist when starting any new web project
  • Project root folder has a lowercase, hyphenated name
  • index.html is at the project root
  • css/ subfolder exists
  • js/ subfolder exists
  • images/ subfolder exists
  • All filenames use lowercase letters only
  • All filenames use hyphens, not spaces
  • All filenames include the correct extension
  • CSS file is linked in the <head> with a correct relative path
  • JavaScript file is linked before </body> with a correct relative path
  • Project folder is opened as a folder in the text editor
  • No file system absolute paths appear in any HTML file
  • Browser developer tools show no path errors on load

Common Misconceptions

Misconception 1
"I can use backslashes in HTML paths."
No. HTML always uses forward slashes /. Backslashes are a Windows file system convention. Web paths, URLs, and HTML attributes use forward slashes on all platforms.
Misconception 2
"File names don't matter as long as the path is right."
They do matter. File names must match exactly, including capitalization, on case-sensitive servers. A file named Photo.jpg and a reference to photo.jpg will fail on Linux servers even though they might work on your Windows or Mac computer during development.
Misconception 3
"I can figure out the path structure later after I build the pages."
It is much easier to set up the project structure first and write correct paths from the beginning than to rename and reorganize files after the project is partially built. Reorganizing files requires updating every path reference in every file that links to them.
Misconception 4
"Absolute paths are safer because they always point to the right place."
File system absolute paths are actually more fragile, not safer. A path like C:\Users\YourName\Documents\site\css\main.css only works on your computer. Relative paths work wherever the project folder is located.
Misconception 5
"The index.html name is just a tradition I can ignore."
For local development, you can use any name and open the file directly. But for published websites, most servers are configured to serve index.html when a folder address is requested. Ignoring this convention causes problems at publication time.
