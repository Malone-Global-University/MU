Chapter 28 — Front Matter

Chapter Introduction and Learning Objectives

Every website you have built so far has contained content that was written directly into HTML files. The text, the links, the images, the form labels — all of it lived inside the HTML you typed by hand. This approach works well for small sites. But many modern websites display content that changes often: news articles, product listings, user profiles, event schedules, blog posts, resource directories. Writing that content by hand into HTML every time something changes would be tedious, error-prone, and difficult to maintain.

There is a better approach. Websites can store content separately from the HTML that displays it, in a format called JSON (JavaScript Object Notation). JavaScript can then read that content and build the page dynamically. For content that lives on a remote server, JavaScript can fetch it over the network. This chapter introduces both JSON and fetching at a beginner level — vocabulary and habits that connect directly to later courses in JavaScript, front-end frameworks, APIs, databases, and full-stack development.

Opening Question

A resource directory page needs to display eight learning resources today. Next year it might need to display two hundred. A teammate wants to add a new resource without touching a single line of HTML. A visitor on a slow connection should never see a blank, unexplained gap on the page while content loads — and if the data fails to arrive at all, the page should still make sense. How do you build a page like that? The answer is to separate the content from the page that displays it, and to teach JavaScript how to load that content, render it, and respond gracefully when something goes wrong. That is exactly what this chapter teaches.

Sections 28.1 – 28.3
The Case for Data: Why Separate Content from Code
Data-Driven Thinking · JSON Syntax · Objects and Arrays
Section 28.1

Why Websites Use Data

Open any large website and look carefully at what you see. A news site has dozens of article summaries, each with a headline, a photo, a category, and a short description. An online store has hundreds of products, each with a name, a price, an image, and a stock status. A school website has a directory of staff, each with a name, a title, a department, and a photo.

If each of those items were written manually into HTML, maintaining the site would be a continuous struggle. Adding a new article would mean opening the HTML file, writing the markup, making sure the structure matches every other article, saving, uploading, and testing. Changing a price would mean hunting through the HTML for the right line. Removing a staff member who left would mean finding every page that mentioned them.

Instead, most websites separate content from presentation. Content lives in a structured data format. Presentation logic lives in HTML, CSS, and JavaScript templates. When the page loads, JavaScript reads the data and builds the HTML automatically. Changing a product price means changing a number in a data file, not editing HTML.

A Key Idea

This separation of data from presentation is one of the most important architectural ideas in web development. Even at a beginner level, learning to work with separate data makes your websites easier to update, extend, and eventually connect to real databases and APIs.

Data-Driven Thinking

When you think about a website as data-driven, you start asking a different set of questions:

  • What are the repeating units of content on this page?
  • What pieces of information does each unit need?
  • Where does that information come from?
  • When should it be loaded?
  • What should the page show while the data is loading?
  • What should the page show if the data fails to load?

These questions guide the work of this entire chapter.

Section 28.2

What JSON Is

JSON stands for JavaScript Object Notation. It is a lightweight text format for storing and exchanging structured data. JSON looks similar to JavaScript objects and arrays, which makes it easy to read and write for anyone who has worked through the earlier chapters of this textbook.

JSON was designed to be readable by humans and easy to parse by machines. It is now the most widely used data format on the web. When one website or application sends data to another — through an API, a database query, or a file — JSON is usually the format used.

JSON files typically have a .json file extension. They are plain text files, which means you can open them in any text editor. They contain no HTML, no CSS, and no JavaScript — just structured data. Here is a simple example of JSON representing a single person:

JSON — a single object
{
  "name": "Amara Osei",
  "role": "Student",
  "year": 2,
  "active": true
}

This should look familiar. The structure uses curly braces, property names in double quotes, a colon between each name and value, and commas between properties. This is the same syntax as a JavaScript object. There are a few important rules for valid JSON:

  • All property names must be in double quotes. Single quotes are not allowed in JSON.
  • Values can be strings (in double quotes), numbers (without quotes), Booleans (true or false), null, arrays, or objects.
  • There must be no trailing commas. A comma after the last property in an object is a JSON error.
  • JSON does not support comments.
Invalid JSON — single quotes, unquoted keys, trailing comma
// INVALID JSON — do not do this
{
  name: "Amara Osei",
  role: 'Student',
  year: 2,
  active: true,
}
Corrected — valid JSON
{
  "name": "Amara Osei",
  "role": "Student",
  "year": 2,
  "active": true
}
Why These Rules Matter

Unlike JavaScript, which is flexible about many syntax issues, JSON is strict. A single missing double quote, a trailing comma, or an unquoted property name will cause a JSON parse error. This strictness is intentional: it keeps JSON simple and consistent across many different programming languages, not just JavaScript. When you write JSON by hand, always validate it. There are free online JSON validators you can paste your JSON into to check for errors.

Section 28.3

Arrays and Objects in JSON

JSON supports two main structures: objects and arrays. These can be nested inside each other in any combination.

JSON Objects

A JSON object is a collection of key-value pairs. Keys are strings. Values can be any valid JSON value. Objects are wrapped in curly braces.

{
  "title": "Introduction to HTML",
  "chapter": 5,
  "published": true,
  "author": "MGU Editorial Team"
}

You can read this as: there is an object with four properties — title, chapter, published, and author.

JSON Arrays

A JSON array is an ordered list of values. Arrays are wrapped in square brackets. Each item is separated by a comma. Array items can be any valid JSON value — strings, numbers, Booleans, objects, or even other arrays.

[
  "HTML",
  "CSS",
  "JavaScript"
]

Arrays of Objects

The most common structure you will encounter is an array of objects. This represents a list of similar items, where each item has the same set of properties.

[
  {
    "id": 1,
    "title": "What Web Development Studies",
    "chapter": 1,
    "part": "Foundations of the Web"
  },
  {
    "id": 2,
    "title": "How the Web Works",
    "chapter": 2,
    "part": "Foundations of the Web"
  },
  {
    "id": 3,
    "title": "Files, Folders, Paths, and Project Structure",
    "chapter": 3,
    "part": "Foundations of the Web"
  }
]

This structure is extremely useful for pages like resource directories, product listings, team member rosters, event schedules, and course guides — anywhere you have a repeating collection of similar items.

Nested Objects

Objects can contain other objects as property values.

{
  "name": "Kofi Mensah",
  "contact": {
    "email": "kofi@example.com",
    "phone": "555-0102"
  },
  "department": "Computer Science"
}

Here, the contact property is itself an object with email and phone properties. To access email in JavaScript after parsing this JSON, you would write person.contact.email.

Reading JSON Structure

Before writing any code, practice reading JSON by answering these questions about a data structure: Is the top-level structure an object {} or an array []? If it is an array, what does each item look like? What are the property names? What type is each value — string, number, Boolean, array, or nested object? These reading habits will serve you well as JSON structures grow more complex.

Section 28.4

Local JSON Files

Before learning to fetch data from a remote server, you will work with local JSON files. A local JSON file lives inside your project folder, just like your CSS or JavaScript files.

Using a local JSON file is good for learning because it removes the complexity of network requests. You can focus on reading and rendering data without worrying about server errors, network delays, or CORS policies — concepts you will encounter when working with remote APIs in later courses.

Creating a JSON File

In your project folder, create a data folder. Inside it, create a file called resources.json.

What You Cannot Do with a Bare File Path

Later in this chapter, you will use fetch() to read this file. One important thing to understand early: fetch() uses HTTP to request files, which means your page must be served by a web server — even locally. You cannot simply open an HTML file by double-clicking it in your file manager and expect fetch() to load a local JSON file. Modern browsers block this for security reasons.

To work with fetch() locally, use a simple local development server. Most code editors, including Visual Studio Code with the Live Server extension, provide this. When your page is served at an address like http://localhost:5500 instead of accessed as a bare file path, fetch() will work correctly. This requirement is not a bug — it is consistent with how websites actually work. Your project runs locally on a small server, just as it would run on a real hosting server when published.

Section 28.5

Introduction to fetch()

The fetch() function is the browser's built-in tool for making network requests from JavaScript. It can request HTML, JSON, images, or any other resource from a URL. At its most basic, fetch() takes a URL as its argument and returns a response.

fetch("data/resources.json");

However, this line alone does nothing visible. It sends the request but does not do anything with the response. To work with the response, you need to understand something about how fetch() works: it is asynchronous.

Why Asynchronous Matters

When a browser sends a network request, it does not freeze and wait for the response before continuing with other work. Waiting could take milliseconds or several seconds depending on server speed, file size, and network conditions. If the browser stopped everything while waiting, the page would become unresponsive.

Instead, the browser sends the request and continues running other JavaScript. When the response arrives, a callback function handles it. This is asynchronous behavior: the request and the response happen at different times, and JavaScript continues working in between. This is different from the synchronous code you have written before, where each line executes before the next one begins.

fetch() handles this through Promises, which are introduced in the next section.

A First Complete fetch() Example

Here is a complete, working example of using fetch() to load and log a JSON file. Read it carefully before studying the explanation below.

fetch("data/resources.json")
  .then(function(response) {
    return response.json();
  })
  .then(function(data) {
    console.log(data);
  })
  .catch(function(error) {
    console.log("Error loading data:", error);
  });
Reading the Example Line by Line
  • Line 1fetch("data/resources.json") sends a request for the file at that path. It returns a Promise.
  • Lines 2–4.then(function(response) { return response.json(); }) — when the response arrives, parse its body as JSON. response.json() also returns a Promise.
  • Lines 5–7.then(function(data) { console.log(data); }) — when the JSON is fully parsed, run this function with the parsed data. At this point, data is a JavaScript array or object you can work with.
  • Lines 8–10.catch(function(error) { ... }) — if anything goes wrong at any step, run this function with the error.

If you open the browser console after loading the page, you should see the parsed array logged there.

Section 28.6

Promises at an Introductory Level

A Promise is a JavaScript object that represents the eventual result of an asynchronous operation. It is called a Promise because it represents a commitment: the result is not here yet, but it will be here eventually — or the operation will fail and the Promise will tell you that too.

State 1
Pending
The operation has started but has not finished yet. The Promise is waiting.
State 2
Fulfilled
The operation completed successfully. The Promise delivers a result value.
State 3
Rejected
The operation failed. The Promise delivers an error.

You work with Promises using the .then() and .catch() methods. .then(callback) runs the callback function when the Promise is fulfilled; the callback receives the result value as its argument. .catch(callback) runs the callback function if the Promise is rejected at any point in the chain; the callback receives the error as its argument.

Chaining Promises

Promises can be chained. When a .then() callback returns another Promise, the next .then() in the chain waits for that inner Promise to resolve. This is exactly what happens with fetch():

fetch("data/resources.json")         // Returns a Promise for a Response
  .then(function(response) {
    return response.json();           // Returns a Promise for parsed JSON
  })
  .then(function(data) {
    // data is the fully parsed JSON value
    renderResources(data);
  })
  .catch(function(error) {
    showError(error);
  });

The chain reads in order: first fetch, then parse, then use the data. If anything fails, catch the error.

A Conceptual Model

Think of a Promise like placing an order at a restaurant. The moment you order, you receive a ticket (the Promise). The food is not ready yet, but the ticket represents the commitment that food is coming. You do not stand frozen at the counter — you go sit down and wait (the browser continues running other JavaScript). When the food is ready, the waiter brings it to your table and you eat (the .then() callback runs with the result). If the kitchen runs out of ingredients, the waiter comes to tell you there is a problem (the .catch() callback runs with the error).

Modern Syntax: async and await

A newer way to write the same logic uses the async and await keywords. You may see this style in documentation and other codebases.

async function loadResources() {
  try {
    const response = await fetch("data/resources.json");
    const data = await response.json();
    renderResources(data);
  } catch (error) {
    showError(error);
  }
}

loadResources();

async before a function declaration marks the function as asynchronous. Inside an async function, await pauses execution of that function — not the entire page — until the Promise resolves, and then returns the resolved value. try/catch handles errors in place of .catch().

Which Style Should You Use?

Both the .then()/.catch() style and the async/await style accomplish the same thing. This textbook uses the .then()/.catch() style in primary examples, since it makes the Promise structure more visible, but shows the async/await style as well so you can read it when you encounter it elsewhere.

From Data to Page: Rendering, States, and Responsibility

Section 28.7

Rendering Data on a Page

Loading data and logging it to the console is only the beginning. The real goal is to display that data as visible HTML on the page. The process follows a clear pattern:

The Render Pattern
  • Step 1Fetch the data.
  • Step 2Parse the JSON.
  • Step 3Loop through the array.
  • Step 4For each item, build an HTML string or create DOM elements.
  • Step 5Insert the built HTML into the page.

Preparing the HTML

Create a container element in your HTML where the rendered content will appear.

<section id="resources-section">
  <h2>Web Development Resources</h2>
  <div id="resources-list"></div>
</section>

The <div id="resources-list"> is intentionally empty. JavaScript will fill it.

Building and Inserting HTML with a Loop

In your JavaScript file, write a function that receives the data array and renders it into the container.

function renderResources(resources) {
  const container = document.querySelector("#resources-list");

  let html = "";

  for (let i = 0; i < resources.length; i++) {
    const resource = resources[i];
    html += `
      <div class="resource-card">
        <h3>${resource.title}</h3>
        <span class="category">${resource.category}</span>
        <p>${resource.description}</p>
        <a href="${resource.url}" target="_blank" rel="noopener noreferrer">
          Visit Resource
        </a>
      </div>
    `;
  }

  container.innerHTML = html;
}
Reading the Function
  • Selectconst container = document.querySelector("#resources-list") selects the empty container div.
  • Startlet html = "" starts with an empty string that will accumulate HTML markup.
  • LoopThe for loop runs once for each resource in the array. resources[i] gives access to the current resource object.
  • BuildInside the loop, a template literal builds an HTML card for the current resource. Each property is inserted using the ${} syntax.
  • Insertcontainer.innerHTML = html replaces the container's content with the assembled HTML string.

Connecting the Fetch to the Render Function

fetch("data/resources.json")
  .then(function(response) {
    return response.json();
  })
  .then(function(data) {
    renderResources(data);
  })
  .catch(function(error) {
    console.log("Could not load resources:", error);
  });

When the page loads, JavaScript requests the JSON file, parses it, and passes the array to renderResources(), which builds and inserts the HTML. The result is a page that displays all five resources as cards, built entirely from data.

Using forEach Instead of a for Loop

Arrays in JavaScript have a built-in forEach method that runs a function once for each item. Many developers prefer it over an index-based for loop for this kind of work.

function renderResources(resources) {
  const container = document.querySelector("#resources-list");

  let html = "";

  resources.forEach(function(resource) {
    html += `
      <div class="resource-card">
        <h3>${resource.title}</h3>
        <span class="category">${resource.category}</span>
        <p>${resource.description}</p>
        <a href="${resource.url}" target="_blank" rel="noopener noreferrer">
          Visit Resource
        </a>
      </div>
    `;
  });

  container.innerHTML = html;
}

The behavior is identical to the for loop version. resources.forEach(function(resource) { ... }) passes each item in the array to the callback function as the resource parameter.

Section 28.8

Loading States

When a page makes a network request, there is a period of time between when the request is sent and when the data arrives. During this time, the container is empty. If a user loads the page on a slow connection, they might see a blank area for several seconds with no explanation. A good user experience shows a loading state during this period — some kind of message or visual indicator that data is on its way.

Implementing a Loading Message

Set a loading message in the container before the fetch() call begins.

const container = document.querySelector("#resources-list");

// Show loading message before the fetch begins
container.innerHTML = "<p class='loading-message'>Loading resources...</p>";

fetch("data/resources.json")
  .then(function(response) {
    return response.json();
  })
  .then(function(data) {
    renderResources(data);   // This replaces the loading message
  })
  .catch(function(error) {
    container.innerHTML = "<p class='error-message'>Could not load resources. Please try again later.</p>";
  });

The loading message is inserted before the fetch starts. When renderResources(data) runs, it replaces container.innerHTML, clearing the loading message. If an error occurs, the catch block replaces the loading message with an error message instead.

Accessible Loading Messages

Add a role="status" attribute to the loading message so that screen readers announce it:

<p class="loading-message" role="status">Loading resources...</p>
Why This Helps

You can add this attribute in JavaScript when building the loading message string. This helps users who rely on screen readers know that something is happening on the page, even though they cannot see it. Some websites show animated spinners or skeleton screens during loading — these are CSS-based techniques. For this textbook, a simple text message is appropriate; more advanced loading UI patterns belong in later front-end development courses.

Section 28.9

Error States

Network requests can fail for many reasons: the server is unreachable, the file does not exist, the JSON is malformed, the user's connection is offline. A well-built page handles these failures gracefully instead of silently showing a blank container.

What Causes a Fetch Error

A .catch() block runs when the request itself fails — for example, a network error or a DNS failure. It does not automatically run for HTTP error responses like 404 (Not Found) or 500 (Server Error). When a server returns a 404, fetch() still considers the request a technical success and does not reject the Promise.

To handle HTTP errors properly, check the response.ok property, which is true for status codes in the 200–299 range and false for anything else.

fetch("data/resources.json")
  .then(function(response) {
    if (!response.ok) {
      throw new Error("Server returned an error: " + response.status);
    }
    return response.json();
  })
  .then(function(data) {
    renderResources(data);
  })
  .catch(function(error) {
    showError(error.message);
  });

if (!response.ok) checks whether the HTTP status code indicates success. If the server returns a 404 or 500, response.ok is false. throw new Error(...) forces the Promise chain into the rejection state, triggering the .catch() block with a descriptive message.

Showing a Helpful Error Message

function showError(message) {
  const container = document.querySelector("#resources-list");
  container.innerHTML = `
    <p class="error-message" role="alert">
      We were unable to load the resources at this time. Please try refreshing the page.
    </p>
  `;
  console.error("Fetch error:", message);
}

Note the role="alert" attribute. This tells screen readers to announce the message immediately when it appears, which is appropriate for error conditions. Log the technical error to the console for debugging purposes. Do not show raw technical error messages to users — they are rarely helpful and can sometimes expose information about your site's internal structure.

Three States of a Data-Driven Component

State 1
Loading
Data has been requested but not yet received. Show a loading message.
State 2
Success
Data arrived and was parsed correctly. Show the rendered content.
State 3
Error
Something went wrong. Show a helpful, accessible error message.

Planning for all three states is a professional habit. Pages that only handle the success case will show users blank or broken content whenever anything goes wrong.

Section 28.10

Responsible Use of External Data

So far, this chapter has used a local JSON file. In real projects, you will often fetch data from external APIs — services provided by other organizations that return data over the web. Before doing so, there are important practices and principles to understand.

What an API Is

An API (Application Programming Interface) is a structured way for one software system to communicate with another. A web API accepts HTTP requests and returns data, usually in JSON format. For example, a weather service might offer an API endpoint at a URL like https://api.example-weather.com/forecast?city=Accra. Your JavaScript sends a request to that URL, and the weather service responds with JSON containing temperature, conditions, and forecast data.

Working with real external APIs is a topic for later courses in JavaScript and full-stack development. This chapter focuses on the patterns for reading and rendering data, which are the same whether the data comes from a local file or a remote API.

ConsiderationWhat to Do
Attribution and terms of service External data sources are owned by people and organizations. Before using data from an external source, read its terms of service. Some data is freely available for any use; some requires attribution; some prohibits commercial use; some requires an API key. For public data you use on your website, attribute the source clearly — this is good practice even when not strictly required.
Sanitizing untrusted data When you insert fetched data into the page using innerHTML, you are trusting that the data does not contain malicious HTML or scripts. For data you control, this is not a concern. For data from third-party sources, treat the content carefully and consider using textContent instead of innerHTML for plain text.
Sensitive or personal data Some APIs return personal data — names, email addresses, phone numbers. Handle this data responsibly. Do not store it beyond what is necessary, and be especially careful about logging personal data to the console in production code.
Rate limits and fair use Many external APIs enforce rate limits — restrictions on how many requests you can make in a given time period. Sending too many requests can get your IP address blocked or your API key revoked. Later courses introduce strategies like caching to reduce unnecessary calls.

Sanitizing in Practice

// Safer for plain text content that should not contain HTML
const titleElement = document.createElement("h3");
titleElement.textContent = resource.title;  // Not innerHTML

Using textContent means any HTML-like characters in the string are treated as plain text, not as markup. For beginner projects using your own controlled data, innerHTML with template literals is fine. As you work with more sources, developing the habit of distinguishing between markup you build and data you receive is important.

Step 4 — Write resources.js

// Store all resources for filtering
let allResources = [];

// Select the container
const container = document.querySelector("#resources-list");

// Show loading state
container.innerHTML = "<p class='loading-message' role='status'>Loading resources...</p>";

// Fetch the data
fetch("data/resources.json")
  .then(function(response) {
    if (!response.ok) {
      throw new Error("Network response was not OK: " + response.status);
    }
    return response.json();
  })
  .then(function(data) {
    allResources = data;
    renderResources(allResources);
    setupFilters();
  })
  .catch(function(error) {
    container.innerHTML = `
      <p class="error-message" role="alert">
        Resources could not be loaded. Please try refreshing the page.
      </p>
    `;
    console.error("Fetch error:", error.message);
  });


// Build and insert cards for a given array of resources
function renderResources(resources) {
  if (resources.length === 0) {
    container.innerHTML = "<p class='no-results'>No resources found for this category.</p>";
    return;
  }

  let html = "";

  resources.forEach(function(resource) {
    html += `
      <div class="resource-card">
        <div class="card-header">
          <h2>${resource.title}</h2>
          <span class="category-badge">${resource.category}</span>
        </div>
        <p class="card-description">${resource.description}</p>
        <a class="card-link" href="${resource.url}" target="_blank" rel="noopener noreferrer">
          Visit Resource <span aria-hidden="true">→</span>
        </a>
      </div>
    `;
  });

  container.innerHTML = html;
}


// Set up filter buttons
function setupFilters() {
  const buttons = document.querySelectorAll(".filter-btn");

  buttons.forEach(function(button) {
    button.addEventListener("click", function() {
      // Update active button styling
      buttons.forEach(function(btn) {
        btn.classList.remove("active");
        btn.removeAttribute("aria-pressed");
      });
      button.classList.add("active");
      button.setAttribute("aria-pressed", "true");

      // Filter and render
      const category = button.getAttribute("data-category");
      if (category === "all") {
        renderResources(allResources);
      } else {
        const filtered = allResources.filter(function(resource) {
          return resource.category === category;
        });
        renderResources(filtered);
      }
    });
  });
}

Chapter 28 — Back Matter

Chapter Summary

  • Section 28.1Modern websites separate content from presentation. Content lives in structured data files; JavaScript reads that data and builds HTML dynamically. Data-driven thinking asks what content repeats, what each item needs, and how the page should behave while loading or on failure.
  • Section 28.2JSON (JavaScript Object Notation) is the standard text format for structured web data. Property names must be double-quoted, trailing commas are forbidden, and comments are not allowed. JSON is stricter than JavaScript object syntax on purpose, so it stays consistent across languages.
  • Section 28.3JSON supports objects (key-value pairs in curly braces) and arrays (ordered values in square brackets), which can nest. Arrays of objects are the most common structure for lists of similar content items, such as resources, products, or team members.
  • Section 28.4Local JSON files live in a project's data folder and are useful for learning without network complexity. fetch() uses HTTP, so even local files must be served by a local development server — a bare file path will not work.
  • Section 28.5fetch() is the browser's built-in function for network requests. It is asynchronous: it sends the request and returns a Promise immediately, without blocking the rest of the page, and the response is handled later through .then().
  • Section 28.6A Promise represents the eventual result of an asynchronous operation, in one of three states: pending, fulfilled, or rejected. .then() handles success, .catch() handles failure, and Promises can be chained. async/await is an alternative syntax for the same behavior.
  • Section 28.7Rendering fetched data means selecting a container, looping through the array with a for loop or forEach, building an HTML string with template literals, and assigning it to container.innerHTML.
  • Section 28.8A loading state shown before the fetch resolves prevents a confusing blank page during network delay. Marking the loading message with role="status" makes it accessible to screen reader users.
  • Section 28.9fetch() only rejects on network failure, not on HTTP error codes like 404 or 500 — checking response.ok and throwing an error handles those cases. A well-built component plans for three states: loading, success, and error, with role="alert" on error messages.
  • Section 28.10An API lets software systems exchange data, usually as JSON. Using external data responsibly means respecting attribution and terms of service, sanitizing untrusted content before inserting it into the page, protecting personal data, and respecting API rate limits.

Key Terms

API
Application Programming Interface. A structured way for software systems to communicate. Web APIs accept HTTP requests and return data, usually in JSON.
array of objects
A JSON or JavaScript array where each item is an object with properties. The most common structure for lists of similar content items.
asynchronous
Describes operations that do not block the rest of a program while waiting. Network requests in JavaScript are asynchronous.
fetch()
A built-in browser function that sends an HTTP request from JavaScript and returns a Promise.
JSON
JavaScript Object Notation. A text-based format for structured data using objects and arrays.
loading state
The visual and structural state of a page component while data is being fetched.
error state
The visual and structural state of a page component when a data fetch has failed.
Promise
A JavaScript object that represents the eventual result of an asynchronous operation. Settled through .then() (success) and .catch() (failure).
rate limit
A restriction imposed by an API provider on how many requests can be made within a given time period.
response.json()
A method on the Response object returned by fetch(). It parses the response body as JSON and returns a Promise that resolves to the parsed value.
response.ok
A Boolean property of the Response object. true if the HTTP status code is in the 200–299 range; false otherwise.
sanitizing
Treating user or external data as potentially unsafe, and handling it so it cannot be misinterpreted as executable code.
textContent
A DOM property that sets or reads the plain text content of an element, treating any HTML characters as literal text rather than markup.

Common Misconceptions

Misconception 1
"JSON and JavaScript objects are exactly the same thing."
JSON is a text-based data format that looks similar to a JavaScript object, but it has stricter rules: property names must be double-quoted, trailing commas are not allowed, comments are not allowed, and functions cannot appear as values. JSON text must be parsed — for example with response.json() or JSON.parse() — before it becomes an actual JavaScript array or object you can work with.
Misconception 2
"If a server returns an error page, like a 404, my .catch() block will run automatically."
fetch() only rejects its Promise — triggering .catch() — when the request itself fails to complete, such as a network outage. A 404 or 500 response is still a completed HTTP exchange, so fetch() treats it as fulfilled. You must check response.ok yourself and throw an error if it is false.
Misconception 3
"Because fetch() is just a function call, the data is available on the very next line of code."
fetch() is asynchronous. It returns a Promise immediately, but the actual response arrives later, whenever the network request completes. Code that needs the data must go inside a .then() callback (or after an await), not on the line directly following the fetch() call.
Misconception 4
"Sanitizing data only matters for professional developers working with sensitive systems — not for a small beginner project."
For your own JSON files that you write and control, the risk is minimal. But the habit of treating external or user-supplied content carefully — for example, using textContent for plain text instead of innerHTML — is worth building early. The moment a beginner project starts pulling in data from an outside source, that habit becomes a real protection rather than an abstract rule.
