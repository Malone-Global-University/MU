Chapter 19 — Front Matter

Chapter Introduction and Learning Objectives

Up to this point, the websites you have built respond to users the way a printed book responds to a reader: they display information, they follow links, they show images, but they do not change based on what the user does while reading. A form submits. A link navigates. But nothing happens in place, nothing updates without a full page reload, and nothing responds to a user action in the moment.

JavaScript changes this. JavaScript is the language that makes pages respond, update, and behave dynamically while a user is looking at them. When a user clicks a button and a message appears without the page reloading, that is JavaScript. When a navigation menu expands on a small screen, that is JavaScript. When a form shows an error message next to the field that needs attention, that is JavaScript. When a list of items filters as a user types, that is JavaScript.

This chapter introduces JavaScript as the behavior layer of the browser. It explains what JavaScript is, how it fits alongside HTML and CSS, how to connect a JavaScript file to an HTML page, how to use the browser console to understand what your code is doing, and how to write and read the most basic JavaScript statements. It introduces common beginner errors with practical debugging guidance, and it closes with a first small interactive page that brings all three languages together.

This is a beginning. The chapters that follow will teach variables, functions, events, the DOM, logic, loops, and forms. This chapter's job is to make JavaScript feel approachable, understandable in its purpose, and connected to the real work of building websites.

Opening Question

A button that changes a message when clicked. A menu that opens on a phone screen. A form that warns you before you submit it with an empty field. None of these require the page to reload. What language makes a page change while a person is still looking at it? By the end of this chapter, you will understand the answer well enough to write — and link, and run, and debug — a small piece of it yourself.

Section 19.1

What JavaScript Is

JavaScript is a programming language that runs in the browser. Unlike HTML, which describes the structure of a document, and CSS, which describes its visual presentation, JavaScript describes behavior: what should happen, when it should happen, and what should change.

JavaScript was created in 1995 by Brendan Eich while working at Netscape Communications. It was designed to add interactivity to web pages that were, at the time, entirely static documents. Despite its early origins and occasional awkward history, JavaScript has become one of the most widely used programming languages in the world. It runs in every major web browser without any installation. It is the only programming language that runs natively in the browser. No other language can do what JavaScript does in the browser without being compiled into JavaScript or running on a server.

Naming Confusion

JavaScript is not Java. The two languages share the first four letters of their names and superficially similar syntax in some areas, but they are distinct languages with different histories, different uses, and different design philosophies. The similar name is a historical marketing decision from the 1990s, not a technical relationship. You will not need Java for this course.

JavaScript is a scripting language. In traditional programming, a program is compiled into machine code before it runs. In scripting, code is read and executed more directly. Browsers contain a JavaScript engine — a program that reads JavaScript code and runs it. The V8 engine in Chrome and Edge, the SpiderMonkey engine in Firefox, and the JavaScriptCore engine in Safari all do this work invisibly whenever a page with JavaScript is loaded.

JavaScript is also a dynamic language. It is flexible about types, forgiving about some syntax details, and can be written in many styles. This flexibility is both a strength and a source of confusion for beginners. This textbook takes a careful, explicit approach: it will explain what each piece of code means, avoid shortcuts that obscure logic, and build habits that make code readable and maintainable.

In the context of web development, JavaScript's primary job is to interact with the browser and the page. It can read what is on the page, change what is on the page, respond to what users do, and communicate with servers. These abilities make it the foundation of every interactive website on the web.

Section 19.2

JavaScript as the Behavior Layer

You have already learned that web pages are built from three layers, each with a distinct job.

Layer 1
HTML — Structure and Meaning
Defines what elements exist on the page: headings, paragraphs, lists, links, images, forms. Gives content semantic identity. Answers: what is this, and how is it structured?
Layer 2
CSS — Presentation and Layout
Controls colors, fonts, sizes, spacing, borders, positioning, responsive behavior, and visual hierarchy. Answers: what does this look like, and how is it arranged?
Layer 3
JavaScript — Behavior and Interaction
Controls what happens in response to user actions, how the page changes over time, how data is validated, how content is loaded. Answers: what does this do, and how does it respond?

These three layers are separate concerns. This separation is intentional and valuable. A page's structure can be understood without knowing its styles. A page's styles can be updated without touching its structure. A page's behavior can be changed without rewriting the underlying HTML. Keeping these layers clean and independent makes websites easier to read, maintain, and debug.

Worked Example
A mobile navigation menu — each layer doing its own job
  • HTMLDefines the list of navigation links and a menu button inside a <nav> element.
  • CSSHides the list by default on small screens and styles the menu button. A class such as .is-open defines what the "open" state looks like.
  • JavaScriptListens for a click on the menu button and toggles the is-open class on the navigation list. It does not write any color values or define any HTML — it only manages the transition between states.

Notice that the JavaScript does not write color values. It does not define the HTML structure. It manages behavior: it listens for the click, and it adds or removes a class. The visual change happens because CSS responds to that class. This is the right pattern.

JavaScript as Enhancement

This separation also means that a page should still be usable even if JavaScript is unavailable or slow to load. Good web development treats JavaScript as an enhancement to an already functional HTML and CSS foundation, not a replacement for it.

Section 19.3

Linking a JavaScript File

Just as CSS is linked from an external file, JavaScript is typically stored in an external .js file and linked to the HTML page.

The Script Element

JavaScript is linked to an HTML page using the <script> element. The <script> element takes a src attribute whose value is the path to the JavaScript file, just as <link> takes an href to a CSS file.

<script src="js/main.js"></script>

Where to Place the Script Element

The placement of the <script> element in the HTML document matters. JavaScript code can interact with the page's HTML elements, but it can only interact with elements that have already been loaded by the browser. If JavaScript runs before the page has finished loading its HTML, the code will try to find elements that do not yet exist.

The simplest and most reliable solution for beginners is to place the <script> element at the end of the <body> element, just before the closing </body> tag.

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>My Page</title>
    <link rel="stylesheet" href="css/main.css">
  </head>
  <body>

    <h1>Welcome</h1>
    <p>This is my page.</p>

    <!-- JavaScript goes at the end of the body -->
    <script src="js/main.js"></script>

  </body>
</html>

By placing the script at the end of the body, all of the HTML elements have been created and loaded before the JavaScript runs. This avoids the common beginner problem of JavaScript trying to select elements that do not exist yet.

A Note on a More Modern Approach

There is a more modern approach using the defer attribute on a <script> element in the <head>, which tells the browser to load the script in parallel but only execute it after the HTML is fully parsed. You will encounter defer in later courses. For now, placing <script> at the end of <body> is clear, reliable, and correct.

The JavaScript File Itself

The JavaScript file is a plain text file saved with a .js extension. It contains JavaScript code, not HTML. There are no tags in a .js file. JavaScript has its own syntax.

A minimal JavaScript file might look like this:

// This is a comment in JavaScript
console.log("The script is running.");

This file does very little: it runs a single line of code that sends a message to the browser console. The console is a place where JavaScript can display information for developers. Users do not see it. You will learn to use it in the next section.

Project Structure with JavaScript

A clean project folder with HTML, CSS, and JavaScript looks like this:

my-project/ index.html css/ main.css js/ main.js images/ hero.jpg
Convention, Not Requirement

The js/ folder is conventional but not required. For small projects, some developers place main.js at the project root. For this textbook, keeping JavaScript in a js/ folder is the recommended practice. It mirrors the way CSS is kept in a css/ folder and reinforces the idea of organized, layered projects.

Section 19.4

The Browser Console

The browser console is one of the most important tools a JavaScript developer uses. It is a panel inside the browser's developer tools where:

  • JavaScript can print messages using console.log()
  • Error messages appear when JavaScript code fails
  • Warnings appear when code behaves unexpectedly
  • You can type and run short JavaScript expressions interactively

You opened the developer tools in Chapter 4. The console is one of the tabs inside that panel.

Opening the Console

To open the developer tools and find the console:

  • In Chrome, Edge, or Firefox: press F12 or Ctrl+Shift+I (Windows/Linux) or Cmd+Option+I (Mac), then click the Console tab.
  • You can also right-click anywhere on the page and choose Inspect, then click Console.

Using console.log()

console.log() is a JavaScript statement that sends a message to the console. It is the most common tool for checking whether your JavaScript is running and what values your code is working with.

console.log("Hello from JavaScript.");
console.log(42);
console.log("The page has loaded.");

Each call to console.log() prints one line to the console. The message in parentheses is what gets printed.

A Habit Worth Building Early

This may seem like a small thing, but console.log() is genuinely one of the most useful techniques in web development debugging. Professional developers use it constantly to understand what their code is doing, what values variables hold, and at what point something is going wrong.

What Errors Look Like in the Console

When JavaScript code contains a mistake — a misspelling, a missing parenthesis, a reference to something that does not exist — the browser will not just ignore the error. It will log a red error message to the console. That message includes:

  • A description of the error
  • The file where the error occurred
  • The line number where the error occurred
Uncaught ReferenceError: consle is not defined
    at main.js:3
Reading the Message

This message tells you exactly where to look. In this case, consle is a typo for console. The error is on line 3 of main.js. Learning to read console error messages is a critical skill. They are not failures — they are the browser's attempt to tell you what went wrong and where. Beginning developers sometimes feel discouraged by red error messages. Experienced developers feel relieved: a clear error message is far easier to fix than mysterious behavior that produces no message at all.

The Interactive Console

You can type JavaScript directly into the console and press Enter to run it immediately. This is useful for quick experiments. Try typing the following in the console on any web page:

2 + 2

The console will show 4. Try:

document.title

The console will show the title of the current page. The console is a live JavaScript environment connected to the page you are viewing. You will use this throughout the course to test small pieces of code before putting them in files.

Section 19.5

Statements and Comments

JavaScript code is made of statements. A statement is one complete instruction. When the JavaScript engine runs your code, it reads and executes statements in order, one at a time, from top to bottom (with exceptions you will learn about later, such as functions and events).

Statements

Each line in the following code is a statement:

console.log("Line one.");
console.log("Line two.");
console.log("Line three.");

JavaScript statements end with a semicolon (;). Semicolons are technically optional in many cases because JavaScript has a feature called automatic semicolon insertion that adds them in certain situations. However, writing your semicolons explicitly is a better habit. It prevents a small category of subtle bugs and makes your intentions clear. All examples in this textbook use explicit semicolons.

Comments

Comments are notes written in the source code that the JavaScript engine ignores. They exist for human readers: you, your teammates, your future self revisiting the code six months later.

There are two types of comments in JavaScript. A single-line comment begins with // and extends to the end of that line:

// This is a single-line comment.
console.log("This runs."); // This comment is after the statement.

A multi-line comment begins with /* and ends with */. Everything between those markers is a comment, even if it spans multiple lines:

/*
  This is a multi-line comment.
  It can span as many lines as needed.
  It does not run as code.
*/
console.log("This still runs.");

Good comments explain why something is done, or note something that might not be obvious to a reader. They do not simply restate what the code says.

// Redundant and unhelpful:
// Log a message
console.log("Hello.");

// Useful:
// Confirm the script loaded before attempting DOM operations.
console.log("Script loaded successfully.");
A Skill That Grows

Writing clear comments is part of writing readable, maintainable code. You will develop better instincts for when and how to comment as your programs grow larger.

Section 19.6

Execution Order

JavaScript executes statements in order, from top to bottom, unless something changes that order. Understanding execution order is fundamental to understanding how your programs behave.

Consider this file:

console.log("First");
console.log("Second");
console.log("Third");

The console output will be:

First
Second
Third

That seems obvious. But consider:

console.log("First");
// console.log("Second");
console.log("Third");

The second statement is commented out, so it does not run. The console output is:

First
Third

And consider:

console.log("A");

function greet() {
  console.log("Hello from greet.");
}

console.log("B");

The output here is:

A
B

The greet() function is defined but never called. Defining a function does not run the code inside it. Calling a function does. You will learn about functions in Chapter 20, but the principle matters now: code only runs when the JavaScript engine reaches it in the execution sequence, or when it is triggered by something like a function call or an event.

This also explains why placing the <script> element at the end of <body> matters. The browser loads and executes JavaScript as soon as it encounters a <script> element. If that element appears in the <head>, the script runs before any of the body content has been loaded — which means the HTML elements do not yet exist in memory, and any JavaScript that tries to find them will fail.

Rule to Remember

Unless you understand defer or event-based initialization, place your <script> element at the end of <body> and write your code in the order you want it to execute.

Section 19.7

alert() and Simple Output at an Introductory Level

Before diving into the full power of JavaScript's interaction with the page, it is worth knowing a few very simple ways that JavaScript can produce visible output. These are useful for early learning and quick experiments, though you will rarely use them in production websites.

alert()

alert() displays a browser dialog box with a message and an OK button. It pauses the page until the user dismisses it.

alert("Welcome to this page.");

When the browser runs this line, a small dialog box appears with the text "Welcome to this page." and an OK button. The page waits until the user clicks OK before continuing.

alert() is one of the oldest JavaScript features. It is rarely used in modern websites because it interrupts the user experience and looks unprofessional. However, it is useful for early experiments because it gives immediate visible proof that JavaScript is running.

confirm()

confirm() displays a dialog box with a message and two buttons: OK and Cancel. It returns a value depending on which button the user clicks. You will use return values more after Chapter 20.

confirm("Do you want to continue?");

prompt()

prompt() displays a dialog box with a text field. The user can type a value and click OK. Again, you will learn how to use the value it returns after Chapter 20.

prompt("What is your name?");
console.log() Is the Right Habit

For debugging and development, console.log() is far more useful than alert(). It does not interrupt the page. It shows values in context. It can display objects and arrays. Get in the habit of using console.log() for development feedback, and reserve alert() for only the earliest experiments.

Section 19.8

JavaScript and Page Loading

There is an important relationship between JavaScript and the process of loading an HTML page. Understanding this relationship helps you avoid a very common category of beginner errors.

When the Page Loads

When a browser loads a web page, it does the following work in roughly this order:

  1. It downloads the HTML file.
  2. It parses the HTML from top to bottom, building the structure of the page in memory.
  3. As it parses, it encounters <link> elements and downloads CSS files.
  4. It applies the CSS to the elements it has built.
  5. At some point, it encounters <script> elements and downloads and executes the JavaScript files.
  6. When the HTML is fully parsed and all subresources have loaded, a load event fires.
The Critical Point

The critical point for beginners is step 5. If the <script> element appears at the top of the page, in the <head>, the JavaScript runs at that moment — before the <body> content has been parsed. The JavaScript cannot interact with elements that do not yet exist in memory.

The Safest Approach for Beginners

Placing <script> at the end of <body> ensures that all HTML has been parsed before the script runs. This is the approach used throughout this textbook.

<body>
  <h1>My Page</h1>
  <p>Content here.</p>

  <script src="js/main.js"></script>
</body>

When the browser reaches the <script> tag at the bottom of the body, all of the HTML above it has been loaded. Any JavaScript that looks for page elements will find them.

The DOMContentLoaded Event (Preview)

There is another pattern you may encounter in examples on the web:

document.addEventListener("DOMContentLoaded", function() {
  // Code here runs after the page has loaded.
});

This approach wraps code in a function that only runs after the page's HTML structure has been fully loaded. It allows a <script> in the <head> to safely interact with page elements. You will learn about events and event listeners in Chapter 21. For now, using <script> at the end of <body> is simpler and achieves the same result.

Section 19.9

HTML, CSS, and JavaScript Working Together

The three languages of the browser are most powerful when they work together as intended: HTML for structure, CSS for presentation, JavaScript for behavior. Let's look at a complete small example to see this collaboration in practice.

The Task

Build a page with a heading, a short message, and a button. When the user clicks the button, the message changes.

The HTML

<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Button Demo</title>
    <link rel="stylesheet" href="css/main.css">
  </head>
  <body>

    <main>
      <h1>Button Demo</h1>
      <p id="status-message">The button has not been clicked yet.</p>
      <button id="change-button">Click me</button>
    </main>

    <script src="js/main.js"></script>

  </body>
</html>

Notice two attributes in this HTML: id="status-message" on the paragraph and id="change-button" on the button. IDs give JavaScript a way to find specific elements on the page.

The CSS

/* css/main.css */

body {
  font-family: Georgia, serif;
  max-width: 600px;
  margin: 2rem auto;
  padding: 0 1rem;
  color: #1a1a1a;
}

h1 {
  font-size: 2rem;
  margin-bottom: 1rem;
}

p {
  font-size: 1.125rem;
  margin-bottom: 1.5rem;
}

button {
  background-color: #1a5276;
  color: white;
  border: none;
  padding: 0.75rem 1.5rem;
  font-size: 1rem;
  border-radius: 4px;
  cursor: pointer;
}

button:hover {
  background-color: #154360;
}

The JavaScript

// js/main.js

// Step 1: Find the elements we need to work with.
var button = document.getElementById("change-button");
var message = document.getElementById("status-message");

// Step 2: Tell the button what to do when clicked.
button.addEventListener("click", function() {
  message.textContent = "The button has been clicked!";
});

This JavaScript does three things:

  1. It finds the button element by its ID.
  2. It finds the paragraph element by its ID.
  3. It attaches a click listener to the button. When the button is clicked, the paragraph's text changes.
Don't Worry About the Details Yet

Do not worry about fully understanding this JavaScript yet. The details of getElementById, addEventListener, and textContent will be explained carefully in Chapters 21 and 22. What matters now is seeing the pattern: HTML defines the elements and their IDs, CSS styles how those elements look, and JavaScript uses the IDs to find the elements and changes them when the user acts. Each language does its part. The layers remain clean.

What the Browser Experience Looks Like

Before clicking: The page shows a heading, a paragraph that reads "The button has not been clicked yet," and a blue button.

After clicking: The paragraph now reads "The button has been clicked!" The button remains where it is. The page never reloaded. The change happened instantly, in place.

This is the fundamental pattern of JavaScript interaction on the web.

Section 19.10

Common Beginner JavaScript Errors

JavaScript code that contains mistakes does not always stop working entirely — sometimes it simply behaves unexpectedly. But many errors cause the browser to stop executing a script and log an error message to the console. Learning to recognize common errors will save you significant time.

Error Type 1: The Script Is Not Linked Correctly
Symptom & Cause

JavaScript appears to do nothing. No output in the console, no interaction. The src path in the <script> element is wrong, and the browser cannot find the file.

Wrong (file is in a js/ folder) 
<script src="main.js"></script>
Right 
<script src="js/main.js"></script>

How to check: Open the Network tab in developer tools and reload the page. If the JavaScript file shows a red 404 error, the path is wrong.

Error Type 2: Typos in Method Names
Symptom & Cause

Red error in the console: Uncaught TypeError: ... is not a function or Uncaught ReferenceError: ... is not defined. A word is misspelled. JavaScript is case-sensitive.

Wrong 
Console.log("Hello.");         // Capital C is wrong
consol.log("Hello.");          // Missing 'e'
console.Log("Hello.");         // Capital L is wrong
Right 
console.log("Hello.");

How to check: Look at the error message and the line number it points to. Read that line carefully, character by character.

Error Type 3: Missing or Mismatched Quotes
Symptom & Cause

Red error in the console: Uncaught SyntaxError: Unexpected token. A string literal is not properly closed.

Wrong 
console.log("Hello.);    // Missing closing quote
console.log('Hello.");   // Mismatched quotes
Right 
console.log("Hello.");
console.log('Hello.');

How to check: Strings must open and close with the same type of quote mark. Most code editors color string literals, which makes this easier to spot visually.

Error Type 4: Missing Parentheses or Curly Braces
Symptom & Cause

Uncaught SyntaxError with a message about an unexpected token, often on a line that looks fine. An opening parenthesis, closing parenthesis, opening brace, or closing brace is missing. JavaScript often does not notice this until a later line, so the error may appear several lines after the actual mistake.

Wrong 
console.log("Hello.";    // Missing closing parenthesis
Right 
console.log("Hello.");

How to check: Many code editors highlight matching pairs of parentheses and braces. Using a good editor helps catch these immediately.

Error Type 5: Script Runs Before the Element Exists
Symptom & Cause

Uncaught TypeError: Cannot read properties of null — the code finds null instead of a real element. The JavaScript tries to find an HTML element by ID before the HTML has loaded.

Wrong (in <head>, no defer) 
<head>
  <script src="js/main.js"></script>  <!-- Runs before body content loads -->
</head>
<body>
  <button id="my-button">Click me</button>
</body>
Right 
<body>
  <button id="my-button">Click me</button>
  <script src="js/main.js"></script>  <!-- Runs after body content loads -->
</body>

How to check: Always place <script> at the end of <body> until you are comfortable with the defer attribute or DOMContentLoaded events.

Error Type 6: Using an ID That Does Not Match
Symptom & Cause

Uncaught TypeError: Cannot read properties of null. The ID in the JavaScript does not exactly match the ID in the HTML.

HTML 
<button id="submitButton">Submit</button>
Wrong 
var btn = document.getElementById("submit-button");  // Wrong: hyphen vs. camelCase
Right 
var btn = document.getElementById("submitButton");   // Matches the HTML exactly

How to check: IDs are case-sensitive and must match exactly. Copy-paste the ID from the HTML rather than retyping it.

Error Type 7: The Console Has Errors From Previous Sessions
Symptom & Cause

Errors appear in the console that do not seem related to your code. The console accumulates messages. Old messages from previous page loads or other experiments may still be visible.

How to fix: Click the clear button in the console (usually a circle-with-slash icon), or press Ctrl+L (Windows/Linux) or Cmd+K (Mac) to clear it. Then reload the page.

Key Terms

JavaScript
A programming language that runs in the web browser, responsible for adding behavior and interaction to web pages.
behavior layer
The role of JavaScript in web development: managing what happens on a page in response to user actions and over time, distinct from HTML structure and CSS presentation.
script element
The HTML element used to link an external JavaScript file to a web page, written as <script src="path/to/file.js"></script>.
JavaScript engine
Software built into each browser that reads and executes JavaScript code. Examples include V8 in Chrome and Edge, SpiderMonkey in Firefox, and JavaScriptCore in Safari.
console
A panel in browser developer tools where JavaScript output, errors, and warnings are displayed. Accessible with console.log().
console.log()
A JavaScript statement that sends a message or value to the browser console. Used for debugging and development feedback.
statement
One complete instruction in a JavaScript program. Statements are separated by semicolons and executed in order from top to bottom.
comment
Text in JavaScript code that the engine ignores, written for human readers. Single-line comments begin with //. Multi-line comments are wrapped in /* */.
semicolon
A punctuation mark (;) used to end JavaScript statements. Technically optional in many cases due to automatic semicolon insertion, but explicitly written semicolons are the recommended practice.
execution order
The sequence in which JavaScript statements run: top to bottom, unless altered by functions, events, or control flow structures.
alert()
A JavaScript function that displays a browser dialog box with a message and an OK button. Useful for early experiments but not appropriate for production websites.
DOMContentLoaded
A browser event that fires when the HTML document has been fully parsed, before stylesheets and images finish loading. Can be used to safely run JavaScript from a <script> in the <head>.
ReferenceError
A JavaScript error that occurs when code references a variable or function name that has not been defined, often caused by a typo.
TypeError
A JavaScript error that occurs when code tries to perform an operation on the wrong type of value — for example, trying to call a method on null because a selected element was not found.
SyntaxError
A JavaScript error that occurs when the code itself is not written correctly — for example, mismatched quotes, missing parentheses, or other structural mistakes.
null
A special value in JavaScript representing the intentional absence of any object or value. When JavaScript looks for an element by ID and cannot find it, it returns null.

Accessibility Check

JavaScript does not remove the need for accessible HTML. It adds to it. Before adding JavaScript to any page, the HTML and CSS should already be accessible. Then JavaScript can enhance the experience without breaking it. Review your Mini-Project page for the following:

Review Checklist
  • Does the page have a meaningful <title>?
  • Does the heading structure use <h1> correctly?
  • Does the button use an actual <button> element, not a <div> or <span> styled to look like a button?
  • Does the button have a clear, descriptive visible label?
  • Is the text readable with sufficient contrast against the background?
  • Does the page work if JavaScript is disabled? (The button may do nothing, but the page content should still be visible and readable.)

This last point matters. JavaScript should enhance a working page, not be required for the page to have any content at all. A page that shows a blank screen when JavaScript is disabled has been built on an unstable foundation.
