Chapter 7 — Front Matter

Chapter Introduction and Learning Objectives

Imagine a library where every book exists, but none of them reference any other. No footnotes. No "see also." No card catalog connecting related works. Every reader would need to already know exactly which book held the answer, because nothing in the building pointed anywhere else. Finding information would depend entirely on luck or memory.

The web does not work that way, and the reason is a single idea so simple it is easy to take for granted: any document can point directly to any other document, anywhere, and a reader can follow that pointer with one click. This idea is called the hyperlink, and it is the feature that gives the World Wide Web its name. In Chapters 5 and 6 you learned how to structure a single page with HTML. This chapter teaches you how to connect pages — to each other, to other websites, and to specific points within a page — and how to organize those connections into a website that makes sense to the people who use it.

Opening Question

A small website has five pages: Home, About, Lessons, Resources, and Contact. A visitor lands on the Resources page from a search engine. How do they get to the other four pages without typing a new address? How does the site let them know which page they are currently viewing? How would a much larger site — one with hundreds of pages organized into categories and subcategories — keep visitors from getting lost? Every one of these questions is answered with the same underlying tool: the link. By the end of this chapter, you will know how to write links correctly, open them responsibly, and arrange them into a navigation system that holds an entire website together.

Section 7.1

Links as the Foundation of the Web

Every HTML page you built in Chapters 5 and 6 was, on its own, an island. It had structure, headings, and meaning, but nothing connected it to anything else. The feature that turns a collection of separate documents into a web is the hyperlink — a piece of text, an image, or another element that, when activated, takes the user to a different location: another page, another part of the same page, a file, or an external resource.

The idea of linked documents predates the web itself. Researchers had imagined systems of cross-referenced information for decades before any of it was built. What made the World Wide Web different was that it combined this idea with a simple, universal way to address any document on any connected computer, and a simple piece of HTML markup — the anchor element, which you will meet in the next section — that any author could use to point to it.

Links matter for two audiences at once. For people, links are how they move through information: from a search result to an article, from a homepage to a product page, from one paragraph to a related idea. For machines, links are how search engines discover new pages in the first place — a search engine finds your site largely by following links to it, and finds the rest of your site by following the links inside it. A page with no incoming links is difficult for anyone, human or machine, to find at all.

A Key Insight

The web is named after its structure, not its appearance. If you drew every web page as a dot and every link as a line connecting two dots, the result would look like a tangled, sprawling web — millions of documents, each pointing to others, with no single center and no required order. This chapter is about learning to add your own pages to that structure correctly, so that they are easy to find, easy to move through, and easy to maintain.

Study Note

It can help to think of a website not as a stack of pages but as a small graph: each page is a node, and each link is an edge connecting two nodes. As you work through this chapter, try sketching the graph of any website you visit. Notice which pages have many incoming links (usually the homepage) and which have only one or two.

Section 7.2

The Anchor Element

HTML creates a link with the anchor element, written as <a>. The anchor element needs one essential attribute to function as a link: href, short for "hypertext reference." The value of href is the destination — the path or address the browser should go to when the link is activated.

Whatever content sits between the opening <a> tag and the closing </a> tag becomes the clickable link. Most often this is text, but it can also be an image or a small group of elements. Whatever it is, that content is what the user sees and clicks.

Example — A Basic Link
<p>
  You can learn more about semantic HTML on the
  <a href="about.html">About page</a>.
</p>

An anchor element without an href attribute is not a functioning link — browsers will display it without the usual link styling and it will not be reachable by keyboard the way a real link is. If an element should behave like a link, it needs href. If it should trigger a script action with no destination, it is usually better written as a <button>, a distinction Chapter 9 returns to.

AttributePurposeExample
hrefThe destination of the link. Required for the element to act as a link.href="lessons.html"
targetWhere to open the destination — the same tab or a new one.target="_blank"
relDescribes the relationship between the current page and the linked page; also used for security and SEO signals.rel="noopener noreferrer"
downloadSuggests that the browser download the linked file rather than navigate to it.download
titleExtra descriptive text, usually shown as a tooltip on hover.title="Opens in a new tab"

By default, browsers style links in a recognizable way — typically underlined and colored, with a different color once a link has been visited. This default styling exists so that links are visually distinguishable from ordinary text without any extra effort from the page author. You will learn to style links deliberately, including their hover and visited states, when Chapter 12 introduces pseudo-classes. For now, leave the default styling alone and focus on writing correct, working href values.

Section 7.3

Relative Links Between Pages

Chapter 3 introduced relative paths for linking stylesheets, scripts, and images. The same idea applies directly to links between pages. A relative link describes a destination relative to the location of the current file — not as a complete address, but as directions from "here."

Three patterns cover the large majority of relative links you will write in a typical project.

Example 7.3.1
Three relative path patterns
  • Same folderIf about.html sits in the same folder as the current page, link to it by filename alone: <a href="about.html">About</a>.
  • SubfolderIf the destination is inside a folder beneath the current one, include the folder name followed by a forward slash: <a href="lessons/intro.html">Intro Lesson</a>.
  • Parent folderIf the destination is one level up from the current file, use two dots and a slash: <a href="../index.html">Home</a>. Each additional ../ moves up one more level.

Relative links have an important advantage: because they describe a path rather than a complete address, an entire project folder can be copied, renamed, or moved to a new server, and every internal link will continue to work — provided the files keep the same positions relative to one another. This is why Chapter 3 emphasized clean, consistent project structure: relative links depend on it directly.

Common Mistake

Relative links describe a path from the current file, not from the project's root folder. A link that works correctly from index.html may need a different path when the same link is copied into a page stored inside a subfolder. When you move a file, check every relative link in it — both links going out of the file and any links elsewhere that point into it. Chapter 4 will show you how to use developer tools to find broken links quickly.

Section 7.4

Absolute Links to Other Sites

An absolute link uses a complete URL: a protocol such as https://, a domain name, and usually a path. Chapter 2 explained URLs in depth as part of how the web works; here, the focus is simply on when and how to use a full URL inside an anchor element.

Relative links only make sense for files that live inside your own project, because they describe a path through your own folder structure. To link to a page that lives anywhere else — a different website entirely — you need the complete address.

Example — An Absolute Link to an External Site
<p>
  Read the official specification on the
  <a href="https://www.w3.org/">W3C website</a>.
</p>
Rule of Thumb

If the destination is a file inside your own project, use a relative link. If the destination is on a different website, use an absolute link with the full address, including https://. Mixing the two up is rare once you ask one question before writing any href: "Does this file live inside my project folder, or somewhere else?"

One more situation calls for an absolute link even within your own site: if you ever reference your own pages from outside contexts — an email, a social post, another company's website — those references need the full address, because there is no "current file" for a relative path to be relative to.

Section 7.5

Opening Links Responsibly

A link does more than point somewhere; it makes a small decision on behalf of the user about how they will get there. Two decisions matter most: whether the link opens in the same tab or a new one, and whether the link's text tells the user what to expect.

New Tabs and the target Attribute

By default, a link replaces the current page. Adding target="_blank" opens the destination in a new tab instead, leaving the original page open. This is sometimes useful — for example, linking to an external reference that a reader might want to consult without losing their place — but it should be used deliberately, not by habit. Opening too many links in new tabs leaves users with a cluttered row of tabs and can be disorienting, especially for people using screen readers, who are not always told that a new tab has opened.

When you do use target="_blank" for a link to another website, pair it with rel="noopener noreferrer". Without it, the newly opened page can, in some browsers, gain a limited ability to access and redirect the original tab — a small but real security and privacy concern. noopener blocks that access; noreferrer additionally withholds referrer information from the destination site.

Example — Opening an External Link Safely
<a href="https://www.example.org/" target="_blank" rel="noopener noreferrer">
  Visit Example.org
</a>

Email and Phone Links

Two special href patterns let a link launch a different application instead of navigating to another page. A mailto: link opens the user's email program with a new message addressed to the given address. A tel: link, most useful on phones, starts a call.

Example — Email and Phone Links
<a href="mailto:hello@example.org">hello@example.org</a>
<a href="tel:+15551234567">(555) 123-4567</a>

Writing Meaningful Link Text

The words inside a link should make sense even if a reader sees them out of context — which is exactly how many screen reader users browse, by pulling up a list of every link on a page in isolation from the surrounding sentences. Link text such as "click here" or "read more" tells that reader nothing about where the link goes.

Avoid This

For our pricing, <a href="pricing.html">click here</a>. — out of context, "click here" is meaningless. Prefer: View our <a href="pricing.html">pricing details</a>. — the link text itself describes the destination.

Section 7.6

Internal Page Links and IDs

Not every link needs to leave the current page. A link can also jump to a specific location within the same page — or within a different page — using a fragment identifier: an href value that begins with a hash symbol, #, followed by the value of an id attribute somewhere on the page.

Every id attribute value must be unique within a page. Once an element has an id, a link anywhere can jump straight to it.

Example — Jumping to a Section on the Same Page
<nav>
  <a href="#ingredients">Ingredients</a>
  <a href="#instructions">Instructions</a>
</nav>

<section id="ingredients">
  <h2>Ingredients</h2>
  <!-- content -->
</section>

<section id="instructions">
  <h2>Instructions</h2>
  <!-- content -->
</section>

A fragment identifier can also follow a filename to jump to a location on a different page: href="resources.html#video-guides" opens resources.html and scrolls directly to the element with id="video-guides".

An Accessibility Pattern: Skip Links

Many sites include a "skip to main content" link as the very first focusable element on the page, hidden visually but reachable by keyboard, linking to #main-content on the same page. This lets a keyboard or screen-reader user bypass a long repeated navigation menu and jump straight to the unique content of the page. Chapter 29 returns to this pattern in more depth as part of accessibility practices.

Section 7.7

Navigation Menus

Chapter 6 introduced <nav> as one of the landmark elements that gives a page meaningful structure. A navigation menu is simply a group of links, wrapped in <nav>, that lets a visitor move between the major pages or sections of a site.

The standard pattern wraps the links in a list — almost always an unordered list, since the order of a navigation menu rarely carries meaning on its own. This is not just convention: a list communicates to assistive technology that the links form a related group, and many screen readers announce how many items the list contains, which helps a listener understand the scope of the menu before navigating through it.

Example — A Site Navigation Menu
<nav aria-label="Main">
  <ul>
    <li><a href="index.html">Home</a></li>
    <li><a href="about.html">About</a></li>
    <li><a href="lessons.html">Lessons</a></li>
    <li><a href="resources.html">Resources</a></li>
    <li><a href="contact.html" aria-current="page">Contact</a></li>
  </ul>
</nav>

Two details in that example are worth noticing. First, aria-label="Main" distinguishes this navigation landmark from any others on the page — useful once a site also has, say, a footer navigation menu. Second, aria-current="page" marks which link represents the page the visitor is currently on, which both sighted users and assistive technology can use to orient themselves. Chapter 26 builds on this idea when it covers active navigation states across a full multi-page site.

Design Note

A navigation menu's markup — the list of links — stays the same regardless of whether it is displayed as a vertical list, a horizontal row, or a collapsible mobile menu. That visual arrangement is entirely the job of CSS, which you will learn beginning in Part III. For now, focus on writing a correct, complete, and logically ordered set of links.

Section 7.8

Breadcrumbs at an Introductory Level

A breadcrumb trail is a small row of links, usually placed near the top of a page, that shows the path from the site's home page down to the page currently being viewed. The name comes from the fairy tale in which a trail of breadcrumbs marks the way back. On a website, a breadcrumb trail marks the way back up the site's hierarchy.

Example — A Breadcrumb Trail
<nav aria-label="Breadcrumb">
  <ol>
    <li><a href="index.html">Home</a></li>
    <li><a href="lessons.html">Lessons</a></li>
    <li>Introduction to Selectors</li>
  </ol>
</nav>

Notice that the trail uses an ordered list, since the sequence from home to current page genuinely matters, and that the final item — the current page — is plain text, not a link, since linking to the page a visitor is already on serves no purpose.

When Breadcrumbs Help

Breadcrumbs earn their place on sites with real depth — a store with categories and subcategories, a documentation site with nested topics, a multi-step lesson library. A small five-page site, like the Student Learning Site running through this textbook, does not need a breadcrumb trail; its full set of pages already fits comfortably in a single navigation menu. Adding breadcrumbs to a flat, shallow site adds visual clutter without adding clarity.

Section 7.9

Site Structure and Information Architecture

Information architecture is the practice of organizing and labeling the content of a site so that people can find what they need. It is a planning activity that happens before — and continues alongside — writing any HTML. A site with strong information architecture feels obvious to navigate; a site with weak information architecture leaves visitors clicking around, unsure where to look.

Two broad patterns describe most small to medium sites:

PatternDescriptionGood Fit For
Flat structureA small number of pages, all reachable directly from one main navigation menu, with no deep nesting.Small business sites, portfolios, personal sites, course companion sites
Hierarchical structurePages organized into categories and subcategories, often several levels deep, usually paired with breadcrumbs.Large documentation sites, online stores, news archives, large institutions

Before writing a single file, it helps to sketch a site map — a simple diagram or outline listing every planned page and how they connect. Restructuring a site map on paper costs minutes; restructuring a site after dozens of pages already link to one another costs hours of careful path-fixing. This is the same lesson Chapter 3 taught about project folders, applied one level up: plan the shape of the whole site, not just the shape of one folder.

Example 7.9.1
A site map for the Student Learning Site
  • HomeThe entry point. Links out to every other top-level page; nothing links into the site more often than this one.
  • AboutA single page describing the purpose of the site. No subpages needed.
  • LessonsA landing page that, in later chapters, may grow links to individual lesson pages — the beginning of a small hierarchy.
  • ResourcesA page listing external and internal resources, later powered by a JSON file in Chapter 28.
  • ContactA page containing the accessible contact form built in Chapter 9.

This is a flat structure: every page is one click from every other page through the same navigation menu. That choice is deliberate, not accidental — for a five-page learning site, a flat structure is simpler to build, simpler to maintain, and easier for a visitor to understand at a glance than a hierarchy would be.

Section 7.10

Building a Multi-Page Link System

Every idea in this chapter comes together in one practical task: building a small set of pages that link to each other correctly and consistently. This section walks through that process from start to finish, using the Student Learning Site site map from Section 7.9.

Project Folder Structure
web-project/
  index.html
  about.html
  lessons.html
  resources.html
  contact.html
Worked Example
From site map to working links
  • PlanConfirm the site map: five pages, flat structure, one shared navigation menu, all files in the same top-level folder.
  • Build the skeletonCreate each file with the basic HTML skeleton from Chapter 5 and a meaningful <title> for every page.
  • Write one navigation menuWrite the <nav> markup from Section 7.7 once, then copy it into every page. Because all five files sit in the same folder, every link in the menu can use a same-folder relative path: href="about.html", href="lessons.html", and so on.
  • Mark the current pageOn each page, add aria-current="page" to the one navigation link that matches that page, and update it as you copy the menu across files.
  • Add internal anchors where usefulOn longer pages, such as Lessons, add id attributes to major sections and a short set of jump links at the top, following Section 7.6.
  • Test every linkOpen each page in the browser and click every link in the navigation menu, confirming it lands on the correct page and that the correct link is marked as current. Chapter 4's developer tools make this faster once a site grows beyond a handful of pages.

A site built this way — small, flat, consistently linked, and carefully tested — is exactly the foundation the rest of this textbook builds on. Part III adds styling and layout to these same pages. Part IV adds JavaScript interaction. Part V revisits navigation again, in Chapter 26, to show how to keep a shared header and footer consistent without duplicating effort across many more pages than five.

Key Terms

hyperlink
A piece of content that, when activated, takes the user to another location: a page, a part of a page, a file, or an external resource.
anchor element
The HTML element, written <a>, used to create a hyperlink.
href
The attribute on an anchor element that specifies the link's destination. Required for the element to function as a link.
relative link
A link whose destination is described as a path from the current file, rather than as a complete address.
absolute link
A link whose destination is a complete URL, including protocol and domain, usually used to link to other websites.
target attribute
An attribute that controls where a link's destination opens, such as a new browser tab when set to _blank.
rel attribute
An attribute describing the relationship between the current page and a linked page; also used for security settings such as noopener noreferrer.
mailto link
A link whose href begins with mailto:, opening the user's email program with a new message.
fragment identifier
A link destination beginning with # that points to an element with a matching id attribute on a page.
id attribute
An attribute giving an element a unique identifying name within a page, used for fragment identifiers and other targeted references.
navigation menu
A group of links, typically wrapped in <nav> and an unordered list, that lets visitors move between a site's major pages or sections.
skip link
An accessible internal link, usually the first focusable element on a page, that lets keyboard and screen-reader users jump directly to the main content.
breadcrumb trail
A row of links showing the path from a site's home page down to the current page, typically marked up as an ordered list.
information architecture
The practice of organizing and labeling a site's content so that people can find what they need.
site map
A diagram or outline listing a site's planned pages and how they connect, sketched before building begins.

Common Misconceptions

Misconception 1
"Any clickable-looking element is a link."
Only an element with a working href — typically an anchor element — behaves like a real link: reachable by keyboard, navigable to a destination, and announced as a link by assistive technology. Styling something to look clickable does not make it a link. If it has a destination, use <a href="...">; if it triggers behavior with no destination, use a <button>, covered further in Chapter 21.
Misconception 2
"Relative paths work the same no matter which file uses them."
A relative path is always relative to the file that contains it. The same href value can point to two completely different destinations depending on where the file using it is located. Moving a file to a different folder without updating its links — or the links pointing to it — is one of the most common sources of broken links.
Misconception 3
"Opening links in a new tab is always more convenient for users."
New tabs can help in specific situations, such as linking to a reference a reader may want to keep open, but used by default they clutter a user's browser and can confuse people using assistive technology, who are not always informed that a new tab has opened. Use target="_blank" deliberately, and pair it with rel="noopener noreferrer".
Misconception 4
"Every website needs a breadcrumb trail."
Breadcrumbs solve a specific problem: helping visitors track their position in a deep, hierarchical site. A small, flat site where every page is one click away through a single navigation menu does not have that problem, and adding breadcrumbs to it adds visual clutter without adding clarity.
Misconception 5
"Site structure can be figured out while building, one page at a time."
Some structure can certainly evolve, but planning a site map in advance — even a rough one — costs far less time than discovering, after dozens of pages already link to one another, that the structure needs to change. Information architecture is a planning discipline, not an afterthought.

  1. Sketch a site map, in outline or diagram form, for a small site of your choosing — for example, a recipe site, a hobby club site, or a personal portfolio. List every planned page and draw or describe which pages link to which.

  3. Identify whether your site would benefit from a breadcrumb trail. Explain your reasoning using the guidance from Section 7.8.
  4. List the navigation menu items, in the order you would place them, and explain your ordering choice.

  3. What does target="_blank" do? What attribute commonly accompanies it, and why?
  4. What is a fragment identifier? What attribute must exist somewhere on the page for it to work?

  6. What is information architecture?

Understanding and Explanation

  1. Explain why the web is sometimes described as a graph of linked documents rather than a simple stack of pages.
  2. Explain why relative links continue to work when an entire project folder is moved to a new server, while absolute links to other sites do not need to change at all.
  3. Explain why link text such as "click here" is considered poor practice, using the idea of a screen reader's link list to support your answer.
  4. Explain the difference between marking up a navigation menu as a plain list of links versus wrapping it in <nav> with an unordered list. What does the extra structure communicate, and to whom?
  5. Explain why a small, flat site generally does not need breadcrumbs, while a large, hierarchical site usually benefits from them.

Application and Analysis

  1. A student builds a five-page site entirely in one folder, then later decides to move all the lesson-related pages into a new lessons subfolder. List every kind of link in the project that might now be broken, and explain, for each kind, what needs to change.
  2. A page includes the following link: <a href="https://www.partner-site.org/" target="_blank">Visit our partner</a>. Identify one improvement you would make to this line and explain why, using ideas from Section 7.5.
  3. A classmate argues that breadcrumbs should appear on every page of every site "just in case it grows later." Write a short response, using the distinction between flat and hierarchical structures from Section 7.9.
  4. Look back at the Student Learning Site map in Section 7.9. Suppose the site later adds ten individual lesson pages under the Lessons section. Would you keep the structure flat or shift toward a hierarchical pattern with breadcrumbs? Justify your answer.
