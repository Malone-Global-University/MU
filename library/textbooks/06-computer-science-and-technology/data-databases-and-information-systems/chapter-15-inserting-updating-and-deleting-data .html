Inserting, Updating, and Deleting Data

Every table you have queried so far was already populated, waiting patiently for a SELECT. But data does not appear by itself. This chapter introduces INSERT, UPDATE, and DELETE, the three commands that add, change, and remove records, along with the habits that keep these powerful commands safe.

Every database you have studied so far has contained data that was already there. The examples in Part III assumed that tables were populated and ready to query. But data does not appear in tables by itself. Someone, or some system, put it there. And once data is there, it may need to change, and eventually it may need to be removed.

This chapter introduces the three SQL commands that modify the contents of a database, INSERT, which adds new records, UPDATE, which changes existing records, and DELETE, which removes records. Together these three commands give you the power to build and maintain a living database.

That power comes with responsibility. A SELECT query that contains a mistake shows you the wrong result, and you simply close it and try again. An UPDATE or DELETE command that contains a mistake can change or erase records permanently. A poorly written UPDATE with no WHERE clause can overwrite every row in a table in a single command. A poorly written DELETE with no WHERE clause can remove an entire table's contents in seconds.

The goal of this chapter is not to make you afraid of data modification. The goal is to make you careful, methodical, and precise. Good database practitioners write modification commands slowly, check their conditions first with a SELECT, test on small datasets, and keep backups before making large changes. By the end of this chapter, you will understand how records are added, changed, and removed, and you will have built the habits that protect databases from avoidable mistakes.

Note for New Students

This chapter assumes you are comfortable reading and writing SELECT queries from Part III. The same tables, the same WHERE clauses, and the same logical thinking apply here. The difference is that these commands change the database rather than simply reading from it, so the stakes for getting the condition right are much higher.

15.1

Databases Change Over Time

A database is not a frozen document. It is a living system. Records are added as new things happen, records are updated as information changes, and records are deleted as things become irrelevant, outdated, or incorrect. Every day, real databases everywhere are being modified by people, forms, applications, automated processes, and system events.

Think about the kinds of changes that happen in common databases.

In a school enrollment system, a student registers for a new course, and that registration is a new record in the Enrollments table. Later, the student drops the course, and that enrollment record is either deleted or updated to reflect a withdrawn status. At the end of the term, final grades are recorded, which is an update to the Submissions or Grades table.

In a library management system, a new book arrives, and a record is inserted into the Resources table. A borrower checks out the book, and a record is inserted into the Loans table. When the book is returned, the loan record is updated with a return date. When a book is damaged beyond repair and removed from the collection, its record may be deleted or marked inactive.

In a store inventory system, a supplier delivers a shipment, and records are inserted into the Inventory table or existing quantity records are updated. When a product is discontinued, its record might be deleted or flagged as inactive.

In every case, the database must be able to grow, change, and shrink. SQL provides three commands for this purpose.

INSERT

Adds a new row to a table.

UPDATE

Changes one or more values in existing rows.

DELETE

Removes one or more rows from a table.

These commands are called data manipulation language, or DML. They operate on the data inside tables. They are different from data definition language commands, which create or drop the tables themselves.

15.2

Adding Records with INSERT

The INSERT statement adds a new row to a table. The basic syntax has two parts, a list of columns and a list of values that match those columns in order.

SQL — INSERT Syntax
INSERT INTO table_name (column1, column2, column3)
VALUES (value1, value2, value3);

Consider the following Students table.

Table 15.1 — The Students table before a new record is added.
student_idfirst_namelast_nameenrollment_datestatus
1AmaraOsei2023-09-01active
2LenaVasquez2023-09-01active
3PriyaMehta2024-01-15active

To add a new student, you would write:

SQL — Inserting a New Student
INSERT INTO Students (student_id, first_name, last_name, enrollment_date, status)
VALUES (4, 'Dawit', 'Bekele', '2024-09-01', 'active');

After this command runs, the Students table contains four rows. The new record appears with the values you provided.

Table 15.2 — The Students table after the new record is inserted.
student_idfirst_namelast_nameenrollment_datestatus
1AmaraOsei2023-09-01active
2LenaVasquez2023-09-01active
3PriyaMehta2024-01-15active
4DawitBekele2024-09-01active

Notice that the column list matches the value list in order. The student_id column receives 4, first_name receives 'Dawit', last_name receives 'Bekele', and so on. If the order or count does not match, the database will return an error, or worse, it will silently store values in the wrong columns.

Specifying Columns Is Best Practice

You do not technically have to list the column names in an INSERT. If you omit them, the database assumes you are providing values for every column in the exact order the table was defined.

SQL — Without Column Names, Risky
-- Without column names: risky
INSERT INTO Students
VALUES (4, 'Dawit', 'Bekele', '2024-09-01', 'active');

This works, but it is fragile. If someone later adds or rearranges columns in the table definition, your INSERT will break or insert data into the wrong columns. Always name your columns explicitly. It makes the query safer, clearer, and easier to maintain.

Text Values Need Quotes

Notice that text values and date values are enclosed in single quotes, while numbers are not.

SQL — Quoting Text and Date Values
INSERT INTO Students (student_id, first_name, last_name, enrollment_date, status)
VALUES (4, 'Dawit', 'Bekele', '2024-09-01', 'active');

A common beginner mistake is to forget quotes around text values, or to use double quotes instead of single quotes. Most SQL databases expect single quotes around text and date literals.

Inserting Multiple Rows

Many SQL databases allow you to insert multiple rows in a single INSERT statement by separating each value list with a comma.

SQL — Inserting Several Rows at Once
INSERT INTO Students (student_id, first_name, last_name, enrollment_date, status)
VALUES
  (4, 'Dawit', 'Bekele', '2024-09-01', 'active'),
  (5, 'Yuna', 'Park', '2024-09-01', 'active'),
  (6, 'Tobias', 'Larsen', '2024-09-01', 'active');

This is more efficient than three separate INSERT statements and is commonly used when loading a batch of new records.

15.3

Required Fields and Default Values

Not every column in an INSERT needs to be included if the column has a default value or is allowed to be null. But columns that are required, meaning they have a NOT NULL constraint and no default, must always be included. If you omit a required column, the database will reject the INSERT with a constraint error.

Columns with Default Values

When a table is designed, certain columns can be given default values. A default value is automatically used when no value is provided in an INSERT.

For example, the status column in the Students table might be defined with a default of 'active'. If so, you can omit status from the INSERT and the database will fill it in automatically.

SQL — Relying on a Default Value
INSERT INTO Students (student_id, first_name, last_name, enrollment_date)
VALUES (4, 'Dawit', 'Bekele', '2024-09-01');
-- status is filled automatically as 'active'

The resulting row will still have status = 'active' because the default kicks in.

Common Default Value Uses

Default values appear frequently in real databases. Here are some common patterns.

Table 15.3 — Common situations where default values are useful.
SituationDefault Value
A status field for new records'active', 'pending', or 'draft'
A created date or timestampThe current date and time
A boolean flag for new recordstrue or false
A numeric field with a starting value0 or 1

These defaults reduce the burden on the person or system entering data and reduce the chance that required information is forgotten.

Auto-Generated Primary Keys

Many tables use an auto-incrementing integer as the primary key. In SQLite, a column defined as INTEGER PRIMARY KEY increments automatically. When you insert a row without providing a value for that column, the database assigns the next available integer.

SQL — Letting the Primary Key Auto-Increment
-- Table defined with: student_id INTEGER PRIMARY KEY AUTOINCREMENT
INSERT INTO Students (first_name, last_name, enrollment_date, status)
VALUES ('Dawit', 'Bekele', '2024-09-01', 'active');
-- student_id is assigned automatically, for example 4

This means you often do not need to track or supply primary key values when inserting records. The database manages them for you.

What Happens When a Required Field Is Missing

If you omit a column that has no default and is defined as NOT NULL, the database will refuse the insert.

SQL — A Rejected INSERT
-- This will fail if last_name is required and has no default
INSERT INTO Students (student_id, first_name, enrollment_date)
VALUES (4, 'Dawit', '2024-09-01');
-- Error: NOT NULL constraint failed: Students.last_name

Reading these error messages carefully is an important skill. The error tells you exactly which constraint was violated and on which column.

15.4

Updating Records with UPDATE

The UPDATE statement changes the values in one or more existing rows. The basic syntax is:

SQL — UPDATE Syntax
UPDATE table_name
SET column1 = value1, column2 = value2
WHERE condition;

The SET clause lists the columns you want to change and their new values. The WHERE clause identifies which rows to change.

Consider the Students table again. Suppose Priya Mehta has been placed on academic leave, and her status should change from 'active' to 'on_leave'.

SQL — Updating One Student's Status
UPDATE Students
SET status = 'on_leave'
WHERE student_id = 3;

After this runs, only row 3 is changed. Rows 1, 2, and 4 are untouched.

Table 15.4 — The Students table after the targeted update.
student_idfirst_namelast_nameenrollment_datestatus
1AmaraOsei2023-09-01active
2LenaVasquez2023-09-01active
3PriyaMehta2024-01-15on_leave
4DawitBekele2024-09-01active
Updating Multiple Columns at Once

You can update more than one column in a single UPDATE by separating the assignments with commas in the SET clause.

SQL — Updating Two Columns at Once
UPDATE Students
SET first_name = 'Priya S.',
    status = 'active'
WHERE student_id = 3;

This updates both first_name and status for row 3 at the same time.

Using Expressions in SET

The new value in an UPDATE does not have to be a literal. It can be an expression that refers to the column's current value or to another column.

Suppose a Products table has a price column, and you need to apply a ten percent price increase to all products in a certain category.

SQL — Updating Using an Expression
UPDATE Products
SET price = price * 1.10
WHERE category = 'textbooks';

This multiplies the existing price by 1.10 for every row where category equals 'textbooks'. The original value of each row's price is used in the calculation.

15.5

Safe Update Conditions

The most dangerous part of UPDATE is the WHERE clause, or rather, the absence of one.

Consider what happens if you run the following:

SQL — Missing WHERE Clause
UPDATE Students
SET status = 'on_leave';

There is no WHERE clause. This updates every single row in the Students table. Every student is now marked 'on_leave'. If this was not your intention, you have a serious problem.

This Is Not Hypothetical

It happens to real practitioners who are working quickly, forget the condition, and press Enter. In a database with thousands of students, the damage happens in milliseconds.

Always Write the WHERE Clause Before the SET Clause

One useful habit is to write the WHERE clause first, even before the SET clause, when you are thinking through an UPDATE. Decide which rows you want to change before deciding what to change.

Another habit is to run the WHERE condition as a SELECT query first, before running the UPDATE. This shows you exactly which rows will be affected.

SQL — Preview, Then Update
-- Step 1: Check which rows match the condition
SELECT student_id, first_name, last_name, status
FROM Students
WHERE student_id = 3;

-- Step 2: If the result looks correct, run the UPDATE
UPDATE Students
SET status = 'on_leave'
WHERE student_id = 3;

This two-step process is one of the most important habits in database work. The SELECT shows you what you are about to change. The UPDATE makes the change.

Be Specific

Prefer conditions that identify rows precisely. Primary keys are the most reliable identifiers because they are unique.

SQL — Precise Versus Risky Conditions
-- Precise: uses primary key
UPDATE Students SET status = 'on_leave' WHERE student_id = 3;

-- Risky: multiple students might share a last name
UPDATE Students SET status = 'on_leave' WHERE last_name = 'Mehta';

The second version will update every student with the last name Mehta. If there are three students with that name, all three will be updated.

Checking the Count Before Updating

Another good habit is to check how many rows your condition matches before running the update.

SQL — Counting Matching Rows First
-- How many rows will this affect?
SELECT COUNT(*)
FROM Students
WHERE last_name = 'Mehta';

If the result is one, you can proceed with confidence. If the result is twelve, you may want to narrow your condition.

15.6

Deleting Records with DELETE

The DELETE statement removes one or more rows from a table. The basic syntax is:

SQL — DELETE Syntax
DELETE FROM table_name
WHERE condition;

For example, suppose a student withdraws from the university and their record should be removed from the Students table.

SQL — Deleting One Student
DELETE FROM Students
WHERE student_id = 4;

After this runs, the row for Dawit Bekele is gone.

Table 15.5 — The Students table after the targeted delete.
student_idfirst_namelast_nameenrollment_datestatus
1AmaraOsei2023-09-01active
2LenaVasquez2023-09-01active
3PriyaMehta2024-01-15on_leave

Row 4 no longer exists. The deletion is permanent unless you have a transaction that can be rolled back, which is covered in Chapter 16.

The Danger of DELETE Without WHERE

Just as with UPDATE, a DELETE without a WHERE clause affects every row.

SQL — Deleting Every Row
DELETE FROM Students;

This removes every student from the Students table. The table structure, meaning the column definitions and constraints, remains. But every single record is gone. This is rarely the intention, and it can cause serious data loss.

Always Include a WHERE Clause

Always include a WHERE clause with DELETE. Always.

Check Before You Delete

Apply the same two-step process used for updates.

SQL — Preview, Then Delete
-- Step 1: Check which rows the condition matches
SELECT student_id, first_name, last_name
FROM Students
WHERE student_id = 4;

-- Step 2: If it looks correct, run the DELETE
DELETE FROM Students
WHERE student_id = 4;

Seeing the row before deleting it gives you a chance to confirm you have the right record.

15.7

Safe Delete Conditions

The same principles that make UPDATE conditions safe apply to DELETE conditions.

Use primary keys when possible. Use SELECT to preview what will be deleted. Count the matching rows before deleting them.

SQL — Counting, Then Deleting
-- How many rows will be deleted?
SELECT COUNT(*)
FROM Loans
WHERE return_date IS NULL AND loan_date < '2020-01-01';

-- If the count looks right, proceed
DELETE FROM Loans
WHERE return_date IS NULL AND loan_date < '2020-01-01';
Soft Delete Versus Hard Delete

In many real systems, records are not actually deleted. Instead, they are marked as inactive or archived. This is called a soft delete. The record remains in the database but is hidden from ordinary queries.

A hard delete uses DELETE FROM and actually removes the row. A soft delete uses UPDATE to set a flag.

SQL — Hard Delete Versus Soft Delete
-- Hard delete: the record is gone
DELETE FROM Students WHERE student_id = 4;

-- Soft delete: the record is kept but marked inactive
UPDATE Students SET status = 'withdrawn' WHERE student_id = 4;

Soft deletes preserve history. If someone asks what happened to student number 4, the record still exists and can be queried. Hard deletes are irreversible. Many organizations prefer soft deletes for important records because deleted data cannot be recovered without a backup.

The choice between soft and hard deletes depends on the system's requirements, legal or policy obligations, and how important historical accuracy is. Some systems use both, soft deletes for routine deactivations, and hard deletes only for records that genuinely must be purged, such as data that privacy regulations require to be fully erased.

15.8

Constraints During Data Changes

Constraints are rules defined when a table is created. They protect data quality by rejecting modification commands that would violate the rules. Every INSERT, UPDATE, and DELETE must pass through these constraints before it can succeed.

Primary Key Constraints

You cannot insert a row that duplicates an existing primary key. The primary key must uniquely identify every row.

SQL — Duplicate Primary Key Rejected
-- student_id 1 already exists
INSERT INTO Students (student_id, first_name, last_name, enrollment_date, status)
VALUES (1, 'New', 'Student', '2024-09-01', 'active');
-- Error: UNIQUE constraint failed: Students.student_id

You also cannot update a primary key value to one that already exists in another row.

NOT NULL Constraints

You cannot insert or update a row to have a null value in a column defined as NOT NULL.

SQL — NOT NULL Violation
UPDATE Students
SET last_name = NULL
WHERE student_id = 2;
-- Error: NOT NULL constraint failed: Students.last_name
Foreign Key Constraints

Foreign key constraints protect referential integrity. You cannot insert a record into a child table with a foreign key value that does not exist in the parent table.

Suppose Enrollments has a student_id foreign key that references Students. You cannot enroll a student who does not exist.

SQL — Foreign Key Violation on Insert
INSERT INTO Enrollments (enrollment_id, student_id, course_id, term)
VALUES (101, 999, 5, 'Fall 2024');
-- Error: FOREIGN KEY constraint failed
-- student_id 999 does not exist in Students

Similarly, you cannot delete a student from the Students table if there are enrollment records in Enrollments that reference that student, unless the foreign key is set to cascade the deletion or set the foreign key to null.

SQL — Foreign Key Violation on Delete
DELETE FROM Students WHERE student_id = 1;
-- Error: FOREIGN KEY constraint failed
-- Enrollment records still reference student_id 1

This is referential integrity in action. The database refuses to leave orphan records, meaning records in a child table that point to a parent record that no longer exists.

CHECK Constraints

Some tables define CHECK constraints that enforce specific conditions on values. For example, a grade column might be constrained to accept only values between 0 and 100.

SQL — CHECK Constraint Violation
UPDATE Submissions SET grade = 115 WHERE submission_id = 45;
-- Error: CHECK constraint failed: grade must be between 0 and 100
Constraints Are Your Allies

When a constraint rejects a modification command, it is protecting the database from bad data. Read the error message, understand which constraint was violated, and fix the command before retrying.

15.9

Testing Before Changing Data

Testing before modifying data is a professional habit. Mistakes in UPDATE and DELETE commands can be difficult or impossible to reverse.

The SELECT-Before-Modify Pattern

Before running any UPDATE or DELETE, run the corresponding SELECT using the same WHERE clause. This previews exactly which rows will be affected.

SQL — Previewing Before an Update
-- Before UPDATE: preview the rows
SELECT student_id, first_name, status
FROM Students
WHERE enrollment_date < '2023-01-01' AND status = 'active';

-- If the result is what you expected, proceed
UPDATE Students
SET status = 'graduated'
WHERE enrollment_date < '2023-01-01' AND status = 'active';
Counting Rows Before Modifying

A quick COUNT(*) query shows how many rows match your condition. If you expected to change three rows but the count returns thirty, stop and re-examine your condition.

SQL — A Sanity Check Count
SELECT COUNT(*)
FROM Loans
WHERE return_date IS NULL;
-- Returns: 47
-- Is 47 the number you expected?
Testing on Small Data First

When working with unfamiliar tables or large datasets, consider adding an extra condition to limit the scope of your first test.

SQL — Testing on One Row First
-- Test on one specific record first
UPDATE Products SET price = price * 1.10 WHERE product_id = 1;

-- Check the result
SELECT product_id, price FROM Products WHERE product_id = 1;

-- If correct, apply to the full set
UPDATE Products SET price = price * 1.10 WHERE category = 'textbooks';

This incremental approach catches mistakes before they affect the whole dataset.

Using Transactions for Safety

Chapter 16 covers transactions in detail, but you should know that wrapping a modification command in a transaction gives you a way to undo it if something goes wrong.

SQL — A Reversible Delete with a Transaction
BEGIN;
DELETE FROM Enrollments WHERE student_id = 4;
-- Review the result before committing
SELECT * FROM Enrollments WHERE student_id = 4;
ROLLBACK; -- Undo the delete if something was wrong
-- Or: COMMIT; -- Make it permanent if everything looks right

Transactions are one of the most important tools for safe data modification. If you are making changes you are not fully sure about, begin a transaction, make the change, review the result, and then either commit or roll back.

15.10

Data Modification Responsibility

The ability to modify database records is significant. In most organizations, not everyone has permission to insert, update, or delete data. Chapter 18 covers permissions and access control in detail. For now, consider the human dimension of data modification.

Why Carefulness Matters

Records in a database represent real things in the world. A student's enrollment record affects their academic standing. A loan record in a library system determines whether a patron owes a late fee. An inventory record determines what a store orders from its supplier. A patient record in a health system influences clinical decisions.

When a record is changed incorrectly, the consequences extend beyond the database. A student may lose credit for a course they completed. A patron may be wrongly charged. A store may order too much or too little of a product. A clinician may see inaccurate information.

Data modification commands must be written and reviewed with the awareness that they affect real systems and real people.

The Principle of Minimum Necessary Change

Change only what needs to change. If a student's address has changed, update the address, and do not update unrelated fields. If a single enrollment record is incorrect, fix that record, and do not write a blanket update that touches many records unless you are certain all of them need the same change.

Documenting What You Changed and Why

In real database work, important modifications are often logged or documented. Who made the change, when, and why? What was the value before, and what is it now? Audit logs, version histories, and change notes help organizations understand what happened to their data and trace problems when they arise.

Backups Before Major Changes

Before running a large batch update or delete, take a backup or export a snapshot of the affected table. This gives you a recovery path if something goes wrong.

  • Preview Before You Act

    Run the matching SELECT first, every time, before any UPDATE or DELETE.

  • Identify Rows Precisely

    Use primary keys or other unique identifiers in your WHERE clause whenever possible, rather than relying on names or categories that might match more rows than expected.

  • Count Before You Commit

    Check how many rows your condition matches, and compare that number to what you expected, before letting the change take effect.

  • Prefer Reversibility

    Use transactions for risky changes, and prefer soft deletes over hard deletes when historical accuracy matters.

  • Respect the Constraints

    Treat constraint errors as protection, not as obstacles, and investigate the underlying cause rather than working around the rule.

Chapter Summary
  • This chapter introduced the three SQL commands that modify the contents of a database. INSERT adds new rows to a table, requiring a list of columns and a matching list of values. Columns with default values or auto-generated keys do not need to be supplied explicitly.
  • UPDATE changes values in existing rows using a SET clause and a WHERE clause. The WHERE clause is essential, since an UPDATE without one affects every row in the table.
  • DELETE removes rows from a table using a WHERE clause. Like UPDATE, a DELETE without a WHERE clause removes all rows.
  • The most important theme of this chapter was caution. Before running any UPDATE or DELETE, use a SELECT with the same WHERE clause to preview which rows will be affected, count the matching rows before modifying them, and test on individual records before applying batch changes.
  • Transactions give you an undo path for risky modifications, and soft deletes are often preferred over hard deletes when historical records matter.
  • Constraints enforced by the database, including primary key uniqueness, NOT NULL, foreign key references, and CHECK conditions, protect data quality by rejecting modification commands that would violate the rules. When a constraint error appears, read the message carefully and understand which rule was broken.
  • Data modification commands are powerful, and the records stored in a database represent real people, organizations, and events. Using INSERT, UPDATE, and DELETE carefully and responsibly is a core skill of good database practice.
Key Terms
INSERT

The SQL command that adds one or more new rows to a table.

UPDATE

The SQL command that changes values in one or more existing rows.

DELETE

The SQL command that removes one or more rows from a table.

SET clause

The part of an UPDATE statement that lists the columns to change and their new values.

WHERE clause

The condition that filters which rows are affected by UPDATE or DELETE. Omitting it causes the command to affect all rows.

Default value

A value automatically assigned to a column when no value is provided in an INSERT.

NOT NULL constraint

A rule that prevents a column from containing null values.

Foreign key constraint

A rule that requires a value in a child table's foreign key column to match an existing value in the parent table's primary key column.

Referential integrity

The property of a database where all foreign key references point to existing records.

Soft delete

Marking a record as inactive or withdrawn instead of physically removing it from the database.

Hard delete

Physically removing a row from a table using DELETE FROM.

Data manipulation language (DML)

The subset of SQL commands that modify the contents of a database, including INSERT, UPDATE, and DELETE.

Auto-increment

A database feature that automatically generates a unique integer primary key for each new row.

Audit log

A record of who changed what data, when, and why.

Coming Next — Chapter 16: Transactions and Data Integrity

Chapter 16 introduces transactions. A transaction is a group of modification commands that are treated as a single unit, so either all of them succeed, or none of them do. Transactions protect databases from partial changes that would leave data in an inconsistent state. You will learn the BEGIN, COMMIT, and ROLLBACK commands, understand atomicity at an introductory level, and see how transactions are used in systems such as banking, enrollment, reservations, and inventory management.

Chapter 16 builds directly on the INSERT, UPDATE, and DELETE commands you have learned here, and it introduces one of the most important ideas in reliable database design, that some changes must succeed or fail together.
