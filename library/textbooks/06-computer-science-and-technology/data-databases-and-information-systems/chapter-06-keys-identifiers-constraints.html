Keys, Identifiers, and Constraints

Every reliable database depends on the ability to identify records without confusion, connect tables without ambiguity, and enforce rules without exceptions. Primary keys, foreign keys, uniqueness constraints, and referential integrity are not optional features. They are the foundation that keeps data trustworthy.

Every time a library system retrieves a patron's borrowing history, it depends on something that is easy to overlook: a reliable way to point to exactly one person. Not someone with a similar name, not a former patron who once had the same address, but this one person, identified without ambiguity. When a hospital looks up a patient's medication record, the stakes are even higher. When an online store updates the status of a specific order, it cannot afford to update the wrong one.

Identifiers make reliable systems possible. Without them, records merge, updates overwrite the wrong rows, relationships between tables break apart, and the entire system becomes untrustworthy. This is not a small technical detail. It is one of the most consequential design decisions in any database.

Chapter 6 explains how databases identify records precisely, protect relationships between tables, and enforce rules that preserve data integrity over time. Students will learn primary keys, foreign keys, candidate keys, natural keys, surrogate keys, uniqueness constraints, required fields, referential integrity, and the reasoning behind each of these tools. By the end of this chapter, the goal is not only to know what these terms mean, but to understand why they exist and what goes wrong when they are missing or poorly chosen.

Building on Chapter 5

Chapter 5 introduced entities, attributes, and relationships at a conceptual level. Chapter 6 moves from concept to implementation, explaining how databases actually enforce the structure that Chapter 5 described. The ideas connect directly: every entity needs a primary key, every relationship between entities is implemented through a foreign key, and the rules that protect both are enforced through constraints.

6.1

Why Records Need Identifiers

Imagine a library that keeps paper records for every book it owns. Each record describes one book: its title, its author, the year it was published, and its current location on the shelves. Now imagine that two books in the library have nearly identical titles. One is Introduction to Programming, published in 2018, and the other is Introduction to Programming, published in 2021. When a patron asks a librarian to reserve the 2021 edition, the librarian needs something more than a title to locate the right record and avoid a mix-up.

In everyday record keeping, people solve this problem with identifiers. A library uses a call number. A hospital uses a patient number. A university uses a student ID. An online store uses an order number. A government uses a passport number. These identifiers are not chosen at random. They are chosen because they point to exactly one record and cannot reasonably be confused with any other.

Databases face the same problem, and they solve it the same way. Every table needs a column or combination of columns that uniquely identifies each record. Without this, several serious problems appear.

  • The Update ProblemIf you want to change the email address for a specific student, you need to be certain you are changing that student's record and not a different student who shares the same name. Without a reliable identifier, the database cannot guarantee you are pointing to the right record.
  • The Retrieval ProblemIf you ask the system to find a specific order, the system needs to find exactly one record. If two orders could match the same search, the system cannot give a reliable answer.
  • The Relationship ProblemChapter 5 introduced relationships between tables. A loan record in a library system connects a patron to a specific book. For that connection to work correctly, the loan table must be able to refer to exactly one patron and exactly one book. If patron identifiers are ambiguous, the connection breaks.
  • The Deletion ProblemWhen you delete a record, you need to delete the right one. If identifiers are unclear, a deletion could remove the wrong record permanently, with no way to undo the loss.

These problems explain why database design treats identifiers as fundamental. A table without a reliable way to identify its records is not a well-designed table. It is a collection of data that cannot be reliably used, updated, or trusted.

6.2

Primary Keys

A primary key is a column or combination of columns in a table that uniquely identifies each row. Every table should have exactly one primary key. The primary key is the table's main identifier, the value that the database and any connected system will use to point to a specific record.

Primary keys must satisfy two requirements.

First, a primary key value must be unique. No two rows in the table may share the same primary key value. If a student table uses a column called student_id as its primary key, then every student in the table must have a different value in that column. Two students cannot both have student_id = 1042.

Second, a primary key value must be not null. A row that has no value in its primary key column cannot be identified at all. The database enforces this automatically. If you try to insert a record without providing a primary key value, and the database is correctly configured, the insertion will fail.

A Simple Example

Consider a table called students with these columns:

Table 6.1 — The students table. student_id is the primary key: every value is unique and none is missing.
student_id first_name last_name email year_enrolled
1001AmaraOseiamara.osei@mgu.edu2022
1002FelixMendezfelix.mendez@mgu.edu2023
1003YukiTakahashiyuki.takahashi@mgu.edu2022
1004SofiaBrunettisofia.brunetti@mgu.edu2024

In this table, student_id is the primary key. Every student has a unique value. No value is missing. If a new student enrolls, they receive a new unique student_id value, and their record can always be found precisely.

Notice that last_name alone could not serve as the primary key in this table. If another student named Osei enrolled, two records would share the same last name and the column would no longer be unique. Similarly, email might seem unique, but if email addresses can change or contain errors, relying on them as primary keys creates risk.

Primary Keys in SQL

When creating a table in SQL, you declare the primary key as part of the table definition:

SQL — Declaring a Primary Key
CREATE TABLE students (
    student_id    INTEGER PRIMARY KEY,
    first_name    TEXT    NOT NULL,
    last_name     TEXT    NOT NULL,
    email         TEXT,
    year_enrolled INTEGER
);

The PRIMARY KEY declaration tells the database to enforce both the uniqueness and the not-null requirement on that column automatically. If you try to insert a second record with student_id = 1001, the database will reject it with an error.

What the Primary Key Does in Practice

The primary key is not just a rule enforced at insertion. It shapes how the entire database works. Other tables that need to refer to a student will use the student_id value to make that reference. Queries that search for a specific student will use student_id to locate the exact record. Systems built on top of the database, such as web applications or reports, will pass student_id values around as the stable way to point to a student.

This means the primary key should be chosen carefully. It should be stable, unique, and appropriate for the table. The rest of this chapter explains the different choices available and the considerations that shape each decision.

6.3

Candidate Keys

Before a primary key is chosen, there may be several columns or combinations of columns that could theoretically serve as the primary key. These possibilities are called candidate keys.

A candidate key is any column or set of columns that is both unique across all rows and never null. A table may have one candidate key, or it may have several. The designer chooses one of them to become the primary key. The others remain as candidates and may be protected with uniqueness constraints even though they are not the primary key.

An Example with Multiple Candidates

Suppose the university issues each student a unique national identification number, and every student is required to have an email address.

Table 6.2 — Three columns in this table qualify as candidate keys: student_id, national_id, and email.
student_id national_id first_name last_name email
1001NID-4421-08AmaraOseiamara.osei@mgu.edu
1002NID-7703-15FelixMendezfelix.mendez@mgu.edu
1003NID-9902-31YukiTakahashiyuki.takahashi@mgu.edu

In this table, three columns are candidates: student_id, which is unique and not null; national_id, which is unique and not null; and email, which is unique and not null (assuming the rule is enforced). Any one of these could serve as the primary key. The designer must choose one. The others become alternate keys, and uniqueness constraints will ensure they remain unique even after the primary key is chosen.

Why the Distinction Matters

Recognizing candidate keys matters because it forces the designer to think explicitly about which properties of a record are truly unique. It also reveals potential redundancy. If both student_id and national_id are unique identifiers, the designer should consider why both exist and which one is more appropriate as the primary key.

It also prevents assumptions. Just because a column appears unique in today's data does not mean it will always be unique. A designer who recognizes candidate keys will ask: is this value guaranteed to be unique for every record that will ever be inserted? Can users change this value? Could two records ever legitimately share this value? These questions shape the final choice.

6.4

Natural Keys and Surrogate Keys

One of the most important design decisions in database work is whether to use a natural key or a surrogate key as the primary key. Both have legitimate uses. Both have significant tradeoffs. Understanding the difference, and the reasoning behind each choice, is essential for designing reliable databases.

Natural Keys

A natural key is a primary key that comes from real-world data about the thing being described. It is a value that the record itself carries, independent of the database.

Examples of natural keys include a national identification number for a citizen record, an ISBN for a book record, an email address for a user account, a vehicle identification number (VIN) for a vehicle record, and a course code such as CS-301 for a course record.

Natural keys have a meaningful advantage: they already exist in the real world, so systems and people can use them without needing to translate between a database identifier and a real identifier. When a librarian receives a book with an ISBN on its cover, they can enter that ISBN directly into the catalog. When a student gives their student ID number at a service desk, the system can look them up immediately.

However, natural keys carry risks.

  • They Can ChangeAn email address that serves as a primary key creates a problem the moment a student changes their email. Every record in every table that refers to that email must be updated, or the database becomes inconsistent. This is called an update anomaly.
  • They Can Be SharedA course code that seems unique might be reused across different departments or different terms if the system evolves in ways the designer did not anticipate.
  • They Can Be MissingNot every record arrives with a guaranteed unique real-world identifier. If some records lack the natural key value, the not-null requirement cannot be met.
  • They Encode AssumptionsA natural key implicitly assumes that the external system assigning those values is consistent, reliable, and will not change its format. If a government changes its national ID format, or if a book publisher reuses ISBNs, the natural key breaks.
Surrogate Keys

A surrogate key is a primary key invented by the database itself, separate from any real-world meaning. It is assigned automatically when a record is inserted and carries no intrinsic meaning beyond identification.

In most database systems, surrogate keys are integers that increment automatically. The first record inserted gets id = 1, the second gets id = 2, and so on. The database handles this assignment without requiring the user to provide a value.

Table 6.3 — A surrogate key. The values 1, 2, and 3 identify records within the database but carry no meaning outside it.
student_id first_name last_name
1AmaraOsei
2FelixMendez
3YukiTakahashi

Surrogate keys avoid most of the risks that affect natural keys. They never change, because there is no external reality that could change them. They are never missing, because the database generates them automatically. They are never shared, because the database ensures uniqueness. They do not encode assumptions about the external world.

The tradeoff is that surrogate keys have no meaning on their own. The number 2 does not tell you anything about Felix Mendez. This is why surrogate keys are typically used together with uniqueness constraints on meaningful columns such as email addresses or national IDs.

How to Choose

In practice, most modern database systems use surrogate keys as primary keys and then protect meaningful unique values with separate uniqueness constraints. This design is safe, flexible, and avoids the common pitfalls of natural keys.

However, there are cases where a natural key is clearly the right choice. If the thing being described already has a stable, guaranteed-unique, never-null, never-changing external identifier, using it as the primary key may be both sensible and efficient. ISBN numbers for books, course codes that are assigned through a rigorous external process, and vehicle identification numbers are examples where natural keys often work well.

The general rule is: prefer surrogate keys unless you have strong reasons to trust that the natural key is stable, unique, and guaranteed to be present for every record. When in doubt, use a surrogate key and protect the meaningful identifiers with uniqueness constraints.

6.5

Foreign Keys

A foreign key is a column in one table that holds the primary key value of a related row in another table. Foreign keys are how relational databases create and enforce relationships between tables.

Chapter 5 introduced the idea that a library loan connects a patron to a book. In a relational database, this connection is implemented through a foreign key. The loan table contains a column that holds the patron's primary key value, and another column that holds the book's primary key value. When the system needs to retrieve the full details of a loan, it uses those foreign key values to look up the correct patron record and the correct book record.

A Simple Example

Consider two tables. The first stores student records, and the second stores course enrollment records.

Table 6.4 — The students table. student_id is the primary key.
student_id first_name last_name email
1001AmaraOseiamara.osei@mgu.edu
1002FelixMendezfelix.mendez@mgu.edu
1003YukiTakahashiyuki.takahashi@mgu.edu
Table 6.5 — The enrollments table. student_id here is a foreign key, pointing to the students table.
enrollment_id student_id course_code term
50011001CS-301Fall 2024
50021002CS-301Fall 2024
50031001MATH-205Fall 2024
50041003CS-301Spring 2025

In the enrollments table, enrollment_id is the primary key. The student_id column is a foreign key. Each value in student_id refers to a row in the students table. Enrollment 5001 belongs to student 1001, Amara Osei. Enrollment 5003 also belongs to student 1001. Student 1003, Yuki Takahashi, appears in enrollment 5004.

The foreign key creates the relationship. Without it, the enrollments table would just be a list of numbers and codes with no connection to the students who enrolled.

Declaring a Foreign Key in SQL
SQL — Declaring a Foreign Key
CREATE TABLE enrollments (
    enrollment_id INTEGER PRIMARY KEY,
    student_id    INTEGER NOT NULL,
    course_code   TEXT    NOT NULL,
    term          TEXT    NOT NULL,
    FOREIGN KEY (student_id) REFERENCES students(student_id)
);

The FOREIGN KEY declaration tells the database that the values in the student_id column of enrollments must match an existing value in the student_id column of students. The database will reject any attempt to insert an enrollment that references a student_id that does not exist in the students table.

What the Foreign Key Protects

The foreign key prevents several kinds of errors that would otherwise silently corrupt the database.

Inserting an enrollment for a nonexistent student. If the students table has no record with student_id = 9999, then inserting an enrollment with student_id = 9999 would create a reference to nothing. The foreign key constraint catches this and rejects the insertion.

Deleting a student who still has enrollments. If a student record is deleted from the students table while enrollment records still reference that student, the enrollment records become orphaned. They refer to a student who no longer exists. The foreign key constraint prevents this deletion unless the related enrollment records are deleted first or updated to point elsewhere.

These protections together are called referential integrity, discussed in detail in section 6.8.

6.6

Uniqueness Constraints

A uniqueness constraint is a rule that prevents duplicate values in a column or combination of columns. It is enforced by the database automatically, just like the primary key's uniqueness requirement.

While the primary key already enforces uniqueness on its column, there are often other columns in a table that should also be unique. Email addresses, national identification numbers, usernames, and course codes are examples of values that should not be duplicated even if they are not the primary key.

Why Uniqueness Constraints Exist Separately from Primary Keys

When a table uses a surrogate key as its primary key, the primary key itself carries no real-world meaning. Uniqueness of the surrogate key does not guarantee uniqueness of meaningful data. Two students could theoretically be inserted with the same email address, and the surrogate key would not catch this because both records would have different primary key values.

A uniqueness constraint on the email column closes this gap:

SQL — Uniqueness Constraints on Non-Key Columns
CREATE TABLE students (
    student_id    INTEGER PRIMARY KEY,
    first_name    TEXT    NOT NULL,
    last_name     TEXT    NOT NULL,
    email         TEXT    UNIQUE,
    national_id   TEXT    UNIQUE
);

Here, email and national_id each have UNIQUE constraints. The database will reject any insertion or update that would create a duplicate in either column, even though student_id is the primary key.

Composite Uniqueness Constraints

Sometimes, a uniqueness constraint applies to a combination of columns rather than a single column. A student might be allowed to enroll in the same course in different terms, but they should not be allowed to enroll in the same course in the same term twice.

Table 6.6 — Both rows are valid because the terms differ. A composite uniqueness constraint on (student_id, course_code, term) would catch an attempt to insert a third row for (1001, CS-301, Fall 2024).
student_id course_code term
1001CS-301Fall 2024
1001CS-301Spring 2025
SQL — Composite Uniqueness Constraint
CREATE TABLE enrollments (
    enrollment_id INTEGER PRIMARY KEY,
    student_id    INTEGER NOT NULL,
    course_code   TEXT    NOT NULL,
    term          TEXT    NOT NULL,
    UNIQUE (student_id, course_code, term),
    FOREIGN KEY (student_id) REFERENCES students(student_id)
);
NULL Values and Uniqueness Constraints

Most database systems treat NULL as an unknown value, not as a specific value. Because of this, many systems allow multiple rows to have NULL in a column that has a uniqueness constraint, reasoning that two unknowns cannot be said to be equal. If a column that has a uniqueness constraint must always have a value, it should also be declared NOT NULL.

6.7

Required Fields

A required field is a column that must have a value for every record. The database enforces this through a NOT NULL constraint, which prevents any row from being inserted or updated without a value in that column.

Not all columns should be required. Some columns represent information that may genuinely be unknown or inapplicable. A middle name is a good example: some people have one, some do not, and requiring a value would either force fabrication or make valid records impossible to insert.

But many columns represent information that is essential for the record to be meaningful. A student record without a last name is incomplete. A book record without a title cannot be found or described. A loan record without a patron is completely meaningless.

SQL — Required and Optional Fields
CREATE TABLE books (
    book_id          INTEGER PRIMARY KEY,
    title            TEXT    NOT NULL,        -- required
    publication_year INTEGER,               -- optional
    isbn             TEXT    UNIQUE         -- unique if provided, but not required
);

In this table, title is required. Every book must have a title. publication_year is optional, because the year might not be known for all books. isbn has a uniqueness constraint but no NOT NULL constraint, meaning an ISBN is not required but must be unique if provided.

Thinking Through Which Fields Are Required

The question of which fields to require is a design decision that should be made thoughtfully, not automatically. Making every field required protects data completeness but makes data entry inflexible and may prevent legitimate records from being inserted. Making every field optional creates gaps in data that undermine reporting and querying.

A useful approach is to ask: what is the minimum information that makes a record meaningful and usable? That minimum set should be required. Everything else can be optional, with documentation explaining when it is expected to be filled in.

For a student record, the minimum meaningful information might include the student's full name and their enrollment term. Email and phone number may be highly desirable but not strictly required if a record can exist without them. Deciding at the database level whether to require, recommend, or allow that gap is an important design choice with real operational consequences.

6.8

Referential Integrity

Referential integrity is the property of a database that guarantees all foreign key values point to existing records. When referential integrity is maintained, every relationship between tables is valid: every enrollment refers to a real student, every loan refers to a real book, every submission refers to a real assignment.

When referential integrity is violated, a database contains orphaned records, which are records that refer to something that no longer exists. Orphaned records corrupt the meaning of the data. An enrollment that refers to a deleted student cannot tell you anything meaningful about who enrolled. Reports that depend on joins between tables will produce wrong results or silently exclude affected records.

What Can Violate Referential Integrity

Two operations can violate referential integrity if the database does not enforce foreign key constraints.

Inserting a child record with a nonexistent parent. If an enrollment record references a student that does not exist in the students table, the enrollment is orphaned at the moment of insertion.

Deleting a parent record that has child records. If a student record is deleted while enrollment records still refer to that student, those enrollment records become orphaned at the moment of deletion.

A database that enforces foreign key constraints prevents both of these automatically.

Handling Deletions with Foreign Keys

When a parent record is deleted, the database must decide what to do with child records that reference it. Database systems offer several options, often called referential actions.

Restrict / No Action

The deletion is rejected if child records exist. This is the safest default. The designer must explicitly remove or reassign child records before the parent can be deleted.

Cascade

When the parent is deleted, all child records that reference it are also deleted automatically. Appropriate when child records have no meaning without the parent.

Set Null

When the parent is deleted, the foreign key column in child records is set to NULL. Appropriate when the relationship is optional and the child can exist independently.

Set Default

When the parent is deleted, the foreign key column is set to a predetermined default value. Less common but available in some systems for specific use cases.

The right referential action depends on what the data means. Choosing cascade when restrict is appropriate can silently destroy data. Choosing restrict when cascade would be appropriate forces tedious manual cleanup. These decisions require careful reasoning about the real-world consequences of deletion.

Referential Integrity in SQLite

SQLite in its default configuration does not enforce foreign key constraints unless explicitly instructed to do so. A designer who knows this will enable enforcement deliberately at the start of each session:

SQL — Enabling Foreign Key Enforcement in SQLite
PRAGMA foreign_keys = ON;
Important for SQLite Users

Without PRAGMA foreign_keys = ON, SQLite accepts foreign key declarations in CREATE TABLE statements but does not enforce them. Records can be inserted that reference nonexistent parents, and parent records can be deleted while children still exist. Always enable this setting when working with SQLite during this course.

6.9

Rules That Protect Data

Primary keys, foreign keys, uniqueness constraints, and required fields are all specific kinds of database constraints: rules that the database enforces automatically to protect the accuracy and consistency of the data it stores. Beyond these, database systems support additional constraint types that can protect data in more specific ways.

Check Constraints

A check constraint is a custom rule that specifies which values are valid for a column. Check constraints can enforce ranges, allowed lists of values, relationships between values, and other business rules.

SQL — Check Constraints
CREATE TABLE students (
    student_id    INTEGER PRIMARY KEY,
    first_name    TEXT    NOT NULL,
    last_name     TEXT    NOT NULL,
    year_enrolled INTEGER CHECK (year_enrolled >= 1900 AND year_enrolled <= 2100)
);

-- A rating column that only accepts values between 1 and 5
rating  INTEGER CHECK (rating >= 1 AND rating <= 5)

-- A status column that only accepts specific values
status  TEXT    CHECK (status IN ('active', 'inactive', 'pending'))

-- A price that must be greater than zero
price   REAL    CHECK (price > 0)

Check constraints protect data quality at the database level, which is more reliable than relying on application code alone. Application code can have bugs, can be bypassed, or can be updated in ways that accidentally remove validation. Database constraints are always enforced, regardless of which application or query tool is used to insert or update data.

Default Values

A default value is a value that the database automatically assigns to a column when an insertion does not provide a value for that column. Default values work together with constraints to manage data completeness.

SQL — Default Values
CREATE TABLE resources (
    resource_id   INTEGER PRIMARY KEY,
    title         TEXT    NOT NULL,
    available     INTEGER DEFAULT 1,
    date_added    TEXT    DEFAULT (date('now'))
);

When a new resource is added without specifying available, the database automatically sets it to 1, meaning available. When date_added is not provided, the database automatically sets it to the current date. Default values reduce the burden on data entry while ensuring that columns have sensible values even when information is not explicitly provided.

Constraints as Documentation

Constraints serve a purpose beyond enforcement. They also document the rules of the database. A designer who reads a table definition with clear constraints immediately understands what values are valid, what relationships exist, and what is required. A table with no constraints provides no such documentation.

When constraints are missing, the rules of the database must be enforced in application code, where they may be inconsistent, incomplete, or invisible to future developers. Encoding the rules in the database itself makes the system more transparent and more reliable over time.

6.10

Choosing Good Keys

Choosing a primary key is one of the most durable decisions in database design. It shapes how the system works, how queries are written, how relationships are built, and how the database evolves over time. A poorly chosen primary key can cause problems that are expensive to fix later, because changing the primary key of a heavily used table requires updating every foreign key that references it, every query that uses it, and every application that passes it around.

The Core Requirements

A primary key must be unique (no two rows share the same value), not null (every row has a value), and stable (the value should never change after the row is inserted). Any candidate key that cannot meet all three requirements should not be chosen as the primary key.

Prefer Surrogate Keys for General Use

For most tables, a surrogate key, typically an auto-incrementing integer, is the best choice. It is always unique, always present, and never needs to change because it has no real-world meaning that could become outdated. The pattern looks like this:

SQL — Recommended Pattern: Surrogate Key with Protected Natural Identifiers
CREATE TABLE patrons (
    patron_id   INTEGER PRIMARY KEY,             -- surrogate key
    name        TEXT    NOT NULL,
    email       TEXT    UNIQUE NOT NULL,           -- natural identifier, protected
    membership  TEXT    CHECK (membership IN ('standard', 'faculty', 'student'))
);
Composite Primary Keys for Junction Tables

A composite primary key uses two or more columns together as the identifier. Composite primary keys can be appropriate in some cases, particularly for junction tables in many-to-many relationships:

SQL — Composite Primary Key in a Junction Table
CREATE TABLE course_resources (
    course_id    INTEGER NOT NULL,
    resource_id  INTEGER NOT NULL,
    PRIMARY KEY (course_id, resource_id),
    FOREIGN KEY (course_id)   REFERENCES courses(course_id),
    FOREIGN KEY (resource_id) REFERENCES resources(resource_id)
);
Checklist: Before Choosing a Natural Key

Before choosing a natural key as the primary key, ask these questions. If the answer to any of them is yes, a natural key is risky and a surrogate key should be preferred instead.

  • Could This Value Change?Could this value ever be updated or corrected after it is first assigned? Email addresses change, names are corrected, and external identifier formats can be revised.
  • Could It Repeat?Could two different records legitimately have the same value in the future, even if no duplicates exist in today's data?
  • Could It Be Missing?Is this value always present for every record, or could some records legitimately lack it?
  • Does It Depend on an External System?Does this value depend on an external system that could change its format, reuse values, or become unavailable?
Document Key Decisions

A design document or schema comment that explains why a particular key was chosen is valuable. Future designers who inherit the database need to understand why the key exists, what it represents, and why alternatives were rejected. Without this documentation, later changes may inadvertently break assumptions that were never written down.

Chapter Summary
  • Primary keys are the fundamental identifiers of rows in a table. Every table should have one, and the primary key must be unique and not null. Primary keys are the basis for all relationships between tables.
  • Candidate keys are all columns or combinations of columns that could theoretically serve as primary keys. Recognizing them helps designers think clearly about what is truly unique and which option is safest to promote.
  • Natural keys come from real-world data and carry meaning outside the database. They are appropriate when the real-world identifier is stable, guaranteed unique, always present, and unlikely to change. Surrogate keys are invented by the database and carry no external meaning. They are safer in most situations.
  • Foreign keys are the mechanism that creates and enforces relationships between tables. A foreign key in one table holds the primary key value of a related row in another table.
  • Uniqueness constraints protect columns that should have unique values even if they are not primary keys. They can apply to individual columns or to combinations of columns.
  • Required fields enforce that specific columns always have values. They prevent incomplete records from being inserted.
  • Referential integrity is the property that all foreign key values point to existing records. Without it, databases accumulate orphaned records that corrupt the meaning of queries and reports.
  • Check constraints and default values extend the database's ability to protect data quality automatically and to document the rules of the system.
  • Good key choices require thinking about stability, uniqueness, presence, and the tradeoffs between natural and surrogate approaches. The most common reliable pattern is a surrogate primary key with uniqueness constraints protecting meaningful natural identifiers.
Key Terms
Primary Key

A column or combination of columns that uniquely identifies every row in a table. Every row must have a value, and no two rows may share the same value.

Candidate Key

Any column or combination of columns that could serve as a primary key, being both unique and not null across all rows. A table may have multiple candidate keys.

Natural Key

A primary key that comes from real-world data about the thing being described, such as a national ID, an ISBN, or an email address.

Surrogate Key

A primary key invented by the database itself, typically an auto-incrementing integer, with no real-world meaning beyond identification.

Foreign Key

A column in one table that holds the primary key value of a related row in another table. Foreign keys implement and enforce relationships between tables.

Uniqueness Constraint

A database rule that prevents any two rows from sharing the same value in a specified column or combination of columns.

NOT NULL Constraint

A database rule that prevents a column from containing a null value. Required fields use this constraint to ensure that every record includes a value.

Referential Integrity

The property of a database that guarantees all foreign key values point to existing records in the referenced table. When maintained, no orphaned records exist.

Orphaned Record

A row whose foreign key value refers to a record that no longer exists or never existed. Orphaned records corrupt the meaning of relationships and queries.

Check Constraint

A custom rule that specifies which values are valid for a column, such as a range of numbers, a list of allowed values, or a logical condition.

Default Value

A value that the database assigns automatically to a column when an insertion does not provide one.

Referential Action

A rule that specifies what happens to child records when a parent record is deleted or updated. Common options include restrict, cascade, set null, and set default.

Alternate Key

A candidate key that was not chosen as the primary key. Alternate keys should be protected with uniqueness constraints.

Composite Key

A primary key or uniqueness constraint that uses two or more columns together to identify a row.

Common Misconceptions
Misconception 1
A unique value in today's data means a column is a good primary key.

Uniqueness in current data does not guarantee future uniqueness. A column that happens to contain no duplicates today may receive duplicates tomorrow. The question is whether the column is guaranteed to be unique by the nature of the data, not whether it happens to be unique right now. Names, addresses, and even email addresses may appear unique in a small dataset but become duplicated as a system grows.

Misconception 2
Foreign keys automatically delete or update related records.

A foreign key constraint by default prevents changes that would violate referential integrity. It does not automatically cascade deletions or updates unless the table definition explicitly specifies a cascade referential action. The default behavior in most systems is to reject the change, not to propagate it. Designers must explicitly choose cascade behavior and should do so carefully.

Misconception 3
A surrogate key eliminates the need to think about natural identifiers.

A surrogate key handles the identification problem within the database, but it does not replace the need to protect meaningful external identifiers. If two student records can be inserted with the same email address because no uniqueness constraint exists, the database contains a data quality problem even though both records have unique surrogate keys. Surrogate keys and uniqueness constraints on natural identifiers work together.

Misconception 4
Primary keys must be integers.

A primary key can be any data type that supports uniqueness and not-null enforcement. Text columns, UUID columns, and composite columns can all serve as primary keys. Integer surrogate keys are the most common choice in practice because they are compact, efficient for indexing, and easy to generate automatically, but they are not the only option.

Misconception 5
Constraints slow down the database, so they should be avoided.

Constraints do add some overhead to insertions and updates, but this overhead is small compared to the cost of discovering and correcting data quality problems caused by missing constraints. A database that allows bad data to be inserted is more expensive to operate over time than one that rejects bad data immediately. The small performance cost of constraints is almost always worth paying.
