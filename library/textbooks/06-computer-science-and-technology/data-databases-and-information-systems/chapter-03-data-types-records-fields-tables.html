Data Types, Records, Fields, and Tables

Every database is built from the same small set of building blocks: fields that describe one thing about a subject, records that collect all those descriptions together, tables that group records of the same kind, and data types that tell the system what kind of value each field holds. This chapter introduces those building blocks carefully, because everything that follows, from SQL to database design to information systems, rests on them.

A database stores information about things. Those things might be students, books, products, appointments, orders, animals, countries, songs, or employees. Whatever the subject, every database uses the same fundamental vocabulary to organize what it knows: fields, records, tables, and data types.

Understanding these building blocks is not optional background knowledge. It is the foundation for everything else in this textbook. A student who cannot read a table, identify a field's data type, or explain why a missing value is different from a zero will struggle with database design, SQL, data quality, and information systems. A student who builds this foundation well will find that later chapters feel natural and connected.

By the end of this chapter, you should be able to read any table and describe its fields, records, and data types clearly. You should recognize text, numbers, dates, Boolean values, categories, and identifiers when you encounter them. You should understand why missing values matter, and you should begin to see tables not as grids of cells, but as structured descriptions of real things in the world.

How This Chapter Connects to What Follows

Data modeling in Part II describes entities and their attributes, which become tables and fields. SQL in Part III queries tables by column name and data type. Data quality problems in Chapter 4 are often data-type problems in disguise. The vocabulary you build here runs through every chapter that follows.

3.1

Fields and Records

For each kind of thing a database stores, it records a consistent set of facts. Every student record might include a name, a date of birth, an enrollment number, an email address, and a department. Every book record might include a title, an author, a publication year, and a subject category. The specific facts a database stores about each thing are called fields.

A field is one piece of information about a subject. It corresponds to a specific characteristic or attribute that the database tracks for every record of that type. A field has a name, which describes what it stores, and a data type, which describes what kind of value it holds.

A record is the complete collection of fields for one specific subject. If the database is tracking students, then one student's complete entry — name, date of birth, enrollment number, email address, department — is one record. Each record describes exactly one instance of whatever the database is tracking.

These two concepts work together. Fields define the categories of information a database collects. Records fill in the actual values for a specific subject.

Reading a Record

Consider a simple example. A school database tracks information about students. For each student, it records five facts:

  • Enrollment number
  • Full name
  • Date of birth
  • Email address
  • Department

These five facts are the fields. Now consider one specific student:

  • 10047
  • Priya Nair
  • 2004-09-15
  • priya.nair@mgu.edu
  • Computer Science

This set of five values, filled in for one student, is one record. The record answers the same five questions the fields ask, but it answers them specifically for Priya Nair.

Fields Are Not Values

It is worth being precise about this distinction early, because confusion between a field and its value causes problems in database design.

A field is a named category of information. "Full name" is a field. "Department" is a field. These are abstract descriptions of what the system tracks. A value is the actual content stored in a field for a specific record. "Priya Nair" is a value. "Computer Science" is a value. These are specific facts about a specific person.

A database defines its fields once and then stores many values for them, one set per record.

3.2

Tables, Rows, and Columns

Fields and records do not float independently. They are organized into a table, which is the primary structure databases use to store information.

A table has a name that describes what kind of thing it stores. Common table names include Students, Books, Products, Orders, Employees, and Courses. The name should clearly identify the subject of every record in the table.

Inside the table, fields appear as columns. Each column has the name of the field it represents. Every cell in a column stores a value for that field. Records appear as rows. Each row represents one specific subject. Every cell in a row stores the value for one field. The result is a rectangular grid where columns represent fields and rows represent records.

A Sample Table

Here is a small students table. It has five columns, one for each field, and four rows, one for each student.

Table 3.1 — The students table: each row is one student, each column is one field.
EnrollmentNumber FullName DateOfBirth Email Department
10047 Priya Nair 2004-09-15 priya.nair@mgu.edu Computer Science
10048 David Osei 2003-11-02 david.osei@mgu.edu Mathematics
10049 Fatima Al-Rashid 2005-01-28 fatima.alrashid@mgu.edu Library Science
10050 Ravi Krishnan 2004-06-10 ravi.krishnan@mgu.edu Computer Science

Reading across row 1, the record tells us: enrollment number 10047, full name Priya Nair, date of birth September 15, 2004, email priya.nair@mgu.edu, department Computer Science. Reading down the Department column, the values tell us: Computer Science, Mathematics, Library Science, Computer Science. This column contains one department value for every student.

Tables Are Not Spreadsheets

Students who have used spreadsheet software such as Microsoft Excel or Google Sheets sometimes confuse a spreadsheet with a database table. They look similar at first glance: both have rows and columns. But they work differently.

A spreadsheet is a flexible grid where any cell can contain any value, a formula, a color, a merged region, or mixed content. A database table is a strictly structured object. Every row follows exactly the same column structure. Every column stores one specific kind of value. Rules and constraints govern what values are allowed, and the database system enforces these rules automatically.

These differences matter for large, shared, and important data. Chapter 7 returns to this comparison in detail, and Chapter 23 covers the practical relationship between spreadsheets and databases.

3.3

Data Types

Every field in a database has a data type. The data type tells the database system what kind of value the field holds and what operations are valid on it.

Data types matter for several reasons. They protect against mistakes: a field defined as a date cannot accidentally store the text "yesterday" or the number -42. They enable correct operations: a database can calculate the average of a numeric field, sort dates chronologically, or search text for a specific word. And they affect storage and performance, because the system stores an integer differently than it stores a long text passage.

The major data types students will encounter in introductory database work are shown below. Each is covered in its own section.

Text
Names, addresses, email, codes

Character sequences where arithmetic is meaningless.

Numeric
Counts, prices, scores, measurements

Quantities where addition, averaging, and comparison make sense.

Date & Time
2024-03-15, 14:30:00

Calendar and clock values with correct sorting and arithmetic.

Boolean
TRUE / FALSE   1 / 0   Yes / No

Exactly two states: true or false, yes or no.

Category
Active, Pending, Completed…

Values from a defined, limited set of allowed choices.

Identifier

A unique label that distinguishes each record in the table.

System Differences

Different database systems use different names for their data types. SQLite uses TEXT, INTEGER, REAL, and BLOB. PostgreSQL adds BOOLEAN, DATE, TIMESTAMP, and many more. MySQL and SQL Server have their own naming conventions. The concepts in this chapter apply universally, even when the exact type names differ across systems.

3.4

Text Data

Text data, sometimes called string data or character data, holds sequences of characters: letters, digits, spaces, punctuation, and symbols. Text fields store values that should be treated as words and phrases rather than as numbers to be calculated.

Common text fields include:

  • Names of people, places, and organizations
  • Addresses, descriptions, and notes
  • Email addresses and URLs
  • Product names and titles
  • Subject codes that mix letters and digits
Fixed-Length and Variable-Length Text

Most database systems distinguish between two kinds of text fields. Fixed-length text fields always store exactly the same number of characters, padding shorter values with spaces. These are used when every value will be the same length, such as two-letter country codes (US, GB, IN). Variable-length text fields store however many characters the actual value needs, up to a defined maximum. These are more common because most real text values vary in length. In SQLite, the standard text type is simply TEXT, accepting variable-length character sequences.

Text Fields Are Not Calculated

Even if a text field contains digit characters, the database does not treat it as a number. The value "42" stored in a text field is not the number 42. It is the two-character sequence consisting of a four and a two. Student enrollment numbers, phone numbers, postal codes, and product codes often contain only digits, but they should be stored as text because no arithmetic should ever be performed on them.

The question to ask is: will the database ever need to perform arithmetic on this field? If the answer is no, text is probably the right choice, even if the values look like numbers.

Text Case and Matching

Database systems differ in whether they treat text matching as case-sensitive. In some systems, "Priya" and "priya" match. In others, they do not. Students should be aware that text searches behave differently depending on the system and its configuration. Chapter 11 covers text filtering in SQL in detail.

3.5

Numeric Data

Numeric data holds values that represent quantities, amounts, measurements, or counts. The defining feature of numeric data is that arithmetic operations, such as addition, subtraction, multiplication, division, average, and sum, make sense on it.

Common numeric fields include:

  • Age, quantity, count, and score
  • Price, salary, and total
  • Measurement values such as height, weight, and temperature
  • Percentage values
Integers and Decimals

Most database systems distinguish between integers (whole numbers without a fractional part) and decimal or floating-point numbers (numbers with a fractional part). An integer field would hold values like 0, 1, 42, -5, or 1000. A decimal field would hold values like 3.14, 0.75, 199.99, or -12.5.

Choosing the right type matters. A field storing the number of students enrolled in a course should be an integer, because half a student makes no sense. A field storing a product price should be a decimal, because prices commonly include cents or fractional currency units.

In SQLite, numeric types are handled flexibly using INTEGER and REAL. Other systems such as PostgreSQL and MySQL offer more specific types, including SMALLINT, BIGINT, NUMERIC, and FLOAT, for different precision needs.

When Numbers Are Not Numeric

Just as text fields can contain digit characters, numeric fields should only hold values where arithmetic makes sense. Enrollment numbers, phone numbers, and identification codes should not be stored as numeric types even if they contain only digits. The test is always: would it ever be meaningful to add these values together, or average them, or find their minimum and maximum? If not, they belong in a text field.

A Note on Numeric Precision

Floating-point numbers are stored using binary approximations, which can introduce tiny rounding errors. For financial data where precision matters, database systems often provide exact decimal types such as DECIMAL or NUMERIC with a specified number of digits. This becomes important in financial applications, but the key point now is that not all decimal storage is identical.

3.6

Dates and Times

Date and time data holds values representing moments, durations, and calendar information. Dates and times are their own data type because they have special properties that text and numbers cannot handle automatically.

Common date and time fields include:

  • Date of birth and enrollment date
  • Order date, appointment date, and submission deadline
  • Publication date and due date
  • Timestamps recording when a record was created or last modified
  • Start time and end time of events or sessions
Why Dates Need Their Own Type

Storing dates as text, for example writing "September 15, 2004", creates serious problems. Text does not sort chronologically. The string "September 15, 2004" would sort before "January 2, 2005" alphabetically because S comes before J, but chronologically it should sort after it. Storing dates as plain integers also prevents the system from easily calculating the number of days between two dates, identifying the day of the week, or determining how old someone is. A proper date type stores the date in a way the system understands, enabling all of these operations.

Standard Date Format: ISO 8601

Database systems almost universally use the ISO 8601 standard for date storage: YYYY-MM-DD. This means a four-digit year, then a two-digit month, then a two-digit day, separated by hyphens.

Table 3.2 — ISO 8601 date format eliminates regional ambiguity and sorts correctly.
Plain-language date ISO 8601 format Sorts correctly?
September 15, 20042004-09-15Yes
January 2, 20052005-01-02Yes
November 30, 20232023-11-30Yes
March 8, 20242024-03-08Yes

This format is unambiguous across different regional conventions. Students from countries that write dates as DD/MM/YYYY and students from countries that write dates as MM/DD/YYYY both read 2004-09-15 the same way.

Date, Time, and Datetime

Most database systems offer separate types. A date field stores only a calendar date: year, month, and day. A time field stores only a time of day: hours, minutes, and sometimes seconds. A datetime or timestamp field stores both a date and a time together. Timestamps are commonly used to record exactly when an event happened, such as when a record was created or when a transaction occurred.

In SQLite, dates are stored as TEXT in ISO 8601 format, as REAL numbers representing Julian day values, or as INTEGER Unix timestamps. SQLite provides built-in functions that handle these correctly when the storage format is consistent.

3.7

Boolean Data

Boolean data holds exactly one of two values: true or false. Boolean fields represent yes-or-no questions, conditions that either apply or do not apply, and flags that mark whether something has happened.

Common Boolean fields include:

  • IsEnrolled — is this student currently enrolled?
  • IsActive — is this account currently active?
  • IsAvailable — is this item available for loan?
  • HasPaid — has this invoice been paid?
  • IsPublished — has this resource been published?
  • IsApproved — has this request been approved?
Boolean Representation

Not every database system uses a native Boolean type. SQLite stores Boolean values as integers: 0 represents false and 1 represents true. PostgreSQL has a native BOOLEAN type. MySQL uses TINYINT(1). Regardless of the underlying storage, the meaning is always the same: the field answers one yes-or-no question about each record. When reading data, students may encounter Boolean values stored as 0 and 1, TRUE and FALSE, Yes and No, Y and N, or T and F.

Boolean Versus Category

A Boolean field works correctly when there are exactly two possibilities and they are truly opposite: enrolled or not enrolled, active or not active, paid or not paid. When there are three or more possibilities, a Boolean is the wrong type. A field representing a student's enrollment status might need to store Active, On Leave, Withdrawn, and Graduated. Four states require a category type, not a Boolean.

3.8

Categories and Codes

Category data holds values from a defined, limited set of possible choices. The set of allowed values is finite and known in advance. Category fields represent classifications, statuses, types, and groups.

Common category fields include:

  • Department (Computer Science, Mathematics, Library Science, Physics)
  • Status (Active, Inactive, Pending, Completed)
  • Priority (Low, Medium, High, Critical)
  • Role (Student, Instructor, Administrator, Librarian)
  • Loan status (Available, Checked Out, Reserved, Damaged)
Why Categories Need Care

Category fields look simple but create common database problems when handled carelessly.

Inconsistency

If one record stores "Computer Science" and another stores "computer science" and another stores "CS", queries counting students by department will produce fragmented, misleading results.

Invalid Values

If any text can be entered in a category field, someone might type "Comuter Science" or "Mathematicss" by accident. The database should restrict category fields to valid values only.

Changes Over Time

Categories sometimes change. A department might rename itself. A status might gain new options. These changes must be managed so old records remain valid.

Codes vs. Full Names

Short codes like "CS" save space and enforce consistency, but they require documentation. Without it, codes become cryptic and hard to maintain.

Storing Categories: References and Codes

One approach is to store the full category name as text and use a constraint or validation rule to limit values to the approved list. A better approach, which Chapters 5 and 6 explore in depth, is to store categories in their own separate table and use a foreign key reference in the main table. This way, the list of valid categories is maintained in one place, and any change to a category name automatically applies to all records that reference it. For now, students should understand that category fields are a distinct kind of data: not free-form text, not numbers, but values from a controlled vocabulary.

3.9

Missing, Unknown, and Not Applicable Values

Real data is rarely complete. People do not always provide every piece of information. Some questions do not apply to every record. Some information is unknown at the time of entry. Databases must have a principled way to handle the absence of a value.

Most database systems represent a missing or absent value with a special marker called NULL. NULL does not mean zero. It does not mean an empty text string. It means: this field has no value for this record. The value is unknown, unavailable, or not applicable.

Three Kinds of Absence

It helps to distinguish between three reasons a value might be absent.

Unknown

The value exists but is not known. A student's date of birth might be unknown if they did not provide it. The date certainly exists, but the database does not have it.

Not Yet Recorded

The value will eventually exist but has not been entered yet. A student may have an email address, but it has not been added to their record.

Not Applicable

The question does not apply to this record at all. A book does not have a volume number. This is different from the volume number being unknown.

NULL in Practice

When a field contains NULL, the database system recognizes that it cannot compare the field to a specific value in the usual way. Testing whether a NULL value equals anything produces an undefined result, not true or false. This has important consequences for queries. Chapter 11 covers how SQL handles NULL values in filtering. For now, the important point is that NULL is not a value, it is the absence of a value, and it must be handled carefully to avoid misleading query results.

Example: A Table with Missing Values

Consider the students table again, now with a phone number field added. Two students have not provided a phone number.

Table 3.3 — NULL in the PhoneNumber field means no value is on record, not that the number is zero.
EnrollmentNumber FullName DateOfBirth Email Department PhoneNumber
10047 Priya Nair 2004-09-15 priya.nair@mgu.edu Computer Science 555-0147
10048 David Osei 2003-11-02 david.osei@mgu.edu Mathematics NULL
10049 Fatima Al-Rashid 2005-01-28 fatima.alrashid@mgu.edu Library Science 555-0219
10050 Ravi Krishnan 2004-06-10 ravi.krishnan@mgu.edu Computer Science NULL

David Osei and Ravi Krishnan have NULL in the PhoneNumber field. This does not mean their phone number is zero, or empty, or invalid. It means the database does not have a phone number on record for them. The NULL marker communicates the absence cleanly.

Required Fields and Optional Fields

Database designers distinguish between fields that must always have a value and fields that may be NULL. A required field is one where NULL is not permitted. An optional field is one where NULL is allowed. A student's enrollment number and full name might be required fields. No student record should exist without them. But a phone number might be optional. Students may or may not provide one. Chapter 6 covers how databases enforce required fields through constraints.

3.10

Identifiers

Every record in a well-designed database needs a way to be uniquely distinguished from every other record. This is the purpose of an identifier field, more formally called a primary key. An identifier is a field, or sometimes a combination of fields, that uniquely identifies each record in a table. No two records in the same table can share the same identifier value. Every record must have one.

Why Identifiers Are Necessary

Consider a students table without a unique identifier. Two students named David Osei enroll at the same university. Without an identifier, the database cannot tell these two records apart. Any query that searches for David Osei will find both records. Any update to David Osei's address might update the wrong one. Any link from another table pointing to David Osei cannot reliably point to the right student. A unique identifier solves this problem by giving every record a distinct label that never repeats and never changes.

Natural Keys

A natural key is an identifier drawn from real-world information that already uniquely identifies the subject. Examples include government-issued identification numbers, passport numbers, ISBN numbers for books, and vehicle identification numbers. Natural keys are attractive because they carry real-world meaning. But they have weaknesses. A natural key might change if the real-world value changes. A passport number changes when a passport expires. A government ID might be corrected if recorded incorrectly. Any change to a natural key that is referenced by other tables creates significant problems. Natural keys can also fail to be truly unique due to errors in external systems.

Surrogate Keys

A surrogate key is an identifier the database generates itself, with no real-world meaning outside the database. The most common kind is an auto-incrementing integer: 1, 2, 3, 4, and so on, assigned automatically each time a new record is added.

Never Changes

Nothing in the real world affects a surrogate key. It is stable for the life of the record.

Always Unique

The database generates the key, so it is guaranteed to be unique within the table.

Compact

A short integer is fast to compare, sort, and use as a reference from other tables.

No Sensitive Data

A surrogate number does not reveal anything private about the record it identifies.

The enrollment numbers in the students table examples above, 10047 through 10050, could function as surrogate keys if the database assigns them automatically and guarantees uniqueness.

Identifiers Are Labels, Not Quantities

By convention, identifier columns are often named with the table name followed by "ID", such as StudentID or BookID. Students sometimes confuse identifiers with numeric data because identifier values often look like numbers. But what would it mean to average two student IDs? Nothing meaningful. Identifiers serve as labels. Whether they are stored as integers or text, they function as unique names, not quantities. Chapter 6 covers primary keys, candidate keys, natural keys, surrogate keys, and foreign keys in full detail.

Chapter Summary
  • A field is one category of information a database tracks about a subject. A record is the complete set of field values for one specific subject. A table organizes records of the same kind into rows and columns, where each column represents one field and each row represents one record.
  • Every field has a data type that defines what kind of value it holds and what operations are valid on it. Choosing the correct data type protects against mistakes, enables correct operations, and supports efficient storage.
  • Text data holds character sequences and is used for names, descriptions, addresses, codes, and any value where arithmetic is meaningless. Even values that look like numbers should be stored as text when they serve as labels or codes rather than quantities.
  • Numeric data holds quantities and measurements where arithmetic operations make sense. Database systems distinguish between integers (whole numbers) and decimal values (numbers with fractional parts). The correct choice depends on whether fractions are possible.
  • Date and time data holds calendar and clock values in a format the database understands. The ISO 8601 standard, YYYY-MM-DD, is universally used for date storage because it sorts correctly and eliminates regional ambiguity. Proper date fields enable date arithmetic and chronological sorting that text storage cannot support.
  • Boolean data holds exactly two possible values representing true and false. Boolean fields answer yes-or-no questions. When a field needs more than two states, a category type is more appropriate.
  • Category data holds values from a limited, predefined set of choices. Inconsistency in category values, including different spellings, abbreviations, and formats for the same category, is a major source of database problems.
  • NULL represents the absence of a value. NULL is not zero and not an empty string. It means the value is unknown, not yet recorded, or not applicable. Databases must handle NULL carefully because testing equality with NULL does not behave like testing equality with a regular value.
  • Identifiers, formally called primary keys, give each record a unique label that distinguishes it from every other record in the table. Surrogate keys, generated by the system, are usually more reliable than natural keys drawn from real-world data.
Key Terms
Field

One category of information that a database tracks about a subject. Corresponds to a column in a table.

Record

The complete collection of field values for one specific subject. Corresponds to a row in a table.

Table

A structured database object that organizes records of the same kind into rows and columns.

Row

One record in a table. Each row represents one specific instance of whatever the table tracks.

Column

One field in a table. Each column represents one category of information and holds values of the same type for every row.

Data Type

The classification that defines what kind of value a field holds and what operations are valid on it.

Text Data

A data type that holds character sequences: letters, digits, spaces, and symbols. Used for names, descriptions, codes, and values where arithmetic is meaningless.

Numeric Data

A data type that holds quantities and measurements where arithmetic operations are valid. Divided into integers and decimal values.

Integer

A whole number with no fractional part.

Decimal

A number with a fractional part, stored with a specified or approximate precision.

Date Data

A data type that holds calendar dates, usually stored in ISO 8601 format (YYYY-MM-DD).

Time Data

A data type that holds times of day: hours, minutes, and seconds.

Datetime / Timestamp

A data type that holds both a date and a time of day together, typically used to record exactly when an event occurred.

ISO 8601

The international standard for representing dates as YYYY-MM-DD, ensuring correct sorting and eliminating regional ambiguity.

Boolean Data

A data type that holds exactly one of two values representing true and false.

Category Data

A data type that holds values from a defined, limited set of allowed choices. Used for classifications, statuses, and types.

NULL

A special marker indicating that a field has no value for a given record. Represents unknown, unrecorded, or not-applicable information. NULL is not zero and not an empty string.

Required Field

A field where NULL is not permitted. Every record must have a value for this field.

Optional Field

A field where NULL is allowed. Records may or may not have a value for this field.

Identifier / Primary Key

A field, or combination of fields, that uniquely identifies each record in a table. No two records may share the same identifier value.

Natural Key

An identifier drawn from real-world information that already uniquely identifies the subject outside the database.

Surrogate Key

A system-generated identifier with no real-world meaning, used to uniquely identify records inside the database.

Common Misconceptions
Misconception 1
Storing numbers as text is always wrong.

This is incorrect. Numbers that function as labels, codes, or identifiers should be stored as text. Phone numbers, postal codes, ISBN numbers, vehicle identification numbers, and student enrollment numbers may contain only digits, but they should be stored as text because arithmetic on them would be meaningless. The correct question is not "does this look like a number?" but "will arithmetic ever be performed on this field?"

Misconception 2
NULL means zero.

This is incorrect, and treating it as such causes real problems in queries and calculations. NULL means the value is absent. A sales total of NULL is not the same as a sales total of zero. A zero means no sales occurred and the value is known. NULL means the total has not been recorded or is unknown. Treating NULL as zero produces incorrect sums, averages, and counts.

Misconception 3
Every row must have a unique value in every column.

This is incorrect. Uniqueness is required only for identifier (primary key) fields. Other fields may repeat values freely. Many students may belong to the same department. Many books may share a subject area. Only the identifier column must be unique across all rows.

Misconception 4
Tables and spreadsheets work the same way.

They look similar but operate very differently. A spreadsheet allows different kinds of values in different cells of the same column. A database table enforces the same data type for every cell in a column. A spreadsheet allows mixed content, colors, merged cells, and formulas anywhere. A database table enforces structure, constraints, and rules consistently across every row.

Misconception 5
A date stored as "January 15, 2023" works the same as a proper date field.

This is incorrect. Text dates do not sort chronologically, cannot be easily compared for ranges, do not enable date arithmetic, and are vulnerable to inconsistencies in how different people write dates. A proper date field using ISO 8601 format eliminates all of these problems and allows the database to work with dates as dates, not as arbitrary strings of characters.
