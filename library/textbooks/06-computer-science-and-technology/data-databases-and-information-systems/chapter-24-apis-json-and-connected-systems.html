APIs, JSON, and Connected Systems

Modern data systems rarely work alone. A hospital records system shares patient data with a pharmacy. A university's enrollment database feeds a financial aid portal and a course scheduling tool. Chapter 24 introduces the ideas and structures that make this communication possible, from API requests and responses to JSON, status codes, authentication, and integration risk.

Before You Begin

Chapter 24 builds on ideas from earlier parts of this textbook. Students should be comfortable with tables, records, fields, data types, SQL queries, and forms and reports before studying this chapter. Chapter 23 introduced CSV files and data exchange between tools. This chapter extends that idea to live, networked communication between systems.

Students do not need to write software to understand this chapter. The focus is on reading, reasoning, and responsible analysis. A student who has studied web development alongside this textbook will find many familiar ideas here, approached from the data perspective.

24.1

Why Systems Need to Communicate

Imagine a university that has three separate information systems: a student records database, a financial aid portal, and a course scheduling tool. Each system has its own database, its own staff, and its own users. Students appear in all three, but each system stores a separate copy of student information.

When a student changes their name, the records office updates the student records database. But the financial aid portal and the scheduling tool still show the old name. Someone has to manually update the other two systems, and until that happens, the information is inconsistent.

Now imagine an alternative. The financial aid portal and the scheduling tool do not store their own copies of student information. Instead, whenever they need a student's name, enrollment status, or ID number, they ask the student records database directly. The student records database is the single source of truth. The other systems simply request data from it when they need it.

This is the basic problem that connected systems solve. Rather than duplicating data across many places and then struggling to keep all copies synchronized, systems can communicate in real time, requesting and sharing data as needed.

Benefits and Costs

The benefits of connected systems are significant. Data remains consistent. Updates happen once. Systems can be built to specialize in what they do best, then rely on other systems for what those other systems do best. New systems can be added without rebuilding everything.

The costs and risks are also real. When systems depend on each other, a failure in one system can affect all the others. Communication between systems must be designed carefully. Data shared across systems must be protected. The rules for who can request what data, and what can be done with it, must be clearly defined.

Central Idea

Understanding how systems communicate is essential for anyone who works with data, whether as a designer, analyst, administrator, or developer. This chapter introduces the structures that make that communication possible and the responsibilities that come with it.

24.2

What an API Is

API stands for Application Programming Interface. The name can sound technical and abstract, but the idea behind it is practical and concrete.

An interface is a defined way of interacting with something. A light switch is an interface for a lighting circuit. You do not need to understand the wiring to use the switch. You just need to know that flipping it up turns the light on and flipping it down turns it off. The switch defines how you interact with the circuit.

An API is the same idea applied to software. An API defines how one piece of software can interact with another. It specifies what requests can be made, what information must be included in those requests, and what responses will come back.

APIs in Daily Life

When a weather app on your phone shows you the current temperature, the app is almost certainly using an API. The app does not store weather data itself. Instead, it sends a request to a weather service, which retrieves the current conditions from its own database and sends the data back. The app receives the data and displays it. The entire exchange happens in a fraction of a second.

When you log into a website using your existing account from another service, that is also an API at work. The website is not checking your password itself. It is asking another service to verify your identity, and that service responds with a confirmation.

When an online store shows you whether an item is in stock, it may be querying a warehouse inventory system through an API. When that store processes your payment, it sends a request through a payment API to a financial system. APIs are everywhere.

Web APIs

For the purposes of this textbook, we will focus on web APIs, which are APIs that communicate over the internet using standard web protocols. These are the APIs that connect databases to websites, mobile apps, third-party services, and other systems. They are the most common kind of API a data professional will encounter.

24.3

Requests and Responses

Every API interaction follows the same basic pattern. One system sends a request, and another system sends back a response. Think of it like a conversation: one party asks a question, and the other party answers. The question must be formed in a way the other party understands, and the answer comes back in a structured format.

The Request

A request is a message from one system to another asking for something. A request typically includes several parts. The method describes what kind of action is being requested:

GET

Requests data without changing anything. A request to retrieve a student's information uses GET.

POST

Sends new data to create something. A request to submit a new enrollment record uses POST.

PUT / PATCH

Sends data to update something that already exists. Changing a student's contact information uses PUT or PATCH.

DELETE

Requests that something be removed. Cancelling a registration uses DELETE.

The URL identifies what the request is directed at. A URL for an API request might look like:

URL — API Request Address
https://api.university.edu/students/10482

Headers carry additional information about the request, including the format the requester expects and an authentication credential. A body is present in POST, PUT, and PATCH requests and carries the data being submitted.

The Response

A response is the message sent back by the API after processing a request. A response typically includes a status code communicating whether the request succeeded or failed (discussed in Section 24.7), headers carrying information about the response, and a body containing the actual data being returned. For a GET request, the body carries the requested records. For an error, the body often explains what went wrong.

The request-response pattern is simple but powerful. An entire system of data exchange, involving dozens or hundreds of systems talking to each other simultaneously, can be built from this basic structure.

24.4

Endpoints

An endpoint is a specific address in an API that handles a particular kind of request. Every API has a collection of endpoints, each one designed to do something specific.

Endpoints are like the counters in a service organization. A university might have a registrar's counter, a financial aid counter, a library counter, and an IT help desk counter. Each counter handles a different kind of request. An API's endpoints work the same way. Each endpoint has an address, and that address corresponds to a specific resource or action.

Example Endpoints

Here are some example endpoints for a fictional university API:

API Endpoints — University System
GET    /students                  — Returns a list of all students
GET    /students/10482            — Returns the record for student 10482
POST   /students                  — Creates a new student record
PUT    /students/10482            — Updates the record for student 10482
DELETE /students/10482            — Deletes the record for student 10482

GET    /courses                   — Returns a list of all courses
GET    /courses/CS201             — Returns the record for course CS201
GET    /courses/CS201/enrollments — Returns all students enrolled in CS201

POST   /enrollments               — Creates a new enrollment record
DELETE /enrollments/88734         — Cancels enrollment 88734
REST Conventions

The structure of these endpoints follows a pattern called REST, which stands for Representational State Transfer. REST is a widely used set of conventions for designing web API endpoints. RESTful APIs use clear, predictable URL structures so that developers can understand how to interact with a system without memorizing every possible command.

Notice how the endpoints reflect the underlying data structure. Students are a resource. Courses are a resource. Enrollments are a resource. The API exposes those resources through consistent, readable addresses. This mirrors the relational database design students studied in Parts II and III of this textbook.

24.5

Parameters

When a request is sent to an API endpoint, it often needs to be more specific than just naming a resource. A request to retrieve all students is rarely useful. A request to retrieve students enrolled in a specific course, in a specific term, sorted alphabetically, limited to fifty results, is much more useful. Parameters are the way API requests communicate these additional details.

Path Parameters

Some parameters are built directly into the URL path. When a URL includes /students/10482, the 10482 is a path parameter. It identifies which specific student is being requested. Path parameters are typically used when a request is directed at a specific, identifiable resource.

Query Parameters

Other parameters are added to the end of the URL as a query string, separated from the path by a question mark and from each other by ampersand characters.

URL — Query Parameters
GET /students?department=computer_science&year=3&sort=last_name

In this example, three query parameters are included: department=computer_science filters results to students in computer science, year=3 filters to third-year students, and sort=last_name requests results sorted alphabetically by last name. Query parameters can be combined in many ways to answer different questions.

Body Parameters

For requests that create or update data, the parameters are often sent in the request body rather than in the URL. A request to create a new student record might include a body like this:

JSON — Request Body
{
  "first_name": "Amara",
  "last_name":  "Osei",
  "department": "computer_science",
  "year":       1
}

The body carries the full set of field values for the new record. The API reads these values, validates them, and uses them to insert a new row in the database.

24.6

JSON Data

When an API sends data back in a response, that data needs to be in a format both systems understand. The most widely used format in modern web APIs is JSON, which stands for JavaScript Object Notation. JSON is a text-based format for representing structured data. It is lightweight, human-readable, and easy for software to parse and generate. Despite its name, JSON is used in virtually every programming language and environment.

JSON Value Types

JSON supports a small set of value types:

  • string
  • number
  • boolean
  • null
  • object
  • array

These types can be nested inside each other in any combination, making it possible to represent complex, hierarchical data in a single structure.

JSON Objects

A JSON object represents a single structured item, similar to a record in a database table. It is wrapped in curly braces, and each field is written as a key and a value separated by a colon. Fields are separated by commas.

JSON — Object (Single Record)
{
  "student_id":       10482,
  "first_name":       "Amara",
  "last_name":        "Osei",
  "department":       "Computer Science",
  "year":             3,
  "active":           true,
  "email":            "amara.osei@mgu.edu"
}

Compare this to a database record. The keys in the JSON object correspond to field names in a table. The values correspond to field values. The structure is familiar.

JSON Arrays

A JSON array represents an ordered list of values, wrapped in square brackets. When an API returns multiple records, it typically wraps them in an array:

JSON — Array (Multiple Records)
[
  {
    "student_id": 10482,
    "first_name": "Amara",
    "last_name":  "Osei",
    "department": "Computer Science",
    "year":       3
  },
  {
    "student_id": 10491,
    "first_name": "Diego",
    "last_name":  "Vargas",
    "department": "Mathematics",
    "year":       2
  },
  {
    "student_id": 10507,
    "first_name": "Priya",
    "last_name":  "Nair",
    "department": "Computer Science",
    "year":       4
  }
]
Nested JSON

JSON objects can contain other objects or arrays as values. This allows complex, hierarchical data to be represented in a single response, often combining what a SQL query would retrieve through multiple joined tables.

JSON — Nested Object with Array
{
  "student_id": 10482,
  "first_name": "Amara",
  "last_name":  "Osei",
  "department": "Computer Science",
  "enrollments": [
    {
      "course_id":   "CS201",
      "course_name": "Data Structures",
      "grade":       "A"
    },
    {
      "course_id":   "CS315",
      "course_name": "Database Systems",
      "grade":       "B+"
    }
  ],
  "contact": {
    "email": "amara.osei@mgu.edu",
    "phone": "555-0193"
  }
}

Here, the enrollments field is an array of objects, one for each course. The contact field is a nested object. This single API response describes one student along with related records, representing relationships that in a relational database would be stored across multiple tables.

Reading JSON

Students should practice reading JSON carefully. The key skills are: recognizing objects, arrays, strings, numbers, booleans, and null values; identifying the keys and their values; tracing nested structures; and understanding what data the response represents. Reading JSON is similar to reading a result table from a SQL query, the goal is to understand what data is present and what it means.

24.7

Status Codes at an Introductory Level

Every API response includes a status code, a three-digit number that communicates whether the request succeeded or failed, and if it failed, what kind of failure occurred. Status codes are standardized, which means a developer who learns what they mean can read any API response and understand its outcome at a glance. Status codes are grouped by their first digit.

2xx — Success
  • 200 OK. The standard success response. The request succeeded and the body contains the requested data or a confirmation.
  • 201 Created. The request succeeded and a new resource was created. Typically returned after a successful POST request.
  • 204 No Content. The request succeeded but the response body is empty. Sometimes returned after a successful DELETE request.
4xx — Client Error
  • 400 Bad Request. The request was malformed or missing required information. The server could not understand it.
  • 401 Unauthorized. The request requires authentication and none was provided, or the provided credentials were invalid.
  • 403 Forbidden. The requester is authenticated but does not have permission to access this resource.
  • 404 Not Found. The requested resource does not exist. The student ID or course ID in the URL was not found in the database.
  • 422 Unprocessable Entity. The request was well-formed but contained invalid data, such as a date field that does not contain a valid date.
  • 429 Too Many Requests. The requester has exceeded the allowed number of requests in a given time period. Rate limiting is discussed in Section 24.9.
5xx — Server Error
  • 500 Internal Server Error. A general server error. The server encountered a problem it did not know how to handle.
  • 503 Service Unavailable. The server is temporarily unable to handle requests, often due to maintenance or overload.
Why Status Codes Matter for Data Work

Status codes are not just technical trivia. When a 404 is returned, the requested record does not exist and no data should be used. When a 401 is returned, the authentication credentials need to be checked. When a 500 is returned, the problem is on the server side and may need to be reported. Understanding status codes allows data workers to diagnose integration problems, identify unreliable responses, and communicate clearly about what went wrong.

24.8

Authentication at a Basic Level

When a system exposes data through an API, not every request should be allowed. Some data is sensitive. Some actions, like deleting records or modifying student information, should be restricted to authorized users. Authentication is the process of proving identity when making an API request. An authenticated request includes some form of credential that tells the API who is making the request.

API Keys

The most common form of API authentication for data-focused work is the API key. An API key is a long string of characters, something like a very complex password, assigned to a specific user or application. When a request is made, the API key is included so the API can identify who is asking.

API keys are typically included in the request header:

Request Header — Authentication
Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...

The API checks the key against its database of registered keys. If the key is valid, the request proceeds. If not, a 401 Unauthorized response is returned.

Why API Keys Must Be Protected

API keys should be treated like passwords. They should never be shared publicly, embedded in code that others can read, or transmitted over insecure connections. If an API key is leaked, anyone who finds it can make requests as if they were the legitimate user, potentially reading sensitive data, creating records, or deleting information. Organizations should regularly audit which systems have API keys, revoke keys that are no longer needed, and rotate keys periodically.

Authentication and Authorization

Authentication establishes identity. It answers the question: who is making this request? It does not, by itself, answer the question: what is this person allowed to do? That is authorization, which was discussed in Chapter 18. In practice, both work together. A request is authenticated to establish identity, and then the system checks that identity's permissions to decide what it is allowed to do.

24.9

Rate Limits and Reliability

A database can answer queries very quickly, but it cannot answer infinitely many queries simultaneously without slowing down or failing. An API that allows unlimited requests is vulnerable to being overwhelmed, whether accidentally by a poorly written application, or deliberately by a malicious actor trying to disable the service. Rate limiting is the practice of restricting how many requests a given user or application can make within a certain time period.

How Rate Limits Work

A weather API might allow up to 1,000 requests per day per API key. A student records API might allow up to 100 requests per minute. When a requester exceeds the rate limit, the API returns a 429 Too Many Requests status code instead of the requested data. The response may also include information about when the limit will reset.

Rate limits have practical consequences for anyone who uses APIs to collect or exchange data. A data analyst who writes a script to collect information from an API needs to account for rate limits. If the script makes requests too quickly, it will hit the limit and start receiving error responses instead of data. The resulting dataset will be incomplete, and the analyst might not notice unless status codes are being checked.

Reliability and Availability

When one system depends on another through an API, the dependent system inherits the other system's reliability problems. If the API being called is unavailable, the system that depends on it cannot function. Well-designed integrations include:

  • Retry logic: If a request fails with a temporary error, the system tries again after a short delay.
  • Timeouts: If a response does not arrive within a reasonable time, the system stops waiting and handles the failure gracefully.
  • Fallback behavior: If the API is unavailable, the system has a way to continue operating, perhaps with cached data or a degraded feature set.
  • Monitoring and alerting: The integration is watched, and administrators are notified when the API is failing or slow.

Students building information systems that depend on external APIs should treat those APIs as components that will occasionally fail, and design accordingly.

24.10

Integration Risks and Responsibilities

Connecting systems through APIs creates opportunities and risks at the same time. Students who work with data should understand both.

Data Quality Risks

When data flows between systems through APIs, quality problems can appear at multiple points. The sending system may have incorrect data. The API may transform it in unexpected ways. The receiving system may interpret fields differently than intended. Treating API-delivered data with the same quality habits from Chapter 4 is essential.

Privacy and Sensitivity Risks

Data shared through APIs often includes personal or sensitive information. Every API that transmits this kind of data is a potential channel for exposure if improperly secured. The connection should always be encrypted (HTTPS), access should require credentials, and only necessary data should be included, an application of the data minimization principle from Chapter 27.

Dependency and Vendor Risks

When an information system depends on an external API, it also depends on whoever operates that API. If the provider changes their pricing, modifies their data format, or shuts down entirely, every dependent system is affected. Organizations should document API dependencies, monitor for changes, and design systems so that dependencies could be replaced.

Data Use and Consent Risks

When data is collected through APIs, questions of consent and appropriate use arise. Data governance principles, discussed in Chapter 27, apply fully to API-based integrations. Data shared through an API is still data that belongs to real people, and the responsibilities that come with handling that data do not disappear because the transmission was automated.

Access and Logging

When sensitive data is retrieved through an API, that access should be recorded for accountability. A data leak through an API can expose thousands or millions of records. API security deserves the same serious attention as database security, covered in Chapter 18.

Chapter Summary
  • Connected systems exchange data through APIs rather than duplicating data across multiple places. This keeps information consistent and allows systems to specialize in what they do best.
  • An API is a defined interface that allows one piece of software to request data or actions from another. Web APIs follow a request-response pattern: a client sends a structured request and a server returns a structured response.
  • Requests include a method (GET, POST, PUT, DELETE), a URL identifying the endpoint, and optionally parameters and a body. Responses include a status code and usually a body containing data or an error explanation.
  • Endpoints are the specific addresses within an API that handle particular kinds of requests. RESTful APIs organize endpoints around resources, mirroring the table and relationship structure of the underlying database.
  • Parameters, whether in the URL path, query string, or request body, allow requests to be filtered, sorted, or customized without multiplying the number of endpoints.
  • JSON is the dominant format for API data exchange, representing structured data as objects, arrays, strings, numbers, booleans, and null values. A JSON object corresponds to a record; arrays hold multiple items; nesting represents joined data.
  • Status codes communicate whether requests succeeded or failed. The 2xx range indicates success; 4xx indicates client errors; 5xx indicates server errors.
  • Authentication protects APIs from unauthorized access, most commonly through API keys that should be treated with the same care as passwords.
  • Rate limits restrict how many requests can be made in a given time period. Well-designed integrations account for rate limits, implement retry logic, and monitor for failures.
  • Integration brings risks including data quality problems, security vulnerabilities, dependency on external providers, and privacy and consent responsibilities. These risks must be understood and managed.
Key Terms
API (Application Programming Interface)

A defined interface that allows one piece of software to communicate with another, specifying what requests can be made and what responses will be returned.

Authentication

The process of proving identity when making an API request, typically through credentials such as an API key.

Authorization

The process of determining what an authenticated identity is allowed to do, based on roles and permissions.

Body

The portion of a request or response that carries data. Request bodies contain data being submitted; response bodies contain requested data or error information.

Endpoint

A specific address within an API that handles a particular kind of request, corresponding to a resource or action.

GET

An HTTP method used to request data without modifying anything.

Headers

Metadata attached to API requests and responses, carrying information such as the authentication credential and the data format expected.

HTTPS

The encrypted form of web communication used to protect data in transit between systems.

JSON (JavaScript Object Notation)

A text-based data format used to represent structured data as objects and arrays, widely used for API data exchange.

JSON Array

A JSON value that contains an ordered list of items, wrapped in square brackets.

JSON Object

A JSON value that contains a collection of key-value pairs representing a structured record, wrapped in curly braces.

Method

The type of action indicated by an API request: GET for retrieving data, POST for creating, PUT or PATCH for updating, and DELETE for removing.

Null (JSON)

A JSON value representing the absence of a value, analogous to NULL in SQL.

OAuth

A standard mechanism for delegating API access, allowing users to grant third-party applications access to their data without sharing passwords directly.

Parameter

Additional information included in an API request to filter, sort, or customize the response. Parameters may appear in the URL path, query string, or request body.

POST

An HTTP method used to submit data and create a new resource.

Query Parameter

A parameter added to the end of a URL after a question mark, used to filter, sort, or paginate API results.

Rate Limiting

A restriction on how many API requests a user or application can make within a given time period, protecting the API from being overwhelmed.

Request

A message sent from a client to an API, specifying what data or action is being asked for.

Response

A message sent back from an API after processing a request, including a status code and usually a data body.

REST (Representational State Transfer)

A set of conventions for designing web API endpoints using clear, resource-oriented URL structures.

Status Code

A three-digit number included in an API response that communicates whether the request succeeded or failed and what kind of outcome occurred.

Common Misconceptions
Misconception 1
APIs are only for programmers.

APIs operate at a technical level, but understanding how they work is essential for anyone who works with data. Data analysts, database administrators, information system designers, and decision-makers all need to understand what APIs do, what risks they introduce, and what questions to ask about integrations. The ability to read a JSON response, interpret a status code, or evaluate an API design is a data literacy skill, not only a programming skill.

Misconception 2
If an API returns data, the data must be correct.

API responses can contain incorrect, incomplete, outdated, or inconsistently formatted data. The source database may have data quality problems. The API may have bugs. The data may have been transformed incorrectly. Data received through an API should go through the same quality checks as data from any other source.

Misconception 3
HTTPS keeps API communication completely secure.

HTTPS encrypts data in transit, which means someone intercepting the communication cannot read it. But HTTPS does not protect against incorrect authentication implementation, leaked API keys, misconfigured permissions, or a database that stores data without adequate access controls. Encryption in transit is one layer of security, not the whole picture.

Misconception 4
Rate limits are just technical obstacles.

Rate limits exist for good reasons. They protect APIs from being overwhelmed, ensure fair access among multiple users, and prevent runaway scripts from causing damage. Respecting rate limits is not merely compliance; it is responsible use of a shared resource.

Misconception 5
Because data transfer is automated, consent and privacy concerns do not apply.

Automated data transfer does not remove human responsibilities. If a person's data is being shared between systems through an API without their knowledge, that is a privacy concern regardless of whether a human or a software process initiated the transfer. The automation makes it easier to share data at scale, which makes privacy and consent considerations more important, not less.
