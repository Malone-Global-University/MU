Indexes and Query Performance at an Introductory Level

A database holding twenty records answers almost any question instantly. A database holding two million records, with real users waiting, is a different kind of system altogether. This chapter introduces performance as a design concern: what happens when a database searches for data, why that search can become slow, and how indexes change the picture entirely.

17.10Performance as a Design Concern

Every database eventually grows large enough that the time it takes to answer a query starts to matter. A database holding twenty student records will answer almost any question in an instant. A database holding two million records, with users waiting for results, becomes a system where design choices have real consequences.

Chapter 17 introduces performance thinking as a design concern, not as an advanced administration topic. Students learn what happens when a database searches through records, why that process can become slow, how indexes make certain searches dramatically faster, and what tradeoffs come with choosing to index a column.

The chapter stays conceptual and practical. It does not teach query optimization tools, advanced execution plans, or database server tuning. It teaches the ideas every database designer needs to understand before a system grows large, and it reinforces that performance is built into good design from the beginning, not patched in afterward.

Note for New Students

This chapter does not require you to memorize internal database engine mechanics. What matters is the conceptual model: searching without help is slow at scale, indexes provide that help, and every help comes with a cost. Keep that exchange in mind as you read.

17.1

Why Query Performance Matters

When a user submits a query, they are not simply asking a question. They are asking a machine to locate specific data in a potentially enormous collection of records, assemble the relevant pieces, apply filters, sort the results, and return an answer. When the database is small, this happens so quickly that the time is invisible. When the database is large and the design is careless, the same operation can take seconds, minutes, or, in poorly designed systems, much longer.

Performance matters for reasons that are easy to understand from everyday experience. A website that takes ten seconds to load a product search loses users. A hospital system that takes thirty seconds to retrieve a patient record during an emergency creates real danger. A school system that takes several minutes to generate a report wastes the time of every administrator who waits for it. A library catalog that responds slowly to every patron search erodes the usefulness of the entire collection.

Database performance is not only about speed as a pleasant convenience. It is about whether a system remains usable as it grows. Almost every database grows over time. Records are added, more users join, more queries run simultaneously. The design choices that work acceptably with ten thousand records may become genuinely painful with ten million.

Central Idea

Understanding performance at an introductory level means understanding one central question: when a query asks the database to find something, how does the database go about looking?

17.2

Searching Through Records

Before thinking about how databases improve their searches, it helps to think carefully about the basic problem. Imagine a stack of five hundred index cards, each containing a student's name, student ID number, department, and enrollment date. You need to find every student in the Engineering department.

There is no shortcut. You pick up the first card, read the department field, check whether it says Engineering, set it aside if it does not, and move to the next card. You repeat this process for every card in the stack. By the time you have checked all five hundred cards, you have found your Engineering students, but you have also touched every card in the collection, even the cards for students in Biology, History, and Languages who could not possibly match your query.

This is exactly what a database does when it has no index on the column you are searching. It examines every row in the table, one by one, checks the value in the column you specified, and either includes or excludes that row from the result. When the table contains five hundred records, this is fast. When the table contains five million records, this is slow in a way that becomes noticeable to anyone waiting for results.

The important thing to understand is that this row-by-row inspection is not a design flaw. It is the natural behavior of a database when it has no faster way to locate data. The question becomes whether the database designer has given the database the tools it needs to search more efficiently.

17.3

Full Table Scans at an Introductory Level

Database professionals use the term full table scan to describe what happens when a database reads every row in a table to answer a query. The name is descriptive: the database scans the full table.

A full table scan is not always a problem. For small tables, it is entirely reasonable. If a table contains a hundred rows, scanning all hundred rows to find a matching record is practically instantaneous. The overhead of setting up a more sophisticated search mechanism would cost more than the scan itself.

Full table scans become a concern when tables are large and when certain queries run frequently. Consider the following scenario.

A university stores its student records in a table called Students. After fifteen years of operation, the table contains three hundred thousand records. An administrative system runs a query every time a student logs in to verify their account and retrieve their record. That query looks something like this:

SQL — Login Lookup Query
SELECT *
FROM Students
WHERE email = 'nguyen.linh@mgu.edu';

If the database has no index on the email column, it must scan all three hundred thousand rows to find the one matching record. Every single login triggers a full table scan. With a hundred students logging in simultaneously during registration period, the database is performing three hundred thousand row reads one hundred times over, all at the same moment.

An index on the email column would allow the database to locate the matching record almost immediately, touching only a tiny fraction of the table instead of all of it.

Worth Remembering

Full table scans are not errors. They are the default behavior. But understanding when they occur, and when they can be replaced by indexed lookups, is the core of introductory performance thinking.

17.4

What an Index Is

An index is a separate data structure that the database maintains alongside a table. It is built on one or more columns, and it stores the values from those columns along with pointers that tell the database exactly where each matching row lives in the table.

The closest everyday analogy is the index at the back of a textbook. A textbook index does not contain the full content of every page. It contains a sorted list of important terms, and next to each term it lists the page numbers where that term appears. When you want to find information about normalization, you do not read the textbook from page one. You go to the index, find the entry for normalization, and jump directly to the relevant pages.

A database index works the same way. When the database needs to answer a query that searches by an indexed column, it does not scan the entire table. Instead, it consults the index, which is organized for fast lookup. The index directs the database to the specific rows that match the query, and the database retrieves only those rows.

The speed difference can be enormous. Searching through an index to find a specific value in a table with one million rows might require examining only twenty or thirty index entries rather than one million table rows. This is possible because database indexes are organized as sorted, hierarchical structures that allow the database to narrow down the search space very quickly with each step, rather than checking one record at a time.

The index is sorted alphabetically. When a query asks for the row where email = 'nguyen.linh@mgu.edu', the database uses the index to go directly to row 2, without reading rows 1, 3, 4, or 5 at all.

The actual internal structure of a database index is more sophisticated than this simple list, using a hierarchical structure called a B-tree that allows very fast binary searching. But the conceptual purpose remains the same: the index stores sorted values and pointers, so the database can locate matching rows without reading the whole table.

Creating an Index in SQL

In SQL, you create an index with the CREATE INDEX statement:

SQL — Creating an Index
CREATE INDEX idx_students_email
ON Students (email);

This tells the database to build and maintain an index on the email column of the Students table. After this statement runs, any query that filters by email can use the index instead of scanning the full table.

The database maintains the index automatically. When new rows are inserted, when email addresses are updated, or when records are deleted, the database updates the index to keep it accurate. This maintenance happens in the background and is transparent to the user.

17.5

Primary Key Indexes

Every table with a primary key benefits from an index on that primary key. In fact, most database systems, including SQLite and PostgreSQL, automatically create an index on the primary key when the table is defined. This happens without any extra instruction from the designer.

The reason is straightforward. Primary keys are the most common way to look up a specific record. Queries join tables using primary and foreign keys constantly. If looking up a primary key value required a full table scan, the overhead would be intolerable.

Consider a simple join:

SQL — Joining on a Primary Key
SELECT Students.name, Enrollments.course_id
FROM Students
JOIN Enrollments ON Students.student_id = Enrollments.student_id
WHERE Students.student_id = 1042;

This query joins two tables on student_id. For the join to work efficiently, the database needs to find the row in Students where student_id = 1042, and all the rows in Enrollments where student_id = 1042. If both tables have indexes on student_id, these lookups are nearly instantaneous. If neither table has indexes, both require full table scans, and the join itself must work through combinations of rows in an inefficient way.

Because primary key indexes are created automatically, students generally do not need to create them manually. But understanding that they exist, and why they matter, is essential for understanding how join queries perform in practice.

17.6

Indexing Foreign Keys

If primary keys are automatically indexed, foreign keys require more attention. In many database systems, including SQLite, foreign keys are not automatically indexed. The designer must decide whether to create indexes on foreign key columns.

This matters because foreign key columns are used constantly in join conditions. Every time you join two tables, the query connects a foreign key in one table to a primary key in another. If the foreign key column has no index, the join may require a full table scan on the child table for every row in the parent table.

Consider a university database with students and their enrollments:

SQL — Table Notes
-- Students table (50,000 rows)
-- Primary key: student_id (automatically indexed)

-- Enrollments table (400,000 rows)
-- Foreign key: student_id references Students(student_id)
-- (no automatic index on this foreign key column)

A query that retrieves all enrollments for a given student:

SQL — Enrollment Lookup by Student
SELECT *
FROM Enrollments
WHERE student_id = 1042;

Without an index on Enrollments.student_id, the database reads all four hundred thousand rows in the Enrollments table to find the records where student_id = 1042. With an index, it goes directly to those rows.

Creating the index is simple:

SQL — Indexing a Foreign Key
CREATE INDEX idx_enrollments_student_id
ON Enrollments (student_id);

As a design habit, database designers commonly create indexes on foreign key columns whenever the table is large and joins on that column will run frequently. This is not a universal rule, but it is a useful starting point.

17.7

Indexes on Search Columns

Primary keys and foreign keys are the most common targets for indexes, but they are not the only ones. Any column that appears frequently in WHERE conditions, particularly on large tables, is a candidate for indexing.

Consider the search scenarios that arise in a library catalog system. A librarian searches for all books by a specific author:

SQL — Search by Author
SELECT * FROM Resources WHERE author_id = 88;

A patron searches for resources tagged with a specific subject:

SQL — Search by Subject
SELECT * FROM Resources WHERE subject = 'Database Systems';

An administrator retrieves all resources added after a certain date:

SQL — Search by Date Added
SELECT * FROM Resources WHERE date_added > '2024-01-01';

If the Resources table contains half a million records, each of these queries could trigger a full table scan without indexes. With indexes on author_id, subject, and date_added, the database can locate matching records quickly.

Creating these indexes follows the same pattern:

SQL — Creating Multiple Search Indexes
CREATE INDEX idx_resources_author_id
ON Resources (author_id);

CREATE INDEX idx_resources_subject
ON Resources (subject);

CREATE INDEX idx_resources_date_added
ON Resources (date_added);

The decision about which columns to index depends on actual usage. Columns that appear in frequent, large-table searches are good candidates. Columns on small tables, or columns that are rarely used in filters, may not benefit enough to justify the cost of maintaining the index.

Composite Indexes

It is also possible to create an index on two or more columns together. This is called a composite index or a multi-column index. Composite indexes are useful when queries regularly filter by two columns together.

For example, if an enrollment system frequently looks up records by both student and term:

SQL — Filtering by Two Columns
SELECT *
FROM Enrollments
WHERE student_id = 1042 AND term_id = 5;

A composite index can help:

SQL — Creating a Composite Index
CREATE INDEX idx_enrollments_student_term
ON Enrollments (student_id, term_id);

The order of columns in a composite index matters. The index is most useful for queries that filter by the leading column first. A composite index on (student_id, term_id) helps queries filtering by student_id alone, or by both student_id and term_id. It is less helpful for queries that filter only by term_id.

A Note on Depth

Composite indexes are an intermediate topic, but understanding that they exist helps students see that index design can become nuanced as systems grow.

17.8

Index Tradeoffs

Indexes are not free. Each index the database maintains comes with costs, and understanding these tradeoffs is part of making good design decisions.

Indexes Speed Up Reads but Slow Down Writes

When a new row is inserted into a table, the database does not only add the row to the table. It also updates every index defined on that table to include the new row's values. If a table has five indexes and a thousand new rows are inserted at once, the database performs a thousand insertions into the table and five thousand index updates.

Similarly, when a row is updated, the indexes on affected columns must be updated. When a row is deleted, the indexes must remove the entry for that row.

For tables that are read far more often than they are written to, this is a reasonable tradeoff. A library catalog might add a few new resources each week but answer thousands of search queries each day. Indexes clearly help there. A system that receives millions of high-speed insertions per hour, such as a sensor logging system or a financial transaction feed, might find that indexes on lightly queried columns create significant write overhead.

This tradeoff can be summarized simply:

Table 17.3 — How indexes affect different database operations.
Operation Effect of indexes
SELECT queries with indexed columnsFaster
INSERT, UPDATE, DELETESlower, since indexes must be updated
Storage spaceMore space used, since index structures occupy disk
Indexes Use Additional Storage

Every index occupies storage space. On large tables with many indexed columns, the indexes themselves can grow to be a significant fraction of the total database size. For introductory purposes, it is enough to know that indexes are not free from a storage perspective, even if storage costs are relatively low in modern systems.

Too Many Indexes Can Hurt Performance

A common mistake by designers who have just learned about indexes is to index every column. This is usually a mistake. An over-indexed table becomes slow to write to and uses far more storage than necessary. Each additional index must be maintained on every data modification.

Good index design means choosing indexes thoughtfully, based on actual queries that the system runs, the size of the tables involved, and the relative frequency of reads versus writes.

17.9

Query Plans at a Basic Level

Modern database systems provide tools that allow designers and administrators to inspect how the database plans to execute a query before it actually runs. This is called a query plan or execution plan.

A query plan describes the steps the database engine intends to take to answer a query. It tells the designer whether the database intends to use an index or perform a full table scan, how it plans to join tables, and what order of operations it will follow.

In SQLite, you can see a simplified version of the query plan using the EXPLAIN QUERY PLAN command:

SQL — Inspecting a Query Plan
EXPLAIN QUERY PLAN
SELECT Students.name, Enrollments.course_id
FROM Students
JOIN Enrollments ON Students.student_id = Enrollments.student_id
WHERE Students.student_id = 1042;

The output might look something like this:

Output — Using Indexes
QUERY PLAN
|--SEARCH Students USING INTEGER PRIMARY KEY (rowid=?)
`--SEARCH Enrollments USING INDEX idx_enrollments_student_id (student_id=?)

This output tells you that the database intends to use the primary key index on Students and the index on Enrollments.student_id. Both lookups are efficient.

If instead the output showed:

Output — Falling Back to Scans
QUERY PLAN
|--SCAN Students
`--SCAN Enrollments

This would mean the database plans to perform a full table scan on both tables, which would be slow for large tables.

Query plan inspection is an advanced skill used by database administrators in professional work. At this introductory level, the important idea is simply that these tools exist. A designer who suspects a query is slow can ask the database how it plans to execute that query, and look for opportunities to add indexes that would allow more efficient lookups.

You Do Not Need Fluency Yet

You do not need to read query plans fluently to benefit from this awareness. Knowing that full table scans are the default, and that indexes change how the database searches, is enough to make much better design decisions.

17.10

Performance as a Design Concern

A common misunderstanding about database performance is that it is an operational problem, something to fix after the system is built and deployed, when users start complaining that queries are slow. This misunderstanding leads to systems that were not designed with performance in mind and require increasingly difficult patches as they grow.

Performance is a design concern. The best time to think about it is when the database is first being designed, when tables are being defined, when the expected data volumes are being estimated, and when the common queries are being planned.

Thinking About Scale from the Beginning

When designing a table, it helps to estimate how many rows it might contain at maturity. A table for a small school club might hold a few hundred rows and need no special indexing at all. A table for student enrollment at a large university system might hold millions of rows and require careful attention to which columns will be searched frequently.

The questions worth asking during design include:

  • How large will this table become over time?
  • Which columns will appear most often in WHERE conditions?
  • Which join conditions will run most frequently?
  • Are there searches that users will perform repeatedly on large data sets?

The answers to these questions guide index decisions before the system goes into production, when changes are easy and inexpensive.

The Query-First Design Habit

One useful habit for database designers is to think about the queries the system must answer before finalizing the table design. Chapter 9 introduced the idea of listing the questions a database must answer as part of requirements gathering. That same practice has a performance dimension.

If you know that the system must frequently answer the question "show me all orders placed by this customer," then you know the Orders table will be filtered by customer_id regularly. You can plan a foreign key and an index on customer_id from the beginning, rather than discovering the need after the system is in production.

If you know that the system must produce a monthly report of all enrollments grouped by department, you can think about how that query will be structured and whether the columns it touches are indexed appropriately.

This is not about premature over-engineering. It is about letting likely usage patterns inform design decisions in a natural way.

Indexes Are Not Magic

Students sometimes assume that adding an index to a column will always improve a query's performance. This is not always true. A database optimizer makes decisions about whether to use an index based on its estimate of how selective the index is, meaning how much it actually narrows down the search.

If a Students table has a column called is_active that stores only two possible values, true or false, and ninety-five percent of records have is_active = true, then an index on is_active may not help much. When a query asks for all active students, the index points to ninety-five percent of the table. The database may decide it is faster to scan the whole table than to use the index and then retrieve ninety-five percent of rows anyway.

Indexes provide the most benefit on columns with high selectivity, meaning columns where each value or small range of values narrows the result set to a small fraction of the table. Email addresses, student ID numbers, and specific category filters on tables with many categories are often highly selective. Boolean flags and columns with only a few possible values are often less selective and less likely to benefit from indexing.

Normalize First, Index Thoughtfully

Good normalization, introduced in Chapter 8, already contributes to performance by reducing data duplication and keeping table sizes smaller. A well-normalized database with thoughtfully placed indexes performs better than an over-denormalized database with indexes added as an afterthought.

The progression for database design should be: model the data correctly, design clean, normalized tables with clear keys and relationships, then think about which columns will be searched frequently on large tables and add indexes where they will genuinely help.

Get the Structure Right First

This order matters. Trying to fix a poorly designed database with indexes is like trying to make a building stable by adding exterior decoration. The structure has to be right first.

Key Terms
Full table scan

The process of reading every row in a table to find records that match a query condition. The default behavior when no index is available on the searched column.

Index

A separate data structure maintained by the database alongside a table. An index stores sorted values from one or more columns along with pointers to the table rows that contain those values, allowing the database to locate matching rows without scanning the full table.

Composite index

An index built on two or more columns together, useful for queries that regularly filter by multiple columns simultaneously.

Selectivity

A property of an index that describes how much it narrows down the result set. An index on a column with many unique values, such as email addresses, has high selectivity and tends to be most useful. An index on a column with only a few possible values, such as a true or false flag, has low selectivity and may offer less benefit.

Query plan

A description of the steps a database engine intends to take to execute a query. Also called an execution plan. Database systems provide tools to inspect query plans, helping designers identify whether queries are using indexes effectively.

EXPLAIN QUERY PLAN

A SQLite command that shows the query plan for a given SQL statement without executing the query itself.

CREATE INDEX

The SQL statement used to define an index on a table column or columns.

Write overhead

The additional work a database must perform to maintain index structures when rows are inserted, updated, or deleted.

B-tree

The hierarchical, sorted data structure most commonly used inside database indexes. Students do not need to understand the internal mechanics of B-trees at this level, but knowing the term is useful for later study.

Common Misconceptions
Misconception 1
Adding indexes always makes queries faster.

Indexes improve performance when a query filters by an indexed column and the index is selective, meaning it narrows the results significantly. If a column has very few distinct values and a query is expected to return most of the table, the database optimizer may choose not to use the index at all and simply scan the table instead. Indexes also slow down insertions, updates, and deletions. Indexing every column is not good design.

Misconception 2
Indexes are only needed for very large databases.

The performance benefit of indexes depends on table size, query frequency, and data volume growth. Even a moderately sized table can become slow if a column is searched frequently without an index. Thinking about likely search patterns during initial design is valuable for any database expected to grow.

Misconception 3
The database always uses an index if one exists.

The database optimizer decides whether to use an index based on its estimate of the cost of using it versus the cost of a full table scan. If using an index would require retrieving almost all rows anyway, the optimizer may decide a full scan is more efficient. The optimizer is usually right, but inspecting query plans can reveal when indexes are not being used as expected.

Misconception 4
Primary key indexes must be created manually.

Most database systems, including SQLite and PostgreSQL, automatically create an index on a table's primary key. This is why primary key lookups and joins on primary keys are fast by default. Foreign key indexes, however, often require manual creation.

Misconception 5
Performance is something to fix after launch.

Retrofitting indexes into a poorly designed database after it is in production is harder, more disruptive, and less effective than thinking about likely query patterns and data volumes during the initial design. Performance is a design concern.

  • E-Commerce Orders. An online store has an Orders table that grows by 30,000 rows per day. Customer service staff look up orders by customer email and by order number. Analysts run weekly reports filtered by order date and status. The database team is concerned about write performance, since the table receives many insertions each day. How would you think about this tradeoff?
  • University Enrollment. A university enrollment system stores 5 million enrollment records accumulated over decades. The most common queries filter by student_id, by course_id, and by term_id. Some queries filter by all three columns together. What indexes would you recommend, and why?
