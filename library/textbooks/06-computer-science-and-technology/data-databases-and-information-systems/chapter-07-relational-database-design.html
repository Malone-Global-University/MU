Relational Database Design

A relational database is not just a collection of tables. It is a designed structure where each table holds one kind of thing, tables connect through shared key values, and the whole system is built to store information cleanly, change reliably, and answer real questions without confusion. This chapter brings together the entities, attributes, and keys from Chapters 5 and 6 and shows how they compose into a working relational database.

Chapters 5 and 6 built the vocabulary of data modeling: entities, attributes, relationships, primary keys, foreign keys, and constraints. This chapter puts those pieces together into a complete picture of relational database design.

The relational model is the organizing principle behind nearly every major database system in use today. Understanding it is not just technical background. It is the foundation for every design decision you will make, every query you will write, and every problem you will diagnose across the rest of this course.

This chapter begins with the model itself and then moves through its practical consequences: how to separate kinds of data into tables, how tables connect through relationships, how many-to-many relationships are handled through junction tables, how to recognize and avoid the trap of repeated groups, and how to design a database so that the queries it will need to answer are natural rather than forced. It closes with an extended exercise in reading a complete relational schema, which is one of the most important skills a database student can develop.

Building on Chapter 5 and Chapter 6

Chapter 5 introduced entities, attributes, and relationships at a conceptual level. Chapter 6 introduced primary keys, foreign keys, and constraints as the mechanisms that enforce those relationships inside a database. Chapter 7 shows how these elements work together in a complete relational design. Chapter 8 will extend this work further into normalization.

7.1

What the Relational Model Is

The relational model is a way of organizing data into tables, where each table holds one kind of thing and tables connect to each other through shared values called keys.

The relational model was introduced in 1970 by a computer scientist named E.F. Codd, who worked at IBM. Codd observed that the software systems of his day stored data in ways that were tangled, redundant, and hard to change without breaking something. He proposed a clean mathematical foundation built on a concept from logic called a relation, which is essentially a set of tuples, meaning a set of rows where each row represents one instance of something.

In practice, a relational database is a collection of tables. Each table has a name, a fixed set of columns with defined types, and any number of rows. Every row in a table represents one record. No two rows in the same table are identical in a properly designed database, because every table has a primary key that makes each row uniquely identifiable.

Tables do not store their connections to other tables as physical links or pointers. Instead, relationships between tables are expressed through matching values. When a row in one table refers to a row in another table, it does so by recording the primary key of that other row in its own column. That column is called a foreign key, and it is the mechanism by which relational databases connect separate tables.

Two Properties That Matter

This design gives the relational model two important properties. First, data is organized clearly and consistently, because each kind of thing has its own table and its own columns. Second, data can be retrieved flexibly, because SQL queries can combine tables in any direction by matching key values, without the database needing to know in advance how any particular query will ask.

The relational model is the dominant approach to database design today. Nearly all major database systems, including SQLite, PostgreSQL, MySQL, SQL Server, and Oracle, are relational. Understanding the relational model is foundational to working with almost any database in any professional context.

7.2

Tables as Relations

In the relational model, a table is a relation. This word has a precise mathematical meaning, but students can understand it well enough through its practical properties.

A table is a collection of rows, all sharing the same structure. Every row has the same columns. Every column holds values of the same type in every row. The order of rows does not carry meaning. The order of columns does not carry meaning. There are no hidden positions or pointers. All the data in a row is right there in its columns.

Here is a simple example. Consider a table called students.

Table 7.1 — The students table. Each row is one student; each column is one kind of information.
student_id first_name last_name email enrollment_date
1001AmaraOseiamara.osei@mgu.edu2023-09-01
1002PriyaNairpriya.nair@mgu.edu2023-09-01
1003TomásReyestomas.reyes@mgu.edu2024-01-15
1004FatouDiallofatou.diallo@mgu.edu2024-01-15

This table has five columns: student_id, first_name, last_name, email, and enrollment_date. It has four rows. Each row represents one student. The student_id column is the primary key, meaning it uniquely identifies each row.

Notice several things about this table. Every column holds one kind of value. The student_id column holds integers. The name columns hold text. The enrollment_date column holds dates. There is no mixing of types within a column. No row repeats another row. And crucially, the table does not try to store anything other than student information. It does not include course details, grade records, or instructor assignments. Those things belong in other tables.

The First Principle of Relational Design

Each table stores one kind of thing. A students table stores student information. A courses table stores course information. An enrollments table stores the fact of enrollment. Mixing different kinds of information into one table is where most relational design problems begin.

7.3

Schemas

A schema is the structure of a database, described independently of any particular data. If a database is a building, the schema is the blueprint. The blueprint describes how many rooms there are, where the doors connect, and what each room is used for, but it does not describe where the furniture is on any particular day.

A database schema specifies the tables in the database, the columns in each table, the data type of each column, the primary key of each table, and the foreign keys that connect tables to each other. It may also specify constraints such as required fields, uniqueness rules, and validation conditions.

Two Ways to Write a Schema

A schema can be expressed in different forms. One common form is as SQL CREATE TABLE statements, which define each table formally in the language the database understands. Another common form is as a diagram showing tables as boxes and relationships as labeled lines connecting them. A schema diagram is sometimes called an entity-relationship diagram or a database diagram, though these terms are used with varying precision across teams and tools.

A third form, useful for learning and discussion, is plain language: a written description of each table, its columns, its key, and its connections to other tables. All three forms convey the same information. The SQL form is precise and executable. The diagram form is visual and fast to scan. The plain-language form is readable by anyone, including non-technical stakeholders.

A Simple Schema in Plain Language

Here is a small schema for a course enrollment system, described in plain language.

The students table stores one row per student, with columns for student ID, first name, last name, email address, and enrollment date. The student ID is the primary key.

The courses table stores one row per course, with columns for course ID, course name, department code, and credit hours. The course ID is the primary key.

The enrollments table records which students are enrolled in which courses, with columns for enrollment ID, student ID, course ID, and enrollment date. The enrollment ID is the primary key. The student ID is a foreign key referencing the students table. The course ID is a foreign key referencing the courses table.

This schema describes three tables, their columns, their keys, and their relationships. It answers a structural question, not a data question. It tells us what the database looks like, not who is enrolled or which courses exist today.

7.4

Separating Kinds of Data

One of the most important habits in relational database design is learning to separate different kinds of data into different tables. This sounds obvious when stated plainly, but it is frequently violated in practice, especially by designers who are used to thinking in spreadsheets.

The reason to separate kinds of data is that different things change at different rates and for different reasons. Students change their email addresses. Courses change their credit hours. Instructors move between departments. If all of this information were stored in a single flat table, changing one piece of information about one thing might require changing many rows. If you change a course name in a flat table that has one row per enrollment, you must update every row that contains that course, across every student who ever enrolled. If you miss even one row, the data becomes inconsistent.

In a relational database, you change the course name exactly once, in the courses table. Every enrollment row that references that course does so through a foreign key, which still points to the same course row. No enrollment rows need to change. The connection is preserved automatically through the key relationship.

The Flat Approach and Its Problems

Here is what a flat, spreadsheet-style approach looks like. All information about students, courses, and enrollments is combined into a single table.

Table 7.2 — A flat table combining student, course, and enrollment data. This design creates serious consistency problems.

The database contains two different answers to a simple factual question. Suppose the course "Database Systems" is renamed "Foundations of Relational Databases." That change must be applied to rows 5001 and 5003. If only one is updated, the database now contains two names for the same course.

Single Source of Truth

The relational approach produces a database where each piece of information is stored in exactly one place. When that information changes, it changes in exactly one place. There is never any ambiguity about which copy is correct. This property is sometimes called a single source of truth, and it is one of the most important goals of good database design.

7.5

Relationships Between Tables

When two tables share a key relationship, they are said to be related. The nature of that relationship depends on how many rows in one table correspond to rows in the other table. Recall from Chapter 5 that there are three fundamental kinds of relationships. In a relational database, each is implemented differently.

One-to-One
A ←→ B

Each row in table A corresponds to at most one row in table B, and vice versa. Less common; used when splitting a table for access control or performance.

One-to-Many
A ←→ B, B, B…

Each row in table A can correspond to many rows in table B, but each row in B corresponds to only one row in A. The most common relationship type.

Many-to-Many
A, A… ←→ B, B…

Each row in A can correspond to many rows in B, and vice versa. Requires a junction table to implement.

One-to-One Relationships

A one-to-one relationship means each row in table A corresponds to at most one row in table B, and each row in table B corresponds to at most one row in table A.

One-to-one relationships are less common than other types, but they appear in certain situations. Sometimes a designer splits a table into two parts for practical reasons: perhaps some columns are accessed very frequently while others are rarely needed and are large in size, or perhaps one set of columns requires stricter access control than another.

As an example, a students table might hold basic identifying information for fast access, while a student_profiles table holds longer biographical fields, profile photos, and preference settings. Each student has exactly one profile, and each profile belongs to exactly one student. To implement this, the student_profiles table contains a student_id column that is both a foreign key referencing students and the primary key of student_profiles.

One-to-Many Relationships

A one-to-many relationship means each row in table A can correspond to many rows in table B, but each row in table B corresponds to at most one row in table A. This is the most common relationship type. An instructor can teach many courses. A department can have many instructors. A student can have many enrollments.

To implement a one-to-many relationship, the foreign key is placed on the many side. If an instructor teaches many courses, the courses table contains an instructor_id foreign key referencing the instructors table.

Tables 7.6 and 7.7 — Instructors and courses. The foreign key instructor_id in courses implements the one-to-many relationship.
instructor_idnamedepartment
I01Dr. MensahCS
I02Prof. KapoorLIB
course_idcourse_nameinstructor_id
CS101Database SystemsI01
CS102Introduction to WebI01
LIB201Research MethodsI02

Dr. Mensah teaches two courses. The instructor_id foreign key in courses creates this connection without duplicating any instructor data.

Many-to-Many Relationships

A many-to-many relationship means each row in table A can correspond to many rows in table B, and each row in table B can correspond to many rows in table A. Students enroll in many courses. Courses have many students. Books can have multiple authors. Authors can write multiple books.

Many-to-many relationships cannot be implemented directly with a single foreign key in either table. If you put a course_id column in the students table, you can only record one course per student. Neither direction works with a single foreign key. The solution is a junction table, which Section 7.6 covers in detail.

7.6

Junction Tables

A junction table is a table whose primary purpose is to represent a many-to-many relationship between two other tables. It holds at least two foreign keys, one pointing to each side of the relationship. Each row in the junction table records one connection between one row in the first table and one row in the second table.

Junction tables are sometimes called associative tables, link tables, or bridge tables. The naming varies across documentation and teams, but the concept is the same.

We have already seen one junction table: the enrollments table from Section 7.4, which connects students to courses. Let us look at a second example to reinforce the pattern.

Books and Authors

A library database needs to record which authors wrote which books. A book can have multiple authors, and an author can have written multiple books. This is a many-to-many relationship. The junction table is book_authors.

Lena Müller (A01) is listed as an author on two books. James Okafor (A02) also appears on two books. Introduction to Data Systems (B01) has two authors. Information Organization (B03) has two authors. The junction table handles all of this without any duplication in the books or authors tables.

Notice that author_order belongs on the junction table, not on either of the other two tables. It describes a property of the specific connection between a book and an author: whether that author is listed first, second, or third. It is not a property of the author alone or the book alone.

Junction Tables Can Carry Their Own Attributes

Any attribute that describes the connection between two things, rather than either thing itself, belongs on the junction table. The enrollment date in an enrollments table describes a specific enrollment event, not the student alone and not the course alone. The author order in a book_authors table describes the specific relationship between a book and an author. When you are unsure where an attribute belongs, ask: does this describe thing A, thing B, or the relationship between them?

Choosing a Primary Key for Junction Tables

Junction tables need a primary key like any other table. Two approaches are common.

The first approach uses a surrogate key, an independent integer column that has no natural meaning but uniquely identifies each row. This is simple and consistent with the surrogate key approach used in other tables.

The second approach uses a composite primary key made of the two foreign key columns together. In book_authors, the combination of book_id and author_id is unique, because a given author can be listed on a given book only once. So (book_id, author_id) can serve as the primary key.

Both approaches are valid. The composite key approach more explicitly enforces uniqueness on the relationship itself. For beginning designers, either is acceptable, but consistency within a project matters.

7.7

Spreadsheet Tables Versus Relational Tables

Students who have used spreadsheets before coming to this course often find relational design counterintuitive at first. In a spreadsheet, putting more information in more columns of the same sheet feels natural. Why break data into separate tables when everything could fit in one place?

This section compares both approaches directly, so students can see what relational design gains and what it asks in return.

Spreadsheet Approach

All related information lives in one worksheet. A row per enrollment includes the student name, email, student ID, course name, course code, department, instructor name, and instructor email together.

This is fast to start. You can see everything at once. But problems emerge quickly:

  • Changing a student email requires finding and updating every row that student appears in.
  • Deleting a course row also destroys the student data embedded alongside it.
  • Adding a new course before any students enroll has nowhere to go.
  • Counting students per course requires scanning many columns for repeated values.
Relational Approach

Students, courses, and enrollments each have their own table. No information is duplicated. Each table stores exactly what it needs to store.

  • Changing a student email is one update to one row in students.
  • Removing a course affects only courses and enrollments, not student data.
  • Adding a new course is inserting one row in courses, no students required.
  • Counting students per course is a simple aggregation on the enrollments table.
The Three Anomalies

The three problems named above, updating a value in multiple places, not being able to delete without destroying unrelated data, and not being able to add data that lacks a pair, are called update anomalies, deletion anomalies, and insertion anomalies. Chapter 8 names and formalizes these concepts in depth. For now, recognizing that flat tables produce all three is enough.

Relational design requires more up-front work. You must think carefully about how to separate kinds of data, design tables, choose keys, and define foreign key relationships. Queries that combine multiple tables require joins, which take time to learn. The trade is worth it. For data that grows over time, changes over time, and is used by many people, the relational model produces databases that are correct, flexible, and maintainable in ways that flat spreadsheets are not.

Spreadsheets remain useful for small, personal, one-time tasks where data quality and consistency are not critical. When data becomes important, shared, or complex, relational design is the better foundation.

7.8

Avoiding Repeated Groups

One specific pattern that damages relational design is the repeated group. A repeated group occurs when a table includes multiple columns that represent the same kind of thing, repeated across columns rather than across rows.

Here is an example. Suppose a designer builds this students table.

Table 7.11 — A table with repeated groups. The course_1 / grade_1 pattern repeats three times across columns. This is a design error.
student_id first_name course_1 grade_1 course_2 grade_2 course_3 grade_3
1001AmaraCS101B+CS102A
1002PriyaCS101A−
1004FatouLIB201BCS101ACS102B+

This design creates serious problems. The table artificially limits students to three courses. A student who enrolls in a fourth course cannot be accommodated without adding more columns, which is a structural change that affects every row and every query. Most rows have empty columns: Amara and Priya leave several cells blank. And queries become awkward: finding all students in CS101 now requires checking three columns instead of one, with a condition like course_1 = 'CS101' OR course_2 = 'CS101' OR course_3 = 'CS101'. As the number of course columns grows, the query grows proportionally.

The Relational Fix

The fix is to separate student information, course information, and grade information into their own tables, using a junction table to represent which students are enrolled in which courses and what grade each received.

Table 7.12 — The corrected design. Every enrollment is one row. Adding a fourth or tenth course for a student is adding one row, not adding columns.
enrollment_idstudent_idcourse_idgrade
50011001CS101B+
50021001CS102A
50031002CS101A−
50041004LIB201B
50051004CS101A
50061004CS102B+
The Numbered Column Warning

When you find yourself numbering column names, course_1, course_2, course_3, stop. Those repeated columns almost certainly belong in separate rows of a related table. Numbered column names are nearly always a sign of a repeated group design error.

7.9

Designing for Reliable Queries

Good relational design makes queries easier to write and easier to get right. Poor relational design makes queries harder to write, more error-prone, and slower to execute.

When designing a database, it helps to think ahead about the questions the database will need to answer. Not every possible query needs to be anticipated, but common queries should be easy to express in SQL.

Here are some questions a student enrollment database might need to answer.

Which courses is a given student enrolled in? This query joins students to enrollments to courses.

How many students are enrolled in each course? This query groups the enrollments table by course_id and counts.

Which students are enrolled in more than three courses this term? This query aggregates enrollments by student and filters on the count.

Which instructors are teaching courses with more than twenty students enrolled? This query joins instructors, courses, and enrollments, groups by instructor, and filters on the enrollment count.

Are there any courses with no students enrolled? This query uses an outer join or subquery to find course rows with no matching rows in enrollments.

Notice that all of these questions are natural and readable when the database is properly designed with separate tables and clear foreign key relationships. The SQL statements flow directly from the table structure. Now consider what those same questions would look like against the flat spreadsheet table from Section 7.7. Questions about courses require scanning across multiple columns. Counting becomes awkward. Instructor data embedded alongside student data makes separation-dependent questions difficult.

Design Checklist for Query Reliability

Before finalizing a schema, ask: Are different kinds of things in different tables? Are table and column names clear enough to read in a SQL query? Are foreign keys defined for every relationship? Are there any repeated groups? Are the questions the system must answer easy to express as SQL against this design? A schema that passes all five checks will serve its queries well.

7.10

Reading a Relational Schema

Reading a schema is a skill that students should practice before they design schemas of their own. A schema is a map of a database's structure. Being able to read one accurately is essential for writing correct queries, designing extensions to existing systems, and identifying design problems before they become data problems.

Here is a full schema for a small library resource database. We will read it together and answer questions about it.

Reading the Schema Together

Let us practice reading this schema by answering questions about it.

How many tables does this database have?

Six: subjects, resources, authors, resource_authors, users, and loans.

Which table connects resources to their subject category?

Resources are connected to subjects directly through the subject_id foreign key in the resources table. This is a one-to-many relationship: one subject can have many resources, but each resource belongs to one subject.

What kind of relationship exists between resources and authors?

Many-to-many. A resource can have multiple authors, and an author can have contributed to multiple resources. The resource_authors table is the junction table that implements this relationship.

What does the author_order column in resource_authors represent?

It records the order in which authors are listed for a given resource. An author_order of 1 means the author is listed first. This attribute belongs on the junction table because it describes the specific relationship between a resource and an author, not the author alone or the resource alone.

What does a null return_date in the loans table mean?

A null return_date means the resource has not yet been returned. This is an intentional use of null to represent an ongoing state. When a resource is returned, the return_date is updated with the date of return. A query looking for currently borrowed resources filters for rows where return_date IS NULL.

Which table would you query to find all resources borrowed by a specific user?

The loans table, filtered by user_id. To also get the resource titles, you would join loans to resources on resource_id.

How would you find all authors of a specific resource?

Join resources to resource_authors on resource_id, then join resource_authors to authors on author_id.

Why Schema Reading Matters

Students who can read schemas confidently can understand other people's database designs, work with legacy systems, plan extensions to existing databases, and write correct SQL without confusion about where data lives. Schema reading is not just preparation for writing queries. It is the foundation for all design work that follows in this course.

Chapter Summary
  • The relational model organizes data into tables, where each table holds one kind of thing and tables connect through shared key values. Introduced by E.F. Codd in 1970, it remains the foundation of nearly all major database systems today.
  • A table is a relation: a collection of rows sharing the same structure, where row order and column order carry no meaning, and data is connected through values rather than physical links.
  • A schema describes the structure of a database, specifying tables, columns, types, primary keys, foreign keys, and constraints, independently of any particular data it contains.
  • The most important habit in relational design is separating different kinds of data into different tables. This eliminates redundancy, produces a single source of truth, and enables changes to be made in exactly one place.
  • One-to-many relationships are implemented by placing the foreign key on the many side. Many-to-many relationships are implemented through a junction table that holds foreign keys to both sides of the relationship.
  • Junction tables can carry their own attributes: data that describes the connection between two things rather than either thing itself.
  • Flat tables create update anomalies, insertion anomalies, and deletion anomalies. Relational design eliminates these by storing each piece of information in exactly one place.
  • Repeated groups, where multiple columns represent the same kind of thing numbered across the table, are a design error. That data belongs in rows of a related table instead.
  • Good relational design makes queries natural to write. Thinking ahead about the questions a database must answer helps verify that the design supports them before any data is entered.
  • Reading a relational schema accurately, tracing foreign key connections, and understanding what each column represents are core skills for every database student.
Key Terms
Relational model

A way of organizing data into tables where each table holds one kind of thing and tables connect through shared key values. Introduced by E.F. Codd in 1970 and the basis of nearly all major database systems today.

Relation

In the mathematical sense, a set of tuples sharing the same attributes. In practice, a table in a relational database.

Schema

The structure of a database, describing its tables, columns, data types, keys, and constraints, independently of any particular data stored in it.

Schema diagram

A visual representation of a database schema, showing tables as boxes and relationships as labeled lines between them.

Single source of truth

The principle that each piece of information should be stored in exactly one place, so there is never ambiguity about which copy is correct when values change.

One-to-one relationship

A relationship in which each row in table A corresponds to at most one row in table B, and each row in table B corresponds to at most one row in table A.

One-to-many relationship

A relationship in which each row in table A can correspond to many rows in table B, but each row in table B corresponds to at most one row in table A. Implemented by placing the foreign key on the many side.

Many-to-many relationship

A relationship in which each row in table A can correspond to many rows in table B, and vice versa. Implemented through a junction table.

Junction table

A table that implements a many-to-many relationship by holding foreign keys to both related tables. Also called an associative table, link table, or bridge table. May carry its own attributes that describe the relationship itself.

Composite primary key

A primary key made of two or more columns together, where the combination of those columns is unique across all rows in the table.

Repeated group

A design error where multiple columns represent the same kind of thing repeated across columns rather than across rows. Repeated groups should be replaced by rows in a related table.

Update anomaly

A problem that occurs when the same information is stored in multiple places and updating one copy without updating all others creates inconsistency.

Insertion anomaly

A problem that occurs when data cannot be added to a flat table without also adding unrelated data that does not yet exist.

Deletion anomaly

A problem that occurs when deleting one piece of information from a flat table unintentionally destroys other information that was stored in the same row.

Practice 7.A
Identify the Relationship Type

For each pair of things below, identify whether the relationship between them is one-to-one, one-to-many, or many-to-many. Write a brief explanation of your reasoning.

  • Students and MGU student ID cards. (Each student has exactly one student ID card, and each card belongs to exactly one student.)
  • Instructors and courses. (An instructor can teach more than one course, but each course section has exactly one instructor.)
  • Books and authors. (Books can have multiple authors, and authors can write multiple books.)

Coming Next — Chapter 8: Normalization at an Introductory Level

Chapter 7 established the principles of relational design: separating kinds of data into tables, connecting tables through keys, and using junction tables for many-to-many relationships. Chapter 8 will build on this foundation by introducing normalization, a more systematic process for identifying and eliminating specific types of design problems.

Normalization begins with a concrete question: what goes wrong when data is duplicated across rows? Chapter 8 will name these problems precisely, update anomalies, insertion anomalies, and deletion anomalies, and introduce formal guidelines called normal forms that tell designers when a table is clean and when it still has problems to solve. The chapter walks through the first three normal forms at an introductory level, using before-and-after table designs so students can see both the problem and the solution clearly.

The goal of normalization is not to follow abstract rules for their own sake. It is to produce databases that store information cleanly, update reliably, and query correctly. Chapter 8 will show how formal normalization supports the practical design instincts built in Chapter 7.
