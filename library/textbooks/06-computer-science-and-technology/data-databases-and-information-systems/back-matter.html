Appendix A

Data Vocabulary Reference

Data is the raw material of information systems. Before students can design databases, write queries, or build reports, they need to share a precise vocabulary for talking about what data is, how it is organized, and why it must be managed with care. This appendix defines the core vocabulary of data and information, organized in the order students are most likely to encounter these ideas.

A.1

Data

Data refers to raw, recorded facts about the world. A measurement, a name, a date, a count, a code, a price, a temperature, a file size, or a checkbox value are all data. Data does not yet carry meaning by itself. The name "Martinez" is data. The number 42 is data. The date 2024-09-01 is data. Each piece of data becomes meaningful only when it is placed in context, organized with other data, or used to answer a question.

Data can be structured or unstructured. Structured data follows a defined format, such as a table with named columns and typed values. Unstructured data, such as a written paragraph, an image, or an audio file, does not follow a fixed format. Most database systems are designed to manage structured data.

Data can also be quantitative or qualitative. Quantitative data consists of numbers that can be measured or counted. Qualitative data describes categories, labels, states, or qualities that are not naturally numeric.

Key Caution

Data is not automatically trustworthy. Data can be wrong, incomplete, inconsistent, outdated, or duplicated. Good data management requires checking data before trusting it.

A.2

Information

Information is data that has been organized, labeled, contextualized, or summarized so that it carries meaning for a specific purpose. A column of numbers labeled "Enrollment Count by Department" is information. A raw list of numbers without labels is merely data.

The transformation from data to information requires structure, context, and human or computational interpretation. Information is always produced for someone. Designing information systems well requires asking: who needs this information, what question does it answer, and what decisions does it support?

A.3

Knowledge

Knowledge is the understanding a person builds from information over time. It includes patterns recognized across many observations, rules learned through experience, and the ability to apply information to new situations. Databases and information systems store data and produce information. They do not store knowledge directly, though they can preserve the rules and patterns that embody organizational knowledge in the form of constraints, workflows, and documented procedures.

A.4

Records

A record is a single organized collection of related fields that describes one instance of something. In a relational database table, a record corresponds to one row. A student record contains all the stored information about one student. Each record in a table should represent exactly one thing of the same kind.

A.5

Fields

A field is a single named unit of data within a record, corresponding to one column within one row. Every field has a name, a data type, and a set of rules governing what values it may hold. Good field names are specific enough to be unambiguous and short enough to be readable in queries. A field named Date is ambiguous. A field named EnrollmentDate is clear. Fields may be required or optional.

A.6

Tables

A table is a structured collection of records organized into named columns. Every row in the table is a record. Every column is a field. All records in the same table follow the same column structure. A well-designed table stores information about one kind of thing and nothing else.

Tables vs. Spreadsheets

A spreadsheet row can contain formulas, merged cells, and mixed content. A database table row is a pure data record subject to strict type and constraint rules. A spreadsheet can change its structure row by row. A database table enforces the same structure for every row.

A.7

Rows

A row is a single record in a table. The terms row and record are used interchangeably in relational database work. A table with five hundred students has five hundred rows. Adding a new student means inserting a new row. Removing a student means deleting a row.

A.8

Columns

A column is a named field that appears in every row of a table. All values in a column share the same data type and follow the same rules. Columns have fixed positions in a table's structure, though queries can select, reorder, and rename columns as needed.

A.9

Data Types

A data type specifies what kind of value a field may hold, how the database stores the value, how much space the value takes, and what operations are valid on the value. Every column in a relational database has a defined data type.

Common data types in relational databases
CategoryExamplesUsed for
TEXTVARCHAR, TEXT, CHARNames, codes, descriptions, identifiers with letters
INTEGERINTEGER, INT, SMALLINTWhole counts, IDs, year values, quantities
DECIMALDECIMAL(p,s), NUMERIC, REALPrices, percentages, averages, measurements
DATE / TIMEDATE, TIME, DATETIME, TIMESTAMPCalendar dates, clock times, event timestamps

Choosing the correct data type for each field is an important part of database design. Storing a phone number as an integer causes problems because leading zeros may be dropped and arithmetic on phone numbers is meaningless.

A.10

Identifiers

An identifier is a value or set of values that uniquely distinguishes one record from all other records in the same table. Every table should have a way to uniquely identify each of its records. The most common identifier pattern is a primary key, which holds a unique, non-null value for every row and should never change once assigned. A surrogate key is an artificial identifier (usually an auto-generated integer) with no real-world meaning. A natural key is drawn from real-world data, such as a library call number.

A.11

Missing Values

A missing value in a relational database is represented by the special marker NULL. NULL means that a value is absent: not recorded, not applicable, or not yet known. Comparing a NULL value using standard equality operators does not work as expected. SQL provides IS NULL and IS NOT NULL for testing absence. Aggregate functions such as AVG and SUM skip NULL values automatically. Missing values can signal data quality problems.

A.12

Validation

Validation is the process of checking that data values conform to defined rules before they are accepted and stored. Validation can be performed at the database level through constraints, at the application level through form checks, or at both levels. Validation rules include type checks, range checks, format checks, presence checks, and referential checks. Strong validation catches errors early, before they become embedded in the database.

A.13

Data Quality

Data quality is a measure of how well data serves its intended purpose. A high-quality dataset is accurate (values are correct), complete (required values are present), consistent (values agree across records and tables), valid (values conform to defined rules), timely (values are current enough for their purpose), and unique (records are not duplicated). Improving data quality requires both technical controls and organizational habits.

Appendix B

Table and CSV Reference

Tables and comma-separated value files are two of the most important structures students encounter when working with data. This appendix describes how both are organized, how they differ, and how data moves between them.

B.1

Table Structure

A relational database table is a rectangular structure organized into named columns and typed rows. Every row follows the same column structure. Every column holds values of the same data type. The table as a whole represents one kind of thing. A table is defined by its schema, specifying column names, data types, constraints, and keys. The schema is set when the table is created with CREATE TABLE and governs every row added afterward. Rows have no guaranteed order unless a query explicitly sorts them.

B.2

Rows and Columns

When designing a table, every column should represent one atomic fact about the thing the table describes. A column called FullName that stores both first and last names is harder to sort and search than separate FirstName and LastName columns. A column storing a comma-separated list of course names violates the principle that each field holds one value and makes joining impossible.

B.3

Headers

Headers are the column names that label each column in a table or CSV file. Good headers are descriptive, consistent, and free of spaces when used in database systems. In a CSV file, headers appear on the first line. Every subsequent line contains one record with values in the same order as the headers.

B.4

CSV Format

A CSV file (comma-separated values) is a plain text file in which each line represents one record and values within each line are separated by commas. CSV is one of the most widely used formats for exchanging structured data between systems.

CSV — Example File
StudentID,FirstName,LastName,EnrollmentDate
1001,Amara,Okonkwo,2023-09-01
1002,Diego,Reyes,2023-09-01
1003,Priya,Sharma,2022-01-15

CSV is not perfectly standardized, which causes problems when files are exchanged between systems. Some files use tabs instead of commas. Some tools enclose all values in quotation marks. Line endings may differ between operating systems.

B.5

Delimiters

A delimiter is the character that separates values within a record in a text-based data file. The comma is most common, but tab-delimited (TSV) and pipe-delimited files are also used. When a value contains the delimiter character, CSV handles this by enclosing the entire value in double quotation marks. A value like "Portland, OR" must be quoted or the embedded comma will be misread as a column separator.

B.6

Importing Data

Importing data means reading data from an external file into a database or data tool. Before importing, check: Does the file have headers? Does the column order match the table? Does each value match the data type of the corresponding column? Are there missing, malformed, or out-of-range values? Importing without checking can produce bad data silently because import tools check data types and structure, not business rules.

B.7

Exporting Data

Exporting data means writing data from a database to an external file. When exporting, consider whether to include headers (almost always yes), which columns to include, whether to filter records, and whether sensitive data needs to be excluded or anonymized before sharing.

B.8

B.9

Spreadsheet-to-Database Issues

Moving data from spreadsheets to databases often reveals hidden problems. Spreadsheet rows may mix student records, summary rows, blank separators, and subtotals. A database table cannot accept this mixture. Spreadsheet columns may hold concatenated values that must be split. Date values vary by system. Numeric values may include currency symbols. Cells containing formulas must be saved as values before import.

B.10

Data Exchange Checklist

  • The file encoding is UTF-8 or another encoding both systems support.
  • The file has a header row with clear, consistent column names.
  • The delimiter is known and matches what the receiving system expects.
  • Values containing the delimiter character are properly quoted.
  • Date values follow ISO 8601 format (YYYY-MM-DD).
  • Numeric values are plain numbers without currency symbols or thousands-separator commas.
  • The column order matches the expected column order, or the import is configured to match by name.
  • Required fields have values; optional fields use empty strings or explicit NULL markers consistently.
  • The file has been reviewed for blank lines, extra columns, truncated rows, or unexpected characters.
  • A small sample of the imported data has been reviewed after import to confirm correctness.
Appendix C

Data Modeling Reference

Data modeling is the process of deciding what a database will represent, how the things it represents relate to each other, and what rules govern the data. A good data model is the foundation of a reliable database. A weak data model produces a system that is hard to query, hard to maintain, and likely to store inconsistent or duplicated data.

C.1

Entities

An entity is a distinct kind of thing about which a system needs to store information. Entities become tables in a relational database. A student is an entity. A course is an entity. A library book is an entity. Identifying entities is the first step in data modeling. Look for the nouns in a system description. Not everything is an entity: the title of a book is an attribute of the Book entity, not a separate entity. An entity becomes a candidate for its own table when it has multiple attributes of its own, or when many other entities refer to it.

C.2

Attributes

An attribute is a piece of information stored about an entity. Attributes become columns in the table that represents the entity. Attributes should be atomic, meaning each holds one piece of information, not a combination. Storing only the attributes the system actually uses reduces unnecessary complexity and avoids storing data that cannot be kept accurate.

C.3

Relationships

A relationship describes how two entities are connected. "A student enrolls in courses" describes a relationship between Student and Course. Relationships have a cardinality, which describes how many instances of one entity can be associated with how many instances of another, and an optionality, describing whether the association is required or not.

C.4

Cardinality

The three cardinality types in relational modeling
TypeMeaningImplementationExample
One-to-OneOne A → at most one B, and vice versaForeign key in either table; uncommonOne student ↔ one library card
One-to-ManyOne A → many B; each B → one AForeign key on the many sideOne department → many students
Many-to-ManyOne A → many B; one B → many AJunction table requiredStudents ↔ Courses via Enrollments
C.5

Optional and Required Relationships

An optional relationship means participation is not mandatory: a book may have no reviews yet. A required relationship means at least one association must exist: every enrollment must refer to a valid student and a valid course. Required relationships are enforced through NOT NULL constraints on foreign keys and referential integrity rules.

C.6

Entity-Relationship Diagrams

An entity-relationship diagram (ERD) is a visual representation of entities, their attributes, and the relationships between them. ERDs communicate data model structure to both technical and non-technical stakeholders. This textbook uses a simplified crow's foot notation. Each box is an entity. Lines connect related entities. Symbols at line ends indicate cardinality and optionality. Attributes are listed inside entity boxes.

Reading an ERD

A single vertical bar ( | ) at a line end means "exactly one." A crow's foot ( << ) means "many." A circle ( ○ ) means "zero or one" (optional). Combine them: a crow's foot with a circle means "zero or many"; a crow's foot with a bar means "one or many."

C.7

Schemas

A database schema is the complete structural definition of a database: all table definitions, column names, data types, constraints, primary keys, foreign keys, and relationships. A schema is the blueprint of the database and is distinct from the data the database contains. Schema diagrams show tables as boxes with column names listed inside, connected by lines representing foreign key relationships.

C.8

Junction Tables

A junction table (also called an associative table, bridge table, or linking table) represents a many-to-many relationship. It contains at minimum the primary key of each connected entity and may also hold attributes specific to the relationship. For example, an Enrollments table connecting Students and Courses might also hold EnrollmentDate and Status.

SQL — Junction Table
CREATE TABLE Enrollments (
    EnrollmentID   INTEGER PRIMARY KEY AUTOINCREMENT,
    StudentID      INTEGER NOT NULL REFERENCES Students(StudentID),
    CourseID       INTEGER NOT NULL REFERENCES Courses(CourseID),
    EnrollmentDate DATE
);
C.9

Design Review Checklist

  • Every entity has a clear, specific purpose and represents exactly one kind of thing.
  • Every attribute is atomic and belongs to the correct entity.
  • Every relationship has a defined cardinality and optionality.
  • Every many-to-many relationship is resolved with a junction table.
  • Every entity has a primary key.
  • Every foreign key references the correct primary key in the correct table.
  • The model can answer the questions the system is required to answer.
  • Sample data has been sketched for at least two or three records per table to confirm the model works.
  • No entity stores information that belongs in a separate table.
  • The diagram has been reviewed by someone who understands the real-world system being modeled.
Appendix D

Keys and Constraints Reference

Keys and constraints are the mechanisms that make databases reliable. Without them, records can be duplicated, references can break, and invalid values can accumulate silently. This appendix defines every key type and constraint used in the textbook.

D.1

Primary Keys

A primary key is a column or combination of columns that uniquely identifies every row in a table. Primary key values must be unique, must never be NULL, and should ideally be stable after assignment. When a primary key consists of two or more columns together it is called a composite primary key, common in junction tables.

SQL — Primary Key Declaration
CREATE TABLE Students (
    StudentID      INTEGER PRIMARY KEY AUTOINCREMENT,
    FirstName      TEXT    NOT NULL,
    LastName       TEXT    NOT NULL,
    EnrollmentDate DATE
);
D.2

Foreign Keys

A foreign key is a column in one table that references the primary key of another table, creating the connections that make databases relational. A foreign key ensures that a value stored in one table actually exists in the table it references. Foreign keys also enforce referential integrity: the rule that references between tables must remain valid.

SQL — Foreign Key Declaration
CREATE TABLE Enrollments (
    EnrollmentID  INTEGER  PRIMARY KEY AUTOINCREMENT,
    StudentID     INTEGER  NOT NULL REFERENCES Students(StudentID),
    CourseID      INTEGER  NOT NULL REFERENCES Courses(CourseID),
    EnrollmentDate DATE
);
D.3

Candidate Keys

A candidate key is any column or combination of columns that could serve as a valid primary key: unique across all rows and never NULL. A table may have more than one candidate key. The designer chooses one as the primary key; others become alternate keys enforced with UNIQUE constraints.

D.4

Natural Keys and Surrogate Keys

A natural key is a primary key drawn from real-world data that already exists and carries meaning, such as an ISBN or a national ID number. Natural keys can be meaningful but may change, may not be guaranteed unique in all cases, and may raise privacy concerns. A surrogate key is an artificial, auto-generated identifier with no real-world meaning, typically an integer. Most professional designs use surrogate keys as primary keys and enforce natural keys with UNIQUE constraints.

D.5

Uniqueness Constraints

A UNIQUE constraint enforces that all values in a column or combination of columns are distinct across all rows. Unlike a primary key, a unique constraint can allow NULL values in most database systems. Use UNIQUE for fields that are not the primary key but must still not repeat, such as email addresses or usernames.

D.6

Required Fields and Referential Integrity

A NOT NULL constraint enforces that a column must always contain a value. Referential integrity means every foreign key value either refers to an existing primary key or is NULL. When referential integrity is enabled, the database prevents inserting rows with non-existent foreign key values, deleting referenced rows, and updating primary keys that are referenced elsewhere. Referential integrity can be configured with RESTRICT, CASCADE, SET NULL, or SET DEFAULT behaviors on delete and update events.

D.7

Constraint Examples

SQL — Constraint Reference Examples
-- Primary key with autoincrement
StudentID INTEGER PRIMARY KEY AUTOINCREMENT

-- Required field
LastName TEXT NOT NULL

-- Unique alternate key
Email TEXT NOT NULL UNIQUE

-- Foreign key with referential integrity
CourseID INTEGER NOT NULL REFERENCES Courses(CourseID)

-- CHECK constraint for valid range
CreditHours INTEGER NOT NULL CHECK (CreditHours BETWEEN 1 AND 6)

-- Default value
EnrollmentDate DATE NOT NULL DEFAULT CURRENT_DATE

-- Composite primary key in junction table
PRIMARY KEY (StudentID, CourseID)
Appendix E

Normalization Reference

Normalization is the process of organizing a relational database to reduce duplication and prevent update problems. This appendix explains the purpose of normalization, the three problems it solves, and the first three normal forms at an introductory level.

E.1

Repeated Data

Repeated data (redundancy) occurs when the same piece of information is stored in more than one place. Repetition creates maintenance problems because every copy must be updated whenever the information changes, and inconsistencies arise when some copies are updated but others are not. Normalization reduces repetition by storing each fact in exactly one place and using foreign keys to reference it.

E.2 – E.4

The Three Anomalies

The three data anomalies caused by poor normalization
AnomalyDefinitionExample
Update anomalyChanging a value requires updating multiple rows; inconsistency arises if any are missedInstructor email stored in every enrollment row; one row missed when email changes
Insertion anomalyCannot insert a record without also inserting other information not yet availableCannot add a student until enrolled in a course, if both share one table
Deletion anomalyDeleting one record unintentionally destroys other informationDeleting the last enrollment for a course also deletes the course itself
E.5

First Normal Form (1NF)

A table is in first normal form when every column holds atomic values (one value per cell), every row is unique with a primary key, and there are no repeating groups of columns. Violations include cells with comma-separated lists and multiple columns for the same kind of data (Phone1, Phone2, Phone3).

E.6

Second Normal Form (2NF)

A table is in second normal form when it is in 1NF and every non-key column depends on the entire composite primary key, not just a part of it. 2NF is only relevant for tables with composite primary keys. A column that depends on only one part of the composite key represents a partial dependency and must be moved to a separate table.

Example

An Enrollments table with composite key (StudentID, CourseID) that also stores CourseName violates 2NF. CourseName depends only on CourseID, not on the full combination. It belongs in the Courses table.

E.7

Third Normal Form (3NF)

A table is in third normal form when it is in 2NF and every non-key column depends directly on the primary key, not on another non-key column. A column that depends on another non-key column is a transitive dependency. A Students table storing both DepartmentID and DepartmentName violates 3NF because DepartmentName depends on DepartmentID, which is non-key. DepartmentName belongs in a Departments table.

E.8

Before-and-After Example

Before normalization — all data in one table
EnrollmentIDStudentIDStudentNameCourseIDCourseNameInstructorName
1001501Amara O.CS101Intro to CSDr. Patel
1002501Amara O.MA201Calculus IDr. Kim
1003502Diego R.CS101Intro to CSDr. Patel

After normalization, this becomes four clean tables: Students, Courses, Instructors, and Enrollments. Every fact is stored once. Changing Dr. Patel's name requires updating one row in the Instructors table. Deleting all CS101 enrollments leaves the course record intact.

E.9

Normalization Checklist

  • Every cell holds one atomic value. No lists, no concatenated values.
  • There are no repeating column groups.
  • Every row has a primary key.
  • In tables with composite keys: every non-key column depends on the full composite key, not a part.
  • No non-key column depends on another non-key column.
  • Every repeated fact has been moved to its own table and referenced by foreign key.
  • Sample data has been checked for update, insertion, and deletion anomalies.
Appendix F

SQL Syntax Reference

This appendix provides a concise reference for the SQL covered in this textbook. Examples use SQLite-compatible syntax. Most examples also work in PostgreSQL, MySQL, and SQL Server with minor adjustments.

F.1

SELECT and FROM

SQL — SELECT and FROM
-- Select specific columns
SELECT FirstName, LastName
FROM Students;

-- Select all columns
SELECT *
FROM Students;
F.2

WHERE

SQL — WHERE Clause
SELECT FirstName, LastName
FROM Students
WHERE EnrollmentDate >= '2023-09-01'
  AND Department = 'CS';

Use AND, OR, and NOT to combine conditions. Use IN for lists, BETWEEN for ranges, LIKE for text patterns, and IS NULL / IS NOT NULL for missing values.

F.3

ORDER BY, GROUP BY, HAVING

SQL — ORDER BY / GROUP BY / HAVING
-- Sort results
SELECT Title, PublicationYear
FROM Books
ORDER BY PublicationYear DESC;

-- Group and count
SELECT DepartmentID, COUNT(*) AS StudentCount
FROM Students
GROUP BY DepartmentID
HAVING COUNT(*) > 50;

WHERE filters rows before grouping. HAVING filters groups after aggregation. Do not swap them.

F.4

JOIN

SQL — INNER JOIN and LEFT JOIN
-- INNER JOIN: only matching rows from both tables
SELECT s.FirstName, s.LastName, c.CourseName
FROM Enrollments AS e
JOIN Students    AS s ON e.StudentID = s.StudentID
JOIN Courses     AS c ON e.CourseID  = c.CourseID;

-- LEFT JOIN: all rows from left table, NULLs for no match
SELECT s.FirstName, s.LastName, c.CourseName
FROM Students AS s
LEFT JOIN Enrollments AS e ON s.StudentID = e.StudentID
LEFT JOIN Courses     AS c ON e.CourseID  = c.CourseID;
F.5

Aliases, Subqueries, Views

SQL — Aliases, Subqueries, Views
-- Column alias
SELECT COUNT(*) AS TotalStudents FROM Students;

-- Subquery in WHERE
SELECT FirstName, LastName FROM Students
WHERE StudentID IN (
    SELECT StudentID FROM Enrollments WHERE CourseID = 'CS101'
);

-- Create a view
CREATE VIEW EnrolledStudents AS
SELECT s.FirstName, s.LastName, c.CourseName
FROM Enrollments AS e
JOIN Students AS s ON e.StudentID = s.StudentID
JOIN Courses  AS c ON e.CourseID  = c.CourseID;

-- Use the view
SELECT * FROM EnrolledStudents;
F.6

INSERT, UPDATE, DELETE

SQL — Data Modification
-- Insert a new row (always name the columns)
INSERT INTO Students (FirstName, LastName, EnrollmentDate)
VALUES ('Priya', 'Sharma', '2023-09-01');

-- Update specific rows (always include WHERE)
UPDATE Students
SET Email = 'priya.sharma@mgu.edu'
WHERE StudentID = 1003;

-- Delete specific rows (always include WHERE)
DELETE FROM Enrollments
WHERE StudentID = 1003 AND CourseID = 'CS101';
Safety Rule

An UPDATE or DELETE without a WHERE clause changes or removes every row in the table. Always include a WHERE clause and test it with a SELECT first.

F.7

Comments and Readable Formatting

SQL — Comments and Formatting
-- Single-line comment

/*
  Multi-line comment
  Author: Course team
*/

-- Readable formatting: uppercase keywords, aligned clauses
SELECT
    s.FirstName,
    s.LastName,
    c.CourseName,
    e.EnrollmentDate
FROM Enrollments AS e
JOIN Students    AS s ON e.StudentID = s.StudentID
JOIN Courses     AS c ON e.CourseID  = c.CourseID
WHERE e.EnrollmentDate >= '2023-09-01'
ORDER BY s.LastName, s.FirstName;

FROM Students     AS s

JOIN Courses      AS c ON e.CourseID  = c.CourseID;

SELECT StudentID, FirstName, LastName
FROM Students
WHERE Email IS NULL;

-- G.10: Report query — enrollment count by course and department
SELECT
    d.DepartmentName,
    c.CourseName,
    COUNT(e.StudentID) AS Enrolled
FROM Enrollments AS e
JOIN Courses     AS c ON e.CourseID     = c.CourseID
JOIN Departments AS d ON c.DepartmentID = d.DepartmentID
GROUP BY d.DepartmentName, c.CourseName
ORDER BY d.DepartmentName, Enrolled DESC;
Appendix H

Database Security Reference

Security is not optional in database work. Every database that stores real information about real people carries responsibility. This appendix summarizes the key security concepts covered in the textbook.

H.1 – H.3

Authentication, Authorization, and Roles

Authentication verifies identity before granting access: are you who you say you are? Strong authentication requires reasonably complex passwords, locked accounts after repeated failed attempts, and secure credential storage using hashing. Authorization determines what an authenticated user may do: now that we know who you are, what are you permitted to access or change? These are distinct steps; a user can be authenticated but still unauthorized for a specific action. A role is a named set of permissions assigned to users corresponding to job functions, making permission management consistent and auditable.

SQL — Granting and Revoking Permissions
GRANT SELECT ON Students TO role_instructor;
GRANT SELECT, INSERT, UPDATE ON Grades TO role_instructor;
REVOKE DELETE ON Students FROM role_instructor;
H.4 – H.5

Permissions and Least Privilege

A permission is a specific allowed action (SELECT, INSERT, UPDATE, DELETE, CREATE, DROP) on a specific database object. Review permissions regularly and revoke what is no longer needed. The principle of least privilege states that users and systems should have access to exactly what they need for their work, and nothing more. Applying least privilege reduces the damage that can result from a compromised account, a programming error, or an accidental action.

H.6 – H.8

Sensitive Data, Access Logs, and SQL Injection

Sensitive data includes medical records, financial information, government identification numbers, passwords, biometric data, and personal communications. Every sensitive field requires: encryption at rest and in transit, restricted access, audit logging, and clear retention and deletion policies. Access logs record who accessed the database, what actions they performed, and when, supporting security monitoring and forensic investigation. SQL injection is a security attack in which malicious SQL is inserted into input that is then executed by a database query. The defense is to always use parameterized queries, never concatenate user input into SQL strings, and never expose raw SQL errors to end users.

H.9 – H.11

Backups, Encryption, and Safe Practices

Backups should be taken regularly, stored separately from the live database, tested by actual restore drills, and retained according to a defined policy. Encryption at rest protects database files on disk. Encryption in transit (TLS/SSL) protects data transmitted over networks. Passwords must never be stored as plain text: always use a secure cryptographic hash.

Safe Database Practices Summary

Never expose database credentials in publicly visible code. Always use parameterized queries. Apply least privilege. Encrypt sensitive columns and connections. Log all access to sensitive tables. Test backups regularly. Review permissions when users change roles. Never run destructive commands without testing first. Document all security decisions.

Appendix I

Information Systems Reference

A database is one component of a larger information system. This appendix defines the components of information systems covered in the textbook.

People

The humans who design, operate, maintain, and use information systems, including designers, administrators, data entry staff, analysts, managers, and the individuals whose data is stored. Systems exist to serve people and depend on people to work correctly.

Processes

The workflows, procedures, and rules that govern how information flows through a system. Well-designed systems make processes clearer, faster, and more consistent. Poorly designed systems force workarounds and introduce errors.

Data

The recorded information the system stores, processes, and presents. Data quality determines the usefulness of the entire system. Even elegant software is undermined by inaccurate, incomplete, or inconsistent data.

Software

The code implementing the system: the DBMS, application server, web interface, reporting engine, validation logic, access control, and supporting libraries. Software connects people, processes, and data.

Hardware

The physical infrastructure on which software runs: servers, storage, networking equipment, desktops, and mobile devices. Cloud computing delegates hardware management to a provider but the physical infrastructure still exists.

Inputs

Data and commands that enter the system from outside: forms filled by users, files uploaded by staff, API calls from connected systems, and scheduled data imports. Input design and validation are critical data-quality control points.

Processing

What the system does with inputs: validating, transforming, storing, retrieving, applying business rules, generating derived values, and producing outputs. Processing is where the work of the information system happens.

Outputs

Results the system produces: reports, dashboards, notifications, data exports, API responses, and displayed records. Good outputs present information clearly and accurately in a form suited to the user's purpose.

Storage

Where the system persists data between sessions. Storage must be reliable, consistent, protected, and queryable. The database is the primary storage component in most information systems.

Feedback

Information a system provides about its own operation: error messages, confirmation messages, audit logs, performance metrics, and evaluation reports. Good feedback helps users correct mistakes and supports continuous improvement.

Workflows

Defined sequences of steps that process a unit of work from initiation to completion, with states, transitions, roles, and rules. Modeling workflows clearly before building systems prevents confusion and missed cases.

Forms

User interface components that collect input and submit it to the system. Forms must be designed with clear labels, appropriate input types, sensible defaults, immediate validation feedback, and accessible design.

Reports

Structured presentations of data retrieved from a database and formatted for a specific audience and purpose. Reports should be designed around the questions they answer and the decisions they support.

Dashboards

Visual displays combining multiple summary values, charts, filters, and indicators on one screen, designed for quick scanning of current-state information rather than deep reading.

Institutional Records

Official data an organization maintains about its activities, members, assets, and history. Subject to legal requirements regarding accuracy, retention, security, and access.

Appendix J

Systems Integration Reference

Modern information systems rarely exist in isolation. They exchange data with other systems, receive updates from external sources, and expose their data through programming interfaces. This appendix covers the integration concepts introduced in the textbook.

J.1 – J.5

APIs, Requests, Responses, Endpoints, Parameters

An API (application programming interface) is a defined way for one software system to communicate with another. Web APIs based on HTTP are the most common form. A request consists of an HTTP method (GET, POST, PUT, DELETE), a URL, optional parameters and headers, and sometimes a JSON body. A response includes an HTTP status code, optional headers, and usually a JSON body. An endpoint is a specific URL that accepts requests for a specific resource or action. Parameters refine requests: path parameters identify specific resources; query parameters provide filtering, sorting, or pagination instructions.

J.6

JSON

JSON (JavaScript Object Notation) is the most widely used format for web API responses and requests. JSON objects are key-value pairs in curly braces. Arrays are lists in square brackets. Values can be strings, numbers, booleans, objects, arrays, or null. String keys and string values are in double quotation marks. Numbers, booleans (true/false), and null are unquoted.

Appendix K

Backup, Recovery, and Maintenance Reference

A database that works today may fail tomorrow. Protecting data requires ongoing maintenance: backups, monitoring, documentation, and planning. This appendix summarizes the maintenance concepts covered in the textbook.

K.1 – K.3

Backup Types and Recovery

The three main backup strategies
TypeWhat it copiesRestore requirementTradeoff
Full backupEntire databaseThis backup aloneSlowest, largest; complete and self-sufficient
Incremental backupChanges since last backup (full or incremental)Last full + all incrementals in orderFastest, smallest per run; complex restore
Differential backupAll changes since last full backupLast full + last differentialGrows over time; simpler restore than incremental

Recovery time objective (RTO) is the maximum acceptable time to restore service after a failure. Recovery point objective (RPO) is the maximum acceptable data loss measured in time. A system with an RPO of one hour must take backups at least every hour. Restore testing means periodically loading a backup into a test environment to verify the database loads correctly and completely. A backup that has never been tested is not a reliable backup.

K.4 – K.9

Retention, Logs, Auditing, Monitoring, Maintenance, Migrations

A retention policy defines how long backup copies are kept. Balance storage cost against the need to recover from events not detected immediately. Transaction logs record every data modification and are the basis for point-in-time recovery. Slow query logs identify performance problems. Auditing is the systematic recording and review of access and modification events for security, compliance, and accountability. Audit logs must be protected from modification. Monitoring continuously observes system health and generates alerts when values exceed thresholds. Migrations are controlled schema changes that should be planned, tested in staging, and applied during a maintenance window with a documented rollback procedure.

K.10

Documentation and Disaster Planning

Good documentation includes an up-to-date schema diagram, descriptions of each table and column, a user and role list with permissions, backup and recovery procedures, maintenance schedules, and contact information for responsible staff. A disaster plan specifies what backups exist and where, the order of recovery steps, who is responsible for each step, expected recovery time, and how to communicate the situation to affected users. Systems without documentation and disaster plans may recover from failures but only slowly, expensively, and with greater risk of permanent data loss.

Appendix L

Data Ethics and Governance Reference

Data systems affect real people. The decisions made in designing, operating, and managing databases have consequences for privacy, fairness, safety, and trust. This appendix summarizes the ethics and governance concepts from the textbook.

L.1 – L.4

Privacy, Consent, Sensitive Data, Data Minimization

Privacy is the right of individuals to control information about themselves and to have it used only for purposes they have consented to. Privacy by design means treating privacy as a design requirement from the beginning, not as an afterthought. Consent must be informed and voluntary: people must understand what data is collected, why, how it is used, with whom it is shared, and how long it is retained. Sensitive data categories include health and medical information, financial accounts, government IDs, biometrics, location history, political opinions, sexual orientation, immigration status, criminal history, and children's data. Data minimization means collecting and retaining only what is genuinely necessary. Data that does not exist cannot be breached.

L.5 – L.7

Retention, Bias, Transparency

Good retention policies specify what categories of data are covered, how long each is retained, what triggers deletion, and how deletion is documented. Data systems can perpetuate or create unfair outcomes when training data reflects historical discrimination, when algorithmic decisions use proxy variables correlated with protected characteristics, or when data quality is worse for some populations. Students should ask: who might be harmed by this system? Are outcomes audited for fairness? Transparency means people understand how decisions affecting them are made. Explainability means specific decisions can be explained to the individuals affected in terms they can understand and challenge.

L.8 – L.10

Governance, Compliance, Risk Management

Data governance is the set of policies, roles, standards, and procedures defining how data is managed. It specifies who owns each category of data, who is responsible for quality, who can authorize access changes, how data is classified by sensitivity, and how compliance is verified. Compliance means conforming to applicable laws and regulations: data protection laws, health data regulations, financial regulations, and educational data protections. Risk management means identifying what can go wrong, estimating likelihood and impact, and taking proportionate steps to reduce or mitigate risks. Common data risks include breach, loss, corruption, compliance failure, and reputational harm. Students who ask "what could go wrong, and who would be harmed?" alongside "how do I make this work?" are better prepared for real professional data work.

Appendix M

Capstone Project Guide

The capstone project brings together everything students have learned in this course. This appendix provides a structured guide for planning, designing, building, and presenting a small information system.

M.1

Choosing a Problem

A good capstone problem involves real-world data management, is bounded enough to complete but complex enough to require multiple tables and meaningful relationships, involves different kinds of users with different access needs, raises real data quality and ethics considerations, and produces outputs that answer genuine questions. Good areas include library resource management, student course enrollment, event registration, help desk ticket tracking, community garden plot assignment, small inventory management, and volunteer schedule coordination.

Scope Guidance

Avoid problems that are too simple (a single-table address book does not require database design) or too large (a full hospital records system is beyond this course's scope). A system with 4 to 8 tables and 5 to 10 meaningful queries is a good target.

M.2 – M.3

Identifying Users and Writing Requirements

Write a brief description of each user type: what is their role, what information do they need to see, what actions do they need to perform, and what should they not be allowed to see or do? Users shape the permission model, interface design, reports, and queries. Functional requirements describe what the system must do. Data requirements describe what information must be stored. Query requirements describe what questions the system must answer. Writing requirements before designing the schema ensures the design serves real needs.

M.4 – M.6

Designing the Data Model, Building the Schema, Creating Sample Data

With requirements in hand, identify entities, list attributes for each, and identify relationship types and cardinalities. Check that the model supports all required queries. Draw an ERD. Translate the data model into CREATE TABLE statements with column names, data types, constraints, primary keys, and foreign keys. Apply normalization checks. Document design decisions. Insert sample data that is realistic enough to test all queries, including edge cases and records with missing optional values. Document the sample data so reviewers can verify query results.

M.7

Writing Useful Queries

  • A basic retrieval query for each major entity.
  • At least one filtering and sorting query per entity.
  • At least one aggregation query with GROUP BY.
  • At least one multi-table join query.
  • At least one many-to-many join through a junction table.
  • At least one query checking for missing or incomplete data.
  • A report query producing a summary useful for decision-making.
M.8 – M.9

Planning Forms, Reports, Permissions, and Backups

For each user type, identify needed input forms and what must be captured and validated at each form. For each query requirement, identify what report format is most useful. Sketch at least one form and one report layout. Write a permissions plan listing which tables each user type may access and what operations they may perform, applying least privilege throughout. Write a backup plan specifying backup frequency, storage location, restore testing schedule, and retention period. Write a maintenance plan listing regular tasks, frequency, and responsible parties.

M.10 – M.11

Writing Documentation and Presenting the System

Good documentation includes the system title and description, a user and role list, an ERD, a schema diagram, table and column descriptions, a query list with explanations, a permissions plan, a backup and maintenance plan, and a list of known limitations. A strong capstone presentation includes a problem description and user walkthrough, a data model walkthrough, a demonstration of key queries running against sample data, a discussion of at least one data quality or ethics consideration, and an honest assessment of limitations. The goal is to demonstrate that data modeling, relational design, querying, permissions, and responsible data thinking all came together in the design.

Appendix N

Glossary

Alphabetical definitions of key terms used throughout the textbook. For fuller explanations, consult the chapter and appendix where the term is introduced.

Access log — A record of who accessed a database, what actions they performed, and when.
Aggregate function — A SQL function that performs a calculation across multiple rows and returns one result. Examples: COUNT, SUM, AVG, MIN, MAX.
Atomicity — The property of a transaction in which all operations succeed together or all fail together, with no partial completion.
Attribute — A named piece of information stored about an entity, corresponding to a column in a relational table.
Authentication — The process of verifying identity before granting access to a system.
Authorization — The process of determining what an authenticated user is permitted to do.
Backup — A copy of a database preserved at a point in time for use in recovery.
Boolean — A data type holding one of two values representing true or false.
Candidate key — Any column or combination of columns that could serve as a valid primary key: unique and never NULL.
Cardinality — The numerical nature of a relationship between entities: one-to-one, one-to-many, or many-to-many.
CHECK constraint — A rule requiring column values to satisfy a specified condition.
Column — A named field shared by all rows in a table, holding values of a defined data type.
Composite key — A primary key consisting of two or more columns combined.
Consent — An individual's informed, voluntary agreement to the collection and use of their data.
Consistency — The property of a database in which all data satisfies defined rules and constraints.
Constraint — A rule enforced by the database that restricts what values may be stored.
CSV — Comma-separated values. A plain text file format in which each line is a record and values are separated by commas.
Dashboard — A visual display combining multiple summary values and charts to provide a current-state overview, designed for quick scanning.
Data — Raw, recorded facts about the world: measurements, names, dates, counts, codes, and values that become meaningful when organized and contextualized.
Data governance — The policies, roles, and procedures defining how data is managed within an organization.
Data minimization — The principle of collecting and retaining only data necessary for a stated purpose.
Data model — A structured description of the entities, attributes, relationships, and rules of a database.
Data quality — A measure of how well data serves its intended purpose, including accuracy, completeness, consistency, and validity.
Data type — A specification of what kind of value a column may hold and what operations are valid on it.
Database — An organized collection of data managed by a database management system, designed for reliable storage, retrieval, and protection.
Database management system (DBMS) — Software that manages the storage, retrieval, organization, and security of database data. Examples: SQLite, PostgreSQL, MySQL, SQL Server.
Default value — A value automatically assigned to a column when no value is explicitly provided during an insert.
DELETE — A SQL statement that removes rows from a table. Always include a WHERE clause.
Delimiter — A character separating values within a record in a text-based data file, most commonly a comma or tab.
Deletion anomaly — A problem in which deleting a record unintentionally destroys other information stored in the same table.
Duplicate record — A row in a table that represents the same real-world thing as another row, creating redundancy and potential inconsistency.
Entity — A distinct kind of thing about which a system needs to store information, corresponding to a table in a relational database.
Entity-relationship diagram (ERD) — A visual representation of entities, attributes, and relationships in a data model.
Encryption — The transformation of data into an unreadable form that requires a key to decrypt, protecting data at rest and in transit.
Endpoint — A specific URL in an API that accepts requests for a specific resource or action.
Field — A single named unit of data within a record, corresponding to a column within a row. Also called an attribute.
First normal form (1NF) — A normalization standard requiring every cell hold one atomic value, every row be unique, and no repeating column groups exist.
Foreign key — A column in one table referencing the primary key of another table, creating a link and enforcing referential integrity.
Form — A user interface component that collects input from users and submits it to an information system.
GROUP BY — A SQL clause dividing query results into groups sharing the same value in specified columns, used with aggregate functions.
HAVING — A SQL clause that filters groups produced by GROUP BY, applied after aggregation. Compare with WHERE, which filters rows before grouping.
Index — A database structure that allows faster retrieval of rows matching specified column values by avoiding full table scans.
Information — Data that has been organized, labeled, or summarized so that it carries meaning for a specific purpose and supports decisions.
Information system — An organized combination of people, processes, data, software, hardware, and rules that collects, stores, processes, and distributes information in support of organizational goals.
Inner join — A SQL join returning only rows where the join condition matches in both tables. Rows with no match are excluded.
INSERT — A SQL statement that adds a new row to a table. Always specify column names explicitly.
Insertion anomaly — A problem in which a record cannot be inserted without also inserting other information that may not yet be available.
JSON — JavaScript Object Notation. A lightweight text format for representing structured data, widely used in web APIs.
Junction table — A table representing a many-to-many relationship by storing the primary keys of both connected tables.
Knowledge — Understanding built from information through experience, pattern recognition, and reasoning over time.
Least privilege — The security principle that users and systems should have access to only what they need for their work, and nothing more.
Left join — A SQL join returning all rows from the left table plus matching rows from the right table, with NULL values where no right-table match exists.
Migration — A controlled, planned, and documented change to a database schema applied in a staging environment before production.
Natural key — A primary key drawn from real-world data that already exists and carries meaning, such as an ISBN or a national ID number.
Normalization — The process of organizing a relational database to reduce duplication and prevent update, insertion, and deletion anomalies.
NOT NULL — A constraint requiring a column to contain a value for every row; used for required fields.
NULL — A special marker representing the absence of a value in a database field: not zero, not empty string, but absent.
ORDER BY — A SQL clause sorting query results by specified columns. ASC is ascending (default); DESC is descending.
Outlier — A data value falling far outside the expected range for its column, requiring investigation to determine if it is an error or a genuine extreme value.
Permission — A specific allowed action on a specific database object, such as SELECT or INSERT on a table.
Primary key — A column or combination of columns uniquely identifying every row in a table. Must be unique, never NULL, and stable.
Privacy — The right of individuals to control information about themselves and have it used only for purposes they have consented to.
Record — A single organized collection of related fields describing one instance of something, corresponding to one row in a table.
Referential integrity — The property of a database in which every foreign key value either refers to an existing primary key or is NULL.
Relational database — A database organized as a collection of related tables connected by keys, queried with SQL.
Retention policy — A rule defining how long data is kept before being deleted, balancing storage cost, legal requirements, and recovery needs.
Role — A named set of permissions that can be assigned to users, corresponding to job functions.
Rollback — The cancellation of a transaction, undoing all changes made within it and restoring the previous state.
Row — A single record in a table. The terms row and record are used interchangeably.
Schema — The complete structural definition of a database: all tables, columns, data types, constraints, keys, and relationships.
Second normal form (2NF) — A normalization standard requiring 1NF plus the condition that every non-key column depends on the full composite primary key, not a part of it.
SELECT — The SQL clause specifying which columns to include in a query result.
Sensitive data — Information whose unauthorized disclosure could harm individuals, violate privacy, or create legal liability.
SQL — Structured Query Language. The standard language for defining, querying, and managing relational databases.
SQL injection — A security attack in which malicious SQL code is inserted into input that is then executed by a database query. Prevented with parameterized queries.
Subquery — A query nested inside another query, used to provide intermediate results in WHERE or FROM clauses.
Surrogate key — An artificially generated identifier with no real-world meaning, typically an auto-incremented integer, used as a primary key.
Table — A structured collection of records organized into named columns, the primary storage unit in a relational database.
Third normal form (3NF) — A normalization standard requiring 2NF plus the condition that every non-key column depends directly on the primary key and not on another non-key column.
Transaction — A unit of database work consisting of one or more operations that succeed or fail together, preserving data integrity.
UNIQUE constraint — A rule requiring that all values in a column or combination of columns are distinct across all rows.
UPDATE — A SQL statement that modifies values in existing rows. Always include a WHERE clause; without it, every row is modified.
Update anomaly — A problem in which changing a value requires updating multiple rows, and inconsistencies arise if any are missed.
Validation — The process of checking that data values conform to defined rules before they are stored.
View — A saved query stored in the database with a name, queried like a table.
WHERE — A SQL clause filtering rows by a condition. Only rows where the condition is true are included in the result.
Workflow — A defined sequence of steps that processes a unit of work from initiation to completion, with states, transitions, roles, and rules.

Ch. 13

Join Query Reading

Answers — Chapter 13 Join Reading
SQL — Join Query
SELECT s.FirstName, s.LastName, c.CourseName
FROM Enrollments AS e
JOIN Students    AS s ON e.StudentID = s.StudentID
JOIN Courses     AS c ON e.CourseID  = c.CourseID
WHERE c.CourseID = 'CS101';

This query returns the first name and last name of every student enrolled in CS101, along with the course name. It uses two INNER JOINs: one to retrieve student details from Students using StudentID, and one to retrieve course details from Courses using CourseID. The WHERE clause limits results to CS101 only.

Ch. 15 – 16

Data Modification and Transaction Safety

Answers — Chapters 15 and 16 Safety Scenarios
  1. A student recorded with the wrong StudentID: do not delete the record if other tables such as Enrollments reference this StudentID. Investigate whether an update to the correct ID is possible, or whether the error can be corrected another way. Deleting a referenced primary key will either fail (if referential integrity is enforced) or create orphaned records (if it is not).
  2. Every DELETE must include a WHERE clause because without one, DELETE removes every row in the table. A DELETE without WHERE on a production database can destroy all records instantly and may not be recoverable.
  3. A library checkout requires recording the loan in the Loans table and updating the resource status to Unavailable. These two operations belong in the same transaction because if only one succeeds, the database is inconsistent: either the book is marked unavailable with no loan record, or a loan record exists but the book is still marked available.
