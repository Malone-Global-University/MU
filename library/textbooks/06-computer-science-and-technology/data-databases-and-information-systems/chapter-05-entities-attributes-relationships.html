Before you can write a single line of SQL, before you can create a single table, before you can store a single record, you need to think carefully about what your database is actually supposed to represent. That thinking has a name. It is called data modeling.

Chapter 5 introduces data modeling as the first and most important step in building a reliable database. You will learn how to identify the real-world things a system needs to store information about, describe what is worth recording about each of those things, and describe how things connect to each other. These ideas form the conceptual foundation that every later chapter depends on.

By the end of this chapter, you should be able to read a plain-language description of a real system and turn it into a structured data model. You should be able to draw a simple diagram showing the pieces of that model and explain why each choice was made. You should also understand why skipping this step leads to messy, confusing, and unreliable databases that are painful to fix after the fact.

Before SQL, Design

Many students are eager to write SQL. Resist that impulse for now. A database that is poorly designed will produce problems that no amount of clever querying can fix. The design chapters come first for good reason. Everything you learn in this chapter will make every SQL chapter that follows more meaningful rather than mechanical.

5.1

What Data Modeling Is

A database is not built by accident. Someone has to decide what information it will store. Someone has to decide how that information will be organized. Someone has to decide what rules will govern it and how its pieces will connect. That process of deciding is called data modeling.

Data modeling is the practice of examining a real-world problem, identifying the information a database system needs to represent, and describing that information in a structured and deliberate way before anything is built in software.

Think about what it means to build a library catalog system. Before writing any code, a designer would need to answer many questions. What kinds of things will the catalog track? Books? Digital resources? Physical magazines? What do we need to know about each book? Its title? Its author? Its subject? Its location on the shelf? How do books and authors connect? Can a book have more than one author? Can an author have written more than one book? What about subjects? Can a book belong to more than one subject?

Each of those questions is a data modeling question. Answer them carefully and the database will be flexible, reliable, and easy to query. Answer them carelessly and the database will be confusing, hard to maintain, and full of problems that are difficult to fix later.

A Conceptual Stage

Data modeling happens at a conceptual level. At this stage, the designer is not yet thinking about which database software to use, which programming language will talk to the database, or what the user interface will look like. The designer is only thinking about the information itself: what things matter, what we need to know about those things, and how those things relate to each other.

The result of data modeling is a structured description of the system, usually expressed as a list of entities, their attributes, and the relationships between them. Those three concepts form the vocabulary of data modeling. The rest of this chapter introduces each one carefully.

Central Idea

The goal of data modeling is to understand a real problem clearly enough to describe it precisely. A good data model captures the right things, with the right descriptions, connected in the right ways, and leaves out everything the system does not need. It is a design discipline as much as a technical one.

5.2

Entities

An entity is a real-world thing that a database system needs to store information about.

The word "thing" is intentional and a little bit vague. Entities can be physical objects, like books, students, or products. Entities can be people, like instructors, customers, or employees. Entities can be places, like classrooms, warehouses, or branches. Entities can be events, like appointments, orders, or registrations. Entities can even be concepts or categories, like subjects, departments, or roles.

The Key Test for an Entity

Does the system need to store multiple pieces of information about this thing, and does it appear more than once? If the answer is yes, it is almost certainly an entity. A thing that is mentioned once as a background fact, rather than tracked as a repeated record, is probably just an attribute belonging to something else.

Consider a university enrollment system. The system needs to track students, each with a name, identification number, program of study, and contact information. Students appear many times. So Student is an entity. The system also needs to track courses, each with a name, code, description, and credit value. Courses appear many times. So Course is an entity. Instructors, too, each with a name, department, and contact details. So Instructor is an entity.

Now compare those to something like "the current semester." That is a piece of context, not a thing the system stores records about in the same way. It might become an entity in a more complex system, but in a simple enrollment system it is background information rather than a primary entity worth its own table.

Naming Entities

Entities are almost always named using a singular noun. The entity is Student, not Students. It is Course, not Courses. This convention exists because an entity describes a category of thing, and any one record in that category is a single instance of the entity. One record in the Student entity describes one student.

Entity names should be clear, specific, and self-explanatory. Avoid vague names like Item, Thing, or Record. Use names that communicate exactly what the entity represents: Book, Appointment, Invoice, InventoryItem, CourseSection.

Entities Across Different Systems

Notice how each kind of system has its own natural set of entities. Data modeling starts by discovering what those entities are for a specific problem.

5.3

Attributes

An attribute is a piece of information that describes an entity.

If an entity is the thing the database tracks, an attribute is what the database records about that thing. Attributes are the descriptive facts that make each instance of an entity distinct and useful. Consider the Student entity in a university system. What do we need to know about a student? We might need to record the student's full name, their student identification number, their date of birth, their email address, their program of study, and their enrollment status. Each of those facts is an attribute of the Student entity.

Attributes correspond to the columns in a database table. When a data model is turned into actual database tables, each entity becomes a table and each attribute becomes a column in that table.

Table 5.2 — Each entity has its own set of descriptive attributes, which become the columns of the eventual table.
Entity Attributes
StudentStudentID, FullName, DateOfBirth, Email, Program, EnrollmentStatus
CourseCourseID, CourseName, CourseCode, Credits, Description
BookBookID, Title, ISBN, PublicationYear, Publisher, PageCount
ProductProductID, ProductName, Price, StockQuantity, Category
Choosing the Right Attributes

Not every possible fact about an entity needs to become an attribute. Designers choose attributes based on the questions the system actually needs to answer. A library catalog might not need to store a book's physical weight or dimensions, even though those are real facts about books. Attributes are chosen because they are relevant to the system's purpose, not because they describe every possible aspect of an entity.

Ask this question for every candidate attribute: Will the system ever need to store, search, sort, or report on this piece of information? If yes, it is probably an attribute. If no, it probably does not belong.

Simple Versus Composite Attributes

Some attributes are simple and hold a single indivisible value, like a student's date of birth or a product's price. Other attributes are naturally composed of multiple parts. A person's full address, for example, is made of a street address, a city, a region or state, a postal code, and a country. These are called composite attributes.

When storing composite attributes, designers usually decide whether to store them as one combined value or break them into their component parts. Separating composite attributes is almost always the better choice.

Table 5.3 — Separating a composite attribute into its component parts allows individual searching, sorting, and filtering on each part.
Approach Example
Combined (problematic)"12 Oak Lane, Millbrook, MB 10421, Canada" — stored as a single text field
Separated (preferred)Street: "12 Oak Lane" · City: "Millbrook" · Region: "MB" · PostalCode: "10421" · Country: "Canada"

A combined address stored as a single text field cannot easily be searched by city alone, sorted by postal code, or filtered by country. Separated attributes make every one of those queries straightforward.

Derived Attributes

Some attributes can be calculated from other stored data. A student's age, for example, can be derived from their date of birth and today's date. The total price of an order can be derived by multiplying the quantity by the unit price. Derived attributes are usually not stored. Instead, they are calculated whenever they are needed.

Storing a derived value creates a risk: if the original data changes, the stored derived value may become incorrect unless it is updated everywhere. In a conceptual data model, it is enough to note that the attribute is derived and record what it is calculated from.

5.4

Relationships

A relationship describes how two or more entities are connected to each other.

Entities rarely exist in isolation. In a school system, students enroll in courses. In a library, patrons borrow books. In a store, customers place orders and orders contain products. Those connections between entities are relationships, and they are one of the most important things a data model must capture.

Relationships are named with a verb or verb phrase that describes the connection from one entity to another. A student enrolls in a course. A patron borrows a book. An instructor teaches a course. The relationship name should be readable as a plain sentence: "A student enrolls in a course." When you read the relationship in both directions, both directions should make sense, because the two directions reveal different useful facts about the connection.

Why Relationships Matter

Relationships are what make relational databases powerful. A system that stores students in one table and courses in another but never connects them cannot answer questions like "Which courses is this student enrolled in?" or "How many students are in this course?" The relationship between students and courses is what allows those questions to be answered.

In the database, relationships are represented using keys: identifiers that allow records in one table to point to related records in another table. Chapter 6 covers keys in detail. For now, the important idea is that every relationship in a data model will eventually need a mechanism in the database that makes the connection real and reliable.

Three Kinds of Relationships

Every relationship between two entities has a cardinality, a description of how many instances of one entity can be associated with instances of the other. There are three basic cardinalities.

Type 1
One-to-One

One instance of an entity is associated with exactly one instance of another entity, and vice versa. Least common in practice.

Type 2
One-to-Many

One instance of an entity can be associated with many instances of another entity, but each of those many instances points back to only one. Most common in practice.

Type 3
Many-to-Many

Many instances of one entity can be associated with many instances of another, and vice versa. Requires a junction table to resolve.

Sections 5.5, 5.6, and 5.7 introduce each cardinality type in depth, with examples and visual diagrams.

5.5

One-to-One Relationships

A one-to-one relationship exists when one instance of an entity is associated with exactly one instance of another entity, and vice versa.

One-to-one relationships are the simplest kind of relationship, but they are also the least common in practice. They often appear when two entities are so closely connected that they could almost be merged into one, but are kept separate for organizational or access reasons.

Example: Student and StudentProfile

Imagine a university system that tracks students and also maintains extended profile information for each student, such as emergency contacts, dietary preferences, and housing preferences. Every student has exactly one profile. Every profile belongs to exactly one student. The Student and StudentProfile entities are in a one-to-one relationship.

ER Diagram — One-to-One Relationship
When to Use One-to-One Relationships

One-to-one relationships are often used when some information is sensitive or accessed much less frequently than the main record, and storing it in a separate table means it does not slow down or complicate queries that do not need it. They are also used when different users or roles should have access to different parts of a record. In many cases, the attributes of both entities could simply be merged into one table, but separating them using a one-to-one relationship can be a deliberate design choice for access control, performance, or clarity.

5.6

One-to-Many Relationships

A one-to-many relationship exists when one instance of an entity can be associated with many instances of another entity, but each instance of the second entity is associated with only one instance of the first.

One-to-many relationships are the most common kind of relationship in relational databases. Most real-world connections between things work this way.

Examples

An instructor teaches many course sections, but each course section (in most systems) has one assigned instructor. The relationship between Instructor and CourseSection is one-to-many. A department has many students enrolled in it, but each student belongs to one department. A publisher publishes many books, but each book has one publisher. The relationship between Publisher and Book is one-to-many.

ER Diagram — One-to-Many Relationship
Representing One-to-Many in a Data Model

In a one-to-many relationship, the "many" side holds a reference back to the "one" side. A Book record holds a reference to the Publisher it belongs to. This reference is called a foreign key, which Chapter 6 explains in detail. For now, understand that the direction of the relationship matters. The "one" side is the parent. The "many" side is the child. Each child record points to the parent it belongs to.

Table 5.4 — In a one-to-many relationship, the one side is the parent and the many side is the child. Each child record points back to its parent.
EntityRoleExample
PublisherOne (parent)Horizon Academic Press
BookMany (child)Each book has one publisher; one publisher has many books
Reading One-to-Many Relationships

When you encounter a relationship description, practice reading it in both directions and asking "how many?" in each direction.

"How many course sections can one instructor teach?" — Many. "How many instructors does one course section have?" — One. That is a one-to-many relationship between Instructor and CourseSection. Getting comfortable with this two-direction reading will serve you throughout the design and SQL chapters.

5.7

Many-to-Many Relationships

A many-to-many relationship exists when one instance of an entity can be associated with many instances of another entity, and one instance of the second entity can also be associated with many instances of the first.

Many-to-many relationships are extremely common in real systems, and they require special treatment in relational databases.

Examples

A student enrolls in many courses. A course has many students enrolled in it. The relationship between Student and Course is many-to-many. A book covers many subjects. A subject can be covered by many books. An event has many attendees. An attendee can attend many events. These are all many-to-many relationships.

Why Many-to-Many Relationships Cannot Be Stored Directly

In a relational database, you cannot directly store a many-to-many relationship between two tables. If you tried to store all of a student's courses inside the Student record, you would end up with a messy, repeated structure that breaks fundamental rules of good data design.

The solution is a junction table, sometimes called a bridge table, an associative table, or a linking table. A junction table sits between the two entities and stores the specific connections between them. For the Student and Course relationship, the junction table might be called Enrollment. Each record in the Enrollment table represents one student's enrollment in one course.

5.8

Required and Optional Relationships

Every relationship has a question on each side: Must every instance participate in this relationship, or is participation optional?

This is called the participation constraint, or sometimes the optionality of the relationship. It tells you whether a record must have a related record on the other side, or whether a record can exist by itself without one.

Required Relationships

A required relationship means that every instance of an entity must participate in the relationship. A record on one side cannot exist without a corresponding record on the other side.

In a school system, every course section must belong to a department. There is no such thing as a course section that belongs to no department. The relationship between CourseSection and Department is required on the CourseSection side. Every loan record in a library system must be linked to a patron. A loan record that is not linked to any patron is meaningless and should not exist.

Optional Relationships

An optional relationship means that participation in the relationship is not required. A record can exist without having a related record on the other side.

In a library catalog, a book may or may not have a review attached to it. The relationship between Book and Review is optional on the Book side. A book can exist in the catalog without any reviews at all. An instructor may or may not be currently assigned to teach a course. If instructors can exist in the system before they are assigned, the relationship between Instructor and CourseSection is optional on the Instructor side.

Why Optionality Matters

Optionality affects how the database enforces rules. A required relationship usually means the database should refuse to save a record that is missing its required connection. An optional relationship means the database should allow records to exist even when the related record is absent.

Understanding optionality early prevents problems later. If you build a database that assumes all instructors must be assigned to courses, but the real system sometimes has instructors in the database before they are assigned, the database will refuse perfectly valid data. Modeling optionality correctly is part of modeling the real world correctly.

Table 5.6 — Participation constraints describe whether every record must have a related record on the other side of a relationship.

Loan → PatronRequiredA loan must link to a patron. A loan with no patron is meaningless.

5.9

Entity-Relationship Diagrams

An entity-relationship diagram, usually called an ER diagram or an ERD, is a visual representation of a data model. It shows entities, their attributes, and the relationships between them in a format that is easier to read and discuss than a plain text description.

ER diagrams are communication tools. They allow database designers to share and discuss a model with other designers, with developers, and with people who will use the system but have no database knowledge at all. A good ER diagram makes the structure of a database visible at a glance.

Basic Notation

Several different notation styles exist for ER diagrams. This textbook uses a readable introductory style throughout, while introducing the professional crow's foot notation where it is helpful. In the notation used here, rectangles represent entities, lines represent relationships, labels describe the relationship, and symbols at the ends of lines describe cardinality and optionality.

Table 5.7 — Common symbols used to mark cardinality and optionality on ER diagram relationship lines.
Symbol Meaning
1 at end of lineExactly one
M or * at end of lineMany
0..1 at end of lineZero or one (optional, at most one)
1..1 at end of lineExactly one (required)
0..* at end of lineZero or more (optional, many)
1..* at end of lineOne or more (required, many)

Crow's Foot Notation

The most widely used professional ER diagram notation is called crow's foot notation, named for the symbol that represents "many" on a relationship line. It looks like a small set of three lines spreading outward like a bird's foot or a fork. Most database design software uses crow's foot notation, so you should be able to recognize it even if you are not yet drawing it.

You Will Encounter Both Notations

This textbook uses a simplified notation for all diagrams and introduces crow's foot symbols where they are helpful. When you use professional database design software such as MySQL Workbench, dbdiagram.io, or Lucidchart, you will see crow's foot notation by default. Recognizing it is more important right now than drawing it perfectly.

5.10

From Real Problems to Data Models

Data modeling is not a purely technical skill. It requires listening carefully to descriptions of real systems, asking good questions, and translating messy human processes into clean structured models. This section works through a complete small example: modeling a system for tracking session registrations at a professional conference.

The Problem Description
Problem Statement

The conference organizer explains: "Our conference has multiple sessions throughout the day. Each session has a title, a start time, a room assignment, and a presenter. Presenters are professionals who submit a short biography and an email address when they register to present. Attendees register for the conference and then sign up for individual sessions they want to attend. We need to track which attendees are signed up for which sessions, and we want to be able to generate a list of attendees for any given session."

Here is the five-step process for turning that description into a data model.

  1. Identify the Entities

    Read the description and look for distinct, real-world things the system must track. From this description: Session (has a title, time, and room), Presenter (has their own information: biography and email), and Attendee (registered people who sign up for sessions). Note that the room is just an attribute of the Session, not a separate entity in this simple system.

  2. List Attributes for Each Entity

    Session: SessionID, Title, StartTime, RoomAssignment. Presenter: PresenterID, FullName, Email, Biography. Attendee: AttendeeID, FullName, Email. Both Presenter and Attendee have a name and email. That does not mean they should be merged. They are different kinds of participants with different roles in the system.

  3. Identify the Relationships

    A Session has one Presenter. A Presenter may present one or more Sessions. This is a one-to-many relationship between Presenter and Session. Attendees sign up for Sessions and a session has many attendees. This is a many-to-many relationship between Attendee and Session.

  4. Resolve the Many-to-Many Relationship

    The many-to-many relationship between Attendee and Session needs a junction table. Call it Registration. Each Registration record represents one attendee's sign-up for one session. The Registration entity connects an Attendee to a Session and adds its own attribute: RegistrationDate.

  5. Review the Model Against the Questions It Must Answer

    Can the system generate a list of attendees for a given session? Yes: query Registration for all records with that SessionID, then look up each AttendeeID. Can it show all sessions a given attendee is registered for? Yes: query Registration for all records with that AttendeeID. Can it show who is presenting each session? Yes: each Session has a PresenterID. The model answers all the questions the problem requires.

ER Diagram — Conference Registration System
The Review Step Is Not Optional

Always finish a data model by checking it against the questions the system must answer. If the model cannot answer one of those questions, something is missing. Adding a question the problem did not ask for is also a warning sign. A data model should be complete enough for the real purpose and lean enough to avoid unnecessary complexity.

Chapter Summary
  • Data modeling is the process of examining a real-world problem and creating a structured description of the information a database must represent, before any database is built. It operates at a conceptual level, independent of software or implementation details.
  • An entity is a real-world thing the database needs to store information about. Entities are named with singular nouns and correspond to tables in a relational database. The key test: does the system need to store multiple pieces of information about this thing, and does it appear more than once?
  • An attribute is a descriptive fact about an entity. Attributes correspond to columns in a database table. Attributes should be chosen based on what the system needs to store, search, sort, or report. Composite attributes should generally be separated into their components. Derived attributes are usually calculated rather than stored.
  • A relationship describes how two entities are connected. Relationships are named with a verb phrase and have a cardinality: one-to-one, one-to-many, or many-to-many. One-to-many is the most common. Many-to-many requires a junction table to resolve.
  • A junction table sits between two entities in a many-to-many relationship and stores individual connections. Junction tables often have their own attributes that belong to the relationship itself rather than to either connected entity.
  • Every relationship has a participation constraint: required (every record must have a related record on the other side) or optional (a record may exist without one). Modeling optionality correctly ensures the database accepts valid data and rejects invalid data.
  • Entity-relationship diagrams are visual tools for representing data models. They use rectangles for entities, lines for relationships, and symbols for cardinality and optionality. Crow's foot notation is the most widely used professional standard.
  • The process for building a data model from a plain-language description follows five steps: identify entities, list attributes, identify relationships, resolve many-to-many relationships with junction tables, and review the model against the questions it must answer.
Key Terms
Attribute

A piece of information that describes an entity. Attributes become columns in a database table.

Cardinality

A description of how many instances of one entity can be associated with instances of another entity in a relationship. The three basic cardinalities are one-to-one, one-to-many, and many-to-many.

Composite Attribute

An attribute that is naturally made of multiple component parts, such as a full address composed of street, city, region, postal code, and country. Usually better stored as separated components.

Data Model

A structured description of the information a system must represent, including its entities, attributes, and relationships, created before any database is built.

Data Modeling

The process of examining a real-world problem and creating a structured description of the information a database system must represent.

Derived Attribute

An attribute whose value can be calculated from other stored data rather than stored directly. Usually not stored, to avoid inconsistency when source data changes.

Entity

A real-world thing that a database system needs to store information about. Entities are named with singular nouns and correspond to tables in a relational database.

Entity-Relationship Diagram (ERD)

A visual representation of a data model showing entities, their attributes, and the relationships between them. A communication tool for designers, developers, and users.

Junction Table

A table used to resolve a many-to-many relationship by storing individual connections between records in two other tables. Also called a bridge table, associative table, or linking table.

Many-to-Many Relationship

A relationship in which one instance of an entity can be associated with many instances of another entity, and vice versa. Requires a junction table in a relational database.

One-to-Many Relationship

A relationship in which one instance of one entity can be associated with many instances of another entity, but each of those many instances is associated with only one instance of the first. The most common relationship type.

One-to-One Relationship

A relationship in which one instance of an entity is associated with exactly one instance of another entity, and vice versa. The least common relationship type.

Optional Relationship

A relationship in which participation is not required. An instance of an entity can exist without having a related instance on the other side.

Participation Constraint

A rule that describes whether every instance of an entity must participate in a relationship (required) or whether participation is allowed but not required (optional).

Relationship

A connection between two or more entities in a data model. Relationships are named with a verb or verb phrase and have a cardinality and participation constraint.

Required Relationship

A relationship in which every instance of an entity must have a corresponding related instance on the other side. The database should reject records that violate this constraint.

Simple Attribute

An attribute that holds a single indivisible value, such as a date of birth or a price.

Common Misconceptions
Misconception 1
"More entities means a more complex database."

The number of entities is not what makes a database complex. A well-modeled database with twelve entities is much easier to work with than a poorly modeled one with three. Complexity comes from muddled relationships, merged entities that should be separate, and attributes that belong in the wrong place. Create the entities the real system requires, not the fewest you can get away with.

Misconception 2
"If two entities share the same attributes, they should be merged into one."

Entities are defined by their role in the system, not by which attributes they happen to share. A Patron and a Presenter both have a name and an email. But they are different kinds of participants in different systems with different relationships to other entities. Merging them would create confusion, messy queries, and fragile rules. Shared attributes are not a reason to merge entities.

Misconception 3
"Many-to-many relationships can be handled without a junction table."

Some students try to store multiple values in a single field to represent a many-to-many relationship, such as storing a list of course codes inside a single Student record field. This approach breaks fundamental principles of table design, makes querying very difficult, and prevents the database from enforcing integrity rules. Many-to-many relationships always require a junction table in a relational database.

Misconception 4
"Optional relationships are less important to model carefully."

Optional relationships are just as important as required ones. If a relationship is optional, the database must allow records to exist without a related record on the other side. Failing to model optionality correctly can result in a database that rejects valid data, or, worse, one that accepts invalid data without any complaint.

Misconception 5
"The ER diagram is the database. Once I draw it, I am done."

An ER diagram is a model, not an implementation. It is a picture of how you plan to organize information. Turning that picture into actual database tables requires further steps: defining keys, writing table creation commands, adding constraints, loading sample data, and testing. The ER diagram is the beginning of the design process, not the end of it.

Comprehension Questions
  • Define data modeling in your own words. Why does a database designer do data modeling before building any tables?
  • What is an entity? Give three examples of entities from three different systems, such as a school system, a retail system, and a healthcare system.
  • What is an attribute? Give four examples of attributes for the entity Product in an online store.
  • Explain the difference between a simple attribute and a composite attribute. Give an example of a composite attribute and describe how you would separate it into its components.
  • What is a derived attribute? Why are derived attributes usually not stored directly in a database?
  • What are the three types of cardinality in entity relationships? For each type, give a real-world example using entities from a system you are familiar with.
  • What is a junction table? Why is it needed to resolve a many-to-many relationship?
  • A junction table often has its own attributes. Give an example of a junction table between two entities and describe two attributes that belong to the junction table itself rather than to either of the connected entities.
  • What is the difference between a required relationship and an optional relationship? Give one example of each from any system you can describe.
  • What is an entity-relationship diagram? What three things does it represent visually?
  • Describe the five-step process for building a data model from a plain-language problem description. What question should you ask at the end to check whether the model is complete?
Application Questions
  • A student says: "I do not need to draw an ER diagram. I can just create the tables directly." What is wrong with this approach? What kinds of problems might arise from skipping the data modeling step?

Describe the entities, their key attributes, and the relationships between them. Identify any many-to-many relationships and explain how you would resolve them.

Coming Next — Chapter 6: Keys, Identifiers, and Constraints

Chapter 5 introduced the three building blocks of every data model: entities, attributes, and relationships. You now know how to identify what a system must track, describe what is recorded about each thing, and show how things are connected.

But the data model is still incomplete. There is a crucial question that Chapter 5 did not fully answer: How does a database tell records apart? If two students have the same name, how does the system know which enrollment belongs to which student? If a book has been borrowed many times, how does each loan record correctly point to the right book?

Chapter 6 answers these questions by introducing keys and identifiers, the mechanisms that allow a database to uniquely identify every record and enforce reliable connections between related tables. Keys are one of the most important design decisions in any database. Chapter 6 will explain primary keys, candidate keys, natural keys, surrogate keys, foreign keys, and the constraints that protect data integrity. Without a solid understanding of keys, even a well-drawn ER diagram cannot be turned into a reliable working database.
