About This Chapter

This is not a test of memorization. It is a demonstration of thinking. A good capstone information system shows that you can reason about what a system needs to do, design structures that support those needs reliably, protect the data and the people it represents, and explain your decisions in plain language. Work carefully. Document your reasoning at every step. The goal is not a perfect system. The goal is a well-reasoned one.

28.1

What Makes a Good Capstone System

Before you choose a problem and begin designing, it helps to understand what a good capstone information system looks like, and what separates it from a collection of tables that technically stores data but does not actually serve anyone well.

Serves Real Users

Every table, query, form, and report exists because a specific user needs to accomplish a specific task. Design elements that serve no user need should not be in the system.

Answers Useful Questions

Before designing tables, write down the questions your system must be able to answer. These questions drive every design decision. If a question cannot be answered by the schema, the design has failed its users.

Protects Data Integrity

A good system enforces constraints, requires necessary fields, prevents orphaned keys, rejects invalid values, and uses transactions where multiple changes must succeed or fail together.

Protects People

Databases hold information about real people. A responsible design considers what data is necessary, who may see it, how it is protected, how long it is retained, and what happens when it needs correction or deletion.

Can Be Maintained

A database that works on day one but cannot be backed up, migrated, or documented is not reliable. A good design includes backup plans, maintenance procedures, and documentation that lets others understand the system.

Is Explained

In professional work, you will almost always need to explain your data system to people who did not help design it. A good capstone comes with a clear explanation of why each decision was made.

Keep all of these qualities in mind as you work through this chapter.

28.2

Choosing a Problem

The first step in designing an information system is choosing a problem worth solving. A good capstone problem involves multiple kinds of things that need to be stored and connected, has identifiable users with real tasks, produces questions that require joins, filters, aggregates, or summaries to answer, and involves data that needs to be validated, protected, and maintained over time.

Problems That Work Well
Library Resource Catalog

Tracks books, digital resources, authors, subjects, users, loans, and due dates. Answers questions about availability, overdue items, and circulation by subject.

Student Course Enrollment

Tracks students, courses, instructors, terms, departments, enrollments, and grades. Answers questions about rosters, prerequisites, and enrollment counts.

Event Registration

Tracks events, sessions, speakers, venues, registrants, and registrations. Manages capacity limits, waitlists, and confirmation records.

Clinic Appointments

Tracks patients, clinicians, appointment types, slots, and prescriptions. Answers questions about upcoming appointments and patient history.

Inventory and Orders

Tracks products, suppliers, inventory levels, customers, orders, and order items. Answers questions about stock levels and pending orders.

Help Desk Tickets

Tracks users, staff, ticket categories, status, assignments, notes, and resolution records. Answers questions about open tickets and resolution times.

Problems to Avoid

Avoid problems that are too simple to generate meaningful design decisions. A table that stores a single list of items with no relationships, no constraints worth enforcing, and no interesting queries is not a suitable capstone problem. Also avoid problems where you do not have enough understanding of the real-world context to design responsibly.

Declaring Your Problem

Once you have chosen a problem, write a one-paragraph description of it. This paragraph should explain what the system does, who uses it, what kinds of information it stores, and what kinds of questions it needs to answer. Everything else in your capstone will flow from this declaration.

Here is an example declaration for a library resource system:

Example Problem Declaration

The MGU Learning Resource System tracks the library's collection of books, digital resources, periodicals, and instructional materials. Library staff use the system to add and update resource records, manage loans to registered users, and generate reports about the collection. Registered users can search the catalog, check availability, and view their current loans. The system must answer questions such as which resources are currently available in a given subject area, which items are on loan and when they are due, which users have overdue items, and how many loans each resource category has generated this term.

28.3

Identifying Users and Requirements

A good information system serves specific people doing specific things. Before designing a single table, you need to understand who uses the system and what they need it to do.

Identifying User Roles

Most information systems have more than one type of user. Each type of user has different tasks, different information needs, and different permissions. Identifying user roles early guides everything from table design to permission planning.

Table 28.1 — User roles for the MGU Learning Resource System.
Role Description
Library AdministratorManages the full system. Adds, edits, and removes resources, user accounts, and records. Runs all reports.
Library StaffAdds new resources, processes loans and returns, generates operational reports. Cannot delete records or change system settings.
Registered UserSearches the catalog, checks availability, views personal loan history, renews loans. Cannot see other users' records.
InstructorSearches the catalog, creates recommended reading lists, views usage reports for their course resources.
System AdministratorManages database backups, user accounts, permissions, and maintenance tasks. Cannot see personal loan records.
Writing Functional Requirements

Functional requirements describe what the system must do. They are written as clear statements of user-facing behavior. A useful format is: The system must allow [role] to [action].

Example functional requirements for the MGU Learning Resource System:

  • The system must allow Library Staff to add a new resource record with title, author or creator, subject area, format, publication year, and a unique identifier.
  • The system must allow Library Staff to record a loan when a registered user borrows a resource.
  • The system must allow Registered Users to search the catalog by title, author, or subject.
  • The system must allow Registered Users to view their own current loans and due dates.
  • The system must allow Library Administrators to run a report showing all overdue loans.
  • The system must prevent a resource from being loaned to a second user while it is already on loan.
  • The system must prevent the deletion of a user account while the user has outstanding loans.
Writing Questions the System Must Answer

At this stage, list every question your system should be able to answer with a query. These questions will drive your table design. If a question cannot be answered, your schema is missing something.

  • Which resources are currently available for loan?
  • Which resources are currently on loan, and to whom, and when are they due?
  • Which loans are overdue as of today?
  • How many times has each resource been borrowed in the last six months?
  • Which subject areas have the largest number of resources?
  • Which resources have never been borrowed?
  • How many resources are in each format category?
  • Which users have borrowed more than five resources in the current term?
Write This List First

Write this question list before you design your tables. Then, after you design your tables, verify that every question can actually be answered by your schema. If it cannot, revise the design. The questions are the test. The schema is the answer.

28.4

Designing the Data Model

With your requirements in hand, you are ready to design the data model. The data model describes the entities in your system, the attributes that describe each entity, and the relationships between entities.

Identifying Entities

An entity is a category of thing your system needs to store information about. Look at your data requirements and your list of questions. Every noun that the system needs to track is a candidate entity. For the MGU Learning Resource System, the entities are Resource, Creator, Subject, Format, User, and Loan.

For each entity, list the attributes that describe it. Focus on the attributes your functional requirements and question list actually need. Do not add attributes speculatively.

Identifying Relationships

Now connect the entities with relationships and note the cardinality of each:

  • A Resource has one Format. Many resources belong to one format.
  • A Resource can have many Creators, and a Creator can be credited on many Resources. This many-to-many relationship requires a junction table: Resource_Creator.
  • A Resource can be classified under many Subjects, and a Subject can include many Resources. This many-to-many relationship requires a junction table: Resource_Subject.
  • A Loan connects one User to one Resource. A User can have many Loans. A Resource can appear in many Loans.

28.5

Building the Schema

The schema is the concrete implementation of your data model. It specifies every table, every column, every data type, every primary key, every foreign key, and every constraint. Read each statement carefully and match each design decision back to the requirements and data model from the previous sections.

SQL — Complete Schema for the MGU Learning Resource System
-- Format lookup table
CREATE TABLE Format (
    format_id     INTEGER PRIMARY KEY,
    format_name   TEXT NOT NULL UNIQUE
);

-- Subject classification table
CREATE TABLE Subject (
    subject_id          INTEGER PRIMARY KEY,
    subject_name        TEXT NOT NULL UNIQUE,
    subject_description TEXT
);

-- Creator table (authors, editors, organizations)
CREATE TABLE Creator (
    creator_id    INTEGER PRIMARY KEY,
    creator_name  TEXT NOT NULL,
    creator_type  TEXT NOT NULL CHECK (creator_type IN ('Person', 'Organization'))
);

-- Resource table (books, PDFs, periodicals, videos, etc.)
CREATE TABLE Resource (
    resource_id         INTEGER PRIMARY KEY,
    title               TEXT NOT NULL,
    publication_year    INTEGER,
    format_id           INTEGER NOT NULL REFERENCES Format(format_id),
    availability_status TEXT NOT NULL DEFAULT 'Available'
                            CHECK (availability_status IN ('Available', 'On Loan', 'Unavailable')),
    date_added          TEXT NOT NULL DEFAULT (date('now'))
);

-- Junction table: Resource and Creator (many-to-many)
CREATE TABLE Resource_Creator (
    resource_id  INTEGER NOT NULL REFERENCES Resource(resource_id),
    creator_id   INTEGER NOT NULL REFERENCES Creator(creator_id),
    creator_role TEXT NOT NULL DEFAULT 'Author',
    PRIMARY KEY (resource_id, creator_id)
);

-- Junction table: Resource and Subject (many-to-many)
CREATE TABLE Resource_Subject (
    resource_id INTEGER NOT NULL REFERENCES Resource(resource_id),
    subject_id  INTEGER NOT NULL REFERENCES Subject(subject_id),
    PRIMARY KEY (resource_id, subject_id)
);

-- User table (registered borrowers, staff, instructors, administrators)
CREATE TABLE User (
    user_id           INTEGER PRIMARY KEY,
    full_name         TEXT NOT NULL,
    email_address     TEXT NOT NULL UNIQUE,
    account_status    TEXT NOT NULL DEFAULT 'Active'
                          CHECK (account_status IN ('Active', 'Suspended')),
    registration_date TEXT NOT NULL DEFAULT (date('now')),
    role              TEXT NOT NULL DEFAULT 'Registered User'
                          CHECK (role IN ('Registered User', 'Staff',
                                          'Instructor', 'Administrator',
                                          'System Administrator'))
);

-- Loan table (borrowing events)
CREATE TABLE Loan (
    loan_id        INTEGER PRIMARY KEY,
    resource_id    INTEGER NOT NULL REFERENCES Resource(resource_id),
    user_id        INTEGER NOT NULL REFERENCES User(user_id),
    date_borrowed  TEXT NOT NULL DEFAULT (date('now')),
    due_date       TEXT NOT NULL,
    date_returned  TEXT
);
Schema Design Decisions Worth Noting

Surrogate primary keys are used throughout. Every table uses an integer _id column as its primary key. Natural keys such as email addresses or titles are subject to change, may contain duplicates, and create performance problems in large join operations. Surrogate keys are stable, simple, and efficient.

CHECK constraints protect the validity of controlled vocabulary fields. The availability_status column in Resource only accepts three specific values. The account_status column in User only accepts two. The role column only accepts five defined roles. If the application tries to insert or update a row with an invalid value, the database will reject it.

Foreign key references enforce referential integrity. A Resource cannot reference a Format that does not exist. A Loan cannot reference a Resource or User that does not exist. In SQLite, foreign key enforcement must be explicitly enabled with PRAGMA foreign_keys = ON; at the start of each connection.

The junction tables Resource_Creator and Resource_Subject use composite primary keys combining both foreign key columns, preventing the same resource from being linked to the same creator or subject more than once.

The date_returned field in Loan is nullable. A NULL value means the resource has not yet been returned. When the resource is returned, an UPDATE statement fills in the actual return date. This is a clean, explicit representation of an ongoing loan.

28.7

Writing Useful Queries

With sample data in place, you can now write the queries that answer the questions your system must be able to answer. This section works through each question with a full SQL query, an explanation of what it does, and the result it would return against the sample data.

Query 1: Which resources are currently available for loan?
SQL — Available Resources
SELECT
    r.resource_id,
    r.title,
    f.format_name,
    r.publication_year
FROM Resource r
JOIN Format f ON r.format_id = f.format_id
WHERE r.availability_status = 'Available'
ORDER BY r.title;

This query joins Resource with Format to add the human-readable format name, then filters for resources marked as Available. It returns the resource identifier, title, format, and publication year, sorted alphabetically.

Query 3: Which loans are overdue as of a given date?
SQL — Overdue Loans
SELECT
    r.title,
    u.full_name       AS borrower,
    u.email_address,
    l.due_date,
    julianday('2024-11-10') - julianday(l.due_date) AS days_overdue
FROM Loan l
JOIN Resource r ON l.resource_id = r.resource_id
JOIN User u     ON l.user_id = u.user_id
WHERE l.date_returned IS NULL
  AND l.due_date < '2024-11-10'
ORDER BY days_overdue DESC;

The julianday() function is SQLite-specific and calculates the difference between two dates as a decimal number of days. The query adds a calculated column showing how many days overdue each loan is relative to the check date.

Query 4: How many times has each resource been borrowed?
SQL — Loan Count per Resource
SELECT
    r.title,
    COUNT(l.loan_id) AS total_loans
FROM Resource r
LEFT JOIN Loan l ON r.resource_id = l.resource_id
GROUP BY r.resource_id, r.title
ORDER BY total_loans DESC;

A LEFT JOIN is used here so that resources with zero loans still appear in the result. Without it, resources that have never been borrowed would be excluded entirely.

Query 5: Which resources have never been borrowed?
SQL — Never-Borrowed Resources (Subquery)
SELECT
    r.resource_id,
    r.title,
    f.format_name
FROM Resource r
JOIN Format f ON r.format_id = f.format_id
WHERE r.resource_id NOT IN (
    SELECT DISTINCT resource_id FROM Loan
)
ORDER BY r.title;

Query 6: How many resources does each subject area contain?
SQL — Resources per Subject
SELECT
    s.subject_name,

FROM Subject s

Query 7: Available resources in a given subject with authors
SQL — Available Computer Science Resources
SELECT
    r.title,
    c.creator_name,
    f.format_name,
    r.availability_status
FROM Resource r
JOIN Resource_Subject rs  ON r.resource_id  = rs.resource_id
JOIN Subject s            ON rs.subject_id   = s.subject_id
JOIN Resource_Creator rc  ON r.resource_id  = rc.resource_id
JOIN Creator c            ON rc.creator_id  = c.creator_id
JOIN Format f             ON r.format_id    = f.format_id
WHERE s.subject_name           = 'Computer Science'
  AND r.availability_status    = 'Available'
  AND rc.creator_role          = 'Author'
ORDER BY r.title;

This is the most complex query in the set. It joins six tables to connect a subject name to available resources, their authors, and their formats. Notice that Data Modeling in Practice would appear twice in the result because it has two authors. Understanding why joins produce extra rows, and how to handle them, is a key skill reinforced here from Chapter 13.

Query 8: Loan activity summary by user
SQL — User Loan Summary with Conditional Aggregation
SELECT
    u.full_name,
    u.role,
    COUNT(l.loan_id)                                                   AS total_loans,
    SUM(CASE WHEN l.date_returned IS NOT NULL THEN 1 ELSE 0 END)  AS returned,
    SUM(CASE WHEN l.date_returned IS NULL     THEN 1 ELSE 0 END)  AS currently_on_loan
FROM User u
LEFT JOIN Loan l ON u.user_id = l.user_id
GROUP BY u.user_id, u.full_name, u.role
ORDER BY total_loans DESC;

This report query uses conditional aggregation with CASE expressions inside SUM to count returned and outstanding loans separately within a single query. Users with no loans still appear due to the LEFT JOIN.

28.9

Planning Permissions, Backups, and Maintenance

A system is not complete when its tables are designed and its queries work correctly. You must also plan who can access what, how the data will be protected, and how the system will be maintained over time.

Permissions Plan

The permissions plan assigns specific database operations to specific user roles, following the principle of least privilege introduced in Chapter 18. Every role should have exactly the permissions it needs, and no more.

Why System Administrators Cannot See Loan Records

System Administrators have infrastructure access but are deliberately separated from loan and user personal records. This separation protects user privacy and reduces the risk that a compromised administrator account exposes personal borrowing histories. This is the principle of least privilege applied to operational roles.

Sensitive Fields

The following fields contain personally identifiable information and require extra protection: User.full_name, User.email_address, and Loan.user_id (which links a person to their borrowing history). These fields should be excluded from any reporting interface that does not require them, should not appear in log files, and should be subject to a clear data retention policy.

Backup Plan

Backup type: Full database backup using SQLite's .backup command or the equivalent for the database engine in use.

Backup schedule: Daily incremental backup at 02:00; full backup every Sunday at 03:00; manual backup before any schema migration or significant data import.

Backup storage: Primary backup on the same server in a separate directory; secondary backup on an offsite server or cloud storage provider; backups older than 90 days archived, backups older than one year deleted unless a retention exception applies.

Restore testing: A restore test is performed monthly by restoring the most recent full backup to a test environment and verifying that all queries return expected results. Restore procedures are documented in the system maintenance guide.

Recovery objectives: Recovery point objective: no more than 24 hours of data loss. Recovery time objective: restore service within four hours of a failure event.

Indexes for Performance
SQL — Recommended Indexes
CREATE INDEX idx_loan_resource_id ON Loan(resource_id);
CREATE INDEX idx_loan_user_id     ON Loan(user_id);
CREATE INDEX idx_loan_due_date    ON Loan(due_date);
CREATE INDEX idx_resource_status  ON Resource(availability_status);
CREATE INDEX idx_user_email       ON User(email_address);

The Loan table's foreign key columns and due date column are likely candidates for frequent filtering and joining. The Resource availability status supports the most common catalog filter. The User email index supports login lookups.

Monthly Data Quality Checks
SQL — Data Quality Audit Queries
-- Resources with no subject classification
SELECT r.resource_id, r.title
FROM Resource r
LEFT JOIN Resource_Subject rs ON r.resource_id = rs.resource_id
WHERE rs.resource_id IS NULL;

-- Resources with no creator
SELECT r.resource_id, r.title
FROM Resource r
LEFT JOIN Resource_Creator rc ON r.resource_id = rc.resource_id
WHERE rc.resource_id IS NULL;

-- Resources marked On Loan with no active loan record
SELECT r.resource_id, r.title
FROM Resource r
WHERE r.availability_status = 'On Loan'
  AND r.resource_id NOT IN (
      SELECT resource_id FROM Loan WHERE date_returned IS NULL
  );

The third query detects a specific integrity problem: a resource marked as on loan with no corresponding active loan record. This could happen if a return was processed incorrectly or a loan record was deleted without updating the resource status. Running these checks monthly catches data problems before they affect operations.

28.10

Documenting and Presenting the System

A complete information system includes documentation that allows other people to understand what the system does, why it is designed the way it is, and how to use it, maintain it, and extend it.

System Overview Document

The system overview document explains the system to a new reader who has not been involved in its design. It should cover: the system name and purpose, user roles and access, a plain-language description of each table and how it connects to others, key queries and what they answer, a list of forms and reports, and maintenance contacts and procedures.

The data description should be written in prose, not as raw SQL. It is a description that a non-technical reader can follow.

Data Dictionary

A data dictionary is a structured reference that describes every table and every column in the database. For each column, record its name, data type, whether it is required or optional, its default value if any, any constraint or validation rule, and a plain-language description of what it stores.

Presenting the System

When presenting a capstone information system to an audience, walk through these steps in order:

  • 1
    Begin with the problem.

    Explain what the system does and who it serves before showing any tables or queries. Your audience needs context before they can evaluate the design.

  • 2
    Show the data model.

    Walk through the entities, attributes, and relationships. Explain why each entity exists and what the relationships between them represent.

  • 3
    Walk through two or three representative queries.

    Explain what each query does in plain language, then show the SQL, then show the result. Choose queries that demonstrate the system's ability to answer its key questions.

  • 4
    Explain one design decision.

    Pick one place where you made a conscious choice, and explain your reasoning. Why a junction table here? Why a surrogate key over a natural key? Why required rather than optional?

  • 5
    Describe your protection and maintenance plan.

    Explain who has access to what, how the data is backed up, and how the system will be maintained and documented over time.

  • 6
    Reflect on one data responsibility question.

    Every system that stores information about people raises questions about privacy, fairness, and responsible use. Name one and explain how your design addresses it.

28.11

Reflecting on Data Responsibility

The final step in any information system project is reflection. Before you call a system complete, ask yourself the hard questions about what the system does to the people it represents.

What data does this system collect?

Make an explicit list of every piece of personal information your system stores. For each field, ask whether it is genuinely necessary for the system to function. Data that is not needed should not be collected. The MGU Learning Resource System collects user names, email addresses, and borrowing history, all of which serve a genuine function. It does not collect home addresses, phone numbers, or payment information, because none of those were required.

Who can see what?

Review your permissions plan. Are there paths through the system through which a user could see information they should not be allowed to see? In the MGU Library System, Registered Users can only see their own loan records. Loan queries in the Registered User role filter by the authenticated user's user_id. This permission boundary must be enforced in the application layer, not only in the database schema.

How long will this data be kept?

Borrowing history is useful for operations, but it is also a record of what people have read. Many privacy advocates and librarians argue that borrowing history should be deleted once a loan is resolved, or at least after a defined retention period. Consider whether your system's retention policy reflects genuine operational need or whether it accumulates records beyond their useful life.

What happens when data is wrong?

Every system will eventually hold incorrect data. A user's name changes. An email address becomes invalid. A loan record is created with the wrong resource. Your system should have a clear, accessible process for identifying and correcting errors, without requiring direct database access for routine corrections.

Could this system cause harm?

A library borrowing record might seem harmless, but borrowing history can reveal political views, medical concerns, religious inquiry, or other private matters. If this data were disclosed without consent, accessed by an unauthorized party, or shared with an external organization, real harm could result.

The Central Responsibility Question

Ask this question about every information system you build: What is the worst thing that could happen if this data were exposed, misused, or lost? The answer should shape your security design, your permission structure, your retention policy, and your backup plan. These are not afterthoughts. They are design requirements.

Chapter Summary
  • A good capstone information system serves real users with real needs, can answer useful questions, protects data integrity, protects the people it represents, can be maintained over time, and is explained clearly to others.
  • Choosing a problem means selecting one with multiple entities, identifiable users, questions requiring joins and aggregates, and data that must be validated and protected.
  • Requirements gathering produces user roles, functional requirements, data requirements, and a list of questions the system must answer. The question list drives every design decision and is used to verify the schema after it is built.
  • Data modeling identifies entities, attributes, relationships, and cardinality. Many-to-many relationships require junction tables. The entity-relationship diagram makes the structure visible before any SQL is written.
  • The schema converts the data model into concrete CREATE TABLE statements with primary keys, foreign keys, data types, NOT NULL constraints, CHECK constraints, and DEFAULT values.
  • Sample data must be internally consistent: availability status, loan records, and date fields must all agree. Sample data is used to verify schema correctness and test every query.
  • Useful queries answer the specific questions from the requirements list. They include single-table filters, multi-table joins, subqueries, aggregation with GROUP BY, conditional aggregation with CASE, and LEFT JOINs to include zero-match rows.
  • Forms plan data entry for each user role, specifying fields, validation rules, and database operations. Reports and dashboards present retrieved data in formats suited to their audiences and decision-making needs.
  • Permissions follow the principle of least privilege. Sensitive fields are identified and protected. A backup plan specifies schedule, storage, restore testing, and recovery objectives. Indexes support the most frequent queries. Monthly data quality checks catch integrity problems early.
  • Documentation includes a system overview and a data dictionary. Presentation follows a clear order: problem first, then data model, then representative queries, then design decisions, then protection plan, then ethical reflection.
  • Every information system that stores data about people carries responsibilities: collect only what is necessary, protect what is collected, define how long it is kept, provide a way to correct errors, and consider the harm that misuse or exposure could cause.
Key Terms
Capstone System

A complete information system designed to demonstrate the full range of skills developed throughout a course, including data modeling, SQL, security, ethics, and documentation.

Data Dictionary

A structured reference document that describes every table and column in a database, including data types, constraints, and plain-language descriptions of what each field stores.

Functional Requirements

Clear statements of what a system must allow users to do, typically written as: "The system must allow [role] to [action]."

Data Requirements

Clear statements of what a system must store, describing the kinds of information each part of the system holds.

Recovery Point Objective

The maximum acceptable amount of data loss in the event of a system failure, expressed as a time period.

Recovery Time Objective

The maximum acceptable amount of time between a system failure and the restoration of service.

System Overview Document

A plain-language description of a system's purpose, users, data structures, queries, interfaces, and maintenance procedures, intended for readers who were not involved in the design.

Conditional Aggregation

The use of CASE expressions inside aggregate functions such as SUM or COUNT to compute different counts or totals for different conditions within a single GROUP BY query.

Problem Declaration

A one-paragraph statement that defines the scope of a capstone system: what it does, who uses it, what it stores, and what questions it must answer.

Database Design Checklist

Use this checklist to verify your capstone schema before writing queries or sample data.

  • Every table has a clearly named primary key.
  • Every primary key is unique and never null.
  • Every foreign key references a valid primary key in the correct table.
  • Every many-to-many relationship has a junction table with a composite primary key.
  • Every required field is marked NOT NULL.
  • Every controlled vocabulary field has a CHECK constraint.
  • Every table name is singular and descriptive.
  • Every column name is lowercase with underscores and clearly descriptive.
  • No column stores multiple values in a single field.
  • No repeating groups of columns exist within any table.
  • Every relationship needed to answer the required question list can be navigated through the schema.
  • The schema has been reviewed against the complete list of required questions.
SQL Query Checklist

Use this checklist to verify each query before including it in your capstone.

  • The query answers a specific question from the requirements list.
  • Every column referenced exists in the table named in FROM or JOIN.
  • Every JOIN includes a valid join condition on matching key columns.
  • WHERE and HAVING conditions filter what they intend to filter.
  • Aggregate functions are used correctly with GROUP BY.
  • LEFT JOIN is used wherever rows with no matches should still appear in the result.
  • Column aliases make the result table readable.
  • The query has been tested against the sample data.
  • The query result has been verified against expected output.
