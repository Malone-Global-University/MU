Designing a Database from a Real Problem

The previous four chapters introduced the building blocks of database design one layer at a time: entities, keys, the relational model, and normalization. This chapter brings them together. The previous four chapters in Part II introduced the building blocks of database design: entities and attributes, keys and constraints, the relational model, and normalization. Each chapter taught one layer of the design process in focused detail. This chapter brings those layers together.

Chapter 9 is a consolidation chapter. It does not introduce many new terms. Instead, it walks through the complete process of designing a small, real database from beginning to end, starting with a plain-language problem description and ending with a reviewed schema, sample data, and a set of questions the database can answer.

By the end of this chapter, you will have practiced the full design process once, from requirements to finished schema. You will be ready to move into SQL with a clear understanding of what a well-designed database looks like before the first CREATE TABLE statement is written.

How to Use This Chapter

Read each section in order. The ten sections follow the real sequence of design decisions, and later sections depend on choices made in earlier ones. As you read, notice not just what is decided but why. The reasoning behind each choice is as important as the choice itself.

9.1

Starting with a Real Problem

Database design does not begin with tables. It begins with a problem.

Someone needs to keep track of something. A school needs to manage student enrollments. A library needs to track which resources have been borrowed. A small organization needs to record its events and attendees. A research team needs to store survey responses. A store needs to manage its product inventory and orders. In each case, there is a real situation with real people, real things, real activities, and real questions that need answering. The job of the database designer is to understand that situation well enough to build a data model that supports it.

Beginners often want to start by drawing tables right away. This is understandable, because tables are the familiar part of databases. But jumping to tables before understanding the problem leads to incomplete designs, forgotten relationships, awkward structures, and systems that cannot answer the questions they were supposed to answer. The disciplined approach is to understand the problem first, then build the model, then draw the tables.

This chapter follows a single running example throughout all ten sections. The example is small enough to understand fully but rich enough to be realistic.

The Running Example: A Community Tutoring Center

The Westbridge Community Tutoring Center is a small nonprofit organization that connects students with volunteer tutors. The center currently tracks everything using paper forms and a shared spreadsheet, but the spreadsheet has become disorganized. Records are duplicated. Columns mean different things to different staff members. Nobody is sure which tutor is assigned to which student. The director wants a proper database.

The center needs to keep track of:

  • Students who come to the center for tutoring
  • Tutors who volunteer at the center
  • The subjects the center offers tutoring in
  • Which tutor is assigned to which student for which subject
  • The tutoring sessions that take place
  • Notes and attendance records from those sessions

This is the starting point. It is a plain-language description of a real situation. It is not yet a data model. Turning it into one is the work of this chapter.

Central Idea

Every database design project starts with a description of a real situation. The designer's first job is to read that description carefully, ask questions, and understand the problem before drawing a single box or writing a single column name.

9.2

Identifying Users and Needs

Before designing any table, a database designer needs to understand who will use the system and what those users need from it. Different users have different relationships to the data. Some users enter data. Some users read data. Some users run reports. Some users have authority to change records. Some users should only see certain information. Understanding these roles helps the designer know what the system must support and what it must protect.

For the Westbridge Tutoring Center, the primary users are:

The Center Director

Needs the full picture: enrollment counts, active tutors, waiting lists by subject, and monthly session totals. Uses summary reports. Assigns tutors to students.

Staff Coordinators

Handle day-to-day record-keeping. Add new students, record new tutors, schedule sessions, and update attendance. Primary data-entry users.

Tutors

Need to see the students assigned to them and review notes from past sessions. Should not see records for other tutors' students.

Students and Families

May eventually view their own records. Out of scope for the initial design, but worth noting for future permissions planning.

Understanding these users reveals several design requirements. The system must record tutor-student assignments, not just separate lists of tutors and students. It must record individual sessions, not just the fact that a relationship exists. It must support notes attached to sessions. It must support summary queries about subjects, session counts, and attendance. And permissions will eventually matter: tutors should see limited views, while coordinators need broader access.

Permissions Are Not Scope Creep

Noting that tutors should see limited data is not premature. Identifying permission boundaries early shapes the design. If a tutor must only see their own students' records, the database needs a clean way to link sessions to tutors through assignments, not through a tangled set of shared tables. Good design anticipates access control even if the implementation comes later.

9.3

Listing Questions the Database Must Answer

One of the most practical tools in database design is a list of questions the system must be able to answer. These questions reveal what data must be stored, what relationships must be captured, and what queries must be possible. Writing questions before drawing tables is a discipline that separates careful designers from impulsive ones.

For the Westbridge Tutoring Center, the coordinator and director identified these questions:

  1. Which students are currently enrolled at the center?
  2. Which tutors are active?
  3. What subjects does the center offer?
  4. Which tutors are qualified to tutor which subjects?
  5. Which student is assigned to which tutor for which subject?
  6. When did a particular tutoring session take place?
  7. How long was each session?
  8. Did the student attend? Did the tutor attend?
  9. What notes did the tutor leave about the session?
  10. How many sessions has a student attended this month?
  11. How many hours has a particular tutor volunteered this semester?
  12. Which subjects have the longest waiting lists?
  13. Which student-tutor pairs have been meeting the longest?
  14. Are there any students who have not had a session in the past thirty days?

These are not SQL queries yet. They are requirements written in plain language. But each question points to data that must be stored. Reading through them reveals that students must be stored as records with a status field; tutors must be stored as records, also with status; subjects must form a separate list rather than repeated text values; there must be a record of which tutors are qualified in which subjects; there must be a record of which student is assigned to which tutor for which subject; and individual sessions must be stored with dates, durations, attendance fields, and notes.

These requirements are the raw material of the data model that follows.

9.4

Identifying Entities

Once the questions are clear, the designer can identify the main entities the database must represent. An entity is a category of real-world thing that the system needs to store information about. Each entity will become a table. Identifying entities means reading the requirements and recognizing the distinct kinds of things being tracked.

From the Westbridge requirements, the entities are:

Student

A person who comes to the center for tutoring. The system needs to store each student as a distinct record. Students appear in most of the fourteen questions from Section 9.3.

Tutor

A volunteer who provides tutoring. Tutors are distinct from students. A person could theoretically be both, but for this system they are treated as separate roles with separate tables.

Subject

An academic subject, such as mathematics, reading, or English as a second language. Subjects are a distinct list that tutors and sessions refer to. They should not be stored as repeated text strings inside other tables, because that would make them impossible to query cleanly and easy to misspell inconsistently.

Assignment

The relationship between a student, a tutor, and a subject. An assignment records that a particular student is working with a particular tutor in a particular subject. This is not just a fact about students or tutors alone. It is a fact about the connection between them, and it carries its own meaningful attributes: a start date, an end date, and a status. That is what makes it an entity rather than a simple join condition.

Session

An individual tutoring meeting. A session has a date, a duration, attendance information for both the student and the tutor, and notes. Sessions are associated with an assignment.

What Makes Something an Entity?

An entity is anything that needs to be stored as a collection of distinct records with its own attributes, even if it is abstract. Assignment and Session are entities even though they represent relationships and events rather than physical objects. Notes and attendance are not entities here; they are attributes of Session. If the requirements were more complex, that might change, but for this design, keeping them as fields on Session is the right level of structure.

9.5

Choosing Attributes

Once the entities are identified, the designer chooses the attributes for each one. An attribute is a specific fact recorded about each instance of an entity. Every attribute should answer the question: what specific information do we need to store about each record of this type? 9.6

Defining Relationships

With entities and attributes chosen, the designer must define the relationships between entities. Relationships describe how records in one table connect to records in another. Defining relationships also means deciding the cardinality: one-to-one, one-to-many, or many-to-many.

The Westbridge database has four relationships:

Student → Assignment — One-to-Many

One student can have many assignments, but each assignment belongs to exactly one student. A student might be assigned to one tutor for mathematics and a different tutor for reading. The foreign key student_id in Assignment connects each assignment to its student.

Tutor → Assignment — One-to-Many

One tutor can have many assignments, but each assignment belongs to exactly one tutor. A tutor working with three different students holds three separate assignment records. The foreign key tutor_id connects each assignment to its tutor.

Subject → Assignment — One-to-Many

One subject can appear in many assignments, but each assignment involves exactly one subject. Mathematics might be the subject for ten different student-tutor pairs. The foreign key subject_id connects each assignment to its subject.

Assignment → Session — One-to-Many

One assignment can have many sessions, but each session belongs to exactly one assignment. A student-tutor pair working on mathematics might meet twenty times. Each meeting is a separate session record, connected by the foreign key assignment_id.

Why Assignment Is Not a Pure Junction Table

The Assignment table does the structural work of resolving the many-to-many relationship between students and tutors, and between students and subjects. But it also carries its own meaningful attributes: start_date, end_date, and status. This is the mark of a true entity. A pure junction table exists only to link two other tables and carries no additional meaning of its own. Assignment is more than that, which is why it receives a surrogate primary key and its own detailed attribute list.

9.7

Choosing Keys and Constraints

With entities, attributes, and relationships defined, the designer must choose keys and constraints. Keys identify records. Constraints protect data quality.

Primary Keys

Each table uses a surrogate integer primary key with a consistent naming convention: student_id, tutor_id, subject_id, assignment_id, session_id. Surrogate keys are stable, simple, and not subject to the problems that natural keys can create when real-world values change. A student's name, phone number, or email can all change. An auto-incremented integer never does.

Foreign Keys
Table 9.6 — Foreign key references
TableForeign Key ColumnReferences
Assignmentstudent_idStudent(student_id)
Assignmenttutor_idTutor(tutor_id)
Assignmentsubject_idSubject(subject_id)
Sessionassignment_idAssignment(assignment_id)

Each foreign key enforces referential integrity. The database will reject an assignment that references a student_id that does not exist in the Student table. It will also prevent a student record from being deleted while assignments still reference it, unless a cascading rule or soft deletion strategy is in place.

Uniqueness Constraints

The subject_name column in Subject should have a uniqueness constraint. There should never be two subject records named "Mathematics." If a duplicate were inserted accidentally, queries and reports would return confusing doubled results. The email columns in Student and Tutor should also have uniqueness constraints if the center uses email as a contact identifier, preventing duplicate registrations.

NOT NULL Constraints

Required fields include first_name and last_name in Student and Tutor; subject_name in Subject; student_id, tutor_id, subject_id, and start_date in Assignment; and assignment_id, session_date, duration_minutes, student_attended, and tutor_attended in Session. Fields such as end_date in Assignment and session_notes in Session are nullable because they represent information that may not exist yet or may not apply.

CHECK Constraints on Status Fields

The status columns in Student, Tutor, and Assignment should only accept specific valid values, enforced with a CHECK constraint. Student and Tutor status: active or inactive. Assignment status: active, completed, or paused. Allowing free text in a status field is a data quality risk. If coordinators type whatever they like, the field will quickly accumulate variations such as "Active," "ACTIVE," "active," "enrolled," and "Enrolled," all of which are different text values but all meant to mean the same thing.

9.10

Reviewing and Revising the Design

A first draft of a database design is rarely the final design. The review step is where the designer checks whether the schema can answer all the questions from Section 9.3, whether any entities or attributes have been missed, whether any constraints have been forgotten, and whether there are any normalization problems.

Checking the Questions

Returning to the fourteen questions from Section 9.3 and checking each one against the schema:

  • Yes
    Which students are currently enrolled? The Student table has a status field; filter for status = 'active'.
  • Yes
    Which tutors are active? The Tutor table has a status field.
  • Yes
    What subjects does the center offer? The Subject table lists all subjects.
  • Gap
    Which tutors are qualified in which subjects? The schema records which tutors have active assignments in which subjects, but not qualifications independently of assignments. A tutor qualified to teach Science with no current Science students is invisible.
  • Yes
    Which student is assigned to which tutor for which subject? The Assignment table records exactly this.
  • Yes
    When did a session take place? The Session table has session_date.
  • Yes
    How long was each session? The Session table has duration_minutes.
  • Yes
    Did the student attend? Did the tutor attend? The Session table has student_attended and tutor_attended.
  • Yes
    What notes did the tutor leave? The Session table has session_notes.
  • Yes
    How many sessions has a student attended this month? Join Student → Assignment → Session, count where student_attended = true and the month matches.
  • Yes
    How many hours has a tutor volunteered? Sum duration_minutes across sessions linked to the tutor's assignments in the current semester.
  • Gap
    Which subjects have the longest waiting lists? The current schema does not include a waiting list at all.
  • Yes
    Which student-tutor pairs have been meeting the longest? Calculate from start_date in Assignment.
  • Yes
    Students with no session in the past thirty days? Check the most recent session date across all assignments for each student.
Identified Gaps

The review reveals two gaps that a more complete design would address.

TutorQualification table. A table recording which subjects each tutor is qualified to teach, independently of current assignments. Attributes would include tutor_id, subject_id, and qualification_date. This would be a many-to-many relationship between Tutor and Subject, resolved through the junction table TutorQualification.

WaitingList table. A table recording students who are waiting for an assignment in a particular subject. Attributes might include student_id, subject_id, request_date, and priority.

Normalization Check

Looking at the five tables in the current design, the schema is in a good state for an introductory-level design. Every table has a single primary key. No table stores repeated groups of values in a single column. Each non-key attribute depends on the full primary key of its table. Each non-key attribute depends on the primary key directly, not through another non-key attribute. The design is clean and does not exhibit the update anomalies, insertion anomalies, or deletion anomalies described in Chapter 8.

Revised Entity List

After the review, the complete entity list for the Westbridge Tutoring Center database is: Student, Tutor, Subject, TutorQualification (new), Assignment, Session, and WaitingList (new). A production design would fully specify TutorQualification and WaitingList with their own attributes, keys, and constraints. For this chapter's purposes, identifying them as gaps and naming them is the right outcome of the review step. A real design project would cycle back through Sections 9.4 through 9.9 to add these entities before finalizing the schema.

Design Is Iterative

Finding gaps during the review step is not a failure. It is the review working as intended. Every real database design project goes through multiple iterations. The review catches problems before they are built into a production system, where fixing them is far more expensive. A designer who finds nothing to improve during a review probably did not look carefully enough.

Chapter Summary
  • Database design does not begin with tables. It begins with a problem description that the designer must understand before drawing anything. The Westbridge Community Tutoring Center example demonstrated how to move from a plain-language situation to a structured model.
  • Identifying users and needs before designing any table reveals what the system must support and what it must protect. Different users need different access, and those differences shape the design from the start.
  • Writing a list of plain-language questions the database must answer is one of the most practical tools in database design. Questions reveal which entities are needed, which relationships must be captured, and which queries must be possible.
  • Entities are the distinct kinds of things a database must store information about. Assignment and Session are entities even though they are abstract, because they carry their own meaningful attributes and require their own tables.
  • Attributes are the specific facts recorded about each entity. Choosing attributes means deciding on data types, which fields are required, which are nullable, and which carry special constraints such as uniqueness.
  • Relationships connect entities. The Westbridge schema uses four one-to-many relationships: Student-to-Assignment, Tutor-to-Assignment, Subject-to-Assignment, and Assignment-to-Session.
  • Keys and constraints protect the data. Surrogate primary keys provide stable identification. Foreign keys enforce referential integrity. Uniqueness constraints prevent duplicates. NOT NULL constraints enforce required fields. CHECK constraints restrict status fields to valid values.
  • A schema diagram shows all tables, columns, primary keys, foreign keys, and constraints in one structured view. It is the blueprint of the database.
  • Sample records make the schema concrete. Creating a small set of realistic data for each table allows the designer to trace queries mentally and verify that the schema can hold real information.
  • The design review checks the completed schema against the original question list. Two gaps were found: the absence of a TutorQualification table and the absence of a WaitingList table. Finding these gaps before implementation is exactly what the review step is designed to accomplish.
Key Terms
Requirements

The stated needs of a system, gathered from users and stakeholders, that define what the database must store and what questions it must answer.

User Roles

The different kinds of people who interact with a system, each with different access needs, data-entry responsibilities, and reporting requirements.

System Questions

Plain-language questions written before any table is designed, used to reveal what data must be stored and what relationships must be captured.

Entity Identification

The process of reading requirements and recognizing the distinct kinds of things a database must store information about, each of which becomes a table.

Attribute Selection

The process of deciding what specific facts to record about each entity, including data types, whether a field is required, and whether special constraints apply.

Relationship Definition

The process of stating how entities connect to each other and what the cardinality of each connection is.

Schema Diagram

A structured representation of a database design showing tables, columns, primary keys, foreign keys, and relationships. The blueprint of a database.

Sample Data

A small set of realistic records created during design to verify that the schema can hold the information it is supposed to hold and that queries can be traced through it.

Design Review

A structured check of the completed schema against the original question list, looking for gaps, normalization problems, missing constraints, and missing entities.

Design Gap

A requirement the current schema cannot satisfy, identified during the design review, pointing to a missing entity, attribute, or relationship.

Soft Deletion

The practice of marking records as inactive rather than removing them from the database, preserving historical data and referential integrity.

Junction-Like Entity

A table that resolves a complex or many-sided relationship between other tables while also carrying its own meaningful attributes. The Assignment table in this chapter is an example.

Table Review
  • Look at the Session sample data in Section 9.9. Session 4 shows that student_attended = false and tutor_attended = true. What does this record mean in plain language? What did the coordinator need to do as a result? How does the database support tracking what happened next?
  • Look at the Assignment sample data. How many assignments does Helena (tutor_id 1) currently have? Which students is she working with, and in which subjects? How would you find this information by reading the table directly, without writing a query?
  • The Subject table includes a level field with values such as "Elementary" and "Middle school." What data quality problem could arise if this field allowed free text with no constraint? What constraint would you add to prevent it?
  • The Student table includes both phone and email columns. Both are nullable. What does it mean, in terms of real-world data, for both fields to be null on a student record? Is this a data quality problem? What policy might the center adopt to address it?

  • The design review also identified a need for a WaitingList table. Add this table to your diagram. What information would it need to record? What foreign keys would it have? What relationship does it create between Student and Subject?

Error Analysis
  • A coordinator tries to enter an Assignment record with a student_id value of 99, but there is no student with that ID in the Student table. What constraint prevents this insertion? What does this protection accomplish for the data in the long run?
  • A staff member enters two Subject records with the same name: one reads "Mathematics" and the other reads "mathematics." Both are accepted if there is no uniqueness constraint. What problems would this duplication cause for queries and reports? What constraint would prevent it?
  • A designer proposes storing tutor qualifications as a comma-separated list in a single text column on the Tutor table, like this: qualifications = "Mathematics, Science, Reading". Identify at least three problems this design would cause. What is the correct relational approach?
  • A designer suggests that instead of linking Session to Assignment, sessions should link directly to the student and tutor using student_id and tutor_id foreign keys on the Session table. What information would this design fail to capture? What problems would it create if a student's tutor changed mid-semester?
