Parts I and II taught you how to think about data and how to design the structures that hold it. You learned what fields and records are. You practiced drawing entity-relationship diagrams. You chose primary keys, defined foreign keys, and thought carefully about constraints. You designed tables to avoid duplication, protect consistency, and support the questions an organization actually needs to answer.

Now comes the moment those skills have been preparing you for.

A relational database stores its data in tables. Those tables sit inside a database management system, a piece of software that organizes and protects the data. To communicate with that system, to ask it questions, to retrieve records, and eventually to add, update, or remove data, you use a language specifically designed for that purpose.

That language is SQL.

SQL is one of the most widely used and durable languages in computing. It has been in continuous use since the 1970s. It runs in databases used by hospitals, banks, governments, universities, retail stores, streaming services, and nearly every other kind of organization that stores structured data. If you work with data in any serious way, whether as a developer, an analyst, a researcher, an administrator, or a system designer, you will encounter SQL.

This chapter introduces SQL carefully and patiently. The goal is not to memorize commands. The goal is to understand what SQL is, why it works the way it does, how to read a query and understand what it asks, and how to write simple, clear queries that answer real questions. Reading before writing is especially important here. Just as learning to read code before writing it helps you understand what programs do, learning to read SQL before writing it helps you understand what databases do.

Note for New Students

This chapter does not require you to have a running database in front of you, though practicing in SQLite alongside the reading is encouraged. The examples are complete and self-contained. You can read every query, predict every result, and answer every practice question using only the tables printed in this chapter.

10.1

What SQL Is

SQL stands for Structured Query Language. It is pronounced either as the three letters S-Q-L or as the word "sequel." Both pronunciations are used by professionals, and neither is wrong.

SQL is a language for working with relational databases. It lets you do four broad categories of things:

Query data

Ask a database to retrieve, filter, sort, join, and summarize records. This is the focus of Part III and by far the most common use of SQL.

Define structures (DDL)

Create tables, specify columns and data types, define keys and constraints, and alter the structure of a database as it grows. Called Data Definition Language.

Manipulate data (DML)

Insert new records, update existing records, and delete records that are no longer needed. Called Data Manipulation Language. Covered in Part IV.

Control access (DCL)

Grant or revoke permissions for different users and roles. Called Data Control Language. Connects to the security ideas in Chapter 18.

In Part III, the focus is almost entirely on querying. Understanding how to retrieve, filter, sort, and summarize data is the foundation of everything else. Once you understand what a database contains and how SQL retrieves it, the other categories become much easier to learn.

SQL and Database Systems

SQL itself is a standard, but every database management system implements it with some variation. The SQL written for PostgreSQL is mostly the same as the SQL written for MySQL or SQLite, but there are differences in available functions, data types, syntax for certain operations, and available features.

This textbook uses SQLite-compatible SQL throughout Part III and Part IV. SQLite is a lightweight, widely available database engine that runs inside applications and requires no server installation. It is excellent for learning because it is simple to set up, works on any computer, and supports all the core SQL features students need. Everything you learn here transfers cleanly to PostgreSQL, MySQL, Microsoft SQL Server, and other systems. You will encounter minor syntax differences, but the core ideas remain exactly the same.

SQL Is Declarative

One of the most important things to understand about SQL is that it is a declarative language. When you write SQL, you describe what result you want, not how the database should go find it.

In a programming language like Python, you often write step-by-step instructions. You might say: start with this list, loop through each item, check if it meets a condition, collect the ones that do, and return them. That is called an imperative style of programming.

SQL works differently. You say: give me all the records from this table where this condition is true, sorted in this order. You describe the result you want, and the database engine figures out the most efficient way to retrieve it. This declarative quality makes SQL readable and expressive. A well-written SQL query reads almost like a question in plain language.

Central Idea

You tell SQL what you want. The database engine decides how to get it. This separation is the key to SQL's expressiveness. A single SQL statement can express a retrieval operation that would take many lines of code in a general-purpose language.

10.2

Databases, Tables, and Result Sets

Before writing a single line of SQL, it helps to be clear about three things that SQL works with: databases, tables, and result sets.

The Database

A database is an organized collection of related data, stored and managed by a database management system. When you connect to a database, you are connecting to a specific, named collection of tables that belong together. A hospital might have one database called patient_records. A library might have one called library_catalog. A university might have one called student_information_system.

In SQLite, a database is typically stored as a single file on disk, such as library.db or school.db. In larger systems like PostgreSQL, a database is a named collection managed by a running server.

Tables

Inside a database, data is organized into tables. You know tables well from Parts I and II. Each table represents one kind of entity, such as students, books, enrollments, or departments. Each table has columns that describe attributes of that entity, and rows that each represent one record. When you write SQL, you almost always name at least one table. The database engine goes to that table to find the data your query asks for.

Result Sets

When you run a SQL query, the database does not return a file, a list, or a simple answer. It returns a result set. A result set is a table, with columns and rows, produced temporarily by the query. It exists as output while you are reading it or using it. It is not stored in the database permanently unless you explicitly save it somewhere.

Key Insight: Output Is a Table

The output of a SQL query is itself a table. It has column names. It has rows. It can be filtered, sorted, and summarized. It can be used as the input to another query. Understanding result sets as tables helps enormously when you start working with joins, subqueries, and views in later chapters.

A Visual Model

Imagine a room with shelves. Each shelf holds one table, which is a stack of records. When you ask SQL a question, it goes to the shelves you named, pulls out the records that match your conditions, arranges them the way you asked, and brings the result back to you as a new temporary table. The shelves do not change. The original data stays where it is. What you receive is a copy of the matching data, organized the way you requested.

10.3

SQL Statements

SQL communicates through statements. A SQL statement is a complete instruction that the database engine can execute. Every statement ends with a semicolon (;), which signals to the database that the statement is complete and ready to run.

In practice, some tools run a SQL query without requiring the semicolon, especially when you are running one statement at a time. But including the semicolon is a good habit. In scripts that contain many statements, semicolons are required to separate them.

The Most Common Statements

In Part III, you will work with the SELECT statement almost exclusively. SELECT is the statement used for querying, for retrieving data from one or more tables. In Part IV, you will also work with INSERT, UPDATE, and DELETE, which change data, and with CREATE TABLE, which creates new tables. For now, every example in this chapter uses SELECT.

How a Statement Is Structured

A SQL statement is made up of clauses. Each clause starts with a keyword, such as SELECT, FROM, or WHERE, followed by more information. Here is the simplest possible SQL statement:

SQL — Minimum Statement
SELECT *
FROM students;

This statement has two clauses. The SELECT clause says what columns to retrieve. The asterisk (*) means all columns. The FROM clause says which table to look in. Together, they say: retrieve all columns from the students table. Every query you write in Part III will be a variation of this basic pattern.

Keywords and Identifiers

In SQL, there are two kinds of words: keywords and identifiers.

Keywords are words that SQL itself defines and that carry special meaning. Examples include SELECT, FROM, WHERE, ORDER, BY, GROUP, HAVING, JOIN, AND, OR, NOT, NULL, IS, IN, and BETWEEN.

Identifiers are names that you give to tables, columns, databases, and views. Examples include students, first_name, enrollment_date, and course_id. These names come from the database schema you designed in Part II.

SQL keywords are conventionally written in uppercase letters. Identifiers are written in lowercase with underscores separating words. This textbook uses that convention throughout. Writing keywords in uppercase makes queries much easier to read, because it immediately distinguishes SQL structure from data content. This is a professional habit worth building from the start.

10.4

The SELECT Clause

The SELECT clause begins every data retrieval query in SQL. It tells the database which columns to include in the result set.

Basic Syntax
SQL — Syntax Pattern
SELECT column_name
FROM table_name;

Suppose a database contains a table called students with columns for student_id, first_name, last_name, email, enrollment_date, and department_id.

Selecting One Column

To retrieve only the first_name column:

SQL
SELECT first_name
FROM students;

The result set would be a single-column table containing the first name of every student in the database, one per row.

Selecting Multiple Columns

To select more than one column, list the column names separated by commas.

SQL
SELECT first_name, last_name, email
FROM students;

The order of columns in the SELECT clause determines the order of columns in the result set. If you write last_name, first_name instead of first_name, last_name, the result set will show last names first.

Selecting All Columns

The asterisk (*) is a shorthand that means "all columns."

SQL
SELECT *
FROM students;

The asterisk is convenient for exploring a table and checking what it contains. But in professional and production contexts, selecting specific columns is almost always better than using *. When you name specific columns, your query is clearer about what it needs, and it stays correct even if someone adds or reorders columns in the table later.

SELECT Never Changes Data

An important property of SELECT is that it never changes the database. No matter how complex your query becomes, a SELECT statement only reads data and returns a result set. The original records remain exactly as they were. This makes SELECT safe to experiment with. You cannot accidentally break anything by writing the wrong SELECT query.

10.5

The FROM Clause

The FROM clause tells the database which table to retrieve data from.

SQL
SELECT first_name, last_name
FROM students;

Here, FROM students points the database to the students table. The name in the FROM clause must exactly match the name of a table that exists in the database. If you type FROM student when the table is called students, the database will return an error telling you the table does not exist. In this chapter, every query uses exactly one table. In Chapter 13, you will learn how to join multiple tables together in a single query.

Logical Execution Order

The actual order in which SQL clauses execute is not the same as the order in which they are written. Even though you write SELECT first, the database processes FROM first. Understanding this logical order helps enormously when writing complex queries.

1 FROM Identify the table
2 WHERE Filter rows
3 GROUP BY Group rows
4 HAVING Filter groups
5 SELECT Choose columns
6 ORDER BY Sort the result
7 LIMIT Restrict row count

The clauses for WHERE, GROUP BY, HAVING, ORDER BY, and LIMIT will be introduced in Chapters 11 and 12. Knowing the logical order now prevents confusion later.

Selecting specific columns produces cleaner, more useful result sets. It communicates clearly what the query is for. 10.7

Column Aliases

The column names in a result set come from the table schema by default. A column alias lets you give a column a different name in the result set without changing the actual table. Aliases use the AS keyword, placed after the column name in the SELECT clause.

SQL — Column Aliases
SELECT title, author_last AS surname, author_first AS given_name
FROM books;
Result — Column headers renamed with aliases.
titlesurnamegiven_name
The Silent ShoreOkaforEmeka
Introduction to EcologyLindqvistSara
Rivers of the WorldCastellanoMarta
Practical Data AnalysisHuangWei
The History of SaltOkaforEmeka
Garden SciencePetrovIrina
Writing Clear ProseLindqvistSara

The underlying data and the table itself are completely unchanged. The alias only affects the result set for this particular query.

Aliases With Spaces

If you want an alias that contains spaces, wrap it in double quotation marks.

SQL — Quoted Alias
SELECT title, copies_owned AS "Copies in Collection"
FROM books;
When to Use Aliases

Aliases are most useful in three situations. First, when a column name is not descriptive enough on its own for the intended reader. Second, when a result set is going to be presented to someone who does not know the database schema. Third, and most importantly, when you use aggregate functions in Chapter 12. Computed values like COUNT(*) or AVG(copies_owned) do not have natural column names, so aliases are how you label them meaningfully.

Aliases Do Not Change the Table

A column alias only affects the result set. It renames the column in the output for this particular query. The column name in the actual books table remains copies_owned regardless of what alias you assign it. Aliases never alter the database.

10.8

SQL Comments

As queries become longer and more complex, they benefit from explanation. SQL supports comments, which are notes written inside a query that the database engine ignores completely. Comments exist for human readers, not for the database.

Single-Line Comments

A single-line comment begins with two hyphens (--). Everything after those two hyphens on that line is ignored.

SQL — Single-Line Comments
-- Retrieve all book titles and their authors
SELECT title,           -- the book's full title
       author_last,     -- author's family name
       author_first     -- author's given name
FROM books;
Multi-Line Comments

A multi-line comment begins with /* and ends with */. Everything between those markers, across as many lines as needed, is treated as a comment.

SQL — Multi-Line Comment
/*
  Query: Book inventory overview
  Purpose: List all books with their genre and available copies
  Table used: books
*/
SELECT title, genre, copies_owned
FROM books;
Commenting Out Lines When Debugging

Comments are useful when debugging. You can temporarily "comment out" part of a query to test whether a specific clause is causing a problem, then restore it when you are done.

SQL — Commenting Out to Debug
SELECT title, genre, copies_owned
FROM books
-- WHERE genre = 'Science'  -- temporarily removed to see all books
;

When you are learning SQL, comments help you think through a query step by step. When you are working on real systems, comments are essential for maintenance. A query written today without explanation may be read by you or someone else months from now when the original context is long forgotten.

10.9

Formatting SQL Readably

SQL does not require any particular spacing, indentation, or line breaks. The following three queries are all syntactically identical and produce exactly the same result:

SQL — All on One Line
SELECT title, author_last, author_first FROM books;
SQL — Clauses on Separate Lines
SELECT title, author_last, author_first
FROM books;
SQL — Each Column on Its Own Line
SELECT
    title,
    author_last,
    author_first
FROM books;

All three are valid SQL. But readable formatting matters enormously for humans. As queries grow in length, well-formatted SQL becomes dramatically easier to read, write, and debug.

Formatting Conventions Used in This Textbook

Keywords are uppercase. All SQL keywords appear in uppercase: SELECT, FROM, WHERE, ORDER BY, GROUP BY, JOIN, AND, OR, NOT, AS, IN, BETWEEN.

Identifiers are lowercase. Table names, column names, and aliases use lowercase letters with underscores: students, first_name, enrollment_date, book_id.

Each major clause starts on its own line. SELECT, FROM, WHERE, ORDER BY, GROUP BY, HAVING, and JOIN each begin a new line.

Column lists are indented. When selecting multiple columns, they are indented under the SELECT keyword.

Here is a longer example that puts all conventions together:

SQL — Full Formatting Example (Chapters 11–12 preview)
-- Find all science books published after 2018
-- that the library owns more than one copy of
SELECT
    title,
    author_last,
    year_published,
    copies_owned
FROM books
WHERE genre = 'Science'
  AND year_published > 2018
  AND copies_owned > 1
ORDER BY year_published DESC;

Even if you do not know all the clauses yet, the formatting makes the structure of this query readable at a glance. The keywords stand out. The column list is clearly separated from the filtering conditions. Each condition is on its own line.

Formatting as Respect

Good formatting is not just about aesthetics. It is about respect: respect for the next person who reads the code, respect for the time of colleagues who debug it, and respect for your own future self who will come back to a query after weeks away. In professional environments, poorly formatted SQL is considered unprofessional, in the same way poorly formatted code is in any programming language. Building the habit of clear formatting from the beginning means you never have to unlearn bad habits later.

10.10

Queries as Questions

The best way to approach a SQL query is to think of it as a question asked of a database in a structured language. Before you write a single line of SQL, ask yourself: what question am I trying to answer? What table holds the relevant data? What columns contain the information I need? What rows should be included or excluded?

Translating Questions into SQL

Consider the following plain-language questions and their SQL equivalents, using the library database.

SQL — Question: What books does the library have?
SELECT title, author_last, author_first
FROM books;
SQL — Question: What are patrons' email addresses?
SELECT first_name, last_name, email
FROM patrons;
SQL — Question: How many copies of each book does the library own?
SELECT title, copies_owned
FROM books;
A Framework for Query Writing

When approaching a new query, work through these steps in order.

  • State the question in plain language

    Write it out in a sentence. "What are the names of all active patrons?" or "Which books were published after 2020?"

  • Identify the table

    Which table holds the records you need? This becomes your FROM clause.

  • Identify the columns

    Which specific pieces of information does the result need? These become your SELECT list.

  • Identify any conditions

    Are you looking for all records, or only records that meet certain criteria? Conditions become your WHERE clause, covered in Chapter 11.

  • Identify any sorting or grouping

    Should results be sorted, grouped, or summarized? These become ORDER BY and GROUP BY clauses, covered in Chapters 11 and 12.

  • Write the SQL

    With the above decisions made, writing the actual SQL is straightforward.

  • Read the query back as a question

    Before running it, ask: does this say what I intended? Is the table name correct? Are the column names correct? Will this return all the records I need, and only the records I need?

The Query Is Not the Answer

A query is a question. The result set is the answer. But answering the right question requires understanding what the database actually contains. A query can be syntactically perfect, with no errors, and still return incorrect or misleading results if the question was wrong. For example, asking "how many books does the library own?" might return the count of rows in the books table. But if the question really means "how many physical copies does the library hold?", the answer requires summing the copies_owned column, which gives a different number entirely. The database gives you what you ask for, precisely.

Chapter Summary
  • SQL (Structured Query Language) is the standard language for working with relational databases. It is a declarative language, meaning you describe what result you want and the database engine determines how to retrieve it.
  • SQL statements are made up of clauses, each beginning with a keyword. Every statement ends with a semicolon. In Part III, the focus is on the SELECT statement, used for querying data.
  • The SELECT clause specifies which columns to include in the result set. The FROM clause specifies which table to look in. Together they form the minimum complete retrieval query. A SELECT query never changes the database.
  • When a query runs, the database returns a result set, which is itself a table with columns and rows. Result sets are temporary and are not stored permanently unless explicitly saved.
  • Column aliases, introduced with the AS keyword, rename columns in the result set for clarity without changing the underlying table.
  • SQL comments, written with -- for single lines or /* */ for multiple lines, add explanatory notes that the database engine ignores. Comments are essential for maintainable queries.
  • Readable SQL formatting, with uppercase keywords, lowercase identifiers, each major clause on its own line, and consistent indentation, makes queries much easier to read, write, and maintain.
  • The right approach to any query is to think of it as a question asked of a database. Working through a plain-language question before writing SQL, step by step, produces clearer and more accurate results.
Key Terms
SQL (Structured Query Language)

The standard language for querying, defining, manipulating, and controlling relational databases.

Declarative language

A style of programming or querying where you describe what result you want, rather than how to produce it step by step. SQL is declarative; Python is generally imperative.

Statement

A complete SQL instruction, ending with a semicolon, that the database engine can execute.

Clause

A component of a SQL statement that begins with a keyword and provides a specific kind of instruction, such as SELECT or FROM.

SELECT clause

The clause in a SQL query that specifies which columns to include in the result set.

FROM clause

The clause in a SQL query that specifies which table to retrieve data from.

Result set

The temporary table produced by executing a SQL query, containing columns and rows determined by the query. Result sets are not stored permanently unless explicitly saved.

Column alias

An alternative name given to a column in a result set using the AS keyword. The alias appears in the output but does not change the underlying table.

Comment

Text inside a SQL statement that the database engine ignores, written for human readers. Single-line comments use --; multi-line comments use /* */.

Keyword

A word reserved by SQL with special meaning, such as SELECT, FROM, WHERE, or JOIN. Written in uppercase by convention.

Identifier

A name defined by the database designer for a table, column, database, or view, such as students or first_name. Written in lowercase with underscores by convention.

SQLite

A lightweight, widely available relational database engine used as the primary SQL environment in this textbook. Requires no server installation.

DDL (Data Definition Language)

The subset of SQL used to define database structures, including creating and altering tables.

DML (Data Manipulation Language)

The subset of SQL used to insert, update, and delete records. Covered in Part IV.

DCL (Data Control Language)

The subset of SQL used to grant and revoke permissions for users and roles. Connects to Chapter 18.

Hint: when two identifiers appear next to each other in a SELECT clause with no comma between them, SQL interprets the second one as an alias for the first.

Real-World Applications

Think about each of the following scenarios and answer the questions using the ideas from this chapter.

Scenario 1 — School Records System

A school database includes a table called students with columns for student_id, given_name, family_name, grade_level, homeroom_teacher, and date_enrolled. A school counselor wants a list of student names to use in a letter-writing project. The list does not need to include any other information. Write the SQL query the counselor should use, and explain why you chose those specific columns.

Scenario 2 — Hospital Appointment System

A hospital information system has a table called appointments with columns including appointment_id, patient_id, doctor_id, department, appointment_date, appointment_time, reason, and status. A hospital administrator runs a query using SELECT * to view a list of upcoming appointments. Describe one potential problem with using SELECT * in this context. What would a more appropriate query look like?

Scenario 3 — A Query Without a Clear Purpose

A colleague hands you the following query and asks if it looks correct:

SQL
SELECT product_name, price, discount_rate
FROM inventory;

The query is syntactically correct. But your colleague says the result "doesn't look right." What questions would you ask them to help diagnose the problem? What kinds of issues might produce an unexpected result from a syntactically correct query?

Scenario 4 — Commenting for a Team

You work on a team that shares SQL queries through a central repository. Your manager asks you to add comments to all existing queries because new team members find them hard to understand. You have the following query:

SQL — Before Comments
SELECT customer_id, order_date, total_amount, shipping_status
FROM orders;

Add appropriate comments to this query. Write at least a short description of what the query does and which table it uses. Decide whether a single-line or multi-line comment style is more appropriate.

Preview of Chapter 11
Coming Next — Chapter 11: Selecting, Filtering, and Sorting Data

Chapter 10 introduced the minimum SQL retrieval query: SELECT and FROM. Every query so far has returned all rows in a table. Chapter 11 changes that.

Chapter 11 introduces the WHERE clause, which lets you filter a result set to include only the rows that meet specific conditions. With WHERE, you move from retrieving all rows to retrieving exactly the rows you need. You will learn comparison operators such as =, <, >, <=, >=, and <>. You will filter text values with equality checks. You will combine conditions using logical operators AND, OR, and NOT. You will use IN to match against a list of values, BETWEEN to match a range, and IS NULL to find records with missing values.

Chapter 11 also introduces ORDER BY for sorting results in ascending and descending order. Chapter 11 is where queries begin to do real work, and where the connection between a question, a condition, and a result set becomes most clear.
