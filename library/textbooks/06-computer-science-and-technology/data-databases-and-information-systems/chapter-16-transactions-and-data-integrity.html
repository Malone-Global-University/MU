Transactions and Data Integrity

A bank transfer is not one change to a database, it is two. A library checkout is not one change, it is two. When several changes describe a single real action, they must succeed together or fail together, and this chapter introduces the tool databases use to guarantee exactly that.

16.10Designing for Reliable Changes

Chapter 15 taught students how to insert, update, and delete records one command at a time. That chapter emphasized caution, and for good reason: a single UPDATE or DELETE can change many records at once, and the results are permanent unless the system is designed to protect against them.

But there is a deeper problem that single commands cannot solve on their own. Some real-world actions require several database changes that must all succeed together, or all fail together. A single transfer of money from one bank account to another is not one database operation. It is two: a withdrawal from one account, and a deposit into another. If the withdrawal succeeds and the deposit fails, the money does not disappear into nothing in real life, but the database will say it did. That is a serious problem.

This kind of problem, where a partial completion leaves the database in a state that does not match reality, is the core problem that transactions are designed to solve.

Chapter 16 introduces transactions as the tool databases use to keep changes reliable, consistent, and safe. Students will learn what data integrity means, why some changes belong together, how transactions work, what commit and rollback do, and what atomicity means at an introductory level. The chapter will also introduce consistency, concurrency at a basic conceptual level, and worked examples drawn from realistic systems: bank transfers, course enrollments, online orders, library checkouts, and appointment reservations.

Note for New Students

By the end of this chapter, you should understand that reliable database systems are not just technically correct. They are designed to protect the truth of the data even when operations fail, connections drop, or multiple users interact with the same records at the same time.

16.1

What Data Integrity Means

Data integrity means that the data stored in a database accurately and consistently represents the reality it is supposed to describe.

That sentence sounds simple, but think about what it requires.

A school's student information system stores enrollment records. If a student is enrolled in a course, the database should say so. If the student drops the course, the enrollment record should be removed or marked inactive. If the student never enrolled, there should be no enrollment record. The data should match the real state of enrollment at all times.

A hospital's appointment system stores patient appointments. If a patient has a confirmed appointment with a specific doctor at a specific time, the database should reflect that. If the appointment is cancelled, the record should change. No two patients should be recorded as having the same appointment slot with the same doctor at the same time.

A library checkout system stores which resources are currently on loan. If a book is checked out, the loan record should exist, and the resource status should indicate unavailability. When the book is returned, the loan should be closed and the resource should be available again.

In all of these cases, data integrity means the database tells the truth. It does not just mean the SQL was written correctly. It means the data reflects reality accurately, completely, and consistently at every moment.

How Data Integrity Breaks Down
Missing Data

Happens when a record that should exist does not. If a student enrolled in a course but no enrollment record was created, the database is missing information.

Incorrect Data

Happens when a record exists but contains wrong values. If a student's grade is recorded as 95 when the actual grade was 59, the data is inaccurate.

Inconsistent Data

Happens when two parts of the database disagree with each other, such as one table saying a student is enrolled while another says the same student has no active enrollments.

Orphaned Data

Happens when a record refers to something that no longer exists, such as a loan record referencing a resource that has been deleted.

Central Idea

Partially complete data happens when a multi-step operation is interrupted before all steps finish. This is the problem that transactions are primarily designed to solve. Data integrity is not a single feature of a database, it is the result of careful design choices, enforced constraints, validation rules, well-written queries, controlled access, and reliable transaction handling. Chapter 16 focuses on the transaction side: how databases ensure that multi-step operations are completed correctly, or not at all.

16.2

Why Some Changes Belong Together

Consider a library resource checkout. When a patron checks out a book, two things need to happen:

  1. A loan record is created, recording who borrowed the book, which book it is, and when the loan started.
  2. The book's availability status is updated to indicate it is on loan.

Both changes must succeed. If the loan record is created but the availability status is not updated, the database says the book is available when it is actually on loan. Another patron could try to borrow the same book, or the system could report the book as available in search results when it is not. The data no longer matches reality.

If the availability status is updated but the loan record is not created, the database says the book is on loan but does not say to whom. The library cannot track who has the book, or when it is due back.

These two changes belong together. Either both happen, or neither should happen.

A More Dramatic Example: A Bank Transfer

A customer transfers 500 from their savings account to their checking account. The database must:

  1. Subtract 500 from the savings account balance.
  2. Add 500 to the checking account balance.

If step one succeeds and step two fails, the savings account loses 500 and the checking account gains nothing. The customer is missing 500. That is not a minor data inconsistency, it is a serious error that causes real harm.

If step two succeeds and step one fails, the checking account gains 500 and the savings account is unchanged. The bank now has a record of money appearing from nowhere. That is also a serious problem.

The only acceptable outcomes are: both changes succeed together, or neither change is applied at all.

A Third Example: Course Enrollment

Consider a course enrollment system. When a student enrolls in a course:

  1. An enrollment record is created linking the student and the course.
  2. The course's current enrollment count is updated.
  3. If the course has a waitlist, the student's position on the waitlist may be removed.

If only some of these steps succeed, the enrollment count may not match the actual number of enrolled students, or a student may appear on the waitlist of a course they are also enrolled in.

These examples all share the same structure. Multiple database changes are logically connected to a single real-world action. The changes must be treated as a unit. That is what a transaction is for.

16.3

Transactions

A transaction is a sequence of database operations treated as a single unit of work. Either all of the operations in the transaction succeed and are saved, or none of them are applied to the database at all.

Transactions give databases a way to handle multi-step operations safely. Instead of executing each INSERT, UPDATE, or DELETE independently and hoping nothing goes wrong in between, the database groups the operations into a transaction, executes them, and then either commits the entire group or rolls it all back.

The basic idea is simple:

  1. Begin the transaction.
  2. Execute the database operations.
  3. If everything succeeded, commit: make the changes permanent.
  4. If anything went wrong, rollback: undo all changes and leave the database as it was before the transaction started.

Here is what a transaction looks like in SQL:

SQL — A Successful Transaction
BEGIN TRANSACTION;

UPDATE accounts
SET balance = balance - 500
WHERE account_id = 101;

UPDATE accounts
SET balance = balance + 500
WHERE account_id = 202;

COMMIT;

If both UPDATE statements succeed, the COMMIT saves both changes permanently. The database now reflects the completed transfer.

If anything goes wrong between BEGIN TRANSACTION and COMMIT, the database can roll back:

SQL — A Rolled-Back Transaction
BEGIN TRANSACTION;

UPDATE accounts
SET balance = balance - 500
WHERE account_id = 101;

-- Something goes wrong here

ROLLBACK;

After the ROLLBACK, the savings account balance is exactly what it was before the transaction started. No change is visible to other parts of the system. The database is as if the transaction never began.

A Note on Different Database Systems

Not every database system requires an explicit BEGIN TRANSACTION. Some systems run each statement as its own automatic transaction by default, meaning each statement is automatically committed unless told otherwise. SQLite, which is used in this textbook for examples, supports explicit BEGIN TRANSACTION, COMMIT, and ROLLBACK. Other common systems, such as PostgreSQL and MySQL, also support explicit transactions. It is worth noting that even a single INSERT, UPDATE, or DELETE statement is technically a transaction in most database systems, it either fully succeeds or fully fails. But when multiple statements need to behave as a unit, explicit transaction control is required.

16.4

Commit

Commit is the operation that makes a transaction's changes permanent. Once a transaction is committed, its changes are saved to the database and become visible to other users and operations.

Before a commit, the changes made inside a transaction may be visible to the session that started the transaction, but other database connections typically cannot see the uncommitted changes. This is an important safety feature. Other parts of the system continue working with the last committed state of the database while the transaction is in progress.

Once commit happens:

  • The changes are written to permanent storage.
  • Other users and processes can see the updated data.
  • The changes cannot be undone with a rollback.

In SQL, committing is as simple as:

SQL — Commit
COMMIT;

Some systems also accept:

SQL — Commit, Alternate Form
COMMIT TRANSACTION;

A commit should only happen after all steps in the transaction have been verified to succeed. Committing too early, before all steps have completed, defeats the purpose of the transaction. If you commit after the first of two required updates and then the second update fails, you have already permanently applied a partial change with no way to roll it back.

Good application code checks for errors after each step before committing. If any step fails, the code issues a rollback instead of a commit.

Here is a simplified example showing how application logic might handle this:

Pseudocode — Checking Before Committing
Begin transaction
    Try: subtract from account A
    If error: rollback and stop

    Try: add to account B
    If error: rollback and stop

    Commit

The commit only happens if both steps succeed without errors. If either step fails, the rollback is triggered and neither change is saved.

16.5

Rollback

Rollback is the operation that undoes all changes made during a transaction that has not yet been committed. When a rollback occurs, the database returns to the state it was in before the transaction began.

Rollback is a powerful safety mechanism. It means that a transaction can begin, make several changes, discover a problem, and then cancel everything without leaving the database in a partially changed state.

In SQL:

SQL — Rollback
ROLLBACK;

Rollback can happen in several situations:

Application Code Triggers a Rollback

If the application detects an error, such as a failed constraint check, an unexpected value, or a programming error, it can explicitly issue a ROLLBACK command.

The Database Engine Triggers an Automatic Rollback

Some systems automatically roll back a transaction if they detect a violation of a constraint or integrity rule that cannot be resolved.

A Connection Is Lost

If the application or network connection drops before a transaction is committed, most database systems automatically roll back the incomplete transaction. The database does not assume the transaction was successful just because it was started.

A System Failure Occurs

If the database server crashes in the middle of a transaction, the recovery process after restart will roll back any transactions that had not yet been committed. The database is restored to its last fully committed state.

Rollback is not an error message or a punishment. It is a feature. It means the database can safely attempt operations without the risk of leaving data permanently broken if something goes wrong.

Why This Matters

Consider what happens without rollback capability. If a bank transfer begins, subtracts from the savings account, and then fails before adding to the checking account, there is no automatic way to undo the subtraction. A developer would have to manually find the affected record and fix it. If many such failures occur, some might go unnoticed. Customers lose money. The bank's records become unreliable. With transactions and rollback, the system automatically handles partial failures. The savings account is not affected unless the entire transfer succeeds.

16.6

Atomicity at an Introductory Level

The word atomic comes from the Greek word for indivisible. An atomic operation is one that cannot be partially completed. Either it happens entirely, or it does not happen at all.

In database systems, atomicity is the property that guarantees a transaction is treated as a single indivisible unit. All of the operations in the transaction succeed together, or none of them are applied.

Atomicity is the first and most fundamental guarantee of reliable transactions.

A Helpful Comparison

Think of atomicity like a light switch. A light switch is either on or off. There is no state where the switch is halfway on, flickering between states, or on for some lights but not others. Atomicity gives database transactions the same binary guarantee: committed, or not committed.

Without atomicity, a transaction might:

  • Complete three of five steps and stop.
  • Apply changes to some tables but not others.
  • Write partial data that violates foreign key constraints or business rules.
  • Leave related records in disagreement with each other.

With atomicity, none of those partial states can persist. The transaction either reaches COMMIT with all steps completed, or the entire thing is rolled back.

A Worked Example With Three Steps

A student completes an online course registration. The system must:

  1. Insert a row into the Enrollments table linking the student and the course.
  2. Update the Courses table to increment the current enrollment count.
  3. Insert a confirmation record into the Notifications table so the student receives a confirmation email.
SQL — A Three-Step Atomic Transaction
BEGIN TRANSACTION;

INSERT INTO Enrollments (student_id, course_id, enrolled_date)
VALUES (1042, 301, '2024-09-01');

UPDATE Courses
SET current_enrollment = current_enrollment + 1
WHERE course_id = 301;

INSERT INTO Notifications (student_id, message, sent_date)
VALUES (1042, 'You are enrolled in Course 301.', '2024-09-01');

COMMIT;

If all three statements succeed, the commit saves all three changes. The student is enrolled, the count is accurate, and the notification is queued.

If the second INSERT fails because the Notifications table has a problem, the rollback cancels the entire transaction. The enrollment record is removed, the course count is unchanged, and no notification exists. The database is exactly as it was before.

Atomicity does not mean the operations happen instantaneously. They happen in sequence. Atomicity means that, from the perspective of the final saved state of the database, either all of the changes exist or none of them do. Intermediate states during execution are temporary and private to the transaction.

16.7

Consistency and Rules

Consistency in database systems means that a transaction takes the database from one valid state to another valid state. A valid state is one in which all defined rules, constraints, and integrity conditions are satisfied.

Before a transaction begins, the database should be in a consistent state. After the transaction commits, the database should still be in a consistent state. The transaction itself may temporarily create intermediate states that violate some rules while in progress, but those intermediate states are hidden inside the transaction. They are never committed to the database as a permanent result.

What Counts as a Consistent State
  • All primary key values are unique.
  • All foreign key references point to records that actually exist.
  • All required fields contain values.
  • All check constraints are satisfied.
  • All unique constraints are satisfied.
  • Business rules defined in the database are respected.

For example, a database might have a rule that a course cannot have more enrolled students than its maximum capacity. This might be expressed as a check constraint:

SQL — A Check Constraint
CHECK (current_enrollment <= max_capacity)

If a transaction tries to enroll a student in a course that has already reached its maximum, the database will reject the transaction. The constraint prevents the database from entering an inconsistent state where enrollment exceeds capacity.

Another example: a foreign key constraint ensures that every enrollment record references a valid student and a valid course. If a transaction tries to insert an enrollment for a student ID that does not exist in the Students table, the foreign key constraint rejects it. The database will not allow an enrollment to reference a non-existent student.

Consistency depends on constraints being defined and enforced. It also depends on application code following business rules that cannot easily be expressed as database constraints. For example, a database constraint cannot easily enforce a rule like "students may not enroll in more than six courses per semester." That rule might be enforced in application code before the transaction is submitted.

How Atomicity and Consistency Relate

Atomicity ensures that transactions do not leave partial changes behind. Consistency ensures that the changes that are saved are valid. Together, they protect the reliability of the database.

16.8

Concurrency at an Introductory Level

Most databases used in real systems are not used by one person at a time. Multiple users connect, query, insert, update, and delete data simultaneously. This creates a situation called concurrency: many operations happening at the same time against the same database.

Concurrency creates challenges for data integrity. When two transactions are running at the same time and accessing or modifying the same data, problems can occur if the database does not manage concurrent access carefully.

Scope of This Section

This section introduces concurrency at a conceptual level. The details of concurrency management, locking strategies, and isolation levels are advanced topics beyond this textbook. But students should understand the basic problem so they appreciate why transaction management matters even in single-operation systems.

The Lost Update Problem

This occurs when two transactions read the same value, both make changes based on that value, and then both write their results back. The second write overwrites the first.

Imagine two library staff members updating the same resource record at the same moment. Staff member A reads the resource record, sees its status as Available, and begins updating it to On Loan. Staff member B also reads the resource record, also sees Available, and begins a different update. When both finish and write their changes, one of the updates will overwrite the other. One change will be silently lost.

The Dirty Read Problem

This occurs when one transaction reads data that another transaction has modified but not yet committed. If the second transaction is later rolled back, the first transaction was working with data that was never actually saved, producing incorrect results.

The Inconsistent Read Problem

This occurs when a transaction reads the same data twice during its execution and gets different values, because another transaction modified and committed the data between the two reads.

Databases handle concurrency through locking and isolation levels. Locking prevents multiple transactions from modifying the same data at exactly the same moment. Isolation levels define how visible one transaction's uncommitted changes are to other transactions.

For the purposes of this chapter, the key idea is this: databases in real systems must be designed and configured to handle multiple simultaneous users safely. Transactions are the mechanism that makes it possible to reason about groups of changes as units. Locking and isolation mechanisms make it possible for those transactions to coexist without corrupting each other's work.

Even in small applications where concurrency seems unlikely, it is worth designing transactions correctly. Web applications can receive many simultaneous requests. Shared library systems may have multiple staff members or students accessing the same records. Correct transaction design is a habit worth building from the beginning.

16.9

Transaction Examples

This section walks through several worked examples of transactions drawn from realistic information systems. For each example, the relevant steps are identified, the SQL is shown, and the behavior under failure is discussed.

Example 1 — Library Book Checkout

Scenario.A patron checks out a library book. The system must record the loan and update the book's availability.

Steps.Insert a row into the Loans table, then update the Resources table to set availability to unavailable.

SQL — Library Checkout Transaction
BEGIN TRANSACTION;

INSERT INTO Loans (resource_id, user_id, loan_date, due_date)
VALUES (5023, 1088, '2024-11-01', '2024-11-15');

UPDATE Resources
SET available = 0
WHERE resource_id = 5023;

COMMIT;

If something goes wrong:The rollback removes the loan record and the resource remains marked as available. No partial checkout state exists.

Note on design.The available field in the Resources table is derived from loan data. In a more sophisticated system, availability might be determined by querying the Loans table rather than storing it separately, a design that avoids update anomalies. But for illustration, the example above shows how the two operations are logically linked.

Example 2 — Bank Account Transfer

Scenario.A customer transfers 200 from their savings account to their checking account.

Steps.Reduce the savings account balance by 200, then increase the checking account balance by 200.

SQL — Bank Transfer Transaction
BEGIN TRANSACTION;

UPDATE Accounts
SET balance = balance - 200
WHERE account_id = 3301
  AND account_type = 'savings';

UPDATE Accounts
SET balance = balance + 200
WHERE account_id = 3302
  AND account_type = 'checking';

COMMIT;

If the savings account has insufficient funds:The first update would complete, but application code or a check constraint could detect the negative balance and issue a rollback before the commit. The savings account balance is restored and no transfer occurs.

If the second update fails:The rollback cancels the reduction of the savings account. No money is lost.

Why this matters.Financial systems were among the earliest use cases for reliable transaction support. The correctness of financial data depends directly on atomic, consistent transaction handling.

Example 3 — Student Course Enrollment

Scenario.A student enrolls in a course. The system must create the enrollment record and update the course's current enrollment count.

Steps.Check that the course has available seats, insert an enrollment record, then increment the course enrollment count.

SQL — Course Enrollment Transaction
BEGIN TRANSACTION;

INSERT INTO Enrollments (student_id, course_id, enrolled_date, status)
VALUES (4451, 201, '2024-08-15', 'Active');

UPDATE Courses
SET current_enrollment = current_enrollment + 1
WHERE course_id = 201;

COMMIT;

If the course is at capacity:A check constraint on the Courses table rejects the UPDATE, or the application detects the capacity limit and issues a rollback. The enrollment record is removed and the count remains unchanged.

If the student is already enrolled:A uniqueness constraint on the Enrollments table rejects the duplicate INSERT. The rollback cancels the transaction. The enrollment count is not changed.

Example 4 — Online Order Placement

Scenario.A customer places an order for a product. The system must create the order record, add order line items, and reduce the product inventory.

Steps.Insert an order record, insert order line items, then reduce the available quantity in inventory.

SQL — Online Order Transaction
BEGIN TRANSACTION;

INSERT INTO Orders (customer_id, order_date, status)
VALUES (7721, '2024-11-05', 'Pending');

INSERT INTO Order_Items (order_id, product_id, quantity, unit_price)
VALUES (LAST_INSERT_ID(), 8801, 2, 24.99);

UPDATE Inventory
SET quantity_available = quantity_available - 2
WHERE product_id = 8801;

COMMIT;

Note on LAST_INSERT_ID().This function retrieves the auto-generated primary key from the most recently inserted row. Its exact syntax varies by database system. In SQLite, the equivalent is last_insert_rowid().

If the inventory is insufficient:Application code detects that the requested quantity exceeds available stock and issues a rollback. No order is created.

If payment is also part of the transaction:A complete e-commerce transaction might include a step to record the payment authorization. If payment fails, the entire transaction rolls back. No order exists and inventory is unchanged.

Example 5 — Appointment Scheduling

Scenario.A patient schedules an appointment with a doctor. The system must create the appointment record and update the doctor's schedule to mark the time slot as occupied.

Steps.Insert an appointment record, then update the schedule to mark the time slot as unavailable.

SQL — Appointment Scheduling Transaction
BEGIN TRANSACTION;

INSERT INTO Appointments (patient_id, doctor_id, appointment_date, time_slot, status)
VALUES (9901, 112, '2024-12-10', '14:00', 'Scheduled');

UPDATE Doctor_Schedule
SET slot_available = 0
WHERE doctor_id = 112
  AND schedule_date = '2024-12-10'
  AND time_slot = '14:00';

COMMIT;

If the time slot was already taken:A uniqueness constraint on Doctor_Schedule rejects the update, or returns an affected row count of zero. Application code detects this and issues a rollback. The appointment record is removed. The patient is notified that the slot is no longer available.

Why this scenario is sensitive.In a concurrent system with many patients booking simultaneously, two patients might attempt to book the same slot at the same moment. Proper locking and isolation levels ensure that only one succeeds and the other receives a clear failure response. This is one of the most common real-world transaction scenarios.

16.10

Designing for Reliable Changes

Designing a database system for reliable changes means thinking about transactions from the beginning, not adding them as an afterthought. Students who understand transactions early will write better queries, design better tables, and build more trustworthy systems.

Here are the key design habits that support reliable transaction behavior.

Identify Which Operations Belong Together

Before writing SQL, think about which real-world actions require multiple database steps. Make a list. For each action, ask: what must all succeed together? Those steps belong in a single transaction.

For a library checkout: the loan creation and the availability update belong together. For a money transfer: both balance changes belong together. For a course enrollment: the enrollment record and the seat count update belong together.

Keep Transactions Short

Long transactions hold locks on data for extended periods. This can slow down other users who need access to the same records. A transaction should do only what is necessary, and commit or rollback as quickly as possible.

Avoid putting user-facing pauses inside a transaction. For example, do not begin a transaction, then wait for a user to fill out a form, then commit after they submit. The database should not hold the transaction open while waiting for human interaction. Start the transaction only when all inputs are ready.

Check for Errors at Each Step

Application code should check whether each database operation succeeded before moving to the next step. If any step returns an error, the code should issue a rollback immediately rather than trying to continue. This is especially important for operations that might fail due to constraint violations, missing records, insufficient inventory, or business rule violations.

Do Not Commit Partial Work Intentionally

Some developers are tempted to commit after each step of a multi-step operation to avoid losing progress. This defeats the purpose of transactions. If the system commits after step two of five and then fails at step three, the database contains permanent partial work with no way to roll it back automatically.

Commit only when all steps are complete. If you are worried about a very long sequence of steps, reconsider whether the sequence can be shortened or broken into smaller logical units with their own complete transactions.

Use Constraints to Support Transaction Integrity

Constraints such as primary keys, foreign keys, uniqueness constraints, and check constraints help transactions fail cleanly when they encounter violations. A transaction that fails due to a constraint violation produces a clear error. The rollback removes the partial changes. The constraint did its job.

Without constraints, a bad transaction might succeed and commit data that violates business rules, integrity requirements, or relational structure. Constraints are a first line of defense. Transactions are the second.

Plan for Failure

Good database design assumes that things will go wrong. Connections drop, servers restart unexpectedly, users provide unexpected input, business rules change. Design transactions to handle failure gracefully. The goal is that after any failure, at any point in the process, the database contains only data that represents a complete and valid state.

The question to ask is: if this transaction is interrupted halfway through, what state does the database end up in? The answer should always be: the state it was in before the transaction started.

Document Transaction Boundaries

When writing documentation or specifications for a database system, explicitly describe which operations are grouped into transactions. This makes the system easier to maintain, debug, and extend. Future developers who work on the system will understand which steps must stay together, and why.

Chapter Summary
  • Data integrity means that the database accurately and consistently represents the reality it is designed to describe. Integrity breaks down through missing data, incorrect data, inconsistent data, orphaned data, and partially completed operations.
  • Some real-world actions require multiple database operations that must all succeed together, or all fail together. A bank transfer, a library checkout, a course enrollment, an appointment booking, and an online order are all examples of multi-step operations where partial completion causes serious problems.
  • A transaction is a sequence of database operations treated as a single unit of work. Either all operations are committed and made permanent, or none of them are applied. The key SQL commands for explicit transaction control are BEGIN TRANSACTION, COMMIT, and ROLLBACK.
  • Commit makes a transaction's changes permanent and visible to other users. Rollback undoes all changes made during a transaction that has not yet been committed, returning the database to its state before the transaction began.
  • Atomicity is the property that guarantees a transaction is treated as an indivisible unit, all operations succeed together or none of them are applied. Consistency means that a transaction takes the database from one valid state to another valid state, satisfying all constraints and rules. Together, atomicity and consistency protect the integrity of the database.
  • Concurrency occurs when multiple transactions run simultaneously against the same database, creating risks such as lost updates, dirty reads, and inconsistent reads. Databases manage concurrency through locking and isolation levels, which are advanced topics, but students should understand the basic problem and the importance of correct transaction design even in systems that seem unlikely to have concurrent users.
  • Designing for reliable changes means identifying which operations belong together, keeping transactions short, checking for errors at each step, never committing partial work, using constraints to support transaction integrity, planning for failure, and documenting transaction boundaries.
Key Terms
Data Integrity

The property of a database in which the stored data accurately, completely, and consistently represents the reality it is designed to describe.

Transaction

A sequence of database operations treated as a single unit of work. All operations either succeed and are committed, or the entire set of changes is rolled back.

BEGIN TRANSACTION

The SQL command that starts an explicit transaction.

COMMIT

The SQL command that makes a transaction's changes permanent and visible to other users. Once committed, changes cannot be undone by a rollback.

ROLLBACK

The SQL command that undoes all changes made during a transaction that has not yet been committed. The database is returned to the state it was in before the transaction began.

Atomicity

The property that guarantees a transaction is treated as an indivisible unit. Either all operations in the transaction succeed and are committed, or none of them are applied.

Consistency

The property that guarantees a transaction takes the database from one valid state to another valid state, satisfying all defined constraints and rules.

Concurrency

The situation in which multiple transactions run simultaneously against the same database. Concurrency requires careful management to prevent data integrity problems.

Lost Update

A concurrency problem in which two transactions read the same data, both make changes, and the second write overwrites the first, silently discarding one of the updates.

Dirty Read

A concurrency problem in which one transaction reads data that another transaction has modified but not yet committed.

Locking

A mechanism databases use to prevent multiple transactions from modifying the same data at exactly the same moment, reducing concurrency-related integrity problems.

Isolation Level

A setting that controls how visible one transaction's uncommitted changes are to other concurrent transactions.

Partial Completion

A state in which some but not all steps of a multi-step operation have been applied. Transactions are designed to prevent partial completion from becoming permanent.

  • What are the two operations that should be grouped into a transaction?

Coming Next — Chapter 17: Indexes and Query Performance at an Introductory Level

Chapter 16 explained how transactions keep multi-step database operations safe and consistent. Chapter 17 turns attention to a different kind of database reliability: performance.

As databases grow larger, queries that once ran quickly may begin to slow down. A query that searches through ten records completes almost instantly. The same query searching through ten million records may take much longer, especially without the right database structures in place. Chapter 17 introduces indexes as structures that help databases find data faster, why primary key indexes exist automatically, when to create additional indexes, and what tradeoffs indexes involve. The chapter will remain conceptual and practical, appropriate for students who are not yet studying advanced database administration, but who need to understand why performance is a design concern from the start.
