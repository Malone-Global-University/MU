Selecting, Filtering, and Sorting Data

A query that returns every row in a table is rarely useful. Real questions demand specific answers: students in a particular grade, books published after a certain year, records with missing values. This chapter teaches the WHERE clause, comparison and logical operators, NULL handling, ORDER BY, and LIMIT — the tools that turn a database into something genuinely useful.

Chapter 10 introduced SQL as a language for asking questions of a database. You learned how to write a basic SELECT statement, choose specific columns, give columns aliases, and read a result set. Every query in that chapter retrieved all the rows in a table.

Real questions are almost never that broad. A librarian does not want a list of every book ever cataloged. She wants books published after a certain year, or books on a particular subject, or books that are currently available. A school administrator does not want every student record. He wants students in a specific grade, or students with missing contact information, or students enrolled in a particular course.

This chapter teaches you how to retrieve specific records rather than everything in a table. You will learn to add conditions that filter which rows appear in a result, to arrange results in a meaningful order, and to limit how many rows come back at once. These skills, filtering and sorting, turn a database into something far more useful than a pile of stored facts.

By the end of this chapter you should be able to write and read SQL queries that select specific columns, filter rows by conditions, combine conditions with logical operators, handle missing values, sort results in ascending or descending order, and limit result sets to a manageable size.

11.2

The WHERE Clause

The WHERE clause appears after the FROM clause and before any grouping or sorting. It specifies a condition that each row must satisfy in order to appear in the result set. Rows that do not satisfy the condition are excluded entirely.

The basic structure of a filtered query looks like this.

SQL — Basic Filtered Query Structure
SELECT column1, column2
FROM table_name
WHERE condition;

The database evaluates the condition for every row in the table. If the condition is true for a row, that row is included in the results. If the condition is false or cannot be determined, that row is excluded.

Here is a query that retrieves only the students in grade 11.

SQL
SELECT student_id, last_name, first_name, grade_level
FROM Students
WHERE grade_level = 11;

The database reads this as: go to the Students table, check every row, and return only the rows where grade_level equals 11.

Result
student_idlast_namefirst_namegrade_level
1001AdeyemiChioma11
1005DialloAminata11
1008OwusuKwame11

Three rows satisfy the condition. Five rows do not, so they are excluded.

The Equals Sign in SQL

Notice that the WHERE clause uses a single equals sign (=) to test for equality. This is different from many programming languages, which use == for comparison and = for assignment. In SQL, = always means comparison inside a WHERE clause.

Clause Order

The SQL clause order for the queries in this chapter follows this pattern.

SQL — Clause Order
SELECT   ...
FROM     ...
WHERE    ...
ORDER BY ...

Each clause has a specific job and a specific place. SELECT chooses columns. FROM names the table. WHERE filters rows. ORDER BY sorts rows. You will learn ORDER BY in Section 11.8.

11.4

Filtering Text

When you filter text columns, you must place the comparison value inside single quotation marks. Text comparisons are case-sensitive in some database systems and case-insensitive in others. SQLite is case-insensitive for ASCII characters by default, but you should not rely on this behavior.

SQL — Finding a student by last name
SELECT student_id, last_name, first_name
FROM Students
WHERE last_name = 'Mensah';
Result
student_idlast_namefirst_name
1003MensahAkosua
Partial Text Matching with LIKE

The = operator requires an exact match. Sometimes you want to find rows where a text column contains a pattern, starts with certain characters, or ends with certain characters. The LIKE operator handles pattern matching.

LIKE uses two special wildcard characters.

CharacterMeaning
%Matches any sequence of zero or more characters.
_Matches exactly one character.

Last names that begin with the letter O:

SQL
SELECT student_id, last_name, first_name
FROM Students
WHERE last_name LIKE 'O%';
Result
student_idlast_namefirst_name
1002OkaforEmeka
1008OwusuKwame

Last names that contain the letter sequence an anywhere:

SQL
SELECT student_id, last_name, first_name
FROM Students
WHERE last_name LIKE '%an%';
Result
student_idlast_namefirst_name
1004BandaTendai

First names that are exactly six characters long, using six underscore wildcards:

SQL
SELECT student_id, last_name, first_name
FROM Students
WHERE first_name LIKE '______';
Result
student_idlast_namefirst_name
1001AdeyemiChioma
1004BandaTendai
Count Your Underscores Carefully

Each underscore matches exactly one character. Six underscores match names of exactly six letters. Always count carefully when using underscore wildcards. A single extra or missing underscore changes which rows appear.

NOT LIKE

You can negate a LIKE condition with NOT LIKE.

SQL
SELECT student_id, last_name, first_name
FROM Students
WHERE last_name NOT LIKE 'O%';

This returns all students whose last name does not begin with O.

11.5

Logical Operators: AND, OR, and NOT

A single condition is often not enough. You may need to combine conditions. SQL provides three logical operators for this purpose: AND, OR, and NOT.

AND

AND requires that both conditions be true. A row must satisfy every condition connected by AND in order to appear in the result.

Students in grade 11 who also have a GPA above 3.5:

SQL
SELECT student_id, last_name, first_name, grade_level, gpa
FROM Students
WHERE grade_level = 11
  AND gpa > 3.5;
Result
student_idlast_namefirst_namegrade_levelgpa
1001AdeyemiChioma113.82
1008OwusuKwame113.55

Diallo (Aminata) is in grade 11 but has a GPA of 2.48, so she is excluded. No grade-10 or grade-12 student appears regardless of GPA.

OR

OR requires that at least one condition be true. A row appears if it satisfies either condition, or both.

Students in grade 10 or grade 12:

SQL
SELECT student_id, last_name, first_name, grade_level
FROM Students
WHERE grade_level = 10
   OR grade_level = 12;
Result
student_idlast_namefirst_namegrade_level
1002OkaforEmeka10
1003MensahAkosua12
1004BandaTendai10
1006NkosiSipho12
1007KariukiWanjiru10

Grade-11 students are excluded because they satisfy neither condition.

NOT

NOT reverses a condition. It returns rows for which the condition is false.

SQL
SELECT student_id, last_name, first_name, advisor_id
FROM Students
WHERE NOT advisor_id = 201;

This is equivalent to WHERE advisor_id <> 201. Both forms work. The <> form is usually cleaner for simple comparisons, but NOT becomes useful with more complex conditions like NOT IN and NOT LIKE.

Combining AND and OR

When you combine AND and OR in a single condition, the order of evaluation matters. SQL evaluates AND before OR, just as multiplication is evaluated before addition in arithmetic. This means the following two queries can produce different results.

SQL — Query A: AND evaluated first
-- Reads as: grade 12, OR (grade 11 AND gpa > 3.5)
SELECT student_id, last_name, first_name, grade_level, gpa
FROM Students
WHERE grade_level = 12
   OR grade_level = 11
  AND gpa > 3.5;
SQL — Query B: parentheses make intent explicit
-- Reads as: (grade 11 or 12) AND gpa > 3.5
SELECT student_id, last_name, first_name, grade_level, gpa
FROM Students
WHERE (grade_level = 12 OR grade_level = 11)
  AND gpa > 3.5;

If Mensah had a GPA of 2.80, Query A would include her (she is in grade 12) while Query B would exclude her (grade 12 but not above 3.5). Parentheses change the meaning of the condition.

Always Use Parentheses with AND and OR

Use parentheses whenever you combine AND and OR in the same condition. Parentheses make your intentions clear, protect you from evaluation-order surprises, and make the query easier to read and maintain.

11.6

IN and BETWEEN

IN

When you want to check whether a column value equals any one of several specific values, you could use multiple OR conditions. The IN operator provides a cleaner way to write the same condition.

SQL — Multiple OR vs. IN
-- Longer form using OR
WHERE grade_level = 10
   OR grade_level = 11
   OR grade_level = 12

-- Cleaner form using IN
WHERE grade_level IN (10, 11, 12)

The IN operator takes a list of values in parentheses and returns rows where the column matches any value in the list.

11.7

Missing Values with IS NULL

Kariuki (Wanjiru), student 1007, has NULL in the gpa column. NULL means the value is unknown, missing, or not yet recorded. It does not mean zero. It does not mean an empty string. It means absence of information.

NULL requires special treatment in SQL because it does not behave like a regular value. You cannot test for NULL using =. The following query does not work as intended.

SQL — Incorrect NULL test
-- This does NOT find rows with a missing GPA
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa = NULL;

This query returns no rows at all, regardless of the data. The reason is that any comparison with NULL using =, <>, or other operators produces an unknown result, not true or false. The database treats unknown as "not true" and excludes those rows.

To test for NULL, you must use IS NULL.

SQL — Correct NULL test
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa IS NULL;
Result
student_idlast_namefirst_namegpa
1007KariukiWanjiruNULL

To find rows where a column has a value, use IS NOT NULL.

SQL
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa IS NOT NULL;
Result — Seven rows, all with actual GPA values
student_idlast_namefirst_namegpa
1001AdeyemiChioma3.82
1002OkaforEmeka2.95
1003MensahAkosua3.67
1004BandaTendai3.10
1005DialloAminata2.48
1006NkosiSipho3.91
1008OwusuKwame3.55
NULL in AND and OR Conditions

When a row contains NULL in a column involved in a comparison, the result of that comparison is unknown, which behaves like false in AND conditions and like false in OR conditions unless the other part of the OR is true.

The query WHERE gpa > 3.5 will silently exclude student 1007 because NULL > 3.5 is unknown. The student is not excluded because her GPA is low. She is excluded because her GPA is unknown.

If you need to include rows where a column might be either missing or above a threshold, you can combine conditions.

SQL — Including NULL rows explicitly
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa > 3.5
   OR gpa IS NULL;
Why NULL Matters in Practice

In real databases, missing values appear frequently. A medical database may have patients whose test results have not been entered yet. A library database may have resources whose subject classification has not been assigned. A school database may have students whose records are incomplete.

Treating missing values carelessly leads to wrong conclusions. If you count all records to find how many students have been assigned a GPA, but some records are NULL, your count will be too high. If you calculate an average GPA without noticing that some records are missing, your average may be slightly off. Data quality and null handling are not just technical details. They affect the reliability of every report, summary, and decision based on the data.

11.8

Sorting with ORDER BY

By default, a SQL query returns rows in no guaranteed order. The database may return them in the order they were stored, the order they were retrieved from an index, or some other internal order that can change without warning. If you need rows in a specific order, you must ask for it explicitly using ORDER BY.

The ORDER BY clause appears at the end of a query, after WHERE.

SQL — ORDER BY structure
SELECT column1, column2
FROM table_name
WHERE condition
ORDER BY sort_column;
Ascending Order

By default, ORDER BY sorts in ascending order: smallest to largest for numbers, earliest to latest for dates, and alphabetical order for text. You can write ASC explicitly to make your intention clear.

SQL — Alphabetical sort
SELECT student_id, last_name, first_name
FROM Students
ORDER BY last_name ASC;
Result
student_idlast_namefirst_name

1003MensahAkosua
1006NkosiSipho
1002OkaforEmeka
1008OwusuKwame
Descending Order

To reverse the sort order, add DESC after the column name.

SQL — Highest GPA first
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa IS NOT NULL
ORDER BY gpa DESC;
Result
student_idlast_namefirst_namegpa
1006NkosiSipho3.91
1001AdeyemiChioma3.82

Notice that the WHERE gpa IS NOT NULL clause is used here so that the student with a missing GPA does not appear at the bottom of the list in an ambiguous position.

Sorting by Multiple Columns

You can sort by more than one column. The database sorts by the first column first. When two rows have the same value in the first sort column, it uses the second column to break the tie.

SQL — Grade level ascending, then last name alphabetically
SELECT student_id, last_name, first_name, grade_level
FROM Students
ORDER BY grade_level ASC, last_name ASC;
Result
student_idlast_namefirst_namegrade_level

1003MensahAkosua12
1006NkosiSipho12

The three grade-10 students are listed first, sorted alphabetically by last name within that group. Grade-11 students follow, then grade-12 students, each group sorted alphabetically.

Sorting by Column Position

Some database systems allow you to sort by the position number of a column in the SELECT list instead of by name. This textbook discourages this practice because the meaning is not obvious without counting columns, and the sort changes silently if you rearrange the SELECT list. Sorting by column name is clearer and safer.

11.9

Limiting Results

Some queries return a very large number of rows. In a real system with thousands or millions of records, you often want to see just the first few results, especially when exploring data or building a report that shows only the top entries. The LIMIT clause restricts how many rows the database returns. It appears at the very end of a query.

SQL — The five students with the highest GPAs
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa IS NOT NULL
ORDER BY gpa DESC
LIMIT 5;
Result
student_idlast_namefirst_namegpa
1006NkosiSipho3.91
1001AdeyemiChioma3.82
1003MensahAkosua3.67
1008OwusuKwame3.55
1004BandaTendai3.10

The database sorted all eligible rows by GPA descending, then returned only the first five.

LIMIT Without ORDER BY

You can use LIMIT without ORDER BY, but the results will be unpredictable. The database will return some rows, but there is no guarantee about which rows or in what order. If you want the highest value, the most recent record, or the earliest entry, you must combine LIMIT with ORDER BY to define what "first" means.

SQL — Comparing unpredictable and predictable LIMIT
-- Unpredictable: could return any 3 rows
SELECT * FROM Students LIMIT 3;

-- Predictable: returns the 3 students with the earliest enrollment dates
SELECT * FROM Students ORDER BY enrollment_date ASC LIMIT 3;
OFFSET

SQLite and many other database systems support OFFSET in combination with LIMIT. OFFSET tells the database to skip a certain number of rows before starting to return results. This is often used in applications that display results across multiple pages.

SQL — Skip first 3 rows, then return the next 3
SELECT student_id, last_name, first_name, gpa
FROM Students
WHERE gpa IS NOT NULL
ORDER BY gpa DESC
LIMIT 3 OFFSET 3;

This returns results in positions 4, 5, and 6 when sorted by GPA descending. OFFSET is useful but introductory here. You will encounter it more in the context of web applications and paginated reports.

11.10

Building Clear Filter Conditions

Writing a working query is only part of the goal. A query should also be readable, correct, and easy to review for errors. This section collects practical habits for building filter conditions that are clear and reliable.

Write One Condition Per Line

When a WHERE clause has multiple conditions, putting each condition on its own line makes the logic visible at a glance.

SQL — Readability comparison
-- Hard to read
SELECT * FROM Students WHERE grade_level = 11 AND gpa > 3.0 AND advisor_id = 202;

-- Easier to read
SELECT student_id, last_name, first_name, grade_level, gpa
FROM Students
WHERE grade_level = 11
  AND gpa > 3.0
  AND advisor_id = 202;
Use Parentheses to Make Logic Explicit

Even when you are confident about evaluation order, adding parentheses makes your conditions self-explanatory.

SQL — Intent made clear with parentheses
WHERE (grade_level = 11 OR grade_level = 12)
  AND gpa > 3.5

Anyone reading this immediately understands: grade must be 11 or 12, and GPA must be above 3.5. Without the parentheses, a reader must remember that AND evaluates before OR.

Handle NULL Deliberately

If a column can contain NULL values, decide explicitly whether you want to include or exclude those rows and write the condition to match your intent.

SQL — Deliberate NULL handling
-- Explicitly excludes students with missing GPAs
WHERE gpa > 3.0 AND gpa IS NOT NULL

-- Explicitly includes students with missing GPAs
WHERE gpa > 3.0 OR gpa IS NULL
Predict Before Running

Before you run a query on a real database, look at the sample data and predict what the result should be. How many rows should appear? Which specific rows? Then run the query and compare. If the result matches your prediction, the query is probably correct. If it does not match, the mismatch tells you where to look for the problem.

Use Comments to Name Your Conditions

For complex queries, SQL comments can explain what each part of the condition is checking.

SQL — Commented conditions
SELECT student_id, last_name, first_name, grade_level, gpa
FROM Students
WHERE grade_level = 11       -- Only juniors
  AND gpa > 3.0              -- Meeting the honors threshold
  AND advisor_id = 202       -- Assigned to this advisor's cohort
  AND gpa IS NOT NULL;      -- Excluding records with missing GPAs

Comments do not affect the query's behavior. They make the query understandable to the next person who reads it, which is often you, three months from now.

Chapter Summary
  • The WHERE clause filters rows by specifying a condition. Only rows where the condition is true appear in the result set. Rows where the condition is false, or where a value is NULL, are excluded.
  • Comparison operators (=, <>, <, >, <=, >=) test relationships between column values and comparison values. They work with numbers, text, and dates.
  • Text filtering uses = for exact matches and LIKE with the % and _ wildcard characters for pattern matching.
  • Logical operators (AND, OR, NOT) combine conditions. AND requires all conditions to be true. OR requires at least one condition to be true. AND evaluates before OR in SQL, so parentheses should be used whenever both operators appear in the same condition.
  • IN provides a clean way to test whether a value matches any member of a list. BETWEEN tests whether a value falls within a range, inclusive of both endpoints.
  • NULL means unknown or missing. It cannot be tested with = or <>. The correct operators are IS NULL and IS NOT NULL. Any comparison with NULL using an ordinary operator produces an unknown result, which causes those rows to be silently excluded from most queries.
  • ORDER BY sorts the result set by one or more columns. ASC specifies ascending order and is the default. DESC specifies descending order. Multiple sort columns can be combined, each with its own direction.
  • LIMIT restricts the number of rows returned. It should almost always be paired with ORDER BY to define which rows are "first."
Key Terms
WHERE clause

The SQL clause that specifies a condition rows must satisfy in order to appear in a result set. Rows that do not satisfy the condition are excluded from the result.

Comparison operator

A symbol that compares two values and produces a true or false result. The comparison operators in SQL are =, <>, !=, <, >, <=, and >=.

LIKE

An operator that tests text column values against a pattern using the % and _ wildcard characters.

Wildcard character

A special character used in pattern matching. % matches any sequence of zero or more characters. _ matches exactly one character.

AND

A logical operator that requires all connected conditions to be true. A row is included only if every condition connected by AND is satisfied.

OR

A logical operator that requires at least one connected condition to be true. A row is included if it satisfies any one of the conditions connected by OR.

NOT

A logical operator that reverses the truth value of a condition. A row where the condition is false will be included; a row where it is true will be excluded.

IN

An operator that tests whether a column value matches any value in a specified list. Cleaner than writing multiple OR conditions for the same column.

BETWEEN

An operator that tests whether a column value falls within a range, inclusive of both the lower and upper endpoints. Equivalent to >= low AND <= high.

NULL

A special marker in SQL indicating that a value is unknown, missing, or not applicable. NULL is not the same as zero, an empty string, or any other specific value.

IS NULL

The correct SQL test for whether a column value is NULL. The expression = NULL does not work; IS NULL must be used instead.

IS NOT NULL

The correct SQL test for whether a column value has a known, non-null value.

ORDER BY

The SQL clause that specifies how the result set should be sorted. Appears at the end of a query, after WHERE.

ASC

Ascending sort order. Smallest to largest for numbers, earliest to latest for dates, alphabetical for text. This is the default sort order when ORDER BY is used.

DESC

Descending sort order. Largest to smallest for numbers, latest to earliest for dates, reverse alphabetical for text.

LIMIT

A SQL clause that restricts the number of rows returned by a query. Should almost always be combined with ORDER BY.

OFFSET

Used with LIMIT to skip a specified number of rows before returning results. Commonly used for paginated query results.

Evaluation order

The sequence in which SQL evaluates logical operators. AND is evaluated before OR unless parentheses override the default order.

Common Misconceptions
Misconception 1
You can test for NULL with the equals sign.

Incorrect: WHERE gpa = NULL. Correct: WHERE gpa IS NULL. The equals sign compares two known values. NULL is not a known value. It is the absence of a value. Comparing anything to NULL with = produces unknown, not true, which causes all rows to be excluded silently.

Misconception 2
LIMIT alone gives you the top rows.

LIMIT 5 without ORDER BY returns five rows, but there is no guarantee about which five rows or in what order. To get the five highest, lowest, most recent, or earliest records, you must combine LIMIT with ORDER BY.

Misconception 3
AND and OR have the same precedence.

AND is evaluated before OR. The condition A OR B AND C is evaluated as A OR (B AND C), not (A OR B) AND C. Use parentheses whenever you combine both operators in the same condition.

Misconception 4
BETWEEN excludes the endpoints.

BETWEEN 10 AND 20 includes both 10 and 20. It is equivalent to >= 10 AND <= 20. If you want to exclude an endpoint, use the comparison operators directly.

Misconception 5
Filtering on a NULL column with a comparison operator returns the NULL rows.

A condition like WHERE gpa > 3.0 does not return the student with a NULL GPA at the bottom. That student is excluded entirely. NULL does not mean zero or very low. It means the value is unknown, and unknown comparisons are treated as not-true in SQL.
