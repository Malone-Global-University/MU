Joining Tables

Relational design separates students from courses, books from borrowers, orders from products. That separation is good design, but it creates a new problem: how do you ask a question that spans more than one table? This chapter teaches you the join, the tool that reunites related data across tables so you can answer the questions a single table never could.

The previous three chapters taught you how to ask questions of a single table. You learned how to select columns, filter rows with WHERE, sort results with ORDER BY, and summarize data with COUNT, SUM, AVG, GROUP BY, and HAVING. Those are essential tools, and you will use them constantly.

But real databases rarely store everything in one table. A school database separates students from courses. A library database separates books from borrowers. A store database separates orders from products from customers. The whole point of relational design, as you studied in Chapters 7 and 8, is to keep different kinds of things in different but connected tables.

That design choice creates a new challenge: when you want to ask a question that spans multiple tables, you need a way to bring those tables together. That is what a join does.

A join combines rows from two or more tables based on a shared value. Usually that shared value is a primary key in one table matched to a foreign key in another table. When you join tables correctly, you can answer questions that no single table can answer on its own.

Joins are one of the most important ideas in the entire database subject. They are also one of the places where students most often get confused, produce unexpected results, or make errors that are hard to notice. This chapter is longer and more patient than most, because joins deserve that attention.

By the End of This Chapter

You should be able to explain why joins are necessary, read and predict the results of join queries, write INNER JOIN queries with correct conditions, work with table aliases, join more than two tables, handle many-to-many relationships through junction tables, understand why joins sometimes produce duplicate rows, combine joins with filters, and recognize the most common join mistakes.

13.1

Why Queries Need Multiple Tables

Imagine a school that stores student records and enrollment records in a relational database. The designers chose to keep these in separate tables, for reasons you studied in Chapters 7 and 8: students belong in a students table, enrollments belong in an enrollments table, and courses belong in a courses table.

Now consider this question: Which students are enrolled in Introduction to SQL?

That question touches two tables. The answer requires student names, which live in the Students table. It requires course enrollment records, which live in the Enrollments table. And it requires the course name Introduction to SQL, which lives in the Courses table.

Central Idea

Joins solve this problem by connecting related records across tables using their shared key values. The enrollment record knows student_id = 1 and course_id = 101. The join finds the student row where student_id = 1 and the course row where course_id = 101, then combines that information into a single result row. This is the fundamental purpose of a join: to reunite information that was intentionally separated for storage but must be combined for a useful answer.

13.2

Keys and Join Relationships

Before writing your first join, you should be clear about the relationship between primary keys and foreign keys. This relationship is what makes a join possible.

A primary key is the column, or combination of columns, that uniquely identifies each row in a table. In the Students table, student_id is the primary key. No two students share the same student_id. Every student has exactly one.

A foreign key is a column in one table that references the primary key of another table. In the Enrollments table, student_id is a foreign key. It refers back to the student_id column in the Students table. Similarly, course_id in Enrollments is a foreign key referencing the course_id in Courses.

When you write a join, you are almost always connecting a foreign key in one table to the primary key of another table. The join condition says: find the row in the other table where the primary key matches this foreign key value.

Enrollments.student_id  →  Students.student_id
Enrollments.course_id   →  Courses.course_id

If you think of it as following a pointer, that is roughly correct. Each enrollment row points to exactly one student and exactly one course. The join follows those pointers to bring back readable names and details.

This is why referential integrity matters so much. If an enrollment row contains a student_id value that does not exist in the Students table, the join will find no matching row for that student. The data becomes broken at the connection point. Well-designed databases with proper foreign key constraints prevent this problem by refusing to insert an enrollment with a student_id that does not exist.

A Habit Worth Building

You should always know which column is the primary key and which column is the foreign key before you write a join. Writing a join without understanding the keys is one of the most common sources of confusion and error.

13.3

INNER JOIN

The most common type of join is the INNER JOIN. It returns only the rows where the join condition is satisfied in both tables. If a row in the left table has no matching row in the right table, it does not appear in the result. If a row in the right table has no matching row in the left table, it does not appear either.

Here is the syntax:

SQL — INNER JOIN Syntax
SELECT columns
FROM table_one
INNER JOIN table_two
  ON table_one.key_column = table_two.key_column;

The ON clause is the join condition. It tells the database which column values must match for two rows to be combined.

Here is a concrete example. This query asks: Which students are enrolled in courses, and what are the course IDs they are enrolled in?

SQL — Joining Students and Enrollments
SELECT Students.first_name,
       Students.last_name,
       Enrollments.course_id,
       Enrollments.grade
FROM Students
INNER JOIN Enrollments
  ON Students.student_id = Enrollments.student_id;

Let us trace through what this query does. For each row in Students, the database looks for rows in Enrollments where Enrollments.student_id matches Students.student_id. When a match is found, the two rows are combined into one result row.

The result looks like this:

First, Amara appears twice because she is enrolled in two courses. The join creates one result row for each matching pair. If a student has three enrollments, that student's name appears three times.

Second, course_id still shows numbers. To get course names, you would need to join the Courses table as well. You will see how to do that shortly.

Third, the result is limited to students who have enrollments. All four students in this dataset happen to have enrollments, so all four appear. But if a student existed in the Students table with no enrollment rows, that student would not appear in the result, because there would be no matching row in Enrollments.

13.4

Join Conditions

The join condition in the ON clause is the most important part of any join query. If you get the condition wrong, the query either returns no rows, returns the wrong rows, or returns a catastrophically large and meaningless result set.

A correct join condition matches a primary key in one table to the corresponding foreign key in another table. Here are the correct conditions for the tables in this chapter:

SQL — Correct Join Conditions
-- Join Students to Enrollments
ON Students.student_id = Enrollments.student_id

-- Join Courses to Enrollments
ON Courses.course_id = Enrollments.course_id

The condition uses the equals operator (=) because you are looking for an exact match. Each enrollment refers to exactly one student, and you want to find that student.

A Word of Caution About Column Names

In the query above, both tables have a column called student_id. SQL requires you to qualify the column name with the table name, or an alias, which you will learn next, whenever the same column name appears in more than one table involved in the query. Writing just student_id when two tables both have that column will cause an error or, worse, ambiguous behavior. Always write TableName.column_name in join conditions.

What Happens Without a Join Condition

If you write a FROM clause with two tables but no ON clause, or with an accidental Cartesian product, the database combines every row in the first table with every row in the second table. If Students has 4 rows and Enrollments has 7 rows, the result would have 4 × 7 = 28 rows, most of which are meaningless combinations. This is called a cross join, and it is almost never what you want. Always include a proper ON condition.

Matching the Right Columns

Occasionally two columns have the same name in different tables but do not actually belong together. For example, if both a Students table and a Products table had a column called status, joining on Students.status = Products.status would be meaningless, even if it produced rows. Always make sure your join condition follows the actual key relationship, not just matching column names.

13.5

Table Aliases

When writing join queries, column references become long. Writing Students.first_name, Enrollments.course_id, and Courses.course_name repeatedly is tedious and makes queries harder to read. Table aliases solve this by giving a table a short nickname for the duration of the query.

You assign an alias in the FROM or JOIN clause by writing the alias immediately after the table name, optionally with the keyword AS:

SQL — Assigning Aliases
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id

Or without AS, which is also valid:

SQL — Aliases Without AS
FROM Students s
INNER JOIN Enrollments e
  ON s.student_id = e.student_id

Once you assign an alias, you use that alias everywhere in the query instead of the full table name. Here is the earlier query rewritten with aliases:

SQL — Rewritten With Aliases
SELECT s.first_name,
       s.last_name,
       e.course_id,
       e.grade
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id;

Common alias conventions are single letters that match the table name, s for Students, e for Enrollments, c for Courses, or short abbreviations, stu, enr, crs. Use whatever makes the query clearest for someone reading it.

One Important Rule

Once you assign an alias in a query, you must use that alias consistently. You cannot mix the full table name and the alias in the same query. If you write FROM Students AS s, then you must use s.first_name, not Students.first_name.

Aliases become especially helpful when joining three or more tables, which you will see in the next section.

13.6

One-to-Many Joins

The most common join relationship is one-to-many. One student can have many enrollments. One course can have many enrollments. One author can write many books. One order can contain many items.

In a one-to-many join, the "one" side has the primary key and the "many" side has the foreign key. The join produces one result row for each record on the "many" side that has a match on the "one" side.

The query below brings together all three tables to answer a more complete question: What are the names of students, the names of the courses they are enrolled in, and their grades?

SQL — Joining Three Tables
SELECT s.first_name,
       s.last_name,
       c.course_name,
       e.grade
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id
INNER JOIN Courses AS c
  ON e.course_id = c.course_id;

Trace through this carefully. The query starts with Students. It joins Enrollments by matching s.student_id = e.student_id. For each matched pair, it then joins Courses by matching e.course_id = c.course_id. Each enrollment row connects one student row and one course row. The final result combines all three.

Now the result shows real names on both sides. Each row represents one enrollment: one student in one course with one grade. Amara appears twice because she has two enrollments. Jonas appears three times because he has three.

This is expected and correct behavior. The join does not compress multiple enrollments into one row. It faithfully reflects each record in the Enrollments table, enriched with names from Students and Courses.

Understanding the Chain

When you join three tables, you form a chain:

Students ──── Enrollments ──── Courses
         (via student_id)  (via course_id)

Enrollments is in the middle because it is the junction between students and courses. Each enrollment knows its student and its course by foreign key references. The join follows both references simultaneously to build each result row.

13.7

Many-to-Many Joins with Junction Tables

A many-to-many relationship exists when records on both sides of a relationship can be associated with many records on the other side. Students can enroll in many courses, and each course can have many students enrolled.

As you learned in Chapter 7, many-to-many relationships cannot be stored directly in two tables. They require a third table called a junction table, also called a bridge table, linking table, or associative table. The Enrollments table in this chapter is exactly that: it sits between Students and Courses and holds one row for every student-course pairing.

The structure of a junction table typically looks like this:

Junction Table
--------------
foreign_key to Table A   (primary key of Table A)
foreign_key to Table B   (primary key of Table B)
(plus any attributes of the relationship itself, such as grade, date, status)

To query a many-to-many relationship, you always join through the junction table. You never try to join Students directly to Courses; you always go through Enrollments.

SQL — The Correct Many-to-Many Pattern
-- This is the correct pattern for a many-to-many join
SELECT s.first_name, s.last_name, c.course_name
FROM Students AS s
INNER JOIN Enrollments AS e   -- join through the junction table
  ON s.student_id = e.student_id
INNER JOIN Courses AS c
  ON e.course_id = c.course_id;

13.8

Why Joins Can Create Duplicate Rows

A common source of confusion is when students expect one row per student but get multiple rows per student, or expect one row per resource but get several.

This is not a bug. This is the join doing exactly what it should.

When you join a "one" side to a "many" side, the result will have one row for each record on the "many" side that finds a match. A student with three enrollments will generate three result rows, because there are three matching enrollment records.

A Simple Way to Think About It

The join does not summarize. It combines. For each matching pair, it produces one row. If one student matches seven enrollment rows, the join produces seven rows for that student.

When Duplicate Rows Are a Sign of a Problem

Duplicate rows are expected and correct in one-to-many and many-to-many joins. But sometimes you join two tables and expect a one-to-one match, and you see unexpected duplication. This usually means one of the following.

The join condition is wrong, matching more rows than intended. A table you thought had unique values for a key column actually has duplicates. You are joining to a table that has multiple rows per entity for reasons you did not anticipate.

If you see more rows than expected, check whether your join condition is correct, check whether the key column on the "many" side truly has the values you expect, and check whether either table has unexpected duplicate key values.

Distinguishing Expected from Unexpected Duplication

Expected: a student appears 5 times because they have 5 enrollments. Unexpected: a student appears 10 times when you only expected 5. The second case usually means a double join, a missing condition, or a table with accidental duplicates.

13.9

Joining and Filtering Together

Join queries can include WHERE clauses, ORDER BY, aggregation, and any other SQL clause you have already learned. The join determines which rows are combined. The WHERE clause then filters those combined rows.

Example: Find All Students Enrolled in Introduction to SQL
SQL — Join With a Filter
SELECT s.first_name,
       s.last_name,
       c.course_name,
       e.grade
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id
INNER JOIN Courses AS c
  ON e.course_id = c.course_id
WHERE c.course_name = 'Introduction to SQL';
Result — Only students enrolled in Introduction to SQL.

The join brought all tables together. The WHERE clause then kept only the rows where the course name matched.

Example: Count How Many Students Are in Each Course
SQL — Join With GROUP BY and COUNT
SELECT c.course_name,
       COUNT(s.student_id) AS student_count
FROM Courses AS c
INNER JOIN Enrollments AS e
  ON c.course_id = e.course_id
INNER JOIN Students AS s
  ON e.student_id = s.student_id
GROUP BY c.course_name
ORDER BY student_count DESC;

SQL — Join With DISTINCT
SELECT DISTINCT s.first_name,
                s.last_name
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id
WHERE e.grade = 'A'
ORDER BY s.last_name;

Notice the use of DISTINCT. Jonas earned A grades in two courses. Without DISTINCT, he would appear twice in the result. DISTINCT removes duplicate result rows. Use it when you want a list of unique entities rather than a count of matching records.

13.10

Common Join Mistakes

Joins give students more trouble than almost any other SQL topic. The following mistakes appear very frequently. Recognizing them will save you significant time.

Mistake 1
Forgetting the ON Clause

Writing two tables in the FROM clause without specifying the join condition produces a cross join, which combines every row in the first table with every row in the second. With small tables this produces a confusing result. With large tables it can produce millions of rows and crash a query.

SQL — Missing ON Clause
-- This is wrong: missing ON clause
SELECT s.first_name, e.grade
FROM Students AS s
INNER JOIN Enrollments AS e;
-- Produces 4 × 7 = 28 rows, all meaningless combinations

Always include ON table_one.key = table_two.key.

Mistake 2
Joining on the Wrong Columns

If you join on two columns that happen to have the same name but do not actually represent the same key relationship, you will get wrong results. The query will run without an error, but the rows it returns will be incorrect combinations.

Always verify that the columns you are joining on actually represent the primary key to foreign key relationship. Do not rely on column names alone.

Mistake 3
Ambiguous Column Names

If a column name exists in more than one table involved in the query and you do not qualify it with a table name or alias, the database will either throw an error or behave unpredictably.

SQL — Ambiguous Reference
-- Ambiguous: student_id exists in both Students and Enrollments
SELECT student_id, first_name
FROM Students
INNER JOIN Enrollments ON Students.student_id = Enrollments.student_id;

Fix this by qualifying every ambiguous column reference:

SQL — Qualified Columns
SELECT s.student_id, s.first_name
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id;
Mistake 4
Confusing INNER JOIN Results With Full Lists

An INNER JOIN only returns rows where a match exists in both tables. If a student has no enrollments, that student does not appear in the join result. If a course has no enrollments, that course does not appear.

If you want to see all students whether or not they have enrollments, you would use a LEFT JOIN, also called LEFT OUTER JOIN. The LEFT JOIN is not covered in depth in this chapter, because this textbook focuses on foundational SQL, but you should know it exists. An INNER JOIN is not the right tool when you need rows from one table even when there is no match in the other.

Mistake 5
Expecting the Join to Deduplicate

As discussed in Section 13.8, a join does not deduplicate. A student enrolled in five courses will appear five times in an INNER JOIN result with Enrollments. This is correct behavior. If you need a unique list of students who have any enrollment at all, use SELECT DISTINCT or restructure as a subquery.

Mistake 6
Wrong Join Order When Filtering

Occasionally students write a WHERE clause that filters out rows before the join has a chance to match them, or they confuse join conditions with filter conditions. A practical rule: use ON for join conditions and WHERE for row filters after joining.

SQL — ON for Joins, WHERE for Filters
-- Correct pattern: ON for join, WHERE for filter
FROM Students AS s
INNER JOIN Enrollments AS e
  ON s.student_id = e.student_id  -- this is the join condition
WHERE e.grade = 'A';              -- this is a row filter

Putting filter conditions inside the ON clause often produces confusing results, especially in outer joins. Keep them separate.

Mistake 7
Skipping the Junction Table in Many-to-Many Joins

If you try to join Students directly to Courses without going through Enrollments, you will either get a cross join or a join on a column that has no meaningful relationship. Always join through the junction table.

SQL — Wrong vs. Correct Pattern
-- Wrong: no junction table
FROM Students AS s
INNER JOIN Courses AS c ON s.student_id = c.course_id; -- meaningless

-- Correct: through the junction table
FROM Students AS s
INNER JOIN Enrollments AS e ON s.student_id = e.student_id
INNER JOIN Courses AS c ON e.course_id = c.course_id;
Chapter Summary
  • A join combines rows from two or more tables based on a shared key value. Joins are necessary because relational databases intentionally separate different kinds of information into different tables. To answer questions that span those tables, you must rejoin the tables using their key relationships.
  • The INNER JOIN returns only rows where a match exists in both tables. The join condition, written in the ON clause, specifies which columns must match. Almost always, this is a primary key in one table matched to a corresponding foreign key in another.
  • Table aliases make join queries shorter and more readable. Assign them in the FROM and JOIN clauses and use them consistently throughout the query.
  • One-to-many joins produce one result row for each record on the "many" side that has a match. A student with three enrollments appears three times in a result that joins Students to Enrollments. This is correct behavior.
  • Many-to-many relationships require a junction table. You never join the two main tables directly. Instead, you join from one main table through the junction table to the other main table.
  • Joins can be combined with WHERE filters, ORDER BY, GROUP BY, COUNT, DISTINCT, and all other SQL clauses to build powerful, meaningful queries. Most real database queries require at least one join.
  • The most common mistakes in join queries are missing ON conditions, joining on wrong columns, using ambiguous column names, expecting INNER JOIN to include unmatched rows, expecting the join to deduplicate automatically, and skipping junction tables in many-to-many relationships.
Key Terms
Join

A SQL operation that combines rows from two or more tables based on a shared value, typically a primary key matched to a foreign key.

INNER JOIN

A join that returns only rows where the join condition is satisfied in both tables. Rows with no match in either table are excluded.

Join condition

The expression in the ON clause that specifies which column values must match for two rows to be combined. Almost always a primary key to foreign key comparison.

Cross join

The result of combining every row in one table with every row in another, produced when no join condition is specified. Almost always a mistake.

Table alias

A short nickname assigned to a table in a query, used to qualify column references and shorten query syntax.

One-to-many relationship

A relationship where one record in a table corresponds to many records in another table. For example, one student can have many enrollments.

Many-to-many relationship

A relationship where records on both sides can correspond to many records on the other side. Requires a junction table.

Junction table

A table that represents a many-to-many relationship by storing one row for each pairing of records from two other tables.

DISTINCT

A keyword that removes duplicate rows from a result set. Used after SELECT.

Referential integrity

The rule that a foreign key value must match an existing primary key value in the referenced table. Prevents broken join relationships.

Cartesian product

See cross join. The mathematical term for combining every row of one set with every row of another.

Outer join

A join variant, LEFT JOIN, RIGHT JOIN, FULL JOIN, that includes rows even when no match exists in the other table. Not covered in depth in this chapter but important to know exists.

Common Misconceptions
Misconception 1
A join always returns one row per entity.

A join returns one row for every matching pair. If a student has five enrollments, the student appears in five rows in a join with Enrollments. This is correct. If you want one row per student, use GROUP BY or DISTINCT.

Misconception 2
Joins are only needed for big or complex databases.

Any relational database with more than one table needs joins to answer cross-table questions. Even a simple two-table database, students and their enrollment records, needs a join to show readable output.

Misconception 3
The order of tables in a join changes the result.

For INNER JOIN, the order of tables does not change which rows are returned. The join is symmetric: swapping FROM A INNER JOIN B for FROM B INNER JOIN A produces the same rows, though possibly in a different order. Use ORDER BY when row order matters.

Misconception 4
If the query runs without error, the join must be correct.

SQL will execute a join even if the join condition is wrong or missing. An accidental cross join runs without error and returns a huge, meaningless result. Always check that your result row count and content make sense.

Misconception 5
You join on column names that match.

Column name matching is unreliable. You should join on columns that represent the actual key relationship, verified by understanding the database design. Two columns named status in two different tables do not belong together just because they share a name.

Misconception 6
INNER JOIN and LEFT JOIN return the same results.

An INNER JOIN excludes rows from either table that have no match. A LEFT JOIN includes all rows from the left table whether or not they have a match in the right table. They produce the same result only when every row in the left table has at least one match.
