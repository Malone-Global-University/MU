Subqueries, Views, and Reusable Queries

You have learned to filter, sort, summarize, group, and join data. Now you learn how to organize that work, name it, and reuse it. This chapter introduces subqueries, views, and common table expressions, the tools that let SQL be organized just like code.

Chapter 14 introduces one of the most important ideas in SQL beyond simple SELECT statements: the idea that queries can build on other queries. Students have already learned how to filter, sort, summarize, group, and join data. Now they learn how to organize that work, reuse it, name it, and make it readable and maintainable.

This chapter introduces subqueries, views, common table expressions, and the broader habit of query readability. These tools do not just make SQL more powerful. They make SQL more organized, more reliable, and much easier for real people to understand when they return to it weeks or months later.

The emphasis throughout is on reading and understanding before writing. Every major concept includes a worked example with a full explanation. Students who finish this chapter should be able to read a complex nested query and explain what each part does, write a named view and explain why it is useful, and choose between different ways of organizing a query based on clarity and simplicity.

Note for Students

The patience you brought to Chapter 13's joins matters here too. Subqueries and CTEs are not harder than joins, they are simply organized differently. Read every example from the inside out, and the logic will follow naturally.

14.1

Queries Can Build on Queries

When you first learn SQL, queries feel like single questions. You ask a table something, and it answers. SELECT name FROM students WHERE enrolled = 1 is one question: give me the names of enrolled students.

But real information systems need to answer harder questions. Sometimes the answer to one question becomes the starting point for another question. Sometimes you need to filter based on a calculated summary. Sometimes you need to compare a row to an average computed from the whole table. Sometimes you want to save a useful query result and treat it like a table you can query again later.

SQL handles all of this by letting queries refer to other queries. A query can use another query's result as its input. A query can be saved as a named object that behaves like a table. A query can be broken into named pieces that are defined at the top and referenced throughout the rest of the query.

This chapter introduces all three ideas:

Subqueries

Queries written inside other queries. They run first, and their result is used by the outer query.

Views

Saved queries stored in the database with a name. Once created, you can query a view just like a table.

CTEs

Common table expressions let you name temporary query results at the top of a statement, then refer to them by name below.

Readability

The broader habit of organizing SQL so that other people, including a future version of you, can understand and trust it.

These tools solve a real problem. As queries grow, they become harder to read, harder to test, and harder to maintain. Breaking a complex query into named, organized pieces makes the whole thing easier to understand, easier to fix, and easier to reuse.

Before introducing these tools, it helps to look at the kinds of questions that motivate them.

A Problem That Needs More Than One Step

Suppose a university library wants to answer this question:

The Question

Which students have borrowed more books than the average student?

This question cannot be answered with a single simple SELECT statement. To answer it, you need to know the average first, and then compare each student's borrowing count to that average. The average is not a single stored value. It has to be calculated from the data.

One approach is to run two separate queries. The first query calculates the average:

SQL — Step One: Calculate the Average
SELECT AVG(loan_count)
FROM student_loan_summary;

Suppose this returns 4.2. Then you run a second query:

SQL — Step Two: Filter Using That Number
SELECT student_name, loan_count
FROM student_loan_summary
WHERE loan_count > 4.2;

This works, but it is awkward. You had to copy a number from one query into another. If the data changes tomorrow, the average changes, and you have to run the first query again to get the new number and paste it into the second query.

A subquery solves this by letting the second query contain the first query inside it. The database calculates the average automatically, and you never have to copy a number by hand.

14.2

Subqueries at an Introductory Level

A subquery is a complete SELECT statement written inside parentheses inside another SELECT statement. The inner query runs first. Its result is used by the outer query.

Here is the two-step loan problem written as a single query with a subquery:

SQL — A Subquery in WHERE
SELECT student_name, loan_count
FROM student_loan_summary
WHERE loan_count > (
    SELECT AVG(loan_count)
    FROM student_loan_summary
);

The inner query is:

SQL — The Inner Query
SELECT AVG(loan_count)
FROM student_loan_summary

This runs first. It calculates the average number of loans across all students. Suppose that average is 4.2. The outer query then uses that result, behaving as though it were written:

SQL — What the Outer Query Effectively Becomes
SELECT student_name, loan_count
FROM student_loan_summary
WHERE loan_count > 4.2

The database handles this automatically. You write the question in one statement. The database figures out what order to execute the parts.

How to Read a Subquery

When you encounter a query with a subquery, read from the inside out.

  • Find the innermost query

    Locate the innermost SELECT statement, enclosed in parentheses.

  • Identify what it produces

    Figure out what that inner query produces. Does it return a single value, a list of values, or a table?

  • Read the outer query

    Understand how the outer query uses that inner result.

This inside-out reading habit is the most important skill for understanding subqueries.

What Subqueries Can Return

Subqueries do not always return the same kind of result. The correct way to use a subquery depends on what it returns.

A single value is the simplest case. The subquery returns exactly one row and one column, and the outer query compares something to that single value. The average-loan example above is a single-value subquery.

A list of values means the subquery returns one column with multiple rows. The outer query typically uses IN to check whether a value appears in that list.

A table means the subquery returns multiple rows and multiple columns. The outer query treats that result as a temporary table and can query it with FROM.

The sections below cover each of these cases.

14.3

Subqueries in WHERE

The most common place to find a subquery is inside a WHERE clause. A subquery in WHERE lets you filter rows based on calculated values or lists produced by another query.

Subquery Returning a Single Value

The loan-average example above is a subquery returning a single value. Here is another example.

Suppose a bookstore database has a table called sales with columns product_id, product_name, and units_sold. A manager wants to find every product that sold more units than the average number of units sold, without running two separate queries:

SQL — Filter by an Average
SELECT product_name, units_sold
FROM sales
WHERE units_sold > (
    SELECT AVG(units_sold)
    FROM sales
);

Read this from the inside out. The inner query calculates AVG(units_sold) from the sales table. This returns one number. The outer query then selects every product where units_sold is greater than that one number.

The outer WHERE clause compares a column value to a single calculated result. This is a very clean use of a subquery.

Subquery Returning a List of Values

Sometimes you want to filter based on whether a value appears in a set that was calculated by another query, not typed by hand.

Suppose a university database has two tables:

A registrar wants to find all students who are enrolled in at least one course. One way to do this is to find the list of student IDs in the enrollments table, and then find every student whose ID appears in that list.

SQL — Subquery with IN
SELECT student_name
FROM students
WHERE student_id IN (
    SELECT student_id
    FROM enrollments
);

The inner query returns a list of student IDs from the enrollments table. In this dataset, that list is: 1, 3, 5. The outer query then selects every student whose student_id appears in that list.

NOT IN with a Subquery

You can reverse the logic with NOT IN. Suppose the registrar wants to find students who are not enrolled in any courses:

SQL — Subquery with NOT IN
SELECT student_name
FROM students
WHERE student_id NOT IN (
    SELECT student_id
    FROM enrollments
);

This returns:

student_name

Warning: NOT IN and Missing Values

NOT IN behaves unexpectedly when the subquery returns any NULL values. If the inner list contains even one NULL, NOT IN will return no rows at all. This is a known source of confusion in SQL.

If the column used in the subquery might contain NULLs, add a WHERE clause to the subquery to exclude them:

SQL — NULL-Safe NOT IN
SELECT student_name
FROM students
WHERE student_id NOT IN (
    SELECT student_id
    FROM enrollments
    WHERE student_id IS NOT NULL
);

This is a good habit any time you write NOT IN with a subquery.

14.4

Subqueries in FROM at an Introductory Level

A subquery can also appear in the FROM clause. When it does, the subquery acts as a temporary table. The outer query reads from that temporary table just as if it were a real table stored in the database.

This is sometimes called a derived table or an inline view.

Why Subqueries in FROM Are Useful

Sometimes you need to summarize data first, and then filter or analyze the summary. You cannot filter on an aggregate function directly in a WHERE clause. For example, this is not valid SQL:

SQL — This Does NOT Work
-- This does NOT work
SELECT student_id, COUNT(*) AS course_count
FROM enrollments
WHERE COUNT(*) > 2;

You cannot use COUNT(*) in WHERE because WHERE is evaluated before the aggregation happens. The solution is to put the aggregation inside a subquery in the FROM clause, and then filter the result in an outer WHERE clause.

Worked Example

Suppose you want to find every student who is enrolled in more than one course.

Step one is to count enrollments per student:

SQL — Step One: Count Enrollments
SELECT student_id, COUNT(*) AS course_count
FROM enrollments
GROUP BY student_id;

This produces:

Result — Enrollment counts per student.
student_idcourse_count
11
32
51

Step two is to filter this result to find only students with more than one course. You can do this by wrapping the first query in a subquery and querying it with FROM:

SQL — Step Two: Filter the Summary
SELECT student_id, course_count
FROM (
    SELECT student_id, COUNT(*) AS course_count
    FROM enrollments
    GROUP BY student_id
) AS enrollment_counts
WHERE course_count > 1;

Read this from the inside out:

  • Inner query groups and counts

    The inner query groups enrollments by student_id and counts rows. It produces a table with two columns: student_id and course_count.

  • Result is aliased

    That result is given the alias enrollment_counts.

  • Outer query filters

    The outer query reads from enrollment_counts and filters for rows where course_count > 1.

Result:

Result — Students enrolled in more than one course.
student_idcourse_count
32

Student 3 (Priya Das) is enrolled in two courses and appears in the result. Students 1 and 5 are each enrolled in only one course and do not appear.

Naming Subqueries in FROM

The alias after a subquery in FROM is required in most database systems. You must give the derived table a name so the outer query can refer to it. In the example above, the alias is enrollment_counts. Without this alias, the query will produce an error.

14.5

What Views Are

A subquery written inside a query is temporary. It exists only while that one query runs. When the query finishes, the subquery is gone.

A view is a way to save a query permanently inside the database and give it a name. Once a view is created, it behaves like a table. You can SELECT from a view, filter it, join it with other tables, and use it in other queries, all without rewriting the underlying query each time.

A view does not store data. It stores the query definition. Every time you query a view, the database runs the underlying query and returns the result. If the underlying tables change, the view always reflects the current data.

Why Views Are Useful

Views serve several important purposes in real database systems.

Reuse

If you write a useful query, you should not have to rewrite it every time you need the same information. A view lets you save the query once and reuse it as many times as you need.

Simplification

Complex queries with multiple joins, filters, and aggregations become difficult to read and maintain. A view can hide complexity behind a simple name.

Security

A view can expose only certain columns or certain rows from an underlying table, letting an administrator protect sensitive columns such as salary or identification numbers.

Consistency

If many parts of an application need the same calculation, defining it once in a view ensures every part uses the same logic.

What a View Is Not

A view is not a copy of the data. It does not have its own storage. It is a saved query, and it always reflects the current state of the underlying tables. This is important because it means:

  • A view is always up to date with the underlying data.
  • Querying a view runs the underlying query every time.
  • If the underlying tables are slow to query, the view may also be slow.

Some database systems offer a related concept called a materialized view, which does store a snapshot of the result for performance reasons. This textbook stays at the introductory level and focuses on regular, non-materialized views.

14.6

Creating and Using Views

Creating a View

The syntax for creating a view is:

SQL — CREATE VIEW Syntax
CREATE VIEW view_name AS
SELECT ...;

The SELECT statement after AS is the query the view represents. Every time someone queries the view, the database runs that SELECT statement.

Worked Example: A Student Enrollment View

Suppose you frequently need to see student names alongside their enrolled course names, joining across three tables: students, enrollments, and courses. This join is useful but verbose to write every time. You can save it as a view.

First, here are the tables:

Create the view:

SQL — Creating the View
CREATE VIEW student_course_enrollment AS
SELECT
    s.student_name,
    c.course_name,
    e.grade
FROM enrollments AS e
JOIN students AS s ON e.student_id = s.student_id
JOIN courses AS c ON e.course_id = c.course_id;

Once this view is created, it lives in the database with the name student_course_enrollment. Now anyone can query it:

SQL — Querying the View
SELECT student_name, course_name, grade
FROM student_course_enrollment;

Filtering a View

You can add a WHERE clause when querying a view, just like a real table:

SQL — Filtering a View
SELECT student_name, course_name, grade
FROM student_course_enrollment

The filter is applied on top of the view's result.

Dropping a View

If you no longer need a view, you remove it with:

SQL — DROP VIEW
DROP VIEW student_course_enrollment;

Dropping a view does not delete any underlying data. It only removes the saved query definition.

Views and Security

Suppose an HR database stores employee records including sensitive information such as salaries and social security numbers. Most staff members need to look up employee names and departments but should not see salary information.

You can create a view that exposes only the safe columns:

SQL — A View That Protects Sensitive Columns
CREATE VIEW employee_directory AS
SELECT employee_id, first_name, last_name, department_name, office_location
FROM employees
JOIN departments ON employees.department_id = departments.department_id;
Structure as Protection

Grant access to the view, not to the underlying table. Staff members can query employee_directory and see names, departments, and office locations, but they cannot see salary data because it is not in the view. This is an example of using database structure, not application code, to protect sensitive information.

14.7

Reusable Query Patterns

As you work with SQL over time, you will notice that certain query patterns appear again and again. Recognizing these patterns helps you read unfamiliar queries and write new ones more confidently.

Here are several of the most common reusable patterns.

Pattern 1 — Filter Based on a Summary

Question: Which items have a value above or below a calculated threshold, like an average or maximum?

Approach: Use a subquery in WHERE to calculate the threshold.

SELECT product_name, price
FROM products
WHERE price > (
    SELECT AVG(price)
    FROM products
);
Pattern 2 — Find Records That Match a Sublist

Question: Which records in one table correspond to records in another table that meet some condition?

Approach: Use a subquery with IN to get the matching list of IDs.

SELECT student_name
FROM students
WHERE student_id IN (
    SELECT student_id
    FROM enrollments
    WHERE course_id = 'CS101'
);
Pattern 3 — Find Records That Have No Match

Question: Which records in one table have no corresponding record in another table?

Approach: Use NOT IN with NULL protection, or structure the problem as a filter.

SELECT student_name
FROM students
WHERE student_id NOT IN (
    SELECT student_id
    FROM enrollments
    WHERE student_id IS NOT NULL
);
Pattern 4 — Summarize, Then Filter

Question: Which groups have a summary value, like a count or total, that meets a condition?

Approach: Use a subquery in FROM to produce the summary, then filter in the outer WHERE.

SELECT department_id, total_students
FROM (
    SELECT department_id, COUNT(*) AS total_students
    FROM students
    GROUP BY department_id
) AS department_counts
WHERE total_students >= 10;
Pattern 5 — Simplify a Complex Join with a View

Question: How can a repeated multi-table join be made easy to use?

Approach: Create a view that encapsulates the join, then query the view.

CREATE VIEW enrollment_details AS
SELECT s.student_name, c.course_name, e.grade, e.enrollment_id
FROM enrollments AS e
JOIN students AS s ON e.student_id = s.student_id
JOIN courses AS c ON e.course_id = c.course_id;

Then query the view as needed.

14.8

Common Table Expressions at an Introductory Level

A common table expression, usually abbreviated CTE, is a way to name a temporary query result at the top of a SQL statement and then use that name in the rest of the query.

CTEs do the same thing as subqueries in FROM, but they are written differently. Many people find CTEs easier to read, especially when a query has more than one named subpart.

The WITH Keyword

CTEs use the keyword WITH followed by a name, the keyword AS, and the query in parentheses. The main query follows afterward.

The general structure looks like this:

SQL — General CTE Structure
WITH cte_name AS (
    SELECT ...
    FROM ...
    WHERE ...
)
SELECT ...
FROM cte_name
WHERE ...;

The CTE named cte_name runs first. The main SELECT below it can use cte_name as if it were a table.

Worked Example: Students Enrolled in More Than One Course

Earlier, this problem was solved with a subquery in FROM. Here it is solved with a CTE instead:

SQL — Same Problem, Written as a CTE
WITH enrollment_counts AS (
    SELECT student_id, COUNT(*) AS course_count
    FROM enrollments
    GROUP BY student_id
)
SELECT student_id, course_count
FROM enrollment_counts
WHERE course_count > 1;

Compare this to the earlier subquery-in-FROM version:

SQL — The Subquery-in-FROM Version, for Comparison
SELECT student_id, course_count
FROM (
    SELECT student_id, COUNT(*) AS course_count
    FROM enrollments
    GROUP BY student_id
) AS enrollment_counts
WHERE course_count > 1;

Both queries produce the same result. The CTE version separates the definition from the usage, which many readers find cleaner, especially as queries grow longer.

Multiple CTEs

You can define more than one CTE in a single WITH clause, separating them with commas. Each CTE can reference the ones defined before it.

SQL — Two CTEs in One Statement
WITH
enrolled_students AS (
    SELECT DISTINCT student_id
    FROM enrollments
),
unenrolled_students AS (
    SELECT student_id
    FROM students
    WHERE student_id NOT IN (
        SELECT student_id FROM enrolled_students
    )
)
SELECT student_name
FROM students
WHERE student_id IN (
    SELECT student_id FROM unenrolled_students
);

This query defines two CTEs. The first, enrolled_students, finds all student IDs with at least one enrollment. The second, unenrolled_students, finds all student IDs not in that first list. The main query then retrieves the names of unenrolled students.

Breaking a complex query into named pieces like this makes it much easier to read and debug one piece at a time.

When to Use CTEs

CTEs are most helpful when:

  • A query has more than one subpart that would otherwise nest deeply inside each other.
  • You want to give a meaningful name to an intermediate result.
  • The same intermediate result is needed more than once in the main query.
  • You are debugging a complex query and want to test each piece separately.

CTEs are not always necessary. For simple single-use subqueries, a subquery in WHERE or FROM is fine. The goal is always readability.

14.9

Query Readability

SQL can be written in many different ways. Two queries can produce identical results but be completely different in how easy they are to read, understand, and maintain. Query readability is not just a style preference. It is a practical concern that affects how reliably databases are used and how easily errors are found and fixed.

Why Readability Matters

Consider who reads SQL queries in a real organization. It is not just the person who wrote them. It is also the same person, months or years later, who has forgotten the details, a colleague who needs to modify the query for a new report, a student or junior staff member who is learning from existing queries, a database administrator reviewing queries for performance or security issues, and an auditor checking that data was handled correctly.

A query that is technically correct but impossible to follow creates real problems for all of these people. A query that is organized, clearly formatted, and thoughtfully named is much easier to verify, trust, and maintain.

Formatting Practices

Good SQL formatting follows a few consistent conventions.

Each major clause on its own line. SELECT, FROM, WHERE, GROUP BY, HAVING, and ORDER BY each begin on their own line.

SELECT student_name, course_name, grade
FROM student_course_enrollment
WHERE grade = 'A'
ORDER BY student_name;

Indent continuation lines. When a clause spans multiple lines, indent the continuation:

SELECT
    student_name,
    course_name,
    grade
FROM student_course_enrollment
WHERE grade = 'A'
ORDER BY student_name;

Align JOIN clauses clearly:

SELECT s.student_name, c.course_name, e.grade
FROM enrollments AS e
    JOIN students AS s ON e.student_id = s.student_id
    JOIN courses AS c ON e.course_id = c.course_id;

Use blank lines to separate logical sections in long queries, and use lowercase or uppercase keywords consistently, whichever your style guide or organization prefers.

Naming Practices

Choose names for aliases, CTEs, and views that describe what the result contains.

Good: enrollment_counts, enrolled_students, department_summary

Poor: temp, t1, result, query1

Descriptive names help readers understand what each part of a query is doing without needing to trace through every detail.

Comments in SQL

SQL supports two comment styles. A single-line comment begins with two dashes:

-- This query finds students enrolled in more than one course
SELECT student_name
FROM students
WHERE student_id IN (
    SELECT student_id
    FROM enrollments
    GROUP BY student_id
    HAVING COUNT(*) > 1
);

A multi-line comment is enclosed between /* and */:

/*
  Report: Unenrolled Students
  Purpose: Identify students with no current course enrollments
  Updated: Term registration review
*/
SELECT student_name
FROM students
WHERE student_id NOT IN (
    SELECT student_id
    FROM enrollments
    WHERE student_id IS NOT NULL
);
What Makes a Good Comment

Comments should explain the purpose of a query, not just repeat what the SQL already says. A comment that says "selects student names" adds nothing. A comment that says "finds students who registered but have not enrolled in any current-term courses" adds real context.

Aliases That Help

When you join multiple tables, use short but clear table aliases. One-letter aliases like s for students and e for enrollments are common and acceptable. But in longer queries, slightly longer aliases like stu and enr can be clearer:

SELECT stu.student_name, crs.course_name, enr.grade
FROM enrollments AS enr
    JOIN students AS stu ON enr.student_id = stu.student_id
    JOIN courses AS crs ON enr.course_id = crs.course_id;

Always choose aliases that make the query easier to read in context.

14.10

Choosing Simple Over Clever

One of the most important habits a database student can develop is the habit of choosing simple SQL over clever SQL.

When you learn subqueries, CTEs, and views, it can be tempting to use them everywhere, nesting queries inside queries, building complicated structures to show off what you know. Resist this temptation.

The goal of SQL is to get accurate answers from data in a way that other people can understand, verify, and maintain. A query that is technically impressive but impossible to follow is a liability, not an asset.

Simpler Is Usually Better

When a question can be answered with a simple single-table query, use a single-table query. When a join is all you need, use a join. Only reach for subqueries, CTEs, or views when they genuinely make the query clearer or solve a problem that simpler tools cannot solve.

Here is an example of unnecessary complexity:

SQL — Overcomplicated
-- Overcomplicated: finding students in department 10
SELECT student_name
FROM students
WHERE student_id IN (
    SELECT student_id
    FROM students
    WHERE department_id = 10
);

This query uses a subquery to filter the same table it is reading from. The subquery adds no value. The simple version is:

SQL — Simplified
SELECT student_name
FROM students
WHERE department_id = 10;

The second version is shorter, clearer, and faster. Use it.

When Complexity Is Justified

Complexity is justified when a question genuinely requires comparing a value to an aggregate, like an average or maximum, calculated from the data, when a question requires matching against a list produced by another query, when a query is long enough that naming its parts with CTEs makes it significantly easier to understand and debug, or when a set of columns or a join is reused so often that a named view would save significant repetition and reduce the risk of inconsistency.

In all other cases, prefer the simpler approach.

Testing Queries in Parts

One practical benefit of subqueries and CTEs is that they let you test a complex query in parts. If you are building a complicated query, write and test the innermost part first. Make sure it returns what you expect. Then wrap it in the next layer and test that. Continue outward.

This step-by-step testing approach is much more reliable than writing the whole query at once and hoping it works. Database students who adopt this habit catch their mistakes much earlier and build confidence in their queries before the queries are used in real systems.

The Habit of Reading Before Writing

Throughout this chapter, worked examples have been presented by reading an existing query first, and then explaining what it does. This is not an accident. Reading SQL well is a prerequisite for writing SQL well.

Before writing a new query, read and understand a similar query that already works. Before submitting a query to a real database, read through every clause and explain to yourself what each piece does, what table or result it reads from, what filter it applies, and what the output should look like. This habit of reading before writing, and explaining before submitting, produces more reliable SQL and fewer mistakes.

The key skill for reading subqueries is reading from the inside out: understand what the inner query produces first, then understand how the outer query uses it.
  • A view is a saved query stored in the database with a name. It behaves like a table and always reflects the current state of the underlying data. Views are useful for reusing complex queries, hiding complexity from users, protecting sensitive data by exposing only certain columns, and ensuring that related parts of an application use consistent query logic.
  • A common table expression uses the WITH keyword to name a temporary query result at the top of a SQL statement. CTEs do the same job as subqueries in FROM but are easier to read when a query has multiple named parts or when the same intermediate result is needed more than once.
  • Query readability is not a luxury. It is a practical requirement for reliable database work. Good SQL is formatted consistently, uses descriptive names for aliases and CTEs, includes comments that explain purpose, and chooses simple approaches over clever ones when the result is the same.
  • The broader lesson of this chapter is that SQL can be organized just like code. The same habits that make programs easier to read and maintain, clear naming, logical structure, separation of concerns, and step-by-step testing, apply equally to database queries.
Key Terms
Subquery

A complete SELECT statement written inside parentheses inside another SELECT statement. The inner query runs first and its result is used by the outer query.

Derived Table

A subquery written in the FROM clause, which is treated as a temporary table by the outer query. Also called an inline view.

Single-Value Subquery

A subquery that returns exactly one row and one column. Used in WHERE clauses with comparison operators such as >, <, or =.

List Subquery

A subquery that returns one column with multiple rows. Used in WHERE clauses with IN or NOT IN.

View

A saved query stored in the database under a name. It behaves like a table and always reflects the current state of the underlying data. It does not store data itself.

CREATE VIEW

The SQL statement used to define and save a view.

DROP VIEW

The SQL statement used to remove a view from the database.

Common Table Expression (CTE)

A named temporary result defined using the WITH keyword at the top of a SQL statement, available for use in the main query that follows.

WITH Keyword

The SQL keyword that introduces one or more common table expressions before the main query.

Query Readability

The quality of a SQL query that makes it easy for people, including the original author later, to understand what the query does, why it is written that way, and how to modify it correctly.

Inline View

Another name for a subquery used in the FROM clause.

NULL Protection in NOT IN

The practice of adding WHERE column IS NOT NULL to a subquery used with NOT IN, because a single NULL in the subquery result causes NOT IN to return no rows.

Common Misconceptions
Misconception 1
A view stores a copy of the data.

A view stores a query definition, not data. Every time you query a view, the database runs the underlying query against the current data. If the underlying tables are updated, the view immediately reflects those changes.

Misconception 2
Subqueries are always the right tool.

Subqueries solve specific problems but are not always necessary. Many questions can be answered with a simple join or filter. Always prefer the simplest approach that correctly answers the question.

Misconception 3
CTEs are stored in the database like views.

CTEs are temporary and exist only for the duration of the single query that defines them. They are not stored and cannot be reused across different queries. Views are stored and can be reused.

Misconception 4
NOT IN always works the same as IN.

NOT IN behaves unexpectedly when the subquery returns NULL values. A single NULL in the subquery result causes NOT IN to return no rows. Always add IS NOT NULL to the subquery when using NOT IN.

Misconception 5
Writing complex queries proves you are good at SQL.

Simple, clear queries that correctly answer the question are better than complex, clever queries that are hard to read and maintain. Query readability is a professional skill, not a compromise.
