Data Quality, Cleaning, and Validation

A database can be carefully designed, its tables properly separated, its keys thoughtfully chosen. And yet, if the data stored inside it is wrong, the entire system fails. This chapter teaches how to recognize, measure, fix, and prevent data quality problems before they undermine everything built on top of them.

A database can be carefully designed. Its tables can be properly separated. Its keys can be thoughtfully chosen. Its relationships can be accurately drawn. And yet, if the data stored inside it is wrong, the entire system fails.

This is not a theoretical concern. It is one of the most practical problems in real information systems. Data arrives from many sources: people typing into forms, files imported from other systems, sensors sending measurements, records copied from older databases, and spreadsheets submitted by departments. Every one of these sources can introduce problems. A name gets misspelled. A date is entered in the wrong format. A required field is left blank. A record is imported twice. A number falls so far outside the normal range that it suggests a typing error rather than a real value.

None of these problems necessarily means the software broke. The software may have accepted every problematic record without complaint. The problem is that the data itself is wrong, and wrong data produces wrong results, wrong reports, wrong decisions, and real harm to the people and organizations that rely on the system.

This chapter teaches students how to recognize, measure, evaluate, fix, and prevent data quality problems. Students will learn what data quality means, why it matters, how to identify specific kinds of problems, how to clean data systematically, and how to design validation rules that catch problems before they enter the database. Students will also learn how to document data quality work so that future users understand the history and condition of a dataset.

A Point Worth Stating Early

This chapter is not about blaming users for mistakes. It is about designing systems and practices that make good data easy to enter and bad data hard to miss.

4.1

Why Data Quality Matters

Data Is Only as Useful as It Is Reliable

When people use a database, they are trusting it. A librarian checking whether a book is available trusts the catalog. A hospital nurse checking a patient's medication list trusts the electronic record. A university administrator generating enrollment reports trusts the student database. A store manager checking inventory before placing an order trusts the inventory system.

That trust is reasonable. It is the whole point of keeping organized records. But trust placed in bad data leads to bad decisions. A book that is marked available when it has already been checked out sends a patron on a wasted trip to the shelf. A medication list with a missing allergy causes a dangerous prescription error. An enrollment report that double-counts students leads to incorrect planning. An inventory count that is off by a factor of ten leads to unnecessary purchases or dangerous shortages.

The phrase that captures this most directly is one that every student of data systems should learn early and remember permanently:

Garbage in, garbage out.

This expression means that a system can only produce useful outputs if it receives accurate inputs. No query, report, or analysis can fix data that was wrong before it entered the database.

The Scale of the Problem

Data quality problems are not rare edge cases. Studies across industries consistently find that a significant percentage of records in real organizational databases contain at least one error. This is not because organizations are careless. It is because data comes from many places, passes through many hands, changes over time, and is entered under time pressure by many people with different understandings of what is expected.

Some problems are introduced at the source: a person fills out a form incorrectly. Some are introduced in transit: data is exported from one system and imported into another, and the formats do not match cleanly. Some develop over time: an address that was correct five years ago is now wrong because the person moved. Some are inherited from older systems: records that predate the current database were migrated without careful checking.

The answer to all of these problems is not simply to hope for perfect input. The answer is to measure data quality, understand where problems come from, clean data systematically, and build validation rules that catch new problems as they arrive.

Data Quality and Decision-Making

Poor data quality has direct consequences for decision-making. When a manager asks a database for the total revenue from a product category, a few duplicate records or missing values can skew the result enough to change the decision. When a researcher analyzes a dataset for patterns, outliers caused by typing errors can produce misleading findings. When a government agency publishes statistics drawn from an administrative database, errors in the underlying records become errors in public knowledge.

Data quality is not a technical concern that only database administrators need to think about. It is a concern for everyone who designs systems, enters data, analyzes results, or makes decisions based on what a database returns.

What Data Quality Work Looks Like

Data quality work takes several forms.

Profiling means examining a dataset to understand its shape, its patterns, its ranges, and its problems before doing anything else with it.

Cleaning means identifying specific errors and correcting them or removing them.

Validation means building rules that check whether new data meets standards before it enters the database.

Documentation means recording what was found, what was changed, and why, so that future users understand the dataset's history.

Each of these activities is covered in this chapter. Together they form the practice of treating data as something that must be actively maintained rather than passively collected.

4.2

Accuracy

What Accuracy Means

Accuracy means that data correctly represents the real-world thing it is supposed to describe. An accurate record of a student's name spells the name the way the student actually spells it. An accurate record of a temperature measurement contains the value actually recorded by the sensor. An accurate list of a customer's purchases includes exactly the purchases that customer actually made, nothing more and nothing less.

Accuracy is one of the most fundamental dimensions of data quality. Without it, nothing else matters very much. A dataset can be complete, consistently formatted, and beautifully structured, but if its values are wrong, it is unreliable.

Sources of Inaccuracy

Inaccurate data comes from many places.

Manual entry errors are the most familiar. A person types a name, address, amount, or date and makes a mistake. Transpositions are common: a person types 91 when they meant 19, or Samntha when they meant Samantha. These errors are easy to make and easy to miss.

Misunderstanding of what a field means causes systematic inaccuracy. If one person entering data understands a field called Enrollment Date to mean the date a student registered for the semester, and another person understands it to mean the date the student first enrolled at the university, the same field will contain two different kinds of values that cannot be meaningfully compared.

Stale data means data that was accurate when it was entered but is no longer accurate because the real world changed. A phone number that was correct two years ago may belong to someone else now. A price that was correct last month may have changed. A person's job title may have changed since the record was created.

System-to-system transfer errors occur when data is moved from one database or file to another and something goes wrong: a field is mapped to the wrong column, a format conversion introduces an error, or a value is truncated because the receiving field is too short.

Automated collection errors happen when data is generated by software rather than entered by people. A sensor records a value that is outside its measurement range. A web form captures incorrect data because a user filled it out in an unexpected way. An automated import pulls a value from the wrong location in a source file.

Detecting and Improving Accuracy

Some inaccuracy is detectable through pattern analysis. A field containing ages should not have a value of 200 or a value of −5. A field containing email addresses should follow a recognizable format. When values fall outside expected patterns, that is a signal worth investigating.

Some inaccuracy is only detectable by comparison with a trusted source. If a student's name in the database does not match the name on the official admission document, the record can be corrected. Some inaccuracy is essentially impossible to detect from the data alone. If a person enters an incorrect but plausible date of birth, the database has no way to know it is wrong. This is why external verification and careful data entry practices matter.

Accuracy can be improved through controlled input methods such as dropdown menus, date pickers, and lookup fields, which reduce the chance that a person types something incorrect. Validation rules reject values that fall outside allowed ranges or formats. Regular audits compare database records against source documents.

4.3

Completeness

What Completeness Means

Completeness means that a dataset contains all the records it should contain, and that each record contains all the fields it should contain. A complete student database has a record for every enrolled student. A complete record for each student includes every required field: name, student number, program, enrollment date, contact information, and so on.

Incompleteness takes two forms: missing records and missing values within records.

Missing Records

Missing records occur when the database does not contain an entry for something that should be there. A library catalog might be missing some books. A customer database might be missing some customers who purchased before the current system was installed. An employee record system might be missing records for contractors who were never entered at the time of hire.

Missing records are often hard to detect unless there is a separate reference to compare against. If you know that a library has 15,000 books and the catalog contains only 14,200 records, you know records are missing. If you have no independent count to compare against, you may not notice the gap.

Missing Values Within Records

Missing values are more commonly encountered. A record exists, but one or more fields are blank or marked as unknown. This can happen for several reasons: the information was never collected, the information is not applicable to this particular record, the information was collected on paper but never entered, or the information exists in another system but was not migrated.

One of the most important practices in database design and data quality work is to distinguish carefully between different kinds of absence. SQL databases use NULL to represent missing values, but a single NULL cannot distinguish between "not applicable," "not known," "not provided," and "not yet determined." Good database design sometimes uses additional fields or status codes to capture which kind of missing is present.

The Impact of Incompleteness

Incomplete data limits what queries can return reliably. If half the records in a table have missing values in the Region field, a report grouped by region will be misleading: the analysis will appear to cover all records but will silently omit or misclassify a large fraction.

Completeness assessment is one of the first steps in any serious data quality review. Before analyzing a dataset, a data practitioner asks: how many records are missing values in each field? Which fields are most affected? Is the pattern of missing values random or systematic?

4.4

Consistency

What Consistency Means

Consistency means that data represents the same real-world facts the same way throughout the database. If a city is recorded as New York in one table and new york in another and N.Y. in a third, the data is inconsistent. If a product is referred to by one identifier in the sales table and a different identifier in the inventory table, those two tables cannot be reliably joined.

Inconsistency is especially dangerous in databases because SQL relies on exact matching. A query that looks for all records from a particular city will miss some of them if the city name is spelled differently in different records.

Types of Inconsistency

Format inconsistency occurs when the same kind of value is stored in different formats. Phone numbers might appear as 555-1234, 5551234, (555) 1234, or +1 555 1234. Dates might appear as 2024-03-15, 15/03/2024, or March 15, 2024. All of these might represent the same information, but a database treating them as text will not recognize them as equivalent.

Case inconsistency occurs when text values are stored with different capitalization. SMITH, Smith, and smith look different to a database engine even though they represent the same thing.

Synonym inconsistency occurs when different words or abbreviations are used for the same concept. A department might be recorded as Computer Science, Comp Sci, CS, or Compsci depending on who entered the record and when.

Cross-table inconsistency occurs when the same fact is stored in multiple places and the copies disagree. This is one of the reasons that good relational database design stores each fact in only one place.

Temporal inconsistency occurs when a record contains dates or timestamps that are logically impossible: an order with a delivery date before its order date, or a record with a modification timestamp earlier than its creation timestamp.

Detecting and Fixing Inconsistency

Detecting format inconsistency often involves looking at the range of distinct values in a field. If a field should contain country names and the list of distinct values includes USA, United States, United States of America, U.S.A., and US, the field has a problem.

Fixing format inconsistency requires deciding on a standard and applying it consistently. In a database context this often means using lookup tables or reference codes rather than free-text fields. Instead of storing a country name as free text, the database stores a standardized country code from a reference list, and the human-readable name is retrieved through a join.

4.5

Validity

What Validity Means

Validity means that data values fall within the rules defined for them. A valid age for an adult library patron is a positive integer somewhere in a realistic human range. A valid email address follows the structure of an email address. A valid enrollment date is a real calendar date, not February 30 or November 45.

Validity is different from accuracy. An age of 25 might be valid (it falls within the allowed range) but inaccurate (the person is actually 30). Validity is about whether a value obeys the rules. Accuracy is about whether a value is true.

Types of Validity Problems

Type invalidity occurs when a value is the wrong data type for its field. A numeric field contains text. A date field contains a value that is not a recognizable date.

Range invalidity occurs when a value is the correct type but falls outside the allowed range. A percentage field contains a value of 150. A rating field that accepts values from 1 to 5 contains a 7.

Format invalidity occurs when a value is the correct type but does not follow the required format. A phone number field requires ten digits but contains eight. A postal code field for US addresses requires five digits but contains letters.

Referential invalidity occurs when a foreign key value refers to a record that does not exist in the referenced table. A loan record references a resource that has been deleted.

Business rule invalidity occurs when a value violates a real-world constraint. A hotel booking with a checkout date before the check-in date. A course with an enrollment count higher than its listed capacity.

Why Validity Problems Occur

Validity problems occur when validation rules are not enforced at the point of data entry. If a form allows any text to be typed into a date field, invalid dates will appear. If an import process does not check foreign keys before loading records, referential invalidity will be introduced. Database constraints, covered in depth in Chapter 6, are one of the most powerful tools for preventing validity problems.

4.6

Duplicate Records

What Duplicates Are

A duplicate record is a record that represents the same real-world thing as another record already in the database. A library catalog with the same book entered twice. A customer database with the same person registered under two slightly different email addresses. An event registration system where the same person submitted the form twice because they were not sure the first submission went through.

Duplicates are one of the most common and damaging data quality problems. They distort counts, inflate totals, cause confusion, waste storage, and make it difficult to know which copy of a record is the most accurate or up-to-date.

How Duplicates Are Created

Duplicates arise from intentional double submission, when a user is unsure whether the first submission went through. They arise from multiple data sources, when the same person appears in records collected through different channels. They arise from system migrations, when an older system already contained duplicates or when the migration process creates new ones. They arise from manual entry without lookup, when a person creates a new record rather than finding the existing one. They arise from case and format differences that cause deduplication checks to miss records that represent the same entity.

Detecting and Handling Duplicates

Detecting obvious duplicates is straightforward: look for records where a primary identifier such as a student number, ISBN, or email address appears more than once. Detecting near-duplicates is harder. Two records might represent the same person but have slightly different spellings of the name, different phone numbers, or different addresses because one record is outdated.

Once duplicates are identified, the options include merging (combining two records into one, keeping the most accurate information from each), deleting (removing one of the duplicates when one is clearly erroneous), and flagging (marking candidates for human review rather than automatically resolving them). Deduplication should always be documented thoroughly. A merge that combines the wrong records can create new errors that are harder to detect than the original duplicates.

4.7

Outliers and Unusual Values

What Outliers Are

An outlier is a value that falls far outside the typical range of values in a field. A dataset of student test scores where most values fall between 50 and 100 but one record contains a score of 9999 is likely to contain an outlier caused by an error. A dataset of transaction amounts where most values are between $5 and $500 but one record shows $5,000,000 may contain either an outlier error or a legitimately unusual but real transaction.

The challenge with outliers is that they can represent either genuine extreme values or errors. Before deciding how to handle an outlier, a data practitioner must investigate it.

Why Outliers Matter

Outliers matter for two distinct reasons. First, they may indicate errors in the data. A recorded temperature of 5000 degrees Celsius from a room sensor is almost certainly an instrument error or a data entry mistake. Second, even when outliers are genuine, they can dramatically affect statistical summaries. A report that states the average transaction amount is $4,500 may be technically correct but deeply misleading if 99% of transactions are under $50 and one transaction is for $4,000,000.

Detecting and Handling Outliers

Simple techniques for detecting outliers include range checking (comparing each value against the known valid range for that field), statistical methods (computing the mean and standard deviation and flagging values more than a certain number of standard deviations from the mean), visualization (plotting the distribution and looking for isolated values), and comparison to related fields (a transaction recorded as $50,000 might be accepted as normal when the database shows the same customer regularly places large orders).

Outliers should never be silently removed without documentation. A decision to remove or adjust an outlier changes the dataset and must be recorded so that future users understand what the cleaned dataset represents.

4.8

Formatting Problems

What Formatting Problems Are

Formatting problems occur when data is stored in a format that does not match what the database or analysis tools expect. A date stored as text rather than as a date type. A number stored with a currency symbol embedded in it, so that $45.00 cannot be used in arithmetic. A phone number stored with spaces, dashes, parentheses, and country codes mixed inconsistently across records.

Formatting problems are extremely common in datasets assembled from multiple sources, entered by multiple people, or migrated from older systems.

Common Formatting Problems

Date format variation: Dates are entered as 2024-03-15, 15/03/2024, March 15 2024, and 3-15-24 by different people. The database stores them all as text, and queries that try to sort or compare dates produce incorrect results.

Number-as-text: Numeric values are stored as text strings because the original format was a CSV file or spreadsheet where the column was formatted as text. A field containing "1,500" with a comma as part of the string cannot be summed or averaged without conversion.

Embedded symbols: Currency symbols, percent signs, or unit labels are embedded in numeric fields. The field contains $45.00 or 95% or 15kg rather than the raw numeric values 45, 0.95, or 15.

Trailing and leading whitespace: A value typed as   Smith   with spaces before and after is not the same string as Smith to a database engine, causing join failures and filter mismatches.

Mixed case: A field contains SMITH, Smith, smith, and sMITH for what should be the same value.

Fixing Formatting Problems

Fixing formatting problems typically involves transformation: converting values from the format they are stored in to a standardized format. Common transformations include converting text dates to proper date types, stripping currency symbols and commas from numeric text fields, trimming leading and trailing whitespace, standardizing case, and standardizing abbreviations to a controlled vocabulary. When done properly, these transformations are documented so that it is clear what the original values were and what transformations were applied.

4.9

Data Cleaning

What Data Cleaning Is

Data cleaning, sometimes called data cleansing or data scrubbing, is the process of identifying problems in a dataset and correcting or removing them to improve data quality. It is not glamorous work, but it is some of the most important work in any data-driven project. Industry practitioners commonly report that data cleaning takes the majority of the time in a data project, sometimes as much as eighty percent.

Cleaning is not the same as deleting. The goal is not to throw away any record that has a problem. The goal is to move the data from its current condition to a more accurate, complete, consistent, valid, and usable condition.

A Systematic Cleaning Process

Data cleaning should follow a systematic process. Jumping in and making changes without a plan leads to confusion, inconsistency, and the introduction of new errors.

  1. Profile the data.

    Before making any changes, examine the dataset. Look at the range of values in each field, count missing values, identify distinct values for categorical fields, and look for obvious formatting problems. This is data profiling, and it is the essential first step.

  2. Document the problems found.

    Write down what problems exist, where they are, and how many records are affected. This documentation is important both for guiding the cleaning work and for later reporting on what was done.

  3. Prioritize the problems.

    Not all problems are equally serious. A missing middle name is less consequential than a missing primary key. Focus cleaning effort where it will have the most impact on the system's reliability.

  4. Decide on corrections.

    For each class of problem, decide on the appropriate correction. Some problems can be fixed systematically. Some require manual review. Some may be unfixable with available information and must be documented as known issues.

  5. Apply corrections to a copy.

    Make changes on a copy of the data rather than the original, so that mistakes can be reversed. Never overwrite raw data with cleaned data without keeping the original.

  6. Validate the corrections.

    After cleaning, re-examine the dataset to confirm that the problems have been addressed and that no new problems have been introduced. Run the same profiling checks done in Step 1 and compare the results.

  7. Document what was done.

    Record exactly what changes were made, why, and when. This documentation is part of the dataset's history and is essential for future users who need to understand the data's provenance.

Cleaning Is Not a One-Time Task

Data cleaning is ongoing, not a one-time event. New data enters the database continuously. New sources introduce new formats and new problems. Data that was accurate when it was entered becomes stale as the real world changes. Regular cleaning cycles, combined with good validation rules at the point of entry, keep a database in usable condition over time.

4.10

Validation Rules and Documentation

What Validation Rules Are

Validation rules are defined checks that data must pass before it is accepted into the database. They are the first line of defense against bad data. Instead of waiting to discover data quality problems after the fact, validation rules prevent many problems from entering the database at all.

Validation rules can be enforced at the user interface layer (a web form checks input before submitting), at the database layer (constraints defined in the schema reject any data that violates the rules, regardless of which application sent it), and at the import layer (an import process checks data against rules before loading). Validation rules at the database layer are the most reliable because they apply universally, regardless of the source.

Types of Validation Rules

Required field rules: A field must not be empty. Enforced with a NOT NULL constraint in SQL.

Data type rules: A field must contain a value of the correct type. Enforced by the database's type system.

Range rules: A numeric value must fall within a specified range. An age must be between 0 and 130. Enforced with CHECK constraints in SQL.

Format rules: A value must match a specified format. An email address must contain an @ symbol. Often handled at the application layer or with pattern-matching checks.

Uniqueness rules: No two records may have the same value in a specific field or combination of fields. Enforced with UNIQUE constraints in SQL.

Referential integrity rules: A foreign key value must reference a record that exists in the referenced table. Enforced with foreign key constraints in SQL.

Business logic rules: A booking cannot have a checkout date earlier than its check-in date. An order cannot be marked as shipped before it is confirmed. Often enforced at the application layer or with carefully designed constraints.

Designing Good Validation Rules

Good validation rules are accurate (they reject what should be rejected and accept what should be accepted), understandable (when a rule rejects a value, the error message tells the user what is wrong and how to fix it), consistent (rules are applied uniformly across all entry points), and documented (every rule is written down so that it can be reviewed, updated, and communicated).

Rules that are too strict frustrate users and lead them to enter placeholder values just to get past the validation. Rules that are too lenient allow bad data to enter. Calibration requires understanding the data, the business process, and the users.

Documentation of Data Quality

Documentation is not optional in serious data work. A database without documentation of its data quality history is a database that future users will not be able to trust or interpret correctly.

Good data quality documentation includes a data dictionary (a description of every table and field, including the meaning of each field, the rules that apply to it, and what NULL means for that field), a cleaning log (a record of what cleaning operations were performed, when, by whom, and what the data looked like before and after), a validation rule registry (a list of all validation rules in the database, where they are enforced, and what they check), and a known issues record (data quality problems that were identified but could not be corrected, along with a description of their likely impact).

Documentation Does Not Need to Be Elaborate

A simple text file or a table in the database itself can serve the purpose of data quality documentation. What matters is that it exists and is kept up to date. Future users, including your future self, will be grateful for it.

Chapter Summary
  • Data quality is not an abstract concern. Every database in everyday use depends on data that is accurate enough to be trusted. Bad data produces bad results even when the software and the queries are perfectly correct. The principle is simple: garbage in, garbage out.
  • Accuracy means that values correctly represent the real-world facts they are supposed to capture. Accuracy problems come from typing errors, misunderstandings, stale data, and transfer failures between systems.
  • Completeness means that all records that should exist do exist, and that all fields that should have values do have values. Different kinds of absence, including not applicable, not known, not provided, and not yet determined, require different handling.
  • Consistency means that the same fact is represented the same way throughout the database. Format inconsistency, case inconsistency, synonym inconsistency, and cross-table inconsistency all undermine queries, joins, and reports.
  • Validity means that values obey the rules defined for their fields: correct data types, correct ranges, correct formats, and correct references to other tables.
  • Duplicate records represent the same real-world thing more than once. They distort counts, inflate summaries, and create confusion about which record is authoritative. Duplicates must be identified, investigated, and handled carefully.
  • Outliers are values far outside the typical range. They may represent errors or genuine extreme values and require investigation before any decision is made about how to handle them.
  • Formatting problems occur when the same kind of value is stored in different formats across records or tables, making comparison, sorting, and analysis unreliable.
  • Data cleaning is a seven-step systematic process: profile, document, prioritize, decide, apply to a copy, validate, and document changes. Cleaning should always preserve the original data.
  • Validation rules prevent data quality problems at the point of entry. The most reliable rules are enforced by the database itself through constraints. Good rules are accurate, understandable, consistent, and documented.
  • Documentation of data quality work, including a data dictionary, cleaning log, validation rule registry, and known issues record, is not optional. A database without quality documentation cannot be safely trusted by future users.
Key Terms
Accuracy

The quality of data that correctly represents the real-world facts it is supposed to describe.

Business rule

A constraint that reflects a real-world requirement, such as requiring a checkout date to be later than a check-in date.

CHECK constraint

A SQL constraint that validates that values in a column meet a specified condition, such as falling within a defined range.

Completeness

The quality of a dataset that contains all the records it should contain, and all the required fields within each record.

Consistency

The quality of data where the same fact is represented the same way in all places it appears, and where related facts across tables do not contradict each other.

Data cleaning

The process of identifying, investigating, and correcting problems in a dataset to improve its quality. Also called data cleansing or data scrubbing.

Data dictionary

Documentation describing each table and field in a database, including field meanings, validation rules, what NULL means for that field, and any known issues.

Data profiling

The practice of examining a dataset to understand its structure, content, value ranges, and quality before analyzing or using it.

Data quality

The degree to which data is fit for its intended use, typically evaluated across dimensions including accuracy, completeness, consistency, and validity.

Duplicate record

A record that represents the same real-world thing as another record already in the database.

Format inconsistency

A condition where the same kind of value is stored in different formats across records, making comparison and analysis unreliable.

Garbage in, garbage out

The principle that a system can only produce reliable outputs if it receives reliable inputs. No query or analysis can fix data that was wrong before it entered the database.

NOT NULL constraint

A SQL constraint that requires a field to have a value in every record, preventing empty entries in required fields.

NULL

The marker used in SQL databases to represent a missing, unknown, or not-applicable value. A single NULL cannot distinguish between different kinds of absence.

Outlier

A value that falls far outside the typical range of values in a field, which may indicate a data entry error or a genuine but extreme real-world value.

Referential integrity

The property of a database where every foreign key value refers to an existing record in the referenced table.

Stale data

Data that was accurate when it was entered but is no longer accurate because the real world has changed.

UNIQUE constraint

A SQL constraint that prevents two records from having the same value in a specified field or combination of fields.

Validation

The process of checking whether data meets defined rules before it is accepted into the database.

Validity

The quality of data where all values obey the rules defined for their fields, including correct types, ranges, formats, and references.

Preview of Chapter 5
Coming Next — Chapter 5: Entities, Attributes, and Relationships

Chapter 4 focused on the data itself: how to recognize when it is wrong and what to do about it. Chapter 5 moves to a different level of the same problem: how to design the structure that holds the data so that it accurately represents the real world.

Chapter 5 introduces data modeling, the practice of identifying the real-world things a system must store information about, describing their properties, and defining how they relate to each other. Students will learn about entities, attributes, and relationships, and they will begin drawing entity-relationship diagrams as a tool for visualizing and communicating database designs.

Good data modeling is one of the most powerful ways to prevent data quality problems before they occur. A well-designed structure makes it natural to enter correct data and difficult to enter incorrect data. A poorly designed structure makes data quality problems almost inevitable, no matter how many validation rules are added afterward. Chapter 5 is where students begin to think like database designers.
