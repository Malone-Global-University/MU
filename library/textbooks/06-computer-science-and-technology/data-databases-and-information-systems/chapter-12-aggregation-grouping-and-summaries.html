Aggregation, Grouping, and Summaries

Individual records tell you about one student, one book, one order, or one appointment. But most real questions about data are not about one thing, they are about many things at once. This chapter introduces aggregate functions, GROUP BY, and HAVING as the tools that turn raw records into the summaries behind every report and dashboard.

Individual records tell you about one student, one book, one order, or one appointment. But most real questions about data are not about one thing. They are about many things at once. How many students enrolled this term? What is the average score on an assignment? Which subject has the most resources? What was the highest loan total last month?

These are summary questions. They collapse many rows into a smaller number of meaningful answers. SQL answers them with aggregate functions and the GROUP BY clause.

This chapter introduces aggregation as the SQL skill that turns raw records into useful information. Students will learn to count, total, average, and find extremes across a dataset. They will learn to group results by category. They will learn to filter groups with HAVING, which works differently from the WHERE clause they already know. And they will connect these query skills to the reports and dashboards that real people use to make decisions.

Note for Students

Aggregation is where SQL becomes genuinely powerful for understanding data, not just retrieving it. The functions in this chapter are short to type, but the thinking behind them deserves your full attention.

12.1

Why Summaries Matter

Raw data is rarely the answer to a question.

Suppose a school stores every assignment submission in a table. On any given day, that table might contain thousands of rows. No one reads thousands of rows to understand how a class is performing. Instead, someone calculates a summary: the average score, the number of submissions, the percentage that met the deadline. The summary turns the raw data into something a teacher, administrator, or student can actually understand and act on.

Summaries appear everywhere in real information systems. A library report shows how many resources were borrowed each month. A hospital dashboard shows average wait times by department. A store system shows total sales by product category. A university system shows enrollment counts by course and term. None of these require looking at every individual row. They require collapsing many rows into a smaller number of meaningful numbers.

In SQL, this collapsing is called aggregation. Aggregate functions compute a single result from a set of rows. The most common ones are COUNT, SUM, AVG, MIN, and MAX.

Central Idea

Understanding aggregation is not just a technical skill, it is a thinking skill. Before writing an aggregate query, a student must understand what question is being asked, what rows should be included in the calculation, and whether the results should be grouped by some category or computed across the whole table. Getting those decisions right is as important as getting the syntax right.

12.2

Counting Records with COUNT

The simplest aggregate function is COUNT. It answers the question: how many rows match this condition?

COUNT(*) counts every row in the result set, including rows with missing values in some columns. COUNT(column_name) counts only rows where that specific column is not null.

Example table: enrollments
enrollment_idstudent_idcourse_idgradeenrolled_date
1101CS101A2024-09-01
2102CS101B2024-09-01
3103CS101NULL2024-09-02
4101CS202A2024-09-01
5104CS202C2024-09-03

Query: How many total enrollment records exist?

SQL — Example Query
SELECT COUNT(*) AS total_enrollments
FROM enrollments;
Result
total_enrollments
5

Query: How many enrollments have a recorded grade?

SQL — Example Query
SELECT COUNT(grade) AS graded_enrollments
FROM enrollments;
Result
graded_enrollments
4

Notice that COUNT(grade) returns 4, not 5, because one row has NULL in the grade column. COUNT(*) counts all rows. COUNT(column_name) skips nulls.

This distinction matters whenever data is incomplete. A count of all rows tells you how many records exist. A count of a specific column tells you how many records have a value in that column. These are different questions, and they can have different answers.

Using COUNT with WHERE

You can combine COUNT with WHERE to count only the rows that meet a condition.

Query: How many students enrolled in CS101?

SQL — Example Query
SELECT COUNT(*) AS cs101_enrollments
FROM enrollments
WHERE course_id = 'CS101';
Result
cs101_enrollments
3
Using DISTINCT with COUNT

Sometimes you want to count unique values, not all values. COUNT(DISTINCT column_name) counts only the distinct non-null values in a column.

Query: How many distinct students are enrolled in any course?

SQL — Example Query
SELECT COUNT(DISTINCT student_id) AS unique_students
FROM enrollments;
Result
unique_students
4

Student 101 appears twice, enrolled in two courses, but is counted only once. The result is 4 unique students, not 5 rows.

12.3

Totals with SUM

SUM adds up all the values in a numeric column across the rows in the result.

Example table: loans
loan_iduser_idresource_iddays_borrowedfine_amount

2202R02141.50

4203R0430.00
5202R05100.75

Query: What is the total amount of fines across all loans?

SQL — Example Query
SELECT SUM(fine_amount) AS total_fines
FROM loans;
Result
total_fines
5.25

Query: What is the total number of days borrowed across all loans?

SQL — Example Query
SELECT SUM(days_borrowed) AS total_days
FROM loans;
Result
total_days
55

SUM ignores null values. If a column has some null entries, those rows are skipped in the total, which is usually the desired behavior.

SUM only makes sense on numeric columns. Trying to sum a text column or a date column will produce an error in most database systems.

12.4

Averages with AVG

AVG computes the arithmetic mean of a numeric column. It adds up all non-null values and divides by the number of non-null rows.

Query: What is the average number of days borrowed per loan?

SQL — Example Query
SELECT AVG(days_borrowed) AS average_days
FROM loans;
Result
average_days
11.0

AVG is straightforward, but students should be aware of a common mistake. Null values are excluded from both the sum and the count that AVG uses internally. If a column has many nulls, the average reflects only the rows that have data, which may not represent the full population.

Watch Closely

If a table of 100 students has assignment scores for only 60 of them, AVG(score) will compute the average over 60 rows, not 100. Whether this is the right answer depends on whether the missing scores should be treated as zero or treated as unknown. SQL assumes unknown and excludes them.

Query: What is the average fine amount?

SQL — Example Query
SELECT AVG(fine_amount) AS average_fine
FROM loans;
Result
average_fine
1.05
12.5

Minimums and Maximums

MIN and MAX find the smallest and largest values in a column.

Query: What is the shortest loan duration?

SQL — Example Query
SELECT MIN(days_borrowed) AS shortest_loan
FROM loans;
Result
shortest_loan
3

Query: What is the largest fine?

SQL — Example Query
SELECT MAX(fine_amount) AS largest_fine
FROM loans;
Result
largest_fine
3.00

MIN and MAX also work on text columns, where they return the alphabetically first and last values according to the database's collation rules. They work on date columns, where they return the earliest and latest dates.

Query: What is the earliest enrollment date in the enrollments table?

SQL — Example Query
SELECT MIN(enrolled_date) AS first_enrollment
FROM enrollments;
Result
first_enrollment
2024-09-01

Like all aggregate functions, MIN and MAX ignore null values.

Summary So Far
FunctionWhat it returns
COUNT(*)Number of rows
COUNT(col)Number of non-null values in a column
COUNT(DISTINCT col)Number of distinct non-null values
SUM(col)Total of all non-null numeric values
AVG(col)Mean of all non-null numeric values

All of these functions collapse many rows into one value. That is the defining feature of an aggregate function.

12.6

Grouping with GROUP BY

The aggregate functions in sections 12.2 through 12.5 all collapsed the entire result set into a single row. But most real summary questions want results broken down by some category. How many enrollments per course? What is the total fines per user? What is the average score per assignment?

GROUP BY divides the rows into groups based on one or more column values, and then applies the aggregate function separately within each group.

Syntax
SQL — Syntax Pattern
SELECT grouping_column, aggregate_function(value_column)
FROM table_name
GROUP BY grouping_column;
Example: Enrollment Count per Course
SQL — Example Query
SELECT course_id, COUNT(*) AS enrollment_count
FROM enrollments
GROUP BY course_id;
Result
course_idenrollment_count
CS1013
CS2022

The database splits the enrollments table into two groups, one for each course_id. Then it counts the rows in each group separately and returns one row of results per group.

Example: Total Fines per User
SQL — Example Query
SELECT user_id, SUM(fine_amount) AS total_fines
FROM loans
GROUP BY user_id;
Result
user_idtotal_fines
2013.00
2022.25
2030.00
Example: Average Days Borrowed per User
SQL — Example Query
SELECT user_id, AVG(days_borrowed) AS avg_days
FROM loans
GROUP BY user_id;
Result
user_idavg_days
20114.0
20212.0
2033.0
The Rule About SELECT and GROUP BY

When a query uses GROUP BY, every column in the SELECT clause must either be listed in the GROUP BY clause or be wrapped in an aggregate function. This rule exists because each group produces exactly one row of output. If you try to include a column that has different values within the same group, the database would not know which value to show.

SQL — Invalid Query
-- BROKEN: enrolled_date is not in GROUP BY and not aggregated
SELECT course_id, enrolled_date, COUNT(*)
FROM enrollments
GROUP BY course_id;

The enrollments table has multiple enrolled_date values within the CS101 group. The database cannot produce one result row for CS101 while also showing a single enrolled_date that represents the whole group. The query will produce an error or unpredictable results depending on the database system.

SQL — Valid Fixes
-- Option 1: Add enrolled_date to GROUP BY
SELECT course_id, enrolled_date, COUNT(*)
FROM enrollments
GROUP BY course_id, enrolled_date;

-- Option 2: Aggregate enrolled_date instead
SELECT course_id, MIN(enrolled_date) AS first_enrollment_date, COUNT(*)
FROM enrollments
GROUP BY course_id;
Grouping by Multiple Columns

You can group by more than one column to get finer-grained summaries.

Query: How many enrollments exist per student per course?

SQL — Example Query
SELECT student_id, course_id, COUNT(*) AS enrollment_count
FROM enrollments
GROUP BY student_id, course_id;

This creates a group for each unique combination of student_id and course_id.

12.7

Filtering Groups with HAVING

WHERE filters individual rows before aggregation. HAVING filters groups after aggregation.

This distinction is important. WHERE runs first and decides which rows enter the aggregation. HAVING runs after the aggregate functions have computed, and decides which groups appear in the final result.

Syntax
SQL — Syntax Pattern
SELECT grouping_column, aggregate_function(value_column)
FROM table_name
GROUP BY grouping_column
HAVING aggregate_function(value_column) condition;
Example: Show Only Courses with More Than Two Enrollments
SQL — Example Query
SELECT course_id, COUNT(*) AS enrollment_count
FROM enrollments
GROUP BY course_id
HAVING COUNT(*) > 2;
Result
course_idenrollment_count
CS1013

CS202 has only 2 enrollments, so it is excluded. CS101 has 3, so it appears.

Example: Show Only Users Whose Total Fines Exceed One Dollar
SQL — Example Query
SELECT user_id, SUM(fine_amount) AS total_fines
FROM loans
GROUP BY user_id
HAVING SUM(fine_amount) > 1.00;
Result
user_idtotal_fines
2013.00
2022.25

User 203 has total fines of 0.00, which does not exceed 1.00, so they do not appear.

WHERE Versus HAVING: A Side-by-Side Comparison
Filters

WHERE — individual rows
HAVING — groups

Runs

WHERE — before GROUP BY
HAVING — after GROUP BY

Can Reference

WHERE — any column
HAVING — aggregate functions and grouped columns

Common Use

WHERE — limiting raw data
HAVING — limiting summary results

A query can use both WHERE and HAVING at the same time.

Example: Among loans made in 2024, show only users whose total fines exceed one dollar

SQL — Example Query
SELECT user_id, SUM(fine_amount) AS total_fines
FROM loans
WHERE loan_id >= 1
  AND fine_amount IS NOT NULL
GROUP BY user_id
HAVING SUM(fine_amount) > 1.00;

Here WHERE removes rows before grouping. Then HAVING filters the grouped results afterward.

12.8

Summary Column Aliases

Aggregate functions produce columns in the result set, but those columns do not have natural names from the original table. Without an alias, the column header might appear as something like COUNT(*) or SUM(fine_amount), which is not readable in a report or dashboard.

Aliases with AS give aggregate columns meaningful names.

SQL — Example Query
SELECT
    course_id,
    COUNT(*)            AS enrollment_count,
    COUNT(grade)        AS graded_count,
    AVG(grade)          AS average_grade
FROM enrollments
GROUP BY course_id;

The alias names enrollment_count, graded_count, and average_grade appear as the column headers in the result set. They make the output clear without requiring the reader to parse SQL syntax.

Aliases also allow you to reference the column cleanly in programming code that reads query results. A column named COUNT(*) is awkward to work with in most languages. A column named enrollment_count is not.

12.9

Row Filters Versus Group Filters

Students sometimes confuse when to use WHERE and when to use HAVING. A useful mental model is to think about what is being filtered, and when in the process that filtering happens.

The Order of SQL Operations

SQL clauses do not execute in the order they are written. The logical order is shown below.

  1. FROM

    Identify the table, or joined tables.

  2. WHERE

    Filter individual rows.

  3. GROUP BY

    Divide remaining rows into groups.

  4. Aggregate Functions

    Compute within each group.

  5. HAVING

    Filter groups.

  6. SELECT

    Choose which columns appear in the result.

  7. ORDER BY

    Sort the final result.

  8. LIMIT

    Limit the number of rows returned.

Understanding this order explains why WHERE cannot reference aggregate functions: aggregates do not exist yet when WHERE runs. Only HAVING, which comes after aggregation, can reference them.

SQL — Invalid Query
-- BROKEN: WHERE cannot reference COUNT(*)
SELECT course_id, COUNT(*) AS enrollment_count
FROM enrollments
WHERE COUNT(*) > 2
GROUP BY course_id;
SQL — Valid Query
-- CORRECT: HAVING references COUNT(*) after aggregation
SELECT course_id, COUNT(*) AS enrollment_count
FROM enrollments
GROUP BY course_id
HAVING COUNT(*) > 2;
When to Use WHERE and When to Use HAVING

Use WHERE to restrict which rows enter the aggregation. This makes the aggregation faster and more focused. Use HAVING to restrict which groups appear in the result after aggregation.

Example: Average loan duration for users who borrowed at least three times

SQL — Example Query
SELECT user_id, AVG(days_borrowed) AS avg_days, COUNT(*) AS loan_count
FROM loans
GROUP BY user_id
HAVING COUNT(*) >= 3;

There is no row-level condition here, so no WHERE is needed. The HAVING clause does the work of keeping only groups with at least three loans.

Example: Average loan duration for resources borrowed in September 2024, for users who borrowed multiple times

If the loans table had a loan_date column, the query might look like this:

SQL — Example Query
SELECT user_id, AVG(days_borrowed) AS avg_days
FROM loans
WHERE loan_date >= '2024-09-01'
  AND loan_date < '2024-10-01'
GROUP BY user_id
HAVING COUNT(*) > 1;

The WHERE clause limits rows to September 2024 loans only. Then the grouping and aggregation happen. Then HAVING keeps only users who appear more than once within that filtered set.

12.10

Summary Queries in Reports

Aggregate queries are the engine behind most reports and dashboards in real information systems. Understanding how aggregate queries map to real outputs makes them easier to design and debug.

Report: Enrollment Summary by Course

A school administrator might need a report showing, for each course, how many students are enrolled, how many have received grades, and the average grade. This maps directly to a summary query.

SQL — Example Query
SELECT
    course_id,
    COUNT(*)        AS total_enrollments,
    COUNT(grade)    AS graded_students,
    AVG(grade)      AS avg_grade,
    MIN(grade)      AS lowest_grade,
    MAX(grade)      AS highest_grade
FROM enrollments
GROUP BY course_id
ORDER BY course_id;

Note the ORDER BY at the end. Reports usually need sorted output. ORDER BY sorts the final results and can sort by grouped columns or by aggregate results.

Example: Sort by enrollment count, largest first

SQL — Example Query
SELECT course_id, COUNT(*) AS enrollment_count
FROM enrollments
GROUP BY course_id
ORDER BY COUNT(*) DESC;
Report: Fine Summary per User, Excluding Zero Balances

A library billing report might want only users who currently owe fines.

SQL — Example Query
SELECT
    user_id,
    COUNT(*)            AS loans_with_fines,
    SUM(fine_amount)    AS total_owed
FROM loans
WHERE fine_amount > 0
GROUP BY user_id
ORDER BY SUM(fine_amount) DESC;

The WHERE fine_amount > 0 removes rows where the fine is zero before grouping. Then HAVING is not needed because the WHERE clause already excluded the zero-fine rows at the individual row level.

Efficiency Note

This approach is more efficient than using HAVING SUM(fine_amount) > 0 because rows are eliminated early. HAVING requires the database to form all groups first and then discard them. WHERE eliminates rows before grouping, which means fewer rows enter the grouping step.

Dashboard Statistic: Total Resources in the Catalog

Sometimes a dashboard needs a single headline number. This is just aggregation without grouping.

SQL — Example Query
SELECT COUNT(*) AS total_resources
FROM resources;

Or a set of headline numbers:

SQL — Example Query
SELECT
    COUNT(*)                        AS total_resources,
    COUNT(DISTINCT subject_id)      AS subjects_covered,
    MIN(published_year)             AS earliest_publication,
    MAX(published_year)             AS most_recent_publication
FROM resources;

This query returns one row with four summary columns, a typical input for a dashboard header.

Chapter Summary
  • This chapter introduced aggregation as the SQL skill that collapses many rows into meaningful summaries. The aggregate functions COUNT, SUM, AVG, MIN, and MAX each answer a different kind of summary question.
  • COUNT(*) counts rows. COUNT(column) counts non-null values. SUM totals numeric values. AVG computes the mean. MIN and MAX find the extremes.
  • GROUP BY divides rows into groups by one or more columns and applies aggregate functions within each group independently. Every column in the SELECT clause must either appear in the GROUP BY clause or be wrapped in an aggregate function, since groups collapse into single rows in the output.
  • HAVING filters groups after aggregation, while WHERE filters individual rows before aggregation. Both can appear in the same query, and each serves a distinct purpose.
  • The order of SQL operations, from FROM through WHERE, then GROUP BY, then HAVING, then SELECT and ORDER BY, explains why WHERE cannot reference aggregate functions and why HAVING can.
  • Aliases give aggregate columns readable names that appear clearly in reports and dashboards.
  • Summary queries are the foundation of most real reporting in information systems. When a teacher checks class averages, when a librarian reviews loan statistics, when an administrator compares enrollments by department, the underlying mechanism is an aggregate SQL query. Designing those queries well means understanding the question being asked, choosing the right rows, choosing the right grouping, and filtering the right things at the right stage.
Key Terms
aggregate function

A SQL function that computes a single result from a set of rows. The core aggregate functions are COUNT, SUM, AVG, MIN, and MAX.

aggregation

The process of combining many rows into one computed value or one row per group.

AVG

An aggregate function that returns the arithmetic mean of all non-null values in a numeric column.

COUNT(*)

An aggregate function that returns the number of rows in the result set, including rows with null values in other columns.

COUNT(column)

An aggregate function that returns the number of non-null values in a specific column.

COUNT(DISTINCT column)

An aggregate function that returns the number of distinct non-null values in a column.

GROUP BY

A SQL clause that divides rows into groups based on one or more column values, allowing aggregate functions to be applied within each group.

group filter

A condition applied after aggregation, using HAVING, that determines which groups appear in the final result.

HAVING

A SQL clause that filters groups after GROUP BY has been applied. It can reference aggregate functions. It runs after GROUP BY and before SELECT.

MAX

An aggregate function that returns the largest non-null value in a column.

MIN

An aggregate function that returns the smallest non-null value in a column.

row filter

A condition applied before aggregation, using WHERE, that determines which rows enter the grouping and aggregation steps.

SUM

An aggregate function that returns the total of all non-null values in a numeric column.

Common Misconceptions
Misconception 1
WHERE can filter on aggregate functions.

Students sometimes write WHERE COUNT(*) > 5 expecting it to filter groups. This does not work. WHERE runs before grouping. At the time WHERE executes, no groups have been formed yet, so no aggregate values exist. Use HAVING to filter on aggregate results.

Misconception 2
COUNT(column) and COUNT(*) always return the same number.

They return the same number only when the specified column has no null values. COUNT(*) counts all rows. COUNT(column) skips nulls. When a column has nulls, these two counts will differ. This distinction is important for data quality analysis.

Misconception 3
AVG automatically handles missing data fairly.

AVG excludes nulls. If some records have null values in the column being averaged, the denominator in the average is the number of non-null values, not the total number of rows. This means the average reflects only the records with data. Depending on the question being asked, this may or may not be the right answer.

Misconception 4
HAVING is just a second WHERE clause.

HAVING and WHERE look similar, but they run at different times and operate on different things. WHERE filters rows before aggregation. HAVING filters groups after aggregation. They are not interchangeable.

Misconception 5
GROUP BY sorts the result.

GROUP BY does not guarantee any particular sort order in the output. To sort aggregated results, you must add an ORDER BY clause explicitly.

Coming Next — Chapter 13: Joining Tables

Chapter 12 has shown how to summarize data within a single table. But real questions often involve data spread across multiple tables. How many resources has each author contributed? Which students are enrolled in courses taught by a specific instructor? What fines are associated with which resources and users?

Answering these questions requires joining tables. Chapter 13 introduces SQL joins as the mechanism for combining related data across tables. Students will learn INNER JOIN, join conditions, table aliases, one-to-many joins, many-to-many joins through junction tables, and the common mistakes that produce duplicate or missing rows. Joining is one of the most important skills in SQL, and it builds directly on the relational design and querying skills from earlier chapters.
