Chapter 4 — Front Matter

Purpose and Overview

Chapter 4 introduces one of the most important and surprising ideas in computer science: all digital information — every photograph, every song, every word, every number, every video, every program — is stored and processed as patterns of just two symbols. This chapter builds patient, visual understanding of binary representation. Students learn bits, bytes, binary counting, decimal-to-binary conversion, text encoding, image representation, sound representation, compression, and the limits of digital representation.

By the end of the chapter, you should understand why representation matters, how computers transform the world into numbers, and what happens when that transformation is imperfect. These ideas sit beneath every file, every format, every network transmission, and every hardware design in computing.

Opening Question

Your phone holds thousands of photographs, hundreds of songs, years of text messages, and every app you have installed — all stored on a chip the size of a fingernail. How? Every one of those photographs, songs, words, and programs has been translated into a long sequence of two symbols: 0 and 1. By the end of this chapter, you will understand exactly how that translation works, why two symbols are sufficient for everything, and what is gained and lost in the process.

Section 4.1

Why Representation Matters

Before a computer can store, process, or transmit anything — a name, a temperature reading, a photograph, a piece of music — it must answer a fundamental question: how do we turn this into something the machine can handle?

That process of turning information into a form a machine can work with is called representation. Every choice about representation has consequences. Some choices make storage smaller. Some make processing faster. Some lose information in the translation. Some preserve everything exactly. Some work for certain kinds of information and fail for others.

The Core Problem

Consider something simple: your name. Your name is a sequence of letters. But a computer does not know what a letter is. It only knows electrical signals — voltage present or voltage absent, current flowing or current not flowing, a magnetic domain pointing one way or another. To store your name, a computer needs a way to connect the human idea of a letter to a pattern of electrical states that the machine can record and retrieve.

That connection — from human meaning to machine-readable pattern and back again — is what representation systems provide.

Everything Is a Number

One of the central choices in digital computing is to reduce everything to numbers, and then to represent those numbers in binary form. Text becomes numbers. Colors become numbers. Musical tones become numbers. Video frames become numbers. Even programs themselves — the instructions that tell the computer what to do — are stored as numbers.

This is a remarkable commitment. It means the same physical hardware that stores a piece of music can also store a photograph or a legal document. The price of this generality is that the computer itself does not know what kind of information it is working with. A file full of numbers might be music or it might be a spreadsheet. The computer relies on software, file formats, and encoding standards to tell it what the numbers mean.

Section 4.2

Bits and Bytes

The Bit

The smallest unit of information in a digital computer is called a bit, short for binary digit. A bit can hold exactly one of two values: 0 or 1. By itself a bit is almost nothing. But bits are never used alone — they are organized into groups, and in groups they become enormously expressive.

Physical Bit Storage — Four Examples

Hard disk drive — A tiny magnetic region on a spinning disk, magnetized in one of two directions.

Solid-state drive (SSD) — A charge level stored in a flash memory cell.

RAM — A flip-flop circuit that can be set to either of two stable states.

Fiber-optic cable — A pulse of light present (1) or absent (0).

The Byte and Storage Units

Eight bits grouped together form a byte. The byte is the standard working unit for most computing purposes. Eight bits can represent 2⁸ = 256 different values — enough for all characters in simple text encoding systems. As computers handle more data, storage is measured in progressively larger units.

Estimating typical file sizes

Bits vs. Bytes — A Common Confusion

Internet speeds are advertised in bits per second (Mbps or Gbps). File sizes are measured in bytes (MB or GB). A 100 Mbps connection transfers about 12.5 MB per second, because there are 8 bits per byte. Always check whether a unit is bits (lowercase b) or bytes (uppercase B).

Section 4.3

Binary Numbers

Computers use binary — the base-2 number system — because it maps perfectly onto two-state physical hardware. The decimal system we use in everyday life is base-10: it uses ten symbols (0–9) and groups quantities in powers of ten. Binary uses base-2: it uses only two symbols (0 and 1) and groups quantities in powers of two. The logic is identical; only the base changes.

Four Key Patterns in Binary Counting

Pattern 1 — The rightmost digit alternates with every step: 0, 1, 0, 1 ...

Pattern 2 — Each column changes half as often as the column to its right. The twos place changes every 2 counts. The fours place every 4. The eights place every 8.

Pattern 3 — The maximum value of n bits is 2ⁿ − 1. Four bits: 0 to 15. Eight bits: 0 to 255. Sixteen bits: 0 to 65,535.

Pattern 4 — Adding one more bit doubles the number of representable values: 4 bits → 16 values; 5 bits → 32 values.

Worked Example 4.4.1
Adding 0101 + 0011 in binary
  • Setup 0101 (decimal 5) + 0011 (decimal 3) ------
  • Ones col1 + 1 = 10. Write 0, carry 1.
  • Twos col0 + 1 + 1 (carry) = 10. Write 0, carry 1.
  • Fours col1 + 0 + 1 (carry) = 10. Write 0, carry 1.
  • Eights col0 + 0 + 1 (carry) = 1. Write 1.
  • Result 1000 (decimal 8) ✓
Answer: 0101₂ + 0011₂ = 1000₂ = 8₁₀
Section 4.5

Decimal and Binary Conversion

Converting by hand builds intuition that a calculator cannot provide. Binary-to-decimal conversion uses the place-value table from Section 4.3. For decimal-to-binary, the most reliable method is repeated division by 2.

The Repeated Division Method

Divide the number by 2 repeatedly, collecting remainders. Read the remainders from bottom to top to obtain the binary representation.

Worked Example 4.5.1
Convert decimal 25 to binary
  • Divide 25 ÷ 2 = 12 remainder 1 12 ÷ 2 = 6 remainder 0 6 ÷ 2 = 3 remainder 0 3 ÷ 2 = 1 remainder 1 1 ÷ 2 = 0 remainder 1
  • Read upRemainders bottom to top: 1 1 0 0 1
  • Check16 + 8 + 0 + 0 + 1 = 25. ✓
Answer: 25₁₀ = 11001₂
Worked Example 4.5.2
Convert decimal 200 to binary
  • Divide 200 ÷ 2 = 100 remainder 0 100 ÷ 2 = 50 remainder 0 50 ÷ 2 = 25 remainder 0 25 ÷ 2 = 12 remainder 1 12 ÷ 2 = 6 remainder 0 6 ÷ 2 = 3 remainder 0 3 ÷ 2 = 1 remainder 1 1 ÷ 2 = 0 remainder 1
  • Read upRemainders bottom to top: 1100 1000
  • Check128 + 64 + 0 + 0 + 8 + 0 + 0 + 0 = 200. ✓
Answer: 200₁₀ = 11001000₂

Section 4.6

Representing Text

Text is made of characters: letters, digits, punctuation, spaces, and special symbols. A computer stores only numbers. The solution is a character encoding — a standardized table that maps characters to numbers. If both the program that writes the text and the program that reads the text use the same encoding table, the text displays correctly. If they use different tables, the characters appear wrong.

ASCII

The earliest widely-used encoding for English text is ASCII (American Standard Code for Information Interchange), developed in the early 1960s. ASCII assigns numbers from 0 to 127 to 128 characters: control codes, printable characters, digits, punctuation, and both cases of all 26 English letters.

Pattern: Uppercase vs. Lowercase

Uppercase A is 65; lowercase a is 97. The difference is exactly 32 for every letter. This is intentional — it makes case conversion simple for programs. The bit at position 5 (value 32) is the only difference between a capital letter and its lowercase version.

  • CharactersH → 72, i → 105, ! → 33

Answer: "Hi!" = 3 bytes = 0100 1000 0110 1001 0010 0001

ASCII's Limitation

ASCII was designed for American English. With only 128 values, it has no room for accented letters (é, ñ, ü), non-Latin alphabets (Arabic, Chinese, Cyrillic, Hindi), mathematical symbols, or emoji. As computing became global, this limitation became a serious problem.

Unicode and UTF-8

Unicode is a modern standard that assigns a unique number (called a code point) to every character in virtually every writing system on Earth — and to emoji, mathematical symbols, musical notation, and much more. As of recent editions, Unicode covers over 140,000 characters across 150+ scripts.

UTF-8 is the most common way of encoding Unicode text as bytes. It is backward-compatible with ASCII: the first 128 Unicode code points use the same numbers as ASCII and store in exactly one byte each. Characters outside the ASCII range use 2, 3, or 4 bytes. This is why most English text files remain small and fast to process in UTF-8 — they use one byte per character — while Chinese or Arabic text uses more bytes per character.

UTF-8 is the dominant encoding for the web, for most programming languages, and for most modern software. Encoding mismatches produce garbled text — question marks, boxes, or strange characters where letters should appear.

Section 4.7

Representing Images

A digital image is made of pixels — tiny colored squares arranged in a rectangular grid. (Pixel is short for "picture element.") If you zoom into a digital photograph far enough, you will see the individual pixels clearly. What looks like a smooth curve in a photograph is actually many tiny squares of slightly different colors arranged to approximate that curve.

An image's resolution is the number of pixels in its grid, expressed as width × height. A common display resolution is 1920 × 1080 pixels. A high-resolution smartphone photograph might be 4000 × 3000 pixels or larger.

RGB Color

The most common system for representing color digitally is RGB — Red, Green, Blue. Every pixel's color is described as a mixture of three components, each stored as an 8-bit number (0 to 255). This gives 256 × 256 × 256 = 16,777,216 possible colors per pixel. Three bytes per pixel is called 24-bit color or true color.

Alpha Channels and Transparency

Some formats store a fourth value per pixel — alpha — representing transparency. Alpha 0 = fully transparent; alpha 255 = fully opaque. PNG supports alpha channels, which is why it is used for logos and graphics with transparent backgrounds. JPEG does not support transparency, which is why JPEG images always have a solid background.

Section 4.8

Representing Sound

Sound is vibrations traveling through air, creating pressure waves that our ears detect. A microphone converts those pressure waves into a continuous electrical signal — a changing voltage that mirrors the shape of the sound wave. A continuous signal is called an analog signal. To store sound digitally, we must convert that continuous analog signal into a sequence of discrete numbers. This process is called analog-to-digital conversion (ADC).

Sampling

The key technique is sampling: measuring the amplitude (height) of the sound wave at regular time intervals and storing each measurement as a number. The sampling rate is how many samples are taken per second, measured in hertz (Hz) or kilohertz (kHz). The bit depth is how many bits store each sample — more bits mean more possible amplitude levels and finer detail.

The Nyquist Theorem

To accurately capture a sound frequency, the sampling rate must be at least twice that frequency. Human hearing extends to roughly 20,000 Hz. CD audio samples at 44,100 Hz — more than twice 20,000 — which is why CD audio captures everything the human ear can hear.

Section 4.9

Compression

Representing images, audio, and video using the methods described above produces very large files. A one-hour uncompressed HD video might require 500 GB or more. Compression reduces file size by finding smarter ways to represent data — using fewer bits without losing important information, or while accepting controlled losses.

Lossless Compression

Lossless compression shrinks a file so that when it is decompressed, the result is bit-for-bit identical to the original. Run-length encoding is one of the simplest techniques: instead of repeating the same value many times, the compressed file stores the value once with a count.

Worked Example 4.9.1
Run-length encoding
  • ProblemA row of pixels: 40 white, then 12 black, then 48 white. Storing every pixel individually requires 100 values.
  • Encoded (white, 40) (black, 12) (white, 48)
  • ResultThree pairs instead of 100 individual values. Dramatic saving for images with large flat areas.
Run-length encoding replaces repetition with (value, count) pairs.

Other lossless methods include dictionary compression (used in ZIP files), which finds repeated patterns and replaces them with short codes, and Huffman coding, which assigns shorter bit sequences to more frequent values. PNG images and ZIP files both use lossless compression. Text compression is always lossless — you cannot afford to lose any characters from a document.

Lossy Compression

Lossy compression achieves much higher compression ratios by discarding some information permanently. The decompressed file is not identical to the original — but if the compression is well-designed, the differences are difficult or impossible for humans to perceive.

JPEG exploits the fact that the human visual system is more sensitive to brightness than to color, and more sensitive to large-scale patterns than to fine high-frequency detail. It divides the image into 8×8 blocks, applies a mathematical transform to each block, rounds away the high-frequency detail, and compresses the result. At high quality settings, losses are nearly invisible. At low quality, the 8×8 block structure becomes visible — the "blockiness" or "JPEG artifacts" you may have noticed in heavily compressed images.

MP3 uses a similar strategy for audio, discarding sounds masked by louder simultaneous sounds, very quiet sounds, and sounds at the extremes of the audible frequency range.

Section 4.10

Limits, Errors, and Approximation in Digital Representation

Digital representation is powerful, but it is not perfect. Every representation system has limits, and understanding those limits is part of working carefully with computers.

Overflow

Each numeric type has a maximum value. If a calculation tries to exceed that maximum, the result wraps around or triggers an error — a problem called overflow. An 8-bit unsigned integer can hold values from 0 to 255. If you add 1 to 255 in a system without overflow protection, the result is 0, not 256.

Worked Example 4.10.1
8-bit overflow
  • Setup 255 = 1111 1111 (8 bits) + 1 = 0000 0001 ───────────────── 256 = 1 0000 0000 ← 9 bits
  • StorageOnly 8 bits are stored. The leading 1 is discarded.
  • Result Stored: 0000 0000 = 0
Adding 1 to the maximum 8-bit value produces 0, not 256. This is integer overflow.
Real-World Overflow: Ariane 5, 1996

The Ariane 5 rocket was destroyed shortly after launch because of an overflow error. A value that was safely within range in the Ariane 4 software exceeded the representable range in the Ariane 5's different hardware, causing the flight computer to crash and the rocket to self-destruct. Overflow errors have real consequences.

Floating-Point Approximation

Computers typically represent real numbers using floating-point representation. Floating-point numbers are stored in binary, which means they can represent fractions whose denominators are powers of 2 — like 1/2, 1/4, 1/8 — exactly. But fractions whose denominators are not powers of 2 — like 1/10 — cannot be stored exactly in binary, just as 1/3 cannot be stored exactly in decimal.

Worked Example 4.10.2
Floating-point arithmetic in Python
  • Expected0.1 + 0.2 should equal 0.3
  • Actual0.1 + 0.2 = 0.30000000000000004
  • WhyNeither 0.1 nor 0.2 has an exact binary representation. The tiny approximation errors accumulate in the addition.
  • ConsequenceThe comparison 0.1 + 0.2 == 0.3 may evaluate to False in most programming languages.
Floating-point arithmetic introduces tiny approximation errors. Financial software uses exact decimal arithmetic to avoid this.

Other Limits

Sampling limits: Digital audio captures sound at a finite number of samples per second. Frequencies above half the sampling rate cannot be represented accurately — they produce aliasing, where high-frequency content appears at incorrect lower frequencies. Similarly, fine image detail smaller than one pixel is simply lost.

Encoding errors: If a file is saved with one encoding (say, UTF-8) but opened by software expecting a different encoding (say, Windows-1252), the bytes are matched to the wrong characters, producing garbled text. Text files should always be labeled with their encoding.

Lossy compression artifacts: Once an image has been saved as JPEG at low quality, the discarded detail cannot be recovered. Medical imaging, scientific photography, and archival records should not use lossy compression.

Chapter 4 — Back Matter

Chapter Summary

  • Sec 4.1Representation is the translation layer between human meaning and machine processing. All digital information is reduced to numbers stored in binary, which is why the same hardware can store text, images, and programs.
  • Sec 4.2A bit holds 0 or 1. Eight bits form a byte. Storage is measured in KB, MB, GB, TB, PB. Internet speeds use bits per second; file sizes use bytes — pay attention to the units.
  • Sec 4.3Binary uses base-2 place values: 1, 2, 4, 8, 16, 32, 64, 128 ... To read binary, sum the powers of 2 for every position containing a 1.
  • Sec 4.4Counting in binary follows four key patterns. Adding in binary uses carry logic at the threshold of 2 instead of 10. More bits means exponentially more representable values: adding one bit doubles the count.
  • Sec 4.5To convert decimal to binary, divide by 2 repeatedly and read remainders upward. To convert binary to decimal, sum the place values for positions containing a 1.
  • Sec 4.6Text is stored using character encoding tables. ASCII covers 128 characters for American English. Unicode covers every writing system on Earth. UTF-8 encodes Unicode as bytes, backward-compatible with ASCII. Encoding mismatches produce garbled text.
  • Sec 4.7Images are grids of pixels. Each pixel's color is stored as three bytes in RGB: one each for red, green, and blue intensity (0–255). Uncompressed size = width × height × 3 bytes. Alpha channels add transparency.
  • Sec 4.8Sound is digitized by sampling the amplitude of the analog wave at regular intervals. CD audio samples at 44,100 Hz with 16-bit depth. Uncompressed audio is large; MP3 uses lossy compression to reduce size.
  • Sec 4.9Lossless compression (ZIP, PNG, FLAC) allows perfect reconstruction. Lossy compression (JPEG, MP3) permanently discards information for higher compression ratios. Lossy is never appropriate for text, code, or archival records.
  • Sec 4.10Digital representation has real limits: overflow wraps values that exceed the maximum; floating-point cannot exactly represent many decimal fractions; lossy compression is permanent; encoding mismatches corrupt text; high-frequency detail is lost at low sampling rates.

Key Terms Review

bit
The smallest unit of digital information. Holds exactly one of two values: 0 or 1.
byte
A group of eight bits. The standard unit of storage in most computing contexts. Can represent 256 different values (2⁸).
binary
The base-2 number system, using only the digits 0 and 1. Used internally by all digital computers.
representation
The process of encoding information (text, images, sound) in a form a computer can store and process.
ASCII
American Standard Code for Information Interchange. A 7-bit encoding of 128 characters for American English, developed in the 1960s.
Unicode
A universal standard assigning unique numbers to every character in every writing system on Earth. Over 140,000 characters across 150+ scripts.
UTF-8
The dominant variable-length encoding for Unicode text. Backward-compatible with ASCII. Used on most of the web.
code point
The number assigned to a specific character in Unicode.
character encoding
A standardized mapping between characters and numeric values that allows computers to represent text.
pixel
The smallest unit of a digital image. A tiny colored square in a rectangular grid.
resolution
The number of pixels in a digital image, expressed as width × height (e.g., 1920 × 1080).
RGB
Red, Green, Blue. The standard system for representing pixel colors using three values (0–255 each) for the intensity of each color channel.
alpha channel
A fourth per-pixel value representing transparency, used alongside RGB in formats like PNG.
analog signal
A continuous signal that varies smoothly, such as a sound wave or the output of a microphone.
sampling
Measuring the amplitude of an analog signal at regular intervals to produce a sequence of digital values.
sampling rate
The number of samples taken per second during digitization, measured in Hz or kHz. CD audio: 44,100 Hz.
bit depth
The number of bits used to store each audio sample. Higher bit depth means more amplitude levels and less quantization error.
compression
A technique for reducing the number of bits needed to represent data, either losslessly or with acceptable losses.
lossless compression
Compression from which the original data can be perfectly reconstructed. Used for text, code, and archival purposes.
lossy compression
Compression that permanently discards some data for higher compression ratios. Used for photographs (JPEG) and audio (MP3).
run-length encoding
A lossless compression technique that replaces runs of repeated values with a (value, count) pair.
overflow
An error that occurs when a calculated value exceeds the maximum representable value for the bit width being used.
floating-point
A binary method for representing real numbers. Cannot exactly represent many decimal fractions, introducing tiny approximation errors.
aliasing
The distortion that results when a signal is sampled too slowly to represent its high-frequency content accurately.
