Chapter 18 — Front Matter

Chapter Purpose and Overview

Every time you search for a product online, check your grade in a school portal, look up a library book, withdraw money from an ATM, or log into a streaming service, a database is running behind the scenes. Databases are organized systems for storing, retrieving, updating, and protecting data. They are one of the most important technologies in modern computing — and yet, from the user's perspective, they are completely invisible.

This chapter introduces databases as systems, not merely as storage containers. You will learn how data is organized into tables, how rows and columns represent real-world records, how keys connect related information, how queries retrieve exactly the data you need, how SQL expresses those queries in a precise language, and how database management systems ensure that data stays consistent and reliable even when thousands of users access it at once.

The chapter also introduces information systems — the larger organizational structures that combine databases, software, people, and processes to support decision-making in schools, hospitals, banks, governments, stores, libraries, and every other institution that manages information.

Opening Question

When a student logs into a school portal and sees their grade for last week's assignment, what happened between clicking the login button and seeing that grade appear on screen? The answer involves a database receiving a query, searching millions of records in a fraction of a second, returning exactly the right information, and sending it to the browser — all without the student seeing any of it. Understanding that system is what this chapter is about.

Section 18.1

Why Databases Matter

Before databases existed, organizations stored information in filing cabinets, paper ledgers, and card catalogs. A hospital might keep patient records in one cabinet, billing records in another, and appointment schedules in a third. Finding all the information about one patient required searching multiple places. Updating an address meant changing it in every location where it appeared. Mistakes were common. Information got lost.

Early computer programs tried to solve this by storing data in files — text files, spreadsheet files, or custom binary files. This was better than paper, but it still created problems. Each program managed its own files in its own format. Two programs that needed to share data had to copy information back and forth, which caused inconsistencies. If a student's address changed, the admissions program might be updated but the billing program might not. The same data existed in multiple places with different values. This problem is called data redundancy — the same information stored in multiple places in ways that can become contradictory.

A database solves these problems by providing a single organized system for storing and accessing data. Instead of scattered files managed by individual programs, a database creates a central repository with consistent structure, controlled access, and reliable retrieval.

What a Database Is

A database is an organized collection of related data. The data is structured so that it can be stored, searched, updated, and retrieved efficiently. A database is not just a pile of information — it is information arranged according to a design that reflects how the information relates to the real world.

A database management system, or DBMS, is the software that manages the database. The DBMS handles storing the data, executing queries, controlling access, maintaining consistency, and protecting the data from errors or corruption. Users and programs interact with the DBMS rather than directly with the raw data files. Together, the database and the DBMS form a database system.

Why Organization Matters

Disorganized data is almost as useless as no data at all. If a student named "James Smith" appears twice with different GPAs, which is correct? A well-designed database eliminates this by storing each piece of information exactly once and connecting pieces together through relationships — which is what the rest of this chapter explains.

Section 18.2

Tables, Records, and Fields

The most common type of database in business, education, and web development is the relational database. A relational database organizes information into tables that are connected by relationships. The concepts of tables, records, and fields are the foundation of everything that follows.

A table is a structured collection of related data organized into rows and columns. Each table represents one type of entity — one kind of thing the database stores information about. A school database might have a Students table, a Courses table, a Teachers table, and an Enrollments table. You would not mix student information and course information in the same table because they are different kinds of entities with different properties.

Records and Fields

Each row in a table is called a record. A record represents one instance of the entity the table describes. In a Students table, each row is one student. Each column is called a field. A field represents one property or attribute of the entity.

Data Types and Constraints

Every field has a data type — the kind of value it is allowed to contain. Data types include Integer (whole numbers), Decimal or Float (numbers with decimal points), Text or String (character sequences), Date, and Boolean (true/false). Data types matter because they determine what operations are valid. You can calculate an average GPA because it is a number. You can sort LastName alphabetically because it is text.

Constraints are rules that the database enforces automatically. Common constraints include NOT NULL (a field must have a value), UNIQUE (no two records can share the same value in this field), and range constraints (GradeLevel might only permit values 9 through 12). Constraints enforce data quality without relying on whoever enters the data to remember the rules.

Database vs. Spreadsheet

A spreadsheet allows any value in any cell with little enforcement. A database table enforces consistent structure: every record has exactly the same fields, every field has a defined type, and constraints prevent illegal values. This strict structure is what makes reliable querying across millions of records possible.

Section 18.3

Keys and Identifiers

One of the most important design decisions in a database is choosing how to uniquely identify each record. In a table of students, two students might have the same first name, or even the same full name. If you search for "James Smith," how does the database know which James Smith you mean? The solution is a key — a field whose value uniquely identifies each record.

Primary Keys

A primary key is a field whose value is unique for every record and never changes. In the Students table, StudentID is the primary key. Every student has a different StudentID. Even if two students are both named Maria Garcia, they will have different StudentIDs — perhaps 1001 and 2047 — and the database can distinguish them precisely.

Properties of a Good Primary Key

• Unique — No two records share the same primary key value.

• Never null — Every record must have a primary key value.

• Stable — The primary key does not change after it is assigned.

• Simple — A single field is preferred over a combination of fields.

Natural Keys vs. Surrogate Keys

A natural key is drawn from real-world data that naturally identifies the entity — for example, an ISBN for a book. A surrogate key is an artificial identifier created by the database, such as StudentID: 1001. Surrogate keys are often preferred because real-world identifiers can change (a person's name, email, or phone number) while surrogate keys never need to be updated. Once 1001 is assigned to Maria Garcia, it stays with her record permanently.

Foreign Keys and Referential Integrity

A foreign key is a field in one table that refers to the primary key of another table. Foreign keys create connections between tables. Consider a Courses table with a TeacherID field — TeacherID is a foreign key that references the TeacherID primary key in the Teachers table. Each course record points to the teacher who teaches it.

Referential integrity is the rule that every foreign key value must match a real primary key in the referenced table. If a foreign key in the Enrollments table contains StudentID 9999, but there is no student with ID 9999 in the Students table, the DBMS rejects that record. This prevents orphaned records — records that refer to things that do not exist.

Section 18.4

Relationships Between Tables

The power of a relational database comes from relationships — the connections between tables that allow you to combine information from multiple sources in a single query. There are three fundamental types of relationships.

One-to-Many

A one-to-many relationship means that one record in Table A can be associated with many records in Table B, but each record in Table B is associated with only one record in Table A. One teacher can teach many courses, but each course has only one teacher. The foreign key lives in the "many" side of the relationship — TeacherID lives in the Courses table, not the Teachers table.

Many-to-Many

A many-to-many relationship means that records in each of two tables can be associated with multiple records in the other. One student can enroll in many courses, and one course can have many students enrolled. Many-to-many relationships cannot be stored directly with two tables alone. They require a junction table — a linking table that sits between the two main tables and holds foreign keys to both.

Many-to-Many: Students ↔ Courses through Enrollments
LoanID (PK)StudentID (FK)CourseID (FK)
E0011001C101
E0021001C102
E0031002C101
E0041003C103

The Enrollments table is the junction table. Each row records one enrollment — one student in one course. Student 1001 (Maria) is enrolled in both C101 and C102. Course C101 has both student 1001 and 1002. By searching this table, you can find all courses a student is in, or all students in a course.

One-to-One

A one-to-one relationship means each record in Table A is associated with exactly one record in Table B, and vice versa. These are less common and are sometimes used to split a large table for performance, or to separate sensitive information from general information.

Entity-Relationship Diagrams

Database designers use Entity-Relationship Diagrams (ER Diagrams) to plan a database's structure visually before writing any code. An ER Diagram shows the tables (entities), their fields (attributes), primary keys, and the relationships between tables. Reading ER Diagrams is a foundational skill for any database work.

Section 18.5

Queries

A query is a request for data from a database. Queries are the primary way that users and programs interact with a database. A query specifies what data you want, where to find it, what conditions it must meet, and how to organize the results.

What Queries Do

OperationWhat It DoesPlain-English Example
SelectionFilter which rows are returned based on a condition"Give me all students in grade 11."
ProjectionChoose which columns appear in the result"Show me only first name and last name."
JoiningCombine rows from multiple tables into one result"Show each student's name and their courses."
AggregationCompute statistics across records"What is the average GPA of 10th graders?"
SortingOrder results by one or more fields"List all students alphabetically by last name."
InsertionAdd new records to a table"Add a new student: Leon Mbeki, grade 12."
UpdateModify existing records"Change Maria Garcia's GPA to 3.30."
DeletionRemove records from a table"Remove the record for student ID 1004."

The result of a retrieval query is a result set — a temporary table containing the records and fields that match the query. This result set is shown to the user or processed by a program. It is a view of the data, not a stored copy — querying does not change what is in the database.

Every time you search for a product, look up your grades, see a recommended list, or log in to a personalized homepage, a query is executing behind the scenes. The web application sends a query to the database, receives a result set, and uses that data to build what you see on screen.

Section 18.6

SQL at an Introductory Level

SQL — pronounced either "S-Q-L" or "sequel" — stands for Structured Query Language. SQL is the standard language for communicating with relational databases. It is used to create tables, insert data, query data, update records, and delete records.

SQL is a declarative language — you describe what you want, and the database figures out how to retrieve it. You do not write step-by-step loops; you state your request, and the DBMS executes it. SQL is one of the most widely used languages in computing, and learning its basics gives you a skill that applies across dozens of database systems.

SELECT — Retrieving Data

The SELECT statement retrieves data. Its basic form specifies which columns to return and which table to read from. To retrieve all columns, use * (asterisk, meaning "all").

SELECT: Retrieve first and last names of all students
SELECT FirstName, LastName
FROM Students;

The WHERE clause filters which rows are included. It supports comparison operators =, <, >, <=, >=, <> (not equal), and logical operators AND, OR, and NOT.

WHERE: Grade 11 students with GPA above 3.0

JOIN — Combining Tables

The JOIN clause combines rows from two tables based on a related column. This is the feature that makes relational databases powerful — it allows you to retrieve connected information from multiple tables in one query.

JOIN: Each student's name and their enrolled courses
SELECT Students.FirstName, Students.LastName, Courses.CourseName
FROM Students
JOIN Enrollments ON Students.StudentID = Enrollments.StudentID
JOIN Courses ON Enrollments.CourseID = Courses.CourseID
ORDER BY Students.LastName;

INSERT, UPDATE, DELETE

Modifying Data
-- Add a new student record
INSERT INTO Students (StudentID, FirstName, LastName, GradeLevel, GPA)
VALUES (1006, 'Leon', 'Mbeki', 12, 3.65);

-- Update an existing record
UPDATE Students
SET GPA = 3.30
WHERE StudentID = 1001;

-- Remove a record
DELETE FROM Students
WHERE StudentID = 1004;
Critical Warning — Always Use WHERE with UPDATE and DELETE

Running UPDATE Students SET GPA = 4.0 without a WHERE clause sets every student's GPA to 4.0. Running DELETE FROM Students without a WHERE clause deletes every student. Always double-check that your WHERE condition is correct before running UPDATE or DELETE.

Section 18.7

Database Management Systems

A database management system (DBMS) is the software layer that sits between users or applications and the raw data stored on disk. Users and programs never interact directly with data files; they always go through the DBMS, which handles all the complexity of actually storing and retrieving data.

What a DBMS Does

ResponsibilityWhat the DBMS Handles
Storage managementOrganizes data on disk in efficient structures; maintains indexes for fast retrieval
Query processingParses SQL, creates an execution plan, runs the query, returns the result set
Access controlEnforces permissions — read, write, admin — for different users and roles
Concurrency controlCoordinates simultaneous access by multiple users without conflicts
Transaction managementEnsures groups of operations either fully succeed or fully cancel
Integrity enforcementEnforces data types, constraints, primary keys, foreign keys, referential integrity
RecoveryRestores a consistent state after a crash, without leaving data partially written

Common Database Systems

Relational DBMSs (RDBMS) store data in tables with SQL as the query language. They are the most common type for business and web applications. Major relational database systems include MySQL (widely used for web applications; open-source), PostgreSQL (powerful open-source system known for standards compliance), SQLite (a lightweight single-file database common in mobile apps and small applications), Microsoft SQL Server (enterprise RDBMS used in many business systems), and Oracle Database (used in large organizations).

NoSQL DBMSs store data in formats other than tables — documents, key-value pairs, graphs, or wide columns. They are used when data does not fit well into a tabular structure or when extreme scale is required. Examples include MongoDB (document store) and Redis (key-value store). NoSQL systems are beyond the scope of this chapter, but they are important to know exist.

Section 18.8

Transactions and Consistency

One of the most important and subtle problems in database design is ensuring that the data remains correct when multiple operations happen together — especially when something goes wrong in the middle.

The Bank Transfer Problem

The Problem Without Transactions
A bank transfers $500 from checking to savings: two operations required
  • Step 1Subtract $500 from the checking account balance
  • Step 2Add $500 to the savings account balance
  • ProblemWhat if the system crashes after Step 1 but before Step 2? Checking has been debited $500, but savings has not been credited. $500 has disappeared.
  • ConclusionBoth operations must either both succeed or both fail. There is no acceptable half-completed state. This is what a transaction enforces.

Transactions: Commit and Rollback

A transaction is a group of database operations that are treated as a single unit. A transaction either commits — all operations complete successfully and changes are permanently saved — or rolls back — all operations are cancelled and the database returns to its state before the transaction began.

The Bank Transfer as a Transaction
BEGIN TRANSACTION;

UPDATE Accounts
SET Balance = Balance - 500
WHERE AccountID = 'CHK-001';

UPDATE Accounts
SET Balance = Balance + 500
WHERE AccountID = 'SAV-001';

COMMIT;  -- makes all changes permanent

If the system crashes between the two UPDATE statements, the DBMS rolls back the transaction on recovery, undoing the first UPDATE. The database returns to the pre-transfer state. No money is lost or created.

Transaction Lifecycle
  BEGIN TRANSACTION
         │
         ▼
    [Operation 1] ──FAIL──► ROLL BACK
         │                  (database returns to
         ▼                   pre-transaction state)
    [Operation 2] ──FAIL──► ROLL BACK
         │
         ▼
    [Operation N]
         │
         ▼
       COMMIT
  (all changes permanent)

The ACID Properties

Database transactions are designed to guarantee four properties, summarized by the acronym ACID.

A
Atomicity
All operations in a transaction succeed, or none do. There is no partial execution. (From "atom" — indivisible.)
C
Consistency
A transaction takes the database from one valid state to another. All constraints and relationships remain intact.
I
Isolation
Concurrent transactions behave as if they run one at a time. One transaction cannot see the incomplete state of another.
D
Durability
Once a transaction commits, its changes are permanent — even if the system crashes immediately after the commit.
Worked Example 18.2
Race condition: Two customers, one last seat
  • Without isolationCustomer A reads: seats available = 1. Customer B reads: seats available = 1. Both proceed. Both book. Result: two bookings for one seat. Double-booked.
  • With isolationCustomer A's transaction begins and locks the row. Customer B's transaction must wait. Customer A commits. Seats = 0 is now permanent. Customer B reads: seats = 0. Condition fails. Customer B is told no seats are available.
  • ResultOne booking for one seat. No double-booking. Isolation prevents the race condition.
Isolation: concurrent transactions cannot interfere with each other's intermediate state.

Section 18.9

Backups and Reliability

Even a perfectly designed database with ACID transactions can lose data if hardware fails, software has bugs, a fire destroys a data center, or a ransomware attack encrypts the data files. Backups are copies of the database that can be used to restore data if the original is lost or corrupted. ACID guarantees consistency within a running system; backups guarantee survival when the system itself is destroyed.

Types of Backups

Backup TypeWhat It CopiesAdvantageDisadvantage
Full backupThe entire database at a point in timeSimple to restore — one copy is all you needSlow and storage-intensive for large databases
Incremental backupOnly data changed since the last backupFast, small filesRestoration requires last full backup plus every incremental since
Differential backupAll data changed since the last full backupGrows over time but simpler to restore than incrementalGrows larger the longer you wait between full backups
Transaction log backupThe record of every transaction, continuouslyAllows recovery to almost any point in timeRequires more storage and management

Key Reliability Measures

Organizations must decide how often to take backups, how long to keep them, and where to store them. Storing backups on the same server as the database is dangerous — if the server fails, the backup fails too. Reliable systems store backups in separate physical locations or in cloud storage.

Two key measures guide backup strategy. Recovery Point Objective (RPO) is how much data is acceptable to lose in the worst case — if you back up every hour, you could lose up to one hour of data. Recovery Time Objective (RTO) is how long it takes to restore service after a failure.

Replication goes beyond backups by maintaining one or more real-time copies of the database on separate servers. If the primary server fails, a replica takes over immediately, minimizing downtime. High-reliability database systems — used by banks, hospitals, airlines, and large web services — combine ACID-compliant transactions, multiple levels of backup, geographic replication, and regular recovery drills to verify that backups actually work.

Testing Backups

Making backups is not enough. Organizations must regularly test that their backups can actually be restored. A backup that cannot be restored is not a backup — it is a false sense of security. Discovering a backup is corrupted during an actual disaster is far worse than discovering it during a routine test.

Section 18.10

Information Systems in Organizations

A database is a technical system — it stores data and answers queries. But databases do not exist in isolation. They are embedded in information systems — larger systems that combine databases, software applications, user interfaces, networks, people, processes, and policies to support the work of organizations.

An information system is more than software. It includes data (the information itself), hardware (servers, computers, storage), software (the DBMS, application programs, interfaces), people (users, administrators, developers, decision-makers), processes (the workflows and procedures that govern how information is collected and used), and policies (rules about who can access what, how long data is retained, and what can be done with it).

Types of Information Systems

TypePurposeExample
Transaction Processing System (TPS)Records day-to-day operations as they happenPoint-of-sale system; bank ATM transactions; hospital patient check-in

Database Administrators

Database administrators (DBAs) are the specialists who design, maintain, and protect database systems. Their responsibilities include designing database schemas, writing and optimizing queries, managing user permissions, monitoring performance, planning backup and recovery procedures, applying security patches, and ensuring compliance with data privacy regulations. Database administration is a specialized career path within computing, covered further in Chapter 28.

Data Governance and Policy

Organizations must make decisions about how their data is governed. Who is allowed to access which tables and fields? How long is data retained before deletion? What regulations apply — for example, student records are governed by FERPA in the United States, and medical records are governed by HIPAA. How is sensitive data (Social Security numbers, medical diagnoses, financial details) protected? Who is responsible when data is lost, stolen, or misused?

These are not purely technical questions — they are legal, ethical, and organizational questions. The database is the technical system; data governance is the human system that determines how the database is used responsibly. Understanding data governance connects directly to Chapter 22 (Privacy, Identity, and Digital Safety) and Chapter 28 (Ethics, Society, Careers, and the Future of Computing).

Chapter 18 — Back Matter

Chapter Summary

  • Sec 18.1Databases solve the problems of data redundancy and scattered files by providing a single organized system for storing and accessing data. A DBMS is the software that manages the database. Databases power nearly every digital system you use.
  • Sec 18.2A relational database organizes data into tables. Each row is a record representing one instance; each column is a field representing one attribute. Fields have data types. Constraints enforce data quality automatically.
  • Sec 18.3A primary key uniquely identifies each record and never changes. A foreign key in one table references the primary key of another, creating a connection. Referential integrity ensures every foreign key points to a real record.
  • Sec 18.4Tables are connected through relationships: one-to-many (one teacher, many courses), many-to-many (students and courses, through a junction table), and one-to-one. ER Diagrams plan database structure visually.
  • Sec 18.5A query is a request for data. Queries filter rows (selection), choose columns (projection), combine tables (joining), compute statistics (aggregation), and sort results. The output is a result set — a temporary view of the data.
  • Sec 18.6SQL is the standard language for relational databases. SELECT retrieves data; WHERE filters rows; ORDER BY sorts results; aggregate functions compute statistics; GROUP BY aggregates by category; JOIN combines tables; INSERT, UPDATE, and DELETE modify data.
  • Sec 18.7A DBMS handles storage, query processing, access control, concurrency, transaction management, integrity enforcement, and recovery. Major relational systems include MySQL, PostgreSQL, SQLite, SQL Server, and Oracle.
  • Sec 18.8A transaction groups operations into a unit that either commits (all succeed, changes permanent) or rolls back (all cancelled, database returns to previous state). ACID properties — Atomicity, Consistency, Isolation, Durability — ensure transaction reliability.
  • Sec 18.9Backups create copies that allow recovery after failure. Types include full, incremental, differential, and transaction log. RPO and RTO measure acceptable data loss and recovery time. Replication maintains real-time synchronized copies on separate servers.
  • Sec 18.10Information systems combine databases with software, hardware, people, processes, and policies to support organizations. Types include TPS, MIS, DSS, CRM, and ERP. DBAs design and protect database systems. Data governance determines how organizations manage access, retention, and responsible use.

Key Terms Review

ACID
The four properties of reliable transactions: Atomicity, Consistency, Isolation, and Durability.
aggregate function
A SQL function that computes a summary value across records: COUNT, AVG, SUM, MAX, MIN.
atomicity
All operations in a transaction succeed, or none do. No partial execution.
backup
A copy of database data made so it can be restored if the original is lost or corrupted.
commit
The action that makes a transaction's changes permanent in the database.
consistency
A transaction takes the database from one valid state to another; all constraints remain intact.
constraint
A rule enforced by the database on field values, such as NOT NULL, UNIQUE, or data type limits.
data governance
Organizational policies determining how data is accessed, retained, protected, and used responsibly.
data redundancy
The same information stored in multiple places, which can lead to inconsistencies.
data type
The kind of value a field is permitted to hold: integer, decimal, text, date, boolean, etc.
database
An organized collection of related data structured for efficient storage, retrieval, and management.
database administrator
A specialist who designs, maintains, optimizes, and protects database systems.
DBMS
Database management system — software that manages the database, handling storage, queries, concurrency, integrity, and recovery.
durability
Committed changes survive system crashes and are permanently stored.
ER Diagram
Entity-Relationship Diagram — a visual plan of a database's tables, fields, and relationships.
field
A column in a database table representing one attribute of the entity being described.
foreign key
A field in one table that references the primary key of another table, creating a relationship.
information system
A system combining database, software, hardware, people, processes, and policies to collect, store, and distribute information for an organization.
isolation
Concurrent transactions behave as if they run one at a time; no transaction sees another's incomplete state.
JOIN
A SQL clause that combines rows from two or more tables based on a related column.
junction table
A table used to implement a many-to-many relationship by holding foreign keys to both related tables.
many-to-many
A relationship where records in each of two tables can be associated with multiple records in the other.
one-to-many
A relationship where one record in Table A can be associated with many records in Table B, but not vice versa.
primary key
A field that uniquely identifies each record in a table and never changes.
query
A request for data from a database specifying what to retrieve, from where, under what conditions, and in what order.
record
A row in a database table representing one instance of the entity being stored.
referential integrity
The rule that every foreign key value must match a valid primary key in the referenced table.
relational database
A database organized into tables connected by relationships, queried using SQL.
replication
Maintaining synchronized copies of a database on separate servers for reliability and performance.
result set
The temporary table of records returned by a query; a view of the data, not a stored copy.
roll back
Cancelling all operations in a transaction and returning the database to its previous state.
SQL
Structured Query Language — the standard language for querying and manipulating data in relational databases.
surrogate key
An artificial primary key created by the database that has no meaning outside the database.
table
A structured collection of related data organized into rows and columns in a relational database.
transaction
A group of database operations treated as a single unit that either fully commits or fully rolls back.
