Chapter 11 — Front Matter

Purpose and Overview

Every program works with data. A music app tracks thousands of songs. A school system stores tens of thousands of student records. A navigation app holds the coordinates of millions of roads. In each case, the program does not just store data — it organizes data so that the right piece can be found, updated, added, or removed quickly and reliably. The way a program organizes its data is called a data structure.

Part II of this textbook introduced you to variables that hold a single value — a number, a string, a Boolean. But most real programs deal with collections of values. A playlist is a collection of song titles. A gradebook is a collection of scores. A contact list is a collection of name-and-phone-number pairs. Each kind of collection calls for a different organizational strategy, and choosing the right strategy is one of the central skills of computer science.

This chapter introduces the most important data structures you will encounter as a beginning computer scientist: lists and arrays, strings as sequences, records, dictionaries and maps, stacks, and queues. The goal is not to make you an expert programmer of every structure — that depth belongs in Data Structures and Algorithms. The goal is to give you a clear mental model of each structure, a sense of what problems it solves, and enough familiarity to read, trace, and reason about code that uses them.

Opening Question

A teacher needs to store the names of every student in a class, then add a new student who enrolled late, then find and remove a student who transferred out. Which way of organizing the names would make each of these tasks easiest? A plain numbered list? A sorted alphabetical list? A special collection that supports fast search? By the end of this chapter you will know how to think through exactly these kinds of decisions — and why the structure you choose shapes every operation that follows.

Section 11.1

Why Data Structures Matter

Imagine you need to store the names of all 300 students at a school. You could create 300 separate variables — student1, student2, and so on all the way to student300. But this quickly becomes unmanageable. What if you need to find a student's name? You would have to check all 300 variables one by one. What if you need to add a student? You have to invent a new variable. What if you need to sort the names alphabetically? You would be facing hundreds of separate operations with no clear way to loop through them.

The problem is not the data itself — it is the organization. A data structure is a way of arranging and storing data in a computer's memory so that it can be used efficiently. The right data structure makes certain operations fast and easy; the wrong one makes the same operations slow and error-prone. Choosing a data structure is one of the most consequential decisions a programmer makes.

Data structures matter for several practical reasons. First, they determine which operations are natural. A list makes it easy to go through items one by one in order. A dictionary makes it easy to look up a value by name. A stack makes it easy to undo the last action. Second, data structures determine efficiency. Some structures find items quickly; others find them slowly. Some add items quickly; others require rearranging everything. As programs grow large — handling millions of records, responding in milliseconds — the choice of structure becomes critical. Third, data structures shape code. Programs built around well-chosen structures are readable, maintainable, and easier to debug. Programs built around poorly chosen structures become tangled.

Core Idea

A data structure is an organized way to store and access a collection of data. The right structure for a problem depends on what operations the program needs to perform on the data — not just how much data there is.

Connection to Earlier Chapters

In Chapter 7 you learned about variables — containers that hold a single value. In Chapter 9 you used lists briefly to pass collections of items around in functions. This chapter gives those ideas a proper foundation. Every data structure is ultimately a way to extend the single-variable idea to collections of values.

Section 11.2

Lists and Sequences

The most fundamental data structure for an ordered collection is a list. A list is a sequence of items arranged in a specific order. The order matters: the first item is distinct from the second, which is distinct from the third, and so on. Items in a list are also called elements.

A list can hold any kind of item: numbers, strings, Boolean values, or even other lists. In Python, a list is written using square brackets, with items separated by commas.

Example 11.2.1
Creating lists in Python
  • Code scores = [88, 95, 71, 100, 83] names = ["Alice", "Bob", "Carmen", "David"] mixed = [42, "hello", True, 3.14] empty = []
  • Explainscores holds five integers. names holds four strings. mixed holds items of different types. empty is a valid list with no items.

Mutability

Python lists are mutable — you can change their contents after they are created. You can add items, remove items, and replace items. This is different from strings, which are immutable (you cannot change individual characters in place). Mutability matters because it affects what your program can do with data after creating it.

Section 11.3

Indexes

The index of an element is its position within the list. Computer science counts positions starting at zero, not one. This is called zero-based indexing, and it is a convention used across virtually all modern programming languages. If a list has five elements, their indexes are 0, 1, 2, 3, and 4. There is no index 5.

Zero-based indexing exists for a mathematical reason rooted in how memory addresses are calculated, but for now the rule is simply: the first element is at index 0, the second at index 1, and so on. The last element is at index (length − 1).

Example 11.3.1
Reading and writing by index
  • Setupcolors = ["red", "green", "blue", "yellow"]
  • Read colors[0] → "red" colors[2] → "blue" colors[3] → "yellow"
  • Write colors[1] = "lime" # Now: ["red", "lime", "blue", "yellow"]
  • Lengthlen(colors) → 4
Indexes run from 0 to len(list) − 1. Assignment by index replaces the existing element.

Index Errors

If you ask for an index that does not exist, the program raises an index error — a runtime error that stops execution. For a list of five elements (indexes 0–4), requesting index 5 or index -6 will cause an error. Always be aware of the valid range of indexes before accessing a list.

Off-By-One Errors

One of the most common mistakes with lists is the off-by-one error — accessing index (length) instead of (length − 1). If scores has 5 elements, valid indexes are 0–4. Writing scores[5] causes an IndexError at runtime. Always double-check that your index does not exceed (length − 1).

Negative Indexing

Python allows negative indexes as a shorthand for counting from the end. Index -1 refers to the last element, -2 to the second-to-last, and so on. This is a convenience feature: colors[-1] is the same as colors[len(colors) - 1].

Slicing at an Introductory Level

A slice extracts a contiguous portion of a list and returns it as a new list. In Python, the syntax is list[start:stop], where start is the index of the first element to include and stop is the index of the first element to exclude. Slicing never modifies the original list.

Example 11.3.2
Slicing a list
  • Setupnums = [10, 20, 30, 40, 50, 60]
  • Slices nums[1:4] → [20, 30, 40] # indexes 1, 2, 3 nums[0:3] → [10, 20, 30] # indexes 0, 1, 2 nums[3:] → [40, 50, 60] # from index 3 to end nums[:2] → [10, 20] # from start to index 1
The stop index is excluded. Omitting start means "from the beginning"; omitting stop means "to the end."

Section 11.4

Arrays at an Introductory Level

In many textbooks and programming languages, the word array refers to a more specific kind of ordered collection than a general list. Understanding the distinction helps you read documentation, understand how computers actually store data, and prepare for Data Structures and Algorithms.

A traditional array has three characteristics that distinguish it from a Python list. First, an array has a fixed size: you specify how many elements it will hold when you create it, and that size cannot change. If you need more space, you must create a new, larger array. Second, every element in an array must be the same type: an array of integers holds only integers, an array of characters holds only characters. Third, elements are stored in contiguous memory locations — they sit right next to each other in the computer's memory, with no gaps.

This contiguous, fixed-size layout gives arrays a significant advantage: accessing any element by index is extremely fast, because the computer can calculate exactly where in memory to look. If the array starts at memory address A and each element takes B bytes, then the element at index i is at address A + (i × B). This is a simple arithmetic calculation that takes the same tiny amount of time regardless of which index you request or how large the array is.

Python's built-in list is more flexible than a traditional array: it can grow, shrink, and hold mixed types. But this flexibility has costs. The internal bookkeeping required to support dynamic resizing and mixed types makes Python lists somewhat slower than raw arrays for very large-scale numerical work. This is why Python's scientific computing library (NumPy) provides a true fixed-type array for applications such as data science and machine learning.

Section 11.5

Strings as Sequences

You have been using strings since Chapter 7, but they are worth revisiting here because a string is not just a piece of text — it is a sequence of characters. Every character in a string has a position, just like every element in a list. You can access individual characters by index, slice substrings, find the length, and loop through characters one at a time. This means everything you learned about lists and indexes applies directly to strings.

The key difference between strings and lists is that strings are immutable: you cannot change an individual character in place. To create a modified version of a string, you must build an entirely new string. This immutability is by design — it makes strings safe to share between different parts of a program without worrying that one part will accidentally modify what another part is reading.

Example 11.5.1
Indexing and slicing a string
  • Setupword = "computing"
  • Index word[0] → "c" word[3] → "p" word[-1] → "g" len(word) → 9
  • Slice word[0:4] → "comp" word[4:] → "uting" word[:3] → "com"
  • Loop for ch in word: print(ch) # prints: c o m p u t i n g (one per line)

Immutability in Practice

Because strings are immutable, operations that seem to "change" a string actually produce a new string and leave the original unchanged. The original variable can then be assigned to the new string if you want the change to persist.

Example 11.5.2
String operations that produce new strings
  • Setupgreeting = "hello world"
  • Upper greeting.upper() → "HELLO WORLD" # greeting is still "hello world" — upper() returns a new string
  • Replace greeting.replace("world", "Python") → "hello Python" # original unchanged
  • Split greeting.split(" ") → ["hello", "world"] # produces a list of substrings

Section 11.6

Records

A list stores many items of the same kind in order. But many real-world entities consist of several different pieces of information that belong together. A student is not just a name — a student record includes a name, a student ID number, a grade level, a list of courses, and a GPA. These different pieces of information about the same entity are called fields, and the entire grouped set of fields is called a record.

Records allow programs to represent real-world entities naturally. Instead of storing names in one list, IDs in another list, and GPAs in a third list (and trying to keep them synchronized), a record keeps all the information about one student bundled together.

In Python, records can be represented in several ways depending on the complexity of the program. At the introductory level, a common approach is to use a dictionary (covered in the next section) for each record, where the field names are keys and the field values are values. Another approach is to use a class, which is covered in later courses. What matters conceptually is the idea: a record groups named fields into a single structured unit that represents one thing in the world.

One record representing one student. Each row is a field: a name (key) paired with a value. All five fields together form a single coherent unit.

Records differ from lists in an important way. A list is accessed by position — you retrieve item number 0, item number 1, and so on. A record is accessed by name — you retrieve the value of the "name" field or the "gpa" field. Accessing by name is more readable and less error-prone, because field names carry meaning while position numbers do not.

In many systems, a large collection of records is organized as a table: each row is one record, and each column is one field. A school database might have a table of thousands of student records, each with the same set of fields but different values. This is exactly how relational databases (covered in Chapter 18) organize information.

Section 11.7

Dictionaries and Maps

Suppose you want to look up a student's grade by their name. A list would require you to search through every element until you find the name you want — potentially checking thousands of entries. A dictionary solves this problem by organizing data as key-value pairs. Each key maps directly to one value, and looking up a value by key is extremely fast regardless of how many entries the dictionary contains.

The metaphor is exact: a physical dictionary maps each word (the key) to its definition (the value). A phone book maps each name (the key) to a phone number (the value). A dictionary data structure generalizes this pattern to any kind of key and any kind of value.

In Python, dictionaries are created using curly braces, with colons separating keys from values and commas separating pairs.

Example 11.7.1
Creating and using a dictionary
  • Create grades = { "Alice": 92, "Bob": 78, "Carmen": 95, "David": 85 }
  • Look up grades["Alice"] → 92 grades["Carmen"] → 95
  • Add grades["Eva"] = 88 # grades now has 5 entries
  • Update grades["Bob"] = 82 # updates Bob's grade from 78 to 82
  • Delete del grades["David"] # removes David's entry entirely
  • Check "Alice" in grades → True "Frank" in grades → False

Dictionaries vs. Lists — When to Use Each

Use a list when order matters and you access items by position. Use a dictionary when you need to access items by name or identifier. If you find yourself maintaining two parallel lists — one of names and one of matching values — that is a strong signal to use a dictionary instead.

Section 11.8

Stacks and Queues

Lists and dictionaries organize data for flexible access. But sometimes the structure of a problem calls for a stricter rule about how items enter and exit the collection. Two fundamental restricted-access structures are the stack and the queue. Both are used throughout computer science, from web browsers to operating systems, from compilers to network routers.

The Stack — Last In, First Out

A stack is a collection in which items are added and removed from the same end, called the top. The rule is simple: the most recently added item is always the next item to be removed. This is called Last-In, First-Out, or LIFO.

The physical analogy is a stack of plates in a cafeteria. You place a clean plate on top of the stack. When someone takes a plate, they take it from the top. The plate added most recently is the one removed first. A plate deep in the stack cannot be reached until all the plates on top of it are removed.

The two fundamental stack operations are push (add an item to the top) and pop (remove the item from the top). In Python, a list can serve as a stack: append() acts as push, and pop() without an argument removes from the end (the top of the stack).

The Queue — First In, First Out

A queue is a collection in which items are added at one end (the back or rear) and removed from the other end (the front). The rule is: the first item added is the first item removed. This is called First-In, First-Out, or FIFO.

The physical analogy is a line of people waiting at a ticket counter. The person who arrived first is served first. New arrivals join at the back of the line. Nobody cuts ahead. The two fundamental queue operations are enqueue (add to the back) and dequeue (remove from the front).

Section 11.9

Choosing a Data Structure

Understanding each data structure individually is the first step. The next step — and the one that actually matters in programming — is knowing which structure to choose when faced with a new problem. The key insight is this: the right data structure is the one whose natural operations match the operations your program needs to perform.

To choose wisely, ask yourself a series of questions about the problem. First, what kind of data are you storing, and how much? Second, how will you access the data? By position? By name? By order of arrival? Third, what operations will you perform most often? Adding items? Removing items? Searching? Iterating in order? Fourth, does order matter? Fifth, do you need fast lookup by a specific identifier?

Decision Framework — Choosing a Data Structure

Use a list when you need ordered items you access by position, and the order of items matters. Good for sequences, histories, and ordered sets.

Use an array when all items are the same type, the size is known and fixed, and you need maximum performance for numerical work.

Use a string when your data is text — a sequence of characters you will process, search, or split.

Use a record when you need to group multiple named fields describing a single real-world entity.

Use a dictionary when you need to look up values by a meaningful name or identifier — not by position. Good for any name-to-value mapping.

Use a stack when the most recent item added must be the first item removed (LIFO). Good for undo, back navigation, and tracking nested operations.

Use a queue when items must be processed in the order they arrived (FIFO). Good for scheduling, waiting lists, and ordered task processing.

Example 11.9.1
Matching a problem to a structure — three scenarios
  • Scenario AA music app needs to store a user's playlist: 50 songs, played in order, with the ability to add songs and skip to the next one.
    Best choice: List. Order matters. Access is by position (current position in playlist). New songs are added to the end.
  • Scenario BA system needs to store student grades indexed by student name, and look up any student's grade instantly.
    Best choice: Dictionary. Lookup is by name, not position. Fast key-based access is the core operation.
  • Scenario CA web server receives requests from users. Each request must be handled in the order it was received. New requests queue up when the server is busy.
    Best choice: Queue. First request received must be handled first (FIFO). New requests join the back; the server handles from the front.

Section 11.10

Data Structures in Real Software

Data structures are not abstract exercises — they are the working machinery of every piece of software you use. Understanding which structures appear in real systems helps you see why the concepts in this chapter are foundational rather than academic.

Web browsers use stacks to implement the Back button: each URL you visit is pushed onto a history stack; pressing Back pops the most recent URL. They also use queues to manage resources being downloaded in the background.

Search engines use sophisticated structures to index billions of web pages, but at their core they maintain enormous dictionaries: each word in their index maps to a list of pages where that word appears. Looking up a search term means looking up its key in this dictionary.

Social networks store user profiles as records — each user has an ID, a name, a list of posts, a list of friends, and other fields. Friend lists are themselves lists of user IDs. Finding a user's profile quickly requires dictionary-style lookup by user ID.

Video games store the positions, attributes, and states of every game object. An enemy might be a record containing location coordinates, health points, speed, and current action. Lists of enemies are processed every frame to update movement and check collisions.

Operating systems use queues to schedule processes. When several programs want to use the CPU at the same time, a scheduling queue determines whose turn it is. They also use stacks to manage function calls: each call pushes a frame onto the call stack, and returning from a function pops it off.

Databases organize data as large tables of records. Each row is one record; each column is a field. Lookup by primary key (like a student ID or product code) is implemented with structures designed for fast key-based search, building on the dictionary concept at a much larger scale.

Network routers maintain routing tables — dictionaries that map destination IP addresses to the next network hop. Incoming packets are often placed in queues while the router decides where to forward them.

Chapter 11 — Back Matter

Chapter Summary

  • Sec 11.1A data structure is an organized way to store and access a collection of data. The right structure depends on what operations the program needs, not just how much data exists. Poor structure choices make programs slow and hard to maintain.
  • Sec 11.2A list is an ordered collection of elements that can be any type. In Python, lists are mutable and support operations including append, insert, remove, pop, membership testing, and iteration. The elements have a defined sequence that persists.
  • Sec 11.3Each element in a list has an index — its position, counted from zero. Valid indexes run from 0 to (length − 1). Negative indexes count from the right. Accessing an out-of-range index causes a runtime IndexError. Slicing extracts a contiguous portion as a new list.
  • Sec 11.4A traditional array has a fixed size and holds elements of a single type, stored in contiguous memory. This layout enables constant-time index access via address arithmetic. Python lists are more flexible but traditional arrays remain important in performance-critical and scientific computing contexts.
  • Sec 11.5A string is a sequence of characters, accessed by index and slice just like a list. Strings are immutable: string methods return new strings rather than modifying the original. Common operations include upper, lower, strip, replace, split, and membership testing with in.
  • Sec 11.6A record groups multiple named fields into a single unit representing one real-world entity. Fields have names and types. Records are accessed by field name, not position. A table organizes many records with the same fields into rows and columns — the foundation of database design.
  • Sec 11.7A dictionary maps unique keys to values, enabling fast lookup by name. Keys must be unique and immutable. Values can be any type. Dictionaries are ideal when you need to retrieve data by a meaningful identifier rather than a position number.
  • Sec 11.8A stack follows Last-In, First-Out (LIFO): the most recently pushed item is the next to be popped. A queue follows First-In, First-Out (FIFO): the first item enqueued is the first dequeued. Both can be implemented with Python lists. Stacks model undo/back; queues model waiting lines and scheduling.
  • Sec 11.9Choosing the right data structure requires understanding the operations the program needs. The key questions are: how will items be accessed (by position, by name, by order of arrival)? What operations are most frequent (add, remove, lookup, iterate)? Does insertion or removal order matter?
  • Sec 11.10Data structures appear throughout real software: stacks in browsers and compilers, dictionaries in search engines and routers, records in databases and social networks, queues in operating systems and print servers. Recognizing these uses shows that data structures are the practical machinery of computing.

Key Terms Review

data structure
An organized way to store and access a collection of data so that specific operations can be performed efficiently.
list
An ordered, mutable collection of elements accessible by index. Elements can be any type and can be added, removed, or changed.
element
A single item stored in a list or other collection.
index
The position of an element in a list or string, counted from zero. Valid indexes run from 0 to (length − 1).
zero-based indexing
The convention that the first element in a sequence is at position 0, the second at position 1, and so on.
slice
A contiguous portion of a list or string, specified with a start and stop index (stop is excluded). Returns a new object.
array
A collection of fixed size holding elements of a single type, stored in contiguous memory. Enables constant-time access by index via address arithmetic.
mutable
Can be changed in place after creation. Python lists are mutable; Python strings are not.
immutable
Cannot be changed in place after creation. Strings are immutable; operations on them produce new strings.
record
A group of named fields representing a single real-world entity. Accessed by field name rather than index position.
field
One named piece of information within a record. A student record might have fields for name, ID, grade level, and GPA.
dictionary
A collection of key-value pairs. Each key is unique and maps to exactly one value. Lookup by key is very fast.
key
The identifier used to look up a value in a dictionary. Keys must be unique and immutable within a dictionary.
value
The data associated with a key in a dictionary. Values can be any type and may repeat across different keys.
stack
A collection following Last-In, First-Out (LIFO) order. Items are pushed to and popped from the top. Used for undo and navigation history.
queue
A collection following First-In, First-Out (FIFO) order. Items are enqueued at the back and dequeued from the front. Used for scheduling and waiting lines.
LIFO
Last-In, First-Out. The ordering principle of a stack: the most recently added item is the next to be removed.
FIFO
First-In, First-Out. The ordering principle of a queue: the first item added is the first item removed.

Debugging and Error Analysis

Each item describes a mistake. Identify the error and write a corrected version or explanation.

Mistake 1 — Off-by-one index
grades = [88, 95, 71]; print(grades[3])

The list has 3 elements at indexes 0, 1, 2. Index 3 does not exist. What error will occur at runtime? Write the correct index to access the last element.

Mistake 2 — Modifying a string character in place
s = "hello"; s[0] = "H"

Strings are immutable. This raises a TypeError. How would you correctly create the string "Hello" from "hello"?

Mistake 3 — Duplicate key in a dictionary
d = {"a": 1, "b": 2, "a": 3}; print(d["a"]) # expects 1

Keys must be unique. The second "a" overwrites the first. What value will print? Explain why.

Mistake 4 — Using pop() on a queue
queue = ["A", "B", "C"]; queue.pop() # intends to dequeue

pop() without an argument removes from the end, giving "C" — which is the most recently added item (LIFO). This is stack behavior, not queue behavior. Write the correct dequeue operation for the front.

Mistake 5 — Accessing a missing dictionary key
grades = {"Alice": 90}; print(grades["Bob"])

Accessing a key that does not exist raises a KeyError. How could you safely check whether "Bob" is in the dictionary before accessing it?
