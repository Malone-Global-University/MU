Chapter 10 — Front Matter

Purpose and Overview

"Everyone knows that debugging is twice as hard as writing a program in the first place. So if you're as clever as you can be when you write it, how will you ever debug it?"

Brian Kernighan, computer scientist

Programs rarely work correctly the first time. This is not a sign that you are doing something wrong — it is the normal, expected, universal experience of every programmer who has ever lived. From beginners writing their first ten-line program to expert engineers building systems with millions of lines of code, mistakes happen. The question is not how to avoid them entirely, but how to find them, understand them, and fix them.

This chapter teaches you to do exactly that. You will learn the three major categories of programming errors, how to read the messages a programming environment produces when something goes wrong, and a set of reliable strategies for finding mistakes when error messages are not enough. You will also learn how to test programs deliberately and thoroughly — not just running them once and hoping, but designing specific tests that check whether your code handles all the situations it will encounter. Finally, you will learn about program design: how to think about structure and organization before and during the writing of code, so that programs are easier to read, easier to test, and easier to fix.

These skills — debugging, testing, and design — are what separate programs that merely seem to work from programs that actually do.

Opening Question

You write a program that calculates student averages. You run it, it produces numbers, and the numbers look reasonable. Is the program correct? How would you know? You would need to check specific inputs against known expected outputs, test what happens with edge cases like an empty list or a list with a single score, and verify that the logic handles every boundary correctly. By the end of this chapter, you will have a systematic method for answering that question — for any program you write.

Section 10.1

Mistakes Are Part of Programming

Before learning how to fix mistakes, it helps to understand what mistakes actually are and why they happen so consistently. A program is a precise set of instructions. Computers follow those instructions exactly as written — not as intended, not as assumed, but exactly as written. This creates an immediate source of tension: human beings are imprecise thinkers who work with intentions, and computers are precise executors that work with instructions. The gap between intention and instruction is where almost every programming mistake lives.

Where Mistakes Come From

SourceDescriptionExample
ComplexityMany interacting parts — variables, loops, functions, conditions — each a potential failure pointA variable updated in one function has unexpected effects in another
Invisible assumptionsUnstated beliefs about the program's environment that live in the programmer's head, not the codeAssuming a list will never be empty; assuming user input will be a number
Translation difficultyMoving from a human description to precise code requires many small decisions, any of which can be wrong"Calculate the average" requires deciding how to handle zero count, rounding, and input format
AccumulationA small mistake early in a program can produce effects only visible much laterWrong initialization on line 5 causes a crash on line 40
The Right Attitude

Experienced programmers make mistakes constantly. What distinguishes them is not that they make fewer mistakes — it is that they have good strategies for finding and fixing them efficiently. Debugging is a skill. When something goes wrong, a program is telling you something. Your job is to figure out what it is telling you.

Section 10.2

Syntax Errors

Every programming language has a set of rules called its syntax — the grammar of the language. A syntax error occurs when code violates these rules. Syntax errors are caught before the program runs: Python reads the code, detects the violation, reports it, and refuses to execute anything. You cannot have a running program with a syntax error in it.

This makes syntax errors the easiest category to deal with. They are caught immediately, they tell you roughly where the problem is, and fixing them is usually a matter of correcting a typo or misplaced character.

Debugging Principle

When fixing a syntax error, read slowly. Many syntax errors are a single missing or wrong character. Train your eye to look for what is absent, not just what is present. Also note: the line Python reports is where it noticed the error — check a few lines before that line if the reported line looks correct.

Section 10.3

Runtime Errors

A runtime error occurs while a program is running, when a syntactically correct operation cannot be completed. The program starts, runs until it reaches an impossible situation, then crashes and produces an error message called an exception. Runtime errors depend on conditions: the same line of code might run successfully one hundred times and fail on the hundred-and-first, when conditions happen to be different.

Common Runtime Errors

Runtime Error — ZeroDivisionError
Division by zero
  • Codeaverage = total / count
  • When it failsWhen count is 0. If count is 5, the line runs fine. If count is 0, Python raises ZeroDivisionError and stops.
  • Why it happensThe code is syntactically legal. The bug is a condition that the programmer did not guard against.
Error TypeMeaningCommon Cause
ZeroDivisionErrorA number was divided by zeroUnguarded division where denominator can be 0
IndexErrorA list or string position does not existscores[3] when list has only 3 items (indices 0–2)
NameErrorA variable or function has not been definedUsing a variable before assigning it a value
TypeErrorAn operation was used on the wrong type"Score: " + 85 — cannot add string and integer
FileNotFoundErrorA file at the given path does not existTypo in filename, file in wrong folder
KeyErrorA dictionary key does not existAccessing a key that was never added to the dictionary
ValueErrorRight type, wrong contentint("hello") — cannot convert a word to an integer
Debugging Principle

When you encounter a runtime error, do not just look at the crashing line. Trace backward. Ask how the values in play got to where they are. A variable might be assigned the wrong value twenty lines earlier, causing a crash only when that value is finally used.

Section 10.4

Logic Errors

A logic error is a mistake in the reasoning of a program. The program has correct syntax. It runs without crashing. It produces output. But the output is wrong. Logic errors do not produce error messages. No alarm sounds. No exception is raised. The program runs to completion and silently delivers wrong results — this is what makes them the hardest category to find.

Common Logic Errors

Logic Error 1
Off-by-one error — loop boundary wrong by exactly one
Intended: print 1 through 10. Actual: prints 0 through 9 
for i in range(10):
    print(i)

The code is legal. Python produces no error. But the intent was range(1, 11). Off-by-one errors are among the most common logic errors in all of programming.

Logic Error 2
Wrong operator — boundary condition uses > instead of >=
Intended: pass if score is 60 or higher. A score of exactly 60 fails. 
if score > 60:
    print("Pass")

60 > 60 is False. The boundary should be >=. The program runs and produces plausible output. The bug is invisible until you test the exact boundary value.

Logic Error 3
Wrong variable — divides total by itself instead of count
average is always 1.0 because total / total = 1 
total = 0
count = 0
for value in scores:
    total = total + value
    count = count + 1
average = total / total   # Should be: total / count
Logic Error 4
Missing update inside a loop — largest never changes
The comparison runs but the update never happens 
largest = scores[0]
for score in scores:
    if score > largest:
        pass   # Forgot: largest = score
print("Largest:", largest)
Why Logic Errors Are Hard to Find

Logic errors are hard because the programmer's mental model is wrong. When you wrote the code, you believed it was correct. That belief makes the error invisible: you look at the code and see what you intended, not what is actually written. This is why testing matters so much — running the program against known expected outputs reveals logic errors that careful reading might miss entirely.

Section 10.5

Reading Error Messages

Many beginning programmers look at error messages with dread and skip over them. This is a mistake. Error messages contain real, useful information about what went wrong. Learning to read them efficiently is one of the most practical skills in debugging.

Anatomy of a Python Error Message

Typical Python Runtime Error Message 
Traceback (most recent call last):
  File "grades.py", line 14, in calculate_average
    average = total / count
ZeroDivisionError: division by zero
Part of MessageWhat It Tells You
Traceback (most recent call last):Python is about to show the sequence of function calls that led to the error
File "grades.py", line 14The file name and line number where the crash occurred
average = total / countThe actual line of code being executed when the error occurred
ZeroDivisionError: division by zeroThe error category and a plain-language description of the problem
Reading Tracebacks

For longer tracebacks with multiple function calls, read from the bottom up. The actual error is always at the bottom. The lines above it show the chain of function calls that led there. Start your investigation at the crash point (bottom), then trace upward only if you need to understand how the program arrived there.

Debugging Strategies
Seven Reliable Methods for Finding and Fixing Bugs
Section 10.6

Debugging Strategies

Reading error messages is the first step. But not every bug produces a helpful error message — logic errors produce none at all — and even with an error message, you may not immediately understand what went wrong. The following strategies form a reliable toolkit for finding and fixing bugs at every level of complexity.

Strategy 1: Read the Code Carefully and Slowly

The simplest debugging strategy is also the most frequently skipped: read the code. Read it slowly, line by line, as if you have never seen it before. Do not read what you intended to write. Read what is actually written. A surprisingly high percentage of bugs can be found this way. Look for wrong variable names (especially when two variables have similar names), wrong operators (> versus >=, and versus or), missing updates inside loops, reversed conditions, and correct actions placed in the wrong location.

Strategy 2: Trace the Program by Hand

Pick up a pencil and paper. Work through the program manually, step by step, exactly as the computer would. Keep a table tracking the value of every important variable at every step. This technique — called hand tracing or desk checking — is powerful because it forces you to follow the actual logic of the code rather than the logic you believe the code has.

Worked Example 10.6.1
Hand tracing to reveal a logic error — finding the largest value
  • Program 
    scores = [3, 7, 2, 9, 4]
largest = 0   # The bug is here
for score in scores:
    if score > largest:
        largest = score
print("Largest:", largest)
  • Trace (positive list)
    Stepscorelargestscore > largestUpdate?
    Start—0——
    130Truelargest = 3
    273Truelargest = 7
    327Falseno change
    497Truelargest = 9
    End—9—Output: 9 ✓
  • Edge case trace: all negatives

    Now try with scores = [-5, -3, -8, -1]:

    Stepscorelargestscore > largestUpdate?
    Start—0——
    1-50Falseno change
    2-30Falseno change
    3-80Falseno change
    4-10Falseno change
    End—0—Output: 0 ✗ (should be -1)
  • Bug foundInitializing largest = 0 assumes all values will be positive. The fix: initialize to the first element of the list — largest = scores[0]. This is a logic error that hand tracing reveals clearly. No error message would ever catch it.

Strategy 3: Add Print Statements

One of the most practical debugging tools available at any level is the simple print() function. Add print statements at key points to display the values of variables as the program runs. This makes the computation completely visible. When the bug is fixed, remove or comment out the debugging print statements.

Print Debugging — Making the Computation Visible 
def calculate_average(scores):
    total = 0
    count = 0
    for score in scores:
        total = total + score
        count = count + 1
        print(f"  After adding {score}: total={total}, count={count}")
    print(f"Computing: {total} / {count}")
    return total / count

Strategy 4: Simplify the Problem

If a bug is hard to find in a complex program, try to reproduce it in a simpler context. Extract just the suspected function, remove everything else, and test it in isolation with a few simple inputs. If the function works correctly alone, the bug is in how it is called or how input is prepared — not inside the function itself.

Strategy 5: Check Your Assumptions

Most bugs exist because a programmer assumed something that was not true. When stuck, write down the assumptions your code makes and then check each one explicitly.

Common Assumptions Worth Questioning

→ Is the list ever empty?

→ Can any numeric value be zero or negative?

→ Can the user enter unexpected characters or spaces?

→ Does this function always return a value for every possible input?

→ Is this variable always set before it is used?

Strategy 6: Work Backward from Wrong Output

When you have wrong output but no error message, start at the output and work backward. Ask: what value produced this output? What code set that value? What values did that code depend on? Keep tracing backward until you find the step where something went wrong. This reverse-tracing technique is particularly useful for logic errors.

Strategy 7: Take a Break

When you stare at the same code for a long time, your brain fills in what it expects to see rather than what is there. You stop reading the code and start reading your memory of writing the code. Taking a break of fifteen minutes, an hour, or a day — returning fresh — frequently makes previously invisible bugs immediately obvious. If you have been debugging for a long time without success, stopping is often the most efficient path forward.

Section 10.7

Test Cases

Debugging is reactive: something goes wrong, and you hunt for the cause. Testing is proactive: before or after writing code, you deliberately run it with specific inputs to verify that it produces correct outputs. A test case is a specific input combined with its expected output. You run the program with the input and compare the actual output to the expected output. If they match, that case is handled correctly. If they do not match, there is a bug.

The key discipline is that test cases should be designed before running the program when possible, not after. Deciding what the correct output should be before you see what the program produces forces you to think clearly about the specification. Once you have already seen the output, it is easy to unconsciously rationalize it as correct.

Testing vs. Running

Running a program once and seeing plausible-looking output is not testing — it is hoping. Real testing means deliberately choosing inputs that cover different scenarios and verifying outputs against carefully determined correct values known in advance. One successful run proves nothing about edge cases or boundary conditions.

Section 10.8

Edge Cases

Typical test cases verify that a program handles normal inputs correctly. Edge cases test the boundaries, extremes, and special situations that a program might encounter — the cases most likely to reveal hidden bugs and violated assumptions.

Most programs are written with typical inputs in mind. The code that handles typical inputs is usually correct because it is the code the programmer thought about most carefully. Edge cases are the inputs the programmer did not think carefully about — and they often expose assumptions that were wrong.

Sections 10.9 – 10.10
Incremental Development and Program Design

Section 10.9

Incremental Development

The best time to prevent debugging difficulty is while writing code, not after. Incremental development is a programming approach that builds programs in small, tested pieces rather than writing everything at once and then trying to make it work.

The core principle: write a little, test it, fix any problems, then write a little more. This approach keeps the gap between working code and untested code very small. If a bug appears after you add a new piece, you know the bug is almost certainly in that new piece — not buried somewhere in fifty lines you wrote all at once.

The Alternative — And Why It Fails

Writing the entire program, running it for the first time, and finding it does not work forces you to debug a large untested system where the bug could be anywhere. Every component is suspect. Beginning programmers often feel it is more efficient to write everything before testing — in practice, the debugging time at the end far exceeds the testing time that could have been distributed throughout.

Worked Example 10.9.1
Building a grade statistics program incrementally — five steps
  • Step 1

    Write and test the simplest possible version: read a list and print it.

    scores = [88, 92, 74, 95, 81, 67, 78]
print("Scores:", scores)

    Does it print correctly? Yes. Move on.

  • Step 2

    Add the calculation of the total.

    scores = [88, 92, 74, 95, 81, 67, 78]
total = sum(scores)
print("Total:", total)

    Is the total correct? 88+92+74+95+81+67+78 = 575. Yes. Move on.

  • Step 3

    Add the average calculation.

    total = sum(scores)
count = len(scores)
average = total / count
print("Average:", round(average, 2))

    Is the average correct? 575 / 7 ≈ 82.14. Yes. Move on.

  • Step 4

    Add highest and lowest scores.

    highest = max(scores)
lowest = min(scores)
print("Highest:", highest)
print("Lowest:", lowest)

    Correct? Yes. Move on.

  • Step 5

    Clean up output, remove intermediate prints, finalize.

    If a bug appeared at step 3, it was almost certainly in the code added at step 3 — not buried in everything written since the beginning. Each step was a small, verified addition.

Incremental Development Rules

Start with the simplest version that does something visible. Make each addition as small as possible. Test after every addition. Keep a working version at all times — never write so much untested code that you cannot return to a working state.

Section 10.10

Documentation and Program Design

Writing correct code is necessary. Writing code that others — and your future self — can understand is equally important. Documentation and program design are the practices that make programs readable, maintainable, and collaborative.

Forms of Documentation

FormWhat It IsWhere It Appears

Writing Good Comments

Example 10.10.1
Useless comment vs. useful comment
Useless — just repeats what the code already says

Writing Docstrings

A Well-Documented Function 
def calculate_average(scores):
    """
    Calculate and return the arithmetic mean of a list of numbers.

    Parameters:
        scores: a list of numbers (may be empty)

    Returns:
        The arithmetic mean of the values in scores.
        Returns 0.0 if the list is empty.
    """
    if len(scores) == 0:
        return 0.0
    return sum(scores) / len(scores)

Program Design Questions Before Coding

Ask These Before Writing a Line of Code

1. What is this program supposed to do? (State it clearly in plain language.)

2. What are the inputs? What values, files, or information does it receive?

3. What are the outputs? What should the program produce?

4. What are the major steps? (Write pseudocode or a task list first.)

5. What functions are needed? Which parts of the work are distinct enough to be separated?

6. What can go wrong? Identify edge cases before they become bugs.

Design, Testing, and Debugging Are Connected

Good program design makes testing easier — well-decomposed functions with clear purposes can be tested individually. Good design makes debugging easier — clear names and useful comments make it much faster to understand what each piece of code is supposed to do and to notice when it is not doing it. Design, testing, and debugging are not separate activities. They are three aspects of a single discipline: writing programs that actually work.

Chapter Summary

  • Sec 10.1Mistakes are normal and inevitable. The gap between intention and instruction is where almost every programming error lives. Debugging is a skill: the right attitude is curiosity, not frustration.
  • Sec 10.2Syntax errors violate the grammar rules of a programming language. They are caught before the program runs and must be fixed before execution can begin. Common causes: missing colons, unmatched delimiters, bad indentation, misspelled keywords, unclosed strings.
  • Sec 10.3Runtime errors occur during execution when a syntactically correct operation cannot be completed. They depend on conditions and produce exceptions. Common types: ZeroDivisionError, IndexError, NameError, TypeError, FileNotFoundError.
  • Sec 10.4Logic errors are mistakes in reasoning. The program runs without crashing but produces wrong output. No error message is produced. Logic errors require testing and careful tracing to find. Common examples: off-by-one errors, wrong operators, wrong variables, missing loop updates.
  • Sec 10.5Error messages and tracebacks contain real information. Read them carefully. For tracebacks, read from the bottom up — the actual error type and description is always at the bottom. The reported line may not always be the true source of the error.
  • Sec 10.6Seven debugging strategies: read slowly; trace by hand; add print statements; simplify the problem; check your assumptions; work backward from wrong output; take a break. These strategies work together and apply to every level of programming.
  • Sec 10.7A test case is a specific input paired with its expected output. Design test cases before running the program when possible. Running a program once is not testing. Systematic test cases build justified confidence in a program's correctness.
  • Sec 10.8Edge cases test boundaries and extremes: empty collections, single elements, boundary values, negative numbers, very large values, all-same values. Edge cases reveal the hidden assumptions a program makes and the bugs those assumptions can cause.
  • Sec 10.9Incremental development means writing small pieces and testing each one before adding more. It keeps bugs small and isolated. Writing all the code before testing is far less efficient because the debugging time at the end outweighs the saved testing time during development.
  • Sec 10.10Documentation makes programs understandable. Useful comments explain reasoning, not just what the code does. Docstrings describe function purpose, parameters, and return values. Naming is design. Thinking about structure, inputs, outputs, and edge cases before writing code makes everything afterward easier.

Key Terms Review

syntax error
An error caused by code that violates the grammar rules of a programming language. Detected before the program runs; prevents execution.
runtime error
An error that occurs during program execution when a syntactically correct operation cannot be completed. Also called an exception.
logic error
A mistake in a program's reasoning. The program runs and produces output, but the output is wrong. No error message is produced.
exception
The technical term for a runtime error in Python. When Python cannot complete an operation, it raises an exception and stops execution.
traceback
The error report Python produces when a runtime error occurs, showing the sequence of function calls that led to the crash. Read from the bottom up.
debugging
The process of finding and fixing errors in a program. Debugging is a skill that improves with practice and systematic strategies.
hand tracing
A debugging technique in which a programmer follows a program's logic step by step on paper, tracking variable values at each step.
print debugging
A technique in which print statements are added to a program to display variable values and confirm execution paths during a run.
test case
A specific input paired with its expected output, used to verify that a program or function behaves correctly.
edge case
An input at the boundary or extreme of a program's expected range — empty collections, exact boundary values, negatives — that often reveals hidden bugs.
off-by-one error
A logic error in which a loop, count, or boundary condition is wrong by exactly one. One of the most common programming mistakes.
incremental development
A programming approach in which code is written in small steps, each tested before the next is added. Keeps bugs small and isolated.
documentation
Written explanations of a program's purpose, design, and behavior. Includes comments inside code, docstrings for functions, and external guides.
comment
A note written inside code using # that the interpreter ignores, used to explain purpose, reasoning, or structure.
docstring
A string literal placed at the start of a function body that serves as documentation: describing the function's purpose, parameters, and return value.
program design
The practice of thinking about structure, naming, decomposition, inputs, outputs, and edge cases before and during the writing of code.

For each description, identify the error type: syntax error, runtime error, or logic error. Explain your answer in one sentence.

  1. A program is supposed to print the numbers from 1 to 5. It prints 0, 1, 2, 3, 4.
  2. Python refuses to run a program and reports SyntaxError: expected ':' on line 6.
  3. A program calculates a shopping discount. When the discount is 100%, it crashes with ZeroDivisionError.
