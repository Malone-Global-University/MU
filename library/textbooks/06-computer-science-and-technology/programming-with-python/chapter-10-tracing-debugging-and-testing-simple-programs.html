Chapter 10 — Front Matter

Chapter Introduction and Learning Objectives

Every programmer, from beginner to expert, writes programs that do not work correctly on the first attempt. This is not a sign of failure. It is a normal part of programming. What separates programmers who grow from programmers who get stuck is not whether they make mistakes. It is whether they have the skills and habits to find and fix those mistakes.

This chapter is about exactly those skills. You have now written programs with variables, expressions, input and output, conditionals, and loops. Your programs have grown more complex. That means there are more places for something to go wrong — and more reasons to develop a disciplined approach to reading, tracing, and testing your programs.

Opening Idea

A program that runs without crashing is not the same thing as a program that is correct. Python will happily execute a flawed plan from beginning to end and print a confident, wrong answer. The only defense against this is a set of habits: tracing code carefully, reading error messages closely, testing with inputs you already know the answer to, and building programs in small, verified pieces. This chapter teaches those habits before your programs grow large enough that mistakes become hard to find.

Section 10.1

Why Programmers Trace Code

When you read a sentence, your eyes move forward and your brain follows the meaning. When you trace code, you follow the computer — one instruction at a time — and track what changes.

Tracing is the practice of reading your program the way Python executes it: in order, keeping track of variable values, following the path through conditions and loops, and noting what gets printed or returned.

Beginners often skip tracing. They write code, run it, and then feel confused when the output is wrong. The confusion comes from not knowing what the program actually did. Tracing eliminates that confusion.

Example 10.1.1
score = 50
bonus = 10
score = score + bonus
print(score)

If you trace this program, you follow it line by line:

  • Line 1: score is assigned the value 50.
  • Line 2: bonus is assigned the value 10.
  • Line 3: Python evaluates score + bonus, which is 50 + 10 = 60. The result is assigned back to score. Now score is 60.
  • Line 4: Python prints 60.
Output
60

That was easy because the program is small. But the same approach applies when programs are larger. You read each line in the order Python runs it, you update your mental picture of every variable, and you follow every branch and every loop iteration.

Study Note

Tracing is not about memorizing what Python does. It is about thinking carefully about what each line means. Every skilled programmer traces code — sometimes in their head, sometimes on paper, sometimes by adding extra print statements to watch variables as a program runs.

Section 10.2

Tracing Tables

A tracing table is a simple tool for keeping track of variable values as a program runs. You create a column for each variable, a column for any output, and a row for each step in the program. Then you fill it in as you read through the code.

Example 10.2.1
count = 0
total = 0

count = count + 1
total = total + 5

count = count + 1
total = total + 8

print(count)
print(total)

The variables are count and total. The tracing table looks like this:

StepLinecounttotalOutput
1count = 00—
2total = 000
3count = count + 110
4total = total + 515
5count = count + 125
6total = total + 8213
7print(count)2132
8print(total)21313

The program prints 2 and then 13.

Now trace a program with a loop:

Example 10.2.2
total = 0
for number in range(1, 5):
    total = total + number
print(total)

range(1, 5) produces the sequence 1, 2, 3, 4.

Iterationnumbertotal
Start—0
1st11
2nd23
3rd36
4th410

After the loop ends, print(total) outputs 10.

Why Tracing Tables Matter

When you trace carefully, you often discover the error before you even run the program. Students who skip tracing tend to change code randomly and feel lucky when it works. Programmers who trace code understand why it works — or why it doesn't.

Section 10.3

Reading Error Messages

Python produces an error message whenever it cannot execute your program as written. The error message is not punishment. It is information. Learning to read error messages clearly is one of the most important debugging skills you can develop.

Here is a typical Python error message:

Traceback
Traceback (most recent call last):
  File "quiz.py", line 7, in <module>
    score = score + points
NameError: name 'points' is not defined

Every Python error message has the same structure:

Anatomy
The four parts of a Python traceback
  • TracebackThe word "Traceback" appears at the top. This tells you Python is showing you the sequence of steps that led to the error.
  • File & linePython tells you the file name (quiz.py) and the line number (line 7) where the error occurred. This is where you should start looking.
  • Problem linePython shows you the actual line of code that caused the problem: score = score + points.
  • Error & messageAt the bottom is the most important part — the error type (NameError) and a short message (name 'points' is not defined). This tells you exactly what kind of problem Python encountered.
Reading Strategy

Reading the bottom line first is often the fastest way to understand what went wrong. Once you know the error type, look at the line number, read that line in your program, and ask yourself why that error would occur there.

Section 10.4

Syntax Errors

A syntax error means that Python cannot even understand what you wrote. The code does not follow the rules of the Python language, so Python refuses to run the program at all.

Common causes of syntax errors:

  • Missing parentheses
  • Missing colons after if, for, while, or def
  • Mismatched quotation marks
  • Misspelled keywords
  • Extra or missing punctuation
Example 10.4.1
score = 85

if score >= 60
    print("Passing")

Running this produces:

  File "grades.py", line 3
    if score >= 60
                 ^
SyntaxError: expected ':'

Python points directly at the problem: the colon is missing at the end of the if statement.

Fix
if score >= 60:
    print("Passing")
Procedure
How to handle syntax errors
  • Step 1Read the error message and note the line number.
  • Step 2Look at that line and the line just before it. Python sometimes points one line past the actual mistake.
  • Step 3Look for the most common syntax problems: missing colons, missing parentheses, mismatched quotes.
  • Step 4Fix one problem at a time, then run again.
Good News

Syntax errors are actually the easiest errors to fix because Python tells you almost exactly where to look.

Section 10.5

Runtime Errors

A runtime error is an error that Python does not detect until the program is actually running. The syntax was valid enough for Python to start, but something went wrong during execution.

Common runtime errors include:

  • Dividing by zero
  • Using a variable name that was never assigned a value
  • Trying to convert a string like "hello" to an integer
  • Using the wrong type of value where a different type is expected
Example 10.5.1 — ZeroDivisionError
items = 0
total = 100
average = total / items
print(average)
Traceback (most recent call last):
  File "average.py", line 3, in <module>
    average = total / items
ZeroDivisionError: division by zero

Python ran the first two lines fine, but failed at line 3 because dividing by zero is impossible.

Example 10.5.2 — ValueError
age = int("twenty")
print(age)
Traceback (most recent call last):
  File "age.py", line 1, in <module>
    age = int("twenty")
ValueError: invalid literal for int() with base 10: 'twenty'

The string "twenty" cannot be converted to an integer. Python expected digits, not letters.

Example 10.5.3 — NameError
print(result)
Traceback (most recent call last):
  File "result.py", line 1, in <module>
    print(result)
NameError: name 'result' is not defined

Python has never seen a variable named result. You are trying to use a variable that does not exist yet.

Procedure
How to handle runtime errors
  • Step 1Read the error type and message.
  • Step 2Find the line number in your code.
  • Step 3Ask: what value does the variable hold at that point? What operation is being performed? What could go wrong with those inputs?
  • Step 4Trace backward from the error to find where the bad value came from.
Section 10.6

Logic Errors

A logic error is the most difficult kind of error to find. The program runs without crashing, but it produces the wrong result. Python cannot detect logic errors because, from Python's perspective, the program is valid and complete. The mistake is in your thinking, not in the syntax or the types.

Example 10.6.1
# Calculate the average of three scores
score1 = 80
score2 = 90
score3 = 70

average = score1 + score2 + score3 / 3
print(average)

Expected output: 80.0

Actual output: 193.33333333333334

The program ran perfectly. No errors. But the answer is wrong.

The problem is order of operations. Division has higher precedence than addition, so Python evaluated score3 / 3 first, producing 23.333..., and then added score1 + score2 + 23.333... to get 193.333....

Fix
average = (score1 + score2 + score3) / 3
print(average)

Now the output is 80.0.

Another classic logic error involves = versus ==:

password = "open123"

if password = "open123":   # This is actually a SyntaxError in Python
    print("Welcome!")

Python actually catches this specific mistake as a syntax error, but the confusion between assignment (=) and comparison (==) causes many logic errors in conditions that are harder to spot.

Example 10.6.2
# Count how many scores are passing (60 or above)
scores = [55, 70, 45, 90, 60]
passing = 0

for score in scores:
    if score > 60:       # Bug: should be >= 60
        passing = passing + 1

print(passing)

With the > instead of >=, a score of exactly 60 is not counted as passing. The program runs without error, but the result is wrong for any input containing a score of exactly 60.

Procedure
How to handle logic errors
  • Step 1Trace the program carefully. Build a tracing table and follow each step.
  • Step 2Add print statements to watch variable values as the program runs.
  • Step 3Test with inputs where you already know what the correct output should be.
  • Step 4Compare what you expected with what actually happened, line by line.
Why This Matters

Logic errors are dangerous precisely because nothing alerts you to them. A program with a syntax error or runtime error announces itself. A program with a logic error confidently hands you a wrong answer that looks correct unless you check it.

Section 10.7

Print-Style Debugging

Print-style debugging is the practice of adding temporary print() statements to your program so you can watch variables change as the program runs. It is one of the oldest and most reliable debugging techniques in programming.

When a program produces a wrong result and you cannot see why, add print statements to show you what the program is actually doing.

Example 10.7.1 — A program that should sum only even numbers
numbers = [1, 2, 3, 4, 5, 6]
total = 0

for number in numbers:
    if number % 2 == 0:
        total = total + number

print(total)

Suppose you are not sure whether the condition is working correctly. Add a print statement inside the loop:

numbers = [1, 2, 3, 4, 5, 6]
total = 0

for number in numbers:
    print("Checking:", number, "  total so far:", total)   # Debug print
    if number % 2 == 0:
        total = total + number

print("Final total:", total)
Output
Checking: 1   total so far: 0
Checking: 2   total so far: 0
Checking: 3   total so far: 2
Checking: 4   total so far: 2
Checking: 5   total so far: 6
Checking: 6   total so far: 6
Final total: 12

Now you can see exactly what happens at every step. The program is working correctly: it adds 2, 4, and 6, and the final total is 12.

Rules for Print-Style Debugging

Add a label to every debug print so you know which print statement produced which line.

Print the variable name alongside its value: print("total:", total) is more useful than print(total).

Remove or comment out debug print statements when you are done. Leaving them in makes programs messy.

Add debug prints at the point where you suspect something goes wrong, not randomly throughout the program.

Print-style debugging is powerful because it shows you the actual values your program uses, not the values you assumed it would use.

Section 10.8

Test Cases

A test case is a specific input, together with the output you expect for that input. Writing test cases before you are done coding is one of the most important habits in programming.

A function to test
def is_passing(score):
    if score >= 60:
        return True
    else:
        return False

Test cases for this function:

Input (score)Expected OutputReason
90TrueWell above passing
60TrueExactly at the boundary
59FalseOne point below passing
0FalseLowest possible score
100TruePerfect score

To test each case manually:

print(is_passing(90))    # Expected: True
print(is_passing(60))    # Expected: True
print(is_passing(59))    # Expected: False
print(is_passing(0))     # Expected: False
print(is_passing(100))   # Expected: True

If any of these prints the wrong value, you have found a bug.

Why Test Cases Matter

Without test cases, you can only say "I think my program is correct." With test cases, you can say "My program produces the right output for these specific inputs." Test cases give you evidence, not just hope.

Good test cases cover:

  • A typical case in the middle (score of 90 above)
  • The boundary between two categories (score of 60 and 59)
  • Minimum and maximum realistic values
  • Any special or unusual input you can think of
Section 10.9

Edge Cases

An edge case is an input that is at the boundary, extreme, or unusual end of what a program might receive. Edge cases are where programs most often fail, because programmers tend to test typical cases and forget the unusual ones.

Summing a List
Possible Edge Cases
What if the list is empty? What if it has exactly one item? What if all items are zero? What if all items are negative?
Dividing Numbers
Possible Edge Cases
What if the second number is zero?
Checking a Password
Possible Edge Cases
What if the user enters an empty string? What if the user enters a very long string?
Counting Characters
Possible Edge Cases
What if the word is empty? What if the word has only one character?

Consider this program that finds the largest number in a list:

def find_largest(numbers):
    largest = 0
    for number in numbers:
        if number > largest:
            largest = number
    return largest

Test with typical input:

print(find_largest([3, 7, 2, 9, 1]))   # Expected: 9

Output: 9. Looks correct.

Now test with an edge case — all negative numbers:

print(find_largest([-3, -7, -2, -9, -1]))   # Expected: -1

Output: 0. Wrong.

The Bug

largest = 0 assumes the largest value is at least zero. But if all numbers are negative, the function never finds anything larger than 0 and incorrectly returns 0.

A better approach
def find_largest(numbers):
    largest = numbers[0]    # Start with the first item, not zero
    for number in numbers:
        if number > largest:
            largest = number
    return largest

Now the function works correctly even when all numbers are negative.

The Habit

The habit of asking "what if the input is unusual?" and then testing that unusual input catches many bugs that normal testing misses.

Section 10.10

Incremental Development

Incremental development means building a program in small pieces and testing each piece before moving to the next. Instead of writing the entire program and then running it for the first time, you write a small portion, test it, confirm it works, and then extend it.

Incremental development makes debugging far easier. When you write 50 lines and run the program for the first time, an error could be anywhere in those 50 lines. When you write 5 lines, test them, confirm they work, and then write 5 more, you always know that any new error is almost certainly in the new code you just added.

Here is an example. Suppose you want to build a program that reads a list of quiz scores from the user, then prints the total, the average, and the highest score.

Build Process
Building a quiz score summary, one piece at a time
  • Step 1Collect the scores and confirm they are stored correctly before adding any calculation logic.
  • Step 2Add the total. Run the program. Confirm the total is correct.
  • Step 3Add the average. Run the program. Check the average.
  • Step 4Add the highest score. Run the program one final time.
Step 1 — Collect the scores
scores = []
count = int(input("How many scores? "))

for i in range(count):
    score = float(input("Enter score: "))
    scores.append(score)

print("Scores entered:", scores)
Step 2 — Add the total
scores = []
count = int(input("How many scores? "))

for i in range(count):
    score = float(input("Enter score: "))
    scores.append(score)

total = sum(scores)
print("Total:", total)
Step 3 — Add the average
scores = []
count = int(input("How many scores? "))

for i in range(count):
    score = float(input("Enter score: "))
    scores.append(score)

total = sum(scores)
average = total / count
print("Total:", total)
print("Average:", average)
Step 4 — Add the highest score
scores = []
count = int(input("How many scores? "))

for i in range(count):
    score = float(input("Enter score: "))
    scores.append(score)

total = sum(scores)
average = total / count
highest = max(scores)

print("Total:", total)
print("Average:", average)
print("Highest:", highest)

Each step added one new idea. Each step was tested. If something broke, it was clear exactly where the new problem was introduced.

The Incremental Development Mindset

Get one part working before building the next.

Test early and test often.

Never add new code to code that is already broken.

If something stops working, think about what you just changed.

Key Terms

syntax error
An error that prevents Python from parsing your code because it violates the rules of the Python language. Always detected before the program runs.
runtime error
An error that occurs while a program is executing. The syntax was valid, but something went wrong during execution.
logic error
An error in which the program runs without crashing but produces the wrong output because the programmer's reasoning was incorrect.
traceback
The error report Python displays when a runtime error occurs, showing the file name, line number, and type of error.
tracing
The practice of reading program code one step at a time, tracking variable values, loop iterations, and branch decisions exactly as Python would execute them.
tracing table
A structured table used to record variable values, output, and execution steps as you trace through a program.
print-style debugging
The practice of adding temporary print() statements to a program to display variable values and help locate the source of a bug.
test case
A specific input paired with the expected output, used to verify that a program or function behaves correctly for that input.
edge case
An input that is at the boundary or extreme of what a program might receive — often where bugs hide.
incremental development
A programming practice in which you build a program in small pieces, testing each piece before adding the next.
NameError
A runtime error that occurs when Python encounters a variable name that has not been assigned a value.
ValueError
A runtime error that occurs when a function receives a value of the right type but an inappropriate value.

Worked ExampleBuilding and Testing a Quiz Score Analyzer

This worked example shows how to build a small program incrementally, test it with test cases, and use debugging to find and fix a logic error.

Goal: Write a program that takes a list of five quiz scores, then prints the total, the average, and whether the student is passing (average of 60 or above).

Step 1 — Define the scores and compute the total
scores = [70, 55, 80, 65, 90]
total = 0

for score in scores:
    total = total + score

print("Total:", total)

Test this. Expected total: 70 + 55 + 80 + 65 + 90 = 360. Output: Total: 360. Correct.

Step 2 — Add the average
scores = [70, 55, 80, 65, 90]
total = 0

for score in scores:
    total = total + score

average = total / len(scores)
print("Total:", total)
print("Average:", average)

Expected average: 360 / 5 = 72.0. Output: Average: 72.0. Correct.

Step 3 — Add the passing check
scores = [70, 55, 80, 65, 90]
total = 0

for score in scores:
    total = total + score

average = total / len(scores)
print("Total:", total)
print("Average:", average)

if average > 60:
    print("Result: Passing")
else:
    print("Result: Not Passing")

Test with the existing scores. Average is 72.0, which is above 60, so "Passing" is printed. Correct.

Step 4 — Test edge cases

Try an average of exactly 60. Change the scores to [60, 60, 60, 60, 60].

Expected output: Passing (60 is a passing average).

Bug Found

The condition is average > 60, not average >= 60. With an average of exactly 60, the program prints "Not Passing." That is a logic error.

Fix
if average >= 60:
    print("Result: Passing")
else:
    print("Result: Not Passing")

Lesson

The boundary case (average of exactly 60) revealed the logic error. Without testing that case, the program would have appeared correct.

Defining and Calling Functions

Chapter 10 closes the control flow unit by giving you the skills to verify and repair programs you write. You now have a systematic way to trace, classify errors, test, and debug. Chapter 11 opens Part III by introducing functions — named, reusable blocks of code that let you organize larger programs. You have been writing short scripts where all the code runs from top to bottom. Functions allow you to give a block of code a name, call it from multiple places, and avoid repeating the same instructions throughout a program. You will learn how to define a function, how to call a function, how execution moves into and back out of a function, and how functions help you organize programs that would otherwise become long and hard to follow. The tracing and debugging skills you built in Chapter 10 will carry directly into Chapter 11, because function calls add a new dimension to tracing: you will track not just variable values, but where in the program execution is happening.
