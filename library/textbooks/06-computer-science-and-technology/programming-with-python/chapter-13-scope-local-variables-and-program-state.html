Chapter 13 — Front Matter

Chapter Introduction and Learning Objectives

When you write a function, the variables inside it do not automatically connect to variables outside it. Python keeps them separate on purpose. This separation is called scope, and understanding it will save you hours of confusion and debugging time.

This chapter explains how scope works, why it exists, what happens to variables when a function finishes running, what global variables are and why you should use them carefully, and how to design functions that communicate clearly through parameters and return values rather than through hidden shared variables.

Opening Question

Two different functions in the same program both use a variable named total. Does changing total inside one function change the total inside the other? Does it change a total that exists in the main program? The answer depends entirely on scope — and getting the answer wrong is one of the most common sources of confusing bugs for new programmers. By the end of this chapter, you will be able to answer this question correctly every time, and you will know how to design functions that avoid the confusion in the first place.

The Foundation: Why Scope Exists, Local Variables, and Lifetime

Section 13.1

Why Scope Matters

Imagine you are working at a large company. Every department has its own filing cabinet. A folder labeled "report" in the accounting department is a completely different folder from one labeled "report" in the marketing department. Neither department accidentally reads or changes the other's files. The same name in two places refers to two different things.

Python uses this same idea for variables inside functions. When you write a variable inside a function, Python treats it as belonging to that function. A variable named total inside one function is not the same variable as total inside a different function, and it is not the same as a total that might exist in the main body of your program.

This system is called scope. Scope answers the question: which part of the program can see and use this variable?

Why Scope Is Not Just a Technicality

New programmers sometimes treat scope as a strange rule to memorize. But scope exists for a very practical reason: it prevents functions from accidentally interfering with each other.

Think about what would happen without scope. If every variable in every function were shared, then calling one function might accidentally overwrite a variable that another function was relying on. Tracking down that kind of bug — where two completely unrelated functions share a name by accident — would be a nightmare. Programs would become fragile and unpredictable.

Scope gives every function a private working space. You can use any variable name you want inside your function without worrying about whether you are accidentally changing something in another function or in the main program.

A First Example

Read this program carefully:

def greet_user(): message = "Hello, welcome to the program." print(message) greet_user() print(message) # What happens here?

When you run this program, Python prints the greeting. But then the second print(message) causes an error:

NameError: name 'message' is not defined

The variable message was created inside greet_user. It belongs to that function. Once the function finishes, message is gone. The main program cannot see it at all.

This is scope working exactly as intended.

Study Note

An error like this is not a sign that something is broken. It is Python correctly enforcing a boundary. Learning to read this kind of error and recognize it instantly as a scope issue — rather than a mystery — is one of the most useful skills you will build in this chapter.

Section 13.2

Local Variables

A local variable is a variable that is created inside a function. It belongs to that function and only exists while that function is running.

Here is how to recognize a local variable: if you see an assignment statement inside a function definition — that is, a line with = that creates a new name — that variable is local to that function.

def calculate_discount(price, rate): discount_amount = price * rate # local variable final_price = price - discount_amount # local variable return final_price

In this function, discount_amount and final_price are both local variables. They are created when calculate_discount is called and they cease to exist when the function returns.

The parameters price and rate are also local. Parameters behave exactly like local variables — they are created when the function call happens and they disappear when the function ends. You will read more about this in Section 13.4.

Local Variables Are Invisible Outside Their Function

You cannot access a local variable from outside the function that owns it. Attempting to do so raises a NameError.

def make_greeting(name): greeting = "Good morning, " + name + "!" return greeting result = make_greeting("Priya") print(result) # Works fine: prints "Good morning, Priya!" print(greeting) # NameError: name 'greeting' is not defined

The variable greeting is local to make_greeting. After the function returns, it does not exist anywhere you can reach.

Two Functions Can Use the Same Variable Name

Because local variables belong to their own functions, two different functions can use the same variable name without any conflict. They are completely separate variables.

def calculate_area(length, width): result = length * width return result def calculate_perimeter(length, width): result = 2 * (length + width) return result area = calculate_area(5, 3) perimeter = calculate_perimeter(5, 3) print(area) # 15 print(perimeter) # 16

Both functions use a variable named result. These are two completely different variables. Neither function knows or cares that the other one also uses the name result. Each result lives in its own private space.

This is one of the great advantages of functions and scope: you can name your variables clearly and naturally without constantly worrying about name collisions.

Section 13.3

Variable Lifetime

A variable's lifetime is the period during which it exists in memory. For local variables, the lifetime is short and well-defined:

  • A local variable is created when the function is called and execution reaches the line that assigns it.
  • A local variable is destroyed when the function returns.

Each time you call a function, its local variables are created fresh. They do not carry values from a previous call.

Each Call Is Independent

def count_up(): counter = 0 counter = counter + 1 print(counter) count_up() # prints 1 count_up() # prints 1 count_up() # prints 1

Every time count_up is called, a fresh counter variable is created, set to 0, incremented to 1, and printed. The counter does not accumulate across calls because the variable is destroyed when the function returns.

If you want a value to persist between function calls, you need a different approach — such as passing values through parameters and return values, or (carefully) using a global variable. You will learn about global variables in Section 13.5.

Tracing the Lifetime of a Local Variable

It helps to trace lifetime explicitly. Consider this program:

def square(n): result = n * n return result x = square(4) y = square(7) print(x, y)

Here is what happens step by step:

StepEventVariables That Exist
1Program starts(none yet)
2square(4) is calledn = 4 created
3result = 4 * 4 executesn = 4, result = 16
4return result executes; function endsn and result are destroyed
5x = 16 in the main programx = 16
6square(7) is calledn = 7 created (new copy)
7result = 7 * 7 executesn = 7, result = 49
8return result executes; function endsn and result are destroyed
9y = 49 in the main programx = 16, y = 49
10print(x, y) executesprints 16 49

Notice that n and result are created twice — once for each call — and destroyed twice. They never accumulate. The main program's variables x and y persist because they live outside any function.

Section 13.4

Parameters as Local Variables

This section makes a point that is easy to overlook: parameters behave exactly like local variables.

When you call a function, Python creates local variables for each parameter and assigns them the values you passed as arguments. From that moment on, the parameter is just a local variable with an initial value.

def double(number): number = number * 2 return number x = 5 result = double(x) print(x) # still 5 print(result) # 10

When double(x) is called:

Step-by-Step
What happens when double(x) runs
  • Step 1Python creates a local variable called number and sets it to 5 (the value of x).
  • Step 2Inside the function, number = number * 2 changes the local number to 10.
  • Step 3The function returns 10.
  • Step 4The local variable number is destroyed.
  • Step 5The variable x in the main program was never touched. It is still 5.
Critical Point

Assigning a new value to a parameter inside a function does not change the original variable that was passed in. The parameter is a local copy.

Verify Your Understanding

Read this program and predict the output before continuing.

def adjust(value, amount): value = value + amount print("Inside function:", value) score = 80 adjust(score, 10) print("Outside function:", score)
Inside function: 90 Outside function: 80

The function prints 90 because it added 10 to its local copy of value. But score in the main program never changed — it is still 80. The assignment inside the function affected only the local variable value, not the variable score that was passed in.

Section 13.5

Global Variables at an Introductory Level

A global variable is a variable that is created in the main body of a program — outside of any function. It exists for the entire lifetime of the program.

program_name = "Grade Tracker" # global variable def show_title(): print("Welcome to", program_name) show_title()

Here program_name is created outside any function, so it is global. The function show_title can read it.

Functions Can Read Global Variables

By default, a function can read a global variable. Reading means looking up its value without changing it.

tax_rate = 0.08 def calculate_total(price): tax = price * tax_rate return price + tax print(calculate_total(50)) # 54.0 print(calculate_total(100)) # 108.0

The function calculate_total reads tax_rate from the global space. This works without any special instruction.

Modifying a Global Variable Requires the global Keyword

If a function tries to assign a new value to a global variable without special permission, Python creates a new local variable with that name instead. This is a very common source of confusion.

count = 0 def increment(): count = count + 1 # This causes an error increment()

Running this produces:

UnboundLocalError: local variable 'count' referenced before assignment

Python saw the assignment count = count + 1 inside the function and decided count must be a local variable. But then it tried to read count on the right side of = before any local count had been given a value. That is a contradiction, and Python raises an error.

To allow a function to modify a global variable, you use the global keyword:

count = 0 def increment(): global count count = count + 1 increment() increment() increment() print(count) # 3

The line global count tells Python: "When I use the name count in this function, I mean the global variable count, not a new local one."

Be Careful with Global Variables

The global keyword works, but it should be used with caution. Here is why.

When a function modifies a global variable, it creates a hidden connection between the function and the rest of the program. This hidden connection is called a side effect — you will read about side effects in Section 13.7.

Hidden connections make programs harder to understand and debug. If something goes wrong with count, you have to check every function that uses global count to find the problem. As programs grow, this becomes very difficult.

The better design is almost always to pass values in through parameters and receive values back through return values. Use global variables for configuration values or constants that never change — not for values that functions modify.

# Better design: no global needed def increment(count): return count + 1 count = 0 count = increment(count) count = increment(count) count = increment(count) print(count) # 3

This version is clearer. The function does not reach into global space. You can read count = increment(count) and immediately understand that count is being updated.

Common Mistake

Writing global count inside every function as a habit, just to avoid an error. The presence of many global declarations scattered across a program is usually a sign that the program's design should be reconsidered — not a problem to be patched with more global statements.

Section 13.6

Program State

Program state is the collection of all variable values at a specific moment during execution. Every variable that exists, everywhere in the program, contributes to the program's state.

Understanding state is important because bugs are almost always a matter of state. When your program produces the wrong output, it is because some variable somewhere has the wrong value. Debugging means finding which variable has the wrong value and figuring out how it got there.

State Changes Over Time

State is not fixed. It changes every time an assignment statement executes.

balance = 500 def deposit(amount): global balance balance = balance + amount def withdraw(amount): global balance balance = balance - amount deposit(200) withdraw(75) print(balance) # 625

The state of balance changes three times: once when it is created, once after deposit, and once after withdraw. At any moment during execution, the state of the program includes the current value of balance plus anything else that exists.

Local State and Global State

Local variables contribute to the local state of the currently running function. Once the function returns, that local state disappears. Global variables contribute to the global state, which persists across all function calls.

When you have a bug, it helps to ask: "Which piece of state is wrong? Is it a local variable inside a function? Is it a global variable? Is it a value that was passed through a parameter?"

Debugging Tip

When tracking down a bug, sketch the program's state at each step on paper — a quick table of variable names and values is often faster than re-reading the code from the top. This habit, introduced fully in Chapter 10, becomes even more valuable once functions and scope are involved.

Section 13.7

Side Effects

A side effect is any change that a function makes to something outside its own local scope. If a function changes a global variable, prints output, writes to a file, or modifies a list that was passed in, those are side effects.

Side effects are not always bad. Printing output is a side effect, and printing output is often exactly what you want. The important thing is to know when your functions have side effects and to design them thoughtfully.

Functions with Side Effects

log = [] # global list def record(entry): log.append(entry) # side effect: modifies global list record("User logged in") record("File opened") record("File closed") print(log)
['User logged in', 'File opened', 'File closed']

The function record has a side effect: it modifies the global list log. This is a reasonable design for a logging system, but it is worth knowing that the function is doing more than just computing a value — it is changing shared state.

Side Effects Can Make Programs Hard to Trace

The danger of side effects is that they create invisible connections between different parts of your program. A function that only takes parameters and returns values is easy to test: call it with specific inputs and check the output. A function that also modifies global state is harder to test because you also need to check what it did to the global state.

Compare these two versions of the same idea:

# Version A — side effect via global variable total = 0 def add_score(points): global total total = total + points add_score(10) add_score(25) print(total) # 35
# Version B — no side effect, uses parameter and return value def add_score(total, points): return total + points total = 0 total = add_score(total, 10) total = add_score(total, 25) print(total) # 35

Both produce the same result. But Version B is easier to understand, test, and reuse. The function add_score in Version B has no side effects — you give it two values and it gives you one value back. You can test it independently, use it in different programs, and reason about it without thinking about global state.

When Side Effects Are Acceptable

Printing is a side effect, and it is common and intentional. Functions like greet_user() or display_menu() exist specifically to produce output. That is their job, and producing output is a perfectly reasonable side effect.

Side effects on global data structures like lists and dictionaries can also be reasonable in carefully controlled situations. You will learn more about this when you study lists and objects in later chapters.

Guideline

If your function is computing a value, prefer returning that value over storing it in a global variable. Save global state for things that genuinely need to persist across many function calls and cannot easily be passed through parameters.

Section 13.8

Functions with Clear Inputs and Outputs

The clearest functions are those that follow a simple pattern:

  • They receive all the information they need through parameters.
  • They compute or transform that information.
  • They send results back through a return value.
  • They do not read or modify global variables unnecessarily.
  • Their behavior depends only on their inputs, not on hidden state elsewhere.

A function like this is sometimes called a pure function — though you do not need to memorize that term right now. The key idea is that the function's behavior is predictable and testable.

Example: Comparing Two Designs

Suppose you want to compute the letter grade for a numeric score.

# Design A — using a global variable score = 0 # global def get_grade(): if score >= 90: return "A" elif score >= 80: return "B" elif score >= 70: return "C" elif score >= 60: return "D" else: return "F" score = 85 print(get_grade()) # B

This works, but get_grade depends on a global variable. If you want to test it with a different score, you have to change the global variable first. If another part of your program also uses a global named score, you might have a conflict.

# Design B — using a parameter def get_grade(score): if score >= 90: return "A" elif score >= 80: return "B" elif score >= 70: return "C" elif score >= 60: return "D" else: return "F" print(get_grade(85)) # B print(get_grade(92)) # A print(get_grade(55)) # F

Design B is better. The function clearly states what it needs (a score), clearly states what it produces (a grade letter), and does not reach into global space. You can call it with any score at any time without worrying about global state.

Constants Are a Reasonable Exception

One acceptable use of global variables is for constants: values that are set once at the top of a program and never changed. Constants make sense as globals because they are not being modified — they are just shared reference values.

MAX_SCORE = 100 PASSING_SCORE = 60 PROGRAM_NAME = "Grade Tracker" def is_passing(score): return score >= PASSING_SCORE def show_header(): print("=== " + PROGRAM_NAME + " ===")

Here MAX_SCORE, PASSING_SCORE, and PROGRAM_NAME are global constants. The functions read them but do not modify them. This is a clean and readable pattern. By convention, Python programmers write constants in ALL_CAPS to signal that they should not be changed.

Design Principle

Receive information through parameters. Compute a result. Return that result. Reserve global variables for constants or for program-wide state that genuinely cannot be passed through parameters.

Tracing Scope

How to Trace Scope

Section 13.10

Common Scope Errors

Scope errors are some of the most confusing errors for new programmers because the error message often points to a line that looks correct. Understanding what Python is actually telling you will help you fix these bugs quickly.

Error Type 1 — NameError: Using a Local Variable Outside Its Function

def setup(): username = "admin" setup() print(username) # NameError: name 'username' is not defined

What happened: username was created inside setup and belongs to that function. Once setup returns, username no longer exists.

Fix: Return the value and capture it in the main program.

def setup(): username = "admin" return username username = setup() print(username) # admin

Error Type 2 — UnboundLocalError: Assigning to a Global Without global

points = 0 def earn_points(n): points = points + n # UnboundLocalError earn_points(10)

What happened: Python saw the assignment points = ... inside earn_points and decided points must be a local variable. But then it tried to read points on the right side before any local points had been assigned a value.

Fix option A: Use the global keyword.

points = 0 def earn_points(n): global points points = points + n earn_points(10) print(points) # 10

Fix option B (usually better): Pass the value through parameters and return values.

def earn_points(points, n): return points + n points = 0 points = earn_points(points, 10) print(points) # 10

Error Type 3 — Expecting a Parameter Change to Affect the Caller

def reset(value): value = 0 score = 100 reset(score) print(score) # 100, not 0

What happened: value is a local variable (a copy of score). Assigning 0 to value changes the local copy. The variable score in the caller is untouched.

Fix: Return the new value and assign it in the caller.

def reset(value): return 0 score = 100 score = reset(score) print(score) # 0

Error Type 4 — Accidentally Relying on a Global Variable That Changes

multiplier = 2 def scale(value): return value * multiplier print(scale(5)) # 10 multiplier = 10 print(scale(5)) # 50, not 10!

What happened: scale reads multiplier from global scope. When multiplier changed, scale started producing different results — even though the function itself was never modified. This makes the function unpredictable.

Fix: Pass the multiplier as a parameter.

def scale(value, multiplier): return value * multiplier print(scale(5, 2)) # 10 print(scale(5, 10)) # 50

Now the function's behavior depends only on its inputs, not on a hidden global.

Error Type 5 — Confusing Same-Named Variables in Different Scopes

def process(): result = "inside function" print(result) result = "main program" process() print(result)
inside function main program

What happened: There are two completely separate variables both named result. The one inside process is local and exists only during the function call. The one in the main program is in the global scope and is unaffected by the function. This is not a bug — it is scope working correctly — but it can be confusing if you are not expecting it.

Recommendation

When possible, avoid using the same variable name in the main program and inside functions for different things. Clear naming reduces confusion even when scope technically keeps them separate.

Key Terms

scope
The region of a program where a variable can be seen and used. Variables in Python have either local scope (inside a function) or global scope (the main program body).
local variable
A variable created inside a function. It belongs to that function and cannot be accessed from outside it.
global variable
A variable created in the main body of a program, outside any function. It is accessible throughout the program.
variable lifetime
The period during which a variable exists in memory. Local variables exist only while their function is running. Global variables exist for the entire program run.
program state
The complete collection of all variable values at a given moment during program execution.
side effect
A change that a function makes to something outside its own local scope, such as modifying a global variable, printing output, or modifying a shared data structure.
global keyword
The Python keyword global, used inside a function to declare that a name refers to a global variable rather than a new local variable.
constant
A variable whose value is set once and not changed during the program. By convention, constants are written in ALL_CAPS.
UnboundLocalError
An error that occurs when Python treats a name as local (because it appears on the left side of an assignment) but the code tries to read the variable before it has been assigned a local value.
NameError
An error that occurs when code tries to use a name that does not exist in the current scope.

Common Misconceptions

Misconception 1
"If I change a parameter inside a function, the original variable changes too."
A parameter is a local variable that starts out holding a copy of the value you passed in. Reassigning the parameter with = changes only that local copy. The caller's original variable is completely unaffected. (Note: this is different from modifying a mutable object like a list through its methods, which you will explore in later chapters — but simple reassignment, as covered here, never affects the caller.)
Misconception 2
"Once I write 'global x' in one function, every function can now treat x as global."
The global declaration is local to the function it appears in. If you want a second function to also modify the same global variable, that function needs its own global statement. Each function's scope rules are independent.
Misconception 3
"Using global variables is always bad practice and should never be done."
Global variables are a completely reasonable choice for constants — values set once and never changed, like MAX_SCORE or PROGRAM_NAME. The caution in this chapter is specifically about global variables that functions modify, because that creates hidden dependencies that are hard to track. Reading a global constant is not a problem at all.
Misconception 4
"A NameError and an UnboundLocalError are basically the same thing."
A NameError happens when a name does not exist anywhere in the current scope — for example, trying to use a local variable from a function that already returned. An UnboundLocalError happens specifically when Python has already decided a name is local (because it is assigned somewhere in the function) but the code tries to read that name before any local assignment has actually happened. The fixes are related but the causes are different, and recognizing which one you are looking at speeds up debugging.
