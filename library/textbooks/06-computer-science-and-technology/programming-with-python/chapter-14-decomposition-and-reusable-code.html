Chapter 14 — Front Matter

Chapter Purpose and Learning Objectives

Programs are not written all at once. They are designed in pieces, built in pieces, tested in pieces, and improved in pieces. Learning to decompose a problem into well-named, focused functions is one of the most important skills a programmer develops.

In Chapters 11, 12, and 13, you learned how to define functions, give them parameters, receive return values, and understand how variables live inside and outside functions. You now know the mechanics of how functions work.

This chapter is about something different: how to think with functions. A function is not only a way to avoid rewriting the same lines. It is a unit of thought. When you give a function a clear name and a single purpose, you are doing more than organizing code. You are saying: this problem has a piece called this, and that piece does exactly that. The program becomes a conversation between named ideas instead of a wall of instructions.

This chapter teaches you how to break a large problem into smaller ones — a practice called decomposition — and how to build programs from focused, reusable, well-named functions. It also teaches you how to spot duplicated code and improve it. By the end of this chapter, you will be able to design a multi-function program before you write it, build and test it piece by piece, and improve messy code into something readable and maintainable.

Opening Idea

When you write a function, you are creating a named abstraction. The rest of the program does not need to know how the function works. It only needs to know what it is called and what it does. This is how programmers manage programs that are too large to hold in one thought — and it is the central idea this chapter builds from beginning to end.

The Problem and the Tool: Complexity, Decomposition, and Helper Functions

Section 14.1

Programs Grow in Complexity

When you first started writing Python programs, a complete program might have been five or ten lines:

Python
name = input("What is your name? ")
print("Hello,", name)

That fits in your head all at once. You can read it, trace it, and understand it in seconds.

Now consider a program that asks a user for their name, age, and city, checks whether their age makes them eligible for a discount, calculates the discounted price based on three different membership tiers, formats a receipt, and saves the result to a file.

That program might be 80 or 100 lines long. If you write it as one continuous block — no functions, no structure, everything in sequence from top to bottom — something uncomfortable happens. You can no longer hold the whole thing in your head at once. When a bug appears on line 67, you have to trace back through every earlier variable to understand what went wrong. When you want to change how discounts are calculated, you have to find every place where discount logic appears and update each one. When you want to test part of the program in isolation, you cannot, because every part depends on every other part.

This is what programmers mean when they say a program has grown in complexity. Complexity is not just length. A 200-line program that is well-organized is often easier to work with than a 50-line program that does everything in one tangled block.

Functions are the main tool for managing complexity. Instead of one long block, you write many small, focused pieces. Each piece has a name. Each piece does one thing. You can read, test, and fix each piece independently.

The Key Insight

When you write a function, you are creating a named abstraction. The rest of the program does not need to know how the function works. It only needs to know what it is called and what it does. This is how programmers manage programs that are too large to hold in one thought.

Section 14.2

Decomposition

Decomposition means breaking a large problem into smaller subproblems, each of which is easier to solve on its own.

You already do this in everyday life. When someone asks you how to make a meal, you do not describe every muscle movement your hands will make. You describe the steps: gather the ingredients, prepare the vegetables, cook the protein, combine everything, plate and serve. Each step is a named piece of the larger task. You think about each piece at the right level of detail.

Programming decomposition works the same way.

Suppose your task is to write a program that reads student names and scores from the user, calculates the class average, identifies the highest score, and displays a summary. That is one big task. But look at the natural pieces:

  • Get the student data from the user
  • Calculate the average of the scores
  • Find the highest score
  • Display the summary

Each of those is a smaller, cleaner task. Each one can become a function. When you put them together, the main program reads almost like a plain-English description of what it does:

Python
def main():
    students = get_student_data()
    average = calculate_average(students)
    highest = find_highest(students)
    display_summary(students, average, highest)

main()

Notice that main() does not show how any of those things work. It only shows what happens, in order, at the level of named ideas. The details live inside each function. This is decomposition working at its best.

What Decomposition Produces
Programs that are easier to work with at every stage
  • ReadEasier to read, because each function has a name that explains its purpose
  • TestEasier to test, because each function can be called and checked in isolation
  • DebugEasier to debug, because problems are localized to one function
  • ModifyEasier to modify, because changing one function does not automatically break others
  • ReuseEasier to reuse, because functions that solve general problems can be used in multiple programs
Study Note

The goal of decomposition is not to make programs longer by adding extra functions for no reason. The goal is to identify the natural structure of the problem and give each piece of that structure its own function.

Section 14.3

Helper Functions

When you decompose a program, you often find that some functions do part of the work while others coordinate the pieces. The functions that do specific, focused work are called helper functions.

A helper function typically:

  • Solves one specific subproblem
  • Is called by another function rather than directly from the main program
  • Has a clear, descriptive name that says exactly what it does
  • Takes only the information it needs and returns only what the caller needs

Here is a small example. Suppose you are building a program that converts temperatures. You could write it as one block:

Python — Version 1: Everything in One Place
choice = input("Convert (C)elsius or (F)ahrenheit? ").upper()
temp = float(input("Enter temperature: "))

if choice == "C":
    result = (temp * 9 / 5) + 32
    print(f"{temp}°C = {result:.1f}°F")
elif choice == "F":
    result = (temp - 32) * 5 / 9
    print(f"{temp}°F = {result:.1f}°C")
else:
    print("Invalid choice.")

This works for a simple case. But what happens if you want to convert temperatures in three different places in your program? You would have to copy this logic each time. And if the formula ever needed to change, you would have to find every copy.

Now look at the same program decomposed with helper functions:

Python — Version 2: Decomposed with Helper Functions
def celsius_to_fahrenheit(celsius):
    return (celsius * 9 / 5) + 32

def fahrenheit_to_celsius(fahrenheit):
    return (fahrenheit - 32) * 5 / 9

def get_temperature():
    return float(input("Enter temperature: "))

def get_conversion_choice():
    return input("Convert (C)elsius or (F)ahrenheit? ").upper()

def display_conversion(original, result, from_unit, to_unit):
    print(f"{original}°{from_unit} = {result:.1f}°{to_unit}")

def convert_temperature():
    choice = get_conversion_choice()
    temp = get_temperature()

    if choice == "C":
        result = celsius_to_fahrenheit(temp)
        display_conversion(temp, result, "C", "F")
    elif choice == "F":
        result = fahrenheit_to_celsius(temp)
        display_conversion(temp, result, "F", "C")
    else:
        print("Invalid choice.")

convert_temperature()

This version is longer. That might seem like a problem. But read convert_temperature() on its own. It reads almost like a plain description of what happens: get the choice, get the temperature, do the right conversion, display the result. You do not need to understand the math to follow the logic.

More importantly: celsius_to_fahrenheit and fahrenheit_to_celsius are now reusable anywhere in a larger program. If you later build a program that processes weather data from a file, you can call those same functions. Nothing needs to change.

Good Helper Functions

Do one thing. Have a name that says what that one thing is. Take parameters for the information they need. Return values instead of printing (usually). Can be tested independently.

Section 14.4

Top-Down Design

One useful strategy for decomposition is called top-down design. In top-down design, you start from the big picture and break it into pieces before you write any code.

The process works like this:

  • Describe the program's overall purpose in one sentence.
  • Identify the major steps the program must take.
  • For each major step, ask: does this step have its own sub-steps?
  • Keep breaking down until each piece is small enough to write as a single, focused function.
  • Write and test the functions from the smallest pieces up.

Here is an example. Suppose you need to write a program that processes a list of exam scores, reports the average, and tells each student whether they passed or failed.

Step 1 — Overall Purpose

Process exam scores and report results for each student.

Step 2 — Major Steps
  • Collect student names and scores
  • Calculate the class average
  • For each student, determine pass or fail
  • Display the full report
Step 3 — Sub-Steps for Each

Collect student names and scores: ask how many students; for each student, ask for name and score; store them together.

Calculate the average: add up all the scores; divide by the number of students.

Determine pass or fail: compare each student's score to the passing threshold (say, 60); return a label, "Pass" or "Fail".

Display the full report: show the class average; for each student, show name, score, and pass/fail result.

Step 4 — Functions
collect_students()
calculate_average(scores)
pass_or_fail(score)
display_report(students, average)
Step 5 — Main Program
def main():
    students = collect_students()
    scores = [s["score"] for s in students]
    average = calculate_average(scores)
    display_report(students, average)

main()

The top-down approach gives you a plan before you write code. This is important because without a plan, it is easy to start writing and then discover that your functions need information they do not have, or that two functions are doing the same work, or that your program structure does not match the problem structure.

Experienced programmers do not always write out the decomposition formally on paper — but they do think through it, often before they type the first line.

Section 14.5

Bottom-Up Testing

After you have your decomposition plan, the most reliable way to build the program is to start from the smallest pieces and test each one before combining them.

This is called bottom-up testing or sometimes incremental development.

Here is why this order matters. If you write all the functions at once and then run the program for the first time, you might get an error. Where is it? Could be anywhere. You have no way to know which piece is working and which is broken. Debugging becomes a search through the entire program.

But if you write and test one function at a time, starting from the smallest helpers, you always know what is working. When a new function fails, you know the problem is in that function — not in something you wrote three days ago.

Bottom-Up Testing in Practice

Write and test the smallest, most independent helpers first. For example, if your program uses a function that checks whether a score is passing:

Python
def pass_or_fail(score):
    if score >= 60:
        return "Pass"
    else:
        return "Fail"

Test it immediately, before you write anything else:

Python
print(pass_or_fail(85))   # Should print: Pass
print(pass_or_fail(45))   # Should print: Fail
print(pass_or_fail(60))   # Should print: Pass

These small test calls confirm the function works. Now you can rely on it. When you later call pass_or_fail inside a larger function, you do not need to wonder whether the problem is here — you already know it works.

Then write and test calculate_average:

Python
def calculate_average(scores):
    if len(scores) == 0:
        return 0
    return sum(scores) / len(scores)

print(calculate_average([80, 90, 70]))   # Should print: 80.0
print(calculate_average([100, 50]))      # Should print: 75.0
print(calculate_average([]))             # Should print: 0

Notice the empty list test. When you test functions in isolation, it is natural to think about edge cases: what happens if the list is empty? Testing that now means you will not be surprised by it later.

Write and test each function, working your way up to the full program. When you finally assemble everything in main(), most of the bugs will already be gone.

Section 14.6

Single Responsibility

One of the clearest rules for writing good functions is the single responsibility principle: each function should do one thing.

This sounds simple. In practice, it takes discipline, because it is tempting to combine related things into one function when writing quickly.

Look at this function:

Python
def process_student(name, score):
    grade = ""
    if score >= 90:
        grade = "A"
    elif score >= 80:
        grade = "B"
    elif score >= 70:
        grade = "C"
    elif score >= 60:
        grade = "D"
    else:
        grade = "F"
    print(f"{name}: {score} ({grade})")
    total = score + 10   # bonus points
    print(f"With bonus: {total}")

This function is doing at least three things: calculating a letter grade, printing results, and applying a bonus. Each of those is a separate responsibility. When you want to change how grades are calculated, you have to edit a function that also handles printing and bonuses. When you want to reuse the grade calculation somewhere else, you cannot — it is stuck inside a function that also prints things.

Compare that to three focused functions:

Python
def letter_grade(score):
    if score >= 90:
        return "A"
    elif score >= 80:
        return "B"
    elif score >= 70:
        return "C"
    elif score >= 60:
        return "D"
    else:
        return "F"

def apply_bonus(score, bonus=10):
    return score + bonus

def display_student_result(name, score):
    grade = letter_grade(score)
    bonus_score = apply_bonus(score)
    print(f"{name}: {score} ({grade}), with bonus: {bonus_score}")

Now each function does exactly one thing. letter_grade can be used anywhere in the program that needs a grade. apply_bonus can be adjusted without touching display logic. display_student_result reads clearly because its sub-steps have names.

A Useful Test for Single Responsibility

If you struggle to write a one-line description of what a function does, and that description includes the word and, the function might be doing too much.

"Calculates the letter grade" — clear, single responsibility.

"Calculates the letter grade and displays the result and applies the bonus" — this function is trying to do too much.

The single responsibility principle does not mean every function must be tiny. Some functions genuinely coordinate several steps, and that coordination is their one responsibility. A main() function that calls several helpers is doing one thing: running the program in order. What the principle prevents is functions that mix concerns — that calculate and display, or validate and format, or compute and save, all in one block.

Section 14.7

Naming Functions Clearly

Function names are not just labels. They are documentation. When you read code you wrote three weeks ago, a well-named function tells you exactly what it does without requiring you to read its body.

Use a Verb That Describes What the Function Does

Python — Unclear vs. Clear
# Unclear
def data(score):
    ...

def result(name, score):
    ...

# Clear
def calculate_grade(score):
    ...

def display_student_result(name, score):
    ...

Predicate Names for True/False Functions

For functions that return a True/False answer, start the name with is_, has_, or can_:

Python
def is_passing(score):
    return score >= 60

def has_bonus_eligibility(score):
    return score >= 85

def is_valid_age(age):
    return 0 <= age <= 120

These names make conditions read like natural language:

Python
if is_passing(score):
    print("Congratulations!")

if has_bonus_eligibility(score):
    print("You qualify for a bonus.")

Specific Names Rather Than Generic Ones

Python — Too Generic vs. Specific
# Too generic
def process(x):
    ...

def calculate(a, b):
    ...

# Specific
def process_exam_score(score):
    ...

def calculate_discounted_price(price, discount_rate):
    ...

Avoid Abbreviations Unless Universally Understood

Python
# Unclear abbreviations
def calc_avg_sc(sc_list):
    ...

# Clear
def calculate_average_score(scores):
    ...

Keep names honest. A function named get_user_name should return a name. If it also validates the name and saves it to a file, the name is misleading.

Study Note

Good function names make programs readable without comments. When you find yourself writing a comment to explain what a function does, consider whether a better name would make the comment unnecessary.

Section 14.8

Avoiding Duplicated Code

Duplicated code is one of the clearest warning signs that a program needs refactoring. When you see the same logic written in two or more places, any future change to that logic must be made in every copy. Forgetting to update one copy creates a bug that is especially hard to find.

Here is a program with duplicated code:

Python — Duplicated Logic
print("--- Morning Report ---")
total = 0
count = 0
for score in morning_scores:
    total += score
    count += 1
if count > 0:
    average = total / count
    print(f"Average: {average:.1f}")
else:
    print("No scores.")

print("--- Afternoon Report ---")
total = 0
count = 0
for score in afternoon_scores:
    total += score
    count += 1
if count > 0:
    average = total / count
    print(f"Average: {average:.1f}")
else:
    print("No scores.")

The calculation block is repeated exactly. If you decide to round to two decimal places instead of one, you have to change it in two places. If you have morning, afternoon, and evening reports, you would have three copies.

The fix is to extract the repeated pattern into a function:

Python — Extracted into a Function
def print_score_report(label, scores):
    print(f"--- {label} ---")
    if len(scores) == 0:
        print("No scores.")
        return
    average = sum(scores) / len(scores)
    print(f"Average: {average:.1f}")

print_score_report("Morning Report", morning_scores)
print_score_report("Afternoon Report", afternoon_scores)

Now the logic exists in exactly one place. Change it once and the change applies everywhere the function is called.

Warning Signs
Common signs that code should be extracted into a function
  • Copy-pasteYou have copied and pasted a block of code
  • Repeating patternYou see the same pattern appearing in multiple places with only small differences (usually a name, value, or parameter)
  • Needs a commentA block of code is long enough to need a comment explaining what it does — that comment is probably a function name waiting to be written
  • Hard to findYou find yourself searching through a long function to find where a specific piece of logic is
The Rule of Thumb

If you write the same thing twice, consider a function. If you write it three times, write a function.

Section 14.9

Planning a Program with Functions

Before you start writing code for a program of any real size, it helps to plan the function structure. This does not need to be formal. A few notes or a quick sketch is enough.

Here is a planning process you can use:

Step 1 — Describe the Program in Plain Language

Write one or two sentences describing what the program does, not how it works.

Example: The program asks a user to enter a list of grades, then displays the average, highest grade, lowest grade, and how many students passed.

Step 2 — Identify the Main Tasks

List the things the program must do. Do not write code yet — just identify the tasks.

  • Collect grades from user input
  • Calculate the average
  • Find the highest and lowest grade
  • Count how many grades are passing
  • Display the full report
Step 3 — Decide Which Tasks Become Functions

Each major task is likely a function. Some tasks have sub-steps that become helper functions.

get_grades()               → returns a list of grades
calculate_average(grades)  → returns the average
find_highest(grades)       → returns the highest grade
find_lowest(grades)        → returns the lowest grade
count_passing(grades)      → returns the number of passing grades
display_report(grades)     → prints the full summary
main()                     → calls everything in order
Step 4 — Sketch the Main Function

Before writing the helpers, sketch what main() will look like:

def main():
    grades = get_grades()
    display_report(grades)

main()

Or, if display_report needs the computed values passed to it:

def main():
    grades = get_grades()
    average = calculate_average(grades)
    highest = find_highest(grades)
    lowest = find_lowest(grades)
    passing_count = count_passing(grades)
    display_report(grades, average, highest, lowest, passing_count)

main()
Step 5 — Write and Test Functions from the Bottom Up

Start with the simplest helpers. Test each one. Then move to functions that call other functions. Finish with main().

Study Note

This planning step takes only a few minutes for small programs. For larger programs, it can save hours of frustrating debugging later. A plan does not need to be perfect — it will change as you build — but starting with a structure is far better than writing code without one.

Section 14.10

Refactoring Repeated Code

Refactoring means improving the structure of code without changing what it does. A program that was written quickly and works correctly might still benefit from refactoring — especially if it has duplicated code, unclear names, functions that do too much, or logic that would be hard to change later.

Refactoring is not rewriting. You do not throw away working code. You reorganize it, give things better names, and extract repeated patterns into functions. The behavior of the program stays exactly the same.

Here is a before-and-after refactoring example.

Python — Before Refactoring
name1 = input("Enter student 1 name: ")
score1 = float(input("Enter student 1 score: "))
if score1 >= 60:
    label1 = "Pass"
else:
    label1 = "Fail"
print(f"{name1}: {score1} — {label1}")

name2 = input("Enter student 2 name: ")
score2 = float(input("Enter student 2 score: "))
if score2 >= 60:
    label2 = "Pass"
else:
    label2 = "Fail"
print(f"{name2}: {score2} — {label2}")

name3 = input("Enter student 3 name: ")
score3 = float(input("Enter student 3 score: "))
if score3 >= 60:
    label3 = "Pass"
else:
    label3 = "Fail"
print(f"{name3}: {score3} — {label3}")

This program works, but it has three identical blocks. Adding a fourth student means copying the block again. Changing the passing threshold from 60 to 65 means updating three places.

Python — After Refactoring
PASSING_SCORE = 60

def get_student():
    name = input("Enter student name: ")
    score = float(input("Enter student score: "))
    return name, score

def pass_or_fail(score):
    if score >= PASSING_SCORE:
        return "Pass"
    return "Fail"

def display_result(name, score):
    label = pass_or_fail(score)
    print(f"{name}: {score} — {label}")

def process_students(count):
    for _ in range(count):
        name, score = get_student()
        display_result(name, score)

process_students(3)

The refactored version is shorter, clearer, and easier to change. Adjusting the number of students is one argument. Changing the passing threshold is one variable. Adding a new display format means changing one function.

Method
A systematic approach to refactoring
  • Step 1Make sure the original code works before you start. Do not refactor broken code.
  • Step 2Identify the repeated pattern.
  • Step 3Decide what the pattern needs as input and what it produces as output.
  • Step 4Extract the pattern into a function with a clear name.
  • Step 5Replace every copy of the pattern with a call to the new function.
  • Step 6Run the program again and confirm it still works correctly.
  • Step 7Repeat for the next opportunity.
Study Note

Refactoring is a habit, not a one-time event. Professional programmers return to old code and improve it. Programs that are never refactored tend to become harder and harder to maintain over time.

Worked Example

Planning and Building a Grade Report Program

This worked example walks through the full process: planning with decomposition, writing helper functions, testing bottom-up, and producing a clean final program.

The Task

Write a program that:

  • Asks the user how many students are in the class
  • For each student, asks for their name and score
  • Calculates and displays the class average
  • Identifies and displays the highest and lowest scores
  • Shows each student's result as Pass or Fail (passing is 60 or above)
Step 1 — Plan the Functions
collect_students(count)     → returns list of (name, score) pairs
calculate_average(scores)   → returns average as a float
find_highest(scores)        → returns highest score
find_lowest(scores)         → returns lowest score
pass_or_fail(score)         → returns "Pass" or "Fail"
display_report(students)    → prints the full report
main()                      → coordinates everything
Step 2 — Write and Test the Simple Helpers
Python
def pass_or_fail(score):
    if score >= 60:
        return "Pass"
    return "Fail"

# Quick tests
print(pass_or_fail(75))   # Pass
print(pass_or_fail(50))   # Fail
print(pass_or_fail(60))   # Pass
Python
def calculate_average(scores):
    if len(scores) == 0:
        return 0.0
    return sum(scores) / len(scores)

# Quick tests
print(calculate_average([80, 90, 100]))   # 90.0
print(calculate_average([55, 65]))         # 60.0
print(calculate_average([]))               # 0.0
Python
def find_highest(scores):
    if len(scores) == 0:
        return None
    return max(scores)

def find_lowest(scores):
    if len(scores) == 0:
        return None
    return min(scores)

# Quick tests
print(find_highest([70, 85, 60, 95]))    # 95
print(find_lowest([70, 85, 60, 95]))     # 60
Step 3 — Write the Data Collection Function
Python
def collect_students(count):
    students = []
    for i in range(count):

Chapter 14 — Back Matter

Chapter Summary

  • Section 14.1Programs grow in complexity as they add features, data, and logic. Functions are the primary tool for managing that complexity by creating named abstractions — pieces the rest of the program can rely on without knowing how they work internally.
  • Section 14.2Decomposition means identifying the natural structure of a problem and assigning each piece its own function. Decomposed programs are easier to read, test, debug, modify, and reuse.
  • Section 14.3Helper functions do focused, specific work. They are called by other functions and typically have clear, descriptive names, narrow parameters, and return values rather than side effects.
  • Section 14.4Top-down design starts from the big picture and breaks problems into smaller pieces before writing code, following a five-step process from overall purpose down to a sketch of main().
  • Section 14.5Bottom-up testing builds and tests functions from the simplest helpers first, building confidence and isolating bugs before combining pieces into larger structures.
  • Section 14.6The single responsibility principle says each function should do one thing. Functions that mix concerns — calculating and displaying, validating and saving — are harder to reuse, test, and change.
  • Section 14.7Naming functions clearly is not cosmetic. Verb-based names, predicate names like is_passing, and specific rather than generic names all contribute to programs that explain themselves without comments.
  • Section 14.8Duplicated code is a warning sign. When the same logic appears in multiple places, a future change must be made in every copy. Extracting repeated patterns into functions eliminates duplication and makes programs easier to maintain.
  • Section 14.9A five-step planning process — plain-language description, major tasks, function assignment, sketching main(), then bottom-up implementation — turns a vague program idea into a clear, testable structure before any code is written.
  • Section 14.10Refactoring is the practice of improving the structure of working code without changing its behavior. Refactoring is a normal, healthy part of programming that prevents complexity from accumulating over time.

Key Terms

decomposition
The practice of breaking a large problem into smaller, more manageable subproblems, each of which can be solved independently.
helper function
A function that performs a specific, focused piece of work and is called by other functions rather than directly by the user or the main program.
top-down design
A planning approach in which you identify the overall structure of a program and break it into smaller pieces before writing any code.
bottom-up testing
A development approach in which you write and test the smallest, most independent functions first, then combine them into larger structures.
single responsibility principle
The guideline that each function should have one clearly defined purpose.
refactoring
Improving the internal structure of existing, working code without changing its external behavior.
duplicated code
Code that appears in two or more places in a program with only minor differences; a sign that the repeated logic should be extracted into a function.
abstraction
Hiding the details of how something works inside a function so that the rest of the program only needs to know what it does, not how.
main function
A function, typically named main(), that coordinates the overall flow of a program by calling other functions in order.

Mini-ProjectMenu-Driven Utility Program

Project Description

Build a menu-driven utility program that offers the user a set of tools, accepts their choice, runs the chosen tool, and returns to the menu until the user decides to quit.

Your program must be designed around functions. The program structure should be visible in main() without needing to read every line of every function.

Requirements

The menu must offer at least four of the following tools:

  • Temperature converter (Celsius ↔ Fahrenheit)
  • Tip calculator (bill amount, tip percent, number of people)
  • Word analyzer (count vowels, consonants, length)
  • Grade calculator (multiple scores → average and letter grade)
  • Loan interest estimator (principal, annual rate, years → total interest owed)
  • BMI calculator (weight in kg, height in m → BMI and category)
  • Password strength checker (checks length, uppercase, lowercase, digits)

Program Structure Requirements

  • Each tool must be implemented using at least two functions (a calculation function and a display or input function)
  • No calculation logic should appear inside main()
  • No function should have more than one primary responsibility
  • All function names must be clear and descriptive
  • Duplicated code is not acceptable — if two tools share a pattern, extract it

Menu Requirements

  • The menu must display all available options
  • The user must be able to choose a tool by number
  • After a tool finishes, the menu must display again
  • The user must be able to quit cleanly

Robustness

  • Invalid menu choices must produce a helpful message and return to the menu
  • Numeric inputs that fail conversion must be caught and handled

Chapter 14 taught you to organize programs using functions you write yourself.
