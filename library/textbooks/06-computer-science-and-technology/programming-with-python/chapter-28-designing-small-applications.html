Chapter 28 — Front Matter

Chapter Introduction and Learning Objectives

You have spent the last several chapters building skill with individual tools: classes, objects, methods, attributes, file input and output, exception handling, functions, and data structures. You can write a function that returns a value. You can define a class that holds data and provides methods. You can open a file and read records from it.

Now it is time to combine those tools into something larger. This chapter is about application design — the process of thinking through what a program needs to do before writing most of its code, organizing the pieces so they work together cleanly, and building toward a working program step by step rather than all at once.

A small console application is a complete program that a real user can run, interact with through text menus, and use to accomplish something useful. It might manage a list of contacts. It might track a personal budget. It might keep a log of exercise sessions. The exact purpose is less important than the process: you learn to plan, design, build, test, and document a program that holds together. This chapter prepares you for Chapter 30, where you will build your own capstone project from start to finish.

Chapter Purpose

Every earlier chapter gave you one or two tools to practice in isolation. This chapter teaches you how to assemble those tools into a program that does something real. The contact manager built across this chapter is not a toy — it is a working application you could actually use. Study it as a complete design, not just as a collection of code snippets.

Section 28.1

From Exercises to Applications

Throughout this book, most programs you wrote were exercises. An exercise is designed to practice one skill at a time. It is short, focused, and complete when it produces the right output. Exercises are essential. They build the fluency you need to write larger programs.

But applications are different from exercises in several important ways.

Difference 1
Applications serve a user
An exercise exists to demonstrate that you can write a loop or define a class. An application exists because someone finds it useful. The user's needs shape every decision you make.
Difference 2
Applications run repeatedly
An exercise runs once and finishes. An application runs every day, or whenever the user needs it. It must handle many different inputs and store data between sessions.
Difference 3
Applications have multiple parts
An exercise is usually one function or loop. An application has menus, data structures, file operations, validation, multiple functions, and possibly classes. A design step helps you understand what those parts are before writing.
Difference 4
Applications can fail subtly
If an exercise produces wrong output, you see it immediately. An application can quietly store bad data or miss edge cases. Testing a complete application requires more thought than testing a single function.

None of this means applications are beyond you. It means they require a different working style: a little more planning, a little more structure, a little more patience before writing code. The good news is that the skills transfer directly. Every application is built from the same loops, functions, classes, files, and data structures you have already studied. Designing an application is learning to arrange those pieces well.

The Key Shift

In exercises, you wrote code to satisfy a specification given to you. In applications, you must first figure out the specification yourself — by writing requirements. The rest of application design flows from that step.

Section 28.2

Requirements

A requirement is a statement of what a program must do. Requirements are written before code. They describe the program from the outside, from the user's perspective, without specifying how the program works internally. Good requirements answer three questions: What can the user do with this program? What information does the program manage? What should happen in unusual situations?

Consider a simple contact manager. A beginner might describe it as: "A program that stores contacts." That is a direction, not a requirement. Here is a more useful set:

Example 28.2.1
Contact Manager Requirements
  • FeaturesThe program must allow the user to: (1) add a new contact with name, phone, and email; (2) view all contacts in alphabetical order by name; (3) search for a contact by name; (4) delete a contact by name; (5) quit the program.
  • PersistenceThe program must save contacts to a file so they persist between sessions. It must load contacts from the file when the program starts. It must handle the case where no file exists yet (first run).
  • Error casesThe program must reject a new contact if the name field is empty. It must inform the user if a search finds no matching contact.

These requirements are specific enough to guide design. They list features, describe the information involved, and mention edge cases. They do not say anything about which functions to write or how the menu looks. Those are implementation decisions, not requirements.

Writing requirements before coding has several benefits: you discover missing features before you have written code around wrong assumptions; you have a checklist for testing; and if you are working with an instructor or collaborator, requirements give you a shared understanding of what you are building.

Common Requirement Mistakes

MistakePoor ExampleBetter Version
Too vague"The program should work well.""The program must display an error message if the user enters a name with no characters."
Mixing in implementation"The program should use a list of dictionaries.""The program must store contacts with a name, phone, and email field." (How is your choice.)
Forgetting error casesRequirements cover only the happy path.Add explicit requirements for bad input, missing files, and empty search results.
Section 28.3

User Stories at an Introductory Level

A user story is a short sentence that describes a feature from the user's point of view. The format is:

User Story Format

As a [user], I want to [do something] so that [I get some benefit].

For a beginner project, you can use a simplified version and drop the "so that" clause if the benefit is obvious.

User stories help you think about features from the outside rather than from the implementation. When you ask "what does the user want?" you stay focused on the actual purpose of the program. Here are user stories for a simple budget tracker:

  • →As a user, I want to add an expense with a description and amount.
  • →As a user, I want to see a list of all my expenses.
  • →As a user, I want to see my total spending so far.
  • →As a user, I want to clear all expenses and start fresh.
  • →As a user, I want my expenses saved so they are still there tomorrow.

Each story maps to one or more requirements. The first story becomes: "The program must allow the user to add an expense with a text description and a dollar amount. The amount must be a positive number." User stories are most useful when you are not sure what features to include. If you already have a clear picture, you can move directly to requirements.

Section 28.4

Designing Program Flow

Program flow is the sequence of steps a running program goes through. For a console application, the flow almost always follows the same pattern: load data, show menu, get choice, perform action, repeat until quit. This is called the application loop.

Before writing code, sketch the flow in plain text. Here is a sketch for the contact manager:

Start
  Load contacts from file (or start with empty list if no file)

  Loop:
    Display menu:
      1. Add contact
      2. View all contacts
      3. Search for contact
      4. Delete contact
      5. Quit
    Get user choice

    If choice is 1: collect name, phone, email -> add to contacts list
    If choice is 2: display all contacts sorted by name
    If choice is 3: ask for name -> search and display result
    If choice is 4: ask for name -> remove from contacts list
    If choice is 5: save contacts to file -> exit loop

End

This sketch is not code yet. It is a plan. Reading it against your requirements reveals decisions you still need to make: What data structure holds contacts? What file format? How do you sort alphabetically? What happens for an invalid menu choice? Answering these questions during design, rather than during coding, makes the coding phase much smoother.

The Main Loop in Python

Translating the sketch to Python produces a skeleton — short but complete in structure:

def main():
    contacts = load_contacts()

    running = True
    while running:
        print_menu()
        choice = input("Enter your choice: ").strip()

        if choice == "1":
            add_contact(contacts)
        elif choice == "2":
            view_contacts(contacts)
        elif choice == "3":
            search_contacts(contacts)
        elif choice == "4":
            delete_contact(contacts)
        elif choice == "5":
            save_contacts(contacts)
            print("Contacts saved. Goodbye.")
            running = False
        else:
            print("Invalid choice. Please enter 1 through 5.")

main()
Keep main() Short

The main() function manages flow and delegates to named functions. It does not implement any feature itself. If main() contains all the logic for every feature, it becomes long, hard to read, and hard to test. Delegating to named functions keeps each feature isolated and testable.

Section 28.5

Menus and User Interaction

A menu is how a console application communicates its options and collects the user's choice. A good menu is clear, consistent, and forgiving of small mistakes.

Displaying a Menu

def print_menu():
    print()
    print("--- Contact Manager ---")
    print("1. Add contact")
    print("2. View all contacts")
    print("3. Search for contact")
    print("4. Delete contact")
    print("5. Quit")
    print()

Putting the menu in a function means you call it at the top of every loop iteration without repeating the print statements. Blank lines before and after give breathing room in the terminal, especially after a long list of contacts.

Collecting and Validating a Choice

Always validate the user's menu choice. A user might type a letter, a space, or a number that is not on the menu:

def get_menu_choice(valid_choices):
    """Ask for a choice until the user enters a valid one."""
    while True:
        choice = input("Enter your choice: ").strip()
        if choice in valid_choices:
            return choice
        print(f"Please enter one of: {', '.join(valid_choices)}")

Collecting Input for a Feature

Each feature has its own input-collection step. Notice what add_contact() does and does not do — it collects and validates input, builds the dictionary, appends it, and prints a confirmation. It does not print the menu and does not save the file. Each function has exactly one job:

def add_contact(contacts):
    """Collect contact information and add it to the list."""
    print("\n--- Add Contact ---")

    name = input("Name: ").strip()
    if not name:
        print("Name cannot be empty. Contact not added.")
        return

    phone = input("Phone: ").strip()
    email = input("Email: ").strip()

    contact = {"name": name, "phone": phone, "email": email}
    contacts.append(contact)
    print(f"Contact '{name}' added.")
Always Give Feedback

After adding a contact, print a confirmation. After a search finds nothing, say so. After deleting, confirm the deletion. Silence leaves users wondering whether their action worked. Every feature function should end with a message to the user, whether the operation succeeded or failed.

Section 28.6

Choosing Data Structures

Before writing any feature code, decide how your program will represent its data. A poor data structure can make every operation awkward. A good one makes most operations natural. Consider three representations for contacts:

OptionWhat It Looks LikeBest When
List of dictionaries [{"name": "Alice", "phone": "555-1234", "email": "…"}, …] Data is flat. Operations are mostly iteration. CSV saving is natural.
List of objects [Contact("Alice", "555-1234", "…"), …] The data has meaningful behavior — display methods, validation, formatting.
Dictionary keyed by name {"Alice": {"phone": "555-1234", …}, …} Lookup by name must be very fast and names are guaranteed unique.

For this chapter's contact manager, a list of dictionaries (or a list of objects) is the right choice — simple, flexible, easy to iterate, and maps naturally to CSV rows. Use the simplest structure that satisfies your requirements. You can always refactor to a more complex structure later if you need it.

General Principles
When to reach for each data structure
  • ListWhen order matters or when you need to iterate over all items.
  • DictionaryWhen you need fast lookup by a named key.
  • ClassWhen the data has associated behavior — methods that belong to the item, not to the collection.
  • TupleWhen the data is fixed, positional, and should not change.
  • SetWhen you only care about membership and uniqueness, not order or repetition.
Section 28.7

Combining Functions and Classes

A clean pattern for small applications is: use a class to represent a single piece of data (one contact, one expense, one task), and use standalone functions for operations on collections of that data (add to list, search list, display list). This keeps each class focused on one thing and keeps the application logic in functions that can be called from main().

class Contact:
    """Represents a single contact with name, phone, and email."""

    def __init__(self, name, phone, email):
        self.name = name
        self.phone = phone
        self.email = email

    def display(self):
        """Print this contact's information."""
        print(f"  Name:  {self.name}")
        print(f"  Phone: {self.phone}")
        print(f"  Email: {self.email}")

    def to_dict(self):
        """Return this contact as a dictionary for saving."""
        return {"name": self.name, "phone": self.phone, "email": self.email}

The display() and to_dict() methods are behaviors specific to a contact, so a class is appropriate. Every function that shows a contact just calls c.display() rather than repeating print statements. Every function that saves contacts just calls c.to_dict().

When to Avoid Classes

Do not use classes just to feel like you are writing object-oriented code. If your data has no meaningful behavior beyond __init__, a dictionary is simpler. Classes are most valuable when the data has behavior attached to it.

Section 28.8

Saving and Loading Data

A program that forgets everything when it closes is not very useful. Most practical applications need persistence: the ability to save data between sessions. For a beginner application, CSV is an excellent choice — easy to read and write with Python's csv module, human-readable in a text editor, and familiar from spreadsheets.

import csv
import os

FILENAME = "contacts.csv"
FIELDNAMES = ["name", "phone", "email"]


def save_contacts(contacts):
    """Save all contacts to a CSV file."""
    with open(FILENAME, "w", newline="") as file:
        writer = csv.DictWriter(file, fieldnames=FIELDNAMES)
        writer.writeheader()
        for c in contacts:
            writer.writerow(c.to_dict())
    print(f"Contacts saved to {FILENAME}.")


def load_contacts():
    """Load contacts from CSV file. Return empty list if file does not exist."""
    contacts = []
    if not os.path.exists(FILENAME):
        return contacts
    with open(FILENAME, "r", newline="") as file:
        reader = csv.DictReader(file)
        for row in reader:
            contacts.append(Contact(row["name"], row["phone"], row["email"]))
    return contacts
Three Details Worth Noting

save_contacts() opens in write mode ("w"), overwriting the file — correct, because you are saving the complete current state. load_contacts() uses os.path.exists() before opening, handling the first-run case without crashing. The newline="" argument is required by Python's csv module to prevent extra blank lines on Windows.

When to Save

StrategyHow It WorksBest For
Save on quitCall save_contacts() only when the user selects Quit.Simple programs. Acceptable risk of losing data if the program crashes.
Save after every changeCall save_contacts() after every add, edit, or delete.Programs where data loss is unacceptable. Slightly slower but much safer.

For a beginner project, saving on quit is fine. JSON is a useful alternative format — especially when your data has nested structures that do not fit neatly into rows and columns — but CSV is simpler and human-readable for flat records like contacts.

Section 28.9

Testing the Application

Testing a complete application requires more effort than testing a single function, because an application has many paths through it. The most reliable approach for a beginner application is a manual testing checklist based on your requirements. Work through every item deliberately.

Example 28.9.1
Contact Manager Manual Testing Checklist
  • SetupRun with no contacts file — does it start without errors? Does the menu display correctly?
  • AddingAdd a contact — does it appear in View All? Try adding with empty name — is it rejected? Add multiple contacts — do all appear?
  • ViewingView all — are contacts sorted alphabetically? View all when empty — does the program say so?
  • SearchingSearch for existing name — correct contact appears? Search partial name — partial matches found? Search missing name — "not found" message?
  • DeletingDelete existing contact — disappears from View All? Delete missing name — handled gracefully?
  • PersistenceAdd contacts, quit, restart — are contacts still there? Delete a contact, quit, restart — is it still gone?
  • MenuEnter invalid choice — does program ask again? Enter letters instead of number — handled?

Work through every item. When something fails, fix it and recheck that item and any related items. Test one feature at a time during development, not only at the end. If you just wrote add_contact(), test it immediately. Catching a bug in a 20-line function is much easier than finding it later in a 200-line program.

Thinking About Edge Cases

Edge cases are inputs at the boundary of normal behavior. You do not have to handle every possible case in a beginner project, but you should think through the most likely ones and decide whether to handle them or to note them as known limitations. For example: "This program does not support contacts with the same name. If two contacts have identical names, both will appear in search results and the first match will be deleted." That is an honest, useful note.

Section 28.10

Documentation and User Instructions

Documentation is writing that helps users and future programmers understand your program. For a small console application, you need two kinds: user documentation (instructions for running and using the program) and code documentation (comments and docstrings that explain how the code works).

User Documentation — the README

A short README file in the same folder as your program is enough. It should answer: What does this program do? How do I run it? What can I do in the program? Where is the data saved? Are there known limitations?

Contact Manager
===============

A simple console application for storing and retrieving contact information.

How to run:
  python contact_manager.py

What you can do:
  1. Add a new contact (name, phone, email)
  2. View all contacts in alphabetical order
  3. Search for a contact by name
  4. Delete a contact by name
  5. Quit (contacts are saved automatically)

Data storage:
  Contacts are saved in contacts.csv in the same folder as the program.

Known limitations:
  - Two contacts with the same name may cause unexpected behavior when deleting.
  - Phone and email are stored as text and are not validated for format.

Code Documentation

A docstring is a string literal placed immediately after a function or class definition. Python treats it as the function's documentation. A comment explains why the code does something, not just what it does:

def search_contacts(contacts):
    """
    Search for contacts by partial name match.

    Parameters:
        contacts (list): A list of Contact objects.

    Prints all contacts whose names contain the search term.
    Returns nothing.
    """
    ...

# Use newline="" to prevent csv module from adding extra blank lines on Windows
with open(FILENAME, "w", newline="") as file:
    ...
What Goes Where

Good code documentation does not repeat what the code already says clearly. You do not need a comment explaining that contacts.append(new_contact) appends a contact. You do need a comment explaining a non-obvious decision, a workaround, or a limitation. For a beginner project, aim for: a module-level docstring, a docstring on every function and class, and comments on any code that is not immediately obvious.

Worked Example

Complete Contact Manager — Full Source and Breakdown

Here is the complete contact manager, putting all the pieces from sections 28.1–28.10 together. Study it as a complete design before reading the breakdown below.

"""
contact_manager.py

A simple console application for storing and managing contacts.
Contacts are saved to and loaded from contacts.csv.

Usage:
    python contact_manager.py
"""

import csv
import os

FILENAME = "contacts.csv"
FIELDNAMES = ["name", "phone", "email"]


# ── Data Model ────────────────────────────────────────────────────────────────

class Contact:
    """Represents a single contact with name, phone, and email."""

    def __init__(self, name, phone, email):
        self.name = name
        self.phone = phone
        self.email = email

    def display(self):
        """Print this contact's information."""
        print(f"  Name:  {self.name}")
        print(f"  Phone: {self.phone}")
        print(f"  Email: {self.email}")

    def to_dict(self):
        """Return this contact as a dictionary (for CSV saving)."""
        return {"name": self.name, "phone": self.phone, "email": self.email}


# ── File Operations ───────────────────────────────────────────────────────────

def load_contacts():
    """
    Load contacts from the CSV file.
    Returns an empty list if the file does not exist.
    """
    contacts = []
    if not os.path.exists(FILENAME):
        return contacts
    with open(FILENAME, "r", newline="") as file:
        reader = csv.DictReader(file)
        for row in reader:
            contacts.append(Contact(row["name"], row["phone"], row["email"]))
    return contacts


def save_contacts(contacts):
    """Save all contacts to the CSV file, overwriting previous data."""
    with open(FILENAME, "w", newline="") as file:
        writer = csv.DictWriter(file, fieldnames=FIELDNAMES)
        writer.writeheader()
        for c in contacts:
            writer.writerow(c.to_dict())


# ── Feature Functions ─────────────────────────────────────────────────────────

def add_contact(contacts):
    """Collect input and add a new Contact to the list."""
    print("\n--- Add Contact ---")
    name = input("Name: ").strip()
    if not name:
        print("Name cannot be empty. Contact not added.")
        return
    phone = input("Phone: ").strip()
    email = input("Email: ").strip()
    contacts.append(Contact(name, phone, email))
    print(f"Contact '{name}' added.")


def view_contacts(contacts):
    """Display all contacts sorted alphabetically by name."""
    print("\n--- All Contacts ---")
    if not contacts:
        print("No contacts stored yet.")
        return
    sorted_contacts = sorted(contacts, key=lambda c: c.name.lower())
    for c in sorted_contacts:
        c.display()
        print()


def search_contacts(contacts):
    """Search for contacts by partial, case-insensitive name match."""
    print("\n--- Search Contacts ---")
    term = input("Search name: ").strip().lower()
    if not term:
        print("No search term entered.")
        return
    results = [c for c in contacts if term in c.name.lower()]
    if not results:
        print("No contacts found matching that name.")
    else:
        print(f"{len(results)} result(s):")
        for c in results:
            c.display()
            print()


def delete_contact(contacts):
    """Remove the first contact whose name exactly matches the input."""
    print("\n--- Delete Contact ---")
    name = input("Enter exact name to delete: ").strip()
    for i, c in enumerate(contacts):
        if c.name.lower() == name.lower():
            contacts.pop(i)
            print(f"Contact '{c.name}' deleted.")
            return
    print(f"No contact named '{name}' found.")


# ── Menu ──────────────────────────────────────────────────────────────────────

def print_menu():
    """Display the main menu."""
    print()
    print("=== Contact Manager ===")
    print("1. Add contact")
    print("2. View all contacts")
    print("3. Search for contact")
    print("4. Delete contact")
    print("5. Quit")
    print()


def get_choice():
    """Ask for a menu choice until the user enters 1 through 5."""
    while True:
        choice = input("Your choice: ").strip()
        if choice in ("1", "2", "3", "4", "5"):
            return choice
        print("Please enter 1, 2, 3, 4, or 5.")


# ── Main ──────────────────────────────────────────────────────────────────────

def main():
    """Run the contact manager application."""
    contacts = load_contacts()
    print(f"Loaded {len(contacts)} contact(s).")

    running = True
    while running:
        print_menu()
        choice = get_choice()

        if choice == "1":
            add_contact(contacts)
        elif choice == "2":
            view_contacts(contacts)
        elif choice == "3":
            search_contacts(contacts)
        elif choice == "4":
            delete_contact(contacts)
        elif choice == "5":
            save_contacts(contacts)
            print("Contacts saved. Goodbye.")
            running = False


main()

Design Breakdown

Structure
Four clear sections
Data model, file operations, feature functions, and main. This organization makes it easy to find any part of the program without searching.
main() length
15 lines
Loads data, runs the loop, delegates to feature functions, and stops. A short, readable main() is a sign of good design.
Single responsibility
Each function does one job
add_contact() adds. save_contacts() saves. get_choice() validates input. No function tries to do too much.
User feedback
Every action gets a response
Every feature function either confirms success or explains the outcome. The user always knows what happened.
Missing file
Handled gracefully
load_contacts() checks for file existence before opening — no try/except needed, because a missing file on first run is expected, not exceptional.
Style
Consistent throughout
All functions have docstrings. All names are lowercase with underscores. Indentation is four spaces. The file is easy to read and extend.

TASKS_FILE = "tasks.txt"

def load_tasks():
    tasks = []
    try:
        with open(TASKS_FILE, "r") as file:
            for line in file:
                line = line.strip()
                if line:
                    tasks.append(line)
    except FileNotFoundError:
        pass
    return tasks

def save_tasks(tasks):
    with open(TASKS_FILE, "w") as file:
        for task in tasks:
            file.write(task + "\n")

def add_task(tasks):
    description = input("Task description: ").strip()
    if description:
        tasks.append(description)
        print("Task added.")
    else:
        print("No description entered.")

def view_tasks(tasks):
    if not tasks:
        print("No tasks.")
        return
    for i, task in enumerate(tasks, start=1):
        print(f"{i}. {task}")

def complete_task(tasks):
    view_tasks(tasks)
    if not tasks:
        return
    try:
        number = int(input("Complete which task number? "))
        if 1 <= number <= len(tasks):
            done = tasks.pop(number - 1)
            print(f"Completed: {done}")
        else:
            print("Invalid number.")
    except ValueError:
        print("Please enter a number.")

def main():
    tasks = load_tasks()
    running = True
    while running:
        print("\n1. Add  2. View  3. Complete  4. Quit")
        choice = input("Choice: ").strip()
        if choice == "1":
            add_task(tasks)
        elif choice == "2":
            view_tasks(tasks)
        elif choice == "3":
            complete_task(tasks)
        elif choice == "4":
            save_tasks(tasks)
            print("Saved. Goodbye.")
            running = False
        else:
            print("Invalid choice.")

DebuggingDebugging Practice

Each program below has a bug. Identify the bug, explain why it causes a problem, and write the corrected code.

Bug 1

def delete_contact(contacts):
    name = input("Name to delete: ").strip()
    for c in contacts:
        if c.name.lower() == name.lower():
            contacts.remove(c)
            print(f"Deleted {c.name}.")
Hint
This code mostly works but has a subtle problem.
Think about what happens if two contacts have the same name, and whether modifying a list while iterating over it can cause issues. The safer approach is to iterate by index (using enumerate) and return as soon as the first match is found, or to collect indices separately before modifying.

Bug 2

def load_contacts():
    contacts = []
    with open("contacts.csv", "r", newline="") as file:
        reader = csv.DictReader(file)
        for row in reader:
            contacts.append(Contact(row["name"], row["phone"], row["email"]))
    return contacts
Hint
What happens if contacts.csv does not exist?
This function will raise a FileNotFoundError on first run when no file exists yet. Add either an os.path.exists() check before opening or a try/except FileNotFoundError block that returns an empty list.

Bug 3

def add_contact(contacts):
    name = input("Name: ").strip()
    phone = input("Phone: ").strip()
    email = input("Email: ").strip()
    if not name:
        print("Name cannot be empty.")
    contacts.append(Contact(name, phone, email))
    print("Contact added.")
Hint
There is a logic error, not a syntax error.
When the name is empty, the function prints an error message but continues to append the contact anyway. The contacts.append() call should be inside an else branch, or the function should return immediately after printing the error.

Chapter Summary

  • Section 28.1Applications differ from exercises in four ways: they serve a user, run repeatedly, have multiple parts, and can fail in slow subtle ways. Designing an application requires a little more planning and structure, but the underlying tools are the same ones you have already learned.
  • Section 28.2Requirements are written before code and describe the program from the user's perspective. Good requirements cover features, the information the program manages, and error cases. Avoid requirements that are too vague, that mix in implementation decisions, or that cover only the happy path.
  • Section 28.3User stories describe features in the format "As a user, I want to [do something]." They help you generate and prioritize requirements by keeping focus on what the user actually needs.
  • Section 28.4Program flow for a console application follows the application loop: load data, show menu, get choice, perform action, repeat until quit. Sketching this flow before writing code reveals design decisions and gaps early. The resulting main() function is short and delegates all feature work to named functions.
  • Section 28.5Menus should be clear and forgiving. Define menu display and choice validation as separate functions. Each feature function collects its own input, validates it, performs its action, and gives the user feedback. Silence after an action is always wrong.
  • Section 28.6Data structure choice shapes the entire program. Use the simplest structure that satisfies your requirements. Lists work for ordered iteration; dictionaries for fast lookup; classes when the data has meaningful behavior; sets for membership and uniqueness.
  • Section 28.7A clean pattern: use classes to represent individual data items, standalone functions to operate on collections. Keep classes focused on one thing. Avoid classes when a dictionary would be simpler.
  • Section 28.8CSV is a reliable, readable format for persistent data. Save on quit for simplicity; save after every change for safety. Always handle the missing-file case on first run.
  • Section 28.9Test a complete application with a manual checklist derived from your requirements. Test incrementally as you build. Think about edge cases and decide whether to handle them or document them as known limitations.
  • Section 28.10Documentation includes user-facing instructions (a README) and code-facing explanations (docstrings and comments). Both matter. A docstring describes what a function does and what it expects. A comment explains why a non-obvious decision was made.

Key Terms

application loop
A while loop in a console application that repeatedly shows a menu, collects input, and performs actions until the user quits.
code documentation
Docstrings and comments inside a program that explain how the code works, what functions expect, and why non-obvious decisions were made.
edge case
An input or situation at the boundary of normal behavior, such as an empty list, a missing file, or a value of zero.
incremental testing
The practice of testing each function as it is written rather than waiting until the entire program is complete.
manual testing checklist
A list of specific inputs and expected results used to verify that an application satisfies its requirements.
menu
A display of numbered or labeled options that a console application presents to the user at each step of the application loop.
persistence
The ability of a program to save data between sessions, usually by writing to a file.
program flow
The sequence of steps a running program follows, from startup through each possible user action to shutdown.
README
A short text file in the same folder as a program that explains how to run it, what it does, and any known limitations.
requirement
A statement of what a program must do, written from the user's perspective before implementation begins.
user documentation
Written instructions for running and using a program, intended for the person who will use it.
user story
A short sentence describing a feature from the user's point of view: "As a user, I want to [do something] so that [I get some benefit]."
