Chapter 27 — Front Matter

Chapter Introduction and Learning Objectives

Objects are not just containers for data. They are participants in a program — things that know their own condition and can take actions based on it.

Chapter 26 introduced the core ideas of object-oriented programming: classes as blueprints, objects as individual instances, the __init__ method for setup, attributes for stored values, and methods for behavior. You wrote your first classes and created your first objects.

This chapter goes deeper. The focus is on object state: the collection of attribute values that an object carries at any moment, and how methods read and change that state over time. You will learn how to write methods that inspect what an object knows, methods that update what an object holds, and how to design objects that behave consistently and clearly.

By the end of this chapter, you should be able to read and trace programs where multiple objects exist and interact, build objects that maintain state across many method calls, work with lists of objects, give objects useful string representations, and decide when a class-based design is genuinely clearer than a dictionary-based one.

This chapter is still introductory. It does not cover inheritance, class methods, static methods, or operator overloading. Those belong to a more advanced course. The goal here is to use classes comfortably for the kinds of programs that appear naturally in a first Python course.

Section 27.1

Object State

When you hold a library card, that card has a state. It might currently be active or expired. It might have two books checked out. It might have a fine of zero dollars or a fine of four dollars. The card's state changes over time as you borrow books, return them, and pay fines.

An object works the same way. Its state is the collection of values stored in its attributes at a given moment. State changes whenever a method modifies an attribute.

Here is a simple LibraryCard class:

Python — LibraryCard class 
class LibraryCard:

    def __init__(self, owner_name):
        self.owner = owner_name
        self.books_checked_out = 0
        self.fine_amount = 0.0
        self.active = True

When you create a card:

card = LibraryCard("Priya Nair")

The object immediately has a state:

AttributeValue
owner"Priya Nair"
books_checked_out0
fine_amount0.0
activeTrue

This state is private to that object. If you create a second card:

card2 = LibraryCard("Omar Castillo")

That object has its own completely separate state. Changing card has no effect on card2.

State is what makes objects useful. A dictionary could hold the same four values. But a class lets you attach methods — behaviors — directly to the data, so the object knows how to manage itself.

Key Idea

Tracing the state of an object across a sequence of method calls is one of the most important skills in object-oriented programming. Before you can debug or extend a class, you must be able to predict what each method call does to the object's attributes. Tracing tables, introduced in the exercises of this chapter, make this concrete.

Section 27.2

Instance Attributes

Instance attributes are the variables attached to a specific object. They are created using self.attribute_name inside the class. Each object gets its own copy.

You saw in Chapter 26 that __init__ creates instance attributes when the object is first built. But instance attributes can also be created or modified in any method — though it is best practice to create all of them in __init__ so you always know what attributes an object has.

Here is a fuller example:

Python — BankAccount with three instance attributes 
class BankAccount:

    def __init__(self, account_holder, starting_balance):
        self.holder = account_holder
        self.balance = starting_balance
        self.transaction_count = 0

Three instance attributes: holder, balance, and transaction_count. Every BankAccount object will have all three.

account_a = BankAccount("Kenji Watanabe", 500.0)
account_b = BankAccount("Amara Diallo", 1200.0)

After these two lines, account_a.holder is "Kenji Watanabe" and account_a.balance is 500.0, while account_b.holder is "Amara Diallo" and account_b.balance is 1200.0. The attributes belong to each object individually.

You can read instance attributes directly:

print(account_a.balance)   # 500.0
print(account_b.holder)    # Amara Diallo

You can also set them directly from outside the class:

account_a.balance = 600.0
Design Caution

Setting attributes directly from outside the class works in Python, but doing it frequently is usually a sign of poor design. Methods give you a controlled, tested way to change state. Direct attribute assignment bypasses any logic the class is meant to enforce. Use methods when the attribute change has rules attached to it.

Section 27.3

Methods That Read State

Some methods exist only to report on an object's current state. They read attributes and return or display information. They do not change anything.

Python — BankAccount with read-only methods 
class BankAccount:

    def __init__(self, account_holder, starting_balance):
        self.holder = account_holder
        self.balance = starting_balance
        self.transaction_count = 0

    def get_balance(self):
        return self.balance

    def get_holder(self):
        return self.holder

    def is_overdrawn(self):
        return self.balance < 0

    def summary(self):
        print(f"Account holder: {self.holder}")
        print(f"Balance: ${self.balance:.2f}")
        print(f"Transactions: {self.transaction_count}")

None of these methods change any attribute. They only report.

account = BankAccount("Kenji Watanabe", 500.0)
print(account.get_balance())     # 500.0
print(account.is_overdrawn())    # False
account.summary()
Output 
500.0
False
Account holder: Kenji Watanabe
Balance: $500.00
Transactions: 0
Terminology

Methods that only read state are sometimes called accessor methods or getter methods in other programming languages. Python does not require those terms, but understanding the concept is useful: some methods observe, some methods change. Keeping them separate in your thinking makes classes easier to design and test.

Section 27.4

Methods That Change State

Other methods modify attributes. These are the actions an object can perform. Each call moves the object to a new state.

Python — BankAccount with deposit and withdraw 
class BankAccount:

    def __init__(self, account_holder, starting_balance):
        self.holder = account_holder
        self.balance = starting_balance
        self.transaction_count = 0

    def deposit(self, amount):
        self.balance = self.balance + amount
        self.transaction_count = self.transaction_count + 1

    def withdraw(self, amount):
        self.balance = self.balance - amount
        self.transaction_count = self.transaction_count + 1

    def get_balance(self):
        return self.balance

    def summary(self):
        print(f"Account holder: {self.holder}")
        print(f"Balance: ${self.balance:.2f}")
        print(f"Transactions: {self.transaction_count}")

Now trace this sequence:

account = BankAccount("Kenji Watanabe", 500.0)
account.deposit(200.0)
account.withdraw(75.0)
account.summary()

Notice how each method call moves the object to a new state. The object remembers every change because it stores the current values in its own attributes between calls. This is the essential power of objects. A plain function cannot remember results between calls unless you pass data back and forth explicitly. An object holds its own data and updates it with every action.

Section 27.5

Validation Inside Methods

Methods can contain logic that checks whether an action should be allowed before carrying it out. This is one of the main reasons to use methods instead of setting attributes directly. A bank account probably should not allow withdrawals larger than the current balance:

Python — withdraw with validation 
def withdraw(self, amount):
    if amount <= 0:
        print("Withdrawal amount must be positive.")
    elif amount > self.balance:
        print("Insufficient funds.")
    else:
        self.balance = self.balance - amount
        self.transaction_count = self.transaction_count + 1

Now the method enforces a rule. You cannot accidentally set the balance to a negative number by calling withdraw() incorrectly. The logic lives in one place, inside the class.

Python — full BankAccount with validation 
class BankAccount:

    def __init__(self, account_holder, starting_balance):
        self.holder = account_holder
        self.balance = starting_balance
        self.transaction_count = 0

    def deposit(self, amount):
        if amount <= 0:
            print("Deposit amount must be positive.")
        else:
            self.balance = self.balance + amount
            self.transaction_count = self.transaction_count + 1

    def withdraw(self, amount):
        if amount <= 0:
            print("Withdrawal amount must be positive.")
        elif amount > self.balance:
            print("Insufficient funds.")
        else:
            self.balance = self.balance - amount
            self.transaction_count = self.transaction_count + 1

    def get_balance(self):
        return self.balance

    def summary(self):
        print(f"Account holder: {self.holder}")
        print(f"Balance: ${self.balance:.2f}")
        print(f"Transactions: {self.transaction_count}")
Testing validation 
account = BankAccount("Amara Diallo", 300.0)
account.deposit(-50)
account.withdraw(500.0)
account.withdraw(100.0)
account.summary()
Output

Key Principle

When you write validation inside methods, you build an object that polices its own integrity. The failed deposit and failed withdrawal did not change the state. Only the valid withdrawal updated the balance and count. This is much safer than relying on the calling code to always pass correct values — which it won't, eventually.

Section 27.6

Objects Interacting with Other Objects

Objects do not have to live in isolation. A method can receive another object as an argument, call methods on that object, or hold a reference to another object as an attribute.

Here is a Student class and a Classroom class that manages students:

Python — Student class 
class Student:

    def __init__(self, name, grade):
        self.name = name
        self.grade = grade

    def is_passing(self):
        return self.grade >= 60

    def summary(self):
        status = "passing" if self.is_passing() else "failing"
        print(f"{self.name}: {self.grade} ({status})")
Python — Classroom class 
class Classroom:

    def __init__(self, course_name):
        self.course = course_name
        self.students = []

    def add_student(self, student):
        self.students.append(student)

    def count_students(self):
        return len(self.students)

    def class_average(self):
        if len(self.students) == 0:
            return 0.0
        total = 0
        for student in self.students:
            total = total + student.grade
        return total / len(self.students)

    def print_roster(self):
        print(f"--- {self.course} ---")
        for student in self.students:
            student.summary()
        print(f"Class average: {self.class_average():.1f}")
Using both classes together 
s1 = Student("Fatima Al-Rashid", 88)
s2 = Student("Lucas Moreau", 54)
s3 = Student("Yuna Park", 73)
s4 = Student("Dev Sharma", 91)

room = Classroom("Introduction to Programming")
room.add_student(s1)
room.add_student(s2)
room.add_student(s3)
room.add_student(s4)

room.print_roster()
Output 
--- Introduction to Programming ---
Fatima Al-Rashid: 88 (passing)
Lucas Moreau: 54 (failing)
Yuna Park: 73 (passing)
Dev Sharma: 91 (passing)
Class average: 76.5
What to Notice
Three key observations about object interaction
  • ReferencesClassroom does not copy student data. It stores references to the Student objects. The same student object lives in both the variable s1 and inside the classroom's list.
  • Reading acrossThe class_average method reaches into each student object to read student.grade. One object's method is free to read the public attributes of another.
  • DelegationThe print_roster method calls each student's summary() method. Classroom delegates the display work to Student. Neither class needs to know the full internal logic of the other.
Section 27.7

Lists of Objects

One of the most practical patterns you will use repeatedly is a list of objects. Many real programs manage collections of similar things: a list of students, a list of accounts, a list of products, a list of tasks.

Python — Task class and a list of tasks 
class Task:

    def __init__(self, description, priority):
        self.description = description
        self.priority = priority
        self.completed = False

    def complete(self):
        self.completed = True

    def display(self):
        status = "Done" if self.completed else "Pending"
        print(f"[{status}] ({self.priority}) {self.description}")


tasks = []

tasks.append(Task("Buy groceries", "low"))
tasks.append(Task("Finish lab report", "high"))
tasks.append(Task("Reply to emails", "medium"))
tasks.append(Task("Review Chapter 27 notes", "high"))

# Mark one task complete
tasks[1].complete()

# Display all tasks
for task in tasks:
    task.display()
Output 
[Pending] (low) Buy groceries
[Done] (high) Finish lab report
[Pending] (medium) Reply to emails
[Pending] (high) Review Chapter 27 notes

Working with lists of objects follows the same patterns as working with lists of integers or strings. You can iterate, filter, sort, and accumulate. The difference is that each element is an object with its own attributes and methods.

Python — filtering and counting with object lists 
def find_high_priority(task_list):
    result = []
    for task in task_list:
        if task.priority == "high":
            result.append(task)
    return result

def count_pending(task_list):
    count = 0
    for task in task_list:
        if not task.completed:
            count = count + 1
    return count

urgent = find_high_priority(tasks)
print("High priority tasks:")
for task in urgent:
    task.display()

print(f"\nPending tasks: {count_pending(tasks)}")
Output 
High priority tasks:
[Done] (high) Finish lab report
[Pending] (high) Review Chapter 27 notes

Pending tasks: 3
Section 27.8

String Representations

When you print a plain object without any special setup, Python shows something like this:

<__main__.BankAccount object at 0x7f3c2a1b4d60>

This is not very helpful. It tells you the class name and a memory address, but nothing about the object's actual state. Python gives you a way to fix this: the __str__ method. When Python needs to convert an object to a string — for printing or for use in an f-string — it calls __str__ if you have defined one.

Python — BankAccount with __str__ 
class BankAccount:

    def __init__(self, account_holder, starting_balance):
        self.holder = account_holder
        self.balance = starting_balance
        self.transaction_count = 0

    def deposit(self, amount):
        if amount > 0:
            self.balance = self.balance + amount
            self.transaction_count = self.transaction_count + 1

    def withdraw(self, amount):
        if 0 < amount <= self.balance:
            self.balance = self.balance - amount
            self.transaction_count = self.transaction_count + 1

    def __str__(self):
        return f"BankAccount({self.holder}, ${self.balance:.2f})"


account = BankAccount("Kenji Watanabe", 500.0)
account.deposit(100.0)
print(account)
print(f"Account details: {account}")
Output 
BankAccount(Kenji Watanabe, $600.00)
Account details: BankAccount(Kenji Watanabe, $600.00)

__str__ is one of Python's special methods, also called dunder methods because of the double underscores on each side of the name. You have already used __init__. __str__ is the most immediately useful beginner one after it.

When you have a list of objects, printing the list without __str__ gives you a list of memory addresses. With __str__, you get readable information:

accounts = [
    BankAccount("Kenji Watanabe", 500.0),
    BankAccount("Amara Diallo", 1200.0),
    BankAccount("Priya Nair", 75.0),
]

for account in accounts:
    print(account)
Output 
BankAccount(Kenji Watanabe, $500.00)
BankAccount(Amara Diallo, $1200.00)
BankAccount(Priya Nair, $75.00)

Section 27.9

Dictionaries Versus Objects

At this stage, you have two tools for grouping related data: dictionaries and objects. Understanding when to use each one will improve your program designs.

Consider storing information about a product. Here is the dictionary approach:

Python — dictionary approach 
product = {
    "name": "Spiral Notebook",
    "price": 3.49,
    "quantity": 50
}

print(product["name"])
product["quantity"] = product["quantity"] - 1

def is_low_stock(p):
    return p["quantity"] < 10

print(is_low_stock(product))

Here is the class approach:

Python — class approach 
class Product:

    def __init__(self, name, price, quantity):
        self.name = name
        self.price = price
        self.quantity = quantity

    def sell_one(self):
        if self.quantity > 0:
            self.quantity = self.quantity - 1
        else:
            print(f"Cannot sell {self.name}: out of stock.")

    def restock(self, amount):
        self.quantity = self.quantity + amount

    def is_low_stock(self):
        return self.quantity < 10

    def __str__(self):
        return f"Product({self.name}, ${self.price:.2f}, qty={self.quantity})"


product = Product("Spiral Notebook", 3.49, 50)
product.sell_one()
print(product)
print(product.is_low_stock())
Output 
Product(Spiral Notebook, $3.49, qty=49)
False
Use a Dictionary When
Data is simple and flat
No behavior is attached to the data. You are building a quick lookup table, counting occurrences, or mapping keys to values. The data will be passed around and read in many places with no logic attached. Data arrives from files or APIs as key-value records.
Use a Class When
Data and behavior belong together
You need to enforce rules about how data changes. You will create many similar objects, each maintaining its own state. The program becomes clearer when you can write account.deposit(100) instead of update_account(account, "deposit", 100).
A Useful Test

If you find yourself writing many functions that all take a dictionary as their first argument and do something to it, that is a sign a class might be cleaner. The dictionary plus its functions is already acting like an object — you just have not given it a class yet. Neither approach is always right. Real programs often use both. A class might hold a dictionary inside it as an attribute. A function might return a list of objects.

Section 27.10

Designing Clear Object Behavior

Writing a class is not only about knowing the syntax. It is about designing something that will be easy to use correctly and hard to use incorrectly. Here are guidelines for designing clear object behavior.

Design Principles
Six guidelines for clear class design
  • One responsibilityA BankAccount should manage its own balance. It should not also manage customer addresses, send emails, or parse CSV files. Keep each class focused on one thing. This is the single responsibility principle applied to objects.
  • All attributes in __init__Do not create new instance attributes in random methods throughout the class. If you look at __init__ and cannot see all the attributes the object will have, the class is harder to understand and debug. Every attribute the object needs should be visible in one place.
  • Methods for state changesUse methods to change state, not direct attribute assignment from outside the class. The more an object controls its own state changes, the more you can trust its state is valid.
  • Write __str__It costs almost nothing and makes testing, debugging, and printing vastly more pleasant. Do it for every class you write.
  • Small focused methodsA method that does three unrelated things is harder to test, understand, and reuse than three methods that each do one thing. If a method is getting long, look for natural sub-tasks to break out.
  • Test each method independentlyBefore connecting objects together, make sure each individual method works correctly on its own. Create a small object, call one method, and check whether the state matches what you expected.
Python — poor design: attribute created outside __init__ 
# Confusing: attributes appear unpredictably
class Task:
    def __init__(self, description):
        self.description = description

    def set_priority(self, p):
        self.priority = p   # appears here for the first time — hard to find
Python — clear design: all attributes in __init__ 
# Clear: all attributes declared upfront
class Task:
    def __init__(self, description):
        self.description = description
        self.completed = False
        self.priority = "medium"
Chapter 27 — Worked Example

A Complete InventoryItem Class

Here is a class built from scratch, following all the principles from this chapter.

Worked Example 27.1
Planning the class

Goal: Represent a single item in a store inventory.

  • Attributes: name, unit price, quantity in stock
  • Actions: restock, sell, apply discount, check low stock
  • Display: readable __str__
Python — complete InventoryItem class 
class InventoryItem:
    """Represents a single product in a store inventory."""

    def __init__(self, name, unit_price, quantity):
        self.name = name
        self.unit_price = unit_price
        self.quantity = quantity

    def restock(self, amount):
        """Add units to inventory."""
        if amount <= 0:
            print("Restock amount must be positive.")
        else:
            self.quantity = self.quantity + amount
            print(f"Restocked {self.name}. New quantity: {self.quantity}")

    def sell(self, units):
        """Remove units from inventory if available."""
        if units <= 0:
            print("Sale quantity must be positive.")
        elif units > self.quantity:
            print(f"Not enough {self.name} in stock. Available: {self.quantity}")
        else:
            self.quantity = self.quantity - units
            total = units * self.unit_price
            print(f"Sold {units} x {self.name} for ${total:.2f}.")

    def apply_discount(self, percent):
        """Reduce unit price by a percentage (0 to 100)."""
        if percent <= 0 or percent >= 100:
            print("Discount must be between 0 and 100 percent.")
        else:
            reduction = self.unit_price * (percent / 100)
            self.unit_price = self.unit_price - reduction
            print(f"New price of {self.name}: ${self.unit_price:.2f}")

    def is_low_stock(self, threshold=10):
        """Return True if quantity is below the threshold."""
        return self.quantity < threshold

    def total_value(self):
        """Return the total value of all units in stock."""
        return self.quantity * self.unit_price

    def __str__(self):
        return (f"InventoryItem({self.name}, "
                f"${self.unit_price:.2f}/unit, "
                f"qty={self.quantity})")
Using the class 
item = InventoryItem("USB Cable", 8.99, 25)
print(item)

item.sell(5)
item.sell(30)
item.restock(20)
item.apply_discount(10)
print(item)
print(f"Total stock value: ${item.total_value():.2f}")
print(f"Low stock? {item.is_low_stock()}")
Output 
InventoryItem(USB Cable, $8.99/unit, qty=25)
Sold 5 x USB Cable for $44.95.
Not enough USB Cable in stock. Available: 20
Restocked USB Cable. New quantity: 40
New price of USB Cable: $8.09
InventoryItem(USB Cable, $8.09/unit, qty=40)
Total stock value: $323.60
Low stock? False

What to Notice

Every state transition is predictable, because every method either validates before acting or reports the problem clearly. The failed sell(30) left the object completely unchanged. The caller knows what happened from the printed message. The object's integrity was never compromised.

Chapter 27 — Back Matter

Chapter Summary

  • Section 27.1Object state is the collection of attribute values an object holds at any moment. State changes as methods are called. Tracing state across a sequence of method calls is one of the most important skills in object-oriented programming.
  • Section 27.2Instance attributes belong to individual objects. Each object gets its own copy. All instance attributes should be created in __init__ so the object's structure is visible in one place. You can read and set attributes directly, but prefer methods when changes have rules attached.
  • Section 27.3Methods that read state inspect attributes and return or display information without modifying the object. Understanding which methods read and which methods change is important for reasoning about what a sequence of calls will do.
  • Section 27.4Methods that change state update one or more attributes. Every update is a state transition. The object tracks its own current condition between calls, which is the essential power of objects over plain functions.
  • Section 27.5Validation inside methods protects the integrity of the object's state. By checking conditions before modifying attributes, a method ensures the object is never left in an invalid state through an incorrect call.
  • Section 27.6Objects can interact by passing one object as an argument to a method of another, or by holding lists of objects as attributes. The Classroom holding Student objects is a typical example. Each class cooperates without needing to know the full internal logic of the other.
  • Section 27.7Lists of objects are one of the most common and practical patterns in object-oriented Python. They work just like lists of any other value, with the added ability to call methods on each element. Filter, count, and accumulate over them using the same loop patterns you already know.
  • Section 27.8The __str__ method defines what an object looks like when printed. Writing a good __str__ costs little effort and makes programs much easier to read, test, and debug, especially when working with lists of objects.
  • Section 27.9Dictionaries are best for simple key-value data with no behavior. Classes are best when data and behavior belong together, when you need validation, and when many similar objects will maintain their own independent state. Real programs often use both.
  • Section 27.10Clear object design means: one responsibility per class, all attributes in __init__, use methods to change state, always write __str__, keep methods small and focused, and test each method independently before composing objects together.

Key Terms

object state
The collection of values stored in an object's instance attributes at a given moment. State changes whenever a method modifies an attribute.
instance attribute
A variable attached to a specific object, created with self.attribute_name inside the class, carrying a separate value for each instance.
accessor method
A method that reads and returns information about an object's state without changing any attribute. Sometimes called a getter method in other languages.
mutator method
A method that changes one or more of an object's instance attributes, moving the object to a new state.
state transition
The change in an object's attribute values caused by calling a method that modifies them. Each mutator call is a state transition.
__str__ method
A special method that defines how an object is converted to a readable string, called automatically by print() and f-strings.
dunder method
Any of Python's special methods with double underscores on each side of the name, such as __init__ and __str__. Short for "double underscore."
validation
Logic inside a method that checks whether an operation is allowed before carrying it out, protecting the object's integrity.
list of objects
A list where each element is an instance of a class. Iterated and processed using the same loop and filter patterns as any other list.
single responsibility
A design principle stating that each class should be responsible for one focused area of the program, not a mixture of unrelated concerns.

class Jar:

    def __init__(self, capacity):
        self.capacity = capacity
        self.contents = 0

    def fill(self, amount):
        space = self.capacity - self.contents
        added = min(amount, space)
        self.contents = self.contents + added

    def empty(self):
        self.contents = 0

    def is_full(self):
        return self.contents == self.capacity

    def __str__(self):
        return f"Jar({self.contents}/{self.capacity})"

class LightSwitch:
