Chapter 20 — Front Matter

Chapter Introduction and Learning Objectives

You now know Python's four main built-in collection types: lists, tuples, sets, and dictionaries. You have also built nested structures and used them to model real information. What you have not yet practiced is the most important skill that comes after knowing the tools: choosing the right tool for the job.

This chapter teaches you how to think before you code. You will compare structures by their properties, practice designing data representations on paper before writing a line of Python, and learn to recognize when a first design needs to be reworked. By the end, you should be able to look at a problem, sketch a data shape, justify your choice, and write programs that are clearer because the data is organized well.

Opening Question

A teacher wants to track which students have submitted an assignment, a store wants to look up a product's price by name, and a researcher wants to count how many times each word appears in a document. Each of these problems could technically be solved with a list — but each one has a different structure that fits it far better. What separates a program that merely works from a program whose data is organized well? By the end of this chapter, you will have a framework for answering that question every time you sit down to write code.

Section 20.1

Data Structure Choice Matters

When you are new to programming, any structure that works feels like a good choice. If you can store ten names in a list, you use a list. If you have seen dictionaries solve a problem like this before, you try a dictionary. The program runs. You move on.

This approach has a real cost. Programs that use the wrong structure for their data become harder to read, harder to fix, and slower to run as they grow. A choice that works for ten items can become genuinely painful with ten thousand. A structure that stores data correctly can still make your code confusing if it hides the meaning of what you are tracking.

Good programmers treat data design as a real step in solving a problem. Before writing code, they ask a short set of questions:

  • What kind of data am I storing?
  • How will I need to access it?
  • Does order matter?
  • Are values allowed to repeat?
  • Will I need to look things up by name or by position?
  • Will this data change, or should it stay fixed?
  • How will I read this structure six weeks from now?

This chapter gives you a framework for answering these questions.

Study Note

None of the structures in this chapter are new. The skill being built here is judgment — the habit of comparing options deliberately instead of defaulting to whichever structure you used last time.

Section 20.2

Reviewing the Four Main Collection Types

Before comparing structures, review what each one is.

List

A list is an ordered, mutable sequence. You can store any mix of values, access them by index, add items, remove items, and change items in place.

scores = [88, 72, 95, 61, 84]
scores.append(77)
scores[0] = 90

Lists are the most general-purpose collection. When you are not sure which structure to use, you will often reach for a list first.

Tuple

A tuple is an ordered, immutable sequence. Once created, it cannot be changed.

dimensions = (1920, 1080)
point = (3.5, -2.1)

Tuples signal that a group of values belongs together as a fixed unit. They are useful for coordinates, settings, records returned from a function, and any case where you want Python to prevent accidental modification.

Set

A set is an unordered collection of unique values. Duplicates are automatically discarded. Sets support fast membership testing and set operations like union and intersection.

seen_ids = {1001, 1002, 1003}
seen_ids.add(1004)
seen_ids.add(1001)   # Already there; set does not change
print(1002 in seen_ids)   # True

Sets are useful when you care about whether something exists, not where it is or how many times it appears.

Dictionary

A dictionary is an unordered collection of key-value pairs. You look up values by key, not by position. Keys must be unique and immutable.

student = {
    "name": "Amara",
    "grade": 10,
    "gpa": 3.7
}
print(student["name"])

Dictionaries are the natural structure for named records, lookup tables, and counting problems. They are the most powerful general-purpose mapping structure in Python.

Section 20.3

Ordered Versus Unordered Data

One of the first questions to ask about your data is whether order matters.

When order matters

  • A sequence of steps that must happen in a specific order.
  • A list of moves in a game, recorded as they happen.
  • A ranked leaderboard where first place is different from tenth place.
  • Lines of text in a file, which must stay in reading order.

For ordered data, use a list or a tuple. Lists are for ordered data that changes. Tuples are for ordered data that should stay fixed.

When order does not matter

  • A collection of student IDs that have submitted an assignment.
  • A set of unique tags applied to a document.
  • A lookup table that maps country codes to country names.
  • A counter that tracks how many times each word appears in a text.

For unordered data, use a set (if you only care about membership) or a dictionary (if each item has an associated value).

A common mistake

Many beginners store unordered data in lists and then write slow, complicated code to search through them.

Checking membership in a list — works, but slow for large lists
submitted_ids = [1001, 1002, 1003, 1004]
if 1002 in submitted_ids:
    print("Already submitted")
Checking membership in a set — faster and clearer
submitted_ids = {1001, 1002, 1003, 1004}
if 1002 in submitted_ids:
    print("Already submitted")

When you only need to know whether something is present and you never need to know where it falls in a sequence, a set is a better fit than a list.

Section 20.4

Unique Versus Repeated Values

Ask yourself: does this data allow duplicates, or must every value appear at most once?

Duplicates are allowed and meaningful

A list of quiz scores for a class allows repeated values. Two students can both score 85. The duplicate values carry real information — two different students earned that score.

quiz_scores = [85, 72, 85, 90, 61, 72, 85]

Here a list is correct. A set would silently discard the duplicates, losing information.

Duplicates must be prevented

A collection of registered usernames should not allow the same username twice. A set of unique items sold today should not count the same item more than once in its membership.

registered_usernames = {"amara2024", "bento_k", "chen_li"}

A set enforces uniqueness automatically. You do not have to write extra code to prevent duplicates.

Duplicates should be counted

Sometimes duplicates carry information not as individual entries, but as counts.

Counting word frequencies
word_counts = {}
words = ["apple", "banana", "apple", "cherry", "banana", "apple"]

for word in words:
    if word in word_counts:
        word_counts[word] += 1
    else:
        word_counts[word] = 1

print(word_counts)
# {'apple': 3, 'banana': 2, 'cherry': 1}

When duplicates are meaningful as frequencies, a dictionary with counts is the right structure.

Section 20.5

Lookup by Position and Lookup by Key

A second major axis of comparison is how you access the data.

Position-based access

Lists and tuples let you access items by their index. Position has meaning. The first item is different from the third item because of where it sits.

temperatures = [72.1, 68.4, 75.0, 80.2, 79.9]
print(temperatures[0])   # First reading
print(temperatures[-1])  # Most recent reading

This works well when items are naturally ordered and position carries meaning: days of the week, steps in a process, readings taken in sequence.

Key-based access

Dictionaries let you access items by a meaningful name. You do not need to remember that the student's GPA is stored at index 2. You look it up by the key "gpa".

student = {"name": "Amara", "grade": 10, "gpa": 3.7}
print(student["gpa"])

Key-based access is clearer whenever the data has named fields. Compare:

Tuple version — you must remember which index means what
student = ("Amara", 10, 3.7)
print(student[2])   # What is index 2? You have to remember.
Dictionary version — the key is self-documenting
student = {"name": "Amara", "grade": 10, "gpa": 3.7}
print(student["gpa"])   # Clear

When names convey meaning more clearly than positions, use a dictionary.

Section 20.6

Mutable and Immutable Data

Mutable data can be changed after it is created. Immutable data cannot.

StructureMutable?
ListYes
DictionaryYes
SetYes
TupleNo

Use immutability as a signal

Tuples are not just a less-capable version of lists. They communicate something: these values belong together and should not change.

A coordinate pair like (latitude, longitude) should be a tuple. You are not going to modify one component of a location mid-program. The pair is a single unit of fixed information.

An RGB color like (255, 128, 0) should be a tuple. Colors in a design palette do not change at runtime.

Configuration settings loaded once at startup can reasonably be stored in a tuple to prevent accidental modification.

WINDOW_SIZE = (1280, 720)      # Tuple: this should not change during the program
player_position = [640, 360]   # List: this changes constantly

Using tuples for fixed data and lists for changing data makes your intent visible in the code itself.

Section 20.7

Modeling Real-World Records

Many programming problems involve tracking records: things with multiple named properties. A student has a name, a grade level, and a GPA. A product has a name, a price, and a quantity in stock. A contact has a first name, a last name, a phone number, and an email address.

Python gives you two main ways to model a single record before you reach classes: a tuple or a dictionary.

Tuple as a record

student = ("Amara", 10, 3.7)

This works, but the meaning of each position is not visible in the code. You must remember the order: name first, grade second, GPA third. This is fragile. If you later add a field in the middle, every piece of code that unpacks the tuple may break.

Tuple records work best when the structure is small, stable, and well-known — like a coordinate or a date.

Dictionary as a record

student = {
    "name": "Amara",
    "grade": 10,
    "gpa": 3.7
}

A dictionary record is more readable. The keys document themselves. You can add a new field without breaking existing code that reads other fields. The cost is slightly more typing.

Rule of Thumb

For most beginner-to-intermediate programs, dictionary records are clearer and safer than tuple records unless the structure is very simple.

Section 20.8

Combining Structures Safely

Real programs rarely use one structure in isolation. They combine them. The most common combinations are described below.

List of dictionaries — a table of records

students = [
    {"name": "Amara", "grade": 10, "gpa": 3.7},
    {"name": "Bento", "grade": 11, "gpa": 3.2},
    {"name": "Chen Li", "grade": 10, "gpa": 3.9}
]

for student in students:
    print(student["name"], student["gpa"])

This models a table where each row is a dictionary. It is one of the most useful patterns in practical Python.

Dictionary of lists — grouped items by category

class_roster = {
    "grade_10": ["Amara", "Chen Li", "Fatima"],
    "grade_11": ["Bento", "Kenji", "Lucia"]
}

print(class_roster["grade_10"])

This models groups. Each key names a category and maps to the list of items in that category.

Dictionary of dictionaries — nested records indexed by key

inventory = {
    "apple": {"price": 0.99, "quantity": 150},
    "banana": {"price": 0.59, "quantity": 80},
    "cherry": {"price": 3.49, "quantity": 40}
}

print(inventory["banana"]["price"])

This models a lookup table where each entry has multiple named properties.

Rules for combining safely

Design Rules
Four rules for combining structures without creating confusion
  • Rule 1Keep nesting shallow. If you need to go three or four levels deep to access a value, the structure is probably too complex. Consider using a class or breaking the data into separate, simpler structures.
  • Rule 2Use consistent shapes. Every dictionary in a list of dictionaries should have the same keys. Inconsistency makes iteration unpredictable and debugging painful.
  • Rule 3Name clearly. A variable named data tells you nothing. A variable named students, inventory, or word_counts tells you what it contains and makes the structure more readable.
  • Rule 4Write a small example before building the full structure. Before you process 500 records, build a version with 3 records and make sure your code works on the small case.
Section 20.9

Data Structure Tradeoffs

No structure is best for everything. Every choice involves tradeoffs.

NeedBest fitWhy
Ordered sequence that changesListMutable, indexed, flexible
Ordered sequence that should not changeTupleImmutable, signals fixed data

Counting occurrencesDictionary with integer valuesNatural accumulator pattern

The Honest Answer

For most beginner Python programs, you will use lists and dictionaries for nearly everything. Lists handle sequences. Dictionaries handle records and lookups. Sets and tuples are valuable but more specialized.

The skill being built in this chapter is not memorizing a formula. It is the habit of pausing before you write code and asking: what shape does this data want to have?

Section 20.10

Sketching Data Before Coding

One of the best habits a programmer can develop is sketching data structures on paper — or in a comment at the top of a file — before writing any code.

This sketch does not need to be formal. It just needs to answer: what does one piece of this data look like? What does a collection of it look like?

Example: planning a contact book

Suppose you are building a contact book that stores names, phone numbers, and email addresses. You want to look up a contact by name.

Sketch 1
List of tuples
  • Sketchcontacts = [("Amara Johnson", "555-1234", "amara@mail.com"), ("Bento Kowalski", "555-5678", "bento@mail.com")]
  • ProblemTo look up Amara, you would have to loop through the entire list comparing names. That is slow and cumbersome for a lookup task.
Sketch 2
List of dictionaries
  • Sketchcontacts = [{"name": "Amara Johnson", "phone": "555-1234", "email": "amara@mail.com"}, {"name": "Bento Kowalski", "phone": "555-5678", "email": "bento@mail.com"}]
  • Better, butEach record is self-documenting. But lookup by name still requires looping.
Sketch 3
Dictionary of dictionaries, keyed by name
  • Sketchcontacts = {"Amara Johnson": {"phone": "555-1234", "email": "amara@mail.com"}, "Bento Kowalski": {"phone": "555-5678", "email": "bento@mail.com"}}
  • ResultNow lookup by name is direct: print(contacts["Amara Johnson"]["phone"])

You would arrive at Sketch 3 faster if you asked the right questions before writing code:

  • How will I look up contacts? By name.
  • Then name should be the key.
  • What information goes with each name? Phone and email.
  • Then each entry should be a dictionary with those two fields.

The sketch is not the program. It is the plan that makes the program obvious.

Section 20.11

Refactoring Data Structures

Sometimes you build a program with one structure and later discover it is the wrong choice. This is normal. The ability to recognize a weak design and improve it is a real programming skill.

Signs that a data structure needs rethinking

  • You loop through a list every time you need to find something by name. A dictionary would make lookup direct.
  • You use a list but never use indexes — you only check if item in list. A set would be more appropriate.
  • You use parallel lists: one for names, one for scores, one for grades. A list of dictionaries would keep related data together.
  • You use a dictionary but keep forgetting which key holds which value because the keys are not descriptive. Rename them.
  • Your nested structure requires three levels of indexing to get to a single value. Flatten or simplify.

Refactoring parallel lists into a list of dictionaries

Before
names  = ["Amara", "Bento", "Chen Li"]
scores = [88, 72, 95]
grades = ["B", "C", "A"]
The Problem

These three lists are coupled. If you sort scores, names and grades go out of sync unless you are careful.

After
students = [
    {"name": "Amara",   "score": 88, "grade": "B"},
    {"name": "Bento",   "score": 72, "grade": "C"},
    {"name": "Chen Li", "score": 95, "grade": "A"}
]

Now each student's data travels together. Sorting by score keeps name and grade attached to the correct student.

Refactoring a lookup list into a dictionary

Before
product_names  = ["apple", "banana", "cherry"]
product_prices = [0.99, 0.59, 3.49]

# To find the price of banana:
index = product_names.index("banana")
price = product_prices[index]
After
prices = {"apple": 0.99, "banana": 0.59, "cherry": 3.49}
price = prices["banana"]

The second version is shorter, clearer, and faster.

Chapter 20 — Back Matter

Chapter Summary

In this chapter you learned how to evaluate data structure choices rather than simply accepting the first structure that comes to mind.

  • Section 20.1Programs that use the wrong structure for their data become harder to read, harder to fix, and slower as they grow. Good programmers ask a deliberate set of questions about order, uniqueness, access, and mutability before writing code.
  • Section 20.2Lists are ordered and mutable. Tuples are ordered and immutable. Sets are unordered and unique. Dictionaries are unordered collections of key-value pairs accessed by key rather than position.
  • Section 20.3Ordered sequences belong in lists or tuples. Unordered collections where position has no meaning belong in sets or dictionaries. Checking membership in a set is faster and clearer than searching a list.
  • Section 20.4Duplicates that are meaningful belong in a list. Duplicates that must be prevented belong in a set. Duplicates that should be counted belong in a dictionary that maps each value to its frequency.
  • Section 20.5Position-based access through lists and tuples works well when position itself carries meaning. Key-based access through dictionaries is clearer whenever data has named fields, since keys document themselves.
  • Section 20.6Lists, dictionaries, and sets are mutable; tuples are not. Using a tuple signals that a group of values belongs together as a fixed unit that should not change during the program.
  • Section 20.7A single record can be modeled as a tuple or a dictionary. Tuple records are fragile because positions are not self-documenting. Dictionary records are usually clearer and safer for most beginner-to-intermediate programs.
  • Section 20.8The most common compound patterns are a list of dictionaries for tables of records, a dictionary of lists for grouped items, and a dictionary of dictionaries for named records indexed by key. Combine structures safely by keeping nesting shallow, using consistent shapes, naming clearly, and testing with small examples first.
  • Section 20.9Every structure involves tradeoffs among order, uniqueness, lookup speed, and mutability. For most beginner Python programs, lists and dictionaries handle the large majority of needs, while sets and tuples serve more specialized purposes.
  • Section 20.10Sketching a data structure before coding — asking how data will be looked up and what fields belong together — leads to better designs faster than writing code first and discovering problems later.
  • Section 20.11Parallel lists, lookup loops over lists, and overly deep nesting are signs that a data structure needs to be refactored. Replacing parallel lists with a list of dictionaries, and replacing lookup lists with dictionaries, produces shorter, clearer, and faster code.

Key Terms

mutable
A data structure that can be changed after it is created. Lists, dictionaries, and sets are mutable.
immutable
A data structure that cannot be changed after it is created. Strings and tuples are immutable.
ordered
A collection where the position of each item is meaningful and preserved. Lists and tuples are ordered.
unordered
A collection where items have no defined position. Sets and dictionaries (before Python 3.7) are unordered. Python 3.7+ dictionaries preserve insertion order but are not typically used for position-based access.
unique
A collection that does not allow duplicate values. Sets enforce uniqueness automatically.
key-value pair
A single entry in a dictionary, consisting of a key used for lookup and an associated value.
record
A collection of named fields describing a single object or entity. Dictionaries are commonly used as records in Python.
list of dictionaries
A list in which each element is a dictionary with the same set of keys, modeling a table of records.
dictionary of lists
A dictionary in which each value is a list, modeling groups of items organized by category.
parallel lists
Multiple lists used together so that items at the same index are related. Generally a design smell; lists of dictionaries are usually cleaner.
refactoring
Restructuring existing code or data to improve clarity, organization, or efficiency without changing what the program does.
data design
The process of choosing and sketching appropriate data structures before writing code.
