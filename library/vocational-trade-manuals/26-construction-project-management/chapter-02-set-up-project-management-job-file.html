Job Overview

This project qualifies you to build and test one controlled project-management job file — the organized document system that every later project in this book files into, references, and updates. The finished work product is one indexed, version-controlled job file, physical or digital, with tested document recovery, a clear separation between active and archived documents, and an issued procedure the project team has acknowledged.

You will select the file platform, set access levels and confidentiality rules, build the complete folder or section structure, establish a naming convention and status terms, create the document register and revision log, load the approved Project 01 scope baseline and supporting documents, create the field-record and log templates every later chapter depends on, test recovery of representative documents, correct defects, back up the system, and issue the procedure to the project team.

This is the second project in the book. It builds directly on the scope baseline approved in Project 01, and it establishes the controlled structure that Project 03's work breakdown structure, Project 05's baseline schedule, Project 07's procurement register, Project 09's meeting records, Project 11's change log, and every later project's records and logs will be filed into.

Where This Skill Fits

The work breakdown structure in Project 03 is filed into the sections this project creates, and its dictionary is cross-referenced to the scope statement, estimate, and permit plan already loaded here. The baseline schedule in Project 05, the procurement register in Project 07, the meeting records in Project 09, the RFI log in Project 10, and the change log in Project 11 all use the document-register, naming-convention, and revision-log conventions established in this project without redefining them.

The active-versus-archive discipline taught here — so that a superseded draft is never mistaken for the current controlling document — is the same discipline the schedule baseline in Project 05 and the change-control system in Project 11 depend on to prevent hidden delay and unauthorized scope expansion.

Scope and Intended Use

This project covers:

  • selecting the approved file platform and defining access levels for owner, management, field staff, accounting, subcontractors, suppliers, designer, and consultants
  • creating confidentiality, cybersecurity, backup, retention, and document-release rules
  • building the complete controlled folder and section structure for contract, design, permits, schedule, procurement, cost, meetings, safety, quality, daily field records, communication, and closeout documents
  • establishing a file-naming convention, status terms, a document register, and a revision log
  • loading the approved Project 01 scope baseline, contract, and supporting Book 23–25 documents, and verifying indexes match the files received
  • creating photograph, field-note, and log templates used by every later project
  • testing recovery of representative current and superseded documents, correcting defects, and backing up the system
  • issuing the job-file procedure to the project team and recording acknowledgment

This project does not authorize the learner to:

  • change, reinterpret, or expand the scope, contract terms, or any approved project document while filing it; the job file organizes controlling documents, it does not create or revise them
  • grant document access that conflicts with the confidentiality, cybersecurity, or contractual notice requirements established by the company or the executed agreement
  • treat a personal or informal storage location as an acceptable substitute for the approved job-file platform
  • delete a superseded document rather than archiving it, except where company retention policy specifically requires deletion

Any conflict between this project's file structure and the company's existing document-control policy must be identified and resolved in writing before the job file is issued to the project team.

Trade Note

A job file is only as reliable as its ability to answer one question under pressure: "what is the current, approved version of this document, right now?" Every step in this project exists to make that question answerable in seconds, not minutes, for every controlled document category the project produces.

Setup Standard

Do not load a single controlled document (Step 24) until the platform, access levels, confidentiality rules, and complete folder or section structure (Steps 1–17) are established. A job file loaded before the structure is complete must be reorganized before documents are considered filed rather than merely stored.

Shop Tip

Test recovery before you trust the system. A job file that looks organized but cannot return one current drawing, one contract document, one inspection record, one photograph, and one pending RFI within a minute, on request, is not yet a working job file — it is only a folder structure.

Active Documents, Superseded Drafts, and the Archive

Every controlled document category in the job file must have exactly one current, active version, and every prior version must be moved to the archive rather than deleted or left alongside the active version. Store the active and archived locations so that a person retrieving a document cannot mistake a superseded draft for the current controlling record.

Recordkeeping Cue — Assign an Identifier Before a Document Is Filed

Do not file a document into the job file without first assigning it a register identifier, date, and revision. A document filed without an identifier cannot be tracked in the document register, cannot be distinguished from a later revision, and cannot be reliably recovered when it is needed.

Critical Version-Control Cue — Separate Active and Archive Locations

Do not store a superseded document in the same folder or section as the current active version. Where the platform does not physically separate them, mark the superseded document's status as superseded in the document register and remove it from any location a field user would search first. A superseded document mistaken for the active version can cause work to proceed from outdated drawings, an outdated schedule, or an outdated scope.

Field Coordination Cue — Confirm Offline or Field Access

Confirm that current drawings and emergency information remain accessible from the jobsite even if the primary platform is temporarily unavailable, whether through a printed set, a synced offline copy, or an approved field device. Do not rely solely on a connection-dependent system for information required during an emergency or an inspection.

Company Policy Cue — Follow the Company's Retention and Backup Rules

Where the company's policy specifies a particular backup frequency, retention period, or cybersecurity control, follow that policy exactly rather than substituting a personal or informal method. Confirm the policy before the folder structure is built, since retrofitting retention rules after documents are loaded is significantly more error-prone.

E. Preparation — Select the Platform and Set Access Rules

  1. Select the approved file platform, physical binder, cloud system, project-management application, or coordinated hybrid system. Confirm the selection matches company policy before proceeding.
  2. Identify access levels for owner, management, field staff, accounting, subcontractors, suppliers, designer, and consultants. Record each role's permitted access individually.
  3. Create confidentiality, cybersecurity, backup, retention, and document-release rules. Base each rule on the company's confirmed policy, not an informal assumption.
  4. Create the project root folder using the approved project number and name. Confirm the number and name match the executed agreement and permit application exactly.
  5. Create a project-information index with address, client, contacts, contract number, permit numbers, and emergency information. Verify each entry against the source document it comes from.

F. Build the Controlled Section Structure

  1. Create controlled sections for contract, scope, estimate, budget, insurance, bonds, licenses, and notices. Follow the section schedule in Section B above.
  2. Create sections for architectural, structural, civil, landscape, plumbing, electrical, HVAC, energy, and specialty drawings. Match section names to the disciplines present in the Book 23 drawing package.
  3. Create sections for specifications, addenda, bulletins, sketches, calculations, submittals, product data, samples, and shop drawings. Keep each document type in its own labeled subsection.
  4. Create sections for permits, inspections, test reports, correction notices, approvals, and certificates. Match section names to the Book 25 permit-and-inspection categories.
  5. Create sections for WBS, schedule, lookaheads, procurement, deliveries, equipment, and logistics. Reserve these sections for Projects 03 and 05–07 without populating them yet.
  6. Create sections for subcontracts, purchase orders, supplier quotes, invoices, payment applications, lien releases, and job-cost reports. Reserve these sections for later commercial records.
  7. Create sections for meetings, RFIs, clarifications, decisions, changes, allowances, alternates, and claims-related notices. Reserve these sections for Projects 09–11.
  8. Create sections for safety plans, orientations, inspections, incidents, competent-person records, and emergency procedures according to access rules. Apply the confidentiality rules from Step 3 to this section specifically.
  9. Create sections for quality plans, checklists, mockups, deficiency logs, photographs, punch lists, and acceptance records. Reserve these sections for Projects 13–14.
  10. Create sections for daily reports, manpower, weather, visitors, deliveries, equipment, delays, and field notes. Confirm the field team can access this section from the jobsite.
  11. Create sections for owner communication, trade communication, supplier communication, design communication, and formal notices. Reserve these sections for Project 17.
  12. Create closeout sections for warranties, manuals, training, spare materials, keys, as-built notes, final photographs, approvals, and owner handoff. Reserve these sections for Projects 18–19.

G. Naming, Registers, and Loading Source Documents

  1. Establish a file-naming convention using date, document type, discipline, subject, revision, and status. Record the convention in writing before any document is named under it.
  2. Establish status terms such as draft, submitted, under review, approved, approved as noted, rejected, superseded, record, and closed. Use only these defined terms in the document register.
  3. Create a document register with identifier, title, originator, date, revision, status, distribution, and current-file location. Begin the register before any document is loaded.
  4. Create a revision log showing what changed, why, by whom, and when it became effective. Link each revision-log entry to its document register identifier.
  5. Create a transmittal system for controlled document distribution. Confirm the transmittal record captures date, recipient, and document identifier.
  6. Create separate active and archive locations so superseded documents cannot be mistaken for current documents. Confirm the separation is structural, not merely a label.
  7. Upload or file the approved scope, contract, Book 23 drawing package, Book 24 estimate, and Book 25 compliance package. Assign a register identifier to each document before filing it.
  8. Verify that drawing and specification indexes match the files received. Record any missing or unindexed document as an open item.
  9. Create a photograph protocol with date, location, direction, subject, phase, and related issue identifier. Confirm the protocol matches the field-note form created in the next step.
  10. Create a field-note form that distinguishes observation, conversation, decision, direction, and follow-up. Confirm the form's categories match the definitions in the Reference Conventions table.
  11. Create logs for RFIs, submittals, procurement, deliveries, changes, allowances, inspections, deficiencies, punch items, and closeout. Create each log as an empty, correctly structured template for the projects that will populate it.
  12. Create a required-document matrix showing who creates, reviews, approves, distributes, and archives each record. Assign one responsible role to each function per document category.

H. Verify, Test, and Issue the Job File

  1. Establish daily, weekly, monthly, milestone, and closeout filing responsibilities. Assign each responsibility to a named role, not a general team.
  2. Create an offline or field-access method for current plans and emergency information. Confirm the method functions without a live connection where required.
  3. Test recovery of one current drawing, one contract document, one inspection record, one photograph, and one pending RFI. Record the time and result of each retrieval attempt.
  4. Test that a superseded document is clearly separated from the active set. Confirm a test user cannot mistake the superseded document for the current version.
  5. Correct broken links, duplicate files, ambiguous names, missing permissions, and inaccessible records. Re-run the affected recovery test after each correction.
  6. Back up the system and test restoration of a sample file. Record the backup method, frequency, and the result of the restoration test.
  7. Issue the job-file procedure to the project team and record acknowledgment. Confirm every stakeholder identified in Step 2 has received and acknowledged the procedure.

Common Job File Inspection

Inspect every job file using the same checks regardless of project size or complexity.

  • Access check: confirm every stakeholder role has a defined access level and that no role has access exceeding the confidentiality rules established in Step 3.
  • Structure check: confirm every section in the section schedule exists and contains the correct document category.
  • Naming and register check: confirm every filed document follows the naming convention and has a complete document register entry.
  • Version-control check: confirm every superseded document is archived and separated from the active set.
  • Recovery check: confirm the five representative documents in Step 32 and the superseded-document test in Step 33 were successfully completed.
  • Backup check: confirm the backup was tested and a sample file restoration succeeded.

Complete the Job File Inspection Checklist in the Inspection Checklist section below before beginning Project 03.

Failure Points and Troubleshooting

A Field User Cannot Find the Current Drawing Under Time Pressure

Probable cause: the naming convention from Step 18 was not followed consistently, or the active and archive locations from Step 23 are not structurally separated.
Confirmation: check the document register for the drawing's current status and location, and confirm the naming convention was applied to the actual filed document.
Correction: rename or relocate the document to conform to the convention, update the register, and repeat the recovery test from Step 32 for that document type.
Reinspection: repeat the recovery check across all five representative document types, not only the affected drawing.

A Superseded Document Is Mistaken for the Current Version

Probable cause: the superseded document was not moved to the archive location, or its status was not updated to superseded in the document register.
Confirmation: check the document register status field and the document's physical or digital location against the active-versus-archive rule from Step 23.
Correction: move the document to the archive location, update its status, and confirm the active location now contains only the current version.
Reinspection: repeat the superseded-document separation test from Step 33.

A Stakeholder Has Access to Confidential Information Outside Their Role

Probable cause: the access levels from Step 2 were not correctly applied to the platform's permission settings, or a section was created without the confidentiality rule from Step 3.
Confirmation: check the platform's actual permission settings against the recorded access level for the affected role.
Correction: correct the permission setting immediately, document the correction, and notify the responsible manager of the exposure and correction.
Reinspection: repeat the access check across every role and section, not only the affected one.

A Backup Restoration Fails or Returns an Incomplete File

Probable cause: the backup method or frequency from Step 35 does not match the platform's actual behavior, or the tested sample file was not representative of the full document set.
Confirmation: re-run the backup and restoration test using a different sample file and record the result.
Correction: correct the backup configuration, confirm with the platform provider or company IT policy if needed, and repeat the restoration test until it succeeds.
Reinspection: repeat the backup and restoration test on a schedule consistent with company policy.

A Trade or Supplier Cannot Locate the Document They Were Told to Reference

Probable cause: the transmittal record from Step 22 was not completed, or the required-document matrix from Step 29 does not identify who distributes that document category to external parties.
Confirmation: check the transmittal record for the document and confirm whether the external party was included in the distribution.
Correction: issue the transmittal, confirm receipt, and update the required-document matrix to assign clear distribution responsibility for that category going forward.
Reinspection: repeat the transmittal check for the affected document category across all external parties who require it.

Field Staff Continue Using an Informal Storage Location Instead of the Job File

Probable cause: the job-file procedure from Step 36 was not issued or acknowledged by the affected field staff, or the approved platform is less accessible in the field than the informal alternative.
Confirmation: check the acknowledgment record from Step 36 and interview field staff about the offline or field-access method from Step 31.
Correction: reissue the procedure, correct the field-access method if it is genuinely impractical, and confirm acknowledgment from every affected field staff member.
Reinspection: repeat the field-access check and confirm documents created after the correction appear in the job file rather than the informal location.
