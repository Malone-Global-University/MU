Job Overview

This project qualifies you to convert an approved residential scope statement into a work breakdown structure — the controlling document that divides the project into deliverable-based work packages that can be assigned, scheduled, cost-coded, inspected, and closed. The finished work product is one approved, version-controlled WBS supported by a WBS dictionary that defines every package's scope, deliverable, exclusions, responsible party, references, acceptance criteria, and closeout requirements.

You will begin with the approved scope baseline from Project 01, divide the project into phases and work packages, define a measurable deliverable for every lowest-level package, assign responsibility and cost codes, link each package to the relevant drawing, specification, and estimate reference, identify concealed, long-lead, and schedule-critical packages, and reconcile the completed structure against the scope statement, the estimate, and the permit-and-inspection plan before obtaining approval.

This is the third project in the book. It converts the scope baseline from Project 01, organized in the job file from Project 02, into the deliverable structure that every later planning document depends on: the sequencing logic in Project 04, the baseline schedule in Project 05, the two-week lookahead in Project 06, procurement in Project 07, and cost coding in Project 15 all trace their structure back to the WBS produced here.

Where This Skill Fits

The WBS produced here is the structural link between the approved scope and every schedule, procurement, cost, and quality document that follows. The sequencing logic in Project 04 orders the exact work packages defined here. The baseline schedule in Project 05 assigns durations and dates to these same packages. The job-cost system in Project 15 aligns cost codes assigned here with commitments, actual cost, and billing.

The discipline of defining a measurable deliverable for every lowest-level package — taught in this chapter — is the same discipline that later prevents vague schedule activities such as "work on house" from entering the baseline schedule.

Scope and Intended Use

This project covers:

  • beginning with the approved Project 01 scope baseline and current project documents
  • identifying the final project deliverable and dividing the project into major phases
  • breaking each phase into systems, areas, floors, rooms, or work packages that can be managed independently
  • identifying demolition, temporary work, permit, submittal, procurement, and trade installation packages
  • defining a measurable deliverable and clear boundaries for every lowest-level work package
  • identifying owner work, utility work, separate contracts, allowances, alternates, and unit-price work within the structure
  • assigning one primary responsible party, cost codes, schedule identifiers, and procurement identifiers to each package
  • linking each package to the relevant drawing, specification, scope, and estimate reference
  • identifying quality, inspection, concealment, long-lead, high-risk, and closeout requirements for each package
  • creating a WBS dictionary and reconciling the structure against the scope, estimate, and permit plan before approval

This project does not authorize the learner to:

  • redesign, add to, or remove approved project scope while building the WBS; the WBS structures the approved scope, it does not change it
  • assign dates, durations, or calendar sequence to any work package; date assignment belongs to Project 04 and Project 05
  • invent cost-code values, markup rules, or accounting categories that conflict with the company accounting system
  • approve the WBS baseline without the review and authority steps identified in Step 31 of the procedure below

Any conflict discovered between the scope statement, drawings, specifications, or estimate while building the WBS must be identified and resolved in writing before the WBS baseline is approved.

Trade Note

Every work package in this chapter is built from the approved scope baseline — never from an assumed understanding of "what usually happens" on a similar project. Treating an unreviewed habit or shortcut structure as controlling, rather than the approved scope and current project documents, is the single most common source of the omitted-work and duplicated-cost problems diagnosed in this chapter's troubleshooting section.

Setup Standard

Do not begin assigning cost codes and responsibility (Steps 15–17) until the phase breakdown and package identification (Steps 1–10) are complete. A WBS built out of sequence must be revised before it is submitted for review.

Shop Tip

Write every lowest-level package deliverable as a completed result, not an activity. "Kitchen cabinets installed, leveled, and adjusted" is a deliverable. "Install cabinets" describes an activity but leaves the finished condition undefined.

Package Boundaries and Deliverables

Every work package must have a start condition, a finish condition, and a measurable completed deliverable. Define package boundaries so that no scope item from the approved statement is omitted and no scope item is counted inside more than one package. A package left ambiguous at its boundary becomes either omitted work discovered in the field or duplicated cost discovered in the job-cost report.

Documentation Cue — Record the Baseline Date, Revision, and Approver

Every version of the WBS must carry a baseline date, a revision number, and the name and role of the approver. Do not distribute an undated or unnumbered draft as though it were the approved baseline. Recording only the current version, without the revision history, allows a superseded draft to be mistaken for the controlling structure later in the project.

Critical WBS Cue — No Package Without an Assigned Deliverable

Do not create a work package that lacks a measurable, completed deliverable. A package defined only by a vague verb such as "work on house" or "finish interior" cannot be scheduled, cost-coded, inspected, or closed with confidence. Revise any such package before the WBS is submitted for review.

Field Coordination Cue — Review With Field Supervision and Key Trades

Review the draft WBS with field supervision, the estimator, the scheduler, procurement, accounting, quality, safety, and key trades as applicable before freezing the baseline. A structure built without field review often groups packages in a way that does not match how the work is actually sequenced or supervised.

Cost-Code Instruction Cue — Follow the Company Accounting System Exactly

Assign cost codes using the company accounting system's exact structure. Do not invent a parallel cost-code scheme for convenience. A work package cost-coded outside the accounting system's structure cannot be reconciled with the job-cost report in Project 15.

E. Preparation — Establish the Deliverable and Phase Structure

  1. Begin with the approved scope baseline and current project documents. Confirm the version in use matches the current baseline recorded in the job file.
  2. Identify the final project deliverable as WBS level one. State the completed project as the single top-level deliverable every package rolls up to.
  3. Divide the project into major phases such as preconstruction, mobilization, sitework, structure, enclosure, MEP, interiors, exterior work, commissioning, and closeout. Use these as the starting phase list.
  4. Adjust phase names to the actual project rather than forcing unused categories. Remove or rename a phase that does not apply to the approved scope.
  5. Break each phase into systems, areas, floors, rooms, or work packages that can be managed independently. Divide each phase to a level field supervision can assign and track.
  6. Identify demolition, temporary work, protection, access, and enabling work as separate deliverables. Do not bury enabling work inside a permanent-work package.
  7. Identify permit, submittal, procurement, utility, inspection, testing, and owner-decision packages. Record each as its own work package rather than an implied task.
  8. Identify each trade installation package from start condition through accepted completion. State the condition that begins the package and the condition that completes it.
  9. Include layout, material handling, installation, testing, correction, cleanup, and handoff within each package where applicable. Confirm the package deliverable accounts for all of these where relevant.
  10. Define a measurable deliverable for every lowest-level work package. State the deliverable as a completed, verifiable result.

F. Define Package Boundaries and Third-Party Work

  1. Avoid vague verbs that do not describe a completed result. Replace activity-only phrasing with a stated finished condition.
  2. Define package boundaries so work is neither omitted nor counted twice. Check adjoining packages for overlap and gaps at every boundary.
  3. Identify owner work, utility work, separate contracts, and excluded work in the structure. Mark each item with its correct category so it is not mistaken for a contractor-responsible package.
  4. Identify allowances, alternates, unit-price work, and potential changes as controlled branches rather than hidden tasks. Give each its own identifiable branch in the structure.
  5. Assign one primary responsible party to each work package. Confirm the assignment matches the responsibility matrix from Project 01.
  6. Identify supporting, reviewing, approving, inspecting, and informed parties. Record these roles alongside the primary responsible party for each package.
  7. Link each work package to the relevant drawing, specification, scope, and estimate reference. Record the exact document and section reference, not a general description.

G. Assign Codes and Identify Risk Factors

  1. Assign cost codes aligned with the Book 24 estimate and company accounting system. Use the accounting system's exact code structure.
  2. Assign schedule activity identifiers that can be rolled up to the WBS. Confirm each identifier maps back to exactly one WBS package.
  3. Assign procurement package identifiers for material-dependent work. Link each material-dependent package to its procurement identifier for use in Project 07.
  4. Assign quality-control and inspection requirements to each applicable work package. Record the requirement at the package level rather than as a separate unlinked list.
  5. Identify required submittals, samples, shop drawings, mockups, and owner selections. Record these against the specific package they affect.
  6. Identify predecessor inputs and successor handoffs without yet assigning dates. Record the logical relationship only; leave date assignment to Project 04 and Project 05.
  7. Identify work that will be concealed and requires preclosure documentation. Flag each concealed package for the quality plan in Project 13.
  8. Identify long-lead, high-risk, high-value, safety-critical, and schedule-critical packages. Record the reason for each flag.
  9. Identify closeout deliverables generated by each trade or supplier. Record these against the applicable package for use in Project 18.

H. Build the Dictionary, Reconcile, and Approve the WBS

  1. Create a WBS dictionary defining code, title, scope, deliverable, exclusions, owner, references, acceptance, and closeout for each package. Complete one dictionary entry per work package without exception.
  2. Compare the WBS with the scope statement and confirm every scope item appears. Record any scope item not represented by a package.
  3. Compare the WBS with the estimate and investigate cost codes without work packages or work packages without budget. Record every mismatch found.
  4. Compare the WBS with the permit and inspection plan. Confirm every regulated package is represented and correctly flagged.
  5. Review the WBS with field supervision, estimator, scheduler, procurement, accounting, quality, safety, and key trades as applicable. Distribute the draft far enough in advance for informed review.
  6. Revise packages that are too broad to control or too detailed to manage efficiently. Adjust package size based on reviewer feedback before freezing the baseline.
  7. Freeze the approved WBS baseline and assign its revision. Record the baseline number and date directly on the WBS-approval record.
  8. Distribute the WBS and dictionary to the project team. Confirm distribution against the stakeholder list from Project 01.
  9. Require approved scope changes to update the WBS before schedule and budget changes are finalized. State this rule directly in the WBS documentation.

Common WBS Inspection

Inspect every work breakdown structure using the same checks regardless of project size or complexity.

  • Deliverable check: confirm every lowest-level package has a measurable, completed deliverable stated as a result, not an activity.
  • Boundary check: confirm no scope item is omitted and no scope item is counted inside more than one package.
  • Responsibility check: confirm exactly one primary responsible party is assigned to each package.
  • Code check: confirm every package carries a cost code, schedule identifier, and, where applicable, a procurement identifier aligned with the estimate and accounting system.
  • Reference check: confirm every package is linked to its governing drawing, specification, scope, and estimate reference.
  • Reconciliation check: confirm the WBS has been compared against the scope statement, estimate, and permit plan and that every identified gap has been resolved.
  • Approval check: confirm written approval was obtained from the authorized party and that the baseline number and date are recorded before distribution.

Failure Points and Troubleshooting

A Scope Item Is Discovered With No Work Package During Construction

Probable cause: the reconciliation against the scope statement in Step 28 was not completed, or the item was assumed to be implied within another package rather than given its own deliverable.
Confirmation: re-run the comparison between the current WBS and the current approved scope statement and check whether the item appears in any package's dictionary entry.
Correction: create the missing work package, assign responsibility and codes, and route the WBS revision through the required review before it is used for scheduling or procurement.
Reinspection: repeat the full reconciliation check between the WBS and the scope statement.

The Same Work Is Billed or Scheduled Under Two Different Packages

Probable cause: the package boundaries in Step 12 were not clearly defined, or two overlapping packages were created for the same physical work during the phase breakdown.
Confirmation: check the WBS dictionary entries for both packages and compare their stated deliverables and boundaries.
Correction: merge or redefine the overlapping packages, correct the associated cost codes, and update the schedule and cost documents that reference the affected packages.
Reinspection: repeat the boundary check across the full phase, not only the two affected packages.

Two Parties Each Assume the Other Owns a Work Package

Probable cause: the package was assigned no responsible party in Step 15, or it lists a general party such as "trades" instead of one primary responsible party.
Confirmation: check the WBS dictionary entry for the package and confirm whether exactly one primary party is assigned.
Correction: assign one primary responsible party to the package, document the decision, and distribute the updated WBS to all affected parties.
Reinspection: repeat the responsibility check across the full structure, not only the disputed package.

A Cost Code Has No Corresponding Work Package, or the Reverse

Probable cause: the reconciliation against the estimate in Step 29 was not completed, or a cost code was assigned without confirming a matching package exists in the structure.
Confirmation: re-run the comparison between the current WBS cost codes and the current estimate cost codes and check the reconciliation log for the item.
Correction: resolve the discrepancy by creating the missing package, correcting the cost code, or removing the unsupported code, and update both documents.
Reinspection: repeat the full reconciliation check between the WBS and the estimate.

A Regulated Work Package Is Missing Its Inspection Requirement

Probable cause: the reconciliation against the Book 25 permit-and-inspection plan in Step 30 was not completed, or the package was not flagged as concealed or inspection-required in Step 24.
Confirmation: check the permit-and-inspection plan for the specific inspection and confirm whether the corresponding WBS package carries the correct flag.
Correction: add the missing inspection requirement to the package's dictionary entry and notify the quality plan and inspection log in Project 13 and Project 16.
Reinspection: repeat the permit-and-inspection reconciliation check for the full structure.

Field Staff Are Working From a Superseded WBS Draft

Probable cause: the superseded draft was not moved to the archive location, or the approved baseline was not distributed to every party identified in Step 34.
Confirmation: check the document register for the baseline number and revision currently in the active location, and confirm distribution records against the project team list.
Correction: remove the superseded draft from active use, redistribute the current approved baseline, and confirm receipt from affected parties.
Reinspection: repeat the approval and distribution check for the current baseline.

A Work Package Is Too Broad to Assign or Too Detailed to Manage

Probable cause: the phase breakdown in Steps 3–5 was not carried to a level field supervision can assign and track, or Step 31's review did not identify the sizing problem before the baseline was frozen.
Confirmation: ask field supervision whether the package can be assigned to one crew with one measurable completion condition; a package that cannot be answered clearly is sized incorrectly.
Correction: split an overly broad package into smaller deliverable-based packages, or combine overly detailed packages that cannot be efficiently tracked separately, and update the affected cost and schedule codes.
Reinspection: repeat the deliverable and boundary checks on the revised packages.
